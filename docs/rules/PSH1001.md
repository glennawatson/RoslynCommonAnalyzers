# PSH1001 — Avoid allocating zero-length arrays

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Allocations |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled (where the API exists) |
| Options | `performancesharp.prefer_collection_expressions` (default `true`) |

## Cause

A zero-length array is allocated with `new T[0]` or `new T[] { }`.

```csharp
// 👎 reported
int[] none = new int[0];
string[] names = new string[] { };
int[][] table = new int[0][];

// 👍 C# 12+ target-typed positions (default)
int[] none = [];
string[] names = [];
int[][] table = [];

// 👍 everywhere else (or with the option off)
var buffer = System.Array.Empty<int>();
Send(System.Array.Empty<byte>());
```

## Rule description

A zero-length array can never hold an element, yet every `new T[0]` evaluation
allocates a fresh object. `Array.Empty<T>()` returns one shared cached instance
per element type — and on C# 12+ an empty collection expression (`[]`) targeting
an array type **compiles to exactly that same shared instance**, so both forms
avoid the allocation entirely.

An array creation is reported only when **all** of the following hold:

- its first rank has a single size that is the literal `0` with no initializer
  (or an empty one), or its sizes are omitted and the initializer contains zero
  expressions (`new T[] { }`);
- it is not inside an attribute argument, where a method call is not a valid
  constant; and
- the element type can be a type argument — pointer, function-pointer, and
  ref-struct element types are skipped.

Multi-dimensional creations (`new T[0, 0]`) are not reported because neither
replacement can produce them. A jagged `new T[0][]` **is** reported.

### Which replacement the fix offers

With `performancesharp.prefer_collection_expressions = true` (the default) on
C# 12 or newer, the fix emits `[]` when the creation sits in a position where
the collection expression unambiguously produces the same array:

- an initializer of an **explicitly array-typed** local, field, or property;
- a `return` statement or expression body whose declared return type is that
  array type; or
- a simple assignment whose target has exactly the created array type.

Everywhere else the fix emits `System.Array.Empty<T>()` instead, because `[]`
would not compile or could change meaning there:

- `var` declarations (no target type);
- **arguments** — `[]` can change overload resolution when several collection
  overloads exist;
- covariant targets (`object[] xs = new string[0];`) — `[]` would produce an
  `object[]` instance instead of the original `string[]`;
- member access on the creation, `foreach` sources, and similar non-target-typed
  positions.

Set `performancesharp.prefer_collection_expressions = false` to always get
`System.Array.Empty<T>()`. On language versions before C# 12 the fix always uses
`Array.Empty<T>()` regardless of the option.

### Availability detection

This rule targets a **runtime API**, so it is gated on `System.Array` exposing
an `Empty` member in the referenced framework (probed once per compilation). On
frameworks without the API it reports nothing rather than suggesting a call
that does not exist.

## How to fix violations

Apply the code fix: it inserts `[]` in safe target-typed positions (C# 12+,
option on) and `System.Array.Empty<T>()` — reusing the creation's original
element type syntax — everywhere else.
