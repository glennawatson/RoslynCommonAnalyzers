# PSH1007 - Pass large readonly structs by in reference

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Allocations |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `readonly struct` larger than the configured size is passed by value, in a signature whose author
can change it to `in` without breaking anything.

```csharp
// Snapshot is a 40-byte readonly struct.
internal static long Score(Snapshot snapshot) => snapshot.Total; // PSH1007
```

## Rule description

Passing a large struct by value copies every byte of it at every call. An `in` parameter passes a
readonly reference instead, keeping the same "the callee cannot change my value" contract for a
fraction of the cost.

```csharp
internal static long Score(in Snapshot snapshot) => snapshot.Total;
```

The rule is deliberately hard to trigger, because most "just pass it by `in`" advice is wrong. `in`
is a pessimization at least as often as it is a win, and it is frequently not even legal. Everything
below is a case the rule stays quiet on.

### Where `in` would be slower

- **Small structs.** A struct of 16 bytes or fewer is passed in **registers** by the System V
  (Linux/macOS) and ARM64 ABIs. Forcing it behind an `in` makes the caller spill it to the stack and
  pass a pointer — strictly worse. The threshold defaults to 32 bytes and has a **hard floor of 17**:
  configuring a smaller value raises it back to the floor rather than honouring it, because the same
  assembly runs on every ABI.
- **Structs that are not `readonly`.** The compiler cannot prove a member access leaves an `in`
  parameter alone, so it copies the whole struct defensively before each one — costing more than the
  copy it saved. That is [PSH1003](PSH1003.md)'s complaint, and this rule never suggests walking into
  it.
- **`ref struct` types**, including `Span<T>` and `ReadOnlySpan<T>`. These are already a reference to
  somewhere else; wrapping one in another reference buys an indirection and nothing else. Worse, it
  changes ref-safety: an `in ReadOnlySpan<T>` may alias its argument's storage, so callers that
  compiled fine before can stop compiling.
- **The SIMD types** — `Vector<T>`, `Vector64/128/256/512<T>`, `Vector2/3/4`, `Quaternion`, `Plane`,
  `Matrix3x2`, `Matrix4x4`. `Vector256<T>` and `Vector512<T>` in particular are readonly structs
  comfortably over the threshold, so a size-only rule flags them. The entire BCL vector surface —
  including the 64-byte `Matrix4x4` — passes these by value and never by `in`. They are built for
  code that inlines, where an `in` buys nothing.
- **Cheap framework handles** — `CancellationToken`, `Memory<T>`, `ReadOnlyMemory<T>`, `ValueTask`,
  `ImmutableArray<T>`. Small, and already a reference to their real payload.

### Where `in` would not compile

The rule only reports a signature that can actually take an `in`. Each of these is a compiler error,
so reporting it would be advice that does not build:

- An **`async` method** (CS1988) or an **iterator** (CS1623) cannot have an `in` parameter at all.
- A parameter **captured by a lambda, local function, or query** cannot become a reference (CS1628).
- A parameter the body **assigns** (CS8331) or **passes as a `ref`/`out` argument** (CS8329) is
  written to, and an `in` is readonly.
- An **override** or an **interface implementation** must keep the signature it inherits. So must a
  **partial** member's other half, and an **`extern`** one matches a native API.
- An **attribute constructor** would break every use of the attribute (CS8358).
- A **primary constructor** parameter cannot be `in` and still be read from an instance member
  (CS9109), which is what the parameter is for.

`virtual` and `abstract` members, operators, delegates, lambdas, and `[UnmanagedCallersOnly]` methods
are left alone too — each is a contract that something else depends on the shape of.

### Public API

Adding `in` changes a signature from `T` to `T&`, which is a **binary break**: already-compiled
callers stop working. So by default the rule only reports members that are **not** externally
visible. Set `in_parameter_include_public_api = true` if you own every caller — an application rather
than a library — and want the public surface reported too.

## How to fix violations

Add `in` to the parameter. Callers do not have to change, because `in` is optional at the call site.

## When to suppress

Two cases the analyzer cannot see, both of which the compiler catches the moment you apply the change:

- The method is **converted to a delegate** (a method group, `Func<>`, `Comparison<T>`) anywhere in
  the solution. Ref-kind has to match, so the conversion stops compiling (CS0123).
- The method is called with a **`dynamic` argument**. That still compiles, then throws
  `RuntimeBinderException` at run time.

Note also that where callers pass a **property** or a **literal**, the compiler materialises a
temporary at the call site and copies it anyway, so the `in` saves nothing there.

## Configuration

```ini
# Rule-specific
performancesharp.PSH1007.in_parameter_minimum_size = 32
performancesharp.PSH1007.in_parameter_excluded_types = My.Lib.Handle, OtherStruct
performancesharp.PSH1007.in_parameter_include_public_api = false

# Project-wide
performancesharp.in_parameter_minimum_size = 32
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `in_parameter_minimum_size` | positive integer (estimated bytes) | `32` | The smallest struct worth an `in`. Values below `17` are raised to it. |
| `in_parameter_excluded_types` | comma-separated type names | built-ins only | Types the rule leaves alone. Matched on the bare or the full name. |
| `in_parameter_include_public_api` | `true`, `false` | `false` | Report externally visible members, accepting the binary break. |

The size is **estimated** from the declared fields with sequential-layout padding, because Roslyn does
not expose runtime layout. A type whose size cannot be determined is never reported.

## Performance

The clean path is syntactic: a parameter that already carries a modifier, or whose type is spelled
with a built-in keyword, is rejected without binding anything — which is nearly every parameter in a
codebase. Only a parameter whose type turns out to be a large readonly struct pays for the size
estimate, the interface walk, and the body scan. An EventPipe CPU trace puts the analyzer at 0.2% of
sampled duration on a clean corpus.
