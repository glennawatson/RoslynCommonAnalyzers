# PSH1017 — Properties should not copy a collection on every read

**Category:** Allocations
**Severity:** Warning
**Code fix:** None — caching the result, returning a read-only view, or turning the
property into a method is a design decision the author has to make.

A property reads like a field. Callers put it in a loop condition, index it in the
body, and pass it around without ever suspecting they are paying for it. A getter
that ends in `ToArray()`, `ToList()`, or a clone allocates and copies the whole
collection on *every* read, so the innocent-looking loop below copies `n` elements
`2n + 1` times:

```csharp
for (var i = 0; i < Items.Count; i++)   // one copy per iteration
{
    Use(Items[i]);                      // and another
}
```

## What is reported

The getter's returned expression — the expression body, the `get =>` body, or a
`return` in the `get` block — is one of:

| Shape | Why it copies |
| --- | --- |
| `_items.ToArray()` / `.ToList()` / `.ToHashSet()` / `.ToDictionary(...)` | Materializes a new collection each read |
| `(int[])_values.Clone()` | Allocates a duplicate array each read |
| `new List<T>(_items)` | The seeding constructor copies the source in |
| `new[] { _first, _second }` | A fresh array each read |

The property's own type must be a collection (an array, or a non-`string`,
non-`ref struct` `IEnumerable`), and the allocation must really be a copy: a
`To*` call has to return a collection, a `Clone` has to be called on one, and an
object creation has to be a `System.Collections.Generic` or
`System.Collections.Concurrent` type seeded from a source.

## Silent shapes

- **Auto-properties** — the getter returns the backing field.
- **A cached field** — `=> _items` allocates nothing, and neither does
  `=> _cache ??= _items.ToArray()`, which builds the copy once.
- **An immutable snapshot the getter did not just build** — returning an existing
  `ImmutableArray<T>` field is a field read, not a copy.
- **A read-only wrapper** — `new ReadOnlyCollection<T>(_list)` wraps the list
  instead of copying it. `System.Collections.ObjectModel` is deliberately outside
  the reported set, because a cached view is one of the fixes this rule suggests.
- **A capacity constructor** — `new List<int>(8)` sizes a collection, it does not
  copy one.
- **Indexers and `set` accessors** — only a property's `get` is examined.
- **A copy inside a lambda or a local function in the getter** — the block scan
  walks statements only, so an allocation that belongs to somebody else's return
  value is never mistaken for the property's.

## How to fix it

- Cache the copy in a field and hand back the same instance.
- Expose a `ReadOnlyCollection<T>` or an `ImmutableArray<T>` built once.
- Turn it into a method (`GetItems()`), so the cost is visible at the call site.

## Configuration

| Key | Default | Meaning |
| --- | --- | --- |
| `performancesharp.PSH1017.excluded_properties` | *(empty)* | Comma-separated property names the rule leaves alone. |
| `performancesharp.excluded_properties` | *(empty)* | The project-wide fallback for the same setting. |

```ini
[*.cs]
performancesharp.PSH1017.excluded_properties = Defaults, Corners
```

A property that copies a genuinely small, fixed-size collection is the case this
exists for.

## Example

```csharp
public int[] Items => _items.ToArray();          // PSH1017

private readonly ReadOnlyCollection<int> _view;  // fixed: built once
public IReadOnlyList<int> Items => _view;

public int[] GetItems() => _items.ToArray();     // fixed: the cost is at the call site
```
