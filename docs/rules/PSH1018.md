# PSH1018 — Do not build an array for a params parameter

**Category:** Allocations
**Severity:** Warning
**Code fix:** Unwraps the array and passes the elements directly (supports Fix All)

A `params` parameter already means "the compiler will collect these arguments into an array for
you". Writing the array out by hand at the call site allocates exactly the array the compiler was
going to build anyway — and hides which overload you are actually calling.

```csharp
void Log(params object[] args) { }

Log(new object[] { user, order });   // PSH1018
Log(user, order);                    // fixed — the compiler builds the same array
```

An empty one is worse than merely redundant. `Log()` lets the compiler pass the shared empty array;
`Log(new object[0])` allocates a fresh zero-length array on every call.

```csharp
Log(new object[0]);            // PSH1018
Log(new object[] { });         // PSH1018
Log(Array.Empty<object>());    // PSH1018
Log();                         // fixed — no allocation at all
```

## What is reported

The last argument must be an array written at the call site, landing in a `params` **array**
parameter's slot:

- `new object[] { a, b }` — an explicit array with an initializer.
- `new[] { a, b }` — an implicitly typed array.
- `new object[0]`, `new object[] { }`, `Array.Empty<object>()` — an empty array.

## Silent shapes

The rule only fires when unwrapping the array provably means the same call. These are left alone:

**The array does not go to a `params` parameter.** `void Take(object[] values)` is asking for an
array, and it gets one. There is no compiler-built array to remove.

**An overload would capture the unwrapped call.** This is the trap the rule exists to avoid:

```csharp
void Log(params object[] args) { }
void Log(object a, object b) { }

Log(new object[] { a, b });   // not reported — Log(a, b) would call the *other* Log
```

To rule this out the analyzer **speculatively binds the rewritten call** and reports only when it
still resolves to the same method. Nothing is offered on a guess.

**A single element that is itself the array.** With one element, the unwrapped call would have the
right number of arguments for the *normal* (non-expanded) form, which the compiler prefers — so the
element would be passed *as* the array rather than *inside* one:

```csharp
void Log(params object[] args) { }

Log(new object[] { values });  // not reported — Log(values) passes values as the whole array
Log(new object?[] { null });   // not reported — Log(null) passes a null array
```

**A covariant array.** `string[]` converts to `object[]`, so the compiler passes it straight through
rather than rebuilding it. Unwrapping would change the array's runtime type from `string[]` to
`object[]`:

```csharp
Log(new string[] { "a", "b" });  // not reported
```

**An array that is not written at the call site**, and an array whose elements were never written:

```csharp
var values = new object[] { a };
Log(values);                     // not reported — nothing was allocated at the call site
Log(new object[3]);              // not reported — those three nulls are not in the source
```

**A named argument**, because the array is not sitting in a positional `params` slot:

```csharp
Log(args: new object[] { a, b });  // not reported
```

## `params` collections

C# 13 allows `params` on collection types other than arrays — `params ReadOnlySpan<T>`,
`params List<T>`, and so on. **The rule skips them.** An array argument binds to a `params
ReadOnlySpan<T>` parameter in *normal* form, through a conversion: the compiler passes your array as
the span rather than building anything, so there is no compiler-built allocation to remove and
unwrapping it would change what is passed. The rule requires the `params` parameter's type to be an
array before it reports anything, so this is skipped structurally rather than by testing a language
version.

```csharp
void Log(params ReadOnlySpan<object> args) { }

Log(new object[] { a, b });   // not reported
```

Collection expressions (`Log([a, b])`) are also left alone. That spelling already leaves the storage
choice to the compiler, which is the outcome this rule is asking for.

## Example

```csharp
void Write(string format, params object[] args) { }

Write("{0} {1}", new object[] { name, count });  // PSH1018
Write("{0} {1}", name, count);                   // fixed — leading arguments stay put
```
