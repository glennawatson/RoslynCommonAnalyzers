# PSH1019 — Slice arrays with AsSpan instead of the range indexer

**Category:** Allocations
**Severity:** Warning
**Code fix:** Slices in place with `AsSpan` or `AsMemory` (supports Fix All)

The range indexer on an array **allocates**. `data[1..5]` is not a view — it is a
new array with the elements copied into it. When the copy is then handed to
something that only wanted to *read* a run of elements, it was pure waste:
`AsSpan(1..5)` and `AsMemory(1..5)` produce the same run with no allocation and
no copy.

```csharp
Parse(data[1..5]);          // PSH1019 — allocates a 4-element array and copies into it
Parse(data.AsSpan(1..5));   // fixed — a window onto data

static void Parse(ReadOnlySpan<byte> bytes) { }
```

The rule looks at the **converted type** of the indexer expression, so it fires
exactly where the compiler was already going to turn that fresh array into a span
or a memory — an argument, a local, a return — and nowhere else.

## What is not reported

- **A slice the caller keeps as an array.** `byte[] copy = data[1..5];` asked for
  an array, and got one. There is nothing to rewrite, and the rewrite would not
  compile anyway.
- **A mutable `Span<T>` or `Memory<T>` target.** This is the rule's central safety
  guard, and it is not about the allocation — it is about what the two mean:

  ```csharp
  Span<byte> s = data[1..5];   // s is a private copy; writing to it leaves data alone
  Span<byte> s = data.AsSpan(1..5);   // s is a window; writing to it changes data
  ```

  Those are different programs. Swapping one for the other would be a behavior
  change dressed up as an optimization, so a mutable target is left alone even
  though the allocation is just as real. A `ReadOnlySpan<T>` or `ReadOnlyMemory<T>`
  target cannot write at all, so the view and the copy answer every question the
  same way — and those are what the rule reports.
- **Anything that is not a one-dimensional array.** `text[1..5]` on a string
  already allocates a string and has no span-shaped alternative in that position;
  `span[1..5]` on a span never allocated in the first place.

## Proving the API exists

`MemoryExtensions.AsSpan(T[], Range)` and `AsMemory(T[], Range)` do not exist on
netstandard2.0 or .NET Framework. The rule resolves them from the analyzed
compilation at compilation start and registers no syntax action when they are
absent, and the rewritten call is then **bound speculatively** — including a check
that its result still converts to the type the use site asked for — before any
diagnostic is raised. A target framework that cannot express the fix never sees
the warning.

## Performance

The clean path is syntactic: an element access whose single argument is not a
range expression is rejected before anything is bound.

The rule carries no expression-tree guard, unlike its sibling span rules, and
deliberately: the compiler already refuses a range indexer and a range expression
inside an expression tree (CS8790, CS8792), so a slice this rule could report can
never appear in one, and a guard would be code that never runs.
