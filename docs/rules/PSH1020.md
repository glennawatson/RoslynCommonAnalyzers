# PSH1020 — Prefer a jagged array over a multidimensional one

**Category:** Allocations
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

The CLR gives a single-dimensional array a fast access path that a
multidimensional array never gets. `a[i]` on a `T[]` is a bounds check the JIT can
often prove away entirely and then a direct address computation. `a[i, j]` on a
`T[,]` goes through a slower helper, multiplies through the row stride at run
time, and the bounds check does not get elided the same way. In a tight loop over
a large grid the difference is easy to measure.

A jagged `T[][]` is an array of single-dimensional arrays, so every row is back on
the fast path.

```csharp
int[,] grid = new int[rows, cols];   // PSH1020
int[][] grid = new int[rows][];      // each row indexes at full speed
```

Reported at the sites where the shape is actually **chosen**: a field, a local, a
parameter, a property, a method's return type, and an array creation.

## Why there is no code fix

Changing the shape is not a local edit, and pretending otherwise would produce a
fix that compiles and quietly means something else:

- every `a[i, j]` becomes `a[i][j]`;
- the allocation becomes a loop that fills each row;
- `GetLength(1)` stops meaning what it meant, and `Length` changes from the total
  element count to the row count;
- a rectangular array **guarantees** that every row has the same length. A jagged
  array does not. Code that relied on that guarantee now has to establish it.

Those are decisions about the program, not mechanical rewrites, so the rule
reports and leaves them to the author.

## What is not reported

Merely **naming** a multidimensional type is not reporting-worthy — there is
nothing to change at those sites, and very often the array belongs to an API the
author does not own:

```csharp
typeof(int[,])          // not reported
(int[,])boxed           // not reported
List<int[,]>            // not reported
```

A jagged `int[][]` is of course never reported: it is what the rule is asking for.
`int[,][]` is, on its rectangular outer specifier.

## Trade-offs worth knowing

A jagged array is not free: it allocates one object per row plus the outer array,
so it costs more objects and one more indirection per access, and the rows are not
guaranteed to be contiguous in memory. For a small, short-lived grid that is read
once, the rectangular form can genuinely be the better choice. This rule is about
the common case — a grid that gets indexed in a loop — where the per-access win
dominates. Where it does not, turn the rule off for that project in
`.editorconfig`.

## Proving the API exists

Nothing to prove: a jagged array needs no API and no target framework support, so
this rule reports identically everywhere, netstandard2.0 and .NET Framework
included. It is the one rule in its batch with no availability gate, because there
is no suggestion that could fail to exist.

## Performance

Purely syntactic. An array type with no rank specifier of two or more dimensions
is rejected without touching the semantic model, and the rule never binds anything
at all.
