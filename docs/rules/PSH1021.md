# PSH1021 — Do not force garbage collection

**Category:** Allocations
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

The .NET garbage collector self-tunes: it watches allocation rate and survival
and schedules collections when they pay off. An explicit `GC.Collect()` overrides
that and forces a collection now — a blocking, stop-the-world pass that suspends
every managed thread, walks the heap, and usually promotes objects that were about
to die anyway into an older generation, so the *next* natural collection has more
to do. `GC.WaitForPendingFinalizers()` compounds it by blocking the caller until
the finalizer queue drains. Both almost always lower throughput and add latency
spikes; the runtime collects better when left alone.

```csharp
GC.Collect();                    // PSH1021
GC.Collect(2);                   // PSH1021 — any overload
GC.WaitForPendingFinalizers();   // PSH1021
```

The call is matched syntactically on `GC.Collect` / `GC.WaitForPendingFinalizers`,
then bound to confirm the containing type is `System.GC`. A method named `Collect`
on a different type — including a user type also called `GC` — is never reported.

## What is not reported

A forced-GC call whose enclosing method also samples allocations — it reads
`GC.GetAllocatedBytesForCurrentThread`, `GC.GetTotalAllocatedBytes`, or
`GC.GetTotalMemory` — is left alone. Settling the heap before reading an allocation
counter is the measurement itself and cannot be written any other way, so the whole
`GC.Collect(); GC.WaitForPendingFinalizers(); GC.Collect();` ritual in such a method
is not reported.

```csharp
public static long Measure(Action op)
{
    GC.Collect();                   // not reported — this method samples allocations below
    GC.WaitForPendingFinalizers();
    GC.Collect();

    var before = GC.GetAllocatedBytesForCurrentThread();
    op();
    return GC.GetAllocatedBytesForCurrentThread() - before;
}
```

The sampling call is bound too, so a same-named method on another type does not
exempt anything.

## Why there is no code fix

Deleting the call is not always the right edit. In a memory-diagnostics harness, a
leak test, or a benchmark's setup phase, a deterministic collection is exactly what
the author wants, and the correct pattern there
(`GC.Collect(); GC.WaitForPendingFinalizers(); GC.Collect();`) is intentional. Apart
from the allocation-measurement case exempted above, the rule cannot tell that apart
from a collection wedged into hot product code, so it reports and leaves the decision
to the author.

## Proving the API exists

Nothing to prove: `GC.Collect` and `GC.WaitForPendingFinalizers` have existed on
every target framework since the first release, so this rule reports identically
everywhere, netstandard2.0 and .NET Framework included.

## Performance

The no-diagnostic path is syntactic. An invocation whose member name is not
`Collect` or `WaitForPendingFinalizers`, or whose receiver is not spelled `GC`, is
rejected before the semantic model is touched; only a matching shape is bound to
confirm the type.
