# PSH1106 — Index collections directly instead of using LINQ element access

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Collections |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A `System.Linq.Enumerable` `First()`, `Last()`, or `ElementAt(index)` call on a
receiver whose static type can be indexed directly.

```csharp
// 👎 reported
var first = list.First();
var last = list.Last();
var third = list.ElementAt(2);

// 👍
var first = list[0];
var last = list[list.Count - 1];
var third = list[2];
```

## Rule description

`First()`, `Last()`, and `ElementAt()` on a list-like receiver allocate an
enumerator and dispatch through `IEnumerable<T>` where the indexer reads the
element directly.

An invocation is reported only when **all** of the following hold:

- it is the member-access extension form: `x.First()` and `x.Last()` with an
  empty argument list, or `x.ElementAt(index)` with exactly one argument; the
  predicate overloads are never reported;
- it binds to `System.Linq.Enumerable`; and
- the receiver's static type is indexable: a rank-1 array, `string`, or a type
  that is or implements `IList<T>`/`IReadOnlyList<T>`.

`Last()` is additionally reported only when the static type also exposes a
constant-time `Count`/`Length`, because its replacement reads the count.
A receiver typed as plain `IEnumerable<T>` is **not** reported.

## How to fix violations

Use the indexer instead. The provided code fix rewrites `x.First()` to `x[0]`,
`x.ElementAt(i)` to `x[i]`, and `x.Last()` to `x[x.Count - 1]` (or
`x[x.Length - 1]` for arrays and strings).

The `Last()` rewrite duplicates the receiver into the index expression, so the
code fix is only offered when the receiver is side-effect-free to repeat — an
identifier, `this`, or a simple member-access chain. The analyzer still reports
`Last()` on a complex receiver (for example a method call); hoist the receiver
into a local first, then apply the indexer by hand or rerun the fix.
