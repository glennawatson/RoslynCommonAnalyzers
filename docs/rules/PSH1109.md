# PSH1109 - Merge consecutive Where calls

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Collections |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

Each `Where` adds an iterator object and a delegate invocation per element. Two consecutive filters
combine into a single predicate joined with `&&`, halving the per-element overhead.

## Example

```csharp
return values.Where(value => value > 0).Where(value => value < 10);
```

Use:

```csharp
return values.Where(value => (value > 0) && (value < 10));
```

## Notes

The analyzer uses syntax-first matching and then verifies both calls resolve to `System.Linq.Enumerable`;
`IQueryable` chains are left alone because query providers translate and optimize the whole expression.
The rule only fires when both predicates are expression-bodied lambdas with exactly one parameter, so
the index-taking `Where` overload is never reported. The code fix keeps the first lambda's parameter
and rewrites the second predicate's parameter references to that name; it is skipped (the analyzer
still reports) when the first parameter's name already occurs in the second predicate, where a rename
could silently change what an identifier refers to.
