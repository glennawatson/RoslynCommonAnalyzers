# PSH1120 — Do not materialize a sequence just to enumerate it

**Category:** Collections
**Severity:** Warning
**Code fix:** Removes the trailing `ToList()`/`ToArray()` call (supports Fix All)

`ToList` and `ToArray` copy every element into a new collection. When that
copy's only consumer is a `foreach` loop, the buffer — and every growth copy
made while filling it — is thrown away as soon as the loop ends. Enumerating
the source directly does the same work without the allocation.

Only the parameterless `System.Linq.Enumerable` extensions are reported;
custom `ToList`/`ToArray` overloads and calls that pass arguments stay clean,
as does `await foreach`.

**Mutation guard:** a common reason to materialize before a loop is to survive
modifying the source inside it — removing matched elements, for example, would
otherwise throw during enumeration. The rule walks the receiver down to its
root identifier (`items` in `items.Where(...).ToList()`) and stays silent when
the loop body mentions that identifier *in any way*, read or write. This is
deliberately blunt: the rule does not try to prove which mentions are safe, so
some harmless loops are skipped rather than risking a fix that introduces a
"collection was modified" exception. When the receiver has no identifiable
root (a parenthesized expression, for example), the guard has nothing to match
and the copy is reported.

## Example

```csharp
foreach (var item in items.Where(i => i > 0).ToList())   // PSH1120
{
    Process(item);
}

foreach (var item in items.Where(i => i > 0))            // fixed
{
    Process(item);
}

foreach (var item in items.ToList())                     // not reported:
{                                                        // the body mentions
    items.Remove(item);                                  // the source again
}
```
