# PSH1122 — Use a sorted set's Min and Max properties

**Category:** Collections
**Severity:** Warning
**Code fix:** Replaces `set.Min()` with `set.Min` (supports Fix All)

`SortedSet<T>` and `ImmutableSortedSet<T>` keep their elements in order, so the
smallest and the largest are already sitting at the ends of the tree. Their `Min`
and `Max` **properties** walk straight to them in O(log n) — effectively free.

`Enumerable.Min()` and `Enumerable.Max()` know nothing about the receiver. They
take an `IEnumerable<T>`, allocate an enumerator, and compare every single element
to rediscover what the set already knows. On a set of a million elements that is a
million comparisons instead of a pointer walk.

```csharp
sortedSet.Min();   // PSH1122: scans the whole set
sortedSet.Min;     // fixed: reads the smallest element directly
```

## Reported shapes

- The receiver's **static** type is `SortedSet<T>` or `ImmutableSortedSet<T>`. The
  property does not exist on `ISet<T>` or `IEnumerable<T>`, so a set behind an
  interface is left alone — the suggested rewrite would not compile.
- The call is the **parameterless** `System.Linq.Enumerable` extension.

## Silent shapes

- **The selector overload** — `set.Min(x => x.Key)` asks for the smallest *key*,
  not the smallest element. Different question, different answer.
- **The comparer overload** — `set.Min(comparer)` asks for the smallest element
  under a comparer that need not be the one the set is ordered by.
- **Any other receiver** — a `HashSet<T>` or a `List<T>` has nothing better to
  offer than the scan.
- **`Queryable` sources** — the call binds to `System.Linq.Queryable`, and
  rewriting an expression tree changes what the provider sees.
