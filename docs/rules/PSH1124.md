# PSH1124 — Use a linked list's First and Last properties

**Category:** Collections
**Severity:** Warning
**Code fix:** Reads the list's own node property (supports Fix All)

A `LinkedList<T>` holds a reference to both of its ends. Its `First` and `Last` properties return
those nodes in constant time, with no enumeration at all.

The `Enumerable.First()` and `Enumerable.Last()` extensions know nothing about the receiver they are
handed. They see an `IEnumerable<T>`:

- `Last()` **walks every node in the list** to arrive at the one the list could have pointed to
  straight away — O(n) instead of O(1).
- `First()` stops at the first element, so it is not a walk; but it still reaches for the enumerator
  through the interface, which boxes the list's struct enumerator on the heap. The property reads a
  field.

```csharp
var newest = history.Last();        // PSH1124 — walks the whole list
var newest = history.Last.Value;    // fixed — reads the end node
```

## The property returns a node, not a value

This is the part to get right. The extension returns `T`; the property returns
`LinkedListNode<T>`. They are not interchangeable, and the fix is **not** `list.Last`:

```csharp
LinkedList<string> history;

string s = history.Last();       // T
string s = history.Last.Value;   // T   ✔ what the code fix writes
string s = history.Last;         // LinkedListNode<string> — does not compile
```

The code fix always appends `.Value`, so the rewritten expression has the same type as the call it
replaces.

## Empty-list behaviour changes — read this before applying the fix

On an **empty list** the two shapes do not fail the same way:

| Expression | Empty list |
| --- | --- |
| `list.Last()` | throws `InvalidOperationException` |
| `list.Last.Value` | `Last` returns `null`, so `.Value` throws `NullReferenceException` |

**Both still throw.** The fix cannot turn an empty list into a silently wrong answer, a wrong value,
or a `default` that flows on unnoticed — the failure stays a failure, at the same place, on the same
line. That is why the fix is offered.

What changes is the **exception type**. If you catch the empty case by type, the fix will break that
catch and you need to see it:

```csharp
try
{
    return history.Last();          // throws InvalidOperationException when empty
}
catch (InvalidOperationException)
{
    return null;
}
```

Rewritten to `history.Last.Value`, that `catch` no longer runs. Code in this shape should guard on
the count instead, which is clearer than either version:

```csharp
return history.Count == 0 ? null : history.Last.Value;
```

If your codebase relies on the exception type at these call sites, this rule is not one you want on.

## Silent shapes

**`FirstOrDefault()` / `LastOrDefault()`** are never reported. They answer an empty list with
`default(T)` and no exception, and no property read reproduces that. The nearest rewrite,
`list.Last?.Value`, has type `T?` rather than `T` — a different expression type, which for a value
type means the surrounding code changes too. The rule stays out of it rather than offering a fix that
does not slot in.

**Predicate overloads** (`First(x => ...)`) are asking a different question and are left alone.

**Any other receiver.** The receiver's *static* type must be `LinkedList<T>`; the properties do not
exist on the interfaces it implements, so a list handed around as `ICollection<T>` or
`IEnumerable<T>` is not reported. `List<T>`, arrays, and everything else are not reported either.

**`Queryable` sources.** The call must bind to `System.Linq.Enumerable`, which keeps expression-tree
providers out: rewriting a query the provider is meant to translate would change what it sees.

## The suggestion is proved, not assumed

Before reporting, the analyzer looks up `First`/`Last` on the receiver's own type symbol, confirms it
is a property, and confirms the node type it returns really does expose `Value` — all against the
compilation actually being analyzed. The replacement is therefore known to exist on the target
framework in front of it, rather than inferred from the type's name.

## Example

```csharp
using System.Collections.Generic;
using System.Linq;

public sealed class History
{
    private readonly LinkedList<string> _entries = new();

    public string Oldest => _entries.First();       // PSH1124
    public string Newest => _entries.Last();        // PSH1124
}
```

```csharp
    public string Oldest => _entries.First.Value;   // fixed
    public string Newest => _entries.Last.Value;    // fixed
```
