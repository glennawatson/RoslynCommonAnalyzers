# PSH1125 — Do not enumerate the same sequence twice

**Category:** Collections
**Severity:** Warning
**Code fix:** None — materializing changes the method's allocation behavior, which is the author's call.

A lazy sequence is a recipe, not a result. Walking it a second time re-runs
whatever produced it — the query, the file read, the `yield return` — and
against a one-shot iterator it yields nothing at all. The bug is invisible at
the call site: the same method body is perfectly correct for a `List<T>` and
wrong for a lazy sequence.

## What counts as a walk

- The sequence is the source of a `foreach`.
- The sequence roots a chain ending in an eager terminal — `Any`, `Count`,
  `First`, `Sum`, `ToList`, and the rest of the `System.Linq.Enumerable`
  operators that consume their source immediately.

Enumeration is proved, not assumed: an eager-looking call is counted only once
it binds to a real `System.Linq.Enumerable` extension. **Passing the sequence
to another method is not treated as a walk** — the rule does not guess at what
it cannot see.

## Example

```csharp
void Report(IEnumerable<Order> orders)
{
    if (!orders.Any())      // first walk
    {
        return;
    }

    foreach (var order in orders)   // PSH1125: second walk
    {
        Handle(order);
    }
}

void Report(IReadOnlyList<Order> orders)   // clean: the type promises a collection
```

## Silent shapes

A false positive here is noise, so the rule is deliberately narrow.

- **The declared type must be exactly `IEnumerable<T>` or the non-generic
  `IEnumerable`.** Anything carrying a materialized-collection contract —
  `List<T>`, `T[]`, `ICollection<T>`, `IReadOnlyCollection<T>`, `IList<T>`,
  `IReadOnlyList<T>`, `HashSet<T>` — can be walked twice safely and is never
  reported.
- **A candidate that is written to is dropped entirely** — reassigned, or
  passed by `ref`/`out`. A later walk is then a walk of something else.
- **A candidate initialized from a materializing call is dropped** —
  `= source.ToList()`, a `new`, an array creation, a collection expression.
  The variable already holds the result.
- **Two walks that cannot both run are not counted as a pair** — the arms of an
  `if`/`else`, different `switch` sections, the branches of a `?:`, a `try`
  body versus its `catch`.

## Fixing it

Either take a collection type in the signature, so the contract says the
sequence can be walked twice, or materialize once (`var list = orders.ToList();`)
and walk the list. Both change what the method allocates, which is why there is
no automatic fix.
