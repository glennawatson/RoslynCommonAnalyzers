# PSH1127 — Clear an array instead of filling it with its default

**Category:** Collections
**Severity:** Warning
**Code fix:** Rewrites the call to `Array.Clear` (supports Fix All)

`Array.Fill(buffer, default)` writes the default into every element, one store
at a time. `Array.Clear` hands the whole block to the runtime, which zeroes it
with a single memset the JIT can vectorize.

## Example

```csharp
Array.Fill(buffer, default);           // PSH1127
Array.Clear(buffer);                   // fixed

Array.Fill(buffer, 0, start, count);   // PSH1127
Array.Clear(buffer, start, count);     // fixed
```

Both the whole-array `Fill(array, value)` and the ranged
`Fill(array, value, startIndex, count)` overloads are recognized, with the fill
value written as `default`, `default(T)`, `null`, `false`, or the `0` literal.

## The value is judged against the element type

The literal alone never decides it. `Array.Fill(new object[4], 0)` stores a
*boxed zero*, which is not `null` — clearing that array would store nulls
instead, so the call is **never reported**.

| Fill value | Qualifies when the element type is |
| --- | --- |
| `0` / `false` | a non-nullable value type |
| `null` | a reference type, or a nullable value type |
| `default` / `default(T)` | anything — it is the default by definition |

## Gating and the fallback

`Array.Fill` and `Array.Clear` are both resolved once per compilation, so the
rule costs nothing where `Fill` does not exist — it is .NET Core 2.0+ and is
absent on .NET Framework and netstandard2.0.

The whole-array `Clear(Array)` overload is .NET 6+. Where it is missing, the
fix falls back to `Clear(array, 0, array.Length)` — and then only when the array
expression is a **repeatable name**, since that rewrite mentions the array
twice and an expression that could have side effects must never be evaluated
twice.
