# PSH1200 — Compare strings without allocating case-converted copies

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Strings |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

Two strings are compared by first case-converting **both** operands with
`ToLower()`, `ToUpper()`, `ToLowerInvariant()`, or `ToUpperInvariant()`.

```csharp
// 👎 reported
if (left.ToLower() == right.ToLower())
{
    // ...
}

// 👍
if (string.Equals(left, right, System.StringComparison.OrdinalIgnoreCase))
{
    // ...
}
```

## Rule description

Each `ToLower()`/`ToUpper()` call allocates a full converted copy of its operand on
every comparison. The `StringComparison` overloads compare in place without
allocating. The following shapes are reported when **both** sides call the **same**
parameterless case-conversion method on `string`:

- `a.ToX() == b.ToX()` and `a.ToX() != b.ToX()`
- `a.ToX().Equals(b.ToX())` (instance `Equals(string)`)
- `string.Equals(a.ToX(), b.ToX())` (static two-argument form)

Single-sided conversions (`a.ToLower() == "abc"`) are **not** reported. There the
conversion is doing real one-sided normalization work, and rewriting to a
case-insensitive comparison would change which operand's casing wins — the rule
stays on the shapes where the intent is unambiguously "compare ignoring case".
Mismatched methods on the two sides (`a.ToLower() == b.ToUpper()`) are likewise
skipped.

### Ordinal vs culture caveat

The fix maps `ToLower`/`ToUpper` to `StringComparison.OrdinalIgnoreCase` and the
invariant variants to `StringComparison.InvariantCultureIgnoreCase`. Ordinal
ignore-case is **not** character-for-character identical to comparing `ToLower()`
results for every culture and character (Turkish dotted/dotless `I`, ligatures,
and other one-to-many case mappings can differ). It is still the recommended
replacement: compare-after-conversion was almost never chosen for those edge
semantics, and the `StringComparison` overloads make the intended semantics
explicit while removing the allocations.

## How to fix violations

Call `string.Equals` with an explicit case-insensitive `StringComparison` and drop
the conversion calls. The provided code fix does this, negating the call for `!=`.
