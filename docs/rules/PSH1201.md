# PSH1201 — Use the char overload for single-character strings

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Strings |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled (where the char overloads exist) |

## Cause

A single-character plain string literal is passed to a string search method that
has a char overload, in a call shape that is already ordinal.

```csharp
// 👎 reported
if (path.Contains("/"))
{
    // ...
}

// 👍
if (path.Contains('/'))
{
    // ...
}
```

## Rule description

The string overloads of `Contains`, `StartsWith`, `EndsWith`, `IndexOf`, and
`LastIndexOf` run full string-search setup for a one-character argument; the char
overloads scan directly. Only **ordinal-equivalent** call shapes are reported, so
the fix never changes behavior:

- `s.Contains("x")` — `Contains(string)` is ordinal already;
- `s.StartsWith("x", StringComparison.Ordinal)` and
  `s.EndsWith("x", StringComparison.Ordinal)` — only with an explicit
  `StringComparison.Ordinal` argument;
- `s.IndexOf("x", StringComparison.Ordinal)` and
  `s.LastIndexOf("x", StringComparison.Ordinal)`.

### Why bare `StartsWith("x")` / `IndexOf("x")` are not flagged

`StartsWith(string)`, `EndsWith(string)`, `IndexOf(string)`, and
`LastIndexOf(string)` use the **current culture**, while the char overloads are
**ordinal**. Rewriting `s.IndexOf("x")` to `s.IndexOf('x')` silently changes the
comparison semantics, so those shapes are deliberately left alone. Non-ordinal
comparisons (`StringComparison.OrdinalIgnoreCase`, culture comparisons) are also
skipped — a char cannot express them.

Only plain `"x"` literals qualify: verbatim, raw, and interpolated strings are not
reported, keeping the rule to arguments a char literal can replace verbatim.

### Availability detection

The char overloads are probed once per compilation on `string`'s member list
(`Contains(char)`, `StartsWith(char)`, `EndsWith(char)`, `IndexOf(char)`,
`LastIndexOf(char)`), and each shape is gated on its own overload. On targets
where an overload does not exist (for example `Contains(char)` on
.NET Framework / `netstandard2.0`) the corresponding shape reports nothing.

## How to fix violations

Pass the character instead of a single-character string, and drop the
`StringComparison.Ordinal` argument — the char overloads are ordinal and take no
comparison. The provided code fix does both, escaping the char literal as needed.
