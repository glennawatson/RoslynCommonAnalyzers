# PSH1204 — Test for empty strings by length

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Strings |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A string is compared against the empty string — the literal `""` or
`string.Empty` — with `==` or `!=`.

```csharp
// 👎 reported
if (name == "")
{
    // ...
}

// 👍
if (name is { Length: 0 })
{
    // ...
}
```

## Rule description

`x == ""` runs the full string-equality path — reference checks, a length
comparison, and for `string.Empty` a static field read on top. Checking `Length`
against zero is a single field read.

A `==`/`!=` comparison is reported when **all** of the following hold:

- one operand is the literal `""`, or a member access that binds to the
  `string.Empty` field;
- the other operand's type is `string`; and
- the operator binds to string's built-in equality — a user-defined `==` that
  takes a string operand on another type is not reported.

When the empty string is on the left (`"" == x`), the rewrite still targets the
value operand `x`.

## The three replacements do not agree on null

This is the whole reason the rule has an option instead of an opinion. The four
forms answer differently when the string is null:

| `x` | `x == ""` | `x is { Length: 0 }` | `x.Length == 0` | `string.IsNullOrEmpty(x)` |
| --- | --- | --- | --- | --- |
| `null` | `false` | `false` ✅ | **throws** ⚠️ | `true` ⚠️ |
| `""` | `true` | `true` | `true` | `true` |
| `"a"` | `false` | `false` | `false` | `false` |

Only the **pattern** agrees with `== ""` on every row: null compares unequal to
`""`, and a property pattern never matches null. `x.Length == 0` throws a
`NullReferenceException` where the original quietly answered `false`, and
`string.IsNullOrEmpty(x)` answers `true` where the original answered `false` —
each is a behavior change, not a rewrite.

### How the rule refuses to make that change for you

The **analyzer** — not the code fix — decides which replacement is allowed,
because only the analyzer has the semantic model and therefore the operand's
**nullable flow state**. It resolves your configured style against that state and
records the permitted style on the diagnostic; the fix emits what it is told.

- The operand's flow state is **not-null** (nullable analysis is on, and the
  compiler has already proven the value cannot be null at that point — whether
  because it is a non-nullable `string`, or because an earlier `is null` check
  narrowed it): the configured style is honored. All three forms agree here,
  because the null row cannot happen.
- Anything else — a `string?`, a value that might be null, or **any file with
  nullable analysis switched off** (whose flow state is not a proof of anything,
  merely an absence of information): the style silently degrades to `pattern`,
  which is an exact equivalent for every input.

So `length` and `is_null_or_empty` are best understood as "use this **where it is
provably identical**". In a project without nullable reference types enabled,
that is nowhere, and the rule keeps emitting the pattern. Turning the option on
never changes what a fixed file does when a string is null.

## Configuration

| Option key | Values | Default |
| --- | --- | --- |
| `performancesharp.PSH1204.empty_string_style` | `pattern`, `length`, `is_null_or_empty` | `pattern` |
| `performancesharp.empty_string_style` | same, applied project-wide | `pattern` |

```ini
[*.cs]
# House style is the framework helper — but only where it cannot change null behavior.
performancesharp.PSH1204.empty_string_style = is_null_or_empty
```

| Value | The fix emits | Notes |
| --- | --- | --- |
| `pattern` (default) | `x is { Length: 0 }` / `x is not { Length: 0 }` | Exact equivalent for every input. Needs C# 9. |
| `length` | `x.Length == 0` / `x.Length != 0` | Offered only where the operand is provably not null. Compiles on any language version. |
| `is_null_or_empty` | `string.IsNullOrEmpty(x)` / `!string.IsNullOrEmpty(x)` | Offered only where the operand is provably not null. Compiles on any language version. |

An unset or unrecognized value falls back to `pattern`. A rule-specific key wins
over the project-wide one.

The `length` fix parenthesizes an operand that binds looser than member access,
so `a + b == ""` becomes `(a + b).Length == 0` and not `a + b.Length == 0`.

### Language version

The diagnostic is reported for every language version. The `pattern` fix needs
C# 9 (for the `not` pattern), so on an older language version it is not offered —
and because a file below C# 9 gets no pattern fix, a project pinned there is
exactly where `length` earns its keep: it compiles anywhere, and is still only
offered where the operand cannot be null.

### Expression trees

Comparisons inside lambdas converted to
`System.Linq.Expressions.Expression<TDelegate>` are **not** reported. Query
providers receive `== ""` as an expression tree and must translate it themselves,
and property patterns are illegal in expression trees, so the fix could not apply
there anyway.

## How to fix violations

Take the code fix. It emits the replacement your configuration asked for wherever
that replacement is provably identical, and the null-safe pattern everywhere else.

## Performance

The clean path is syntax-only: a comparison with no `""` or `.Empty` operand is
rejected before the semantic model is touched. The option is read only after a
violation is certain, so a clean file never pays for the per-tree option lookup,
and the operand's type and its nullable flow state come from the same single
`GetTypeInfo` call.
