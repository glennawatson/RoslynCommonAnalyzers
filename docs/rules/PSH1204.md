# PSH1204 — Test for empty strings by length

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Strings |
| Severity | Warning |
| Code fix | Yes (C# 9+) |
| Default | Enabled |

## Cause

A string is compared against the empty string — the literal `""` or
`string.Empty` — with `==` or `!=`.

```csharp
// 👎 reported
if (name == "")
{
    // ...
}

// 👍
if (name is { Length: 0 })
{
    // ...
}
```

## Rule description

`x == ""` runs the full string-equality path — reference checks, a length
comparison, and for `string.Empty` a static field read on top. Checking `Length`
against zero is a single field read. (the rule/the rule-inspired.)

A `==`/`!=` comparison is reported when **all** of the following hold:

- one operand is the literal `""`, or a member access that binds to the
  `string.Empty` field;
- the other operand's type is `string`; and
- the operator binds to string's built-in equality — a user-defined `==` that
  takes a string operand on another type is not reported.

When the empty string is on the left (`"" == x`), the rewrite still targets the
value operand `x`.

### The fix is an exact null-safe equivalent

The code fix rewrites `x == ""` to `x is { Length: 0 }` and `x != ""` to
`x is not { Length: 0 }`. These preserve behavior for **every** input, including
null:

| `x` | `x == ""` | `x is { Length: 0 }` | `x != ""` | `x is not { Length: 0 }` |
| --- | --- | --- | --- | --- |
| `null` | `false` | `false` | `true` | `true` |
| `""` | `true` | `true` | `false` | `false` |
| `"a"` | `false` | `false` | `true` | `true` |

Null compares unequal to `""`, and a property pattern never matches null — the
two forms agree on all three rows. This is why the fix uses the pattern rather
than `x.Length == 0`, which throws on null. The pattern is parenthesized when the
surrounding context is itself an expression (for example inside `&&` or `?:`).

### Expression trees

Comparisons inside lambdas converted to
`System.Linq.Expressions.Expression<TDelegate>` are **not** reported. EF-style
query providers receive `== ""` as an expression tree and must translate it
themselves, and property patterns are illegal in expression trees, so the fix
could not apply there anyway.

### Language version

The diagnostic is reported for every language version, but the code fix is only
offered on C# 9 or later, where the `not` pattern exists. Older files keep the
diagnostic without a fix.

## How to fix violations

Replace the comparison with the null-safe length pattern. The provided code fix
does this, negating the pattern with `not` for `!=`.
