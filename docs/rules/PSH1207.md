# PSH1207 — Specify StringComparison for culture-sensitive string operations

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Strings |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A culture-sensitive `string` search or comparison is called with only string
arguments, letting it bind to the overload that consults the current culture on
every call.

```csharp
// 👎 reported — culture-sensitive by default
if (path.StartsWith(prefix))
{
    // ...
}

// 👍
if (path.StartsWith(prefix, System.StringComparison.Ordinal))
{
    // ...
}
```

## Rule description

`StartsWith`, `EndsWith`, `IndexOf`, and `LastIndexOf` with a single `string`
argument, and the static `string.Compare(string, string)`, all bind to their
culture-sensitive overloads. Those walk `CultureInfo` and the collation tables on
every call, which is dramatically slower than an ordinal comparison and — because
the result depends on the ambient culture — is usually not what the caller
intended. Passing an explicit `StringComparison` selects the comparison directly;
`StringComparison.Ordinal` is a direct char-by-char check.

Only the all-string shapes are reported, so the rule never fires where the default
is already ordinal or where no `StringComparison` overload applies:

- The char overloads (`IndexOf('x')`, `StartsWith('x')`) are ordinal.
- `Contains(string)` is ordinal by default.
- `Equals(string)` and `==` are ordinal.
- Any call that already passes a `StringComparison` has a different argument count
  and is left untouched.

The rule is gated once per compilation on `System.StringComparison` existing, so it
costs nothing on targets that do not expose it.

### Ordinal vs culture caveat

The fix appends `StringComparison.Ordinal`, which changes a culture-sensitive call
to an ordinal one. This is the performance-motivated choice and the semantics the
vast majority of `StartsWith`/`IndexOf`/`Compare` calls actually want. When a call
genuinely needs linguistic comparison, specify `StringComparison.CurrentCulture`
(or the invariant variant) instead — the point of the rule is that the comparison
should be **explicit**, not left to the culture-sensitive default.

## How to fix violations

Add a `StringComparison` argument. The provided code fix appends
`System.StringComparison.Ordinal`.
