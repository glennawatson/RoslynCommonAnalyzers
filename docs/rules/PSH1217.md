# PSH1217 — Do not copy a sequence to an array just to read it

**Category:** Strings
**Severity:** Warning
**Code fix:** Removes the copying call (supports Fix All)

`string.ToCharArray()` and `ReadOnlySpan<T>.ToArray()` allocate an array and copy
every element into it. When that array is handed straight to something the
original would have satisfied on its own, the allocation and the copy buy
nothing:

| Written | Reported because | Fixed |
| --- | --- | --- |
| `foreach (var c in s.ToCharArray())` | a string is already enumerable, and the compiler lowers `foreach` over one to an indexed loop | `foreach (var c in s)` |
| `s.ToCharArray().Length` | a string has a `Length` | `s.Length` |
| `s.ToCharArray()[i]` | a string has an indexer | `s[i]` |
| `Use(s.ToCharArray())` where `Use` also takes `string` or `ReadOnlySpan<char>` | that overload reads the same characters without the copy | `Use(s)` |

The same four shapes are reported for `ReadOnlySpan<T>.ToArray()`, against the
span's own `Length`, indexer, and `ReadOnlySpan<T>` overloads.

## Example

```csharp
foreach (var c in text.ToCharArray()) { }   // PSH1217 — allocates a char[] per call
foreach (var c in text) { }                 // fixed
```

## What is not reported

The consumers above are a whitelist, so a copy that is genuinely needed never
matches one. A copy that is **stored** in a local or field, **returned**,
**assigned**, or handed to an API that only takes an array stays exactly as it
is. So does a **mutated** copy — `chars[0] = 'x'` writes to a storage location a
string and a `ReadOnlySpan<T>` do not have, and turning a mutable buffer into a
read-only one is the opposite of a fix. Making a mutable copy is what
`ToCharArray` is *for*, and this rule leaves that job alone. (Building a string
by copying, mutating, and re-wrapping is [PSH1209](PSH1209.md)'s business.)

Two narrowings worth knowing:

- **`foreach` over a span's `ToArray()` is not reported.** A `ReadOnlySpan<T>` is
  a `ref struct`. Enumerating one directly keeps it alive for the whole loop, and
  a loop body containing an `await` or a `yield` then does not compile. The
  array's copy is what makes those loops legal, so the rule leaves them alone.
  The span's `Length`, indexer, and argument shapes have no such problem and are
  reported.
- **The argument shape is confirmed by binding the rewrite, not by finding an
  overload.** Passing a `string` where a `char[]` used to go does not always
  reach the overload you expect: a `Use(object)` overload wins over a
  `Use(ReadOnlySpan<char>)` one, because a reference conversion outranks the
  user-defined conversion the span relies on. The rule speculatively binds the
  rewritten call and only reports when it provably lands on a sequence-taking
  overload with the same shape and the same return type.

## Performance

The clean path is syntactic: an invocation that is not a bare `x.ToCharArray()`
or `x.ToArray()`, or whose result is not in one of the four consumer positions,
is rejected before anything is bound — which is every call in a codebase but a
handful.
