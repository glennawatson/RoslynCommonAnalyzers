# PSH1218 — Search with a start index instead of allocating a substring

**Category:** Strings
**Severity:** Warning
**Code fix:** Slices with `AsSpan` instead (supports Fix All)

`text.Substring(i).IndexOf(value)` copies the whole tail of the string before it
looks at a single character. The search itself never needed the copy — it only
needed to start at `i`. `text.AsSpan(i)` produces that same tail with no
allocation, and the search runs over it in place.

```csharp
if (text.Substring(i).IndexOf('/') >= 0) { }   // PSH1218 — allocates the tail
if (text.AsSpan(i).IndexOf('/') >= 0) { }      // fixed
```

`IndexOf`, `LastIndexOf`, `Contains`, `StartsWith`, and `EndsWith` are all
reported.

## Why `AsSpan(i)` and not `IndexOf(value, i)`

The string overload that takes a start index looks like the obvious fix and is
not, because **it changes what the returned number means**:

```csharp
var a = text.Substring(i).IndexOf('/');   // relative to the tail
var b = text.IndexOf('/', i);             // relative to the whole string — b == a + i
```

Any code that used the result as an index would silently start reading from the
wrong place. Nor can it be patched up with a `- i`: on a miss both forms return
`-1`, and `-1 - i` is not `-1`.

A span search has no such problem. It reports its hit **relative to the span**,
which is exactly the basis the substring had, and returns `-1` on a miss just the
same. So the rewrite is value-preserving whether the result is tested as a
boolean or used as an index, and the fix is offered in both cases without any
offset arithmetic.

`string.LastIndexOf(value, i)` is a worse trap still — it searches **backward**
from `i` toward the start of the string, which is not what the substring form
does at all. `AsSpan(i).LastIndexOf(value)` searches the slice, which is.

## What is not reported

- **Culture-sensitive searches.** A span search is ordinal; `text.IndexOf("ab")`
  and `text.StartsWith("ab")` are not. Rewriting them would quietly change the
  comparison, so they are left alone — [PSH1207](PSH1207.md) is the rule that
  asks for an explicit `StringComparison` there. Only calls whose semantics carry
  across untouched are reported: the `char` overloads and `Contains(string)`,
  which are already ordinal, and any overload that names its own
  `StringComparison`, which the span overload takes as well.
- **Searches whose span overload does not exist on the target framework.** The
  rewritten call is bound before the rule fires, so `StartsWith(span, char)` —
  which only arrives in .NET 10 — is reported on .NET 10 and silently skipped on
  .NET 8. The same check covers `AsSpan` not being in scope.
- **A substring used for more than the search.** A substring assigned to a local
  and then both searched and returned is not a throwaway, and never matches the
  shape.
- **Searches inside an expression tree.** An expression tree cannot hold a
  `ref struct`, so a span slice there would not compile.
- **`Substring(start, length)`.** Only the open-ended tail slice is rewritten.

## Performance

The clean path is syntactic: an invocation whose receiver is not a
`x.Substring(i)` call, or whose member name is not one of the five searches, is
rejected before anything is bound. The whole rule is switched off at compilation
start when `System.MemoryExtensions` is not referenced.
