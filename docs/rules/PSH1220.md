# PSH1220 — Do not pass a length that reaches the end

**Category:** Strings
**Severity:** Warning
**Code fix:** Drops the length argument (supports Fix All)

`s.Substring(i, s.Length - i)` says "from `i`, take everything that is left". So
does `s.Substring(i)`, without the arithmetic — and the arithmetic is the part
that goes wrong later, when `i` changes meaning or an offset creeps in and nobody
notices the length no longer lines up.

```csharp
text.Substring(start, text.Length - start);   // PSH1220
text.Substring(start);                        // fixed
```

The same shape is reported on `AsSpan`, `AsMemory`, and the span and memory
`Slice` methods:

```csharp
text.AsSpan(start, text.Length - start);   // PSH1220
data.Slice(start, data.Length - start);    // PSH1220
```

## The arithmetic is proved, not guessed

Exactly two forms are reported:

- a length of `recv.Length - start` whose subtrahend is **the very expression**
  passed as the start;
- a length of `recv.Length` paired with a start of `0`.

A length that merely *happens* to reach the end at run time proves nothing and is
never touched:

```csharp
text.Substring(start, length);   // not reported — may or may not reach the end
```

Forms that do reach the end by a longer route are also not reported:

```csharp
text.Substring(i + 1, text.Length - i - 1);   // not reported
```

That is a deliberate false negative. Missing it costs nothing; a rule that guessed
at those would eventually guess wrong.

## Both spellings must be safe to drop

The fix deletes one reading of the receiver and one of the start, so both have to
be expressions that can be evaluated one time fewer without the program noticing.
A name, a dotted name path, or a literal qualifies. A call does not:

```csharp
GetText().Substring(i, GetText().Length - i);   // not reported
```

Dropping the length there would drop a `GetText()` call — a change in behavior
whenever the method does anything at all.

## Only slices that mean "to the end"

The one-argument overload is only equivalent on the types where it is *defined* to
run to the end: `string`, the `MemoryExtensions` slices, and the span and memory
`Slice` methods. Somebody else's `Slice(int, int)` is not assumed to have a
sibling that means the same thing — their `Slice(int)` might well take that many
elements — so a user-defined slice is left alone however neatly the arithmetic
lines up.

## Proving the API exists

`string.Substring(int)` is as old as `string` itself, so **the string form of this
rule reports on every target framework, netstandard2.0 and .NET Framework
included** — staying silent there would be a false negative, not a safety measure.
The span forms are gated by construction: `AsSpan` does not exist on those targets,
so the code cannot be written there in the first place, and the semantic gate would
refuse it anyway.

In every case the shortened call is **bound speculatively** — same name, same
containing type, same return type — before the diagnostic is raised, so the fix is
known to compile.

## Performance

The clean path is syntactic: an invocation that does not carry exactly two
positional arguments, or whose member name is not one of the four slices, is
rejected before anything is bound.
