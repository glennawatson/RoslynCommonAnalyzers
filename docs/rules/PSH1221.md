# PSH1221 — Ask whether a string starts with a value, not where the value is

**Category:** Strings
**Severity:** Warning
**Code fix:** Rewrites to `StartsWith` (supports Fix All)

`text.IndexOf(value) == 0` walks the whole string looking for a match it is going
to **reject everywhere except position zero**. `StartsWith` compares at position
zero and stops. On a long string with no match, the difference is the entire
string.

```csharp
if (text.IndexOf('/') == 0) { }   // PSH1221 — scans to the end to answer a question about index 0
if (text.StartsWith('/')) { }     // fixed
```

`!= 0` is reported too, and becomes a negated prefix test:

```csharp
if (text.IndexOf('/') != 0) { }   // PSH1221
if (!text.StartsWith('/')) { }    // fixed
```

## The comparison basis never moves

This is the rule where culture bites, so the invariant is worth stating plainly:
**whatever comparison the `IndexOf` call was doing, the `StartsWith` call does the
same one.** Three shapes qualify, and no others.

| Reported | Becomes | Why it is safe |
| --- | --- | --- |
| `IndexOf(char) == 0` | `StartsWith(char)` | both are **ordinal** |
| `IndexOf(string, StringComparison) == 0` | `StartsWith(string, StringComparison)` | the **same** `StringComparison` argument moves across |
| `IndexOf(string) == 0` | `StartsWith(string)` | both are **current-culture** |

The third row is the one to read twice. `string.IndexOf(string)` is
culture-sensitive by default — and so is `string.StartsWith(string)`. So the
rewrite leaves the comparison basis exactly where the author put it. What the rule
will **never** do is turn that into `StartsWith(value, StringComparison.Ordinal)`:
that would be a behavior change wearing an optimization's clothes.
[PSH1207](PSH1207.md) is the rule that asks for an explicit `StringComparison`
there, and it stays missing here, exactly as written.

## What is not reported

- **`IndexOf` with a start index.** `text.IndexOf("ab", start) == 0` is not asking
  a prefix question at all.
- **`IndexOf(char, StringComparison)`.** There is no `StartsWith(char,
  StringComparison)` to carry the comparison into, so the shape is refused rather
  than approximated.
- **A comparison against anything but `0`.** `>= 0` is a *contains* question;
  `== 1` is something else again.
- **A search inside an expression tree**, where changing the call shape can change
  how a provider translates it.

## Proving the API exists

`string.StartsWith(char)` **does not exist on netstandard2.0 or .NET Framework** —
while `string.IndexOf(char)` compiles happily on both. Suggesting the char form
there would hand the author a diagnostic they cannot act on.

So the rule resolves `string`'s `StartsWith` overloads from the analyzed
compilation, once, at compilation start, and gates **each shape on its own
overload**. On netstandard2.0 the char form is therefore silent while the
`StringComparison` form — which does exist there — still reports and still fixes.
The rewritten call is then bound speculatively before the diagnostic is raised.

## Performance

The clean path is syntactic: a comparison whose operands are not a zero literal and
an `IndexOf` call is rejected before anything is bound. The whole rule is switched
off at compilation start when `string` offers no usable `StartsWith` overload at
all.
