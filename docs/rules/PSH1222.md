# PSH1222 — Concatenate slices without materializing them

**Category:** Strings
**Severity:** Warning
**Code fix:** Concatenates the spans (supports Fix All)

`string.Concat(a.Substring(i), b)` copies the characters twice. Once into the
substring, which is allocated for the purpose; then again into the result, at
which point the substring is garbage. The span overloads of `string.Concat` take
the slice itself and copy once.

```csharp
string.Concat(a.Substring(1), b);           // PSH1222 — allocates the substring, then copies it again
string.Concat(a.AsSpan(1), b.AsSpan());     // fixed — one copy, straight into the result
```

## Every argument moves, or none do

`string.Concat` has **no overload that mixes a `string` with a
`ReadOnlySpan<char>`**. So the rewrite has to turn every argument into a span: the
slices become `AsSpan(...)` with their arguments untouched, and the plain strings
gain an `AsSpan()`.

That is not a compromise. `AsSpan()` on a string allocates nothing and copies
nothing — it is a length and a pointer — and a `null` string yields an empty span,
which is exactly what `Concat` was already doing with it. The result is one copy
instead of two.

## What is not reported

- **A concatenation with no slice in it.** `string.Concat(a, b)` has nothing to
  save.
- **The `object` and `params` overloads.** Neither has a span counterpart, so
  there is nowhere to rewrite them to.
- **Concatenation written with `+`.** The compiler already lowers that, and this
  rule does not second-guess it.
- **A concatenation inside an expression tree**, which cannot hold a `ref struct`.

Two to four arguments are reported, which is the arity the span overloads offer.

## Proving the API exists

`string.Concat(ReadOnlySpan<char>, ReadOnlySpan<char>)` arrived with .NET Core 2.1
and **does not exist on netstandard2.0 or .NET Framework** — where the reported
code compiles perfectly well. The rule probes `string`'s member list at compilation
start and registers no syntax action at all when the span overloads are absent, so
the suggestion is never made where it cannot be taken.

The all-span rewrite is then **bound speculatively** — every parameter of the
resolved overload must be a `ReadOnlySpan<char>` — before the diagnostic is raised.

## Performance

The clean path is syntactic: an invocation whose member name is not `Concat`, or
whose arguments include no `Substring` call, is rejected before anything is bound.
