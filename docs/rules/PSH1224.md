# PSH1224 — Convert bytes to hex in one call

**Category:** Strings
**Severity:** Warning
**Code fix:** Rewrites to `Convert.ToHexString` (supports Fix All)

`BitConverter.ToString(bytes).Replace("-", "")` builds the hex string **twice**.
First the hyphen-separated form, which is allocated and filled; then a second
string, allocated to take the hyphens back out again. For a 32-byte hash that is
two strings and two full passes to produce one answer that was never going to have
hyphens in it.

`Convert.ToHexString` writes the result once.

```csharp
BitConverter.ToString(hash).Replace("-", "");   // PSH1224 — two strings, two passes
Convert.ToHexString(hash);                      // fixed — one string, one pass
```

The offset-and-count form lines up as well:

```csharp
BitConverter.ToString(hash, 2, 4).Replace("-", "");   // PSH1224
Convert.ToHexString(hash, 2, 4);                      // fixed
```

## The two produce the same text

Both emit **upper-case** hex, two characters per byte, and both throw
`ArgumentNullException` on a null array — so the rewrite is a straight
substitution, not an approximation.

A case conversion chained after the call keeps its place, and simply gets a cheaper
string to work on:

```csharp
BitConverter.ToString(hash).Replace("-", "").ToLowerInvariant();   // PSH1224
Convert.ToHexString(hash).ToLowerInvariant();                      // fixed — still lower-case
```

`string.Empty` is recognized as the empty replacement, just as `""` is.

## What is not reported

- **The separated form kept as-is.** `BitConverter.ToString(hash)` on its own is a
  different string, and a perfectly reasonable one.
- **Replacing a different separator.** `Replace("-", ":")` is not building hex the
  hard way; it is building something else.
- **A `ToString` on anything but `BitConverter`.** The receiver is bound, not
  guessed at.

## Proving the API exists

`Convert.ToHexString` **arrived in .NET 5**. The hand-rolled chain is exactly what
an author on netstandard2.0 or .NET Framework *has* to write — reporting it there
would be telling them to call something that does not exist.

So the method is resolved from the analyzed compilation at compilation start and
the rule registers no syntax action when it is absent. The rewritten call is then
**bound speculatively** before the diagnostic is raised, and the fix prefers the
unqualified `Convert` spelling but falls back to the fully qualified one, so a file
without `using System;` still gets a fix that compiles.

## Performance

The clean path is syntactic: an invocation whose member name is not `Replace`, or
whose receiver is not a `ToString` call, is rejected before anything is bound.
