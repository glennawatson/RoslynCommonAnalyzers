# PSH1225 — Decode bytes to a string in one call

**Category:** Strings
**Severity:** Warning
**Code fix:** Rewrites to `Encoding.GetString` (supports Fix All)

`new string(encoding.GetChars(bytes))` allocates a `char[]`, decodes into it, and
then copies the whole thing again into the string that actually gets returned. The
buffer is garbage the moment the constructor returns. `Encoding.GetString` decodes
straight into the string.

```csharp
new string(Encoding.UTF8.GetChars(bytes));   // PSH1225 — a throwaway char[] and a second copy
Encoding.UTF8.GetString(bytes);              // fixed
```

The index-and-count form maps across the same way:

```csharp
new string(Encoding.ASCII.GetChars(bytes, 0, count));   // PSH1225
Encoding.ASCII.GetString(bytes, 0, count);              // fixed
```

## The two decode identically

`GetChars` and `GetString` run the same decoder, with the same fallback and the
same handling of a partial sequence at the end of the input, so the resulting
string is character-for-character what the buffer held. The rewrite reuses the
arguments untouched.

## What is not reported

- **A char buffer that is kept as a buffer.** `Encoding.UTF8.GetChars(bytes)` on
  its own is a `char[]` somebody wanted.
- **A string built from a buffer that came from somewhere else.**
  `new string(chars)` is not a decode.
- **A `GetChars` on something that is not an `Encoding`.** The receiver is bound,
  not guessed at.
- **A decode inside an expression tree.**

## Proving the API exists

The sibling overload is **probed, never assumed**. The rule does not hard-code
`Encoding`'s overload list: it takes the `GetChars` that actually bound, looks for
a `GetString` on the same type whose parameter list matches it exactly, and then
binds the rewritten call. A custom `Encoding` subclass that adds a `GetChars` with
no matching `GetString` is left alone, because there is nothing to rewrite it into.

**This rule reports on netstandard2.0 and .NET Framework, and that is correct.**
Unlike its neighbours in this range, its suggestion is not new —
`Encoding.GetString(byte[])` has been there since .NET Framework 1.1. The author on
an old target can act on the diagnostic and the fix compiles, so staying silent
there would be a false negative invented for its own sake. The gate that matters
for this rule is the sibling-overload probe, not a framework version.

## Performance

The clean path is syntactic: an object creation that is not `new string(...)` with
a single `GetChars` argument is rejected before anything is bound. The whole rule is
switched off at compilation start when `System.Text.Encoding` is not referenced.
