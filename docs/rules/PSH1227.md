# PSH1227 тАФ Use the purpose-built call

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Strings |
| Severity | Info |

## Cause

A general-purpose call is configured to do one particular thing, where a
purpose-built call says the same thing directly and does a little less work.

```csharp
// ЁЯСО reported
var order = string.Compare(a, b, StringComparison.Ordinal);
Debug.Assert(false, message);

// ЁЯСН
var order = string.CompareOrdinal(a, b);
Debug.Fail(message);
```

## Rule description

Two shapes are reported.

- **`string.Compare(a, b, StringComparison.Ordinal)`** selects the ordinal
  comparison at run time and then compares. `string.CompareOrdinal(a, b)` is that
  comparison with nothing to select. The two return the same ordering тАФ including
  the same handling of `null` тАФ so the rewrite preserves behavior exactly.
- **`Debug.Assert(false, message)`** tests a condition that is always false only to
  fail. `Debug.Fail(message)` is the failure with no condition. The detail-message
  overload maps too: `Debug.Assert(false, message, detail)` becomes
  `Debug.Fail(message, detail)`.

The dedicated call is the clearer statement of intent and is never slower.

## What is not reported

**An ordinal `Compare` compared against zero.** `string.Compare(a, b, StringComparison.Ordinal) == 0`
asks for equality, not ordering, and is a separate, already-reported shape. This
rule stands down there so the two never both fire on the same expression.

```csharp
// Not reported here: an equality test is a different rule's shape.
if (string.Compare(a, b, StringComparison.Ordinal) == 0) { }
```

**Any non-ordinal `Compare`.** A case-insensitive or culture-sensitive comparison
is not what `CompareOrdinal` does, so it is left alone. A two-argument
`string.Compare(a, b)` тАФ which has no `StringComparison` тАФ is not this rule's shape
either.

**A real assertion.** `Debug.Assert(condition, message)` with an actual condition
is a genuine check and is never touched. A message-less `Debug.Assert(false)` is
not reported, because `Debug.Fail` needs a message.

## How to fix violations

Take the code fix. It renames the call to the purpose-built member and drops the
argument that is no longer needed тАФ the `StringComparison` for `CompareOrdinal`,
the always-false condition for `Debug.Fail` тАФ keeping the receiver you wrote.

## Performance

The clean path is a name check on each invocation; nothing binds until the callee
names `Compare` or `Assert`. Each shape is gated once per compilation on its
replacement resolving тАФ `string.CompareOrdinal` and `Debug.Fail` respectively тАФ so
a project that cannot use one pays nothing for it.
