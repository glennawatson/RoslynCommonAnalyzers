# PSH1311 — Remove a pass-through async state machine

**Category:** Concurrency
**Severity:** Warning
**Code fix:** Removes `async` and returns the task directly (supports Fix All)

An `async` method whose whole body is one tail-position await of another task
compiles to a state machine that exists only to forward that task: every call
pays the machine's setup and an extra completion hop just to finish one task
when another does. Dropping `async` and returning the inner task hands the
caller the same task with none of that overhead.

Three shapes are reported, on methods and local functions returning exactly
`Task` or `Task<T>`: an expression body `=> await X`, a body that is only
`return await X;`, and a `Task`-returning body that is only `await X;`. The
awaited task's type must be identical to the declared return type — no
covariance, no `ValueTask` — so the rewrite never changes the signature or
inserts a conversion. A trailing `.ConfigureAwait(...)` is stripped by the
fix: with nothing after the await, there is no continuation left for it to
configure. Lambdas and `async void` methods are never reported.

## Behavior notes

Removing the state machine moves synchronous failures: an exception the
callee throws before handing back its task — an argument check, a faulting
property getter — now surfaces at the call site instead of inside the
returned task, and the method no longer appears as its own frame in async
stack traces. That shift is only safe to reason about when nothing else
happens around the await, which is why the rule reports single-statement
pass-through bodies and nothing more.

## Example

```csharp
public async Task<int> CountAsync() => await _inner.CountAsync();   // PSH1311
public Task<int> CountAsync() => _inner.CountAsync();               // fixed
```
