# PSH1313 — Call the async overload from an async method

**Category:** Concurrency
**Severity:** Warning
**Code fix:** Awaits the async overload (supports Fix All)

Blocking inside an `async` method wastes the thread that was about to be
released. On a thread pool of finite size, that is how an application which
looks asynchronous starves — and then deadlocks — under load. The async
overload exists precisely so the thread can go and do something else while the
work is outstanding.

## Reported shapes

| Blocking call | Replacement |
| --- | --- |
| `task.Result` | `await task` |
| `task.Wait()` | `await task` |
| `task.GetAwaiter().GetResult()` | `await task` |
| `x.Foo(args)` with a fitting `FooAsync` | `await x.FooAsync(args)` |

The task-blocking forms require the receiver to bind to a real task type;
`.Result` is reported only on a `Task<T>` or `ValueTask<T>`, the only shapes
that have a blocking result to read.

## The sibling is proved, not guessed

For the fourth row, a same-named `…Async` is not enough. The candidate must:

- be declared on **the same type** as the synchronous method,
- **agree on staticness**,
- **await to exactly the type the synchronous call returned** — `void` to
  `Task`, `T` to `Task<T>`, or the `ValueTask` equivalents, and
- **accept the arguments already written**: it leads with the same parameter
  types, and any parameters it adds beyond them must be optional.

A type carrying a `FooAsync` that does not fit is therefore never suggested.
Resolutions are memoized per method symbol, so an async-heavy file pays for
each lookup once.

`Dispose` is left alone — the `await using` rewrite belongs to
[PSH1310](PSH1310.md).

## Example

```csharp
async Task<string> LoadAsync(Stream stream)
{
    var header = ReadHeaderAsync(stream).Result;     // PSH1313
    var header = await ReadHeaderAsync(stream);      // fixed

    return File.ReadAllText(_path);                  // PSH1313: ReadAllTextAsync fits
    return await File.ReadAllTextAsync(_path);       // fixed
}
```

## Silent shapes

The **nearest enclosing function** decides the context, not the outermost one,
so a synchronous local function or lambda declared inside an async method stays
clean — there is no `await` to be had there.

## Known limitation

The rule proves that the shapes line up; it cannot prove that the synchronous
member actually blocks. A type whose `Foo`/`FooAsync` pair matches structurally
— same signature, same awaited result — but where `Foo` does no I/O and never
waits on anything can still be reported.
