# PSH1314 — Read and write streams through the memory-based overloads

**Category:** Concurrency
**Severity:** Warning
**Code fix:** Rewrites the call via `AsMemory` (supports Fix All)

The array-based `ReadAsync`/`WriteAsync` overloads have to allocate state to
carry the array, the offset and the count across the await. The
`Memory<byte>` overloads carry all three in the value itself and return a
`ValueTask`, so a call that completes synchronously — on a buffered stream,
most of them — allocates nothing at all.

## Example

```csharp
await stream.ReadAsync(buffer, offset, count);                   // PSH1314
await stream.ReadAsync(buffer.AsMemory(offset, count));          // fixed

await stream.WriteAsync(buffer, 0, length, token);               // PSH1314
await stream.WriteAsync(buffer.AsMemory(0, length), token);      // fixed
```

A trailing cancellation token is carried over unchanged, and `ReadAsync` is
matched against `Memory<byte>` while `WriteAsync` is matched against
`ReadOnlyMemory<byte>`.

## Gating

The memory overloads are .NET Core 2.1+, so they are never assumed.
`Memory<byte>`, `ReadOnlyMemory<byte>`, and a `Stream.ReadAsync` that really
does take one are all resolved at compilation start, and the rule registers
nothing at all when the framework has none of them. The replacement is then
resolved off the receiver's **own** type hierarchy, so a stream that does not
expose a memory overload is not reported either.

## Silent shapes

**Only a directly awaited call is reported.** The array overloads return `Task`
and the memory ones return `ValueTask`, so rewriting a call whose task is stored
in a variable — or passed to a combinator — would change that variable's type.
Awaiting in place is the only shape where the swap is invisible. A
`ConfigureAwait` sitting between the call and its `await` is fine.

```csharp
Task read = stream.ReadAsync(buffer, 0, count);   // clean: the task's type would change
await stream.ReadAsync(buffer, 0, count).ConfigureAwait(false);   // PSH1314
```
