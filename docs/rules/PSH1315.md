# PSH1315 ŌĆö Do not block on a task that may not be complete

**Category:** Concurrency
**Severity:** Warning
**Code fix:** Awaits the task, where an `await` would compile (supports Fix All)

`Result`, `Wait()`, and `GetAwaiter().GetResult()` all park the calling thread
until the task finishes. Where a `SynchronizationContext` is in play ŌĆö a UI app,
legacy ASP.NET ŌĆö the task's continuation needs that same thread and cannot have
it, and the two wait for each other forever. On the thread pool nothing
deadlocks outright, but the blocked worker is one fewer worker able to complete
the very task it is waiting on, and under load that is how a pool starves.

The rule reports the wait wherever it is written ŌĆö a synchronous method is
exactly where the deadlock lives, so restricting the rule to `async` methods
would miss the interesting half of it. Where the enclosing function is already
`async`, the code fix awaits. Where it is not, the diagnostic stands with no fix:
how far up the call chain `async` should go is the author's decision, and a code
action that guessed would produce something that does not build.

## Reported shapes

| Blocking call | Fix |
| --- | --- |
| `task.Result` | `await task` |
| `task.Wait()` | `await task` |
| `task.GetAwaiter().GetResult()` | `await task` |
| `task.ConfigureAwait(false).GetAwaiter().GetResult()` | `await task.ConfigureAwait(false)` |
| `task.Wait(timeout)`, `task.Wait(cancellationToken)` | reported, never rewritten |

`Task`, `Task<T>`, `ValueTask`, and `ValueTask<T>` are all covered. Blocking a
`ValueTask` is the worse of the two ŌĆö it may only be consumed once ŌĆö and rules
that only know about `Task` miss it entirely.

`Wait(timeout)` gives up after an interval and `Wait(cancellationToken)`
abandons the wait when the token fires. A plain `await` does neither, so those
two are reported without a fix rather than rewritten into something that means
something else.

The receiver is bound before anything is reported: a `Wait()` on a
`SemaphoreSlim` is not a blocking task wait, and is left to
[PSH1313](PSH1313.md), which will suggest `WaitAsync`.

## Silent shapes

### The guarded fast path

A wait a completion check has already proved cannot block is not a defect. It is
the whole point of `IsCompletedSuccessfully`, and reporting it would be telling
the author to make correct, allocation-free code worse:

```csharp
if (pending.IsCompletedSuccessfully)
{
    pending.GetAwaiter().GetResult();   // consumes the result; cannot block
    return;
}

await pending;
```

The guard is recognized when the task is checked with `IsCompleted` or
`IsCompletedSuccessfully` and the wait sits somewhere control can only reach with
that check having held:

- inside the `if` body the check gates, and inside the `else` of a negated check;
- in either arm of a ternary on the check ŌĆö `t.IsCompleted ? t.Result : await t`;
- to the right of an `&&` whose left side proves it, or an `||` whose left side
  disproves it;
- after an early-return guard that has already left the method ŌĆö `if
  (!t.IsCompleted) { return; }` ŌĆö where "left" means the guard's body returns,
  throws, breaks, or continues;
- after the task was awaited outright.

Conditions are read through parentheses, `!`, `&&`, and `||`, so
`if (enabled && !t.IsCompleted) { return; }` guards what follows it.

**Not recognized, and therefore still reported:** a check routed through a local
(`var done = t.IsCompleted;`), a pattern (`t.IsCompleted is true`), a
`Status == TaskStatus.RanToCompletion` comparison, and a check the wait does not
name identically. A check on a task that is *fetched by calling something* ŌĆö
`if (Load().IsCompleted) { ŌĆ” Load().Result ŌĆ” }` ŌĆö guards nothing and is reported,
because each call returns a different task.

### An awaiter's own `GetResult()`

The awaiter pattern requires `GetResult` to be synchronous, and the compiler only
calls it once `IsCompleted` is true. Blocking there is the contract, not a
defect, so a `GetResult` on a type that implements `INotifyCompletion` and offers
`IsCompleted` is left alone. Note that neither `IsCompleted` nor `GetResult` is an
interface member ŌĆö the awaiter pattern is structural ŌĆö so this needs its own rule.

### Members whose signature the author does not own

An override, an explicit interface implementation, and an implicit one are all
left alone when they are synchronous: making them `async` would mean changing a
base type or an interface the author may not own, and a diagnostic nobody can act
on is noise.

This is what quietly covers `IDisposable.Dispose`. A type that must drain an
async resource from a synchronous `Dispose` cannot be told to await, because
`IDisposable` owns that signature. Where the type *can* offer `IAsyncDisposable`
instead, [PSH1310](PSH1310.md) already says so at the call site ŌĆö which is the
place where something can actually be done about it. (A `Dispose(bool)` of the
author's own is not an interface member and *is* reported: implementing
`IAsyncDisposable` and draining in `DisposeAsyncCore` is a fix available to
whoever wrote it.)

### `Main`

The entry point is the outermost frame. Nothing is queued behind it for the
thread it parks, no `SynchronizationContext` is installed yet, and bridging from
synchronous to asynchronous is precisely what it is for. A `Main` that the
compilation resolves as its entry point ŌĆö including the one synthesized for
top-level statements ŌĆö is not reported. A method merely *named* `Main` in a
library is.

### Generated code

Not analyzed.

## Not a port of the rule

the analyzer's the rule reports the guarded fast path and an awaiter's own `GetResult`, and
does not cover `ValueTask` at all. Both false positives force real code to carry
suppressions. PSH1315 is silent on both.
