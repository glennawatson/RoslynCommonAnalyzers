# PSH1315 — Do not block on a task that may not be complete

**Category:** Concurrency
**Severity:** Warning
**Code fix:** Awaits the task, where an `await` would compile (supports Fix All)

`Result`, `Wait()`, `GetAwaiter().GetResult()`, `RunSynchronously()`,
`Task.WaitAll(…)` and `Task.WaitAny(…)` all park the calling thread until the
task finishes. Where a `SynchronizationContext` is in play — a UI app, legacy
ASP.NET — the task's continuation needs that same thread and cannot have it, and
the two wait for each other forever. On the thread pool nothing deadlocks
outright, but the blocked worker is one fewer worker able to complete the very
task it is waiting on, and under load that is how a pool starves.

The rule reports the wait wherever it is written — a synchronous method is
exactly where the deadlock lives, so restricting the rule to `async` methods
would miss the interesting half of it. Where the enclosing function is already
`async`, the code fix awaits. Where it is not, the diagnostic stands with no fix:
how far up the call chain `async` should go is the author's decision, and a code
action that guessed would produce something that does not build.

## Reported shapes

| Blocking call | Fix |
| --- | --- |
| `task.Result` | `await task` |
| `task.Wait()` | `await task` |
| `task.GetAwaiter().GetResult()` | `await task` |
| `task.ConfigureAwait(false).GetAwaiter().GetResult()` | `await task.ConfigureAwait(false)` |
| `Task.WaitAll(a, b)` | `await Task.WhenAll(a, b)` |
| `Task.WaitAny(a, b)`, result discarded | `await Task.WhenAny(a, b)` |
| `task.Wait(timeout)`, `task.Wait(cancellationToken)` | reported, never rewritten |
| `Task.WaitAll(tasks, timeout)`, `Task.WaitAll(tasks, cancellationToken)` | reported, never rewritten |
| `Task.WaitAny(a, b)`, result used | reported, never rewritten |
| `task.RunSynchronously()` | reported, never rewritten |

`Task`, `Task<T>`, `ValueTask`, and `ValueTask<T>` are all covered.

### The four shapes that are reported but never rewritten

`Wait(timeout)` gives up after an interval and `Wait(cancellationToken)`
abandons the wait when the token fires; the `WaitAll` overloads that take a
timeout or a token do the same. A plain `await` does neither, so all four are
reported without a fix rather than rewritten into something that means something
else.

`Task.WaitAny` returns the **index** of the task that finished, where
`Task.WhenAny` returns the task itself. Where that index is used —
`var i = Task.WaitAny(a, b);` — the rewrite would not compile and, worse, a
rewrite that did compile would not mean the same thing, so no fix is offered.
Where the value is discarded, `await Task.WhenAny(a, b)` waits for exactly what
the blocking call waited for and the fix is offered.

`RunSynchronously()` runs a cold task's delegate on the calling thread. There is
no mechanical fix: `await task` on a task nothing ever started waits forever, so
the task has to be *created* differently — `Task.Run`, or an `async` method
called directly — and only its author can say which. The rule reports it and
stops there.

The call is bound before anything is reported: a `Wait()` on a `SemaphoreSlim` is
not a blocking task wait, and is left to [PSH1313](PSH1313.md), which will suggest
`WaitAsync`. A `WaitAll` on a helper of your own is not `Task.WaitAll`, and is not
reported. `WaitAsync` itself is never reported — it is the non-blocking API, and
the fix rather than the defect.

### A guard has to cover what the wait actually blocks on

The [guarded fast path](#the-guarded-fast-path) below silences a wait a completion
check has already proved cannot block. The combinators block on their arguments
rather than on a receiver, so the proof has to cover those:

- `Task.WaitAll` parks the thread until **every** task it was handed is done, so
  it is silent only when every one of them was proved complete. A check on one of
  five says nothing about the other four, and the wait is still reported.
- `Task.WaitAny` returns the moment **any** one of them is done, so a proof about
  a single argument is enough to make it a no-op, and it is silent.

## Silent shapes

### The guarded fast path

A wait a completion check has already proved cannot block is not a defect. It is
the whole point of `IsCompletedSuccessfully`, and reporting it would be telling
the author to make correct, allocation-free code worse:

```csharp
if (pending.IsCompletedSuccessfully)
{
    pending.GetAwaiter().GetResult();   // consumes the result; cannot block
    return;
}

await pending;
```

The guard is recognized when the task is checked with `IsCompleted` or
`IsCompletedSuccessfully` and the wait sits somewhere control can only reach with
that check having held:

- inside the `if` body the check gates, and inside the `else` of a negated check;
- in either arm of a ternary on the check — `t.IsCompleted ? t.Result : await t`;
- to the right of an `&&` whose left side proves it, or an `||` whose left side
  disproves it;
- after an early-return guard that has already left the method — `if
  (!t.IsCompleted) { return; }` — where "left" means the guard's body returns,
  throws, breaks, or continues;
- after the task was awaited outright.

Conditions are read through parentheses, `!`, `&&`, and `||`, so
`if (enabled && !t.IsCompleted) { return; }` guards what follows it.

**The check and the wait must be in the same method body.** The search stops at
the enclosing method, at a lambda, and at a local function; a guard on one side
of that boundary does not cover a wait on the other. See
[the guard analysis is intra-method](#the-guard-analysis-is-intra-method) below,
which is the limitation this produces in real code.

**Not recognized, and therefore still reported:** a check routed through a local
(`var done = t.IsCompleted;`), a pattern (`t.IsCompleted is true`), a
`Status == TaskStatus.RanToCompletion` comparison, and a check the wait does not
name identically. A check on a task that is *fetched by calling something* —
`if (Load().IsCompleted) { … Load().Result … }` — guards nothing and is reported,
because each call returns a different task.

### An awaiter's own `GetResult()`

The awaiter pattern requires `GetResult` to be synchronous, and the compiler only
calls it once `IsCompleted` is true. Blocking there is the contract, not a
defect, so a `GetResult` on a type that implements `INotifyCompletion` and offers
`IsCompleted` is left alone. Note that neither `IsCompleted` nor `GetResult` is an
interface member — the awaiter pattern is structural — so this needs its own rule.

### Members whose signature the author does not own

An override, an explicit interface implementation, and an implicit one are all
left alone when they are synchronous: making them `async` would mean changing a
base type or an interface the author may not own, and a diagnostic nobody can act
on is noise. This exemption — and `Main`, below — is about the enclosing
**member**, not about the blocking shape, so it covers every shape alike: a member
that cannot be made `async` cannot await a `Task.WaitAll` either.

This is what quietly covers `IDisposable.Dispose`. A type that must drain an
async resource from a synchronous `Dispose` cannot be told to await, because
`IDisposable` owns that signature. Where the type *can* offer `IAsyncDisposable`
instead, [PSH1310](PSH1310.md) already says so at the call site — which is the
place where something can actually be done about it. (A `Dispose(bool)` of the
author's own is not an interface member and *is* reported: implementing
`IAsyncDisposable` and draining in `DisposeAsyncCore` is a fix available to
whoever wrote it.)

### `Main`

The entry point is the outermost frame. Nothing is queued behind it for the
thread it parks, no `SynchronizationContext` is installed yet, and bridging from
synchronous to asynchronous is precisely what it is for. A `Main` that the
compilation resolves as its entry point — including the one synthesized for
top-level statements — is not reported. A method merely *named* `Main` in a
library is.

### Generated code

Not analyzed.

## The guard analysis is intra-method

The guard analysis looks only inside the method the wait is written in. **A guard
in the caller is not seen.** When the completion check is at the call site and the
consume is hoisted into a helper, PSH1315 reports the helper, and the code needs
a suppression.

This is the shape. It is the standard allocation-free async fast path: take the
synchronous result without a state machine when the task is already done, and
hand off to an `async` continuation when it is not.

```csharp
// caller — guards
var pending = observers[i].OnNextAsync(value, cancellationToken);
if (!pending.IsCompletedSuccessfully)
{
    return AwaitOnNextRemainderAsync(pending, observers, i + 1, value, cancellationToken);
}

ConsumeCompleted(pending);   // only reached when the task is already done

// helper — consumes
[MethodImpl(MethodImplOptions.AggressiveInlining)]
private static void ConsumeCompleted(ValueTask pending) =>
    pending.GetAwaiter().GetResult();   // PSH1315 reports here
```

The wait inside `ConsumeCompleted` cannot block: every caller has already checked
`IsCompletedSuccessfully` and taken the `async` path when it did not hold. PSH1315
does not know that, because the check is one frame up.

### Suppressing it

Suppress on the helper, not on the file or the assembly, and state the invariant
the reader has to check:

```csharp
[MethodImpl(MethodImplOptions.AggressiveInlining)]
[SuppressMessage(
    "Concurrency",
    "PSH1315:Do not block on a task that may not be complete",
    Justification = "Consumes a ValueTask that is already complete. This method is private, and every " +
                    "call site checks IsCompletedSuccessfully first and takes the async path when the " +
                    "task is not done, so GetResult() blocks nothing here.")]
private static void ConsumeCompleted(ValueTask pending) =>
    pending.GetAwaiter().GetResult();
```

What makes that justification checkable, and what a reviewer should confirm before
accepting it:

- the helper is `private`, so the call sites are all in the one type and can be
  read;
- every one of them guards the argument with `IsCompleted` or
  `IsCompletedSuccessfully` and does not fall through to the helper when the check
  fails;
- the helper does not reassign the parameter before waiting on it, and does not
  defer the wait into a lambda that runs later.

If any of those stops being true, the suppression is wrong and the wait can block.

### Why the rule does not follow the guard into the caller

This is a deliberate choice, not an oversight. To stay silent here the rule would
have to prove it has seen **every** call site, and that proof is not available in
general:

- `private` is the only accessibility whose callers are all inside the current
  compilation. `internal` is not — `InternalsVisibleTo` hands the method to an
  assembly this compilation never sees — and anything more visible is open to the
  world.
- Even within a type, a helper's name can be converted to a delegate, or passed to
  reflection as a string. Neither carries a guard the analyzer can read.
- The parameter would also have to be proved to still hold the caller's task at
  the point of the wait.

Getting any of that wrong does not produce a false positive; it produces a **false
negative** — a genuine sync-over-async blocking wait that the rule silently
approves, in the exact place such a bug is hardest to see. A rule that reports a
provably safe wait costs one reviewable suppression. A rule that guesses across a
call boundary costs a deadlock nobody was warned about. PSH1315 takes the first.
