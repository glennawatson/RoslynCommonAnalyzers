# PSH1316 - Consume a ValueTask exactly once

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | Yes for the loop form (move the producing call into the loop) |
| Default | Enabled |

## Cause

A `ValueTask` instance is consumed more than once - awaited across loop iterations, or through a copy.

```csharp
ValueTask vt = ReadAsync();
for (int i = 0; i < count; i++)
{
    await vt; // PSH1316: every iteration after the first reuses a spent token
}
```

## Rule description

A `ValueTask` is a single-use handle to a pooled `IValueTaskSource` token. The first consume - an
`await`, `.Result`, `.GetAwaiter()`, or `.AsTask()` - invalidates that token and returns it to the pool.
Consuming the same instance a second time reads a recycled token and silently returns another caller's
result.

The rule is deliberately narrow. It reports only the two shapes a dataflow pass cannot see:

- a `ValueTask` local **declared outside a loop and consumed inside it**, where the loop body does not
  reassign it first, so every iteration after the first reuses a spent token;
- a `ValueTask` local **copied into a second local** where both are consumed.

It does not report a local assigned inside the loop (a fresh `ValueTask` each iteration), a
`.Preserve()`d instance (safe to consume repeatedly), or a single consume. The rule stays silent unless
`System.Threading.Tasks.ValueTask` resolves.

## How to fix violations

Give each iteration a fresh `ValueTask` by moving the producing call into the loop.

```csharp
for (int i = 0; i < count; i++)
{
    ValueTask vt = ReadAsync();
    await vt;
}
```

If the value genuinely must be read again, call `.Preserve()` once, which returns a `ValueTask` that is
safe to await repeatedly.

The code fix moves the producing declaration into the loop body, and is offered only when the local is
used nowhere but inside the loop. The copy form is not fixed, because the intent is ambiguous.

## When not to use the rule

There is no correct use of a spent `ValueTask` token, so the rule has no routine exemption. If a custom
`ValueTask` source documents multi-consume support, suppress the rule at that call.
