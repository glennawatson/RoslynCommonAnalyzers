# PSH1401 — Attribute types should be sealed

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | ApiSelection |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A non-abstract class deriving (directly or indirectly) from `System.Attribute`
is not `sealed`.

```csharp
// 👎 reported
public class MyAttribute : System.Attribute
{
}

// 👍
public sealed class MyAttribute : System.Attribute
{
}

// 👍 abstract bases exist to be derived from
public abstract class ValidationAttribute : System.Attribute
{
}
```

## Rule description

Reflection-based attribute lookups (`GetCustomAttributes` and friends) are
cheaper on sealed attribute types: when an attribute type is sealed, the runtime
can match it by exact type and never has to consider derived attribute types
during the inheritance search. Most attribute types are never subclassed, so the
`sealed` modifier is free performance.

Indirect inheritance is also reported — a class deriving from an intermediate
attribute base (`class CustomAttribute : MyBaseAttribute` where
`MyBaseAttribute : Attribute`) is still an unsealed attribute type. Abstract
attribute classes are never reported; they exist precisely to be derived from.

### Breaking-change caveat

Sealing a type that is subclassed in **another assembly** is a breaking change
for that assembly. This rule reports **declarations, not usages** — mirroring
the ecosystem's equivalents — so it cannot see subclasses outside the current
compilation. If your attribute type is public API that consumers are expected
to derive from, either make it `abstract` or leave it unsealed deliberately.

## How to fix violations

Add the `sealed` modifier. The provided code fix inserts `sealed` after any
accessibility modifiers so the modifier order stays valid; Fix All seals every
reported attribute type in the document in one pass.
