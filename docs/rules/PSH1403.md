# PSH1403 — Do not initialize fields to their default value

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | ApiSelection |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A field initializer restates the value the runtime already guarantees: the
field type's default.

```csharp
// 👎 reported
private int _count = 0;
private bool _ready = false;
private string? _name = null;
private State _state = State.None; // enum member with value 0

// 👍
private int _count;
private bool _ready;
private string? _name;
private State _state;
```

## Rule description

The runtime zero-initializes every field before any constructor body runs, so
an explicit default initializer emits initialization IL that repeats that work
in **every** constructor of the type, for nothing.

An initializer is reported when it is:

- a `default` literal, or `default(T)` of the field type; or
- a compile-time constant equal to the field type's default value: `null` for
  reference and nullable value types, numeric zero (including `0.0f`, `0.0`,
  and `0m`), `false`, `'\0'`, or an enum member whose underlying value is zero.

Each declarator of a multi-variable declaration is checked independently —
`int a = 0, b = 1;` reports only `a`.

The following are **never** reported:

- **Negative floating-point zero** (`-0.0f` / `-0.0`): it is a different bit
  pattern from the default `+0.0`, so the initializer is not redundant.
- **Null-forgiving initializers** (`= null!`): the `!` documents an intentional
  "starts null but treated as not-null" contract, so removing it would delete
  information (and reintroduce nullable warnings).
- **`const` fields**: the initializer is mandatory.

### Why struct fields are excluded

Fields declared inside `struct` (and `record struct`) declarations are skipped
entirely. Struct field initializers only run when an **explicitly declared
constructor** runs — `default(S)`, array elements, and field defaults bypass
them — so a struct field initializer is not a straightforward restatement of
guaranteed zero-initialization, and removing one can be a semantic change
depending on how instances are created.

## How to fix violations

Delete the initializer. The provided code fix removes only the reported
declarator's initializer, leaving other declarators in the same declaration
untouched; Fix All removes every redundant initializer in the document in one
pass.
