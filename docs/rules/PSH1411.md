# PSH1411 — Seal types that nothing derives from

**Category:** ApiSelection
**Severity:** Warning
**Code fix:** Adds the `sealed` modifier (supports Fix All)

A call on an unsealed class has to go through a virtual dispatch unless the JIT
can prove no override exists. `sealed` states that up front: virtual and
interface calls on the type devirtualize, its members become inlining candidates,
and the optimizations downstream of an inline stop being blocked.

A `private`, `file`, or `internal` class that **nothing in the compilation
derives from** can be sealed for free, because the compilation can see every type
that could possibly derive from it.

```csharp
internal class Widget { }          // PSH1411
internal sealed class Widget { }   // fixed
```

## What is not reported

- A class that is already `sealed`, or is `static` or `abstract`.
- A **record** — [SST1800](SST1800.md) owns record sealing.
- A class **any type in the compilation derives from**, including through a
  constructed generic base (`class Gadget : Widget<int>`) and including a
  subclass emitted by a source generator.
- A class **named by a generic constraint** (`where T : Widget`). A sealed class
  is not a valid constraint (CS0701), so the fix would not compile. Constraints
  on local functions count too.
- A class declaring a **new `virtual` member** (CS0549 in a sealed class) or a
  **new `protected` member** (CS0628). An `override` is fine on both counts and
  is not a reason to stay quiet.
- A **`public` or `protected` class**, by default — see below.

## The visibility rules

Sealing a class that something outside this build derives from is a **breaking
change**, and this compilation cannot see that derivation. Two cases follow.

**`public` and `protected` classes** may be derived from by any assembly that
references this one, so they are not reported unless you opt in with
`include_public`. Accessibility is judged on what is actually reachable, not on
the keyword: a `public` class nested in an `internal` one is not visible outside
the assembly, so it *is* reported.

**`internal` classes** are only safe while the assembly keeps its internals to
itself. If the assembly carries an **`[InternalsVisibleTo]`** — which most do,
for a test project — a friend assembly may derive from an internal class, and
that subclass is nowhere in this compilation. So when an `[InternalsVisibleTo]`
is present, internal classes are skipped entirely. `private` and `file` classes
are still reported: a friend assembly gains internal access, never private
access, and a file-local type is invisible even in its own assembly.

The practical consequence is worth stating plainly: **in a project with an
`[InternalsVisibleTo]`, this rule reports only `private` nested and `file`
classes.** That is the correct answer, not a limitation — the internal ones
genuinely cannot be judged from here.

## Configuration

```ini
# Rule-specific
performancesharp.PSH1411.include_public = false

# Project-wide
performancesharp.include_public = false
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `include_public` | `true`, `false` | `false` | Report externally visible classes too, accepting that sealing one is a breaking change for any assembly that derives from it. |

Turn it on for an **application**, where nothing outside the build derives from
your types. Leave it off for a **library**.

## When to suppress

The compilation is not the whole world. Two things it cannot see:

- A **runtime-generated subclass** — Castle DynamicProxy, Moq, NSubstitute, an
  ORM's lazy-loading proxy, a DI container's interceptor. These derive from your
  class at run time, and sealing it turns a working test or a working mapping
  into a run-time failure rather than a compiler error.
- A type derived from in a **friend assembly**, which the `[InternalsVisibleTo]`
  check above is there to protect but which `include_public = true` deliberately
  overrides for public types.

## Performance

The derived set is built **once** for the whole compilation, not per candidate.
The symbol action that visits a class checks its shape, its accessibility, and
its members first; only a class that survives all three asks the index whether
anything derives from it, and the first such question triggers a single pass that
records every source type's base type and every class a type parameter is
constrained to. Every candidate after that is a hash lookup. Asking "does
anything derive from this?" by re-walking the compilation per type would be
quadratic; this is one pass over the compilation's types.

The clean path costs a base-type read and a special-type compare. A class whose
base is `object`, or is sealed, or comes from metadata can never be a candidate,
so nothing is recorded and nothing is allocated for it — and a compilation with
no candidate at all never builds the index.

The diagnostic is reported from the class's own symbol action rather than at
compilation end, so it is a **local** diagnostic: it appears live in the IDE
without full-solution analysis, and the "Seal the type" fix can be offered on it.
