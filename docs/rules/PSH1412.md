# PSH1412 — Use Random.Shared instead of allocating a Random

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | ApiSelection |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A `Random` is allocated with the **parameterless** constructor in a compilation
where `Random.Shared` exists.

```csharp
// 👎 reported
var value = new Random().Next(1, 7);

// 👍
var value = Random.Shared.Next(1, 7);
```

## Rule description

`Random.Shared` is a thread-safe instance the runtime has already made. Taking it
costs nothing: no allocation, no initialization, no seeding.

Allocating your own is worse than merely redundant:

- **Two instances created close together can produce the same sequence.** A
  `new Random()` seeds itself from the clock. Historically that meant that
  several created inside one tight loop — or one per web request — could all be
  seeded from the same tick and hand back identical numbers.
- **A single instance shared across threads is not safe.** `Random`'s instance
  methods are not thread-safe; concurrent calls can corrupt its internal state and
  make it return zeros. Cache a `Random` in a static field and you have written
  that bug. `Random.Shared` is explicitly safe to use from any thread.

So the shared instance is not just faster — it is the correct default.

### A cached, long-lived instance is still reported

```csharp
// Still reported: the allocation happens once, but the field is not thread-safe.
private static readonly Random Rng = new Random();
```

The allocation happens once, so it is not the cost that matters here. A `Random`
in a shared static field is a data race waiting for a second thread, and
`Random.Shared` carries none of that hazard. The rule reports it, and the fix is
the same.

## What is not reported

**A seeded `Random`.** `new Random(42)` asks for a *reproducible* sequence — a
test, a replayable shuffle, a deterministic simulation — and `Random.Shared`
cannot give one. The rule matches the argument-free constructor and nothing else.

```csharp
// Not reported: the seed is the whole point.
var deterministic = new Random(seed);
```

**A subclass of `Random`.** `new StubRandom()` binds to a constructor whose
containing type is the subclass, so the rule leaves it alone: its overrides are
presumably why it exists.

**Anything on a framework without the API.** `Random.Shared` arrived in .NET 6.
The rule probes the compilation for the static `Shared` property and registers no
action at all when it is absent, so on .NET Framework or `netstandard2.0` it is
silent — no diagnostic, and no code fix that would not compile.

## How to fix violations

Take the code fix. The type is written back exactly as you wrote it — a
`new System.Random()` becomes `System.Random.Shared` — and a target-typed `new()`
becomes `Random.Shared`, which the rule has already confirmed resolves at that
position.

A variable that is later reassigned is no obstacle: `Random.Shared` is a `Random`
like any other, so `var rng = Random.Shared;` followed by `rng = new Random(seed);`
compiles exactly as before.

## Performance

The rule costs one type lookup per compilation on a framework that has no
`Random.Shared`, and nothing else. Where the API does exist, an explicit
`new Foo()` is rejected on a string comparison of the written type name before
anything is bound.
