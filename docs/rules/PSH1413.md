# PSH1413 — Read the Unix epoch from the framework

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | ApiSelection |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

The Unix epoch is constructed by hand where the framework already holds it as a
field.

```csharp
// 👎 reported
var epoch = new DateTime(1970, 1, 1, 0, 0, 0, DateTimeKind.Utc);

// 👍
var epoch = DateTime.UnixEpoch;
```

## Rule description

`DateTime.UnixEpoch` and `DateTimeOffset.UnixEpoch` are the value you are
spelling out — already UTC, folded at compile time, and impossible to mistype.

Spelling it out is what invites the bug below.

### The kindless epoch is the bug this rule is really for

```csharp
// Reported — and this one is a latent bug, not a style nit.
var epoch = new DateTime(1970, 1, 1);
```

That `DateTime` has `DateTimeKind.Unspecified`, not `Utc`. An Unspecified
`DateTime` is treated as **local** the moment anything converts it —
`ToUniversalTime()`, a subtraction against a UTC value, most serializers. The
epoch then silently moves by the machine's UTC offset, which produces a
timestamp that is wrong by hours, reproduces on one developer's machine and not
another's, and changes with the season in any zone with daylight saving.

Replacing it with `DateTime.UnixEpoch` gives you the UTC epoch, which is what the
code meant. **This means the fix corrects behavior rather than preserving it** —
deliberately, and only for a value that was already wrong.

### What is matched

Reported shapes:

| Written | Rewritten |
| --- | --- |
| `new DateTime(1970, 1, 1, 0, 0, 0, DateTimeKind.Utc)` | `DateTime.UnixEpoch` |
| `new DateTime(1970, 1, 1)` | `DateTime.UnixEpoch` (see above — a correction) |
| `new DateTime(1970, 1, 1, 0, 0, 0)` | `DateTime.UnixEpoch` (same) |
| `new DateTimeOffset(1970, 1, 1, 0, 0, 0, TimeSpan.Zero)` | `DateTimeOffset.UnixEpoch` |

Every numeric component is read as a **constant**, so a named
`const int EpochYear = 1970;` matches exactly as the literal does. A computed
value matches nothing. The parameters are read off the **bound constructor**,
which is what tells a seventh `int` (a millisecond) apart from a `DateTimeKind`
in the same position, and what rejects any overload the rule cannot vouch for — a
`Calendar`, a tick count, an existing `DateTime`.

## What is not reported

**An explicitly non-UTC epoch.** `DateTimeKind.Local` names a different instant.
`DateTimeKind.Unspecified`, written out, is a choice — the author has said what
they mean, and the rule does not argue with it. Neither is `DateTime.UnixEpoch`,
so neither is rewritten.

```csharp
// Not reported: both say what they mean.
var local = new DateTime(1970, 1, 1, 0, 0, 0, DateTimeKind.Local);
var unspecified = new DateTime(1970, 1, 1, 0, 0, 0, DateTimeKind.Unspecified);
```

**A non-zero offset.** `new DateTimeOffset(1970, 1, 1, 0, 0, 0, TimeSpan.FromHours(1))`
is not the epoch. Only `TimeSpan.Zero` is matched.

**Any other date**, and any epoch with a non-zero time component.

**Anything on a framework without the API.** The two `UnixEpoch` fields arrived
with .NET Core 2.1 and .NET Standard 2.1. Each is probed for separately in the
compilation: a target framework with neither registers no action at all, and one
that has only one of them reports only that one. On .NET Framework and
`netstandard2.0` the rule is silent.

## How to fix violations

Take the code fix, which replaces the allocation with `DateTime.UnixEpoch` or
`DateTimeOffset.UnixEpoch`, keeping the qualification you wrote (a
`new System.DateTime(...)` becomes `System.DateTime.UnixEpoch`).

If you were relying on the Unspecified kind — that is, if some downstream code
depends on the epoch being reinterpreted as local time — do not take the fix.
Fix the downstream code instead: an epoch that means different instants on
different machines is not a value, it is a bug with a birthday.

## Performance

The clean path is two type lookups per compilation, then nothing: an allocation
with fewer than three arguments, an initializer, or a type written as anything but
a name is rejected before the semantic model is touched.
