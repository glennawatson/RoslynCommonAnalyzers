# PSH1414 — Mark members that do not touch instance state as static

**Category:** ApiSelection
**Severity:** Warning
**Code fix:** For a `private` member, adds `static` and repairs `this.`-qualified call sites (supports Fix All); an `internal` member is reported without an automatic fix

An instance method that never reads `this` still takes it — a hidden argument
passed at every call, and a receiver the JIT must prove non-null before it can
dispatch. `static` says what the member actually is, and lets the call go
direct.

## Example

```csharp
private int Score(Order order) => order.Total * 2;          // PSH1414
private static int Score(Order order) => order.Total * 2;   // fixed
```

## What counts as using instance state

A member uses instance state when it mentions `this` or `base`, or names any
**non-static member of its own type or a base type**. An unqualified recursive
call does not count — the member still binds to itself once it is static.

## Silent shapes

The rule is deliberately conservative about what it will touch.

- **Only `private` and `internal` members are reported.** Both have all of their
  call sites inside the assembly. A `public` or `protected` member is left alone —
  making it static is a breaking change to a surface other code binds to.
- **`virtual`, `abstract`, and `override` members** are bound to instance
  dispatch by definition.
- **Members a framework requires to stay instance are skipped.** A member carrying
  a test attribute (xUnit, NUnit, MSTest, TUnit), a BenchmarkDotNet benchmark or
  lifecycle attribute, or a serialization callback attribute is left alone, as is
  every member of a type marked as a test fixture or a BenchmarkDotNet diagnostics
  host — the runner or serializer reaches those on an instance, and `static` would
  break it. An *unrelated* attribute — `[Obsolete]`, an inlining hint — does **not**
  exempt a member that could plainly be static.
- **Auto-properties are skipped** — they *are* instance state.

## Test and benchmark projects

The .NET SDK ships its own analyzer that suggests the same change, but it has no
knowledge of test or benchmark frameworks: it will flag a `[Benchmark]` method
that touches no instance state, and making that method static breaks
BenchmarkDotNet, which requires an instance. PSH1414 is framework-aware and stays
silent on those members, so it is the safer choice inside test and benchmark
projects. To rely on it there, turn the built-in rule off for those folders and
enable PSH1414:

```ini
# in a tests/ or benchmarks/ .editorconfig
dotnet_diagnostic.CA1822.severity = none
dotnet_diagnostic.PSH1414.severity = warning
```

## When the fix is offered

The diagnostic can stand without the fix, and for an `internal` member it always
does: its callers can live in any file in the assembly, out of the fix's view, so
no automatic rewrite is offered (making an assembly-visible member static is a
judgement call in any case). For a `private` member, adding `static` is offered
only when every call site is provably safe:

- the member's type must be declared in a **single file**, so all of the call
  sites are visible, and
- every reference must be **unqualified or `this`-qualified**. An unqualified
  call keeps binding; a `this.Foo(...)` call is rewritten to `Foo(...)` in the
  same edit. A call through some *other* instance (`other.Foo()`) would stop
  compiling, so where one exists the fix is withheld and the diagnostic is left
  for a human.
