# PSH1416 — Cache the serializer options instead of building them per call

**Category:** ApiSelection
**Severity:** Warning
**Code fix:** None.

The serializer caches its per-type metadata **against the options instance** it
was handed. A fresh `JsonSerializerOptions` on every call throws that cache away
and re-derives the whole contract — reflection, converter lookup and all — for a
type it has already seen. This is one of the most expensive accidental costs in
a serialization path.

## Example

```csharp
// PSH1416: a new options instance, and a new metadata cache, on every call
public string Write(Order order) =>
    JsonSerializer.Serialize(order, new JsonSerializerOptions { WriteIndented = true });

// fixed: derived once, reused for the lifetime of the process
private static readonly JsonSerializerOptions Options = new() { WriteIndented = true };

public string Write(Order order) => JsonSerializer.Serialize(order, Options);
```

## What is reported

Only **per-call** construction. A field or property *initializer* runs once and
is left alone, as is anything built in a constructor — so the cached
`static readonly JsonSerializerOptions` the rule is steering toward never
reports itself. An **expression-bodied property**, which does run on every read,
is reported.

The rule is gated on `System.Text.Json.JsonSerializerOptions` resolving in the
compilation, so it costs nothing where the serializer is not referenced.

## Why there is no code fix

Hoisting the construction has to invent a field name, choose where to put it,
and — the part no analyzer can settle at the creation site — be sure the options
are never mutated afterwards. `options.Converters.Add(...)` against a now-shared
instance would change *behavior*, not just move an allocation.
