# PSH1417 — Do not compute an expensive argument for an assertion

**Category:** ApiSelection
**Severity:** Warning
**Code fix:** None — wrapping the work in `#if DEBUG`, or restructuring so the cost is only paid on failure, is a design choice.

`Debug.Assert` carries `[Conditional("DEBUG")]`, so **the call is compiled away
in release builds — but its arguments are evaluated before the call, so the work
that produced them is not.** An assertion whose message interpolates state, or
whose condition calls something costly, pays that cost in production for a check
that no longer runs.

## Example

```csharp
Debug.Assert(items.All(IsValid), $"bad batch: {string.Join(",", items)}");   // PSH1417 ×2
Debug.Assert(count >= 0, "count must not be negative");                      // clean
Debug.Assert(value is not null, nameof(value));                              // clean
```

Each expensive argument is reported on its own — the condition and the message
are separate costs.

## What makes an argument expensive

- It is a **non-constant interpolated string** — it interpolates state, which
  allocates and may box.
- It **contains a call or an object creation** anywhere in its subtree.

## Silent shapes

Anything the compiler folds to a **constant** is free and is never reported.
That is what keeps `nameof(x)` clean — an invocation in syntax only, with a
constant value — along with constant messages and constant interpolated strings.
