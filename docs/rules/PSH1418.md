# PSH1418 — Reuse a shared client instead of constructing one per call

**Category:** ApiSelection
**Severity:** Warning
**Code fix:** None.

Every `HttpClient` owns its own connection pool. One constructed for a single
request opens a fresh connection, then abandons the socket to the operating
system's timed-wait state when it is disposed. A client created per call therefore
leaks a port per call and drains the ephemeral port range under load, and the
throughput of the whole process collapses long before that.

The cloud service clients carry the same cost in another shape: each construction
builds a transport pipeline, authentication state, and metadata caches, and the
next construction rebuilds all of it from scratch — a re-negotiated connection and
a re-authenticated token per call. All of the covered types are documented as
thread-safe and intended to be cached for the lifetime of the process, so a single
long-lived instance shares its connections and caches across every call.

## Covered types

| Client | Namespace |
| --- | --- |
| `HttpClient` | `System.Net.Http` |
| `BlobServiceClient` | `Azure.Storage.Blobs` |
| `QueueServiceClient` | `Azure.Storage.Queues` |
| `ShareServiceClient` | `Azure.Storage.Files.Shares` |
| `DataLakeServiceClient` | `Azure.Storage.Files.DataLake` |
| `TableServiceClient` | `Azure.Data.Tables` |
| `ServiceBusClient` | `Azure.Messaging.ServiceBus` |
| `EventHubProducerClient` | `Azure.Messaging.EventHubs.Producer` |
| `EventGridPublisherClient` | `Azure.Messaging.EventGrid` |
| `CosmosClient` | `Microsoft.Azure.Cosmos` |
| `SecretClient` | `Azure.Security.KeyVault.Secrets` |
| `KeyClient` | `Azure.Security.KeyVault.Keys` |
| `CertificateClient` | `Azure.Security.KeyVault.Certificates` |

Each type is resolved in the analyzed compilation before anything is reported, so
the rule stays silent about any client whose package the project does not
reference — a same-named type of your own is never confused with it.

## Example

```csharp
// PSH1418: a new client, a new connection pool, and an abandoned socket per call
public async Task<string> Fetch(string url)
{
    using var client = new HttpClient();
    return await client.GetStringAsync(url);
}

// PSH1418: a new service client, a new pipeline, and a re-authenticated connection per call
public async Task Send(string connection, ServiceBusMessage message)
{
    await using var client = new ServiceBusClient(connection);
    await using var sender = client.CreateSender("queue");
    await sender.SendMessageAsync(message);
}

// fixed: one shared instance of each, reused for the lifetime of the process
private static readonly HttpClient Client = new HttpClient();

public Task<string> Fetch(string url) => Client.GetStringAsync(url);
```

## What is reported

Only the two shapes that prove — with no data-flow analysis — that the instance
dies with the call:

- a `using` or `await using` declaration or statement over the construction;
- the construction used directly as the receiver of the call it feeds, such as
  `new HttpClient().GetStringAsync(url)` or
  `new BlobServiceClient(connection).GetBlobContainerClient("data")`.

Everything else is silent by construction, because it may already be the single
shared instance the rule is steering toward:

| shape | example |
| --- | --- |
| a static field | `static readonly HttpClient Client = new HttpClient();` |
| an instance field | `readonly CosmosClient _client = new CosmosClient(conn);` |
| a property initializer | `HttpClient Client { get; } = new HttpClient();` |
| a factory return | `HttpClient Create() => new HttpClient();` |
| an argument | `Consume(new HttpClient());` |
| a plain local | `var client = new HttpClient();` |
| an injected client | `public C(BlobServiceClient client) => _client = client;` |

The assembly entry point is exempt: a process that is about to exit exhausts
nothing.

## How to fix

Where a dependency-injection container is available, register the client as a
singleton and inject it. For `HttpClient` specifically, register and inject
`IHttpClientFactory` and create clients from it — the factory rotates the
underlying handlers so connections do not go stale; the diagnostic message names
the factory when its type is referenced by the compilation.

Where no container can be assumed — the usual case for a library — share one
`static readonly` instance for the lifetime of the process; every covered type is
thread-safe:

```csharp
private static readonly HttpClient Client = new HttpClient();

private static readonly BlobServiceClient Blobs = new BlobServiceClient(connectionString);
```

If stale DNS is a concern for a long-lived shared `HttpClient`, set
`SocketsHttpHandler.PooledConnectionLifetime` so pooled connections are recycled
on a schedule without tearing down the client.

## When not to use the rule

Turn it off for tests and benchmarks, where a short-lived client is created,
exercised, and discarded before the process exits — the connection cost never
materializes. Put the relaxation in an `.editorconfig` scoped to that folder:

```ini
# tests/.editorconfig
[*.cs]
dotnet_diagnostic.PSH1418.severity = none
```

## Performance

The whole rule is gated at compilation start on `HttpClient` resolving — every
covered client's SDK layers on the HTTP stack, so a project that references none
of them registers no syntax action at all. The clean path is a parent-shape check
and a token comparison; each service-client type is resolved lazily, the first
time its simple name appears in a per-call shape, and the result (present or
absent) is cached for the compilation, so a project without a client's package
pays at most one failed lookup for it.
