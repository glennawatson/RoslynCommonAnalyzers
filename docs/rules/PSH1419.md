# PSH1419 — Use the built-in cross-platform time-zone API

**Category:** ApiSelection
**Severity:** Warning
**Code fix:** Rewrites `TZConvert.GetTimeZoneInfo(id)` to
`System.TimeZoneInfo.FindSystemTimeZoneById(id)`. The two id-conversion calls are
reported without a fix.

The TimeZoneConverter package existed to paper over a gap the framework has since
closed. Before .NET 6, `TimeZoneInfo.FindSystemTimeZoneById` only understood the id
style of the platform it ran on — Windows ids on Windows, IANA ids on Linux and
macOS — so a program that carried ids across platforms reached for the package to
translate between them.

Since .NET 6 the runtime does that itself. `TimeZoneInfo.FindSystemTimeZoneById`
accepts an IANA id on Windows and a Windows id on Linux and macOS, and
`TimeZoneInfo.TryConvertIanaIdToWindowsId` and
`TimeZoneInfo.TryConvertWindowsIdToIanaId` convert between the two id styles
directly. On a supported framework the package is one more dependency to restore,
load, and keep current for work the framework already does.

## Example

```csharp
using TimeZoneConverter;

// PSH1419: the package resolves an id the framework now resolves itself
TimeZoneInfo eastern = TZConvert.GetTimeZoneInfo("Eastern Standard Time");

// PSH1419: the package converts between id styles the framework now converts itself
string windows = TZConvert.IanaToWindows("America/New_York");
string iana = TZConvert.WindowsToIana("Eastern Standard Time");
```

```csharp
// fixed: the built-in lookup, which accepts either an IANA or a Windows id
TimeZoneInfo eastern = TimeZoneInfo.FindSystemTimeZoneById("Eastern Standard Time");

// the built-in id conversions
TimeZoneInfo.TryConvertIanaIdToWindowsId("America/New_York", out string windows);
TimeZoneInfo.TryConvertWindowsIdToIanaId("Eastern Standard Time", out string iana);
```

## What is reported

A call bound to a method on `TimeZoneConverter.TZConvert`:

| Converter call | Built-in replacement |
| --- | --- |
| `TZConvert.GetTimeZoneInfo(id)` | `TimeZoneInfo.FindSystemTimeZoneById(id)` |
| `TZConvert.IanaToWindows(ianaId)` | `TimeZoneInfo.TryConvertIanaIdToWindowsId(ianaId, out …)` |
| `TZConvert.WindowsToIana(windowsId)` | `TimeZoneInfo.TryConvertWindowsIdToIanaId(windowsId, out …)` |

The receiver is confirmed by binding, so a `GetTimeZoneInfo` method of your own is
never confused with the package's.

## How to fix

The code fix is offered for `GetTimeZoneInfo` only, because that call and
`FindSystemTimeZoneById` have the same shape — one string in, a `TimeZoneInfo` out —
so the rewrite is a clean one-to-one swap.

The id-conversion calls are reported without a fix. `IanaToWindows` and
`WindowsToIana` return the converted id, whereas the built-in helpers return a
`bool` and write the id through an `out` parameter, so replacing them changes the
surrounding expression — a `TryConvert…(id, out var converted)` call and a check of
its result — which is left to a human to write.

## When not to use the rule

Turn it off for a project that must run on a framework older than .NET 6, where
the built-in cross-platform id resolution does not exist and the package is still
doing real work. The rule already stays silent when the replacement is missing, but
a codebase that multi-targets can suppress it for the down-level target:

```ini
# .editorconfig
[*.cs]
dotnet_diagnostic.PSH1419.severity = none
```

## Performance

The whole rule is gated at compilation start on `TimeZoneConverter.TZConvert`
resolving — a project that does not reference the package registers no syntax
action at all. It is gated a second time on the replacement existing:
`System.TimeZoneInfo.FindSystemTimeZoneById`, with the two conversion helpers probed
separately so the id-conversion calls are reported only on a framework that has
them. The clean path is a single switch over the invoked simple name; only a call
that spells one of the three converter methods is bound, and only then is its
receiver confirmed.
