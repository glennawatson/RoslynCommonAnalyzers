# PSH1420 — Do not hold a per-invocation client in a function class field

**Category:** ApiSelection
**Severity:** Warning
**Code fix:** None.

An isolated-worker function class is constructed once per invocation, so anything
it keeps in an instance field lives for a single call. A heavyweight client held
there is therefore rebuilt and discarded on every request.

An `HttpClient` opens its own connection pool, then abandons the socket to the
operating system's timed-wait state when the instance is collected — a client held
per invocation leaks a port per call and drains the ephemeral port range under
load, and the throughput of the whole worker collapses long before that. The cloud
service clients carry the same cost in another shape: each construction builds a
transport pipeline, authentication state, and metadata caches, and the next
invocation rebuilds all of it from scratch.

All of the covered types are documented as thread-safe and intended to be cached
for the lifetime of the process, so a single shared instance carries its
connections and caches across every invocation.

## Covered types

| Client | Namespace |
| --- | --- |
| `HttpClient` | `System.Net.Http` |
| `BlobServiceClient` | `Azure.Storage.Blobs` |
| `QueueServiceClient` | `Azure.Storage.Queues` |
| `ShareServiceClient` | `Azure.Storage.Files.Shares` |
| `DataLakeServiceClient` | `Azure.Storage.Files.DataLake` |
| `TableServiceClient` | `Azure.Data.Tables` |
| `ServiceBusClient` | `Azure.Messaging.ServiceBus` |
| `EventHubProducerClient` | `Azure.Messaging.EventHubs.Producer` |
| `EventGridPublisherClient` | `Azure.Messaging.EventGrid` |
| `CosmosClient` | `Microsoft.Azure.Cosmos` |
| `SecretClient` | `Azure.Security.KeyVault.Secrets` |
| `KeyClient` | `Azure.Security.KeyVault.Keys` |
| `CertificateClient` | `Azure.Security.KeyVault.Certificates` |

Each type is resolved in the analyzed compilation before anything is reported, so
the rule stays silent about any client whose package the project does not
reference — a same-named type of your own is never confused with it.

## What is reported

An instance field or instance auto-property whose type is one of the covered
clients, in a class that declares at least one method carrying
`Microsoft.Azure.Functions.Worker.FunctionAttribute`:

```csharp
using System.Net.Http;
using System.Threading.Tasks;
using Microsoft.Azure.Functions.Worker;

public class ImageFunctions
{
    // PSH1420: a new HttpClient per invocation, and an abandoned socket per call
    private readonly HttpClient _client = new HttpClient();

    [Function("Resize")]
    public Task<string> Resize(string url) => _client.GetStringAsync(url);
}
```

The construction itself is not what is flagged — the field is, because the
per-invocation instance that owns it is the problem. The assignment can be an
initializer, a constructor assignment, or an injected value; all of them tie the
client's lifetime to a single call.

Everything that already shares one instance is silent:

| shape | example |
| --- | --- |
| a static field | `private static readonly HttpClient Client = new HttpClient();` |
| a static property | `public static HttpClient Client { get; } = new HttpClient();` |
| a computed property | `public HttpClient Client => _shared;` |
| a non-client member | `private readonly ILogger _logger;` |
| a class with no function method | a plain service type constructed once and injected |

The rule is silent unless both the worker function attribute and `HttpClient`
resolve in the compilation, so a project that is not an isolated worker registers
no work at all.

## How to fix

Hold one shared instance instead of one per invocation. Where a dependency-injection
container is available — the usual case for a worker — register the client as a
singleton and inject it, and it outlives the per-invocation function class:

```csharp
public class ImageFunctions
{
    private readonly BlobServiceClient _blobs;

    public ImageFunctions(BlobServiceClient blobs) => _blobs = blobs; // singleton from DI

    [Function("Resize")]
    public Task Resize() => _blobs.GetBlobContainerClient("images").CreateIfNotExistsAsync();
}
```

For `HttpClient` specifically, register and inject `IHttpClientFactory` and create
clients from it — the factory rotates the underlying handlers so connections do not
go stale; the diagnostic message names the factory when its type is referenced by
the compilation. Where no container is used, a `static` field shared for the
lifetime of the process works too, because every covered type is thread-safe.

## Relationship to the per-call construction rule

The per-call construction rule (PSH1418) reports a client whose position proves it
dies with the call — a `using` over the construction, or a construction used
directly as a call receiver — and deliberately leaves a field assignment alone,
because a field is usually long-lived. In a function class it is not: the instance
is per-invocation. This rule covers exactly that gap, and reports the member rather
than the construction, so the two never fire on the same span.

## When not to use the rule

Turn it off where a short-lived client never accumulates cost — tests and
benchmarks that create, exercise, and discard a client before the process exits.
Put the relaxation in an `.editorconfig` scoped to that folder:

```ini
# tests/.editorconfig
[*.cs]
dotnet_diagnostic.PSH1420.severity = none
```

## Performance

The whole rule is gated at compilation start on both `FunctionAttribute` and
`HttpClient` resolving, so a non-worker project registers no syntax action. Per
class the clean path is pure syntax — one member scan that bails unless the class
has both a `Function`-named method and a client-typed instance member — and only a
matching class pays for binding. Each service-client type is resolved lazily, the
first time its simple name appears on a candidate member, and the result is cached
for the compilation, so a project without a client's package pays at most one
failed lookup for it.
