# PSH1500 — Prefer TypedResults over Results in a minimal API handler

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | AspNetCore |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (where `TypedResults` exists) |

## Cause

A minimal-API handler produces its result through the untyped
`Microsoft.AspNetCore.Http.Results` factory (`Results.Ok`, `Results.NotFound`,
`Results.Created`, …) where the matching `TypedResults` member is available.

```csharp
// 👎 reported — Results.Ok returns IResult; the concrete Ok<Todo> shape is erased
app.MapGet("/todo/{id}", (int id) => Results.Ok(store.Find(id)));

// 👍 TypedResults.Ok returns Ok<Todo>; the status code and body type are on the signature
app.MapGet("/todo/{id}", (int id) => TypedResults.Ok(store.Find(id)));
```

## Rule description

`Results.X(...)` returns `IResult`. That hides the concrete response type behind
an interface, so the framework has to *infer* the produced status code and body
type when it builds the endpoint's metadata (the description OpenAPI and API
Explorer read). The matching `TypedResults.X(...)` returns the concrete result
type — `Ok<T>`, `NotFound`, `Created<T>`, `NoContent`, and so on — which is
strongly typed and self-describing: the metadata comes straight off the return
signature, with nothing to infer at runtime and a compile-time check that the
body type matches.

The analyzer binds each candidate invocation and reports it only when the
invoked method's **containing type is exactly** `Microsoft.AspNetCore.Http.Results`
(a same-named factory in your own code is never touched — the type identity is
checked, not the name text). Only the `Results.X(...)` call span is reported.
This is a local shape: the analyzer looks at the call, not at whether it sits
inside a `Map*` handler, because the `Results` factory exists to produce a
minimal-API result in the first place.

### Availability detection

This rule targets a **runtime API**, so it is gated on
`Microsoft.AspNetCore.Http.TypedResults` resolving in the referenced framework
(detected once per compilation via `Compilation.GetTypeByMetadataName`). A
non-web project, or one whose ASP.NET Core version predates `TypedResults`,
reports nothing rather than suggesting a type that does not exist. A `Results`
member is reported only when `TypedResults` exposes a member with the same name,
so the suggestion is always actionable. No target framework string is parsed.

## How to fix violations

Call the matching member on `TypedResults` instead of `Results`.

**There is no automatic code fix.** Swapping the call is often that simple, but
not always: a handler that returns more than one result shape must also declare
its return type as a `Results<TResult1, …, TResultN>` union so every branch has a
concrete type, for example:

```csharp
// before — one branch NotFound, one branch Ok<Todo>, return type inferred as IResult
app.MapGet("/todo/{id}", (int id) =>
{
    var todo = store.Find(id);
    return todo is null ? Results.NotFound() : Results.Ok(todo);
});

// after — the union return type names both concrete results
app.MapGet("/todo/{id}", Results<Ok<Todo>, NotFound> (int id) =>
{
    var todo = store.Find(id);
    return todo is null ? TypedResults.NotFound() : TypedResults.Ok(todo);
});
```

Because that return-type change is a judgment call rather than a mechanical
member swap, it is left to you.
