# PSH1501 — Use the two-parameter middleware delegate

**Category:** AspNetCore
**Severity:** Warning
**Code fix:** None. The rewrite reshapes the delegate and moves the `next`
parameter into the outer lambda, which is not a mechanical edit.

Inline middleware registered as a nested delegate is a
`Func<RequestDelegate, RequestDelegate>`:

```csharp
app.Use(next => async context =>
{
    // ... work before the rest of the pipeline ...
    await next(context);
});
```

The outer delegate runs once, when the pipeline is built. The inner delegate it
returns closes over `next` and becomes the request handler, so the captured
`next` is carried as an extra closure alongside the handler on every request
through that branch of the pipeline.

The two-parameter overload hands both the context and the next delegate in
directly, so there is no nested delegate to return and nothing to capture:

```csharp
app.Use(async (context, next) =>
{
    // ... work before the rest of the pipeline ...
    await next(context);
});
```

This form is a `Func<HttpContext, RequestDelegate, Task>`. It reads the same and
does the same work with one fewer allocation on the hot path.

## What is reported

Only the inline nested-lambda shape, where the extra closure actually exists:

- the single argument to `Use` is a **single-parameter** lambda, and
- its body returns another lambda or anonymous method — directly, wrapped in
  parentheses, or through a `return` inside a block body.

```csharp
app.Use(next => context => next(context));            // reported
app.Use(next => async context => { await next(context); }); // reported
app.Use(next => { return context => next(context); }); // reported
```

## What is not reported

| shape | why |
| --- | --- |
| `app.Use(async (context, next) => ...)` | already the two-parameter overload |
| `app.Use(next => next)` | returns the next delegate unchanged — no nested closure |
| `app.Use(next => Wrap(next))` | the handler is produced elsewhere, not an inline lambda |
| a `Use` on a type that is not an `IApplicationBuilder` | not the middleware pipeline |

The whole rule stays silent unless
`Microsoft.AspNetCore.Builder.IApplicationBuilder` is referenced, so a non-web
project pays nothing.

## How to fix

Move the `context` parameter up beside `next`, drop the nesting, and call the
next delegate the same way:

```csharp
// before
app.Use(next => async context =>
{
    await next(context);
});

// after
app.Use(async (context, next) =>
{
    await next(context);
});
```

## Performance

The rule is gated at compilation start on `IApplicationBuilder` resolving, so a
project that does not reference ASP.NET Core registers no syntax action. The
clean path is a name check on the invoked member, an argument-count check, and a
syntactic probe of the argument lambda's shape; the semantic model is consulted
only once the syntax already matches, to confirm the call binds to the legacy
`Func<RequestDelegate, RequestDelegate>` overload on a builder that implements
`IApplicationBuilder`.
