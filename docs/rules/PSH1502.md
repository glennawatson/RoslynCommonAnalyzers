# PSH1502 — Route handler returns a deferred sequence

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | AspNetCore |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (in ASP.NET Core projects) |

## Cause

A web route handler — a minimal-API map lambda or a controller action — returns a
value typed as `IEnumerable<T>` (or `Task<IEnumerable<T>>` /
`ValueTask<IEnumerable<T>>`) whose returned expression is a **deferred query**:
an `IQueryable<T>` or a lazy LINQ chain such as `Where`/`Select`/`OrderBy`.

```csharp
using Microsoft.AspNetCore.Mvc;

public sealed class WidgetsController : ControllerBase
{
    private readonly IQueryable<Widget> _query;

    // 👎 reported — the returned IQueryable is enumerated synchronously while serializing the response
    [HttpGet]
    public IEnumerable<Widget> Get() => _query.Where(w => w.Active);
}
```

```csharp
using Microsoft.AspNetCore.Builder;

// 👎 reported — a lazy in-memory LINQ result handed back from a minimal-API handler
app.MapGet("/widgets", () => items.Where(w => w.Active));
```

```csharp
// 👍 stream the results asynchronously
app.MapGet("/widgets", () => Stream());

async IAsyncEnumerable<Widget> Stream() { /* ... */ }

// 👍 materialize before returning
[HttpGet]
public async Task<IEnumerable<Widget>> Get() => await _query.Where(w => w.Active).ToListAsync();
```

## Rule description

When a route handler hands the response serializer a sequence that has not yet
been enumerated, the JSON serializer drives that enumeration **synchronously on
the request thread**. If the sequence is an `IQueryable<T>`, the database round
trip runs there; if it is a lazy LINQ chain, the per-element projection or
filtering runs there. While the thread is blocked on that work it cannot serve
other requests, so under load the thread pool starves and latency climbs.

Returning `IAsyncEnumerable<T>` instead lets the framework stream the results
asynchronously, and materializing the query first (for example with
`ToListAsync`/`ToArrayAsync`) moves the work off the serialization path and lets
the handler `await` it.

Two shapes are reported:

- **Minimal API** — a lambda passed to a `Microsoft.AspNetCore.Builder` map
  method (`MapGet`, `MapPost`, `MapPut`, `MapDelete`, `MapPatch`, or `Map`) whose
  return type is `IEnumerable<T>` (or the task-wrapped forms) and whose returned
  expression is deferred.
- **MVC** — a public instance method on a `Microsoft.AspNetCore.Mvc.ControllerBase`
  subclass whose return type is `IEnumerable<T>` (or the task-wrapped forms) and
  whose returned expression is deferred. A method marked `[NonAction]`, a private
  or static method, and a method with a materialized return type are not reported.

### What is not reported

The rule stays quiet unless it can prove the returned value is deferred, so it
does not fire on an already-materialized collection even when that collection is
typed as `IEnumerable<T>`:

```csharp
// 👍 not reported — the query is materialized before it leaves the handler
public IEnumerable<Widget> Get() => _query.Where(w => w.Active).ToList();
```

It also leaves `IAsyncEnumerable<T>` handlers alone — that is the correct
streaming shape — and never reports a handler that returns a concrete
`List<T>`, an array, or an `IReadOnlyList<T>`.

### Availability detection

The rule is gated on `Microsoft.AspNetCore.Builder.WebApplication` or
`Microsoft.AspNetCore.Mvc.ControllerBase` resolving in the referenced framework
(detected once per compilation via `Compilation.GetTypeByMetadataName`). A
project that references neither registers no analysis and pays nothing. No target
framework string is parsed.

## How to fix violations

- Return `IAsyncEnumerable<T>` so the framework streams the results without
  blocking the request thread, **or**
- Materialize the query before returning it — `await query.ToListAsync()` for a
  provider that supports it, or `query.ToList()` / `query.ToArray()` for an
  in-memory sequence — so the work happens off the serialization path.
