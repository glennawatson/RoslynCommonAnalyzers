# PSH1503 — Prefer output caching over response caching

**Category:** AspNetCore
**Severity:** Warning
**Code fix:** None. Output caching is configured differently — see [How to fix](#how-to-fix).

The response-caching middleware is a client-driven cache: it only writes and
honors HTTP `Cache-Control` headers. In practice that means it caches nothing a
request's headers forbid, skips authenticated requests and anything that is not a
`GET`/`HEAD`, and hands the application no key or eviction control — a cache the
server cannot actually manage. Under load it is easy to end up with a "cache" that
never hits.

Output caching (.NET 7+) is the server-side answer. It stores the rendered
response on the server under keys and tags that you choose, caches requests the
header-driven cache would refuse, and can be invalidated on demand by tag. It is
the recommended approach for app-controlled server-side caching.

## What is reported

A single, local registration of the legacy middleware:

- a call to `AddResponseCaching(...)` that binds to the response-caching
  service-collection extensions;
- a call to `UseResponseCaching(...)` that binds to the application-builder
  extensions.

The invoked method is bound before anything is reported, so a method of your own
that merely shares the name `AddResponseCaching` or `UseResponseCaching` is never
confused with the framework one.

## Example

```csharp
// PSH1503: the legacy, client-header-driven cache
builder.Services.AddResponseCaching();

var app = builder.Build();

app.UseResponseCaching();
```

```csharp
// fixed: server-controlled output caching
builder.Services.AddOutputCache();

var app = builder.Build();

app.UseOutputCache();
```

## How to fix

Output caching is not a drop-in rename — it is opt-in per endpoint, so there is no
mechanical code fix:

1. Replace `AddResponseCaching()` with `AddOutputCache()` (optionally configuring
   policies on the `OutputCacheOptions`).
2. Replace `UseResponseCaching()` with `UseOutputCache()`. Place it after
   `UseRouting()` and before the endpoints, as with the response-caching
   middleware.
3. Opt endpoints in. Nothing is cached until an endpoint asks for it:

   ```csharp
   app.MapGet("/products", GetProducts).CacheOutput();

   // or on a controller action / Razor page
   [OutputCache(Duration = 60)]
   public IActionResult Products() => View();
   ```

4. Where you need to evict, tag entries and invalidate the tag:

   ```csharp
   builder.Services.AddOutputCache(options =>
       options.AddPolicy("products", policy => policy.Tag("products")));

   // later
   await outputCacheStore.EvictByTagAsync("products", cancellationToken);
   ```

## When not to use the rule

Keep the response-caching middleware — and turn this rule off — when the goal is
specifically to emit and honor HTTP caching headers for downstream/proxy caches
rather than to cache on the server. Scope the relaxation to the project or folder
that needs it:

```ini
# .editorconfig
[*.cs]
dotnet_diagnostic.PSH1503.severity = none
```

## Performance

The whole rule is gated at compilation start on the output-caching API resolving
(`OutputCacheServiceCollectionExtensions` or `OutputCacheOptions`), so a project
that cannot adopt the suggestion — or is not a web app at all — registers no syntax
action and pays nothing. The clean path is a single method-name token comparison;
only a name-matched invocation is bound to confirm its containing type, so the
common no-diagnostic path never touches the semantic model.
