# PSH1505 — Handle exceptions in an IExceptionHandler, not an MVC exception filter

**Category:** AspNetCore
**Severity:** Warning
**Code fix:** None.

An MVC exception filter — a class implementing `IExceptionFilter` or
`IAsyncExceptionFilter` — runs inside the MVC action-invocation pipeline. It only
sees exceptions raised once routing has selected an action and the heavier filter
pipeline has been entered, and it is invoked per matched action. That is the wrong
place for cross-cutting error handling: exceptions thrown by middleware, routing,
or model binding never reach it, and the work runs inside the per-action filter
path.

`IExceptionHandler` (`Microsoft.AspNetCore.Diagnostics`) is the centralized
primitive. Registered with `AddExceptionHandler<T>()` and activated with
`UseExceptionHandler()`, it is invoked once by the middleware pipeline for any
unhandled exception in the request, with less per-request overhead than the filter
path.

## Example

```csharp
using Microsoft.AspNetCore.Mvc.Filters;

// PSH1505: cross-cutting handling inside the MVC filter pipeline
public class GlobalExceptionFilter : IExceptionFilter
{
    public void OnException(ExceptionContext context)
    {
        context.Result = Problem(context.Exception);
        context.ExceptionHandled = true;
    }
}
```

```csharp
using Microsoft.AspNetCore.Diagnostics;
using Microsoft.AspNetCore.Http;

// fixed: one handler the pipeline runs once for any unhandled exception
public sealed class GlobalExceptionHandler : IExceptionHandler
{
    public ValueTask<bool> TryHandleAsync(
        HttpContext httpContext,
        Exception exception,
        CancellationToken cancellationToken)
    {
        // write the response, then return true to mark the exception handled
        return ValueTask.FromResult(true);
    }
}

// Program.cs
// builder.Services.AddExceptionHandler<GlobalExceptionHandler>();
// app.UseExceptionHandler();
```

## What is reported

Only a class that names one of the two filter interfaces **directly in its own
base list**, confirmed by binding:

- `class Foo : IExceptionFilter`
- `class Foo : IAsyncExceptionFilter`
- the fully qualified or `using`-aliased spellings of either

The diagnostic points at the class name. A class that implements both interfaces
is reported once.

Silent by construction:

- a class whose base list names neither interface;
- a class that reaches a filter interface only through a base class or a derived
  interface — the class that names it directly is the one reported, so nothing is
  reported twice;
- a same-named `IExceptionFilter` from another namespace — the written interface is
  bound and confirmed to be the ASP.NET Core one, so a look-alike of your own is
  never confused with it;
- a class implementing the modern `IExceptionHandler`.

## When not to use the rule

A filter that genuinely needs action-specific context — it inspects the selected
action or its arguments, or is applied to one controller by attribute — is a
deliberate choice that `IExceptionHandler` cannot replicate, and the suggestion
does not apply there. Turn the rule off where that is the intent:

```ini
[*.cs]
dotnet_diagnostic.PSH1505.severity = none
```

## Performance

The whole rule is gated at compilation start on
`Microsoft.AspNetCore.Diagnostics.IExceptionHandler` resolving — the modern
replacement must exist for the suggestion to be actionable — and on at least one of
the two filter interfaces resolving. A non-web project registers no syntax action
at all. When active, the clean path is a base-list walk with a token comparison per
base type; a class is bound only after its base list syntactically names one of the
filter interfaces, so the semantic model is touched only for real candidates.
