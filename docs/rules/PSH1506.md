# PSH1506 — Synchronous read or write of the HTTP request or response body

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | AspNetCore |
| Severity | Warning |
| Code fix | Yes (conditional) |
| Default | Enabled (in ASP.NET Core projects) |

## Cause

The HTTP request or response body is read or written with a **synchronous** stream
call on the request thread:

- `Read`, `ReadByte`, `Write`, `WriteByte`, `Flush`, or `CopyTo` on
  `HttpRequest.Body` / `HttpResponse.Body`, or
- `ReadToEnd`, `ReadLine`, or `ReadBlock` on a `StreamReader` (or `StreamWriter`)
  constructed over one of those streams.

The receiver may be the body property written inline, or a local variable
initialised from it.

```csharp
using System.IO;
using Microsoft.AspNetCore.Http;

public sealed class UploadHandler
{
    // 👎 reported — the whole request body is read synchronously on the request thread
    public string Read(HttpContext ctx) => new StreamReader(ctx.Request.Body).ReadToEnd();

    // 👎 reported — a synchronous write of the response body
    public void Write(HttpContext ctx, byte[] payload)
        => ctx.Response.Body.Write(payload, 0, payload.Length);
}
```

```csharp
// 👍 await the async overload instead
public async Task<string> ReadAsync(HttpContext ctx)
    => await new StreamReader(ctx.Request.Body).ReadToEndAsync();

public async Task WriteAsync(HttpContext ctx, byte[] payload)
    => await ctx.Response.Body.WriteAsync(payload, 0, payload.Length);
```

## Rule description

The request and response bodies are backed by the network connection. A
synchronous call on them waits on socket I/O while holding the pooled thread that
was serving the request — the classic sync-over-async blocking that starves the
thread pool under load: the threads that should be picking up new requests are
instead parked on connections. The synchronous form also buffers the payload
**without a bound**, so a large or slow transfer is held in memory in full.

On top of the performance cost, ASP.NET Core disables synchronous request and
response body I/O by default, so these calls also **throw at runtime** unless that
safeguard is explicitly turned off.

Awaiting the asynchronous overload (`ReadAsync`, `WriteAsync`, `FlushAsync`,
`ReadToEndAsync`, and their siblings) hands the thread back to the pool while the
I/O is outstanding, so the server keeps serving other requests.

### What is reported

- A synchronous I/O call whose receiver **binds to** `HttpRequest.Body` or
  `HttpResponse.Body` — directly, through a `StreamReader`/`StreamWriter` wrapping
  it, or through a local initialised from either.
- Only when the matching asynchronous overload actually exists on the receiver's
  type. The suggested member (`ReadAsync`, `ReadToEndAsync`, …) is resolved on the
  receiver, so a type that has no async form of the call is never reported.

### What is not reported

- A synchronous call on any other stream (a `MemoryStream`, a file stream, …) — the
  receiver has to be the HTTP body.
- A call that is already asynchronous.

### Availability detection

The rule is gated on `Microsoft.AspNetCore.Http.HttpRequest` resolving in the
referenced framework (detected once per compilation via
`Compilation.GetTypeByMetadataName`). A project that does not reference ASP.NET
Core registers no analysis and pays nothing. No target framework string is parsed.

## How to fix violations

Await the asynchronous overload of the call, for example:

- `stream.Read(buffer, offset, count)` → `await stream.ReadAsync(buffer, offset, count)`
- `stream.Write(buffer, offset, count)` → `await stream.WriteAsync(buffer, offset, count)`
- `stream.Flush()` → `await stream.FlushAsync()`
- `new StreamReader(body).ReadToEnd()` → `await new StreamReader(body).ReadToEndAsync()`
- reading a whole JSON body → `await JsonSerializer.DeserializeAsync<T>(body)`

### Code fix

An automatic fix is offered **only when the enclosing method, local function, or
lambda is already `async`**, so the inserted `await` is legal, and only when the
asynchronous sibling awaits to the synchronous call's own result and accepts the
very arguments already written. The rewritten call is bound speculatively before
the fix is offered, so a fix that would not compile is never suggested.

A call whose async form has a different signature (for example `ReadByte()`, which
has no single-byte async counterpart) and a call in a synchronous method are
reported without a fix — moving the method to `async` is a decision only its author
can make.
