# PSH1600 — Delegate captures the loop variable inside a component render loop

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Blazor |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (in Blazor projects) |

## Cause

A lambda or anonymous method inside a `for`/`foreach` render loop reads a variable
declared by that loop — the iteration variable, a `for` counter, or a local
declared in the loop body. Because the captured value changes every iteration, the
compiler allocates a distinct closure and delegate for each row, and the whole set
is rebuilt on every render.

```razor
@* 👎 reported — a fresh closure over `item` is allocated for every row on every render *@
@foreach (var item in Items)
{
    <button @onclick="() => Select(item)">Select @item.Name</button>
}
```

```csharp
// 👎 reported — the hand-written render override has the same per-row capture
protected override void BuildRenderTree(RenderTreeBuilder builder)
{
    foreach (var item in Items)
    {
        builder.AddAttribute(1, "onclick", EventCallback.Factory.Create(this, () => Select(item)));
    }
}
```

```razor
@* 👍 precompute a per-item model that owns the handler, so no closure is built per render *@
@foreach (var row in Rows)
{
    <button @onclick="row.OnSelect">Select @row.Name</button>
}
```

## Rule description

A component builds its render output by running `BuildRenderTree` on every render.
A `.razor` component's `@foreach`/`@for` body compiles into exactly that shape, so a
`@onclick="() => Select(item)"` inside the loop becomes an
`EventCallback.Factory.Create(this, () => Select(item))` whose lambda captures the
iteration variable.

When the captured value changes each iteration, the compiler cannot cache the
closure: it allocates a new display-class instance and delegate for **every row on
every render**. A list of N rows therefore costs N closure allocations each time the
component re-renders — and Blazor re-renders on every state change. The new delegate
identities also change the render diff, so the framework treats each handler as a
new attribute and re-attaches it.

Measured on .NET 9, one render pass over a loop whose `@onclick` captures the
iteration variable allocates a fixed ~128 bytes per row (the loop-variable closure
plus the delegate and event-callback wrapper); at 1,000 rows that is ~128 KB of
garbage produced on **every** render, repeated for the lifetime of the component.
Precomputing a per-item model that owns the handler removes the per-row closure
entirely.

### What is not reported

The rule fires only when the delegate captures a variable declared **by the loop**,
so the shapes whose identity does not change per row are left alone:

```csharp
// 👍 not reported — loop-invariant lambda; its identity is the same every iteration
foreach (var item in Items)
{
    builder.AddAttribute(1, "onclick", () => Select(0));
}

// 👍 not reported — a method group captures no loop variable
foreach (var item in Items)
{
    builder.AddAttribute(1, "onclick", Refresh);
}

// 👍 not reported — the delegate is hoisted out of the loop and reused
Action handler = () => Select(0);
foreach (var item in Items)
{
    builder.AddAttribute(1, "onclick", handler);
}
```

A lambda that reads only a field or a variable from outside the loop is also left
alone, because it does not vary per row. When a capturing lambda is nested inside
another anonymous function, only the outer one is reported — it is the delegate that
is rebuilt per iteration.

### Availability detection

The rule is gated on
`Microsoft.AspNetCore.Components.Rendering.RenderTreeBuilder` resolving in the
referenced framework (detected once per compilation via
`Compilation.GetTypeByMetadataName`). A project that does not reference Blazor
registers no analysis and pays nothing. The loop's containing method is confirmed to
be a `BuildRenderTree` method whose single parameter is that render-tree builder, so
the report never leaves that render path. Generated code is analyzed, because a
`.razor` component's render body is generated; the diagnostic maps back to the
original `.razor` line.

## How to fix violations

- Precompute a per-item model that owns the handler (for example an `EventCallback`
  or `Action` field created once), and bind that member in the loop, **or**
- Bind a method group that does not capture the iteration variable, **or**
- Cache the delegate in a field when a single shared handler is enough, so one
  delegate is reused across renders instead of one being allocated per row.
