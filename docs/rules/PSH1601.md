# PSH1601 — JavaScript-interop call issued once per loop iteration

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Blazor |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (in Blazor projects) |

## Cause

An `IJSRuntime` or `IJSObjectReference` interop call — `InvokeAsync` or
`InvokeVoidAsync` — is written directly inside a `for`/`foreach` loop, so it runs
once per item.

```csharp
using System.Collections.Generic;
using Microsoft.JSInterop;

public sealed class Highlighter
{
    // 👎 reported — one interop call per row
    public async Task ApplyAsync(IJSRuntime js, List<Row> rows)
    {
        foreach (var row in rows)
        {
            await js.InvokeVoidAsync("app.highlight", row.Id);
        }
    }
}
```

```csharp
// 👍 batch the whole collection into a single interop call
public async Task ApplyAsync(IJSRuntime js, List<Row> rows)
{
    var ids = new int[rows.Count];
    for (var i = 0; i < rows.Count; i++)
    {
        ids[i] = rows[i].Id;
    }

    await js.InvokeVoidAsync("app.highlightAll", ids);
}
```

## Rule description

Under the Interactive Server hosting model a component runs on the server and each
interop call is marshalled to the browser over the SignalR circuit. A call placed
inside a loop therefore turns a collection of N items into **N separate network
round-trips**: the circuit latency is paid once per item, and the circuit itself is
saturated with chatter that delays every other update. The cost is invisible on a
WebAssembly host, where interop is a local call, but the same component running on a
server circuit stalls.

Passing the whole collection to a single JavaScript function that does the per-item
work on the client collapses the N round-trips into one.

### What is reported

- An `InvokeAsync`/`InvokeVoidAsync` call whose receiver is an `IJSRuntime` or
  `IJSObjectReference` (or a type that implements either), written **directly** in a
  `for`/`foreach` body.

### What is not reported

- A single interop call outside any loop.
- A call nested inside a lambda or local function within the loop — that function's
  own body owns its cost.
- A same-named call on an unrelated receiver.

### Availability detection

The rule is gated on `Microsoft.JSInterop.IJSRuntime` resolving in the referenced
framework (detected once per compilation via `Compilation.GetTypeByMetadataName`). A
project that does not reference Blazor registers no analysis and pays nothing. No
target framework string is parsed.

## How to fix violations

Move the interop out of the loop: build the whole payload first, then make one
interop call that hands the collection to a JavaScript function which loops on the
client. There is no automatic fix — collapsing per-item calls into a batched API is a
design change only the author can make.
