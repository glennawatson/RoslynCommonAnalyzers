# PSH1602 — StateHasChanged is called unconditionally in a render callback

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Blazor |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (in Blazor projects) |

## Cause

A component's `OnAfterRender` / `OnAfterRenderAsync` reaches `StateHasChanged()` on
its unconditional path — the call is not guarded by the `firstRender` parameter or a
boolean state flag.

```csharp
using Microsoft.AspNetCore.Components;

public sealed class Ticker : ComponentBase
{
    // 👎 reported — schedules another render after every render
    protected override void OnAfterRender(bool firstRender)
    {
        StateHasChanged();
    }
}
```

```csharp
// 👍 run the post-render work once, behind the firstRender guard
protected override void OnAfterRender(bool firstRender)
{
    if (firstRender)
    {
        StateHasChanged();
    }
}
```

## Rule description

`OnAfterRender` and `OnAfterRenderAsync` run **after every render pass**. Calling
`StateHasChanged()` on their unconditional path requests yet another render, whose
callback calls `StateHasChanged()` again — a render loop that pins the CPU and, on
Interactive Server, floods the SignalR circuit with a continuous stream of render
deltas.

The callback receives a `firstRender` flag for exactly this reason. Post-render work
that must run once (measuring the DOM, initialising a JavaScript widget) belongs
behind `if (firstRender)`; work that must run only when some state actually changed
belongs behind that state's own guard.

### What is reported

- A `StateHasChanged()` call (bare or on `this`) that `OnAfterRender` /
  `OnAfterRenderAsync` reaches with no intervening `if` or other branch. A call that
  the callback reaches after an early-return guard is still unconditional on the paths
  that continue, and is reported.

### What is not reported

- A call already inside an `if` — whether guarded by `firstRender` or a boolean state
  flag.
- A call deferred into a lambda or local function.
- A `StateHasChanged()` in any other method, or a same-named method on a type that is
  not a component.

### Availability detection

The rule is gated on `Microsoft.AspNetCore.Components.ComponentBase` resolving in the
referenced framework (detected once per compilation via
`Compilation.GetTypeByMetadataName`). A project that does not reference Blazor
registers no analysis and pays nothing. No target framework string is parsed.

## How to fix violations

Guard the call. Put one-time post-render work behind `if (firstRender)`, and put
state-dependent work behind a boolean flag that you set once the state settles, so
the extra render is requested only when it is genuinely needed. There is no automatic
fix — which guard is correct depends on what the render is meant to react to.
