# PSH1603 — Allocation used as a component parameter inside a render loop

| Item | Value |
| --- | --- |
| Package | PerformanceSharp.Analyzers |
| Category | Blazor |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (in Blazor projects) |

## Cause

A non-delegate allocation — a `new` object, an array or collection literal, or a
materializing query (`ToList`/`ToArray`/…) — is passed as a child component's
parameter value inside a `for`/`foreach` render loop.

```razor
@* 👎 reported — a fresh list is allocated for every row on every render *@
@foreach (var row in Rows)
{
    <RowView Tags="new List<string> { row.Category }" />
}
```

The `@foreach` above compiles into a `BuildRenderTree` loop whose component parameter
is set from a `new List<string> { ... }`:

```csharp
foreach (var row in Rows)
{
    builder.OpenComponent<RowView>(0);
    builder.AddComponentParameter(1, "Tags", new List<string> { row.Category });
    builder.CloseComponent();
}
```

```razor
@* 👍 precompute the per-item value once, so its identity is stable across renders *@
@foreach (var row in Rows)
{
    <RowView Tags="row.Tags" />
}
```

## Rule description

A component rebuilds its render output by running `BuildRenderTree` on every render.
When a component parameter's value is allocated inside the render loop, a fresh
instance is created for **every row on every render** — a list of N rows costs N
allocations each pass, straight onto the GC.

The reference identity matters as much as the allocation. A child component compares
each incoming parameter by reference to decide whether it changed; a brand-new
instance every render never equals the previous one, so the child is treated as
changed and **re-rendered even when its data did not move**. Hoisting the value so its
identity is stable — caching it in a field, or precomputing a per-item model once and
reusing it — lets the child skip the redundant render.

This is the non-delegate sibling of [PSH1600](PSH1600.md), which reports the delegate
form of the same defect. A delegate or lambda passed as a component parameter is left
to PSH1600; this rule covers the non-delegate allocation only.

### What is reported

- A `new` object, an array literal (`new[] { … }`, `new T[] { … }`), a collection
  created with an initializer (`new List<T> { … }`), or a materializing query
  (`ToList`, `ToArray`, `ToHashSet`, `ToDictionary`, `ToLookup`) passed as the value
  of a `RenderTreeBuilder.AddComponentParameter(...)` call directly inside a
  `for`/`foreach` in a component's `BuildRenderTree`.

### What is not reported

- A delegate creation (covered by PSH1600) — a `new SomeDelegate(...)` value is left
  alone.
- A plain value, field, or method call that does not allocate — a `ToList`-shaped call
  on a type other than the query type is not a materializer.
- An allocation outside a render loop, or in a loop that is not a component's
  `BuildRenderTree`.

### Availability detection

The rule is gated on
`Microsoft.AspNetCore.Components.Rendering.RenderTreeBuilder` resolving in the
referenced framework (detected once per compilation via
`Compilation.GetTypeByMetadataName`). A project that does not reference Blazor
registers no analysis and pays nothing. No target framework string is parsed.

## How to fix violations

Move the allocation off the per-row render path. Precompute the value once — in the
component's state, on a per-item model, or in a cached field — and pass that stable
reference as the parameter, so the value is allocated once and the child re-renders
only when its data actually changes. There is no automatic fix — where the value
should live depends on the component's data flow.
