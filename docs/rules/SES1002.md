# SES1002 — Password-based key derivation must not use a constant or predictable salt

**Category:** Cryptography
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

PBKDF2 — reached through the `System.Security.Cryptography.Rfc2898DeriveBytes`
constructor or the static `Rfc2898DeriveBytes.Pbkdf2` method — derives a key from
a password and a salt. The salt exists so that the *same* password yields a
*different* derived key for every credential. That per-secret uniqueness forces an
attacker to attack each hash on its own and makes precomputed rainbow tables
useless.

A fixed salt throws that away. Hard-code the salt once and a single precomputed
table cracks every password stored with it, and two users who happen to pick the
same password end up with identical derived keys — visible in the store. The rule
reports the salt argument when it is a value baked into the program:

```csharp
using System.Security.Cryptography;
using System.Text;

// all reported (SES1002):
new Rfc2898DeriveBytes(password, new byte[16], 100_000, HashAlgorithmName.SHA256);          // all-zero buffer
new Rfc2898DeriveBytes(password, new byte[] { 1, 2, 3, 4 }, 100_000, HashAlgorithmName.SHA256); // literal bytes
new Rfc2898DeriveBytes(password, SharedSalt, 100_000, HashAlgorithmName.SHA256);            // static readonly field
Rfc2898DeriveBytes.Pbkdf2(password, Encoding.UTF8.GetBytes("app-salt"), 100_000, HashAlgorithmName.SHA256, 32); // constant string

private static readonly byte[] SharedSalt = new byte[16];
```

The fix is to generate a fresh cryptographically random salt for every credential,
store it alongside the hash, and pass that:

```csharp
byte[] salt = RandomNumberGenerator.GetBytes(16);   // unique per credential, persisted with the hash
byte[] key = Rfc2898DeriveBytes.Pbkdf2(password, salt, 100_000, HashAlgorithmName.SHA256, 32);
```

## What is reported

The salt argument is flagged when it is one of:

- an inline `new byte[N]` with no initializer (an all-zero buffer that nothing
  writes to before the call);
- an inline byte array whose every element is a compile-time constant;
- a reference to a `static readonly` field (allocated once and shared by every
  call);
- `Encoding.GetBytes` over a constant string.

A salt read from a fresh `RandomNumberGenerator.GetBytes` call, a per-instance
field, a method result, or any other runtime value is not reported. The
`Rfc2898DeriveBytes(string password, int saltSize, …)` overloads generate a random
salt internally — their parameter is `saltSize`, not `salt`, so they are never
reported.

Only the salt supplied *at the call* is examined. A fixed salt copied into a local
variable first, or one that flows in from another method, is not tracked — that
would require whole-program data-flow analysis, which this rule deliberately does
not do, to keep it fast and free of false positives.

## Why there is no code fix

There is no safe automatic rewrite. The correct salt is a fresh random value that
must be **persisted with each hash** so the same salt can be supplied again when
the password is later verified; an analyzer cannot invent that storage. Silently
replacing the constant with `RandomNumberGenerator.GetBytes(16)` inline would
derive a different key on every run and lock every user out. The author has to wire
the per-credential salt through storage, so the rule reports and leaves the change
to them.

## Proving the API exists

The rule resolves `System.Security.Cryptography.Rfc2898DeriveBytes` in the analyzed
compilation and registers nothing when it is absent, so a project targeting a
framework without the type receives no diagnostic. The static `Pbkdf2` method is a
member of that same type, so no separate probe is needed.

## Performance

The no-diagnostic path is syntactic. An object creation whose type is not spelled
`Rfc2898DeriveBytes`, or an invocation whose member is not `Pbkdf2`, is rejected
before the semantic model is touched; only a matching shape is bound to confirm the
type and locate the salt argument.
