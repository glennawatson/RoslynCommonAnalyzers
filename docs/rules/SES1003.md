# SES1003 — Password-based key derivation must use a sufficient iteration count

| Item | Value |
| --- | --- |
| Category | Cryptography |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `System.Security.Cryptography.Rfc2898DeriveBytes.Pbkdf2` one-shot call is given a compile-time
constant `iterations` argument that is strictly below the configured floor (default `100000`).

```csharp
using System.Security.Cryptography;

public byte[] DeriveKey(byte[] password, byte[] salt)
    => Rfc2898DeriveBytes.Pbkdf2(password, salt, 1000, HashAlgorithmName.SHA256, 32);
```

## Rule description

PBKDF2 turns a password into a key by running a pseudo-random function over and over. The iteration
count is the only knob that makes that work expensive, and it is a defence against an *offline*
attack: once an attacker has the salt and the derived key — from a database leak, a stolen backup, a
captured file — they guess passwords locally, as fast as their hardware allows, with no server to
rate-limit them. A low iteration count means each guess is cheap, so a commodity GPU grinds through
billions of candidates per second and recovers the password. A high count deliberately slows every
guess by the same factor, turning a minutes-long crack into a computationally infeasible one.

`1000` iterations was the historical default and is now far too low. Modern guidance puts the floor
for PBKDF2-HMAC-SHA256 at 100,000 and above; raise the number you pass as hardware gets faster.

```csharp
public byte[] DeriveKey(byte[] password, byte[] salt)
    => Rfc2898DeriveBytes.Pbkdf2(password, salt, 600000, HashAlgorithmName.SHA256, 32);
```

The rule reports the `iterations` argument whether it is passed positionally or by name, and it
resolves a `const` reference to its constant value, so `Pbkdf2(password, salt, LowConst, …)` is
reported at `LowConst`.

## What is not reported

- **A non-constant iteration count.** When the count is a method parameter, a field, or a value read
  from configuration, its value cannot be judged from the source — it may well be `600000` at
  runtime — so the rule stays silent rather than guess. Read the count from configuration and this
  rule never fires; that is by design, not a gap.

  ```csharp
  // Not reported: the value is unknown at compile time.
  public byte[] DeriveKey(byte[] password, byte[] salt, int iterations)
      => Rfc2898DeriveBytes.Pbkdf2(password, salt, iterations, HashAlgorithmName.SHA256, 32);
  ```

- **A count at or above the floor.** `100000` (the default) and anything larger is fine.

- **The obsolete instance constructors.** `new Rfc2898DeriveBytes(password, salt, 1000)` and the
  `IterationCount` property are marked obsolete by the platform (*"use the static `Pbkdf2` method
  instead"*), which already steers you off them. This rule guards only the modern static one-shot,
  where the count is a direct argument to a single call — a purely local shape that needs no flow
  analysis and carries near-zero false positives.

- **A `Pbkdf2` method on any type other than `Rfc2898DeriveBytes`.** The call is matched by its bound
  symbol, not by the method name, so an unrelated helper named `Pbkdf2` is left alone.

## How to fix violations

Pass a fixed iteration count at or above the floor, and raise it over time. Where the exact number
is a policy you want to tune without recompiling, read it from configuration — the rule then treats
the count as non-constant and stays silent, leaving the policy to your configuration and this rule to
catch hard-coded low numbers. There is deliberately **no code fix**: the right value is a security
decision that depends on your threat model and hardware, not a mechanical substitution.

## Configuration

```ini
# Rule-specific
securitysharp.SES1003.iterations = 100000

# Project-wide (applies to any SecuritySharp rule that reads an iteration floor)
securitysharp.iterations = 100000
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `iterations` | integer `>= 1` | `100000` | The smallest iteration count a constant `Pbkdf2` call may use before it is reported. |

The rule-specific key wins over the project-wide key. An unset, unparsable, or nonsensical value
(below `1`) falls back to the default, so a typo cannot silently disable the rule.

## Performance

The rule gates on `System.Security.Cryptography.Rfc2898DeriveBytes` resolving in the compilation, so
a target framework without it registers nothing and pays nothing. Each invocation is filtered
syntactically first — only a `.Pbkdf2(...)` member call with arguments reaches the semantic model —
and the `.editorconfig` floor is read only after a constant iteration count below any plausible floor
has already been found, keeping the common no-diagnostic path allocation-free.
