# SES1004 — A secret must not be produced from Guid.NewGuid()

| Item | Value |
| --- | --- |
| Category | Cryptography |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A value produced by `Guid.NewGuid()` flows directly into a local, field, property, parameter, or return
whose name reads as a secret.

```csharp
public string CreateApiKey() => Guid.NewGuid().ToString(); // SES1004
```

## Rule description

A GUID is built to be *unique*, not *unguessable*. It carries at most 122 bits of entropy, its layout is
public, and it is routinely logged, embedded in URLs, and returned in responses — all harmless for an
identifier and all fatal for a secret. Minting a bearer token, API key, password, nonce, salt, session id,
one-time code, or password-reset token from `Guid.NewGuid()` hands an attacker a value that is far weaker
than it looks and, on some platforms, is not drawn from a cryptographic source at all.

Because a GUID is a perfectly good *identifier*, the rule does not flag `Guid.NewGuid()` on its own. It fires
only when the GUID's value (optionally stringified through `.ToString(...)`) flows directly into a target
whose name matches a curated, high-precision secret vocabulary, matched on word boundaries:

- `token`, `secret`, `password` / `passwd` / `pwd`, `nonce`, `salt`, `otp`;
- an API key (`apiKey`, `api_key`, `APIKEY`);
- a session id or session key (`sessionId`, `sessionKey`);
- a verification code (`verificationCode`);
- a reset token (`resetToken`).

Matching is on whole words split from the identifier (on case changes, digits, and separators), so an
ordinary GUID used as an id — `correlationId`, `tokenizer`, `sessionValue` — is never touched. The rule
covers only *local* shapes: a declaration initializer, a simple assignment, a return or expression body, or
an argument bound to a secret-named parameter. It does not trace a GUID through intermediate variables,
conditionals, or lambdas.

The diagnostic is reported only when `System.Security.Cryptography.RandomNumberGenerator` is available in the
compilation, so a project that cannot act on the suggestion never receives it.

## How to fix violations

Draw the secret from a cryptographic random source sized for what you need, rather than from a GUID:

```csharp
// A url-safe token:
public string CreateApiKey() => RandomNumberGenerator.GetHexString(64);

// Raw key bytes:
public byte[] CreateKey() => RandomNumberGenerator.GetBytes(32);

// A numeric one-time code:
public int CreateOtp() => RandomNumberGenerator.GetInt32(0, 1_000_000);
```

Use `GetBytes` / `GetInt32` on any framework that has them, and `GetHexString` / `GetString` where they are
available. On frameworks without the static helpers, fill a buffer with
`RandomNumberGenerator.Create().GetBytes(buffer)`.

There is no automatic code fix: the correct replacement call and its size depend on the shape of the secret
being minted.

## Performance

The clean path is a syntax test. The rule registers on invocation expressions, rejects anything that is not a
parameterless `.NewGuid()` call, and then reads the target's name straight from syntax; only a name that
matches the secret vocabulary triggers the semantic bind that confirms the call is `System.Guid.NewGuid`. A
project on a framework without `RandomNumberGenerator` registers nothing at all.
