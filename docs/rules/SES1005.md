# SES1005 — Compare secret values in constant time

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Cryptography |
| Severity | Warning |
| Code fix | Yes (byte-buffer `SequenceEqual` only) |
| Default | Enabled (where `CryptographicOperations` exists) |

## Cause

A secret — an HMAC, signature, authentication tag, token, or hash — is compared
with an ordinary short-circuiting equality: `==`/`!=`, `.Equals`, `object.Equals`,
or a `SequenceEqual`.

```csharp
using System.Linq;

// 👎 reported (byte-buffer content comparison that returns on the first mismatch)
bool ok = mac.SequenceEqual(expectedMac);

// 👎 reported (string token compared with ==)
bool ok = token == expectedToken;

// 👍
bool ok = System.Security.Cryptography.CryptographicOperations.FixedTimeEquals(mac, expectedMac);
```

## Rule description

`==`, `.Equals`, and `SequenceEqual` all return as soon as they find the first
differing byte. When one operand is a secret, an attacker who can submit guesses
and measure how long the check takes learns *where* the first difference is, and
so can recover the expected value one byte at a time — enough to forge a valid
authentication tag without ever knowing the key. The defence is a comparison
whose running time depends only on the buffer length, not on where the first
difference falls: `System.Security.Cryptography.CryptographicOperations.FixedTimeEquals`.

The rule reports every one of these shapes when one operand is named like a
secret and both operands are a comparable type:

- **Operators**: `a == b` / `a != b`.
- **Instance equality**: `a.Equals(b)` and the `string.Equals(b, StringComparison)` overloads.
- **Static equality**: `object.Equals(a, b)`.
- **Sequence equality**: an `Enumerable`, `MemoryExtensions`, `string`, or
  `byte[]` `SequenceEqual(a, b)` / `a.SequenceEqual(b)`.

### Detection heuristic

Detection is a curated, high-precision **name-and-type** match, chosen to keep
false positives near zero — the rule never traces values across method calls
(that would need whole-program dataflow), so it fires only on a comparison whose
operands are visibly secret and visibly a buffer:

- **Name.** One operand's identifier or member name must contain one of
  `hmac`, `signature`, `sig`, `mac`, `tag`, `token`, `hash`, `digest`, or
  `secret` (case-insensitive). The names `expected` and `actual` are ordinary in
  test assertions, so they count only inside a method or local function whose own
  name is verify/validate/check-shaped (contains `verify`, `validate`, `check`,
  `compare`, `authenticate`, or `match`). An operand reached through a call —
  `provided.SequenceEqual(ComputeHmac(key))` — is named from the invoked method
  (`ComputeHmac`).
- **Type.** Both operands must be `byte[]`, a byte span
  (`ReadOnlySpan<byte>`/`Span<byte>`), or `string`.
- **Not a constant.** A comparison against a compile-time constant — such as
  `token == ""` or `token == null` — is an emptiness or hard-coded check, not a
  comparison of two runtime secrets, and is never reported.

### Availability detection

This rule suggests a **runtime API**, so it is gated on
`System.Security.Cryptography.CryptographicOperations` resolving in the analyzed
compilation (probed once per compilation). On a target framework without that
type — `netstandard2.0` or .NET Framework — the rule reports nothing rather than
recommending a method that does not exist.

## How to fix violations

Compare the buffers with `CryptographicOperations.FixedTimeEquals`, whose running
time is independent of the contents:

```csharp
// before
bool ok = mac.SequenceEqual(expectedMac);

// after
bool ok = System.Security.Cryptography.CryptographicOperations.FixedTimeEquals(mac, expectedMac);
```

The provided code fix handles the **byte-buffer `SequenceEqual`** shape — the one
place the rewrite is unambiguous, because `SequenceEqual` already compares by
content and `FixedTimeEquals` is its constant-time twin. The fix spells the type
name fully qualified unless `System.Security.Cryptography` is already imported,
and it is only offered when `FixedTimeEquals` resolves.

The other shapes are reported without a fix, because the rewrite is not
mechanical:

- **`byte[]` `==`/`!=` and `object.Equals`/`.Equals`** compare *references*, not
  contents; turning that into a content comparison is a behaviour change, so
  decide it by hand.
- **`string`** comparisons need the text encoded to bytes first, since
  `FixedTimeEquals` takes byte buffers.
- **`SequenceEqual` with an `IEqualityComparer`** has no constant-time
  equivalent.
