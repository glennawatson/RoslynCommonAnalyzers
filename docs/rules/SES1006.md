# SES1006 ‚Äî Persisted Data Protection keys must be encrypted at rest

**Category:** Cryptography
**Severity:** Warning
**Code fix:** None ‚Äî see [Why there is no code fix](#why-there-is-no-code-fix)

ASP.NET Core Data Protection keeps a key ring ‚Äî the private keys that sign and
encrypt every protected payload (authentication cookies, anti-forgery tokens, and
anything else guarded by `IDataProtector`). By default that key ring is stored in a
per-application location and encrypted at rest with a platform key-encryption
mechanism. The moment you select an explicit key repository with a
`PersistKeysTo...` call, that default protection is switched off: the keys are
written to the chosen store in plaintext.

Anyone who can read the store ‚Äî the file share, the database, the blob container,
the cache, or the registry hive ‚Äî then holds the keys and can forge authentication
cookies, strip anti-forgery protection, or decrypt any protected payload. Moving
keys to a shared location is often exactly what you want (so a web farm shares one
key ring), which makes the missing encryption easy to overlook.

The rule reports the `PersistKeysTo...` call when the same configuration chain has
no `ProtectKeysWith...` call:

```csharp
using Microsoft.AspNetCore.DataProtection;

// reported (SES1006) ‚Äî keys written to the share unencrypted:
services.AddDataProtection()
    .PersistKeysToFileSystem(new DirectoryInfo(@"\\share\keys"));
```

Pair the persistence call with a protection call in the same chain:

```csharp
// not reported ‚Äî the key ring is encrypted before it leaves the process:
services.AddDataProtection()
    .PersistKeysToFileSystem(new DirectoryInfo(@"\\share\keys"))
    .ProtectKeysWithCertificate(certificate);
```

## What is reported

A `PersistKeysTo...` call is flagged when both hold:

- it binds to an extension method whose `this` parameter is
  `Microsoft.AspNetCore.DataProtection.IDataProtectionBuilder` ‚Äî one of
  `PersistKeysToFileSystem`, `PersistKeysToDbContext`,
  `PersistKeysToAzureBlobStorage`, `PersistKeysToStackExchangeRedis`, or
  `PersistKeysToRegistry` (the symbol is bound, so a same-named method on any other
  type is ignored); and
- the enclosing scope contains no `ProtectKeysWith...` call bound to the same
  builder ‚Äî one of `ProtectKeysWithCertificate`, `ProtectKeysWithDpapi`,
  `ProtectKeysWithDpapiNG`, or `ProtectKeysWithAzureKeyVault`.

The "scope" is the configuration lambda body (or nearest enclosing lambda), and for
a bare fluent chain it is the single enclosing statement. This is a purely local,
syntactic scan ‚Äî no data-flow. A persistence call and a protection call written as
separate statements *outside* a shared configuration lambda are deliberately not
correlated.

## Why there is no code fix

There is no single safe rewrite. Which protector to apply is an environment choice:
a certificate needs the concrete certificate, DPAPI and DPAPI-NG are Windows-only,
and a key vault needs the vault key identifier. The analyzer cannot supply any of
these, so the rule reports and leaves the choice to the author.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.DataProtection.IDataProtectionBuilder` in
the analyzed compilation and registers nothing when it is absent, so a project
without ASP.NET Core Data Protection receives no diagnostic and pays no analysis
cost.

## Performance

The no-diagnostic path is syntactic. Only an invocation whose called member is one
of the `PersistKeysTo...` names is bound; the scan for a `ProtectKeysWith...` call
runs only after that call is confirmed to extend `IDataProtectionBuilder`, which is
rare.
