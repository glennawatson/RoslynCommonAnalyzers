# SES1007 — Do not hand-implement a cryptographic primitive

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Cryptography |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (where `System.Security.Cryptography` exists) |

## Cause

A class derives from one of the abstract cryptographic primitive base classes in
`System.Security.Cryptography` — `HashAlgorithm`, `KeyedHashAlgorithm`, `HMAC`,
`SymmetricAlgorithm`, `AsymmetricAlgorithm`, or `DeriveBytes` — which means it
overrides the transform and supplies a hash, keyed hash, cipher, key-exchange, or
key-derivation algorithm of its own.

```csharp
using System.Security.Cryptography;

// 👎 reported (a hash primitive implemented by hand)
public class MyHash : HashAlgorithm
{
    protected override void HashCore(byte[] array, int ibStart, int cbSize) { /* ... */ }
    protected override byte[] HashFinal() => /* ... */;
    public override void Initialize() { }
}

// 👍 use a vetted platform implementation
using var hash = SHA256.Create();
```

## Rule description

These abstract base classes exist to be overridden with the actual algorithm, so
deriving from one is the same as writing your own cryptographic primitive.
Home-grown cryptography is a well-known source of severe, hard-to-spot defects:

- **Timing side channels** — a comparison or table lookup whose duration depends
  on secret data leaks that data a step at a time.
- **Weak or biased constructions** — a hash or MAC that looks plausible but does
  not actually resist collision or forgery.
- **Incorrect padding and block handling** — off-by-one and mode-of-operation
  mistakes that let an attacker recover or tamper with plaintext.
- **Non-constant-time comparisons** of tags and digests.

None of these show up in functional tests: the code round-trips correctly and
still leaves the data unprotected. The platform implementations already handle
them, are widely reviewed, and are hardware-accelerated where possible. Use a
built-in algorithm — `SHA256`, `Aes`, `HMACSHA256`, `RSA`, `Rfc2898DeriveBytes`,
and the rest — or a reviewed cryptographic library.

### What is and is not reported

- **Reported**: a class whose base chain reaches one of the six abstract
  primitive bases, including one reached through a **custom intermediate base**
  (an `abstract class` of your own that derives from the primitive). Each such
  type is flagged, because each one is supplying the primitive by hand.
- **Not reported**: a class that subclasses a **concrete, named algorithm** to
  configure it — `class MyAes : Aes`, `class MyMac : HMACSHA256`, and so on. The
  walk stops at the first already-compiled algorithm in the chain, so reusing a
  vetted implementation (rather than replacing it) is left alone. A class that
  reaches a primitive base only *through* such an algorithm is likewise not
  reported.

### Availability detection

The rule resolves the primitive base types once per compilation. On a target
framework without `System.Security.Cryptography` — where none of the bases exist —
it registers nothing and reports nothing, so a project that cannot derive from
these types pays nothing.

## How to fix violations

Delete the hand-rolled primitive and call a platform algorithm instead. There is
**no automatic code fix**: replacing a bespoke algorithm with a real one is a
design change — the correct replacement depends on what the type was trying to be
(a hash, a MAC, a cipher, or a key-derivation function) and the surrounding call
sites must move to the platform API.

```csharp
// before — a hand-rolled MAC
public class MyMac : HMAC { /* ... */ }

// after — the platform HMAC, configured but not reimplemented
using var mac = new HMACSHA256(key);
byte[] tag = mac.ComputeHash(message);
```

If you genuinely need an algorithm the platform does not ship, take it from a
reviewed cryptographic library rather than implementing the primitive yourself.
