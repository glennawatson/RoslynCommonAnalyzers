# SES1008 — Verify an XML signature against a known key

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Cryptography |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (where `SignedXml` exists) |

## Cause

An XML digital signature is verified with a `SignedXml.CheckSignature` overload
that takes no caller-supplied key — the parameterless `CheckSignature()` or the
single-`bool` `CheckSignature(bool)`.

```csharp
using System.Security.Cryptography.Xml;

// 👎 reported (trusts whatever key is embedded in the document)
bool ok = signedXml.CheckSignature();

// 👍 verify against a key you already trust
bool ok = signedXml.CheckSignature(trustedPublicKey);

// 👍 verify against a certificate you have pinned or chain-validated
bool ok = signedXml.CheckSignature(pinnedCertificate, verifySignatureOnly: false);
```

## Rule description

`SignedXml.CheckSignature()` and `CheckSignature(bool)` verify the signature
against the public key or certificate carried inside the document's own
`KeyInfo` element. That only proves the document was signed by *whoever holds the
embedded key* — not by a party you trust. An attacker can:

1. Take a legitimately signed document.
2. Tamper with its contents.
3. Discard the original signature and re-sign the tampered document with a key
   pair they generated themselves.
4. Replace `KeyInfo` with their own public key or certificate.

The no-key `CheckSignature` then returns `true`, because the signature is
internally consistent with the attacker's embedded key. The signature check has
verified nothing that an attacker could not forge.

The defence is to verify against a key established out of band, so a document
signed with any other key is rejected:

- `CheckSignature(AsymmetricAlgorithm key)` — the expected public key.
- `CheckSignature(X509Certificate2 certificate, bool verifySignatureOnly)` — a
  certificate you have pinned or validated to a trust anchor.

If the signer's key legitimately varies and must be read from the document, use
an overload that returns the signing key or certificate and validate it against a
trust anchor (a pinned key, or a certificate chain terminating in a trusted root)
*before* acting on the `true` result.

### Detection

The rule fires on an invocation whose member name is `CheckSignature` that binds
to a method declared on `System.Security.Cryptography.Xml.SignedXml` whose
parameter list is empty or a single `bool`. The key- and certificate-taking
overloads are never reported, and the `CheckSignatureReturning*` methods carry a
different member name, so they are excluded before any binding.

### Availability detection

`SignedXml` ships in the separate `System.Security.Cryptography.Xml` package on
modern .NET. The rule is gated on that type resolving in the analyzed
compilation (probed once per compilation), so a project that does not reference
the XML-signing package registers no analysis and pays nothing.

## How to fix violations

Pass a key or certificate you trust to `CheckSignature`. Choose the source of
trust deliberately — a pinned public key, or a certificate validated to a root
you control — because that decision is exactly what the no-key overload skips,
which is why no automatic code fix is offered.

```csharp
// before — trusts the embedded KeyInfo
bool ok = signedXml.CheckSignature();

// after — verify against the expected signer's public key
bool ok = signedXml.CheckSignature(trustedPublicKey);
```
