# SES1009 — Passwords must use a slow, salted key-derivation function

| Item | Value |
| --- | --- |
| Category | Cryptography |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A password-like value is hashed with a fast, general-purpose hash — `MD5`, `SHA1`,
`SHA256`, `SHA384`, or `SHA512` — through a one-shot `HashData` or an instance
`ComputeHash`, instead of a password key-derivation function.

```csharp
using System.Security.Cryptography;
using System.Text;

// 👎 reported (fast hash over a password)
byte[] stored = SHA256.HashData(Encoding.UTF8.GetBytes(password));

// 👎 reported (instance ComputeHash over a password)
using var sha = SHA256.Create();
byte[] digest = sha.ComputeHash(passwordBytes);

// 👍 slow, salted password KDF with a work factor
byte[] key = Rfc2898DeriveBytes.Pbkdf2(password, salt, iterations: 600_000, HashAlgorithmName.SHA256, outputLength: 32);
```

## Rule description

General-purpose hashes are engineered to be *fast*. That is the wrong property for
a password: commodity GPUs compute billions of `SHA256` hashes per second, so a
leaked table of password digests falls to an offline dictionary or brute-force
attack in a fraction of the time it took to leak. A salt stops precomputed
rainbow tables and forces the attacker to crack each digest separately, but it
does nothing to slow down the guessing of any single one — a fast hash stays fast.

The defence is a deliberately *slow*, salted password key-derivation function
whose work factor can be tuned upward as hardware improves — `Rfc2898DeriveBytes`
/ `Pbkdf2`, Argon2, scrypt, or bcrypt. Each guess then costs the attacker the same
tunable amount of work it costs you once at sign-in.

This rule is orthogonal to the iteration-count check on a password KDF: that rule
governs the *work factor* of a KDF you are already using, while this one fires
when there is no KDF at all — a raw fast hash standing in for one.

### Detection heuristic

Detection is a curated, high-precision **name-and-API** match, chosen to keep
false positives near zero — the rule never traces values across method calls, so
it fires only when the hashed input is visibly a password:

- **API.** A static one-shot `MD5`/`SHA1`/`SHA256`/`SHA384`/`SHA512`.`HashData(...)`,
  or an instance `.ComputeHash(...)` whose receiver's static type is one of those
  five hash types. `ComputeHash` on an `HMAC`/keyed hash, or on a variable typed as
  the abstract `HashAlgorithm` base, is not this shape and is not reported.
- **Name.** The first argument's identifier or member name — or, when the argument
  is a `System.Text.Encoding.GetBytes(...)` call, the encoded value's name — must
  contain one of `password`, `passwd`, `pwd`, `passphrase`, or `credential`
  (case-insensitive). Hashing arbitrary, non-password data is never flagged.

### Availability detection

The rule resolves the five fast-hash types once per compilation and registers
nothing when none are present, so a target framework that has no such type pays
nothing.

## How to fix violations

Replace the fast hash with a password KDF and persist its salt and work-factor
parameters alongside the derived key so a sign-in can reproduce the derivation:

```csharp
// before
byte[] stored = SHA256.HashData(Encoding.UTF8.GetBytes(password));

// after — a per-password random salt and a high, tunable iteration count
byte[] salt = RandomNumberGenerator.GetBytes(16);
byte[] key = Rfc2898DeriveBytes.Pbkdf2(password, salt, iterations: 600_000, HashAlgorithmName.SHA256, outputLength: 32);
// store salt, iterations, and key together
```

Prefer a memory-hard KDF such as Argon2 where a vetted implementation is
available. There is no automatic code fix because moving off a fast hash is a
storage-format redesign — the salt, work factor, and output length have to be
chosen and persisted — not a mechanical call rewrite.

## Performance

The clean path is a syntax test. The rule registers on invocation expressions,
rejects anything that is not a `.HashData(...)` / `.ComputeHash(...)` member call,
and then reads the first argument's name straight from syntax; only a name that
reads as a password triggers the semantic bind that confirms the algorithm is one
of the fast-hash types. A framework without those types registers nothing at all.
