# SES1102 — Do not accept any server certificate

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Transport |
| Severity | Warning |
| Code fix | None — the intent behind disabling validation cannot be inferred |
| Default | Enabled (where `System.Net.Http.HttpClientHandler` and the member exist) |

## Cause

Code reads `HttpClientHandler.DangerousAcceptAnyServerCertificateValidator` — a
built-in callback that returns `true` for every server certificate. It is almost
always wired into a validation callback:

```csharp
using System.Net.Http;

var handler = new HttpClientHandler();

// 👎 reported (turns off server-certificate validation entirely)
handler.ServerCertificateCustomValidationCallback =
    HttpClientHandler.DangerousAcceptAnyServerCertificateValidator;

var client = new HttpClient(handler);
```

Any reference to the member is reported, including a fully qualified read and a
read into a local, because the member exists only to accept every certificate.

## Rule description

TLS protects a connection only while the client verifies the server's identity:
that the presented certificate chains to a trusted root and matches the host it
connected to. `DangerousAcceptAnyServerCertificateValidator` short-circuits that
check to always succeed, so the client will complete a handshake with **any**
party — including an attacker presenting a self-signed certificate or a
certificate issued for a different host. Once that handshake completes the
attacker sits in the middle of the connection and can read and modify every
request and response. This defeats the entire purpose of using HTTPS.

The rule binds the referenced member to the symbol on
`System.Net.Http.HttpClientHandler`, so an unrelated type that happens to declare
a member of the same name is never flagged. It reports the member-access
reference itself and does not trace where the value is later assigned — a read of
this member has no legitimate purpose, so the reference alone is enough.

## How to fix violations

Remove the accept-any validator and let the platform validate the certificate
chain (the default behaviour when no callback is set):

```csharp
using System.Net.Http;

// default validation — no callback needed
var handler = new HttpClientHandler();
var client = new HttpClient(handler);
```

If a specific self-signed or pinned certificate genuinely must be trusted,
validate *that one certificate* explicitly inside the callback (for example by
comparing its thumbprint or public key) rather than accepting all of them:

```csharp
handler.ServerCertificateCustomValidationCallback = (message, cert, chain, errors) =>
    cert is not null && cert.Thumbprint == ExpectedThumbprint;
```

## Availability detection

The member exists only on target frameworks from .NET Core 2.1 / .NET Standard 2.1
onward. The rule resolves `System.Net.Http.HttpClientHandler` in the analyzed
compilation and confirms the member is present on it (probed once per
compilation); on a framework without the type or the member — `netstandard2.0` or
.NET Framework — it registers nothing, so a project that cannot reference the
member pays nothing and receives no diagnostic it cannot act on.

## Scoping the rule off in test code

A test or diagnostic project that *deliberately* accepts any certificate (for
example, to reach a local server with a self-signed certificate) can turn the rule
off for that project with an `.editorconfig` entry:

```ini
# In the test project's .editorconfig
[*.cs]
dotnet_diagnostic.SES1102.severity = none
```

The analyzer does not try to detect test code itself — there is no reliable local
signal for "this is a test" — so scoping is left to configuration.

## Performance

The no-diagnostic path is syntactic: a member access whose name is not
`DangerousAcceptAnyServerCertificateValidator` is rejected before the semantic
model is touched. Only a name-matching access is bound to confirm it resolves to
the member on `HttpClientHandler`.
