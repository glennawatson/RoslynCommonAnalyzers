# SES1104 — Certificate-chain validation must not be deliberately weakened

**Category:** Transport
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

An `X509Chain` decides whether a certificate can be trusted by building the chain
from the certificate up to a trusted root and checking every link — including
whether any certificate in the chain has been revoked. `X509ChainPolicy` controls
that process, and two of its members can switch the important checks off:

- `RevocationMode` decides whether revocation is checked at all. Setting it to
  `X509RevocationMode.NoCheck` means a certificate that has been **revoked**
  (typically because its private key was stolen) is still accepted.
- `VerificationFlags` lists the chain errors the engine is told to ignore. A value
  that names `AllowUnknownCertificateAuthority` accepts a certificate that does
  **not** chain to a trusted authority — exactly the certificate an attacker forges
  for a man-in-the-middle. `AllFlags` ignores *every* error, which is strictly
  worse.

The rule reports the assigned value when either member is weakened:

```csharp
using System.Security.Cryptography.X509Certificates;

var chain = new X509Chain();

// all reported (SES1104):
chain.ChainPolicy.RevocationMode = X509RevocationMode.NoCheck;                              // revocation off
chain.ChainPolicy.VerificationFlags = X509VerificationFlags.AllowUnknownCertificateAuthority; // untrusted CA accepted
chain.ChainPolicy.VerificationFlags = X509VerificationFlags.AllFlags;                       // every error ignored
chain.ChainPolicy.VerificationFlags =                                                       // suppressing flag in an OR
    X509VerificationFlags.IgnoreEndRevocationUnknown | X509VerificationFlags.AllowUnknownCertificateAuthority;

// object-initializer form is reported too:
var policy = new X509ChainPolicy
{
    RevocationMode = X509RevocationMode.NoCheck,
    VerificationFlags = X509VerificationFlags.AllFlags,
};
```

Leave the checks on. Keep `RevocationMode` at `Online` (or `Offline` where the CRL
is cached) and `VerificationFlags` at `NoFlag`:

```csharp
chain.ChainPolicy.RevocationMode = X509RevocationMode.Online;
chain.ChainPolicy.VerificationFlags = X509VerificationFlags.NoFlag;
```

## What is reported

The assigned value is flagged when it is:

- `RevocationMode` set to `X509RevocationMode.NoCheck`; or
- `VerificationFlags` set to an `X509VerificationFlags` value that names
  `AllowUnknownCertificateAuthority` or `AllFlags`, alone or as one operand of an
  OR-combination (`a | b | …`).

Both the direct member set (`policy.Member = value`) and the object-initializer
set (`Member = value` inside `new X509ChainPolicy { … }`) are covered.

`NoFlag`, `Online`, `Offline`, and narrow flags that only relax a specific
revocation-unknown condition without disabling authority trust (for example
`IgnoreEndRevocationUnknown` on its own) are **not** reported — those do not
suppress genuine chain-of-trust errors. The rule deliberately keeps this set small
to stay high-precision.

Only the value written **at the assignment** is examined. A flags value computed in
another method, or a `RevocationMode` copied through a local first, is not tracked —
that would require whole-program data-flow analysis, which this rule does not do, to
stay fast and free of false positives.

## Why there is no code fix

There is no single safe rewrite. Whether revocation should be `Online` or `Offline`,
and which — if any — errors a particular deployment can legitimately relax, is a
policy decision the author has to make. Silently flipping `NoCheck` to `Online`
could break an air-gapped deployment that has no revocation endpoint, and clearing
`VerificationFlags` could reintroduce an error the author suppressed for an
understood, narrow reason. The rule reports and leaves the correction to the author.

## Proving the API exists

The rule resolves
`System.Security.Cryptography.X509Certificates.X509ChainPolicy` in the analyzed
compilation and registers nothing when it is absent, so a project targeting a
framework without the type receives no diagnostic. The `X509RevocationMode` and
`X509VerificationFlags` enums are the declared types of the two members, so the
matched value's enum is taken straight from the member being set — no separate
probe is needed.

## Performance

The no-diagnostic path is syntactic. An assignment whose target member is not
spelled `RevocationMode` or `VerificationFlags` is rejected before the semantic
model is touched; only a matching shape is bound to confirm the member belongs to
`X509ChainPolicy` and to inspect the assigned value.
