# SES1105 — Bearer and OpenID Connect metadata must not be retrieved over plain HTTP outside development

**Category:** Transport
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

The JWT-bearer and OpenID Connect authentication handlers validate incoming
tokens against the identity provider's metadata: they download the discovery
document and, from it, the JWKS signing keys that decide whether a token's
signature is genuine. `JwtBearerOptions.RequireHttpsMetadata` and
`OpenIdConnectOptions.RequireHttpsMetadata` default to `true` so that download
happens over HTTPS.

Setting `RequireHttpsMetadata = false` lets the handler fetch that metadata over
plain HTTP. Anyone on the network path can then rewrite the discovery document and
substitute signing keys they control; the application will accept every token the
attacker mints against those keys — a full authentication bypass. Disabling the
check is only ever acceptable against a loopback authority during local
development.

The rule reports the `= false` assignment on either option type, written directly
or in an object initializer:

```csharp
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.AspNetCore.Authentication.OpenIdConnect;

// all reported (SES1105):
jwtOptions.RequireHttpsMetadata = false;                       // direct assignment
var opts = new JwtBearerOptions { RequireHttpsMetadata = false };  // object initializer
oidcOptions.RequireHttpsMetadata = false;                      // OpenID Connect options
```

Leave the option at its secure default (or set it explicitly to `true`) in
production and serve the authority over HTTPS:

```csharp
jwtOptions.RequireHttpsMetadata = true;   // the default — metadata is fetched over HTTPS
```

## The development-environment guard

An assignment that is lexically enclosed by an `if` statement or a conditional
expression whose condition calls a method named `IsDevelopment` is treated as a
development-only opt-out and is **not** reported:

```csharp
if (env.IsDevelopment())
{
    jwtOptions.RequireHttpsMetadata = false;   // not reported
}

if (builder.Environment.IsDevelopment())
{
    oidcOptions.RequireHttpsMetadata = false;  // not reported
}
```

This is a purely local, syntactic check of the assignment's enclosing statements:
the guard call must appear in the condition of an ancestor `if`/conditional. No
data-flow or interprocedural analysis is performed, so an environment flag copied
into a local variable first, or a guard hidden behind a helper method, is not
recognized — keep the guard directly around the assignment.

## What is reported

The assignment is flagged when both hold:

- the assigned member is `RequireHttpsMetadata` and its containing type is
  `Microsoft.AspNetCore.Authentication.JwtBearer.JwtBearerOptions` or
  `Microsoft.AspNetCore.Authentication.OpenIdConnect.OpenIdConnectOptions`
  (the symbol is bound, so a same-named property on any other type is ignored);
- the right-hand side is the literal `false`; and
- no enclosing `if`/conditional condition calls `IsDevelopment`.

## Why there is no code fix

There is no safe automatic rewrite. Removing the assignment (or flipping it to
`true`) can break a developer's local loopback setup, and wrapping it in an
`if (env.IsDevelopment())` guard requires an environment accessor the analyzer
cannot conjure or reliably locate. The author has to decide whether the opt-out
belongs behind a development guard or should be dropped entirely, so the rule
reports and leaves the change to them.

## Proving the API exists

The rule resolves `JwtBearerOptions` and `OpenIdConnectOptions` in the analyzed
compilation and registers nothing when neither is present, so a project without
ASP.NET Core authentication receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An assignment whose right-hand side is not the
literal `false`, or whose left-hand side is not spelled `RequireHttpsMetadata`, is
rejected before the semantic model is touched; only a matching shape is bound to
confirm the option type and check for an enclosing development guard.
