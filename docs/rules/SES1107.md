# SES1107 — Do not weaken SQL connection transport security

**Category:** Transport
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A SQL connection reaches the database server over the network. Two connection
settings quietly remove the protection that keeps that traffic private and
trustworthy:

- **`TrustServerCertificate=true`** makes the client accept whatever certificate
  the server presents without validating its chain or name. An attacker who can
  sit on the network path presents a forged certificate, the client trusts it, and
  every query and result — credentials included — flows through the attacker. This
  is a classic man-in-the-middle attack.
- **`Encrypt=false`** (or **`Encrypt=Optional`** on the newer client) turns TLS off
  or makes it merely negotiable, so the connection can travel in cleartext and be
  read or altered in transit.

Keep the server certificate validated and encryption on. Install a certificate the
client already trusts on the server instead of switching validation off.

The rule reports these settings in **local, compile-time** shapes for
`Microsoft.Data.SqlClient` and the legacy `System.Data.SqlClient`.

## What is reported

### A literal connection string

A string **literal** carrying one of the weakening keyword settings, when it is
passed to a `SqlConnection` or `SqlConnectionStringBuilder` constructor or assigned
to a `ConnectionString` property:

```csharp
using Microsoft.Data.SqlClient;

// all reported (SES1107):
var a = new SqlConnection("Server=db;Database=app;TrustServerCertificate=true");
var b = new SqlConnection("Server=db;Encrypt=false");
var c = new SqlConnectionStringBuilder("Server=db;Encrypt=Optional");

var d = new SqlConnection();
d.ConnectionString = "Server=db;TrustServerCertificate=true";
```

Keywords and their values are matched case-insensitively and tolerate surrounding
whitespace (`trustservercertificate = TRUE`). The weakening values are
`TrustServerCertificate` = `true`/`yes`, and `Encrypt` = `false`/`no`/`optional`.

### A builder member

A `SqlConnectionStringBuilder` object-initializer member or property assignment
that sets one of the weakening values:

```csharp
using Microsoft.Data.SqlClient;

// all reported (SES1107):
var e = new SqlConnectionStringBuilder
{
    TrustServerCertificate = true,
    Encrypt = false,
};

var f = new SqlConnectionStringBuilder();
f.TrustServerCertificate = true;
f.Encrypt = SqlConnectionEncryptOption.Optional;
```

`Encrypt = false` is reported on both clients; `Encrypt = SqlConnectionEncryptOption.Optional`
is reported on the modern client, which models `Encrypt` as an option rather than a
`bool`.

## What is not reported

A secure configuration is left alone — encryption on and the certificate validated:

```csharp
var ok = new SqlConnection("Server=db;Encrypt=true;TrustServerCertificate=false");

var okBuilder = new SqlConnectionStringBuilder
{
    Encrypt = SqlConnectionEncryptOption.Strict,   // Strict / Mandatory keep TLS required
};
```

Only a literal supplied at the call, and a constant member value, are examined. A
connection string held in a variable, built by interpolation or concatenation, or
read from configuration is not tracked — that would require whole-program data-flow
analysis, which this rule deliberately avoids to stay fast and free of false
positives. A non-constant member value is also left alone, so a deliberate
environment switch is never second-guessed:

```csharp
string value = "Server=db;TrustServerCertificate=true";
var skipped = new SqlConnection(value);             // not reported — flows through a variable

builder.TrustServerCertificate = isDevelopment;     // not reported — not a constant
```

## Why there is no code fix

There is no safe automatic rewrite. Removing `TrustServerCertificate=true` or
turning encryption back on only works once the server presents a certificate the
client trusts and serves TLS on that endpoint; when it does not, the connection
fails at runtime and the change silently breaks the caller. Whether the server is
ready for validated, encrypted connections is knowledge the analyzer does not have,
so the rule reports and leaves the decision to the author.

## Proving the API exists

The rule resolves `Microsoft.Data.SqlClient.SqlConnection` and
`System.Data.SqlClient.SqlConnection` in the analyzed compilation and registers
nothing unless at least one is present, so a project without a SQL client receives
no diagnostic. The builder and encryption-option types are resolved the same way,
and the builder members are confirmed by binding the instance type, so a same-named
member on an unrelated type is never reported.

## Performance

The no-diagnostic path is syntactic. A connection-string literal is rejected before
the semantic model is touched unless a cheap case-insensitive scan finds a weakening
keyword setting, and an assignment whose target is not `ConnectionString`,
`TrustServerCertificate`, or `Encrypt` is dropped by name first. The compilation is
bound only after a weakening shape has already been found.
