# SES1108 — Do not accept every server certificate from a custom callback

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Transport |
| Severity | Warning |
| Code fix | None — the intent behind disabling validation cannot be inferred |
| Default | Enabled (where `System.Net.Http.HttpClientHandler` and the property exist) |

## Cause

`HttpClientHandler.ServerCertificateCustomValidationCallback` is assigned a callback whose body
unconditionally returns `true`, so it treats every server certificate as valid:

```csharp
using System.Net.Http;

var handler = new HttpClientHandler();

// 👎 reported (accepts any certificate, including a self-signed or wrong-host one)
handler.ServerCertificateCustomValidationCallback = (message, cert, chain, errors) => true;

var client = new HttpClient(handler);
```

The following shapes are all reported: an expression lambda `(...) => true`, a block lambda or
anonymous method whose only result is `return true;`, and a method group to a source method of that
shape.

## Rule description

TLS protects a connection only while the client verifies that the server's certificate chains to a
trusted root and matches the host it connected to. A validation callback that always returns `true`
replaces that check with unconditional acceptance: the client completes a handshake with any server
presenting a self-signed, expired, or wrong-host certificate, and a network attacker positioned between
the client and the server can then read and alter every request and response.

The callback body is inspected only locally — its expression body, or the `return` statements it owns —
so a callback that actually examines the certificate, chain, or `SslPolicyErrors` is never reported.
The rule binds the assignment target, so a same-named property on an unrelated type is not flagged, and
it is gated on `System.Net.Http.HttpClientHandler` and the property resolving, so a project on a target
framework without them pays nothing.

The built-in `HttpClientHandler.DangerousAcceptAnyServerCertificateValidator`, which is a property rather
than a written callback, is reported by [SES1102](SES1102.md) instead, so the two rules do not overlap.

## What is not reported

- A callback that inspects the certificate or the errors — for example
  `(message, cert, chain, errors) => errors == SslPolicyErrors.None`.
- A method group whose referenced source method actually validates, or one whose body cannot be
  inspected locally (it has no single source declaration).
- The built-in accept-any sentinel (see [SES1102](SES1102.md)).

## How to fix violations

Remove the callback and let the platform validate the certificate chain. If a specific self-signed or
pinned certificate must be trusted, compare that one certificate explicitly — for example by thumbprint —
and return `false` for everything else:

```csharp
handler.ServerCertificateCustomValidationCallback = (message, cert, chain, errors) =>
    cert is not null && cert.Thumbprint == PinnedThumbprint;
```
