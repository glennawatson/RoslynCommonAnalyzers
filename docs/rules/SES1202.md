# SES1202 — Do not hard-code a credential value

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Secrets |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A non-empty string literal is written directly into source in a position that
expects a credential — either an argument bound to a credential-named parameter,
or the secret position of a known credential-type constructor.

```csharp
// 👎 reported — literal bound to a credential-named parameter
client.Connect(apiKey: "k3y-9f2a-not-a-recognized-pattern");

// 👎 reported — secret position of a credential-type constructor
var cred = new System.Net.NetworkCredential("svc-account", "literal-password-value");
var key = new Azure.AzureKeyCredential("literal-azure-key-value");

// 👍 read the secret from configuration / environment / a secret store
client.Connect(apiKey: Environment.GetEnvironmentVariable("API_KEY"));
var cred2 = new System.Net.NetworkCredential(user, password);
```

## Rule description

A secret written into source is compiled into the assembly, committed to version
control, and exposed to anyone who can read the repository or decompile the
binary. It cannot be rotated without a rebuild, and a leak of the source is a
leak of the credential.

This rule reports a **string literal** in a credential position in two shapes:

1. **Credential-named parameter (no type gate).** The argument is bound — by
   position or by name — to a parameter whose name matches a curated,
   high-precision set (case-insensitive), including underscore spellings:
   `apiKey`/`api_key`, `password`/`passwd`/`pwd`, `secret`/`clientSecret`,
   `token`/`accessToken`/`sasToken`, `connectionString`,
   `accessKey`/`accountKey`, `privateKey`.

2. **Credential-type constructor (gated on the type resolving).** The argument is
   the secret position of `System.Net.NetworkCredential`,
   `Azure.AzureKeyCredential` (`Azure.Core.AzureKeyCredential`),
   `Azure.Identity.ClientSecretCredential`, or
   `System.ClientModel.ApiKeyCredential`. A generic `key` parameter is treated as
   a secret **only** inside these known types, so an ordinary `key` parameter
   elsewhere (for example a dictionary insert) is never reported. On a target that
   does not reference these types, the constructor shape resolves to nothing and
   costs nothing.

It complements the pattern-based secret rule (SES1201): where that rule matches
the *content* of a literal against known secret shapes, this rule catches a
hard-coded credential by its *position*, even when the text is not a recognizable
pattern.

### What is not reported

- **Any non-literal.** A variable, a constant reference, a configuration lookup,
  or an environment read is the correct way to supply a secret and is never
  reported.
- **An empty string** (`""`) — an empty credential is a separate concern.
- **Obvious placeholders** — a value beginning with `your`, an angle-bracket
  template such as `<your-key>`, an all-same-character mask such as `xxxxxxxx`,
  and a small curated set of words (`changeme`, `placeholder`, `example`,
  `sample`, `dummy`, `test`, and similar).

### Availability detection

The parameter-name shape needs no framework gate; it applies to any method or
constructor. The constructor shape probes each credential type once per
compilation via `Compilation.GetTypeByMetadataName` and stays silent for any type
the analyzed project does not reference — no target-framework string is parsed.

No code fix is offered: replacing a hard-coded secret requires choosing where the
value should come from (configuration, an environment variable, or a secret
manager), which the analyzer cannot decide for you.
