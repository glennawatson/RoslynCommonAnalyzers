# SES1203 — Do not use an empty or missing database connection-string password

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Secrets |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A string literal is a database connection string that logs in with a user name but supplies an empty
password, or no password at all.

```csharp
public sealed class Repository
{
    // 👎 reported — a user is named but the password is blank
    private const string Connection = "Server=db;Database=app;User Id=sa;Password=;"; // SES1203

    // 👎 reported — a user is named and there is no password key at all
    private const string Missing = "Server=db;Database=app;User Id=sa"; // SES1203
}
```

## Rule description

A connection string that authenticates with a user name but no password has zero credential strength:
anyone who can reach the server logs in as that account. The rule reports a **string literal** that is
connection-string shaped and whose password is blank.

A literal is treated as a connection string when it carries **both**:

- a data-source key — `Server`, `Data Source`, `Host`, `Initial Catalog`, or `Database`; and
- a user key — `User ID` / `User Id` or `Uid`.

It is reported when the password is blank, meaning **either**:

- a `Password` or `Pwd` key whose value is empty (`Password=;`, `Password=` at the end, or only
  whitespace); **or**
- no `Password` / `Pwd` key at all.

Keys are parsed case-insensitively by splitting on `;` and the first `=` of each segment, and whitespace
around keys and values is ignored.

### What is not reported

- **A non-empty password.** A connection string that carries a real password value is not a blank-password
  defect and is left alone here — a hard-coded password value is a separate concern.
- **Integrated or trusted authentication.** `Integrated Security=true` / `Integrated Security=SSPI` and
  `Trusted_Connection=true` / `Trusted_Connection=yes` need no password, so a literal that enables either is
  not reported (an explicit `Integrated Security=false` does not exempt the literal).
- **No named user.** A connection string without a `User ID` / `Uid` key is not reported, because there is no
  user-name credential whose strength is at stake.
- **Any non-literal or non-connection-string text.** A variable, a configuration lookup, or ordinary prose is
  never reported.

### Availability detection

Detection is purely syntactic — string literals exist on every target framework, so the rule needs no
semantic model, no framework probing, and no configuration, and it reports on every target framework.

### Known limitation

Only a **string literal** is inspected. The same blank-password shape assembled through a connection-string
builder object (setting an empty `Password` property while a user is set) is not reported, because that
requires cross-statement flow that the syntactic rule deliberately avoids to stay allocation-free and high
precision.

## How to fix violations

Supply a strong password read at run time from a location that is not committed, or switch to integrated
authentication so no password travels in the connection string:

```csharp
public sealed class Repository
{
    private readonly string _connection;

    public Repository(IConfiguration configuration)
        => _connection = configuration.GetConnectionString("App")
            ?? throw new InvalidOperationException("Connection string 'App' is not configured.");
}
```

No code fix is offered: there is no password value for the analyzer to insert, and the choice between a
managed secret and integrated authentication is yours to make.

## Performance

The clean path is a length screen plus two character searches (`=` and `;`); an ordinary literal is rejected
with no allocation. Only a connection-string-shaped literal reaches the single index-based scan that splits it
into `key=value` segments, and that scan allocates nothing — no `Split`, no substrings.
