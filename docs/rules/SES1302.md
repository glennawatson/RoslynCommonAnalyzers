# SES1302 — A shell-executed process must not use a non-constant FileName

**Category:** Injection
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

`System.Diagnostics.ProcessStartInfo.UseShellExecute` decides *who* resolves the
program named in `FileName`. When it is `false`, the process loader launches the
executable directly. When it is `true`, the operating-system **shell** resolves and
parses `FileName` instead: it applies `PATH` lookup, expands the value, and on
Windows can pick a registered file-type handler or open a document or URL.

Handing the shell a filename the program did not fix in source lets an attacker
steer *which* program runs — a value that reaches `FileName` from a request, a
config field, or any other data can select a different executable on the `PATH`,
open an unexpected handler, or smuggle in extra shell behaviour. That is a
command-injection and unexpected-program risk. The rule reports the non-constant
filename in the one shape it can judge locally:

```csharp
using System.Diagnostics;

// reported (SES1302): shell execution + a filename the program did not fix
var byInitializer = new ProcessStartInfo
{
    FileName = fileName,            // non-constant
    UseShellExecute = true
};

var byConstructor = new ProcessStartInfo(fileName)   // non-constant
{
    UseShellExecute = true
};
```

The fix is to stop shell-resolving a data-derived program. Either turn the shell
off so the loader launches a fixed executable directly and pass user data as
arguments:

```csharp
var psi = new ProcessStartInfo
{
    FileName = "/usr/bin/git",     // a fixed, validated program path
    UseShellExecute = false
};
psi.ArgumentList.Add(userSuppliedValue);   // data goes through arguments, never the filename
```

or, when shell execution is genuinely required, restrict `FileName` to a constant,
validated program path.

## What is reported

Within a **single** `new ProcessStartInfo { ... }` object creation, the filename is
flagged when both hold:

- an initializer assignment `UseShellExecute = true` is present, with `true`
  written as the literal (a non-literal such as a variable or `const` reference is
  not treated as enabling the shell); and
- the effective `FileName` is not a compile-time constant. The filename is read
  from an initializer `FileName = ...` assignment when present (it is the last
  write and wins over the constructor), otherwise from the constructor's `fileName`
  argument.

A constant filename — a string literal, or any expression `GetConstantValue`
resolves — is never reported, whether it comes from the initializer or the
constructor. Shell execution with no filename set locally is not reported either.
The `ProcessStartInfo` type and its `UseShellExecute` / `FileName` members are bound
by symbol, so an unrelated type that merely shares the name is not flagged.

## Local by design

Detection is confined to the one object-creation expression. A `UseShellExecute`
or `FileName` set on a separate statement (`psi.FileName = data;`), or a filename
that flows in through a local or another method, is not tracked — that would
require whole-program data-flow analysis, which this rule deliberately does not do,
to keep it fast and free of false positives. It reports the precise, self-contained
shape and leaves flow-sensitive cases to review.

## Why there is no code fix

There is no single safe rewrite. Setting `UseShellExecute = false` changes launch
semantics (no shell resolution, no handler/document/URL support, different
redirection defaults) and may not preserve the author's intent; pinning `FileName`
to a constant requires knowing the intended program. Both are decisions for the
author, so the rule reports and leaves the change to them.

## Proving the API exists

The rule resolves `System.Diagnostics.ProcessStartInfo` in the analyzed compilation
and registers nothing when it is absent, so a project targeting a framework without
the type receives no diagnostic. `UseShellExecute` and `FileName` are members of
that same type, so no separate probe is needed.

## Performance

The no-diagnostic path is syntactic. An object creation with no object initializer,
or whose initializer does not set `UseShellExecute` to the literal `true`, is
rejected before the semantic model is touched; only a matching shape is bound to
confirm the type and locate the filename.
