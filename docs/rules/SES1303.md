# SES1303 — Regular-expression pattern must not be built from non-constant data

**Category:** Injection
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A regular expression has two inputs: the text being searched and the *pattern*
that decides what a match is. When the pattern comes from non-constant data,
whoever supplies that data controls the matching grammar, not just the text. That
is regex injection, and it bites at runtime in three ways:

- **Grammar change** — metacharacters in the data (`.`, `*`, `+`, `|`, `(`, `[`,
  `^`, `$`) turn a pattern meant to match one exact value into one that matches far
  more. An allow-list check built as `new Regex(userValue)` can be widened to match
  anything.
- **Catastrophic backtracking** — a data-supplied pattern with nested quantifiers
  (`(a+)+$` and friends) can make a single match take exponential time, hanging the
  thread and denying service.
- **Capture rewriting** — injected groups shift the capture numbers the surrounding
  code reads back, so `$1`/`Groups[1]` no longer mean what the author intended.

The rule reports the **pattern** argument when it is not a compile-time constant:

```csharp
using System.Text.RegularExpressions;

// all reported (SES1303):
var r1 = new Regex(userPattern);                     // constructor
var r2 = new Regex("^" + data + "$");                // concatenated
var r3 = new Regex($"^{data}$", RegexOptions.IgnoreCase); // interpolated
bool m = Regex.IsMatch(input, userPattern);          // static overload, pattern is the 2nd arg
string s = Regex.Replace(input, userPattern, "X");   // static overload
```

Treat the untrusted text as literal data. Escape it and compose it into a fixed
template, so its characters can only match themselves:

```csharp
var safe = new Regex("^" + Regex.Escape(data) + "$");
bool m2 = Regex.IsMatch(input, "^" + Regex.Escape(userValue) + "$");
```

If you only need an exact or substring comparison, drop the regex entirely and use
`string.Equals`, `StartsWith`, `EndsWith`, or `Contains` with an explicit
`StringComparison`.

## What is reported

The pattern argument is flagged only in these local, high-precision shapes:

- the first argument of `new System.Text.RegularExpressions.Regex(pattern, ...)`;
- the `pattern` argument (the second positional argument) of the static
  `Regex.IsMatch`, `Regex.Match`, `Regex.Matches`, `Regex.Replace`, and
  `Regex.Split(input, pattern, ...)` overloads.

An argument passed by name as `pattern:` is matched wherever it appears. The
constructor or method is **bound** to confirm its type is the real
`System.Text.RegularExpressions.Regex`; a same-named type or method is never
reported. "Non-constant" means the compiler cannot fold the argument to a constant
string — a parameter, local, field, method result, interpolation, or concatenation
that includes any of those. A fully constant expression (a literal, a `const`, or a
concatenation of constants) is clean.

## What is not reported

- **A constant pattern over non-constant input.** Only the pattern argument is
  examined. `Regex.IsMatch(userInput, "^[a-z]+$")` is not flagged here — searching
  attacker-controlled *input* with a fixed pattern is a matching-timeout (ReDoS)
  concern owned by a separate rule, not a pattern-injection defect.
- **Instance calls.** `regex.IsMatch(userInput)` carries no pattern argument; the
  pattern was fixed when the `Regex` was constructed and is judged at that call.
- **Data flow.** A pattern that reaches the call through a variable assigned earlier
  is still reported at the call only when the expression *at the call* is
  non-constant; the rule does not trace values across statements or methods. This
  keeps it fast and free of false positives.

## Why there is no code fix

There is no safe automatic rewrite. Wrapping the data in `Regex.Escape` is the right
remedy when the intent is to match the text literally, but that **changes the
matching semantics** — a pattern the author genuinely meant to be dynamic (built
from a trusted, validated source) would be broken by escaping it. Whether the data
is meant as a pattern or as literal text is knowledge the analyzer does not have, so
the rule reports and leaves the decision to the author.

## Proving the API exists

The rule resolves `System.Text.RegularExpressions.Regex` in the analyzed
compilation and registers nothing when it is absent, so a project without the type
receives no diagnostic. `Regex.Escape`, the suggested remedy, is a static member of
that same type, so its availability follows from the same probe.

## Performance

The no-diagnostic path is syntactic. An object creation whose type name is not
`Regex`, and an invocation whose member name is not one of the five guarded static
methods (or that carries fewer than the input-plus-pattern arguments), are rejected
before the semantic model is touched. Binding and the constant-value check run only
after those cheap filters pass, and the message string is built only once a
violation has been confirmed.
