# SES1304 — Do not build a write path from an archive entry name without a containment check

**Category:** Injection
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

An archive (a `.zip` or a `.tar`) carries a name for each entry, and that name is
attacker-controlled: whoever built the archive chose it. When extraction code joins
that name onto a destination directory and writes there without checking that the
result stays *inside* the directory, a crafted entry escapes it. An entry named
`../../etc/cron.d/task`, or an absolute path like `/home/app/.ssh/authorized_keys`,
is written wherever it points — outside the folder you intended, overwriting
arbitrary files. This is the "zip slip" path-traversal class.

```csharp
using System.IO;
using System.IO.Compression;

foreach (ZipArchiveEntry entry in archive.Entries)
{
    // reported (SES1304): entry.FullName reaches File.WriteAllBytes with no containment check
    File.WriteAllBytes(Path.Combine(destDir, entry.FullName), entry.Open().ReadAllBytes());
}
```

Resolve the combined path and confirm it stays under the destination root before
writing:

```csharp
string destRoot = Path.GetFullPath(destDir);
string target = Path.GetFullPath(Path.Combine(destRoot, entry.FullName));

// require the resolved target to sit under the root, separator-terminated so
// "…/data" cannot be passed off as a prefix of "…/data-evil".
string prefix = destRoot.EndsWith(Path.DirectorySeparatorChar)
    ? destRoot
    : destRoot + Path.DirectorySeparatorChar;

if (!target.StartsWith(prefix, StringComparison.Ordinal))
{
    throw new IOException($"Entry '{entry.FullName}' escapes the destination directory.");
}

File.WriteAllBytes(target, data);
```

## What is reported

The rule reports one tight, local shape: an archive entry name that is joined into a
path **and consumed by a write sink in the same expression**, so there is no room
for a guard between the join and the write. Concretely, all of these must hold:

- the joined value is an archive entry name — a `FullName` on a
  `System.IO.Compression.ZipArchiveEntry`, or a `Name` on a
  `System.Formats.Tar.TarEntry` (the member is **bound** to the archive type, not
  matched by name);
- that entry name is a **direct** argument of `System.IO.Path.Combine(...)`, or a
  **direct** operand of a string `+` concatenation;
- that `Path.Combine`/concatenation is **itself** the destination argument of a
  write sink:
  - `entry.ExtractToFile(<combined>, …)`,
  - `File.Create`, `File.OpenWrite`, `File.WriteAllBytes`, or `File.WriteAllText`
    with `<combined>` as the path,
  - `new FileStream(<combined>, …)`.

The reported span is the `Path.Combine`/concatenation expression. Both archive-entry
types are probed independently: a project that references only one of them still
gets that surface, and a project that references neither is never analysed.

The zip `ExtractToFile`, `FileStream`, and `File.OpenWrite` sinks are **already**
reported by the .NET SDK's built-in archive path-traversal analysis, which
recognises a zip entry as a tainted source but does not recognise
`System.Formats.Tar.TarEntry`. To avoid emitting a second diagnostic on the same
line, this rule reports those three sinks only for **Tar** entries, and reports the
remaining sinks (`File.Create`, `File.WriteAllBytes`, `File.WriteAllText`) for both
zip and Tar entries.

## What is not reported

- **The multi-statement form.** When the entry name is copied into a local (or a
  field) and the write happens in a later statement, deciding whether a containment
  guard sits in between requires data-flow analysis, which this rule deliberately
  avoids. Only the inline, single-expression shape — where a guard is structurally
  impossible — is reported.

  ```csharp
  string name = entry.FullName;
  File.WriteAllBytes(Path.Combine(destDir, name), data); // not reported — flows through a local
  ```

- **A wrapped entry name.** If the entry name is passed through a call before the
  join (`Path.Combine(destDir, Sanitize(entry.FullName))`), it is no longer a direct
  argument and is not reported — the wrapper may be the very containment check.

- **A look-alike member.** A `Name`/`FullName` on any other type (for example
  `DirectoryInfo.Name`), or a `Combine`/`FileStream`/write method on a type other
  than `System.IO.Path`/`System.IO.FileStream`/`System.IO.File`, is bound and
  rejected — a name match alone is never trusted.

## Why there is no code fix

A correct fix has to introduce the destination root, resolve the combined path with
`Path.GetFullPath`, choose a rejection strategy (throw, skip, clamp), and thread the
guarded path back to the sink — decisions that depend on how the surrounding code is
structured and what it should do with a hostile entry. There is no single rewrite
that is safe in every case, so the rule reports and leaves the fix to the author.

## Proving the API exists

The rule resolves `System.IO.Compression.ZipArchiveEntry` and
`System.Formats.Tar.TarEntry` in the analyzed compilation and registers nothing when
neither is present, so a project targeting a framework without them (for example an
older `.NET Framework` target that predates these types) receives no diagnostic.
`System.Formats.Tar` is only available on .NET 7 and later, so the Tar surface simply
does not appear on frameworks that lack it.

## Performance

The no-diagnostic path is syntactic. An invocation whose member name is not a write
sink, and an object creation whose type is not `FileStream`, are rejected before the
semantic model is touched; the destination argument is then checked for an inline
`Path.Combine`/concatenation carrying a `.FullName`/`.Name` member access. Only once
that cheap shape matches are the entry member, the join, and the sink bound.
