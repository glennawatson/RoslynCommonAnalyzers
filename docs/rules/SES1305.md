# SES1305 — Do not build a storage path from an uploaded file name

**Category:** Injection
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

The `FileName` of an uploaded file (`Microsoft.AspNetCore.Http.IFormFile`) is set
entirely by the client. A request can send `..\..\appsettings.json`, a rooted
path such as `/etc/passwd`, or `C:\Windows\System32\...`. Using that value to
build a filesystem path lets the request climb out of the intended upload
directory and read or overwrite arbitrary files — a classic path-traversal
vulnerability.

The rule reports the `.FileName` read when it flows straight into a path-building
sink:

```csharp
using System.IO;
using Microsoft.AspNetCore.Http;

// all reported (SES1305):
Path.Combine(root, file.FileName);          // Path.Combine argument
var full = root + "/uploads/" + file.FileName;  // '+' concatenation forming a path
File.WriteAllBytes(file.FileName, bytes);   // file-creating call
File.Create(file.FileName);
File.OpenWrite(file.FileName);
File.Copy(source, file.FileName);
new FileStream(file.FileName, FileMode.Create);
```

Strip the directory portion with `Path.GetFileName` before combining, or — better
— store the upload under a server-generated name and keep the original only as
metadata:

```csharp
// safe: the directory portion is removed before the path is built
var safe = Path.Combine(root, Path.GetFileName(file.FileName));   // not reported

// safer: a server-generated name that the client cannot influence at all
var stored = Path.Combine(root, Guid.NewGuid().ToString("N"));
```

Note that even after `Path.GetFileName`, an upload directory should be validated
and access-controlled; sanitizing the name defeats traversal but is not a
substitute for storing uploads outside the web root.

## What is reported

The `.FileName` expression is flagged when both hold:

- it binds to a property named `FileName` whose containing type is
  `Microsoft.AspNetCore.Http.IFormFile` (the symbol is bound, so a same-named
  `FileName` on any other type is ignored); and
- it is used directly in one of these sinks:
  - a direct argument to `System.IO.Path.Combine(...)`;
  - an operand of a `+` string concatenation whose chain also contains a string
    literal with a `/` or `\` path separator (so a non-path concatenation such as
    a log or display string is not flagged); or
  - a direct argument to a file-creating call —
    `System.IO.File.Create`/`OpenWrite`/`WriteAllBytes`/`Copy`, or
    `new System.IO.FileStream(...)`.

Because the `.FileName` read must be the **direct** argument, wrapping it in
`Path.GetFileName(file.FileName)` moves it out of the sink's argument position and
the call is not reported.

## Local shape only

This is a purely local, syntactic match against the sink expression; there is no
data-flow or interprocedural analysis. A filename copied into a local variable and
then combined a few lines later, or passed through a helper method, is not
tracked. The rule targets the direct, common shape where an upload name is fed
straight into a path — keep the `Path.GetFileName` call (or the server-generated
name) at that call site.

The concatenation heuristic requires a path-separator literal to be present in the
`+` chain, which also matches URL-building concatenations that use `/`; treat such
a report as a prompt to confirm the value is not used to address a resource by an
attacker-chosen segment.

## Why there is no code fix

There is no single safe rewrite. `Path.GetFileName` removes traversal but keeps a
client-chosen name (which can still collide or leak information), while a
server-generated name changes the stored identity and usually requires storing the
original name elsewhere. The author has to choose the storage strategy, so the
rule reports and leaves the change to them.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Http.IFormFile` in the analyzed
compilation and registers nothing when it is absent, so a project without ASP.NET
Core receives no diagnostic and pays no analysis cost. The suggested
`Path.GetFileName` is available on every framework that also has `IFormFile`.

## Performance

The no-diagnostic path is syntactic. A member access whose name is not `FileName`
is rejected immediately; a `.FileName` that is not in one of the sink shapes is
rejected before the semantic model is touched. Only a matching shape is bound to
confirm the `IFormFile` container and the sink member.
