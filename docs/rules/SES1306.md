# SES1306 — Do not compile or execute non-constant C# via the scripting API

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Injection |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A non-constant string is passed as the C# source to a static scripting entry point —
`CSharpScript.EvaluateAsync`, `CSharpScript.RunAsync`, or `CSharpScript.Create` on
`Microsoft.CodeAnalysis.CSharp.Scripting.CSharpScript`. The value is compiled and run
in the host process, so any runtime data folded into it becomes executable code.

```csharp
using Microsoft.CodeAnalysis.CSharp.Scripting;

public async Task Evaluate(string userInput)
{
    // 👎 reported — runtime data compiled and executed
    await CSharpScript.EvaluateAsync<int>(userInput);

    // 👎 reported — a variable folded into the source string
    await CSharpScript.RunAsync("Console.WriteLine(" + userInput + ");");

    // 👍 a constant, author-written template is trusted and is not reported
    await CSharpScript.EvaluateAsync<int>("1 + 1");
}
```

## Rule description

The scripting API compiles the string it is given and executes it with the host's full
privileges. When that string is a compile-time constant, it is a template the author
wrote and can be trusted. When any part of it comes from runtime data — a request field,
a file, a database value, or a concatenation or interpolation that folds in a variable —
whoever controls that data controls the code that runs. That is arbitrary code execution:
the attacker's input is not treated as data but recompiled and run as a program.

The rule reports the `code` argument of the gated methods when its value is **not** a
compile-time constant:

- a variable or parameter reference;
- a concatenation or interpolation that includes a non-constant part;
- a method result or any other runtime expression.

### What is not reported

- **A constant code channel.** A string literal, a `const` reference, or an expression
  the compiler folds to a constant (such as `"1 + " + "1"`) is a trusted template and is
  never reported.
- **The non-string overloads.** The `Stream`-based `Create` overload is not a source
  string and is outside this rule.
- **A same-named method on another type.** Only methods whose containing type is the
  gated `CSharpScript` class are considered; an unrelated `EvaluateAsync`/`Create`
  elsewhere is ignored.

Detection is local: the constant-versus-runtime decision is made at the call site from
the argument's own constant value. The rule performs no data-flow or interprocedural
tracking, so a runtime value first stored in a local and later passed is judged by the
argument expression at the call.

### How to fix violations

Keep the code channel constant. If the behaviour must vary at runtime, drive it through a
fixed set of vetted operations selected by the data — a lookup, a switch, or a small
interpreter over trusted building blocks — rather than compiling the data as source. Never
build the script from untrusted input.

### Availability detection

The rule probes `Microsoft.CodeAnalysis.CSharp.Scripting.CSharpScript` once per compilation
via `Compilation.GetTypeByMetadataName` and registers nothing when the analyzed project does
not reference the C# scripting package, so a project that cannot call these APIs pays nothing
and never receives a diagnostic it cannot act on. No target-framework string is parsed.

## Performance

The clean path is a syntactic screen: an invocation is ignored unless it is a member call
whose method name is `EvaluateAsync`, `RunAsync`, or `Create` and carries an argument. Only
after that screen passes does the analyzer bind the call, confirm the containing type, and
read the `code` argument's constant value — so an ordinary invocation is rejected with no
allocation and no semantic query.
