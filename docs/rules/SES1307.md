# SES1307 — Do not create predictable temporary files with Path.GetTempFileName

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Injection |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

The static `System.IO.Path.GetTempFileName()` method is called.

```csharp
using System.IO;

public string Save(byte[] payload)
{
    // 👎 reported — predictable name in a shared, world-readable directory
    string path = Path.GetTempFileName();
    File.WriteAllBytes(path, payload);
    return path;
}
```

## Rule description

`Path.GetTempFileName()` creates a zero-byte file with a guessable name — `tmpXXXX.tmp` —
in the process-wide temp directory, which on most systems is readable and writable by
every local user. Two weaknesses follow from that (CWE-377):

- **A time-of-check/time-of-use race.** Between the moment the file is created and the
  moment your code opens it, another local user can delete it and substitute their own
  file or a symlink pointing elsewhere. Your later write then lands on an attacker-chosen
  target, or reads back attacker-controlled data — information disclosure or tampering.
- **A hard 65535-file ceiling.** The API draws from only 65535 candidate names and throws
  an `IOException` once that many undeleted `tmp*.tmp` files exist in the directory,
  turning a slow leak of temp files into a denial of service.

The rule binds the invoked symbol and reports only when the call resolves to the
parameterless `GetTempFileName` whose containing type is `System.IO.Path`. The call is
recognised whether written as a member access (`Path.GetTempFileName()`), fully qualified
(`System.IO.Path.GetTempFileName()`), or as a bare identifier under
`using static System.IO.Path`.

### What is not reported

- **A same-named method on another type.** Only a method bound to `System.IO.Path` is
  reported; an unrelated static or instance `GetTempFileName` elsewhere is ignored.
- **Other `Path` members.** `Path.GetRandomFileName()`, `Path.GetTempPath()`, and the
  rest of `Path` are outside this rule.

Detection is local to the call site. The rule performs no data-flow or interprocedural
tracking.

### How to fix violations

There is no automatic code fix, because `Path.GetTempFileName()` also *creates* the file —
replacing it is not a mechanical swap. Choose the option that matches how you use the file:

- **An unpredictable name.** `Path.GetRandomFileName()` returns a cryptographically random
  file name (available on every target framework). Combine it with `Path.GetTempPath()` and
  open the file with a mode that fails if it already exists, for example:

  ```csharp
  string path = Path.Combine(Path.GetTempPath(), Path.GetRandomFileName());
  using var stream = new FileStream(path, FileMode.CreateNew, FileAccess.ReadWrite);
  ```

- **An isolated directory.** On .NET 7 and later, `Directory.CreateTempSubdirectory()`
  creates a uniquely named directory that only the current user can access, giving every
  file you place in it a private, race-free home:

  ```csharp
  DirectoryInfo dir = Directory.CreateTempSubdirectory();
  string path = Path.Combine(dir.FullName, "payload.bin");
  ```

Always delete the file or directory when you are done with it.

### Availability detection

The rule always runs — `System.IO.Path` is present on every target framework — but the
suggested replacement adapts to the compilation. `Directory.CreateTempSubdirectory` is
named in the diagnostic message only when it resolves in the analyzed compilation
(.NET 7+); otherwise the message suggests the always-available `Path.GetRandomFileName()`.
No target-framework string is parsed.

## Performance

The clean path is a syntactic screen: an invocation is ignored unless its member or
identifier name is `GetTempFileName` and it carries no arguments. Only after that screen
passes does the analyzer bind the call and confirm the containing type is `System.IO.Path`,
so an ordinary invocation is rejected with no semantic query. The replacement suggestion is
computed once per compilation, not per call.
