# SES1308 — Do not create a group- or world-writable file or directory

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Injection |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A constant `UnixFileMode` whose group-write (0o020) or other-write (0o002) bit is set is
supplied to a filesystem permission API — the `mode` argument of `File.SetUnixFileMode` or
`Directory.CreateDirectory`, or the value assigned to `FileInfo.UnixFileMode`,
`DirectoryInfo.UnixFileMode`, or `FileStreamOptions.UnixCreateMode`.

```csharp
using System.IO;

public void Configure(string path, FileInfo info)
{
    // 👎 reported — other-write makes the file world-writable
    File.SetUnixFileMode(path, UnixFileMode.OtherWrite);

    // 👎 reported — an OR-combination that folds in the group-write bit
    File.SetUnixFileMode(path, UnixFileMode.UserRead | UnixFileMode.UserWrite | UnixFileMode.GroupWrite);

    // 👎 reported — the create mode of a new directory is world-writable
    Directory.CreateDirectory(path, UnixFileMode.UserRead | UnixFileMode.UserWrite | UnixFileMode.UserExecute | UnixFileMode.OtherWrite);

    // 👎 reported — assigning an over-permissive mode to the file entry
    info.UnixFileMode = UnixFileMode.GroupWrite;

    // 👍 owner-only access is not reported
    File.SetUnixFileMode(path, UnixFileMode.UserRead | UnixFileMode.UserWrite);
}
```

## Rule description

On Unix, a file's mode carries a write bit for each of owner, group, and other. When the
group-write bit is set, every local account in the file's group can change its contents;
when the other-write bit is set, every local account on the machine can. Whatever the
process later reads back from that path — a configuration file, a script it executes, a
data file it trusts — may have been replaced or tampered with by another user, so an
over-permissive mode is a local privilege-escalation and integrity hole (CWE-732).

The rule reports the mode value at a filesystem permission sink when its **constant** value
has the group-write or other-write bit set. The value is recognized whether it is:

- a single member, such as `UnixFileMode.OtherWrite`;
- an OR-combination, such as `UserRead | UserWrite | GroupWrite`;
- a broad `0o777`-style combo, whether inline or referenced through a `const` field.

The sinks inspected are `File.SetUnixFileMode`, `Directory.CreateDirectory`,
`FileInfo.UnixFileMode`, `DirectoryInfo.UnixFileMode`, and `FileStreamOptions.UnixCreateMode`.

### What is not reported

- **Owner-only or read-only sharing.** A mode that grants write to the owner alone, or that
  grants only read/execute to group and other, has no group- or other-write bit and is never
  reported.
- **A non-constant mode.** A mode read from a variable, parameter, field, or method result is
  not a compile-time constant, so the bit cannot be judged at the call site and nothing is
  reported.
- **A same-named member on another type.** Only the gated BCL filesystem types are considered;
  an unrelated `SetUnixFileMode`/`UnixFileMode`/`UnixCreateMode` on your own type is ignored.

Detection is local: the mode's bits are read from the argument or right-hand-side expression
at the call or assignment site. The rule performs no data-flow or interprocedural tracking, so
a mode first stored in a local and passed later is judged by that later expression, not traced
back to its constant origin.

### How to fix violations

Grant write to the owner only. Drop `GroupWrite` and `OtherWrite` from the mode and keep it as
tight as the file's real sharing needs — for a private file, `UnixFileMode.UserRead |
UnixFileMode.UserWrite`. If a group genuinely needs write access, prefer directory ownership and
group membership over an other-writable mode, and never leave the other-write bit set.

### Availability detection

The rule probes `System.IO.UnixFileMode` once per compilation via
`Compilation.GetTypeByMetadataName` and registers nothing when the analyzed project targets a
framework without it (netstandard2.0, .NET Framework — the type is .NET 7+). A project that
cannot call these APIs pays nothing and never receives a diagnostic it cannot act on. No
target-framework string is parsed.

## Performance

The clean path is a syntactic screen: an invocation is ignored unless it is a member call named
`SetUnixFileMode` or `CreateDirectory` carrying at least two arguments, and an assignment is
ignored unless its target member is named `UnixCreateMode` or `UnixFileMode`. Only after that
screen passes does the analyzer bind the call, confirm the containing type is a gated filesystem
type, and read the mode's constant value — so ordinary code is rejected with no allocation and no
semantic query.
