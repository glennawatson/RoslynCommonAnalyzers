## Cause

`XslCompiledTransform.Load` is called with an `XsltSettings` that enables embedded
script. The settings expression at the call site is one of the script-enabling shapes:

- an object initializer that sets `EnableScript = true`;
- the `new XsltSettings(enableDocumentFunction, enableScript)` constructor with a
  constant `enableScript` of `true`;
- the static `XsltSettings.TrustedXslt`, which turns on both the `document()` function
  and script.


## Rule description

`XslCompiledTransform.Load` compiles the stylesheet it is handed. When the `XsltSettings`
passed alongside it enable script, an embedded script block in the stylesheet is compiled
and executed in the host process with the host's privileges. A stylesheet that comes from,
or is influenced by, untrusted input is then arbitrary code execution: the stylesheet is
not merely transformed data but a program the attacker gets to run.

The rule reports the settings argument of a `Load` call when, at the call site, the
settings enable script:

- **an object initializer** with `EnableScript = true`;
- **a constructor** whose `enableScript` argument is a constant `true` (positional or named);
- **`XsltSettings.TrustedXslt`**, the pre-built settings that enable script.

### What is not reported

- **Script disabled.** `XsltSettings.Default`, an initializer with `EnableScript = false`,
  a constructor with `enableScript` false, or settings that enable only the `document()`
  function are not script execution and are not reported.
- **Settings held in a variable.** Settings first stored in a local or field and passed to
  `Load` later are not followed. Detection is local to the call site; the rule performs no
  data-flow or interprocedural tracking.
- **A same-named method on another type.** Only a `Load` whose containing type is the real
  `System.Xml.Xsl.XslCompiledTransform`, taking the real `System.Xml.Xsl.XsltSettings`, is
  considered; an unrelated `Load`/`XsltSettings` elsewhere is ignored.

### How to fix violations

Load untrusted stylesheets with script disabled â€” pass `XsltSettings.Default`, or leave
`EnableScript` false â€” and never enable script for a stylesheet you do not fully control.
If a trusted stylesheet genuinely needs helper logic, move that logic into vetted, compiled
extension objects supplied by the host rather than script embedded in the stylesheet.

### Availability detection

The rule probes `System.Xml.Xsl.XslCompiledTransform` and `System.Xml.Xsl.XsltSettings`
once per compilation via `Compilation.GetTypeByMetadataName` and registers nothing when
either is absent, so a project that cannot call these APIs pays nothing. No target-framework
string is parsed.

## Performance

The clean path is a syntactic screen: an invocation is ignored unless it is a member call
whose method name is `Load` and carries at least two arguments. Only after that screen passes
does the analyzer bind the call, confirm the containing type, locate the `XsltSettings`
argument, and inspect its shape â€” so an ordinary invocation is rejected with no allocation and
no semantic query.
