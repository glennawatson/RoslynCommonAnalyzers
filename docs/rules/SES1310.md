# SES1310 — Do not bind to a directory without authenticating

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Injection |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `System.DirectoryServices.DirectoryEntry` is constructed so that it binds to the
directory anonymously — either by selecting `AuthenticationTypes.Anonymous`, or by
binding to an `LDAP://` path with an explicitly empty username and password.

```csharp
using System.DirectoryServices;

public DirectoryEntry Connect()
{
    // 👎 reported — anonymous authentication type
    var byType = new DirectoryEntry("LDAP://ad.example.com", "svc", "pw", AuthenticationTypes.Anonymous);

    // 👎 reported — anonymous authentication type set in an object initializer
    var byInitializer = new DirectoryEntry("LDAP://ad.example.com")
    {
        AuthenticationType = AuthenticationTypes.Anonymous,
    };

    // 👎 reported — empty username and password against an LDAP path
    return new DirectoryEntry("LDAP://ad.example.com", "", "");
}
```

## Rule description

An LDAP *bind* is how the application authenticates to the directory. A bind that
supplies no identity — an **anonymous bind** — succeeds as nobody, and two problems
follow:

- **Unauthenticated reads.** Any party who can reach the endpoint can enumerate the
  portion of the directory the anonymous identity is allowed to see. Directory data
  routinely includes user names, email addresses, group membership, and organisational
  structure that should require an authenticated caller.
- **Trusting an unproven peer.** Because the connection carries no proven identity, any
  authorization or trust the application layers on top of that connection is built on
  data it never verified. An attacker who can answer on the directory's behalf (or
  populate it) can steer those decisions.

The rule reports two explicit, local shapes on a `DirectoryEntry` construction:

1. **An anonymous authentication type.** `AuthenticationTypes.Anonymous` passed as the
   `authenticationType` constructor argument (positional or named), or assigned to the
   `AuthenticationType` member in an object initializer. The value is bound and confirmed
   to be the `System.DirectoryServices.AuthenticationTypes.Anonymous` enum member, so a
   same-named member on another type is ignored.
2. **An empty-credential bind.** A construction whose `path` argument is an `LDAP://`
   string literal (case-insensitive) and whose `username` **and** `password` arguments
   are both empty-string literals (`""`) or `null` literals.

Both shapes anchor on the `new DirectoryEntry(...)` expression itself — the explicit form
and the target-typed `new(...)` form are both recognised.

### What is not reported

- **A supplied credential.** A bind that passes a real username and password
  (`new DirectoryEntry("LDAP://…", "svc", "pw")`) is not reported. Only credentials that
  are *explicitly* empty or `null` at the call site count as an anonymous bind.
- **A non-anonymous authentication type.** `AuthenticationTypes.Secure` and every other
  value are left alone; only `Anonymous` is reported.
- **A non-LDAP provider or a non-literal path.** The empty-credential shape requires an
  `LDAP://` string literal; a `WinNT://` path, or a path supplied through a variable, is
  not reported.
- **A same-named type.** A `DirectoryEntry` from another namespace, or an unrelated type,
  is ignored — the created type is bound and must be
  `System.DirectoryServices.DirectoryEntry`.

Detection is local to the construction. The rule performs no data-flow or interprocedural
tracking, so a credential or authentication type flowed in from elsewhere is not followed.

### How to fix violations

There is no automatic code fix, because the correct credentials and authentication type
depend on the deployment. Bind with a real account and a non-anonymous authentication
type so the directory proves who it is talking to:

```csharp
using System.DirectoryServices;

var entry = new DirectoryEntry(
    "LDAP://ad.example.com",
    userName,
    password,
    AuthenticationTypes.Secure | AuthenticationTypes.Sealing | AuthenticationTypes.Signing);
```

Keep the credentials out of source — read them from a secret store or the hosting
environment — and prefer an authentication type that also seals and signs the channel so
the exchange is confidential and tamper-evident.

### Availability detection

The rule is gated on `System.DirectoryServices.DirectoryEntry` and
`System.DirectoryServices.AuthenticationTypes` resolving in the analyzed compilation. A
project that does not reference `System.DirectoryServices` registers nothing and pays
nothing.

## Performance

The clean path is syntactic. An explicit construction is ignored unless its type name is
`DirectoryEntry`; a target-typed `new(...)` is ignored unless it names `Anonymous` or
binds an `LDAP://` literal with empty credentials. Only after that screen passes does the
analyzer bind the created type and confirm the `Anonymous` enum member. The gated types
are resolved once per compilation, not per construction.
