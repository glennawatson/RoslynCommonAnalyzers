# SES1401 — A type resolved from non-constant data must not be instantiated or used as a deserialization target

**Category:** Serialization
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

`Type.GetType(name)` loads whatever type its string names. When that string is a
constant the type is fixed at compile time; when it comes from data an attacker can
influence — a request field, a header, a message body, a config value they control —
the *attacker* chooses which type gets loaded. Feeding that resolved type straight
into `Activator.CreateInstance` or a `Deserialize` call then runs the chosen type's
construction and deserialization logic. A "gadget" type whose constructor or
deserialization callback touches the file system, opens a connection, or starts a
process turns a crafted type name into code execution.

The rule reports the inline `Type.GetType(nonConstant)` expression when it is passed
directly as the `System.Type` argument of one of these calls:

```csharp
using System;
using System.IO;

// reported (SES1401): the type to build is chosen by 'typeName'
Activator.CreateInstance(Type.GetType(typeName));
Activator.CreateInstance(nonPublic: true, type: Type.GetType(typeName));
serializer.Deserialize(Type.GetType(typeName), stream);   // any method named Deserialize taking a Type
formatter.Deserialize(payload, Type.GetType(typeName));   // Type argument in any position
```

The fix is to constrain which types may be resolved. Bind a constant type, or
resolve the name against a fixed allow-list of expected type names and reject
anything else, before you instantiate or deserialize:

```csharp
// only the types you expect can ever be built
static readonly Dictionary<string, Type> Allowed = new()
{
    ["invoice"] = typeof(Invoice),
    ["receipt"] = typeof(Receipt),
};

if (Allowed.TryGetValue(name, out var type))
{
    var value = Activator.CreateInstance(type);
}
```

## What is reported

An argument is flagged when **all** of these hold:

- the outer call is `Activator.CreateInstance(...)` (resolved on `System.Activator`)
  or any method named `Deserialize`;
- the argument bound to that call's `System.Type` parameter is an inline
  `Type.GetType(...)` call (the static factory on `System.Type`);
- the type-name argument of that `Type.GetType` call is **not** a compile-time
  constant.

A constant type name (`Type.GetType("System.Text.StringBuilder")`) is safe and not
reported. `typeof(T)`, an instance `assembly.GetType(name)` (a different method on a
different type), the generic `Activator.CreateInstance<T>()`, and a `CreateInstance`
method that is not `System.Activator`'s are all left alone.

## Inline-only scope

Only the inline shape — the `Type.GetType(nonConstant)` call written directly as the
type argument — is reported. A type first stored in a local and used on a later line:

```csharp
var type = Type.GetType(typeName);
Activator.CreateInstance(type);           // not reported
```

is **not** flagged. Confirming that the local carries an attacker-influenced type
would require data-flow tracking, which this rule deliberately avoids to stay fast
and free of false positives. Prefer the inline form at the sink, or guard the
resolution with an allow-list, so the check the rule performs matches the code.

## Why there is no code fix

There is no safe automatic rewrite. The correct change is to restrict which types may
be instantiated — an allow-list, a constant type, or a validated binder — and only
the author knows which types are legitimate for the call. An analyzer cannot invent
that policy, so the rule reports and leaves the decision to the developer.

## Proving the API exists

The rule resolves `System.Activator` and `System.Type` once per compilation. Both
anchor the match rather than being suggested APIs, and both exist on every target
framework, so they are passed through to the match: when a symbol is absent the
comparison simply never succeeds and the rule stays silent, so nothing that cannot
occur is ever reported.

## Performance

The no-diagnostic path is syntactic. An invocation whose member is not
`CreateInstance` or `Deserialize`, or that carries no argument shaped like
`x.GetType(...)`, is rejected before the semantic model is touched; only a matching
shape is bound to confirm the target method and locate the `Type.GetType` argument.
