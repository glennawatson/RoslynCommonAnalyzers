# SES1402 — Do not load an assembly from raw bytes or a non-constant location

**Category:** Serialization
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

Loading an assembly is code execution. The runtime runs the assembly's module
initializers and makes every type in it callable, all with the full trust of the
process that loaded it. When the assembly's origin is untrusted input, that is an
arbitrary-code-execution sink.

The rule reports the load call in three situations:

```csharp
using System.Reflection;
using System.Runtime.Loader;

// all reported (SES1402):
Assembly.Load(blob);                      // raw bytes: whatever produced 'blob' decides what runs
context.LoadFromStream(stream);           // a stream of assembly bytes: same risk
Assembly.LoadFrom(path);                  // non-constant path: a caller controls which file executes
Assembly.LoadFile(path);                  // non-constant path
Assembly.UnsafeLoadFrom(path);            // non-constant path (also skips some load-context checks)
```

`Assembly.Load(byte[])` and `AssemblyLoadContext.LoadFromStream` take the
assembly as an in-memory buffer, so there is no signature or fixed location to
anchor trust to — whatever produced those bytes (a download, a deserialized blob,
a database column) decides what code runs. `LoadFrom`, `LoadFile`, and
`UnsafeLoadFrom` with a **non-constant** path let a caller-controlled string
decide which file is executed, so a planted or swapped file runs as your process.

## What is reported

The invocation is flagged when the bound method is one of:

- `System.Reflection.Assembly.Load(byte[])` — the raw-bytes overload, **always**
  (loading assembly bytes is inherently a code-execution risk);
- `System.Runtime.Loader.AssemblyLoadContext.LoadFromStream(...)` — **always**; or
- `System.Reflection.Assembly.LoadFrom(...)` / `LoadFile(...)` /
  `UnsafeLoadFrom(...)` whose path argument is **not** a compile-time constant.

The method symbol is bound, so a same-named `Load`/`LoadFrom` method on any other
type is ignored.

## What is not reported

- **The safe identity overloads.** `Assembly.Load(string assemblyName)` and
  `Assembly.Load(AssemblyName)` resolve an already-trusted assembly by name
  through the normal binding rules; they are told apart from the raw-bytes
  overload by their first parameter type and are never reported.

  ```csharp
  Assembly.Load("System.Text.Json");   // not reported — identity, not raw bytes
  ```

- **A constant, hard-coded path.** A literal or `const` path to `LoadFrom` /
  `LoadFile` / `UnsafeLoadFrom` is lower risk and out of scope; only a
  non-constant path (a parameter, a field, a concatenation involving one) is
  reported.

  ```csharp
  Assembly.LoadFrom("plugins/known.dll");   // not reported — constant path
  ```

- **A trusted embedded resource.** When the byte/stream argument is *directly*
  `<assembly>.GetManifestResourceStream(...)`, the bytes ship inside your own
  assembly and the load is left silent.

  ```csharp
  context.LoadFromStream(host.GetManifestResourceStream("Embedded.Plugin.dll"));   // not reported
  ```

  This is a purely local, syntactic recognition of the argument expression: if the
  bytes are copied into a local first, or read through a helper, the origin can no
  longer be told apart locally and the load is reported.

## Guidance

Load only assemblies you ship — referenced normally, or from a fixed, trusted
path. When you must load something sourced at run time, verify a strong-name or
Authenticode signature before loading it, and isolate the load in a dedicated,
collectible `AssemblyLoadContext` so it cannot see or tamper with your own types.

## Why there is no code fix

There is no safe automatic rewrite. Whether the input can be trusted, where a
signature check belongs, and how to isolate the load are decisions only the author
can make, so the rule reports and leaves the change to them.

## Proving the API exists

The rule resolves `System.Reflection.Assembly` in the analyzed compilation and
registers nothing when it is absent, so a project that cannot call these APIs
receives no diagnostic and pays no analysis cost. `AssemblyLoadContext` is
resolved the same way for the `LoadFromStream` check.

## Performance

The no-diagnostic path is syntactic. An invocation whose member name is not one of
the guarded load methods, or which has no arguments, is rejected before the
semantic model is touched; only a matching shape is bound to confirm the declaring
type and inspect the source argument.
