# SES1403 ‚Äî JSON deserialization depth limit must stay within a safe ceiling

| Item | Value |
| --- | --- |
| Category | Serialization |
| Severity | Info |
| Code fix | No |
| Default | Enabled |

## Cause

A `MaxDepth` property of `System.Text.Json.JsonSerializerOptions`,
`System.Text.Json.JsonReaderOptions`, or `System.Text.Json.JsonDocumentOptions` is assigned a
compile-time constant strictly greater than the configured ceiling (default `64`).

```csharp
using System.Text.Json;

public JsonSerializerOptions Build()
    => new JsonSerializerOptions { MaxDepth = 5000 };
```

## Rule description

`System.Text.Json` reads nested objects and arrays recursively, one stack frame per level. To stop a
hostile payload from nesting so deeply that the reader recurses off the end of the thread stack, the
serializer caps the nesting at `MaxDepth`, which defaults to `64`. That cap is a denial-of-service
control: a `StackOverflowException` cannot be caught, so a single crafted request that blows the
stack tears down the entire process ‚Äî every other in-flight request with it.

Raising `MaxDepth` to a large constant hands that control back to the attacker. An input source that
an attacker can influence ‚Äî a request body, an uploaded document, a message off a queue ‚Äî can then
nest to the raised depth and exhaust the stack on demand. The default of `64` already accommodates
realistic document shapes; a value in the hundreds or thousands almost always reflects working around
a specific deep document rather than a considered limit.

```csharp
// Reported at 5000: the deep-nesting DoS guard is opened wide.
var options = new JsonSerializerOptions { MaxDepth = 5000 };

// Also reported: a plain assignment to the same property.
options.MaxDepth = 5000;
```

The rule matches the property by its bound symbol, so it fires on all three option types and on both
the object-initializer member (`{ MaxDepth = N }`) and the plain assignment (`x.MaxDepth = N`).

## What is not reported

- **A value of `0`.** `MaxDepth = 0` tells the serializer to use its default (`64`), so it is safe
  and left alone.

  ```csharp
  // Not reported: 0 selects the framework default of 64.
  var options = new JsonSerializerOptions { MaxDepth = 0 };
  ```

- **A value at or below the ceiling.** The default (`64`) and anything smaller is fine.

- **A non-constant depth.** When the value is a method parameter, a field, or a value read from
  configuration, its size cannot be judged from the source ‚Äî it may well be a safe number at runtime
  ‚Äî so the rule stays silent rather than guess. Reading the limit from configuration is by design a
  supported way to silence the rule.

  ```csharp
  // Not reported: the value is unknown at compile time.
  public JsonSerializerOptions Build(int depth)
      => new JsonSerializerOptions { MaxDepth = depth };
  ```

- **A `MaxDepth` member on any other type.** The property is matched by its bound symbol, not by
  name, so a `MaxDepth` on an unrelated type is left alone.

## How to fix violations

Leave `MaxDepth` at its default, or set it no higher than the deepest document your inputs genuinely
require. If a specific trusted input really is deeply nested, prefer restructuring the data over
raising the global limit; where the limit must be tunable, read it from configuration so the value is
a deployment decision rather than a hard-coded constant ‚Äî the rule then treats it as non-constant and
stays silent. There is deliberately **no code fix**: the right ceiling is a threat-model decision, not
a mechanical substitution.

## Configuration

```ini
# Rule-specific
securitysharp.SES1403.maxdepth = 64

# Project-wide (applies to any SecuritySharp rule that reads a depth ceiling)
securitysharp.maxdepth = 64
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `maxdepth` | integer `>= 1` | `64` | The largest depth a constant `MaxDepth` assignment may set before it is reported. |

The rule-specific key wins over the project-wide key. An unset, unparsable, or nonsensical value
(below `1`) falls back to the default, so a typo cannot silently disable the rule.

## Performance

The rule gates on `System.Text.Json.JsonSerializerOptions` resolving in the compilation, so a target
framework without `System.Text.Json` registers nothing and pays nothing. Each assignment is filtered
syntactically first ‚Äî only a left-hand side that names `MaxDepth` reaches the semantic model ‚Äî and the
`.editorconfig` ceiling is read only after a constant value above the default has already been found,
keeping the common no-diagnostic path allocation-free.
