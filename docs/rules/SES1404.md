# SES1404 — A type must not be instantiated by name from non-constant data

**Category:** Serialization
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

The string overloads of `Activator.CreateInstance(assemblyName, typeName, ...)` and
`Activator.CreateInstanceFrom(assemblyFile, typeName, ...)` load and construct
whatever type the `typeName` string names. When that string is a constant the type is
fixed at compile time; when it comes from data an attacker can influence — a request
field, a header, a message body, a config value they control — the *attacker* chooses
which type gets built. Merely running a hostile type's constructor, before any method
is called on the new instance, can touch the file system, open a connection, or start
a process, so a crafted type name turns into code execution (CWE-470, "unsafe
reflection").

The rule reports the `typeName` argument of these by-name overloads when it is not a
compile-time constant:

```csharp
using System;

// reported (SES1404): the type to build is chosen by 'typeName'
Activator.CreateInstance("MyAssembly", typeName);
Activator.CreateInstance("MyAssembly", typeName, activationAttributes);
Activator.CreateInstance(typeName: typeName, assemblyName: "MyAssembly");   // named / reordered
Activator.CreateInstanceFrom("plugin.dll", typeName);
Activator.CreateInstance("MyAssembly", "MyApp.Handlers." + suffix);         // built from non-constant data
```

The fix is to constrain which types may be instantiated. Pass a constant type name,
or resolve the incoming name against a fixed allow-list of expected type names and
reject anything else, before you construct the instance:

```csharp
// only the types you expect can ever be built
static readonly Dictionary<string, Type> Allowed = new()
{
    ["invoice"] = typeof(Invoice),
    ["receipt"] = typeof(Receipt),
};

if (Allowed.TryGetValue(name, out var type))
{
    var value = Activator.CreateInstance(type);
}
```

## What is reported

An argument is flagged when **all** of these hold:

- the call is `Activator.CreateInstance(...)` or `Activator.CreateInstanceFrom(...)`,
  resolved on `System.Activator`;
- the bound overload leads with two `string` parameters — the assembly reference
  followed by the type name — which selects exactly the by-name overloads;
- the argument bound to that second (`typeName`) parameter is **not** a compile-time
  constant.

A constant type name (`Activator.CreateInstance("MyAssembly", "MyApp.KnownType")`) is
safe and not reported, as is a name held in a `const` local, which the compiler folds
to a constant. This rule is the by-name complement to the inline
`Activator.CreateInstance(Type.GetType(x))` shape, which is reported separately; the
`Type`-taking overloads, the generic `Activator.CreateInstance<T>()`, and a
`CreateInstance` method that is not `System.Activator`'s are all left alone here.

## Direct-shape scope

Only the direct shape is reported: a non-constant expression written as the `typeName`
argument. Any non-constant expression there is flagged — a parameter, a field, a
method result, a concatenation. A `const` name is folded to a constant and is not
flagged. The rule does not track a name across assignments to decide whether an
earlier value was attacker-influenced; that would require data-flow tracking, which
this rule deliberately avoids to stay fast and free of false positives. Guard the
resolution with an allow-list, or pass a constant name, so the check the rule
performs matches the code.

## Why there is no code fix

There is no safe automatic rewrite. The correct change is to restrict which types may
be instantiated — an allow-list, a constant type name, or a validated binder — and
only the author knows which types are legitimate for the call. An analyzer cannot
invent that policy, so the rule reports and leaves the decision to the developer.

## Proving the API exists

The rule resolves `System.Activator` once per compilation and registers nothing when
it is absent, so a target framework without the by-name overloads pays nothing and
never receives a diagnostic it cannot act on. The overload itself is confirmed by
binding the call and checking its first two parameters are both `string`, so only a
real by-name activation is ever reported.

## Performance

The no-diagnostic path is syntactic. An invocation whose member is not
`CreateInstance` or `CreateInstanceFrom`, or that carries fewer than the two leading
arguments, is rejected before the semantic model is touched; only a matching shape is
bound to confirm the target overload and inspect the `typeName` argument.
