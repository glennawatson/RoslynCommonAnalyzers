# SES1405 — A MessagePack typeless deserializer must not reconstruct arbitrary types from untrusted input

**Category:** Serialization
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

MessagePack's *typeless* serialization embeds the full .NET type name of every value
in the payload and reconstructs that exact type on read. That is the same open type
binding that makes `BinaryFormatter` dangerous: whoever writes the bytes decides which
type is instantiated. When the bytes come from an attacker — a request body, a queue
message, a cached blob they can poison — they pick a "gadget" type whose constructor,
deserialization callback, or property setter touches the file system, opens a
connection, or starts a process, and a crafted payload becomes code execution
(CWE-502).

Two local shapes opt into that open type binding, and the rule reports both:

```csharp
using MessagePack;
using MessagePack.Resolvers;

// reported (SES1405): the always-typeless facade reconstructs whatever type the bytes name
object o = MessagePackSerializer.Typeless.Deserialize(untrustedBytes);
object a = await MessagePackSerializer.Typeless.DeserializeAsync(untrustedStream);

// reported (SES1405): referencing a typeless resolver opens the type set for any serializer built on it
var options = MessagePackSerializerOptions.Standard.WithResolver(TypelessObjectResolver.Instance);
var options2 = MessagePackSerializerOptions.Standard.WithResolver(TypelessContractlessStandardResolver.Instance);
```

The fix is to deserialize into a known contract type with the default (typed)
resolver, so only the type you named can ever be built:

```csharp
// the type is fixed at compile time; the payload cannot choose it
var order = MessagePackSerializer.Deserialize<Order>(untrustedBytes);
```

If a small set of shapes is genuinely expected, discriminate on a plain field and map
it to a fixed set of types yourself, rather than letting the payload name the type.

## What is reported

- A call bound to `MessagePackSerializer.Typeless.Deserialize` or
  `MessagePackSerializer.Typeless.DeserializeAsync` — the typeless facade, which is
  always open-typed. The whole call is reported.
- A `.Instance` reference to `MessagePack.Resolvers.TypelessObjectResolver` or
  `MessagePack.Resolvers.TypelessContractlessStandardResolver` — the resolvers that
  reconstruct a value's type from its embedded name. The reference is reported as the
  unsafe configuration, wherever it appears.

A typed `MessagePackSerializer.Deserialize<T>(...)` call built on the default resolver
is safe and not reported, because `T` fixes the type at compile time.

## Local-shape scope

The rule matches only what it can see at the call site. A typeless resolver first
stored in a field or a local and then passed to a serialize/deserialize call on a
later line:

```csharp
IFormatterResolver r = TypelessObjectResolver.Instance;   // reported here, at the reference
var options = MessagePackSerializerOptions.Standard.WithResolver(r);   // not reported: 'r' is a variable
```

is reported at the `.Instance` reference, not at the downstream `WithResolver` call —
tying the variable to the call would require data-flow tracking, which this rule
deliberately avoids to stay fast and free of false positives. Referencing the resolver
is itself the opt-in to open type binding, so that is where the diagnostic lands.

## Why there is no code fix

There is no safe automatic rewrite. Moving off typeless means choosing the contract
type (or the discriminated set of types) the payload is expected to carry, and only
the author knows what that is. An analyzer cannot invent that contract, so the rule
reports and leaves the decision to the developer.

## Proving the API exists

The rule resolves the typeless facade (`MessagePack.MessagePackSerializer+Typeless`)
and the two resolver types once per compilation. A project that does not reference
MessagePack resolves none of them, so nothing is registered and the rule costs
nothing and never reports a shape the project cannot contain.

## Performance

The no-diagnostic path is syntactic. An invocation whose member is not
`Deserialize`/`DeserializeAsync` qualified by `Typeless`, and a member access whose
name is not `Instance` qualified by a typeless resolver name, are rejected before the
semantic model is touched; only a matching shape is bound to confirm the type.
