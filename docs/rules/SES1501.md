# SES1501 — A CORS policy must not allow credentials together with any origin

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A CORS policy built with `CorsPolicyBuilder.AllowAnyOrigin()` responds with
`Access-Control-Allow-Origin: *`, and one built with `AllowCredentials()` responds
with `Access-Control-Allow-Credentials: true`. The Fetch standard forbids that
pair: a browser refuses to expose a credentialed response whose allowed origin is
the wildcard, so the cross-origin call fails. ASP.NET Core enforces the same rule
server-side — applying a policy that sets both throws an
`InvalidOperationException` when a request hits the endpoint, so the route is dead
at run time rather than merely insecure.

The combination is a classic misconfiguration: an attempt to "open up" CORS by
allowing every origin, then bolting on `AllowCredentials()` so cookies or the
`Authorization` header flow — which instead breaks every credentialed cross-origin
request. Had the runtime accepted it, it would send authenticated responses to any
site the user visits.

The rule reports the `AllowCredentials()` call when the same policy also allows any
origin, in a fluent chain or across the statements of one policy lambda:

```csharp
using Microsoft.AspNetCore.Cors.Infrastructure;

// all reported (SES1501):
services.AddCors(options =>
{
    options.AddPolicy("open", policy =>
        policy.AllowAnyOrigin().AllowCredentials());          // fluent chain

    options.AddDefaultPolicy(policy =>
    {
        policy.AllowAnyOrigin();
        policy.AllowAnyHeader();
        policy.AllowCredentials();                            // same lambda body
    });
});
```

Allow credentials only for the specific origins that must send them:

```csharp
options.AddPolicy("trusted", policy =>
    policy.WithOrigins("https://trusted.example")             // not reported
          .AllowCredentials());
```

## What is reported

The `AllowCredentials()` call is flagged when both hold:

- its bound method's containing type is
  `Microsoft.AspNetCore.Cors.Infrastructure.CorsPolicyBuilder` (the symbol is
  bound, so a same-named method on any other builder type is ignored); and
- the enclosing policy scope also contains an `AllowAnyOrigin()` call bound to
  `CorsPolicyBuilder`.

The "policy scope" is the configuration lambda body passed to `AddPolicy` /
`AddDefaultPolicy` (or the nearest enclosing lambda), and for a bare fluent chain
it is the single enclosing statement. This is a purely local, syntactic scan — no
data-flow. Two `CorsPolicyBuilder` calls written as separate statements *outside*
any policy lambda are deliberately not correlated, and neither are calls split
across two different policy lambdas.

## Why there is no code fix

There is no single safe rewrite. Dropping `AllowCredentials()` and dropping
`AllowAnyOrigin()` produce opposite behaviours, and the secure fix — replacing
`AllowAnyOrigin()` with `WithOrigins(...)` — needs the concrete trusted origins,
which the analyzer cannot supply. The author has to choose, so the rule reports and
leaves the change to them.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Cors.Infrastructure.CorsPolicyBuilder` in
the analyzed compilation and registers nothing when it is absent, so a project
without ASP.NET Core CORS receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. Only an invocation whose called member is
spelled `AllowCredentials` is bound; the policy-scope scan for `AllowAnyOrigin`
runs only after that call is confirmed to sit on `CorsPolicyBuilder`, which is rare.
