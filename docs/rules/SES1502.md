# SES1502 — A CORS origin predicate must not unconditionally allow every origin

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

`CorsPolicyBuilder.SetIsOriginAllowed(Func<string, bool>)` registers a predicate
that runs for each request to decide whether the request's `Origin` header is
allowed. The predicate exists to make that decision — to compare the incoming
origin against an allow-list.

A predicate that ignores its argument and always returns `true` throws that
decision away and accepts every origin. It is exactly equivalent to calling
`AllowAnyOrigin`, so the allow-list is a fiction. Paired with `AllowCredentials`
it is worse than pointless: the middleware reflects the caller's origin back in
`Access-Control-Allow-Origin` and sets `Access-Control-Allow-Credentials: true`,
so the browser hands credentialed cross-origin responses — cookies, authenticated
data — to any site that asks. An attacker-controlled page can then read a
logged-in user's data straight from your API.

The rule reports the predicate argument when it always returns `true`:

```csharp
using Microsoft.AspNetCore.Cors.Infrastructure;

// all reported (SES1502):
builder.SetIsOriginAllowed(_ => true);                 // discard expression lambda
builder.SetIsOriginAllowed(origin => true);            // named expression lambda
builder.SetIsOriginAllowed(origin => { return true; }); // block reducing to return true
builder.SetIsOriginAllowed(AllowAll);                  // method group to an always-true method

static bool AllowAll(string origin) => true;
```

Compare the origin against origins you actually trust instead:

```csharp
builder.SetIsOriginAllowed(origin =>
    origin == "https://app.example.com" || origin == "https://admin.example.com");
```

## What is reported

The call is flagged when both hold:

- the invoked method binds to `SetIsOriginAllowed` on
  `Microsoft.AspNetCore.Cors.Infrastructure.CorsPolicyBuilder` (the method is
  bound, so a same-named method on any other type is ignored); and
- its single predicate argument always returns `true`, in one of these local
  shapes:
  - an expression lambda `_ => true` / `origin => true` (parentheses around the
    lambda parameter or around the `true` literal do not matter);
  - a block lambda or anonymous method whose own `return` statements all return
    the literal `true` (with at least one such return) — statements before the
    return, and returns inside nested lambdas or local functions, do not change
    the result; or
  - a method group to a source method or local function of one of the above
    shapes.

Anything that inspects the origin — `origin => origin.StartsWith("https://…")`,
a block with a conditional `return false;` path, or a method group to such a
method — is not reported.

## Local analysis only

The predicate body is inspected purely locally: the expression body, or the
block's own `return` statements. No data-flow or interprocedural tracking is
performed, so a predicate whose result is computed elsewhere and flowed in — for
example a `Func<string, bool>` variable assigned a constant `true` somewhere
above the call — is not recognized. For a method group, only the referenced
method's own body is read; the rule does not follow calls it makes.

## Why there is no code fix

There is no safe automatic rewrite. The correct allow-list is domain knowledge
the analyzer does not have — which origins the application should trust — so it
cannot synthesize a replacement predicate. Removing the call entirely would
change the policy in a way the author may not intend. The author has to supply
the real set of allowed origins, so the rule reports and leaves the change to
them.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Cors.Infrastructure.CorsPolicyBuilder` in
the analyzed compilation and registers nothing when it is absent, so a project
without ASP.NET Core CORS receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An invocation whose member name is not
`SetIsOriginAllowed`, or which does not carry exactly one argument, is rejected
before the semantic model is touched; a lambda argument that is not `=> true` (or
a block that does not reduce to `return true;`) is rejected syntactically too, so
only a genuine always-true shape is bound to confirm the method and its
containing type.
