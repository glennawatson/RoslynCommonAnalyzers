# SES1503 — JWT signature verification must not be disabled on TokenValidationParameters

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A JSON Web Token is trusted for one reason only: its signature proves the token
was minted by the identity provider and has not been altered. Two flags on
`Microsoft.IdentityModel.Tokens.TokenValidationParameters` decide whether that
proof is actually checked:

- `RequireSignedTokens` (default `true`) rejects a token that carries no
  signature at all;
- `ValidateIssuerSigningKey` (default `true`) checks that the key that produced
  the signature is one the application accepts.

Setting either to `false` lets the validation pipeline accept a token whose
signature is missing or was produced by a key the attacker chose. Anyone can then
mint a token with whatever claims they like — any user id, any role — and the
application treats it as authentic. This is a complete authentication bypass and
the single most dangerous way to misconfigure token validation.

The rule reports the `= false` assignment on either flag, written directly or in
an object initializer:

```csharp
using Microsoft.IdentityModel.Tokens;

// all reported (SES1503):
parameters.RequireSignedTokens = false;                                    // direct assignment
parameters.ValidateIssuerSigningKey = false;                               // direct assignment
var p = new TokenValidationParameters { RequireSignedTokens = false };     // object initializer
var q = new TokenValidationParameters { ValidateIssuerSigningKey = false };// object initializer
```

Leave both flags at their secure default (or set them explicitly to `true`) and
supply the issuer signing keys the token must be verified against:

```csharp
var parameters = new TokenValidationParameters
{
    RequireSignedTokens = true,          // the default — an unsigned token is rejected
    ValidateIssuerSigningKey = true,     // the default — the signing key must be one you trust
    IssuerSigningKey = trustedKey,
};
```

## What is reported

The assignment is flagged when both hold:

- the assigned member is `RequireSignedTokens` or `ValidateIssuerSigningKey`, and
  its containing type is
  `Microsoft.IdentityModel.Tokens.TokenValidationParameters` (the symbol is bound,
  so a same-named property on any other type is ignored); and
- the right-hand side is the literal `false`.

## Scope: signature flags only

This rule covers only the two flags that turn off **signature verification**.
The issuer, audience, and lifetime flags (`ValidateIssuer`, `ValidateAudience`,
`ValidateLifetime`) are a separate concern — disabling them weakens *who* and
*when* a token is accepted for, not *whether it is genuine* — and are already
reported by an in-box analyzer, so SES1503 deliberately stays silent on them to
avoid double-reporting.

## Why there is no code fix

There is no safe automatic rewrite. Flipping the flag back to `true` (or deleting
the assignment to fall back to the secure default) is usually correct, but a
token validated with signature checking on also needs an `IssuerSigningKey` (or a
key resolver) that the analyzer cannot supply. The author has to provide the keys
the token is verified against, so the rule reports and leaves the change to them.

## Proving the API exists

The rule resolves `TokenValidationParameters` in the analyzed compilation and
registers nothing when it is absent, so a project that does not reference
`Microsoft.IdentityModel` receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An assignment whose right-hand side is not
the literal `false`, or whose left-hand side is not spelled `RequireSignedTokens`
or `ValidateIssuerSigningKey`, is rejected before the semantic model is touched;
only a matching shape is bound to confirm the containing type.
