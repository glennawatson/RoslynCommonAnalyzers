# SES1504 — A cookie with SameSite=None must be marked Secure

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

`SameSite=None` tells the browser to attach the cookie to cross-site requests,
undoing the same-site default that protects a cookie from being sent from another
origin. That relaxation is only safe when the cookie is also confined to HTTPS: a
`SameSite=None` cookie that lacks the `Secure` attribute is sent over plain HTTP,
where anyone on the network path can read or replay it. Modern browsers refuse the
pairing outright and **drop** a `SameSite=None` cookie that is not also `Secure`,
so the code is both insecure and broken.

The rule reports the `SameSite = SameSiteMode.None` member of a single object
initializer of `Microsoft.AspNetCore.Http.CookieOptions` or
`Microsoft.AspNetCore.Http.CookieBuilder` when that initializer does not also mark
the cookie secure:

```csharp
using Microsoft.AspNetCore.Http;

// reported (SES1504) — no Secure flag in the initializer:
var options = new CookieOptions { SameSite = SameSiteMode.None };
var builder = new CookieBuilder { SameSite = SameSiteMode.None };
CookieOptions targetTyped = new() { SameSite = SameSiteMode.None };
```

Set the securing member alongside `SameSite`:

```csharp
// clean — the cookie is confined to HTTPS:
var options = new CookieOptions { SameSite = SameSiteMode.None, Secure = true };
var builder = new CookieBuilder { SameSite = SameSiteMode.None, SecurePolicy = CookieSecurePolicy.Always };
```

If the cookie does not actually need to travel cross-site, prefer
`SameSiteMode.Lax` or `SameSiteMode.Strict` instead, which do not require the
`Secure` attribute to be accepted.

## The two types expose "secure" differently

`CookieOptions` and `CookieBuilder` control the same `Secure` cookie attribute
through different members, and the rule checks each type's own member:

- **`CookieOptions`** has a `bool Secure`. The initializer is clean when it sets
  `Secure` to anything other than the constant `false` — `Secure = true` secures
  the cookie, and a non-constant value such as `Secure = isProduction` is treated
  as securing so a deployment-driven flag is never flagged. Only an explicit
  `Secure = false` (or no `Secure` member at all) is reported.
- **`CookieBuilder`** has a `CookieSecurePolicy SecurePolicy`. The initializer is
  clean when `SecurePolicy` is set to anything other than
  `CookieSecurePolicy.None` — `Always` and `SameAsRequest` both emit the `Secure`
  attribute. Only `SecurePolicy = CookieSecurePolicy.None`, or no `SecurePolicy`
  member at all, is reported.

## What is reported

The `SameSite` member is flagged when all of the following hold:

- the object being created is `Microsoft.AspNetCore.Http.CookieOptions` or
  `Microsoft.AspNetCore.Http.CookieBuilder` (the created type is bound, so a
  same-named `SameSite` member on any other type is ignored);
- the initializer has a `SameSite` member whose value binds to
  `Microsoft.AspNetCore.Http.SameSiteMode.None`; and
- the same initializer has no securing sibling member for that type
  (`Secure` not set to constant `false` on `CookieOptions`; `SecurePolicy` set to
  a non-`None` value on `CookieBuilder`).

## Single-initializer scope

Detection is local to one object initializer — there is no data-flow or
interprocedural analysis. A `Secure` flag assigned on a later statement is **not**
tracked, so the initializer is still reported:

```csharp
var options = new CookieOptions { SameSite = SameSiteMode.None };  // reported
options.Secure = true;                                             // not consulted
```

Set the securing member inside the initializer, next to `SameSite`, so the secure
pairing is visible at the point the cookie is configured.

## Why there is no code fix

There is no single safe rewrite. Adding `Secure = true` (or
`SecurePolicy = CookieSecurePolicy.Always`) is usually correct, but if the cookie
does not need to be sent cross-site the right fix is to drop `SameSite=None` and
use `Lax`/`Strict` instead. The author has to decide which of those the cookie
needs, so the rule reports and leaves the change to them.

## Proving the API exists

The rule resolves `SameSiteMode`, `CookieOptions`, `CookieBuilder`, and
`CookieSecurePolicy` in the analyzed compilation and registers nothing when the
cookie types are absent, so a project without ASP.NET Core receives no diagnostic
and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An object initializer without a
`SameSite = <...>.None` member is rejected before the semantic model is touched;
only a matching shape is bound to confirm the cookie type, the `SameSiteMode.None`
value, and the absence of a securing sibling.
