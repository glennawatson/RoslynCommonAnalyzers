# SES1505 — The request body size limit must not be removed

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

ASP.NET Core caps the size of an incoming request body so that a single client
cannot stream an arbitrarily large payload into server memory or a temporary
file. That cap is the last line of defence against a trivial denial-of-service:
without it, one connection can keep sending bytes until the process runs out of
memory or the disk fills, taking the whole service down.

The rule reports the two ways that cap is removed.

## The attribute shape

Applying `[DisableRequestSizeLimit]` to a controller or an action turns off the
body-size limit for every request routed to it:

```csharp
using Microsoft.AspNetCore.Mvc;

[DisableRequestSizeLimit]            // reported (SES1505) — whole controller
public class UploadController : ControllerBase
{
    [DisableRequestSizeLimit]        // reported (SES1505) — single action
    [HttpPost("upload")]
    public IActionResult Upload() => Ok();
}
```

The applied attribute is bound to
`Microsoft.AspNetCore.Mvc.DisableRequestSizeLimitAttribute`, so a same-named
attribute from another namespace is ignored.

## The assignment shape

Setting `MaxRequestBodySize` to the literal `null` means "no limit". The rule
reports that assignment — written directly or in an object initializer — when the
assigned member's containing type is
`Microsoft.AspNetCore.Server.Kestrel.Core.KestrelServerLimits` or
`Microsoft.AspNetCore.Http.Features.IHttpMaxRequestBodySizeFeature`:

```csharp
using Microsoft.AspNetCore.Server.Kestrel.Core;
using Microsoft.AspNetCore.Http.Features;

// all reported (SES1505):
limits.MaxRequestBodySize = null;                                 // direct assignment
var l = new KestrelServerLimits { MaxRequestBodySize = null };    // object initializer
feature.MaxRequestBodySize = null;                                // per-request feature
```

Because the assigned member is bound to its symbol, the same property name on any
other type is ignored.

## What is not reported

Keeping a finite cap is the fix, so a numeric assignment is clean:

```csharp
limits.MaxRequestBodySize = 10_485_760;   // 10 MB — not reported
```

Set the limit to the largest legitimate upload you accept, and raise it
deliberately for the specific endpoints that need a larger body rather than
removing it outright.

## Why there is no code fix

There is no safe automatic rewrite. The right cap is application-specific — it
depends on what the endpoint legitimately accepts — so the analyzer cannot pick a
number, and silently deleting the opt-out could break an endpoint that was
intentionally raised. The author has to choose the limit, so the rule reports and
leaves the change to them.

## Proving the API exists

The attribute check is registered only when
`DisableRequestSizeLimitAttribute` resolves in the analyzed compilation, and the
assignment check only when `KestrelServerLimits` or
`IHttpMaxRequestBodySizeFeature` resolves. A project that references neither
surface registers nothing and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An attribute not spelled
`DisableRequestSizeLimit`, and an assignment whose right-hand side is not the
literal `null` or whose left-hand side is not spelled `MaxRequestBodySize`, are
rejected before the semantic model is touched; only a matching shape is bound to
confirm the attribute or containing type.
