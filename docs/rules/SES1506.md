# SES1506 — The developer exception page must be guarded by a development-environment check

**Category:** WebHardening
**Severity:** Info
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

`UseDeveloperExceptionPage` installs middleware that catches an unhandled request
exception and writes the full exception message, stack trace, and request state
straight back to the client. That is a debugging aid for a developer — and a
reconnaissance gift for an attacker. In production it discloses file paths, type
and framework versions, connection strings surfaced in exception messages, and the
internal call graph, all of which narrow the search for a deeper exploit.

The middleware is meant to run only in the Development environment. The call
therefore belongs behind a development-environment guard — an enclosing `if` (or
conditional) whose condition calls `IsDevelopment`:

```csharp
using Microsoft.AspNetCore.Builder;

// reported (SES1506): enabled unconditionally
app.UseDeveloperExceptionPage();

// not reported: guarded by the environment
if (app.Environment.IsDevelopment())
{
    app.UseDeveloperExceptionPage();
}
```

In production, serve a hardened error handler instead:

```csharp
app.UseExceptionHandler("/error");   // returns a generic page, leaks nothing
```

## The development-environment guard

A call that is lexically enclosed by an `if` statement or a conditional expression
whose condition calls a method named `IsDevelopment` is treated as
development-only and is **not** reported:

```csharp
if (env.IsDevelopment())
{
    app.UseDeveloperExceptionPage();          // not reported
}

app = env.IsDevelopment()
    ? app.UseDeveloperExceptionPage()          // not reported
    : app;
```

This is a purely local, syntactic check of the call's enclosing statements: the
guard call must appear in the condition of an ancestor `if`/conditional. No
data-flow or interprocedural analysis is performed, so an environment flag copied
into a local variable first, or a guard hidden behind a helper method, is not
recognized — keep the guard directly around the call.

## What is reported

The invocation is flagged when both hold:

- the invoked method is named `UseDeveloperExceptionPage` and its containing type
  is `Microsoft.AspNetCore.Builder.DeveloperExceptionPageExtensions` (the symbol is
  bound, so a same-named method on any other type is ignored); and
- no enclosing `if`/conditional condition calls `IsDevelopment`.

Both the extension-method form (`app.UseDeveloperExceptionPage()`) and the static
form (`DeveloperExceptionPageExtensions.UseDeveloperExceptionPage(app)`) are
covered, as is the overload that takes a `DeveloperExceptionPageOptions`.

## Why there is no code fix

There is no safe automatic rewrite. Deleting the call can break a developer's local
loop, and wrapping it in an `if (env.IsDevelopment())` guard requires an
environment accessor the analyzer cannot conjure or reliably locate. Swapping it
for `UseExceptionHandler` needs an error-handling path the author has to design.
The rule reports and leaves the decision to the developer.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Builder.DeveloperExceptionPageExtensions`
in the analyzed compilation and registers nothing when it is absent, so a project
without ASP.NET Core hosting receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An invocation whose invoked member is not
named `UseDeveloperExceptionPage` is rejected before the semantic model is touched;
only a matching shape is bound to confirm the containing type and to walk the
ancestor chain for a development guard.
