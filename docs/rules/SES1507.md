# SES1507 — AllowAnonymous and Authorize on the same declaration conflict

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

When one method or type carries both `[AllowAnonymous]` and `[Authorize]`, the
ASP.NET Core authorization pipeline honours the anonymous marker first and
short-circuits: the request is allowed through without authentication and the
`[Authorize]` never runs. The `[Authorize]` reads as protection but enforces
nothing, so an author who added it believing the endpoint is locked down is
mistaken — the endpoint is open.

The rule reports the misleading `[Authorize]` attribute:

```csharp
using Microsoft.AspNetCore.Authorization;

[Authorize]        // reported (SES1507): dead — AllowAnonymous below overrides it
[AllowAnonymous]
public class ReportsController : ControllerBase
{
}
```

Remove whichever marker does not match the intent:

```csharp
// keep it protected:
[Authorize]
public class ReportsController : ControllerBase { }

// or make the anonymous access explicit:
[AllowAnonymous]
public class ReportsController : ControllerBase { }
```

## What is reported

A single method or type declaration (class, struct, record, or interface) that
carries **both** markers is flagged, and the `[Authorize]` attribute is the
reported span:

- both markers in separate attribute lists — `[Authorize]` then `[AllowAnonymous]`;
- both markers in one list — `[Authorize, AllowAnonymous]`;
- either marker written as `[Authorize]`, `[AuthorizeAttribute]`, or fully
  qualified — each attribute is **bound** to its symbol and matched by attribute
  class, never by the written name;
- a subclass of `AuthorizeAttribute` or of `AllowAnonymousAttribute`, because a
  derived attribute is honoured by the runtime exactly as its base is.

## What is not reported

The conflict must sit on **one** declaration. A secured type with a single opened
member — `[Authorize]` on the controller and `[AllowAnonymous]` on one action — is
the intended way to expose one anonymous endpoint on an otherwise-protected
controller, and is not flagged:

```csharp
[Authorize]
public class ReportsController : ControllerBase
{
    [AllowAnonymous]                 // not reported — deliberate, per-action opt-out
    public IActionResult Health() => Ok();
}
```

The rule is purely local and symbol-bound: it inspects the attributes written on
the declaration itself and never follows base types, partial parts in other files,
or convention-based policies. This keeps it fast and free of false positives.

## Why there is no code fix

There is no safe automatic rewrite. Which marker is wrong is the author's intent,
not something the analyzer can infer: removing `[Authorize]` and removing
`[AllowAnonymous]` produce opposite security postures. The rule reports and leaves
the decision to the author.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Authorization.AllowAnonymousAttribute` and
`Microsoft.AspNetCore.Authorization.AuthorizeAttribute` in the analyzed compilation
and registers nothing when either is absent, so a project that does not reference
ASP.NET Core authorization receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. A declaration with fewer than two attributes
is rejected before the semantic model is touched; only when two or more attributes
are present are they bound to confirm the two marker types.
