# SES1508 — A validation method must not fail open

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | WebHardening |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method shaped like a security check — its name begins with `Validate`, `Verify`,
`Authenticate`, `Authorize`, `Check`, `IsValid`, `IsAuthentic`, or `Ensure`, and it
returns `bool`, `Task<bool>`, or `ValueTask<bool>` — catches an exception and returns
success, so a caller that forces the exception is treated as validated.

```csharp
using System.Security.Cryptography;

// 👎 reported: a forced CryptographicException is treated as a valid signature
bool VerifySignature(byte[] data, byte[] signature)
{
    try
    {
        return Rsa.VerifyData(data, signature);
    }
    catch (CryptographicException)
    {
        return true;
    }
}

// 👍 a check that cannot complete did not pass
bool VerifySignature(byte[] data, byte[] signature)
{
    try
    {
        return Rsa.VerifyData(data, signature);
    }
    catch (CryptographicException)
    {
        return false;
    }
}
```

## Rule description

A security check answers one question: *is this allowed?* An exception raised while
answering it — a malformed token that throws while parsing, a signature routine that
rejects bad input by throwing, a key provider that is momentarily unavailable — is a
check that could **not** be completed. Returning `true` in that case fails **open**:
the caller proceeds as if the check passed, and an attacker who can reliably provoke
the exception is authenticated or validated without ever satisfying the check. The
safe direction is to fail **closed** — return `false`, so an unanswerable check is a
failed check.

The rule reports a `catch` clause only when every one of these holds, so it fires on
the fail-open shape and stays quiet on ordinary fail-closed error handling:

- **The method looks like a security check.** Its name begins with one of `Validate`,
  `Verify`, `Authenticate`, `Authorize`, `Check`, `IsValid`, `IsAuthentic`, or
  `Ensure`, and it returns `bool`, `Task<bool>`, or `ValueTask<bool>`. A local
  function shaped this way counts; a lambda does not (a `return` inside a lambda
  leaves the lambda, not the check).
- **The catch is broad or security-relevant.** Either a bare `catch`, a
  `catch (System.Exception)`, or a catch of a cryptographic exception, an
  authentication exception, or a type whose name ends in `SecurityTokenException`. A
  catch of a narrow, non-security exception is left alone.
- **The catch's only effect is to return success.** Either its single statement is
  `return true;` (or `return … FromResult(true);` for a task-returning method), or its
  body is empty and control falls straight through to a trailing `return true;`. A
  catch that also logs, retries, or does anything else before returning is not
  reported.

### Detection heuristic

Detection is a curated, **name-and-shape** match kept local to the one method — the
rule never traces values across method calls (that would need whole-program
dataflow), so it fires only when the fail-open shape is visible in the method itself.
The prefixes and the "returns success" shape are deliberately narrow to keep false
positives near zero; a catch that recovers legitimately, returns a computed result, or
fails closed is never reported.

## How to fix violations

Return `false` from the catch (or from the fall-through) so a check that cannot be
completed is treated as a failure, and let the exception propagate where the caller
genuinely needs to distinguish "invalid" from "could not check".

```csharp
// before
catch (System.Exception)
{
    return true;
}

// after
catch (System.Exception)
{
    return false;
}
```

If the method truly must swallow the exception for a non-security reason, the shape is
the problem: split the best-effort work out of the security-check-named method so the
name no longer promises a verdict it does not deliver.
