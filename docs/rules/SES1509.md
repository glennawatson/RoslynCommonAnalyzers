# SES1509 — Backtracking-prone regular expression has no match timeout

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

The default .NET regex engine matches by backtracking. For most patterns that is
linear, but when a pattern **nests or overlaps quantifiers** — a group repeated by
an unbounded quantifier (`*`, `+`, or `{n,}`) whose body itself repeats or offers an
alternation — the number of ways the engine can try to match one input grows
exponentially with the input length. A single crafted string then makes one call to
`Match`/`IsMatch`/`Replace` run for seconds, minutes, or effectively forever, pinning
a CPU core. On a server that evaluates the regex against request data, one request
denies service to everyone. This is a **regular-expression denial of service
(ReDoS)**, [CWE-1333](https://cwe.mitre.org/data/definitions/1333.html).

The classic evil shapes all share the pattern "repeat something that can already
repeat":

```csharp
using System.Text.RegularExpressions;

// all reported (SES1509):
var r1 = new Regex("(a+)+");                       // '+' over a group ending in '+'
var r2 = new Regex("(a*)*");                       // '*' over a group ending in '*'
var r3 = new Regex("([a-z]+)*");                   // '*' over a repeated class
var r4 = new Regex("(.*)*");                       // '*' over '.*'
bool m1 = Regex.IsMatch(input, "(a|aa)+");         // '+' over overlapping alternatives
var r5 = new Regex(@"(\d+)*");                     // '*' over '\d+'

[GeneratedRegex("([a-z]+)*")]                       // the same shape in a source-generated regex
private partial Regex Evil();
```

## How to fix it

Any one of these removes the diagnostic; pick by intent:

1. **Bound the match time.** Pass a `TimeSpan` so a runaway match is aborted with a
   `RegexMatchTimeoutException` you can catch:

   ```csharp
   var r = new Regex("(a+)+", RegexOptions.None, TimeSpan.FromMilliseconds(200));
   bool m = Regex.IsMatch(input, "(a+)+", RegexOptions.None, TimeSpan.FromMilliseconds(200));
   ```

   On `[GeneratedRegex]` the timeout is the `matchTimeoutMilliseconds` argument:

   ```csharp
   [GeneratedRegex("(a+)+", RegexOptions.None, matchTimeoutMilliseconds: 200)]
   private partial Regex Bounded();
   ```

2. **Use the non-backtracking engine.** `RegexOptions.NonBacktracking` (available on
   .NET 7 and later) matches in time linear in the input, so catastrophic
   backtracking is impossible:

   ```csharp
   var r = new Regex("(a+)+", RegexOptions.NonBacktracking);
   ```

   It does not support backreferences or lookaround; if the pattern needs those,
   use a timeout instead.

3. **Rewrite the pattern** so no repeated group contains its own repetition or
   overlapping alternation. `(a+)+` and `(a|aa)+` both reduce to `a+`; `(.*)*`
   reduces to `.*`. An atomic group `(?>...)` or possessive-style anchoring also
   removes the ambiguity the engine backtracks over.

## What is reported

The pattern argument is flagged only when **all** of these hold, in one of the three
local shapes below:

- **No match timeout is supplied** — no `System.TimeSpan` argument on the
  constructor or static call, and no `matchTimeoutMilliseconds` argument on
  `[GeneratedRegex]`.
- **`RegexOptions.NonBacktracking` is not among the options** — a constant options
  value is inspected for the flag; a combination such as
  `RegexOptions.Compiled | RegexOptions.NonBacktracking` is recognized as safe.
- **The pattern is a compile-time-constant string** (a literal or a `const`
  expression) that carries a nested or overlapping unbounded quantifier: an outer
  `*`, `+`, or `{n,}` applied to a group whose body contains a `*`, `+`, or `{n,}`,
  or a top-level `|`.

The three shapes:

- the first argument of `new System.Text.RegularExpressions.Regex(pattern, ...)`;
- the `pattern` argument of the static `Regex.IsMatch`, `Regex.Match`,
  `Regex.Matches`, `Regex.Replace`, `Regex.Split`, and `Regex.Count(input, pattern, ...)`
  overloads;
- the `pattern` argument of a `[GeneratedRegex(pattern, ...)]` attribute.

The constructor, method, or attribute is **bound** to confirm the real
`System.Text.RegularExpressions` type; a same-named type or method is never reported.
A pattern passed by name (`pattern:`) is matched wherever it appears.

## What is not reported

- **A non-constant pattern.** `new Regex(userInput)` is not judged here — a pattern
  built from data is a regex-injection concern owned by a separate rule. SES1509
  fires only on a **fixed, trusted-but-dangerously-shaped** pattern that is run
  unguarded, so the two rules never overlap.
- **A repeated group with no inner repetition or alternation.** `(abc)+` and
  `(\d{4})+` are linear — a fixed-width or non-repeating body cannot blow up — and
  are left alone.
- **A bounded outer quantifier.** `(a+){2,3}` repeats a bounded number of times, so
  it is not flagged.
- **A non-constant options value.** When the options argument is a runtime variable,
  the rule cannot prove `NonBacktracking` is absent and stays silent, to keep false
  positives near zero.

## Why there is no code fix

There are three different remedies — add a timeout, switch to the non-backtracking
engine, or rewrite the pattern — and they are not interchangeable. A timeout keeps
the backtracking semantics but caps the cost; `NonBacktracking` changes the engine
and rejects backreferences and lookaround; rewriting changes the pattern text. Which
is correct depends on what the pattern must express and how the caller handles a
timeout, and that is knowledge the analyzer does not have. The rule reports and
leaves the choice to the author.

## Proving the API exists

The rule resolves `System.Text.RegularExpressions.Regex`, its `RegexOptions` enum,
and `System.TimeSpan` in the analyzed compilation and registers nothing when any is
absent. The `[GeneratedRegex]` shape is registered only when
`System.Text.RegularExpressions.GeneratedRegexAttribute` (available on .NET 7 and
later) also resolves, so a project targeting an older framework never receives a
diagnostic pointing at an attribute it cannot use.

## Performance

The no-diagnostic path is syntactic. An object creation whose type name is not
`Regex`, an invocation whose member name is not one of the guarded static methods (or
that carries fewer than the input-plus-pattern arguments), and an attribute not
spelled `GeneratedRegex` are all rejected before the semantic model is touched.
Binding, the constant-value checks, and the pattern scan run only after those cheap
filters pass, and the pattern is walked once, character by character, without
allocating.
