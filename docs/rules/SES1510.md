# SES1510 — An MVC controller must not redirect to a non-constant URL

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | WebHardening |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A type that derives from `Microsoft.AspNetCore.Mvc.ControllerBase` calls one of the
raw-URL redirect helpers — `Redirect`, `RedirectPermanent`, `RedirectPreserveMethod`,
or `RedirectPermanentPreserveMethod` — with a URL argument that is not a compile-time
constant.

```csharp
using Microsoft.AspNetCore.Mvc;

public class AccountController : ControllerBase
{
    // 👎 reported: 'returnUrl' can be any host, so the redirect is an open redirect
    public IActionResult LogIn(string returnUrl) => Redirect(returnUrl);

    // 👍 a local-only redirect rejects a non-local target at run time
    public IActionResult LogInSafe(string returnUrl) => LocalRedirect(returnUrl);

    // 👍 a hard-coded literal URL cannot be steered by an attacker
    public IActionResult Home() => Redirect("/home");
}
```

## Rule description

`ControllerBase.Redirect`, `RedirectPermanent`, `RedirectPreserveMethod`, and
`RedirectPermanentPreserveMethod` each copy their argument straight into the response's
`Location` header. When that argument is not a compile-time constant it can carry a
value an attacker supplied — a query-string `returnUrl`, a form field, a header — and
the browser follows it to any host, including an external one.

That is an **open redirect** (CWE-601). A link that appears to point at the trusted
site — because the trusted domain leads the URL — lands the victim on an
attacker-controlled page that harvests credentials or serves malware. The trusted
domain lends the attack credibility, which is exactly why open redirects are a
staple of phishing campaigns.

The rule fires only when:

- **The invoked method is one of the four raw-URL redirect helpers.** `RedirectToAction`,
  `RedirectToRoute`, and `RedirectToPage` take an action, route, or page *name* — not a
  URL — and are never reported. The `LocalRedirect` family is already local-only and is
  never reported.
- **The method is bound to `ControllerBase` (or a subclass).** The invocation is bound
  and its containing type matched by symbol, so a same-named `Redirect` method on an
  unrelated type is ignored.
- **The URL argument is non-constant.** A hard-coded literal URL (or a `const`) cannot be
  steered by an attacker, so it is out of scope and not reported.

### Detection heuristic

Detection is **local** to the single call: the rule reports whenever a non-constant
value reaches one of the guarded redirect helpers, without tracing where that value
came from. It does not attempt whole-program taint tracking from an HTTP-request source
to the redirect sink, so it fires on the visible shape and stays near zero false
positives — the trade-off is that a non-constant target which the surrounding code has
already validated is still reported. The whole rule is gated on
`Microsoft.AspNetCore.Mvc.ControllerBase` resolving, so a project without ASP.NET Core
MVC pays nothing.

## How to fix violations

Prefer `LocalRedirect` (and its `RedirectPermanent`/`PreserveMethod` variants), which
throw when the target is not a local URL, so an external destination can never be
followed:

```csharp
// before
public IActionResult LogIn(string returnUrl) => Redirect(returnUrl);

// after
public IActionResult LogIn(string returnUrl) => LocalRedirect(returnUrl);
```

If the redirect must be able to reach an external site, validate the target against an
allow-list of permitted destinations before redirecting, and redirect only when it
matches.

There is no automatic code fix: switching `Redirect` to `LocalRedirect` changes runtime
behaviour (a non-local URL that previously succeeded now throws), so the correct
remediation is a decision for the author.
