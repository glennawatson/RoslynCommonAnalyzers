# SES1511 — The forwarded-headers trust boundary must not be removed

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

The forwarded-headers middleware rewrites a request's remote IP address, host,
and scheme from the `X-Forwarded-For`, `X-Forwarded-Host`, and
`X-Forwarded-Proto` headers so that an app behind a reverse proxy sees the real
client instead of the proxy. Because *any* client can send those headers, the
middleware trusts them only from a known set of proxies — loopback by default —
so a request that did not pass through a proxy you operate keeps its real
connection values.

Removing that restriction makes the middleware apply forwarded headers from any
source. An attacker can then forge `X-Forwarded-For` to spoof the client IP that
rate-limiting, allow-lists, and audit logs depend on, or forge the host and
scheme to defeat redirect and same-origin checks.

The rule reports two local shapes on a
`Microsoft.AspNetCore.Builder.ForwardedHeadersOptions`.

## The cleared-list shape

Calling `.Clear()` on the trusted-proxy or trusted-network list empties it, so
the middleware trusts every source:

```csharp
using Microsoft.AspNetCore.Builder;

services.Configure<ForwardedHeadersOptions>(options =>
{
    options.ForwardedHeaders = ForwardedHeaders.All;
    options.KnownProxies.Clear();      // reported (SES1511)
    options.KnownNetworks.Clear();     // reported (SES1511)
    options.KnownIPNetworks.Clear();   // reported (SES1511) — the newer trusted-network list
});
```

Both the legacy `KnownNetworks` list and its newer replacement
`KnownIPNetworks` are covered, because clearing either has the same effect.

## The removed-hop-limit shape

`ForwardLimit` caps how many forwarded hops are honoured; setting it to `null`
removes the cap. The rule reports that assignment written directly or in an
object initializer:

```csharp
options.ForwardLimit = null;                                    // reported (SES1511)
var o = new ForwardedHeadersOptions { ForwardLimit = null };    // reported (SES1511)
```

The accessed member is bound to its symbol and its containing type is confirmed
to be `ForwardedHeadersOptions`, so a same-named member on any other type is
ignored.

## What is not reported

Keeping the trust boundary in place is the fix, so populating the lists and
keeping a finite limit are clean:

```csharp
options.KnownProxies.Add(IPAddress.Parse("10.0.0.4"));   // trusted proxy added — not reported
options.ForwardLimit = 2;                                 // finite hop cap — not reported
```

Populate the trusted-proxy and trusted-network lists with only the reverse
proxies you operate, and keep a finite `ForwardLimit` matching how many proxies
a request legitimately traverses.

## Why there is no code fix

There is no safe automatic rewrite. The correct trusted proxies, trusted
networks, and hop count are deployment-specific — the analyzer cannot know which
addresses front your app or how many hops a request makes — and re-adding the
default loopback entry could still be wrong for a container or cloud deployment
where the proxy is on a different network. The author has to supply the real
values, so the rule reports and leaves the change to them.

## Proving the API exists

The rule is registered only when
`Microsoft.AspNetCore.Builder.ForwardedHeadersOptions` resolves in the analyzed
compilation. A project that does not reference the ASP.NET Core forwarded-headers
surface registers nothing and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. A `.Clear()` call whose receiver is not one
of the trusted-proxy/network list members, and an assignment whose right-hand
side is not the literal `null` or whose left-hand side is not spelled
`ForwardLimit`, are rejected before the semantic model is touched; only a
matching shape is bound to confirm the containing type.
