# SES1512 — Sensitive framework diagnostics must not be enabled outside development

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

Several framework switches deliberately widen diagnostic output to include values
that are normally withheld. They are invaluable while debugging locally and a
serious disclosure in production, where the extra detail lands in logs that many
people and systems can read (CWE-215, CWE-532):

- `DbContextOptionsBuilder.EnableSensitiveDataLogging()` writes the concrete
  parameter values bound into every SQL command into the logs — identifiers,
  search terms, and any secret passed as a query parameter.
- `IdentityModelEventSource.ShowPII = true` un-redacts personally identifiable
  information (names, subject ids, claims) in identity and token logs that are
  otherwise scrubbed.
- `IdentityModelEventSource.LogCompleteSecurityArtifact = true` logs the full
  security token, including its signature and embedded claims — enough to replay
  it.

The rule reports the enabling call or assignment:

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.IdentityModel.Logging;

// all reported (SES1512):
optionsBuilder.EnableSensitiveDataLogging();                       // EF parameter values
IdentityModelEventSource.ShowPII = true;                           // identity PII
IdentityModelEventSource.LogCompleteSecurityArtifact = true;       // full tokens
```

Leave these switches off in production. Enable them only under a development guard,
or remove them before shipping.

## The development-environment guard

A call or assignment that is lexically enclosed by an `if` statement or a
conditional expression whose condition calls a method named `IsDevelopment` is
treated as a development-only opt-in and is **not** reported:

```csharp
if (env.IsDevelopment())
{
    optionsBuilder.EnableSensitiveDataLogging();   // not reported
}

if (builder.Environment.IsDevelopment())
{
    IdentityModelEventSource.ShowPII = true;       // not reported
}
```

This is a purely local, syntactic check of the enclosing statements: the guard
call must appear in the condition of an ancestor `if`/conditional. No data-flow or
interprocedural analysis is performed, so an environment flag copied into a local
variable first, or a guard hidden behind a helper method, is not recognized — keep
the guard directly around the switch.

An inline flag passed to `EnableSensitiveDataLogging` is also treated as the
developer's own gate and is not reported, because the value decides at runtime
whether the switch turns on:

```csharp
optionsBuilder.EnableSensitiveDataLogging(env.IsDevelopment());   // not reported
optionsBuilder.EnableSensitiveDataLogging(false);                 // not reported — disabled
```

## What is reported

- **EF Core:** a call to `EnableSensitiveDataLogging` whose containing type is
  `Microsoft.EntityFrameworkCore.DbContextOptionsBuilder` or the generic
  `DbContextOptionsBuilder<TContext>`, made with no argument or the compile-time
  constant `true`.
- **Identity:** an assignment of the literal `true` to the static `ShowPII` or
  `LogCompleteSecurityArtifact` property of
  `Microsoft.IdentityModel.Logging.IdentityModelEventSource`.

Every member is bound to its symbol, so a same-named method or property on any
other type is ignored, and the assignment is reported only when no enclosing
`if`/conditional condition calls `IsDevelopment`.

## Why there is no code fix

There is no safe automatic rewrite. Removing the switch (or flipping it off) can
break a developer's intentional local diagnostics, and wrapping it in an
`if (env.IsDevelopment())` guard requires an environment accessor the analyzer
cannot conjure or reliably locate. The author has to decide whether the switch
belongs behind a development guard or should be dropped entirely, so the rule
reports and leaves the change to them.

## Proving the API exists

The rule resolves the EF Core option-builder types and the identity event source
in the analyzed compilation and registers each check only when its type is
present, so a project without Entity Framework Core, or without the identity
logging package, receives no diagnostic and pays no analysis cost for the
framework it does not use.

## Performance

The no-diagnostic path is syntactic. An invocation whose callee is not spelled
`EnableSensitiveDataLogging`, or an assignment whose right-hand side is not the
literal `true` or whose left-hand side is not one of the two guarded property
names, is rejected before the semantic model is touched; only a matching shape is
bound to confirm the framework type and check for an enclosing development guard.
