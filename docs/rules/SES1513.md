# SES1513 — An authorization result must be checked, not discarded

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | WebHardening |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

`IAuthorizationService.AuthorizeAsync` is called and its `AuthorizationResult` is
thrown away — the call forms a bare (optionally `await`ed) statement, or its result is
assigned to a discard (`_ = …`) — so nothing reads `Succeeded` and the guarded code
runs whether or not authorization passed.

```csharp
using Microsoft.AspNetCore.Authorization;
using System.Security.Claims;
using System.Threading.Tasks;

public class DocumentHandler
{
    private readonly IAuthorizationService _authz;

    public DocumentHandler(IAuthorizationService authz) => _authz = authz;

    // 👎 reported: the result is discarded, so Delete runs for everyone
    public async Task DeleteAsync(ClaimsPrincipal user, Document document)
    {
        await _authz.AuthorizeAsync(user, document, "CanDelete");
        Delete(document);
    }

    // 👍 branch on the outcome before doing the guarded work
    public async Task DeleteAsync(ClaimsPrincipal user, Document document)
    {
        var result = await _authz.AuthorizeAsync(user, document, "CanDelete");
        if (!result.Succeeded)
        {
            throw new UnauthorizedAccessException();
        }

        Delete(document);
    }
}
```

## Rule description

`AuthorizeAsync` does **not** throw when authorization is denied. It answers the
question *is this allowed?* by returning an `AuthorizationResult` whose `Succeeded`
property is `true` or `false`. A call whose result is discarded asks the question and
ignores the answer: the method returns, control falls through to whatever the check
was meant to protect, and the operation proceeds for an unauthorized principal exactly
as it would for an authorized one. The authorization check is present in the source but
has no effect at runtime — a missing-authorization defect (CWE-863) that reviewers
routinely miss because the call *looks* like a guard.

The rule reports an `AuthorizeAsync` invocation only when both of these hold:

- **The call binds to the authorization service.** The invoked method is
  `AuthorizeAsync` whose containing type is
  `Microsoft.AspNetCore.Authorization.IAuthorizationService`, or one of the convenience
  extension overloads declared on that interface (for example
  `AuthorizeAsync(user, policyName)`). An unrelated `AuthorizeAsync` on some other type
  is left alone.
- **The produced value is discarded.** The call — optionally `await`ed, optionally
  wrapped in `.ConfigureAwait(…)` — is used as an expression statement, or it is
  assigned to a discard (`_ = …`). A result stored in a variable, returned, passed as
  an argument, or read (for example `.Succeeded`) is a legitimate use and is not
  reported.

### Detection heuristic

Detection is purely **local** to the single statement: the rule matches the call and
its immediate discard context and never traces the `AuthorizationResult` across method
calls (that would need whole-program dataflow). Because a discarded result is the only
shape reported, a call that keeps its result — the overwhelmingly common case — is
never flagged, keeping false positives near zero. The rule resolves
`IAuthorizationService` once per compilation and registers nothing when the type is
absent, so a project without ASP.NET Core authorization pays nothing.

## How to fix violations

Capture the `AuthorizationResult` and branch on `Succeeded`. Deny the request when it
is `false` — return a forbidden or challenge result from a controller, or throw — and
only continue when it is `true`.

```csharp
// before
await _authz.AuthorizeAsync(user, document, "CanDelete");
Delete(document);

// after
var result = await _authz.AuthorizeAsync(user, document, "CanDelete");
if (!result.Succeeded)
{
    return Forbid();
}

Delete(document);
```

In an ASP.NET Core controller, `AuthorizationResult.Succeeded == false` maps naturally
to `Forbid()` (the principal is authenticated but not permitted) or `Challenge()` (no
identity yet); outside MVC, throw so the operation cannot proceed unauthorized.
