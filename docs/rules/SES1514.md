# SES1514 â€” OpenID Connect PKCE, state, and nonce protections must not be disabled

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None â€” see [Why there is no code fix](#why-there-is-no-code-fix)

The OpenID Connect authorization-code flow defends the sign-in with three
protocol protections, each on by default. Turning one off silently downgrades the
flow and reopens an attack the protocol was designed to close:

- **PKCE** â€” `OpenIdConnectOptions.UsePkce` (default `true`) binds the code
  exchange to the client that began the flow, so an authorization code stolen in
  transit cannot be redeemed by anyone else.
- **State** â€” `OpenIdConnectProtocolValidator.RequireState` and
  `RequireStateValidation` (both default `true`) tie the provider's redirect back
  to the request the browser actually started, which is what stops a cross-site
  request forgery login (CWE-352).
- **Nonce** â€” `OpenIdConnectProtocolValidator.RequireNonce` (default `true`) binds
  the returned id token to this one authentication, so a previously captured token
  cannot be replayed (CWE-294).

The validator is reached through `OpenIdConnectOptions.ProtocolValidator`. The
rule reports the `= false` assignment on any of the four flags, written directly
or in an object initializer:

```csharp
using Microsoft.AspNetCore.Authentication.OpenIdConnect;

// all reported (SES1514):
options.UsePkce = false;                                     // disables PKCE
options.ProtocolValidator.RequireState = false;              // disables the state check
options.ProtocolValidator.RequireStateValidation = false;    // disables state validation
options.ProtocolValidator.RequireNonce = false;              // disables the nonce check
var v = new OpenIdConnectProtocolValidator { RequireNonce = false };  // object initializer
```

Leave every flag at its secure default (or set it explicitly to `true`):

```csharp
options.UsePkce = true;                              // the default
options.ProtocolValidator.RequireState = true;       // the default
options.ProtocolValidator.RequireStateValidation = true;
options.ProtocolValidator.RequireNonce = true;       // the default
```

## What is reported

The assignment is flagged when both hold:

- the assigned member is one of the four flags bound to its owning type â€” `UsePkce`
  on `Microsoft.AspNetCore.Authentication.OpenIdConnect.OpenIdConnectOptions`, or
  `RequireState` / `RequireStateValidation` / `RequireNonce` on
  `Microsoft.IdentityModel.Protocols.OpenIdConnect.OpenIdConnectProtocolValidator`
  (the symbol is bound, so a same-named property on any other type is ignored); and
- the right-hand side is the literal `false`.

## Scope: the protocol flow, not token validation

This rule covers only the PKCE, state, and nonce protections of the OpenID
Connect authorization-code flow. The signature, issuer, audience, and lifetime
flags on `TokenValidationParameters` decide whether an already-issued token is
genuine â€” a separate concern â€” and are not reported here.

## Why there is no code fix

Flipping the flag back to `true` (or deleting the assignment to fall back to the
secure default) is usually correct, but the author disabled it for a reason â€”
often to work around a provider or test setup â€” and restoring the protection may
need matching configuration on both ends. The rule reports and leaves the change
to the author.

## Proving the API exists

The rule resolves `OpenIdConnectOptions` in the analyzed compilation and registers
nothing when it is absent; the state and nonce checks additionally require
`OpenIdConnectProtocolValidator` to resolve. A project that does not use OpenID
Connect authentication receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An assignment whose right-hand side is not
the literal `false`, or whose left-hand side is not spelled `UsePkce`,
`RequireState`, `RequireStateValidation`, or `RequireNonce`, is rejected before
the semantic model is touched; only a matching shape is bound to confirm the
containing type.
