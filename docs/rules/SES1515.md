# SES1515 — A Content-Security-Policy value disables its own protection

**Category:** WebHardening
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A `Content-Security-Policy` (CSP) response header tells the browser where scripts,
styles, and other resources may load from. Its single most valuable job is
**blocking injected inline scripts** — the payload of most reflected and stored
cross-site-scripting (XSS) attacks. Three source values undo exactly that:

- **`'unsafe-inline'`** re-permits inline `<script>` blocks and inline event
  handlers — the thing an injected `<script>` needs to run.
- **`'unsafe-eval'`** re-permits `eval` and other string-to-code evaluation.
- **a bare `*` source** lets script (or any resource) load from any origin.

Any of the three turns the policy into security theatre: the header is present, a
scan sees a CSP, but it no longer stops the attack it exists to stop.

```csharp
// reported (SES1515):
headers.Add("Content-Security-Policy", "default-src 'self'; script-src 'unsafe-inline'");
headers["Content-Security-Policy"] = "script-src 'self' 'unsafe-eval'";
headers.Append("Content-Security-Policy", "default-src *");

// reported — a self-evident policy value even without a header call:
var policy = "default-src 'self'; script-src 'unsafe-inline'";
```

Remove the unsafe source. For the inline scripts you genuinely need, allow them by
a per-response **nonce** or a **content hash** instead of `'unsafe-inline'`:

```csharp
// clean — inline scripts are allowed by nonce, not by 'unsafe-inline':
headers.Add("Content-Security-Policy", $"default-src 'self'; script-src 'self' 'nonce-{nonce}'");

// clean — a locked-down policy:
headers["Content-Security-Policy"] = "default-src 'self'; object-src 'none'; base-uri 'self'";
```

## What is reported

A string literal is reported when **both** of the following hold:

1. It is a CSP value — it contains a policy directive token: `default-src`,
   `script-src`, `style-src`, `object-src`, or `base-uri`; **and** it contains a
   permissive source: `'unsafe-inline'`, `'unsafe-eval'`, or a bare `*`.
2. It is used as a Content-Security-Policy value — **either**:
   - it is the value argument of a two-argument header-set call whose name
     argument is `Content-Security-Policy` (compared case-insensitively), such as
     `Headers.Add("Content-Security-Policy", value)` or
     `.Append("Content-Security-Policy", value)`; or the right side of a
     `Headers["Content-Security-Policy"] = value` indexer assignment; **or**
   - the literal itself **begins** with a directive token (a self-evident policy),
     so it is reported wherever it appears.

Detection is framework-agnostic: the rule matches on the string and the header
name, so it fires the same way regardless of which web stack sets the header.

## What is not reported

The rule is deliberately precise and stays silent on shapes that are not a
permissive policy:

- **Host wildcards.** A subdomain wildcard such as `*.example.com` or a
  scheme-relative wildcard such as `https://*` is a host source, not a bare `*`
  origin, so it is not flagged. Only a standalone `*` — bounded by whitespace, a
  `;`, or the ends of the string — counts as a wildcard source.

  ```csharp
  // clean — a host wildcard, not a bare '*':
  var policy = "default-src 'self' *.example.com";
  ```

- **A different header.** The same permissive value set on any header other than
  `Content-Security-Policy` (for example `Content-Security-Policy-Report-Only`, or
  a custom header) is not reported unless the value is self-evidently a policy.
- **A non-value position.** The literal in the name slot of a header call, an
  argument to a call with more than two arguments, a constructor argument, or a
  plain field/local that neither begins with a directive nor sets the header is
  not reported.
- **A directive mentioned in prose.** A string like
  `"note: script-src 'unsafe-inline' would be unsafe"` contains a directive
  mid-text but neither begins with one nor sets the header, so it is left alone.

## Interpolated and computed policies

The rule inspects string **literals**. A policy assembled by string interpolation
or concatenation (`$"default-src {sources}"`) is not analyzed, because its final
value is not known statically. Prefer building the policy from a literal template
whose unsafe parts are visible, or keep the static portion in a single literal so
this rule can see it.

## Why there is no code fix

There is no single safe rewrite. Dropping `'unsafe-inline'` is usually correct,
but a page that genuinely relies on inline scripts then needs those scripts moved
behind a nonce or hash — a change to both the policy and the markup that emits the
scripts. Replacing a `*` source requires knowing the real set of origins the app
loads from. The author has to make those calls, so the rule reports and leaves the
change to them.

## Performance

The no-diagnostic path is a cheap string scan. A literal is skipped immediately
unless it contains one of the directive tokens, so the vast majority of string
literals cost only that check. No semantic model is consulted: the header shape is
recognized syntactically, so the rule adds no binding cost.
