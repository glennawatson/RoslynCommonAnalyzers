# SES1602 — Do not route AI model output into a process, file, or raw SQL sink

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Ai |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Text read from an AI model response — the `Text` property of
`Microsoft.Extensions.AI.ChatResponse` or `Microsoft.Extensions.AI.ChatMessage` — flows,
inline or through a single immediately-preceding local, into a sink that executes or acts
on it: `Process.Start`, a `ProcessStartInfo.FileName`/`Arguments` assignment, a
`System.IO.File` path, or a raw-SQL `FromSqlRaw`/`ExecuteSqlRaw` command.

```csharp
using System.Diagnostics;
using System.IO;
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.AI;

public async Task Act(IChatClient client, DatabaseFacade database)
{
    ChatResponse response = await client.GetResponseAsync("...");

    // 👎 reported — the model chooses the program that runs
    Process.Start(response.Text);

    // 👎 reported — the model chooses the launched program (object initializer)
    var info = new ProcessStartInfo { FileName = response.Text };

    // 👎 reported — the model chooses a file-system path
    File.ReadAllText(response.Text);

    // 👎 reported — the model builds the executed SQL
    database.ExecuteSqlRaw(response.Text);

    // 👎 reported — one immediately-preceding local still holds the model text
    var command = response.Text;
    Process.Start(command);

    // 👍 not reported — the model result is treated as data, not as the command
    Log(response.Text);
}
```

## Rule description

A language model's response is attacker-influenced data. A prompt-injection payload hidden
in a retrieved document, a tool result, or an earlier user turn can steer the model into
emitting any string the attacker chooses. When that string is used verbatim to launch a
process, name a program or its arguments, build a file-system path, or form a raw SQL
command, the model — and whoever steered it — decides what code, file, or query the host
executes. That turns prompt injection into remote code execution, arbitrary file access, or
SQL injection.

The **source** is proven, never guessed: the analyzer binds a `.Text` member access to the
`Text` property of the gated `ChatResponse` or `ChatMessage` type (so `ChatResponse<T>`,
which inherits the property, matches too). The **sink** is proven by binding its symbol and
containing type. The model text is reported where it reaches one of these local shapes:

- the `fileName` or `arguments` argument of `System.Diagnostics.Process.Start`;
- a `System.Diagnostics.ProcessStartInfo.FileName` or `.Arguments` assignment, including an
  object initializer;
- the leading path argument of a curated, high-signal set of `System.IO.File` members
  (`ReadAllText`, `WriteAllText`, `Open`, `Create`, `Delete`, `Copy`, `Move`, and their
  siblings);
- the `sql` argument of Entity Framework Core's `FromSqlRaw`, `ExecuteSqlRaw`, or
  `ExecuteSqlRawAsync`.

### What is not reported

- **Model text used as data.** Logging, returning, displaying, or storing the response text
  is not a sink and is never reported. Only the process, file, and raw-SQL sinks above are.
- **A non-model value at the sink.** A parameter, field, or unrelated local flowing into the
  same sink is not reported — the source must bind to the AI response `Text` property.
- **A look-alike `Text`.** A `Text` property on any other type is ignored; the containing
  type must be the gated `ChatResponse`/`ChatMessage`.
- **A same-named sink on another type.** A `Start`, `Delete`, `ExecuteSqlRaw`, or `FileName`
  member on an unrelated type is ignored; the sink is confirmed by binding.
- **The content argument of a file write.** Only the path argument is a sink, so writing
  model text as the *contents* of a fixed file (`File.WriteAllText("out.txt", response.Text)`)
  is not reported.

Detection is strictly local — there is no data-flow or interprocedural tracking. A local is
followed exactly one hop, and only when its declaration is the statement immediately before
the sink in the same block, so the local provably still holds the model text at the sink. A
value carried across other statements, reassigned, or passed through a method is outside the
rule.

### How to fix violations

Treat model output as untrusted data and never let it choose what runs.

- Do not pass model text as a program name, command line, or file path. Constrain the model
  to a fixed, vetted set of operations selected by its output, and validate any dynamic value
  against an allow-list before use.
- For processes, pass fixed executables and add each argument to `ProcessStartInfo.ArgumentList`
  (which escapes per platform) rather than to `FileName`/`Arguments`.
- For databases, use parameterized queries — `FromSql`/`ExecuteSql` interpolated forms or
  explicit SQL parameters — so the model value is bound as data, never concatenated into the
  command text.

### Availability detection

The rule probes `Microsoft.Extensions.AI.ChatResponse` and `Microsoft.Extensions.AI.ChatMessage`
once per compilation via `Compilation.GetTypeByMetadataName`. A project that does not reference
`Microsoft.Extensions.AI` registers nothing and pays nothing. Each sink family is registered
only when its type is present in the compilation, and no target-framework string is parsed.

## Performance

The clean path is a syntactic screen. An invocation is bound only when it is a member call
whose name could name a resolved sink family (`Start`, a curated `File` member, or a raw-SQL
method) and carries an argument; an assignment is bound only when its target is named
`FileName` or `Arguments` and its value is model output. Every other invocation and assignment
is rejected with no semantic query, and the single-local hop is evaluated only after the sink
and source are already established.
