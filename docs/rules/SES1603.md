# SES1603 ‚Äî An AI tool declared read-only or non-destructive must not call a state-changing API

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Ai |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method exposed to a model host as a tool declares itself read-only or
non-destructive ‚Äî its `[McpServerTool]` attribute sets `ReadOnly = true` or
`Destructive = false` ‚Äî yet its body calls an API that deletes data, overwrites a
file, starts a process, or writes to a database.

```csharp
using System.IO;
using ModelContextProtocol.Server;

// üëé reported: advertised read-only, but the body deletes a file
[McpServerTool(ReadOnly = true)]
public void Cleanup(string path) => File.Delete(path);

// üëç the same delete tool, honestly declared, so the host prompts before running it
[McpServerTool(Destructive = true)]
public void Cleanup(string path) => File.Delete(path);
```

## Rule description

A host that automates tool use reads a tool's annotations to decide whether it may
invoke the tool **without asking the user**. A tool the model believes only reads
state is a candidate for silent auto-invocation. When such a tool actually mutates
state, the model can trigger irreversible damage ‚Äî deleting files, overwriting data,
launching a process, or issuing a destructive database command ‚Äî entirely on its own,
because the safety hint told the host it was safe to run unattended.

The rule reports a call only when both of these hold, so it fires on the dishonest
shape and stays quiet on an honestly-declared tool:

- **The method is a tool with an explicit safety hint.** It carries `[McpServerTool]`
  (or a subclass) whose `ReadOnly` is set to `true` or whose `Destructive` is set to
  `false`. An **unset** hint is left alone: the attribute stores these hints in
  nullable backing fields, so leaving one unset means the tool made *no* promise and
  the host assumes it may be destructive ‚Äî there is nothing to contradict. An explicit
  `ReadOnly = false` or `Destructive = true` is likewise left alone.
- **The body calls a curated state-changing API.** One of `File.Delete`,
  `File.WriteAllText`, `Directory.Delete`, `Process.Start`,
  `DbCommand.ExecuteNonQuery`, an Entity Framework `ExecuteSqlRaw` /
  `ExecuteSqlInterpolated` / `ExecuteDelete` / `ExecuteUpdate`, or a
  `DbContext.SaveChanges` (each including its async twin).

The Semantic Kernel `[KernelFunction]` attribute is **not** covered: it carries no
read-only or destructive member, so a tool declared with it makes no safety promise
this rule could contradict.

### Detection heuristic

Detection is local to the one method ‚Äî the rule scans only the tool's own body and
never traces values across calls (that would need whole-program dataflow), so it fires
only when the state-changing call is visible in the method itself. Each destructive
sink is matched against the types actually present in the compilation; a project
without a given API (or without the model-tool SDK) registers nothing and is never
reported. The sink list is deliberately narrow to keep false positives near zero.

## How to fix violations

Make the tool's behaviour match its advertised hint ‚Äî move the destructive work out of
a tool that claims to be read-only ‚Äî or drop the read-only/non-destructive claim
(`ReadOnly = true` / `Destructive = false`) so the host prompts the user before running
the tool.

```csharp
// before: claims read-only while mutating
[McpServerTool(ReadOnly = true)]
public void Cleanup(string path) => File.Delete(path);

// after: the destructive nature is declared, so the host will not silently auto-run it
[McpServerTool(Destructive = true)]
public void Cleanup(string path) => File.Delete(path);
```

If the method genuinely only reads state, the call is the bug: replace the
state-changing API with a read-only equivalent so the tool matches its hint.
