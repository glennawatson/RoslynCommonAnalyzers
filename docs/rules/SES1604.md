# SES1604 — Prompt-template input encoding must not be disabled

**Category:** Ai
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A Semantic Kernel prompt template treats every substituted input variable and
function return value as untrusted. Before a value reaches the model it is
HTML/expression-encoded, so text a caller supplied cannot close the current
`<message>` tag or open a new one and take over the conversation. This is the
default, and it is what stops prompt injection through a template variable.

Four members expose an opt-out named `AllowDangerouslySetContent`, and each one
turns that encoding off:

- `Microsoft.SemanticKernel.PromptTemplateConfig` — trusts every value the
  template inserts;
- `Microsoft.SemanticKernel.InputVariable` — trusts one named variable;
- `Microsoft.SemanticKernel.KernelPromptTemplateFactory` and
  `Microsoft.SemanticKernel.PromptTemplates.Handlebars.HandlebarsPromptTemplateFactory`
  — trust everything the factory renders.

Set any of them to `true` and the raw value is inserted verbatim. An attacker who
controls that variable can inject `</message><message role="system">…` to rewrite
the system prompt, leak the surrounding context, or drive tools the user never
authorised. The default (`false`) keeps the value encoded and the injection
inert.

The rule reports the `= true` assignment, written directly or in an object
initializer:

```csharp
using Microsoft.SemanticKernel;

// all reported (SES1604):
config.AllowDangerouslySetContent = true;                                         // direct assignment
var a = new PromptTemplateConfig { AllowDangerouslySetContent = true };            // object initializer
var b = new InputVariable { Name = "input", AllowDangerouslySetContent = true };   // one variable
var c = new KernelPromptTemplateFactory { AllowDangerouslySetContent = true };     // whole factory
```

Leave the flag at its secure default so substituted content stays encoded:

```csharp
// the default — the value is encoded and cannot break out of its slot:
var config = new PromptTemplateConfig();

// only trust content that cannot carry attacker input (a fixed template you
// author, not a user-supplied variable):
var trustedTemplate = new PromptTemplateConfig { AllowDangerouslySetContent = false };
```

## What is reported

The assignment is flagged when both hold:

- the assigned member is `AllowDangerouslySetContent`, and its containing type is
  one of the four Semantic Kernel types above (the symbol is bound, so a
  same-named property on any other type is ignored); and
- the right-hand side is the literal `true`.

A non-literal value — `AllowDangerouslySetContent = isTrustedSource` — is left
alone: the developer has made the trust decision conditional, and the analyzer
cannot evaluate the condition.

## Why there is no code fix

There is no safe automatic rewrite. Deleting the assignment (or setting it back to
`false`) restores encoding, but that is only correct when the inserted content is
in fact untrusted — which is exactly the judgement the author opted out of. If the
value really is trusted, the correct fix is to narrow the trust to that one
`InputVariable` rather than removing it wholesale. The rule reports and leaves the
decision to the author.

## Proving the API exists

The rule resolves `Microsoft.SemanticKernel.PromptTemplateConfig` in the analyzed
compilation and registers nothing when it is absent, so a project that does not
reference Semantic Kernel receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An assignment whose right-hand side is not
the literal `true`, or whose left-hand side is not spelled
`AllowDangerouslySetContent`, is rejected before the semantic model is touched;
only a matching shape is bound to confirm the containing type.
