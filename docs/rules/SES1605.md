# SES1605 — AI instrumentation must not enable sensitive-data capture

| Item | Value |
| --- | --- |
| Category | Ai |
| Severity | Info |
| Code fix | No |
| Default | Enabled |

## Cause

The `EnableSensitiveData` property of a `Microsoft.Extensions.AI` OpenTelemetry
instrumentation client — the chat, embedding, realtime, speech-to-text,
text-to-speech, image, or hosted-file variant — is assigned the compile-time
constant `true`.

```csharp
using Microsoft.Extensions.AI;

IChatClient client = inner
    .AsBuilder()
    .UseOpenTelemetry(configure: o => o.EnableSensitiveData = true)
    .Build();
```

## Rule description

OpenTelemetry instrumentation for generative-AI clients emits operation metadata —
token counts, model names, durations — but leaves the raw inputs and outputs out of
telemetry by default. That default exists because a chat prompt or a model response
routinely carries credentials, access tokens, personal data, and other secrets.

Setting `EnableSensitiveData` to `true` turns that raw content back on. Every prompt
and completion, and every function-call argument and result, is then written
verbatim to whatever backend receives the traces — application logs, an APM service,
a third-party collector. The sensitive content now lives in a second store with its
own access model and retention, which widens the blast radius of a breach and often
crosses a data-residency or compliance boundary that the primary datastore was
designed to respect.

```csharp
// Reported: a plain assignment turns raw capture on.
otel.EnableSensitiveData = true;

// Reported: an object-initializer member does the same.
var otel = new OpenTelemetryChatClient(inner) { EnableSensitiveData = true };
```

The rule matches the property by its bound symbol, so it fires on every listed
instrumentation type — including the generic embedding generator — and on the plain
assignment (`x.EnableSensitiveData = true`), the object-initializer member
(`{ EnableSensitiveData = true }`), and the configure-delegate shape
(`o => o.EnableSensitiveData = true`) alike.

## What is not reported

- **`EnableSensitiveData = false`.** Explicitly turning capture off is the safe
  choice and is left alone.

- **A runtime-computed value.** When the right-hand side is a method parameter, a
  field, or a value read from configuration, its value cannot be judged from the
  source, so the rule stays silent rather than guess. Gating the switch behind a
  development-only configuration flag is by design a supported way to keep the rule
  quiet.

  ```csharp
  // Not reported: the value is unknown at compile time.
  otel.EnableSensitiveData = builder.Environment.IsDevelopment();
  ```

- **An `EnableSensitiveData` member on any other type.** The property is matched by
  its bound symbol, not by name, so a same-named property on an unrelated type is
  left alone.

## How to fix violations

Leave `EnableSensitiveData` at its default, so telemetry carries only metadata and
never raw prompts or responses. If you genuinely need to inspect raw content while
debugging, enable it only in a controlled, non-production environment — gate the
assignment behind an environment or configuration check so the value is a
deployment decision rather than a hard-coded `true`. Because the value is then no
longer a compile-time constant, the rule treats it as unknown and stays silent.

There is deliberately **no code fix**: whether to capture raw content is a
privacy and threat-model decision, not a mechanical substitution.

## Why this is Info

Capturing raw content is a deliberate diagnostics choice that is defensible while
debugging in a controlled environment, so the finding is reported at `Info`. It is a
prompt to confirm the switch is intentional and scoped to development — not a
declaration that the code is broken.

## Proving the API exists

The rule resolves the `Microsoft.Extensions.AI` OpenTelemetry instrumentation types
in the analyzed compilation and registers nothing when none are present, so a
project without `Microsoft.Extensions.AI` receives no diagnostic and pays no
analysis cost.

## Performance

Each assignment is filtered syntactically first — only a left-hand side that names
`EnableSensitiveData` and a right-hand side that is the constant `true` reach the
semantic model — so the common no-diagnostic path stays allocation-free. The
containing type is bound only after that shape matches.
