# SES1606 — Do not fetch model weights over cleartext HTTP

**Category:** Ai
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A model-weights file fetched from an `http://` URL travels the network
unencrypted, so anyone on the path — a shared Wi-Fi hop, a compromised router, a
middlebox — can replace the file in transit with a tampered or backdoored model.
Weights fully determine what a model does at inference: a swapped file can leak the
prompts and data it is given, emit attacker-chosen output, or carry an embedded
deserialization payload that runs when the file is loaded. Nothing at the call site
reveals the substitution — the download simply succeeds with different bytes.

The rule reports the hard-coded URL **literal** wherever it appears, because a
weights extension combined with the `http` scheme is on its own a high-confidence
signal:

```csharp
using System;

// all reported (SES1606):
const string ModelUrl = "http://models.example.com/resnet50.onnx"; // constant
static readonly string Weights = "http://cdn.example.com/llama-7b.gguf"; // field
LoadModel("http://ml.example.com/model.safetensors");              // any loader argument
var uri = new Uri("http://weights.example.com/checkpoint.ckpt");    // new Uri
```

Fetch the weights over `https://` and verify the download against a hash or
signature published by the model's author before loading it:

```csharp
const string ModelUrl = "https://models.example.com/resnet50.onnx";
// after download: compare SHA-256 to the author's published digest, then load.
```

Transport encryption alone proves only that the file arrived unaltered from the
server; it does not prove the server served the model you trust, which is why a
content hash or signature check is the second half of the fix.

## What is reported

A string literal is flagged when **all** of these hold, parsed directly from the
literal text:

- the text begins with `http://` (matched case-insensitively);
- the authority is a non-loopback host;
- the path (ignoring any `?query` or `#fragment`) ends in a model-weights
  extension: `.onnx`, `.gguf`, `.safetensors`, `.pt`, `.pth`, or `.ckpt`.

The literal may be an ordinary, verbatim (`@"…"`), or raw (`"""…"""`) string. The
extension is matched case-insensitively and only at the end of the path, so a
weights extension that appears mid-path is not a match.

```csharp
// reported — path ends in a weights extension, query is ignored:
const string A = "http://models.example.com/resnet50.onnx?token=abc";

// not reported — the extension is not at the end of the path:
const string B = "http://models.example.com/model.onnx/download";
```

Loopback hosts are treated as clean, because cleartext there stays on the machine
and is normal in local development:

- `localhost` and any `*.localhost` host;
- `127.0.0.1`;
- the IPv6 loopback `[::1]`.

```csharp
const string Local = "http://localhost/model.onnx";        // not reported
const string Loop  = "http://127.0.0.1:5000/model.gguf";   // not reported
```

## What is not reported

The `.bin` extension is deliberately excluded. It is a common weights format in
practice, but it is also used for firmware, generic blobs, and unrelated binary
data, so flagging every `http://…/*.bin` URL would produce false positives. The
rule keeps its extension set to formats that are specific to model weights.

An `https://` URL is never reported — the transport is already encrypted. (Verify
the content hash or signature regardless; that is a supply-chain concern this rule
does not enforce.)

Only a literal is examined. A URL held in a variable, built by interpolation or
concatenation, returned by a method, or read from configuration is not flagged —
tracking it would require whole-program data-flow analysis, which this rule
deliberately avoids to stay fast and free of false positives.

```csharp
string url = "http://models.example.com/model.onnx";
LoadModel(url);                 // not reported — flows through a variable
LoadModel($"http://{host}/m.pt"); // not reported — interpolated, not a literal
```

## Relationship to SES1106

SES1106 reports a cleartext `http://` literal only when it reaches an
`HttpClient` request sink (a request method's URL argument, or a `new Uri(...)`
assigned to `HttpClient.BaseAddress`). SES1606 targets the weights literal
independent of the consuming API, so it fires where SES1106 cannot — a constant, a
field, or an argument to an ML loader that never touches `HttpClient`.

To avoid two diagnostics on one span, SES1606 stays silent when the literal is
exactly the shape SES1106 already owns: the URL argument (directly, or wrapped in a
`new Uri(...)`) of an `HttpClient` request method, or a `new Uri(...)` assigned to
`HttpClient.BaseAddress`. Every other placement of a cleartext weights URL is
reported by SES1606.

```csharp
using System.Net.Http;

// deferred to SES1106 (HttpClient sink) — SES1606 stays silent:
await client.GetByteArrayAsync("http://models.example.com/traced.pt");

// SES1606 reports the declaration; the request below uses an identifier, not a literal:
const string ModelUrl = "http://models.example.com/model.onnx"; // reported (SES1606)
await client.GetByteArrayAsync(ModelUrl);
```

The deferral is `HttpClient`-specific: a same-named request method on an unrelated
type is still reported by SES1606, because SES1106 would not fire there.

## Why there is no code fix

There is no safe automatic rewrite. Flipping `http` to `https` only works when the
host actually serves TLS, and even then the fix is incomplete without a hash or
signature check that the analyzer cannot supply. Because the correct remediation
requires knowledge the analyzer does not have — the author's published digest and
whether the endpoint serves HTTPS — the rule reports and leaves the change to the
author.

## Proving the API exists

The rule suggests no API, so it needs no framework probe and fires on any target.
It resolves `System.Net.Http.HttpClient` only to recognise the SES1106 sink shapes
it defers; when that type is absent, no deferral is needed (SES1106 cannot fire
either) and the weights literal is still reported.

## Performance

The no-diagnostic path is syntactic. Every string literal is tested with a cheap
scan of its text — a scheme prefix check, a single walk to locate the path, and an
extension suffix comparison — with no `Uri` allocation. The semantic model is
touched only after a cleartext, non-loopback weights literal has already been
found, and only to decide whether the SES1106 sink deferral applies.
