# SES1701 — Raw HTML is rendered from a non-constant value

**Category:** Blazor
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

Blazor HTML-encodes every value it renders, and that encoding is exactly what stops
an attacker-supplied string from being interpreted as markup. Three sinks opt out of
it and write their value to the page as **raw HTML**:

- **`new MarkupString(value)`** — constructing the raw-markup wrapper.
- **`(MarkupString)value`** — the explicit string-to-`MarkupString` conversion.
- **`RenderTreeBuilder.AddMarkupContent(sequence, value)`** — appending raw markup
  to the render tree.

When `value` is a compile-time constant it is developer-authored markup and is safe.
When `value` is **non-constant** it can carry request data, a database field, or any
other untrusted input — and a `<script>` element or an event-handler attribute inside
it will execute in the victim's session. That is reflected or stored cross-site
scripting (XSS).

```csharp
using Microsoft.AspNetCore.Components;

public sealed class Renderer
{
    // 👎 reported: 'comment' is untrusted and is emitted as live markup with no encoding
    public MarkupString Show(string comment) => new MarkupString(comment);

    // 👎 reported: the explicit conversion is the same raw-HTML sink
    public MarkupString ShowCast(string comment) => (MarkupString)comment;

    // 👍 a compile-time constant is developer-authored markup, not attacker input
    public MarkupString Divider() => new MarkupString("<hr />");
}
```

## What is reported

The value written to any of the three sinks is reported when it is **not** a
compile-time constant. The sink is confirmed by binding: `new MarkupString(...)` and
`(MarkupString)...` must resolve to `Microsoft.AspNetCore.Components.MarkupString`,
and `AddMarkupContent` must resolve to
`Microsoft.AspNetCore.Components.Rendering.RenderTreeBuilder`, so a same-named type or
method on an unrelated type is never flagged.

## What is not reported

- **A constant value.** A string literal, a `const` field, or any other compile-time
  constant is developer-authored markup and is left alone.
- **A sanitized value.** A value wrapped in a call to a method named in the
  [`securitysharp.SES1701.sanitizers`](#configuration) allow-list is treated as
  already-encoded and is not reported.
- **A same-named sink on an unrelated type.** A `MarkupString` type or an
  `AddMarkupContent` method that is not the Blazor one is not a raw-HTML sink.

```csharp
// 👍 clean — the value is produced by an allow-listed sanitizer
//     (securitysharp.SES1701.sanitizers = Sanitize):
public MarkupString Show(string comment) => new MarkupString(Sanitize(comment));
```

## Configuration

```ini
# Rule-specific
securitysharp.SES1701.sanitizers = Sanitize, Encode

# Project-wide (applies to any SecuritySharp rule that reads a sanitizer allow-list)
securitysharp.sanitizers = Sanitize, Encode
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `sanitizers` | comma-separated method names | *(empty)* | A value produced by a call to any listed method is treated as already-encoded and is not reported. The match is on the method's simple name (`Sanitize`, whether called as `Sanitize(x)` or `sanitizer.Sanitize(x)`). |

The rule-specific key wins over the project-wide key. With no allow-list configured,
every non-constant value reaching a raw-HTML sink is reported.

## How to fix violations

Prefer rendering the value as ordinary text so Blazor encodes it:

```razor
@* before — raw, unencoded *@
@((MarkupString)comment)

@* after — encoded by the framework *@
@comment
```

When the value must genuinely be markup, build the `MarkupString` only from markup you
control, or from the output of a trusted HTML sanitizer, and add that sanitizer's
method name to `securitysharp.SES1701.sanitizers` so the rule can see it is safe.

## Why there is no code fix

There is no single safe rewrite. Switching a `MarkupString` to plain text changes what
the page renders (markup the author may have intended stops being markup), and wrapping
the value in a sanitizer requires choosing and wiring up a sanitizer the project trusts.
Both are decisions for the author, so the rule reports and leaves the change to them.

## Performance

The no-diagnostic path is a cheap syntactic prefilter: an object creation or cast is
skipped immediately unless its type name is `MarkupString`, and an invocation unless its
member name is `AddMarkupContent`. The semantic model — binding the sink and reading the
sanitizer allow-list — is consulted only after that prefilter matches and only for a
non-constant value. The whole rule is gated on
`Microsoft.AspNetCore.Components.MarkupString` resolving, so a non-Blazor project
registers nothing and pays nothing.
