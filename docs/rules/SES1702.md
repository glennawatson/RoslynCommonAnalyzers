# SES1702 â€” A JavaScript interop call targets a script-evaluation primitive

**Category:** Blazor
**Severity:** Warning
**Code fix:** None â€” see [Why there is no code fix](#why-there-is-no-code-fix)

A Blazor JavaScript-interop call names the function to run by its identifier string:
`js.InvokeAsync<T>("someFunction", args)` or `js.InvokeVoidAsync("someFunction", args)`
on an `IJSRuntime` or `IJSObjectReference`. When that identifier is a JavaScript
primitive that **evaluates source**, every argument the call forwards becomes
executable script, so any untrusted value in those arguments runs in the browser. The
interop boundary turns into a script-injection channel.

The eval-class identifiers are:

- **`eval`** and **`Function`** â€” evaluate a string as code directly.
- **`document.write`** and **`document.writeln`** â€” write a string into the document,
  where an embedded `<script>` executes.
- **`setTimeout`** and **`setInterval`** â€” compile and run their first argument **only
  when it is a string body**; passed a real function reference they are safe.

```csharp
using Microsoft.JSInterop;

public sealed class Bridge(IJSRuntime js)
{
    // đź‘Ž reported: 'code' is evaluated as script
    public ValueTask Run(string code) => js.InvokeVoidAsync("eval", code);

    // đź‘Ž reported: a string body handed to setTimeout is compiled and run
    public ValueTask Schedule(string body) => js.InvokeVoidAsync("setTimeout", body, 100);

    // đź‘Ť a named function does the work; the data is passed as an ordinary argument
    public ValueTask RunSafely(string data) => js.InvokeVoidAsync("app.render", data);
}
```

## What is reported

The identifier argument is reported when **both** hold:

1. The identifier is a **compile-time constant string** equal to `eval`, `Function`,
   `document.write`, or `document.writeln`; **or** it is `setTimeout` / `setInterval`
   **and** the argument that follows it is typed as `string` (a script body).
2. The call **binds** to an `InvokeAsync` or `InvokeVoidAsync` method whose receiver is,
   or implements, `Microsoft.JSInterop.IJSRuntime` or
   `Microsoft.JSInterop.IJSObjectReference`.

Binding the call means a same-named `InvokeAsync`/`InvokeVoidAsync` on an unrelated type
is never flagged.

## What is not reported

- **A non-constant identifier.** An identifier computed at run time cannot be judged
  from the source, so it is left alone (there is nothing actionable to point at).
- **A benign function name.** Any identifier that is not an eval-class primitive is a
  normal named function and is not reported.
- **`setTimeout` / `setInterval` with a function reference.** When the argument after
  the identifier is not a string body, the call schedules a real function and is safe.

```csharp
// đź‘Ť clean â€” a non-constant identifier is not judged here
public ValueTask Run(string fn, string data) => js.InvokeVoidAsync(fn, data);

// đź‘Ť clean â€” setTimeout is given a function reference, not a string body
public ValueTask Schedule(object handler) => js.InvokeVoidAsync("setTimeout", handler, 100);
```

## How to fix violations

Expose a named JavaScript function that performs the specific work, and invoke **that**
by name, passing the data as ordinary arguments so nothing is evaluated as code:

```javascript
// wwwroot/app.js
window.app = {
  render: (data) => { /* build DOM from data, no eval */ }
};
```

```csharp
// before â€” the argument is evaluated as script
await js.InvokeVoidAsync("eval", "render(" + data + ")");

// after â€” a named function receives the data as data
await js.InvokeVoidAsync("app.render", data);
```

For `document.write` / `document.writeln`, set text through a safe DOM API (for example
assigning `textContent`) inside your named function instead of writing raw markup. For
`setTimeout` / `setInterval`, pass a function reference rather than a string body.

## Why there is no code fix

The safe rewrite is a new named JavaScript function plus a changed call â€” the analyzer
cannot invent the function or know what the evaluated string was meant to do. That is a
decision for the author, so the rule reports and leaves the change to them.

## Performance

The no-diagnostic path is a cheap syntactic prefilter: an invocation is skipped unless
its member name is `InvokeAsync` or `InvokeVoidAsync`. The identifier's constant value is
read only for those calls, the following argument's type only for `setTimeout` /
`setInterval`, and the call is bound only once the identifier is confirmed eval-class. The
whole rule is gated on `Microsoft.JSInterop.IJSRuntime` resolving, so a non-Blazor project
registers nothing and pays nothing.
