# SES1703 — Authorize on a non-routable component enforces nothing

**Category:** Blazor
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

In Blazor, authorization is evaluated as a **routing** concern: the router reads a
routable page's `[Authorize]` when it resolves the route, and only then renders the
page. A component that carries `[Authorize]` but is **not routable** — it has no
`[Route]` (the `@page` directive) — is never reached through routing, so its
`[Authorize]` is never evaluated. The component renders whenever a parent decides to
render it, regardless of the policy. The `[Authorize]` reads as protection but
enforces nothing, so an author who added it believing the component is locked down is
mistaken.

The rule reports the misleading `[Authorize]` attribute:

```csharp
using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.Components;

[Authorize]        // reported (SES1703): this component is not routable, so nothing enforces it
public class AccountPanel : ComponentBase
{
}
```

Put the authorization where it is actually enforced — on the routable page that hosts
the component, or in an `<AuthorizeView>` around the sensitive content — and drop the
`[Authorize]` from the child component:

```csharp
// the routable page carries the policy the router enforces
[Authorize]
[Route("/account")]
public class AccountPage : ComponentBase
{
    // AccountPanel renders inside AccountPage, which the router has already authorized
}
```

## What is reported

A type that derives from `Microsoft.AspNetCore.Components.ComponentBase`, carries
`[Authorize]`, and carries no `[Route]`:

- the marker written as `[Authorize]`, `[AuthorizeAttribute]`, or fully qualified —
  each attribute is **bound** to its symbol and matched by attribute class, never by
  the written name;
- a subclass of `AuthorizeAttribute`, because a derived authorization attribute is
  honoured by the framework exactly as its base is.

The `[Authorize]` applied through the Razor `@attribute [Authorize]` directive is a
class-level attribute on the generated component and is reported the same way.

## What is not reported

- **A routable component.** A type that also carries `[Route]` (`@page`) is a page the
  router authorizes, which is the correct pattern.
- **A non-component type.** `[Authorize]` on a controller, a Razor Pages model, or any
  type that does not derive from `ComponentBase` is out of scope — those are enforced
  by their own pipelines.
- **Abstract components.** A base component that is `abstract` is not the rendered
  leaf; the concrete component that derives from it is where a route (or its absence)
  is decided.
- **Layout components.** A type deriving from `LayoutComponentBase` is a layout, not a
  routed page, and is exempt.

The rule is purely local and symbol-bound: it inspects the attributes written on the
declaration itself and never follows base types or partial parts in other files.

## Configuration

Additional component type names can be exempted with a comma-separated list, either
per rule or project-wide:

```ini
[*.cs]
# exempt specific components (simple name or fully qualified name)
securitysharp.SES1703.exempt_types = MyBaseComponent, App.Components.TrustedPanel

# or exempt them for every SecuritySharp rule that reads an exempt list
securitysharp.exempt_types = MyBaseComponent
```

An entry matches either the component's simple name or its fully qualified name.

## Why there is no code fix

There is no safe automatic rewrite. Whether the component should become routable, be
wrapped in an `<AuthorizeView>`, or have the authorization moved to its host page is
the author's design intent, not something the analyzer can infer. The rule reports and
leaves the decision to the author.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Authorization.AuthorizeAttribute` and
`Microsoft.AspNetCore.Components.ComponentBase` in the analyzed compilation and
registers nothing when either is absent, so a project that does not reference Blazor
components receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. A declaration with no attributes is rejected
before the semantic model is touched; the type is bound only for the rare shape of an
`[Authorize]` without a `[Route]`.
