# SES1704 — HttpContext accessed from an interactive component is stale

**Category:** Blazor
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A component whose definition fixes an **interactive** render mode — Interactive
Server, Interactive WebAssembly, or Interactive Auto — does not execute inside a live
HTTP request. Under interactive rendering:

- `Microsoft.AspNetCore.Http.IHttpContextAccessor` returns `null`, and
- an `HttpContext` captured as a `[CascadingParameter]` is **frozen at the moment the
  circuit started** — its user, claims, and headers are never refreshed as the circuit
  lives on.

Code that reads authorization or request state through either one is therefore acting
on absent or stale data. That is a security bug when it gates access: a check that
"passed" when the circuit opened keeps passing long after the underlying request, its
cookies, and its authentication have changed.

The rule reports the capturing member:

```csharp
using Microsoft.AspNetCore.Components;
using Microsoft.AspNetCore.Components.Web;
using Microsoft.AspNetCore.Http;

[InteractiveServerRenderMode]                 // an interactive render mode is fixed on the definition
public class Dashboard : ComponentBase
{
    [Inject]
    public IHttpContextAccessor Accessor { get; set; }   // reported (SES1704): null under interactive rendering

    [CascadingParameter]
    public HttpContext Context { get; set; }             // reported (SES1704): frozen at circuit start
}
```

Flow the specific values the component needs — the authenticated user through
`AuthenticationStateProvider`, a claim, or a header value — in as parameters instead of
capturing `HttpContext`:

```csharp
[InteractiveServerRenderMode]
public class Dashboard : ComponentBase
{
    [Inject]
    public AuthenticationStateProvider AuthState { get; set; }   // refreshed with the circuit
}
```

## What is reported

On a component (a `ComponentBase`-derived type) whose definition carries a fixed render
mode:

- an `[Inject]` **property or field**, or a **constructor parameter**, typed
  `IHttpContextAccessor`;
- a `[CascadingParameter]` **property or field** typed `HttpContext`.

Every marker is bound by symbol. A fixed render mode is detected as a
`Microsoft.AspNetCore.Components.RenderModeAttribute` on the type: the Razor compiler
emits that attribute for an `@rendermode` declared on the component itself, and the
three interactive modes are the only ones that produce it (static server rendering
emits none).

## What is not reported

- **Static server rendering.** A component with no fixed render mode runs inside the
  request, where `HttpContext` is valid; it is not reported.
- **A non-`HttpContext` dependency.** An injected service or a cascaded value of any
  other type is out of scope.
- **A non-component type.** A plain class that happens to carry a render-mode attribute
  and an accessor is ignored.

### Known limitation

The render mode must be declared **on the component definition** (an `@rendermode` on
the component, which compiles to a `RenderModeAttribute` on the type) for the rule to
decide. A render mode applied only where the component is *used*
(`<Dashboard @rendermode="InteractiveServer" />`) is not visible on the definition, so
such usage is not reported.

## Why there is no code fix

There is no safe automatic rewrite. Replacing an `HttpContext` capture with the right
alternative — `AuthenticationStateProvider`, a specific claim, a header passed as a
parameter — depends on what the component actually reads, which is the author's intent.
The rule reports and leaves the redesign to the author.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Http.IHttpContextAccessor`,
`Microsoft.AspNetCore.Components.RenderModeAttribute`, and
`Microsoft.AspNetCore.Components.ComponentBase` (with the `HttpContext`, `InjectAttribute`,
and `CascadingParameterAttribute` types it matches against) in the analyzed compilation
and registers nothing when any is absent, so a project without interactive Blazor
components receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is selective. A declaration with no attributes, or without a
render-mode attribute, is rejected before its members are scanned; member types are
bound only for members that actually carry `[Inject]` or `[CascadingParameter]`.
