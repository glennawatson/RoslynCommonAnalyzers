# SES1705 — A Blazor component must navigate only to a verified relative URL

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Blazor |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A call to `Microsoft.AspNetCore.Components.NavigationManager.NavigateTo` passes a `uri`
argument that is not a verified relative URL: it is non-constant, an absolute URL
literal (one that carries a scheme), or a protocol-relative literal (one that begins
with two slashes) — and it is not produced by an allow-listed validator.

```csharp
using Microsoft.AspNetCore.Components;

public class LoginModel
{
    private NavigationManager _navigation = null!;

    // 👎 reported: 'returnUrl' can be any host, so the navigation is an open redirect
    public void Continue(string returnUrl) => _navigation.NavigateTo(returnUrl);

    // 👎 reported: an absolute URL leaves the app's origin
    public void Leave() => _navigation.NavigateTo("https://attacker.example/login");

    // 👍 a constant relative URL stays on the current origin
    public void Home() => _navigation.NavigateTo("/home");
}
```

## Rule description

`NavigationManager.NavigateTo` hands its `uri` argument to the browser as the next
location. When that argument is a compile-time-constant **relative** URL — `/counter`,
`counter`, `~/counter`, `#section` — the navigation resolves against the app's own base
URI and stays on the current origin, where it cannot be steered.

An **absolute** URL (`https://attacker.example`), a **protocol-relative** URL
(`//attacker.example`, and the `/\` / `\\` browser-equivalent tricks), or any
**non-constant** value — a query-string parameter, a field read from client state, a
value taken off the wire — can instead send the browser to a host the attacker chose.

That is an **open redirect** (CWE-601). A link that appears to belong to the trusted
app lands the victim on an attacker-controlled page that harvests credentials or serves
malware, and the trusted origin in the original link lends the attack credibility, which
is exactly why open redirects are a staple of phishing.

The rule reports the `uri` argument unless one of the following holds:

- **It is a verified relative literal.** A compile-time constant whose value is not
  absolute (carries no `scheme:`) and not protocol-relative (does not begin with two
  slashes). A leading single slash is treated as a rooted **local** path, not an
  absolute URL.
- **It is produced by an allow-listed validator.** When the argument is a call to a
  method named in `securitysharp.SES1705.validators` (or the project-wide
  `securitysharp.validators`), the result is trusted and the call is left alone.

### Detection heuristic

Detection is **local** to the single call: the rule reports whenever a target that is
not a verified relative literal reaches `NavigateTo`, without tracing where a
non-constant value came from. It does not attempt whole-program taint tracking, so it
fires on the visible shape and stays near zero false positives — the trade-off is that a
non-constant target the surrounding code has already validated is still reported unless
that validation runs through an allow-listed validator. The whole rule is gated on
`Microsoft.AspNetCore.Components.NavigationManager` resolving, so a project without
Blazor pays nothing.

## How to fix violations

Navigate to a relative URL, so the target can never leave the origin:

```csharp
// before
public void Continue(string returnUrl) => _navigation.NavigateTo(returnUrl);

// after — reject any non-local target before navigating
public void Continue(string returnUrl)
{
    var target = EnsureLocal(returnUrl);
    _navigation.NavigateTo(target);
}

private static string EnsureLocal(string url)
    => url.StartsWith("/", StringComparison.Ordinal) && !url.StartsWith("//", StringComparison.Ordinal) ? url : "/";
```

If the navigation genuinely must be able to reach an external site, validate the target
against an allow-list of permitted destinations and navigate only when it matches. Name
the validating method in `securitysharp.SES1705.validators` so the rule treats its
result as trusted.

There is no automatic code fix: whether a target is safe, and how it should be
validated, is a decision for the author.

## Configuration

```ini
# Rule-specific: methods whose result is a trusted (already-validated) navigation target
securitysharp.SES1705.validators = EnsureLocal, ToSafeReturnUrl

# Project-wide (applies to any SecuritySharp rule that reads a validator allow-list)
securitysharp.validators = EnsureLocal
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `validators` | comma-separated method names | *(empty)* | A `NavigateTo` target produced by a call to one of these methods is treated as verified and not reported. |

The rule-specific key wins over the project-wide key. A validator is matched by its
simple method name, so `EnsureLocal(url)` and `helper.EnsureLocal(url)` both qualify.

## Performance

The rule gates on `Microsoft.AspNetCore.Components.NavigationManager` resolving in the
compilation, so a non-Blazor project registers nothing and pays nothing. Each invocation
is filtered syntactically first — only a call named `NavigateTo` reaches the semantic
model — and the `.editorconfig` validator allow-list is read only after a target has
already been found to be neither a verified relative literal nor an obvious plain value,
keeping the common no-diagnostic path allocation-free.
