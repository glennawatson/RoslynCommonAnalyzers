# SES1706 — A Blazor uploaded-file read must have a bounded, server-chosen size limit

| Item | Value |
| --- | --- |
| Package | SecuritySharp.Analyzers |
| Category | Blazor |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A call to `Microsoft.AspNetCore.Components.Forms.IBrowserFile.OpenReadStream` passes a
`maxAllowedSize` argument that is unbounded or client-derived: `long.MaxValue`,
`IBrowserFile.Size`, or a compile-time constant above the configured threshold
(default 10 MB).

```csharp
using Microsoft.AspNetCore.Components.Forms;

public class UploadModel
{
    // 👎 reported: no cap at all — the client can stream any amount
    public void ReadUnbounded(IBrowserFile file) => Consume(file.OpenReadStream(long.MaxValue));

    // 👎 reported: the browser-reported size is untrusted client metadata
    public void ReadClientSized(IBrowserFile file) => Consume(file.OpenReadStream(file.Size));

    // 👍 a fixed, server-chosen limit sized to the largest legitimate upload
    public void ReadBounded(IBrowserFile file) => Consume(file.OpenReadStream(2 * 1024 * 1024));

    // 👍 the no-argument overload keeps the safe ~500 KB default
    public void ReadDefault(IBrowserFile file) => Consume(file.OpenReadStream());
}
```

## Rule description

`IBrowserFile.OpenReadStream` caps the number of bytes it will stream at its
`maxAllowedSize` argument, which defaults to roughly 500 KB so a caller cannot
accidentally read an oversized upload into memory. Raising that cap to an unbounded or
client-controlled value hands the limit back to the attacker, and reading the resulting
stream into a server-side buffer can then exhaust the process's memory or disk and take
the service down — a denial of service (CWE-770, allocation of resources without
limits). The risk is sharpest for interactive server-side components, where the read
happens on the server.

The rule reports the `maxAllowedSize` argument in three shapes:

- **`long.MaxValue`** — removes the cap entirely.
- **`IBrowserFile.Size`** — the length the browser reported for the file. That metadata
  is supplied by the client and is untrusted, so passing it lets the client declare its
  own limit.
- **A constant above the threshold** — a compile-time-constant byte count strictly
  greater than `securitysharp.SES1706.max_bytes` (default `10485760`, i.e. 10 MB).

## What is not reported

- **The no-argument `OpenReadStream()`.** It keeps the framework's safe ~500 KB default,
  so it is never reported. Passing only a named `cancellationToken:` (which also leaves
  `maxAllowedSize` at its default) is likewise left alone.

  ```csharp
  // Not reported: the safe default cap is in force.
  file.OpenReadStream();
  file.OpenReadStream(cancellationToken: token);
  ```

- **A bounded constant at or below the threshold.** A fixed limit sized to the largest
  legitimate upload is exactly the recommended shape.

- **A non-constant size that is not `IBrowserFile.Size`.** A limit read from
  configuration or computed at runtime cannot be judged from the source and may well be a
  safe number, so the rule stays silent rather than guess. Reading the limit from
  configuration is by design a supported way to keep the rule quiet.

- **A `Size` property on any other type.** The property is matched by its bound symbol,
  so a `Size` on an unrelated type is not mistaken for the client-reported file length.

## How to fix violations

Pass a fixed, server-chosen limit sized to the largest upload you actually expect, and
stream the file to disk or an external store rather than buffering the whole thing in
memory:

```csharp
// before
Consume(file.OpenReadStream(long.MaxValue));

// after
const long MaxUploadBytes = 5 * 1024 * 1024;
await using var target = File.Create(path);
await file.OpenReadStream(MaxUploadBytes).CopyToAsync(target);
```

There is deliberately **no code fix**: the right limit is a threat-model decision, not a
mechanical substitution.

## Configuration

```ini
# Rule-specific
securitysharp.SES1706.max_bytes = 10485760

# Project-wide (applies to any SecuritySharp rule that reads a byte ceiling)
securitysharp.max_bytes = 10485760
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `max_bytes` | integer `>= 1` | `10485760` (10 MB) | The largest byte count a constant `maxAllowedSize` may reach before it is reported. `long.MaxValue` and `IBrowserFile.Size` are reported regardless of this value. |

The rule-specific key wins over the project-wide key. An unset, unparsable, or
nonsensical value (below `1`) falls back to the default, so a typo cannot silently
disable the rule.

## Performance

The rule gates on `Microsoft.AspNetCore.Components.Forms.IBrowserFile` resolving in the
compilation, so a project without Blazor registers nothing and pays nothing. Each
invocation is filtered syntactically first — only a call named `OpenReadStream` that
carries an explicit argument reaches the semantic model — and the `.editorconfig`
threshold is read only after a constant size has been found, keeping the common
no-diagnostic path allocation-free.
