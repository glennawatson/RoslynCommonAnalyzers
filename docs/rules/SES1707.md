# SES1707 — Do not hard-code secrets in code that runs in the browser as WebAssembly

**Category:** Blazor
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A Blazor WebAssembly assembly is downloaded to the browser and runs on the client.
Every string it contains is therefore readable by anyone who opens the browser
developer tools or unpacks the app bundle — there is no server boundary left to
protect it. A credential embedded in that code is disclosed the moment the app is
served, and the identical literal that would be safe in server-only code becomes a
guaranteed leak here.

```csharp
// reported (SES1707): this key ships to the browser and is fully readable
[RenderModeInteractiveWebAssembly]
public partial class Report : ComponentBase
{
    private const string ApiKey = "AKIAIOSFODNN7EXAMPLE";
}
```

Keep the secret on the server, expose it only through an authenticated endpoint,
and have the client call that endpoint:

```csharp
// the browser never sees the key; the server holds it and authorizes the call
var data = await Http.GetFromJsonAsync<Report>("api/report");
```

## What is reported

A string literal is flagged when both hold:

- its content matches a high-precision credential shape — the same shapes as the
  general hard-coded-secret rule (an OpenAI-style key, an AWS access key id, a
  GitHub or Slack token, a Google API key, a PEM private-key block, an Azure key
  body, or a connection-string password); and
- the literal is **WebAssembly-reachable**, meaning one of:
  - the compilation is a standalone WebAssembly host — or the client project of a
    Blazor Web App — whose whole assembly is downloaded to the browser; or
  - the literal's enclosing component type carries a render-mode attribute that
    selects **Interactive WebAssembly** or **Interactive Auto** hosting (both run
    on the client).

Server-rendered code is **not** reported: an Interactive Server component, or any
type in a server-only project, never ships that text to the browser.

## Why the render mode matters

The same secret literal is a leak in a WebAssembly-hosted component and harmless in
a server-hosted one, so the rule keys off where the code runs. A component whose
fixed render mode is Interactive WebAssembly or Interactive Auto is compiled into
the downloaded client bundle; a component left server-rendered stays on the server.
The rule reads the render mode from the component's render-mode attribute — the one
the `@rendermode` directive generates on the component type — and reports only the
client-hosted modes.

## Why there is no code fix

Removing a secret safely requires somewhere to move it — an authenticated endpoint,
a configuration source read on the server, a token exchange — that the analyzer
cannot design or locate. The rule reports and leaves the redesign to the developer.
Whatever the fix, treat the leaked value as compromised and rotate it.

## Proving the surface exists

The rule resolves the WebAssembly render-mode marker
(`Microsoft.AspNetCore.Components.Web.InteractiveWebAssemblyRenderMode`) and the
standalone host builder
(`Microsoft.AspNetCore.Components.WebAssembly.Hosting.WebAssemblyHostBuilder`) once
per compilation and registers nothing when neither is present. A project that
targets no WebAssembly hosting receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is a single allocation-free classification of the literal's
text; only a recognized secret shape triggers the reachability check, and that check
runs the render-mode walk just once per matched literal.
