# SES1708 — Circuit detailed errors must not be enabled outside development

**Category:** Blazor
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

A server-side Blazor circuit normally sends the browser only a generic error
identifier when an unhandled exception occurs, keeping the message and stack trace
in the server log. Setting `CircuitOptions.DetailedErrors` to `true` reverses that:
the full exception detail — message, stack trace, and any state surfaced in it, such
as file paths, type and framework versions, and connection strings caught in a
message — is streamed to **every** connected client. That is a debugging aid for a
developer and a reconnaissance gift for an attacker, so it belongs only in the
Development environment.

```csharp
// reported (SES1708): detailed circuit errors go to every client
builder.Services.AddServerSideBlazor(options =>
{
    options.DetailedErrors = true;
});
```

Leave the option at its default in production, or bind it to the environment so it
is enabled only in Development:

```csharp
builder.Services.AddServerSideBlazor(options =>
{
    options.DetailedErrors = builder.Environment.IsDevelopment();
});
```

## What is reported

The `DetailedErrors` property is flagged when it is assigned the constant `true`
and its containing type is
`Microsoft.AspNetCore.Components.Server.CircuitOptions`. Both the direct assignment
(`options.DetailedErrors = true`) and the object-initializer member
(`new CircuitOptions { DetailedErrors = true }`) are covered. The symbol is bound,
so a same-named property on any other type is ignored, and a value that is not the
constant `true` — `false`, or a value read from configuration or the environment —
is not reported.

## Why there is no code fix

The safe replacement is to gate the flag on the environment, which needs an
environment accessor the analyzer cannot conjure or reliably locate. The rule
reports and leaves the decision to the developer.

## Proving the API exists

The rule resolves `Microsoft.AspNetCore.Components.Server.CircuitOptions` in the
analyzed compilation and registers nothing when it is absent, so a project without
server-side Blazor receives no diagnostic and pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An assignment whose target is not named
`DetailedErrors`, or whose right-hand side is not the literal `true`, is rejected
before the semantic model is touched; only a matching shape is bound to confirm the
containing type.
