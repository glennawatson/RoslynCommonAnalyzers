# SES1709 — Do not serialize every claim into the client-readable authentication state

**Category:** Blazor
**Severity:** Warning
**Code fix:** None — see [Why there is no code fix](#why-there-is-no-code-fix)

When a Blazor Web App serializes the server authentication state so a WebAssembly
client can read it, the default serializer emits only the name and role claims — the
minimum the browser needs to render an authorized view. Setting
`AuthenticationStateSerializationOptions.SerializeAllClaims` to `true` copies every
claim on the principal into that state, and because the state is transported to and
stored in the browser it becomes fully readable on the client. Internal identifiers,
access or refresh tokens, security stamps, and personal data that were only ever
meant for the server are then disclosed to anyone with the client.

```csharp
// reported (SES1709): every claim is serialized into the browser-readable state
builder.Services.AddRazorComponents()
    .AddInteractiveWebAssemblyComponents()
    .AddAuthenticationStateSerialization(options =>
    {
        options.SerializeAllClaims = true;
    });
```

Leave it at its default. When the client genuinely needs an extra claim, add just
that claim through the serialization callback rather than shipping them all.

## What is reported

The `SerializeAllClaims` property is flagged when it is assigned the constant
`true` and its containing type is
`Microsoft.AspNetCore.Components.WebAssembly.Server.AuthenticationStateSerializationOptions`.
Both the direct assignment (`options.SerializeAllClaims = true`) and the
object-initializer member are covered. The symbol is bound, so a same-named property
on any other type is ignored, and a value that is not the constant `true` is not
reported.

## Why there is no code fix

Removing the flag can change which claims the client sees, and choosing the right
per-claim subset is an authorization decision the analyzer cannot make. The rule
reports and leaves the decision to the developer.

## Proving the API exists

The rule resolves
`Microsoft.AspNetCore.Components.WebAssembly.Server.AuthenticationStateSerializationOptions`
in the analyzed compilation and registers nothing when it is absent, so a project
that does not serialize WebAssembly authentication state receives no diagnostic and
pays no analysis cost.

## Performance

The no-diagnostic path is syntactic. An assignment whose target is not named
`SerializeAllClaims`, or whose right-hand side is not the literal `true`, is rejected
before the semantic model is touched; only a matching shape is bound to confirm the
containing type.
