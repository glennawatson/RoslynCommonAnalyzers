# SST0019 — Conversion operator declaration parameters should be on unique lines

| | |
| --- | --- |
| **Category** | Readability |
| **Severity** | Warning |
| **Code fix** | Yes — puts each parameter on its own line |
| **Applies to** | `implicit operator Foo(Bar b)` (note: conversion operators always have exactly one parameter, so in practice this rule never reports — included for completeness/symmetry) |

## Cause

The parameters in a conversion operator declaration are split across lines such that some share a line while others are wrapped (a "jagged" layout), instead of all being on one line or each on its own line.

## Rule description

A conversion operator declaration should keep its parameters visually consistent: either everything on a single line, or one parameter per line. Mixed/jagged layouts hurt readability and produce noisy diffs when items are added or removed.

In practice a conversion operator always declares exactly one parameter, so a jagged layout is impossible and this rule never actually fires. It exists for completeness and symmetry with the other list-formatting rules.

## How to fix violations

Put every parameter on its own line (the supplied code fix does this), or collapse the list back onto a single line.

### Example

```csharp
// A conversion operator has a single parameter, so there is nothing to make jagged:
public static implicit operator Foo(Bar b) => new Foo();
```

## When to suppress

Generally don't — this is purely a formatting/readability convention. If a particular file or generated code needs an exception, configure the severity in `.editorconfig` (`dotnet_diagnostic.SST0019.severity = none`) rather than scattering pragmas.
