# SST0020 — Type parameter lists should be on unique lines

| | |
| --- | --- |
| **Category** | Readability |
| **Severity** | Warning |
| **Code fix** | Yes — puts each type parameter on its own line |
| **Applies to** | generic type parameter lists — `class Foo<T1, T2>`, `void M<T1, T2>()` |

## Cause

The type parameters in a type parameter list are split across lines such that some share a line while others are wrapped (a "jagged" layout), instead of all being on one line or each on its own line.

## Rule description

A type parameter list should keep its type parameters visually consistent: either everything on a single line, or one type parameter per line. Mixed/jagged layouts hurt readability and produce noisy diffs when items are added or removed.

## How to fix violations

Put every type parameter on its own line (the supplied code fix does this), or collapse the list back onto a single line.

### Example

```csharp
// 👎 Violates SST0020
class Foo<T1, T2,
    T3> { }

// 👍 OK — each type parameter on its own line
class Foo<
    T1,
    T2,
    T3> { }

// 👍 OK — all on one line
class Foo<T1, T2, T3> { }
```

## When to suppress

Generally don't — this is purely a formatting/readability convention. If a particular file or generated code needs an exception, configure the severity in `.editorconfig` (`dotnet_diagnostic.SST0020.severity = none`) rather than scattering pragmas.
