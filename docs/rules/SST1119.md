# SST1119 — Digit separators should group digits regularly

| Item | Value |
| --- | --- |
| Category | Readability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A numeric literal uses `_` digit separators, but the groups after the first are not all the same width, or that
width is not the base's convention.

```csharp
public int Population => 1_000_00_000; // SST1119
```

## Rule description

Uneven grouping is a misreading hazard, not just an untidy one. `1_000_00_000` looks like a billion and is a
hundred million — the separators actively mislead. The rule reads a literal as regular when every group after
the first shares one width, and that width is the convention for the base: three for decimal, four or two for
hexadecimal and binary.

- The **leading group may be short**, so `1_000_000` and `0x_FF_FF` are both fine.
- A group after the first that differs from its siblings, or a uniform width that is not the base's convention,
  is reported.
- Floating-point literals are left alone; their fractional and exponent parts are not an integer-grouping
  question.

This is disjoint from the separate opt-in rule that reports a long literal carrying no separators at all: a
literal can match at most one of the two.

## How to fix violations

Regroup the digits evenly for the base.

```csharp
public int Population => 100_000_000;
```

The code fix rewrites the literal into even groups of the base's conventional width, dropping the separators
when the number is too short to group regularly (`0xFF_F` becomes `0xFFF`).

## Performance

The clean path is a single pass over the token text. A literal with no `_` is rejected before anything is
counted, and nothing is allocated: the group widths are tallied in place. No semantic model is consulted.
