# SST1145 — Place conditional operators consistently

| Item | Value |
| --- | --- |
| Category | Readability |
| Severity | Warning |
| Code fix | No |
| Default | Leading |

## Cause

When a conditional (ternary) expression wraps across lines, its `?` or `:`
operator is on the wrong end of a line for the configured placement.

```csharp
// 👎 reported (default 'leading' placement)
var x = condition ?
    whenTrue :
    whenFalse;

// 👍
var x = condition
    ? whenTrue
    : whenFalse;
```

## Rule description

A wrapped conditional reads more clearly when its operators are placed
consistently. The default — `leading` — puts a wrapped `?`/`:` at the **start** of
its continuation line, the dominant .NET convention. Only operators that are
actually wrapped are checked, so single-line conditionals (`c ? a : b`) and
operators that stay inline are never reported.

## Configuration

```ini
[*.cs]
# leading (default) or trailing
stylesharp.conditional_operator_placement = leading
# rule-specific override wins over the general key:
stylesharp.SST1145.conditional_operator_placement = leading
```

With `trailing`, the operators are expected at the **end** of the preceding line
instead.

## How to fix violations

Move the `?`/`:` operator to the configured end of the line. (No automatic code fix
— reflowing wrapped expressions is left to the formatter.)
