# SST1148 — Commented-out code should be removed

| Item | Value |
| --- | --- |
| Category | Readability |
| Severity | Warning |
| Code fix | No |
| Default | Disabled (opt-in) |

## Cause

A regular comment conservatively resembles a C# statement.

```csharp
// return result;
```

## Rule description

Source control preserves removed code, while comments should explain the current code.
The classifier is intentionally conservative and ignores documentation comments, file
headers, URLs, and `TODO` or `HACK` markers.

### How the match works (and its limits)

Detection is a deliberately simple textual heuristic, not a parser: a single-line comment is
flagged when its text **ends with `;`, `{`, or `}`**, or **begins with a code keyword**
(`return`, `throw`, `var`, `if (`, …). The diagnostic message names the signal that matched
(for example *“it ends with a ';'”*) so the cause is obvious.

Because of that, a prose sentence can trip the rule — most often when an English clause break
leaves a trailing semicolon:

```csharp
// the subject is seeded with the initial value;   <- flagged: it ends with a ';'
// late subscribers therefore receive that value.
```

This is expected for a heuristic rule. When a flagged comment is genuinely prose, either reword
it so it does not end with `;`/`{`/`}` (for example move the semicolon, or end the line with the
clause), or suppress the single occurrence. The rule is opt-in for exactly this reason.

## How to fix violations

Delete the commented-out statement, or — if the comment is prose — reword it so it does not end
with a statement terminator, or suppress that occurrence.
