# SST1172 тАФ Negated comparisons should use the opposite operator

| Item | Value |
| --- | --- |
| Category | Readability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |
| Analyzer | `ExpressionSimplificationAnalyzer` |

## Cause

A comparison is wrapped in a logical-not (`!`) instead of using the opposite operator.

```csharp
// ЁЯСО reported
return !(a == b);
return !(a != b);
if (!(count < limit)) { }

// ЁЯСН
return a != b;
return a == b;
if (count >= limit) { }
```

## Rule description

Negating a comparison with `!` hides the real intent behind an extra operator. The
opposite operator reads directly: `!(a == b)` is `a != b`, `!(x < y)` is `x >= y`.

Equality inversions (`==`/`!=`) are always reported because they are safe for every
operand type. Relational inversions (`<`, `<=`, `>`, `>=`) are only reported when the
semantic model confirms neither operand is **nullable** or **floating-point** тАФ
inverting `!(a < b)` to `a >= b` does not preserve the result when an operand is
`null` (the lifted comparison is `false`, so both `!(a < b)` and `a >= b` differ) or
`NaN` (every floating-point comparison with `NaN` is `false`). Conditional-access
operands (`a?.X`) are skipped for the same reason.

The rule uses an allocation-free, syntactic-first implementation: the semantic model is
only consulted for relational operators after the cheap syntactic match.

## How to fix violations

Replace the negated comparison with the opposite operator and drop the `!`. The
provided code fix does this automatically.
