# SST1174 — Redundant jump statements should be removed

| Item | Value |
| --- | --- |
| Category | Readability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |
| Analyzer | `RedundantCodeAnalyzer` |

## Cause

A `return;` is the last statement of a void member, or a `continue;` is the last statement
of a loop body — in both cases control already flows to the same place, so the jump does
nothing.

```csharp
// 👎 reported
public void Log(int x)
{
    Console.WriteLine(x);
    return;            // control already leaves the method here
}

for (var i = 0; i < n; i++)
{
    Process(i);
    continue;          // the loop already continues here
}

// 👍
public void Log(int x)
{
    Console.WriteLine(x);
}

for (var i = 0; i < n; i++)
{
    Process(i);
}
```

## Rule description

A value-less `return;` at the tail of a method, local function, accessor, constructor,
operator, or lambda body is unreachable-by-effect — removing it does not change control
flow. The same holds for a `continue;` at the tail of a `for`, `foreach`, `while`, or `do`
loop body.

To stay free of false positives this is a deliberately syntactic, allocation-free subset:
the jump is only reported when it is the **last statement of the block that is directly the
member or loop body**. A `return;`/`continue;` nested inside an `if`, `try`, `switch`, or any
other inner block is never reported, because removing it there could change behavior. The
check is deliberately syntactic, with no control-flow-graph analysis.

## How to fix violations

Delete the redundant statement. The provided code fix does this automatically.
