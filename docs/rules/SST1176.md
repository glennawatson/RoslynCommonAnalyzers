# SST1176 — Members should not be initialized to their default value

| Item | Value |
| --- | --- |
| Category | Readability |
| Severity | Warning |
| Code fix | Yes |
| Default | Disabled (opt-in) |
| Analyzer | `RedundantCodeAnalyzer` |

## Cause

A field, event field, or auto-property is explicitly initialized to the value the runtime
already gives it.

```csharp
// 👎 reported (when enabled)
private int _count = 0;
private string _name = null;
public bool Ready { get; set; } = false;

// 👍
private int _count;
private string _name;
public bool Ready { get; set; }
```

## Rule description

Fields, event fields, and auto-properties start at their type's default value, so writing
`= 0`, `= false`, `= null`, `= default`, or `= default(T)` adds nothing. The check is purely
syntactic — those literal forms are the default of any type they can legally bind to — so no
semantic model is needed. `const` fields are never reported, since they require an
initializer.

This rule is **off by default**: writing the default explicitly is a defensible, intentional
style. Enable it in `.editorconfig`:

```ini
dotnet_diagnostic.SST1176.severity = warning
```

Named members that happen to equal the default (for example an enum member whose value
is `0`) are deliberately not reported.

## How to fix violations

Drop the initializer. The provided code fix removes it (and the auto-property's trailing
`;` when present).
