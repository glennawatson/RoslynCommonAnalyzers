# SST1181 — Redundant overriding members should be removed

| Item | Value |
| --- | --- |
| Category | Readability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |
| Analyzer | `RedundantCodeAnalyzer` |

## Cause

An `override` does nothing but forward to the same base member with the parameters passed
straight through, so it has no effect beyond what the inherited member already provides.

```csharp
// 👎 reported
public override int Add(int x, int y) => base.Add(x, y);

public override int Value
{
    get => base.Value;
    set => base.Value = value;
}

// 👍 — let the inherited member be used directly (delete the override)
```

## Rule description

A method override is reported when its body is exactly `base.Member(args)` (block or
expression bodied) and the arguments re-pass the parameters in order with matching `ref`/`out`/
`in` kinds. A property override is reported when every accessor only forwards to the base
(`get => base.X;`, `set => base.X = value;`).

To stay safe, the rule skips members that carry attributes, `sealed override` members (which
deliberately stop further overriding), and generic methods. The whole check is syntactic, so
the no-diagnostic path binds no symbols.

## How to fix violations

Delete the redundant override so the inherited member is used directly. The provided code fix
does this automatically.
