# SST1201 — Elements should be ordered by kind

| Item | Value |
| --- | --- |
| Category | Ordering |
| Severity | Warning |
| Code fix | Yes (move) |

## Cause

A type's members are not grouped by kind in the conventional order.

## Rule description

Within a type, members appear grouped by kind in this order:

1. Fields
2. Constructors
3. Finalizers (destructors)
4. Delegates
5. Events
6. Enums
7. Interfaces
8. Properties
9. Indexers
10. Methods (including operators and conversions)
11. Structs (including record structs)
12. Classes
13. Records
14. Unions

Nested reference types sort **classes before records before unions**. Unions are
recognised by the `System.Runtime.CompilerServices.IUnion` marker interface, so the
distinction only applies once unions are in play.

```csharp
// 👎 reported — the field follows a method
public class Widget
{
    public void Run() { }
    private int _count;
}

// 👍 fields before methods
public class Widget
{
    private int _count;
    public void Run() { }
}
```

## How to fix violations

Move the member into its kind group. The code fix relocates it just after the last
member of an equal-or-earlier rank. The fix is only offered when the file has no
conditional (`#if`/`#elif`/`#else`/`#endif`) directives, which could otherwise move a
member across a compilation boundary.
