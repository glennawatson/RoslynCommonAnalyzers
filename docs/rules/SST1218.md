# SST1218 — Method overloads should be grouped together

| Item | Value |
| --- | --- |
| Category | Ordering |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

Other members separate the overloads of one method name.

```csharp
public sealed class Writer
{
    public void Write(int value)
    {
    }

    public void Flush()
    {
    }

    public void Write(string value) // reported: separated from the overload above
    {
    }
}
```

## Rule description

Overloads are one idea with several entry points. Scatter them through a type and a reader who found
one has no reason to suspect the others exist — so they add a third, or they change the one they
landed on and leave its twin behind. Keeping the family together is what makes it look like a
family.

The diagnostic lands on the **out-of-place overload** — the one that appears after the interruption —
because that is the member that has to move.

```csharp
public sealed class Writer
{
    public void Write(int value)
    {
    }

    public void Write(string value)
    {
    }

    public void Flush()
    {
    }
}
```

## What counts as one family

Two methods are one family when they share a name **and** the place the ordering rules give them:
the same accessibility, and the same static-ness.

That qualifier matters. [SST1202](SST1202.md) requires the more accessible members first, and
[SST1204](SST1204.md) requires the static ones before the instance ones. If this rule compared
across those lines, a `private` overload sitting correctly below the public members would be
reported here — and moving it up to its family would be reported there. A rule that demands a move
the ordering rules demand back is not a rule, it is a loop. So overloads in different ordering
groups are simply never compared.

```csharp
public sealed class Writer
{
    public static void Write(int value)   // a different group: static
    {
    }

    public void Write(string value)
    {
    }

    public void Flush()
    {
    }

    private void Write(bool value)        // a different group: private. Not reported.
    {
    }
}
```

## What is not reported

- **Comments and blank lines between two overloads.** They are not members, so they never separate a
  family. Two overloads with a paragraph of explanation between them are still together.
- **Constructors.** They already share the type's name, so a reader who found one knows exactly
  where the others live, and the ordering rules pin them to a single block of the type anyway.
  Operators, indexers and conversions are excluded for the same reason.
- **An explicit interface implementation.** Its placement follows the interface it implements, not
  the method name it happens to share.
- **The other parts of a `partial` type.** Each declaration is judged on its own members. Overloads
  split across two files are two files' business, and this rule does not reach across them.
- **Generated code.**

## How to fix violations

The code fix moves the overload back up to sit immediately after the nearest earlier member of its
family. The member moves as a whole node, so the blank line, the comments and the documentation
comment in its leading trivia travel with it.

Fixing one diagnostic moves one member. **Fix All** regroups every family in the type in a single
pass, which reaches the same place without walking there one member at a time.

The fix is **not offered** when the type contains a preprocessor directive. A member carries the
trivia around it when it moves, and moving one past a `#region` or an `#if` would leave the pair
unbalanced. The diagnostic still reports; the move is yours to make.

## Performance

The rule is pure syntax — identifier text and modifier keywords. It never touches the semantic
model and allocates nothing on any path. For each method it scans back to the nearest earlier member
of the same family and stops there, so a type whose members are all distinct pays one comparison per
earlier member and nothing else.
