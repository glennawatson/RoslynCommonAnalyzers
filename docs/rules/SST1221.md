# SST1221 тАФ Constraint clauses should match the type-parameter declaration order

| Item | Value |
| --- | --- |
| Category | Ordering |
| Severity | Info |
| Code fix | Yes |
| Default | Enabled |
| Analyzer | `Sst1221ConstraintClauseOrderAnalyzer` |

## Cause

A generic declaration's `where` constraint clauses are not written in the order of the type parameters
they constrain.

```csharp
// ЁЯСО reported
public class Cache<TKey, TValue>
    where TValue : class
    where TKey : notnull
{
}

// ЁЯСН
public class Cache<TKey, TValue>
    where TKey : notnull
    where TValue : class
{
}
```

## Rule description

Constraint clauses read most easily when they follow the order of the type parameters in the
angle-bracket list, so a reader can scan the parameters and their constraints together. The rule fires
when a declaration has at least two `where` clauses and one of them constrains a type parameter that
appears earlier in the list than the clause before it.

It applies to types, methods, delegates, and local functions. It pairs with SST1127, which keeps each
`where` clause on its own line.

## How to fix violations

The fix reorders the constraint clauses to match the type-parameter list, keeping each clause's own
trivia so the layout is preserved.
