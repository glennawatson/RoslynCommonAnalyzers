# SST1305 — Field names should not use Hungarian notation

| Item | Value |
| --- | --- |
| Category | Naming |
| Severity | Warning (disabled by default) |
| Code fix | No |

## Cause

A field, parameter, or local name appears to use Hungarian notation — a one- or
two-letter lower-case type prefix immediately followed by an upper-case letter
(for example `iCount`, `strName`).

## Rule description

Hungarian notation is avoided in favour of meaningful names. A small allow-list of
common English words (`is`, `to`, `id`, …) is excluded, so names like `isEnabled` are
not flagged.

This rule is **disabled by default** — it is a heuristic and opt-in. Enable it in
`.editorconfig` with `dotnet_diagnostic.SST1305.severity = warning`.

### Allowing extra prefixes

Real projects use short prefixes that are not type abbreviations (`vm` for a view model,
`db`, `wpf`, …). Add them to the allow-list with a comma- or whitespace-separated list, either
for this rule or generally:

```ini
[*.cs]
stylesharp.SST1305.allowed_hungarian_prefixes = vm, wpf, db
# or, shared with any future prefix-based rule:
stylesharp.allowed_hungarian_prefixes = vm, wpf, db
```

The configured prefixes are matched case-insensitively and added to the built-in allow-list.

## How to fix violations

Rename the element to a meaningful name without the type prefix.

## Performance and accuracy

This rule is a heuristic: it infers Hungarian notation from the shape of a name and can
produce false positives (for example on legitimate prefixes outside its allow-list). It is
disabled by default — enable it deliberately and review its findings rather than treating
them as absolute.
