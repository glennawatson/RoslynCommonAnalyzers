# SST1320 — Parameter names should not match the method name

| Item | Value |
| --- | --- |
| Category | Naming |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method parameter has the same name as the method that declares it.

```csharp
public void Foo(int Foo) // SST1320
{
}
```

## Rule description

A parameter that repeats its method's name is almost always a copy-paste slip left over from
renaming one but not the other. It also reads badly at the call site: a named argument such as
`Foo(Foo: x)` looks as though the method were being passed to itself, and it gives the reader no clue
what the value is *for*.

Give the parameter a name that describes its role instead.

```csharp
// 👎 reported
public void Save(string Save) { }
public int Map<T>(T Map) => 0;

// 👍
public void Save(string path) { }
public int Map<T>(T item) => 0;
```

### Scope

- The comparison is **ordinal and case-sensitive**. A camelCase parameter that differs from the
  method only by case is idiomatic and is **not** reported:

  ```csharp
  public int Value(int value) => value; // fine — differs only by case
  ```

- Only an **ordinary method declaration** is considered. Constructors, operators, and local functions
  are out of scope — a constructor parameter that echoes its type, or a local function that echoes its
  own name, is not this rule's business.

## How to fix violations

Rename the parameter to describe what it holds. There is no automatic code fix: the rule cannot know
which meaningful name the parameter should take.
