# SST1321 — Synchronous method names should not end with 'Async'

| Item | Value |
| --- | --- |
| Category | Naming |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |
| Analyzer | `MethodNamingAnalyzer` |

## Cause

A method's name ends with the `Async` suffix, but its return type cannot be awaited, so a caller who
trusts the name will try to `await` a value that is already computed.

```csharp
// 👎 reported
public int FetchAsync() => Load();

// 👍
public int Fetch() => Load();
```

## Rule description

A method is reported when its name ends with `Async` (and is longer than the bare suffix), it is not
declared `async`, and its return type is not awaitable — that is, not `Task`, `Task<T>`, `ValueTask`,
`ValueTask<T>`, `IAsyncEnumerable<T>`, or any type that exposes a parameterless `GetAwaiter`.

This is the inverse of SST1317, which flags an awaitable method that is *missing* the suffix.

Overrides and explicit or implicit interface implementations are skipped, because their names follow
the base member. The name and return type are checked syntactically before the method symbol is bound —
a name that does not end in `Async`, an `async` method, or a syntactically task-like return type never
reaches the semantic model — so the no-diagnostic path stays cheap.

## How to fix violations

Drop the `Async` suffix so the name matches the synchronous result. The shared rename code fix updates
the declaration and every reference. If the method really should be asynchronous, change its return type
instead of its name.
