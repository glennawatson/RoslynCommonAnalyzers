# SST1419 — Remove redundant modifiers

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Partial — the modifier shapes only |
| Default | Enabled |

## Cause

A modifier, or a `checked`/`unchecked` context, has no effect where it is written.

```csharp
public partial class OnlyPart { }   // single-part 'partial'

int Length(string text)
{
    checked                          // guards no arithmetic
    {
        return text.Length;
    }
}
```

## Rule description

Three shapes are reported:

- a `partial` declaration with only one source part — there is no other part for `partial` to join;
- a `sealed` modifier that seals nothing — on a member that is not an override, or on a member of a type
  that is already sealed;
- a `checked` or `unchecked` context whose contents contain no operation an overflow check could change.

The overflow-check context only affects integer arithmetic (`+ - * /`, `%`), increment and decrement, unary
negation, and narrowing numeric casts written **directly** inside it — not the body of a method it calls, and
not floating-point or reference operations. When none of those appear beneath the context, the keyword does
nothing:

```csharp
checked(value)              // reported: no operation to check
unchecked { Log(message); } // reported: no operation to check
```

The check is deliberately conservative: any overflow-capable operation anywhere beneath the context — even
inside a nested `checked`/`unchecked` — keeps it, so a context where overflow is genuinely possible is never
reported.

```csharp
checked(a + b)             // NOT reported: the '+' can overflow
unchecked((ulong)signed)   // NOT reported: the cast can wrap
```

Genuine multipart declarations, meaningful sealed overrides, and contexts that guard real arithmetic are all
left alone.

## How to fix violations

Remove the reported modifier. The provided code fix does this for the `sealed` and `partial` shapes and
supports Fix All.

The redundant `checked`/`unchecked` context has **no code fix**: removing it means lifting its statements or
unwrapping its expression into the enclosing context — a structural rewrite, not the deletion of a token —
and the surrounding formatting is the author's to keep. Delete the keyword (and, for a block form, its
braces) by hand.
