# SST1420 — Trivial properties should be auto-implemented

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A property only reads and writes a private field that is not used elsewhere.

```csharp
private int _value;
public int Value { get => _value; set => _value = value; }
```

An expression-bodied property counts as its own getter, so it is reported too:

```csharp
private readonly int _value = 5;
public int Value => _value;
```

## Rule description

An auto-property expresses trivial storage directly and removes unnecessary state
management. A property qualifies when every accessor does nothing but read the field or
assign `value` to it, and the field is a private, single-variable, non-`volatile`,
attribute-free declaration referenced nowhere outside that property.

The rule stays quiet when:

- an accessor contains logic, or the field is read or written by another member;
- the property has no getter — an auto-property without one does not compile;
- the field is `const`, which is compile-time state rather than a backing store;
- the field's `static` modifier does not match the property's. An instance property over a
  static field shares one slot across every instance, so folding the field away would
  silently give each instance its own storage;
- the containing type is `partial`, because another part may use the field.

## How to fix violations

Convert the accessors to auto-accessors and remove the backing field. The provided code
fix performs both edits, moving any field initializer onto the property.

```csharp
public int Value { get; set; }
public int Value { get; } = 5;
```
