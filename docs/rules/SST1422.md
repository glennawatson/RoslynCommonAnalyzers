# SST1422 — Private fields used only as locals should be local variables

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Disabled (opt-in) |

## Cause

A private field is reset at the start of the only method that references it.

## Rule description

Such a field does not represent object state and can normally be local storage. The
analysis is conservative and excludes partial types, initialized fields, closures, and
fields referenced by more than one method.

The field is reported only when the first statement of its single method writes it with a
plain `=` whose right side does not read the field. A field whose value can flow in from a
previous call is left alone, because a local would lose that state. This excludes lazily
initialized caches (`_x ??= …`, `if (_x is null) _x = …`), compound assignments used as the
first access (`_x += …`), and self-referencing resets (`_x = _x + 1`).

## How to fix violations

Declare an appropriately initialized local variable in the method and remove the field.
