# SST1427 — Protected members of sealed types should not be used

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |
| Analyzer | `RedundantModifierAnalyzer` |

## Cause

A member of a sealed type is declared `protected`, but a sealed type cannot be derived from,
so `protected` grants no access beyond `private`.

```csharp
// 👎 reported
public sealed class C
{
    protected void M() { }
}

// 👍
public sealed class C
{
    void M() { }
}
```

## Rule description

`protected` only matters to derived types. A sealed type has none, so a `protected` member
is reachable exactly as if it were `private`, and the modifier is misleading. The rule reports
the `protected` token on methods, constructors, properties, indexers, events, and fields of a
sealed type.

Members that **override** an inherited member are never reported — an override cannot reduce
the accessibility of the member it overrides, so `protected` is required there. The semantic
model is consulted only after the cheap syntactic check that a `protected` modifier is present.

## How to fix violations

Make the member private by removing `protected` (for `protected internal`, this leaves
`internal`). The provided code fix removes the `protected` modifier.
