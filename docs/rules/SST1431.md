# SST1431 — Static members of a generic type should use a type parameter

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |
| Analyzer | `TypeDesignAnalyzer` |

## Cause

A static, externally visible member of a generic type has a signature that never mentions the
type's type parameters. Callers must pick an arbitrary type argument to reach it
(`Cache<int>.Clear()`), even though the argument is irrelevant.

```csharp
// 👎 reported
public class Cache<T>
{
    public static void Clear()      // 'T' appears nowhere in the signature
    {
    }
}

// 👍 — move the member to a non-generic type
public static class Cache
{
    public static void Clear()
    {
    }
}
```

## Rule description

The rule reports a `public`, `internal`, or `protected` static field, method, property, or
event of a generic type whose signature (return type, parameter types, or field/property type)
references none of the enclosing type's type parameters. Members that do use a type parameter,
and private static helpers, are left alone.

Each enclosing type parameter name is matched against the identifiers in the member's signature
types, behind the cheap static-member-of-generic-type gate.

No automatic code fix is offered, because the right move — relocating the member to a
non-generic type — depends on the surrounding design.

## How to fix violations

Reference a type parameter in the member's signature, or move the member to a non-generic
(possibly companion) type.
