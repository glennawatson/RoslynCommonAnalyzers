# SST1443 - Functions should keep nested flow easy to follow

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled; maximum 15 for functions, 3 for properties and accessors |
| Analyzer | `FunctionComplexityAnalyzer` |

## Cause

A method, constructor, operator, accessor, local function, or expression-bodied property contains more
nesting-heavy control flow than the configured maximum.

## Rule description

Nested-flow complexity weights control flow by how deeply the reader has to nest context. An `if` inside
another `if` costs more than two adjacent guard clauses. Loops, catches, conditionals, gotos, logical
operator chains, lambdas, local functions, and direct recursion all contribute to the score.

Switch expressions and wide switch statements are kept cheap: the switch itself contributes, but each arm
or case is not treated as another nested decision. Nested logic inside an arm or case is still counted.

## Configuration

```ini
stylesharp.max_cognitive_complexity = 15
stylesharp.SST1443.max_cognitive_complexity = 20

stylesharp.max_property_cognitive_complexity = 3
stylesharp.SST1443.max_property_cognitive_complexity = 5
```

The rule-specific values take precedence.

## How to fix violations

Prefer guard clauses, extract deeply nested blocks into named helpers, and keep exceptional cases out of
the main path where that makes the code easier to scan.
