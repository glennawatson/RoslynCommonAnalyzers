# SST1464 - Unwrap else after a branch that does not fall through

**Category:** Maintainability
**Severity:** Warning
**Code fix:** Removes the `else` and hoists its statements after the `if` (supports Fix All)

When the `if` branch is, or ends in, a `return`, `throw`, `break`, or `continue`
statement, control can never fall through into the paired `else`. The `else`
wrapper then only adds nesting; moving its statements to follow the `if`
statement reads straighter. An `else if` is reported too and hoists as a whole
`if` statement.

The check is purely syntactic: only the branch's direct last statement is
inspected, with no flow analysis.

In an `else if` chain every earlier arm must jump too, because the `else`'s
statements can only be hoisted past the whole chain. One arm that falls through
makes the `else` load bearing:

```csharp
if (found)              { next = value; }   // falls through
else if (TryClaim())    { return; }         // jumps
else                    { continue; }       // not reported
```

Hoisting the `continue` out would put it on the path `found` takes too — the
loop would continue where it used to fall through. The `else` is carrying the
chain rather than merely nesting it, so there is no unwrap to make.

The code fix bails out when unwrapping could change scoping: if the `else`
block declares locals or local functions while more statements follow the `if`
in the containing block, or the `if` statement is not directly inside a block,
the diagnostic is still reported but no fix is offered.

## Example

```csharp
int M(int value)
{
    if (value > 0)
    {
        return 1;
    }
    else // SST1464
    {
        return 2;
    }
}
```

Use:

```csharp
int M(int value)
{
    if (value > 0)
    {
        return 1;
    }

    return 2;
}
```
