# SST1467 - Enumerate with foreach instead of driving the enumerator by hand

**Category:** Maintainability
**Severity:** Warning
**Code fix:** Yes

A plain local initialized from a zero-argument `GetEnumerator()` call,
immediately followed by a `while` loop on that local's `MoveNext()` whose body
only reads `Current`, restates what a `foreach` statement compiles to. Beyond
the noise, the manual form loses the disposal `foreach` guarantees: when the
enumerator implements `IDisposable`, `foreach` disposes it even if the body
throws, while the hand-driven loop shown never calls `Dispose` at all.

The analyzer stays narrow so the rewrite is always safe. It skips `using`
declarations, loops whose body uses the enumerator for anything besides
reading `Current`, loops that do not immediately follow the declaration, and
enumerators that are still used after the loop.

The code fix reuses the loop's own variable when the body starts with
`var x = e.Current;` and that is the only `Current` read; otherwise it names
the iteration variable `item` and substitutes every `Current` read. When the
name `item` is already taken around the loop, the diagnostic is reported
without a fix.

## Example

```csharp
int Sum(List<int> values)
{
    var total = 0;
    var e = values.GetEnumerator();
    while (e.MoveNext()) // SST1467
    {
        var value = e.Current;
        total += value;
    }

    return total;
}
```

Use:

```csharp
int Sum(List<int> values)
{
    var total = 0;
    foreach (var value in values)
    {
        total += value;
    }

    return total;
}
```
