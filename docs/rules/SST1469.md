# SST1469 - Do not compare a value type to null

**Category:** Maintainability
**Severity:** Warning
**Code fix:** None

A non-nullable value type can never be null. Comparing one to the null literal
with `==` or `!=` still compiles, because the compiler lifts the operator over
the nullable form of the type, but the answer is fixed: `==` is always false
and `!=` is always true. A comparison that folds to a constant almost always
means the type was misread — the value was assumed to be nullable, or the
declaration is missing a `?`.

The analyzer reports primitives, enums, and user structs that define their own
equality operators. Nullable value types (`int?`), reference types, pointers,
and type parameters are never reported because null is a real value for those.

No code fix is offered because the intent is ambiguous: sometimes the
declaration should become nullable (`int?`) so the check is meaningful, and
sometimes the check itself is dead and should be removed. Only the author
knows which side is wrong.

## Example

```csharp
public bool HasStarted(DateTime start)
{
    return start != null; // SST1469 — always true
}
```

Use a nullable declaration when absence is a real state:

```csharp
public bool HasStarted(DateTime? start)
{
    return start != null;
}
```

Or drop the dead check when the value can never be absent.
