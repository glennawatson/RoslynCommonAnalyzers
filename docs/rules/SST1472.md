# SST1472 — Signatures should not declare too many parameters

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A signature declares more parameters than the configured maximum, which defaults to 7.

```csharp
public Order Create(
    string customer, string street, string city, string postcode, string country,
    decimal total, decimal tax, string currency)
{
}
```

## Rule description

A long parameter list is easy to call wrongly. Adjacent arguments of the same type are silently
swappable — nothing stops `Create(city, street, ...)` from compiling — and every caller has to
re-read the declaration to know what goes where. A list this long is usually a type asking to be
born: the parameters that always travel together belong to something.

```csharp
public sealed record Address(string Street, string City, string Postcode, string Country);

public sealed record Money(decimal Total, decimal Tax, string Currency);

public Order Create(string customer, Address address, Money money)
{
}
```

### What is measured

Methods, constructors (including primary constructors on a class or struct), delegates, local
functions, and indexers. Operators are not: their arity is fixed by the language, so there is
nothing to report and nothing to fix.

### What is not counted

A parameter the caller never writes does not count against the maximum:

- **The extension receiver.** `public static int Sum(this int[] values, ...)` is called as
  `values.Sum(...)`, so the receiver costs the caller no argument. The same is true of the
  receiver on a C# 14 `extension(...)` block, which its members never declare at all.
- **Caller-info parameters.** `[CallerMemberName]`, `[CallerFilePath]`, `[CallerLineNumber]` and
  `[CallerArgumentExpression]` are filled in by the compiler.

### What is not reported

The rule only reports a signature its author can actually change. Where the shape comes from
somewhere else, the fix belongs at that other declaration — and *that* declaration is the one
reported:

- **An override**, and an **interface implementation**, implicit or explicit. Shortening the list
  here is not possible; the base type or the interface is reported instead.
- **A P/Invoke** — `extern`, `[DllImport]` or `[LibraryImport]`. The native API dictates the
  signature.
- **A `Deconstruct` method**, whose parameters mirror the type's own state by contract.
- **A lambda or anonymous method**, whose shape the delegate it is assigned to dictates.
- **The implementing half of a partial member.** Both halves repeat the parameter list; only the
  defining half is reported, so the same problem is not stated twice.
- **A positional record**, by default — see below.

### Positional records and primary constructors

A positional record *is* the parameter object this rule asks for. Reporting
`record Address(string Street, string City, string Postcode, string Country, ...)` would be telling
you to extract a record from a record, so records are exempt by default. Set
`stylesharp.SST1472.check_positional_records = true` to measure them anyway — worth doing if your
records are large enough that they have started hiding sub-structures of their own.

A primary constructor on a **class or struct** is measured, because it is a constructor: a type
that needs ten constructor arguments has ten dependencies, and that is the thing worth seeing. If a
long list of injected dependencies is deliberate in your codebase, raise `max_parameters` rather
than suppressing the rule.

## How to fix violations

There is **no code fix**, deliberately. The fix is to introduce a type — usually a `record` — that
groups the parameters that belong together, and that is a judgement no analyzer can make. Which
parameters travel together, what the new type is called, and whether it should carry behaviour are
all design decisions; and applying the change means rewriting every call site, potentially across
projects, on a signature that may be public API. An automated rewrite here would be a large,
silent, cross-project API change with a good chance of being subtly wrong. The rule reports it and
leaves the change to you.

The usual moves, in rough order of preference:

1. **Extract a parameter object** for the arguments that always travel together.
2. **Split the method.** A long list often means the method does more than one thing, and each
   half needs only some of the arguments.
3. **Move the state.** If most parameters are the same on every call, they are the object's
   state, not the call's — put them on the type.

## Configuration

```ini
# Rule-specific
stylesharp.SST1472.max_parameters = 7
stylesharp.SST1472.check_positional_records = false
stylesharp.SST1472.count_optional_parameters = true

# Project-wide
stylesharp.max_parameters = 7
stylesharp.check_positional_records = false
stylesharp.count_optional_parameters = true
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `max_parameters` | positive integer | `7` | The most parameters a signature may declare. |
| `check_positional_records` | `true`, `false` | `false` | Measure a positional record's parameter list. |
| `count_optional_parameters` | `true`, `false` | `true` | Count a parameter that has a default value. |

Optional parameters count by default: a long tail of them is still a long signature to read, and
still a call site that can drift. Set `count_optional_parameters = false` when only the required
arguments should count — a reasonable choice for an API whose optional arguments are genuinely
rare at the call site. Caller-info parameters are never counted either way.

An unset or unparsable value falls back to the default, so a typo neither disables the rule nor
turns every three-parameter method into a diagnostic.

## Performance

The clean path is a parameter count and one cached per-tree option lookup. Excluding a parameter
can only lower the total, so the declared count is an upper bound: a signature already inside the
limit is rejected before any parameter is inspected. The exclusions are then applied on syntax
alone, and only a signature still over the limit afterwards is bound to a symbol — so clean code
never touches the semantic model. An EventPipe CPU trace of the clean corpus confirms it: no bind
frame appears at all, and the analyzer accounts for 0.0% of sampled duration.
