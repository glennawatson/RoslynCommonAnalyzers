# SST1473 — Floating-point values should not be compared for exact equality

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes — for the NaN comparisons only |
| Default | Enabled |

## Cause

A `float` or `double` is compared with `==` or `!=`, or any value is compared against `float.NaN` /
`double.NaN`.

```csharp
public bool IsComplete(double progress) => progress == 1.0;

public bool IsMissing(double value) => value == double.NaN;
```

## Rule description

Binary floating-point arithmetic rounds. A `double` stores a value in base 2, and most decimal
fractions have no exact base-2 form, so two numbers that are mathematically equal often differ in
their last bits. `0.1 + 0.2 == 0.3` is **false**, and no amount of care at the call site changes
that — the equality is asking a question the type cannot answer.

```csharp
private const double Tolerance = 1e-9;

public bool IsComplete(double progress) => System.Math.Abs(progress - 1.0) < Tolerance;
```

NaN is worse, and it is a different bug. NaN compares **false** against every value under every
operator — including against itself, and including against NaN:

| Expression | Result | |
| --- | --- | --- |
| `x == double.NaN` | always `false` | even when `x` *is* NaN |
| `x != double.NaN` | always `true` | even when `x` *is* NaN |
| `x < double.NaN` | always `false` | as does every other relational operator |
| `x == x` | `false` **only** when `x` is NaN | |
| `x != x` | `true` **only** when `x` is NaN | |

So a comparison against `NaN` never does what it reads as doing, and the last two lines are the
deliberate idiom for detecting NaN without saying so. Both are reported, and both have a fix.

```csharp
public bool IsMissing(double value) => double.IsNaN(value);

public bool IsUsable(double value) => !double.IsNaN(value);
```

### What is not reported

- **`decimal`.** It is a base-10 type and its arithmetic is exact for the values it can represent,
  so `price == total` means what it says. `decimal` is never reported, whatever the configuration.
- **A comparison against a literal zero**, by default — see below.
- **Relational comparisons.** `left < right` on doubles is a perfectly good question, and rounding
  does not make it meaningless. Only `<`, `<=`, `>`, `>=` *against NaN* are reported.
- **A comparison of any other type.** `int`, `string`, an enum, a boxed `object` — none of them
  round, and none is reported.

### Zero comparisons

`value == 0.0` is allowed by default. It is not asking whether two computed values landed on the
same bits; it is asking a different question — *is this negative*, *was this ever assigned*, *is
this the identity element* — and zero is one of the few values a `double` represents exactly. The
same goes for `value != 0`, `value == 0f` and `value == default`.

Set `stylesharp.SST1473.allow_zero_comparison = false` to report those too. It is the right choice
when your zeroes are computed rather than assigned: a sum that *should* cancel out to zero is
exactly the case rounding ruins.

## How to fix violations

**For the general case there is no code fix, deliberately.** The replacement for `a == b` is
`Math.Abs(a - b) < tolerance`, and the tolerance is a property of your problem, not of the code:
it depends on the magnitude of the values, on how many operations they have been through, and on
what "close enough" means to the caller. Picking one silently would be picking wrong. The usual
moves:

1. **Compare against a tolerance** — an absolute one for values of a known scale, or a relative one
   (`Math.Abs(a - b) <= Math.Max(Math.Abs(a), Math.Abs(b)) * epsilon`) when the scale varies.
2. **Change the type.** If the values are money, counts, or anything where an exact answer is the
   requirement, they should be `decimal` or an integer, and the comparison then means what it says.
3. **Compare the input, not the output.** Often the exact value that was *assigned* is still around
   and can be compared before the arithmetic that broke it.

**For NaN there is a code fix**, because the intent is unambiguous:

| Reported | Fix |
| --- | --- |
| `x == double.NaN` | `double.IsNaN(x)` |
| `x != double.NaN` | `!double.IsNaN(x)` |
| `x == x` | `!double.IsNaN(x)` |
| `x != x` | `double.IsNaN(x)` |

Note the inversion in the last two rows, and that it runs the other way from the first two. `x == x`
is true for every value *except* NaN, so it already means `!IsNaN(x)`; `x == double.NaN` is
false for every value *including* NaN, so what it was written to ask is `IsNaN(x)`.

The `x == x` rewrite preserves the value exactly and only says out loud what the code already did.
The `x == double.NaN` rewrite deliberately **changes behaviour**, from an expression that is always
false into the test the author was reaching for. That is the point of the rule: the original cannot
have been what was wanted.

The fix is offered on a `float` as `float.IsNaN`, and is not offered when the operand is nullable
(`double?` has no `IsNaN` overload) or when the operands do something — `Next() == Next()` is
reported as an exact comparison, but it is not a self-comparison, and rewriting it would collapse
two calls into one.

## Configuration

```ini
# Rule-specific
stylesharp.SST1473.allow_zero_comparison = true

# Project-wide
stylesharp.allow_zero_comparison = true
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `allow_zero_comparison` | `true`, `false` | `true` | Leave a comparison against a literal zero (`0`, `0.0`, `0f`, `0x0`, `default`) alone. |

An unset or unparsable value falls back to the default, so a typo neither turns the rule off nor
starts reporting every `x == 0` in the file.

## Related rules

[SST1474](SST1474.md) reports identical operands on both sides of an operator. It defers the
floating-point `x == x` / `x != x` shape to this rule, so the NaN idiom is reported once, by the
rule that can fix it.

## Performance

The clean path never touches the semantic model. The four relational operators leave on two name
comparisons unless an operand is literally spelled `NaN`; the two equality operators first reject an
operand no floating-point value could be — a `null`, a `bool`, a string or a char literal — and then
a zero literal, whose zero-ness is read straight off the token's characters without parsing or
boxing it. Only what survives all of that is bound, and the per-tree settings are read only once a
zero literal has actually been seen. The structural self-comparison test runs last of all, on the
report path, where its cost buys the code fix.
