# SST1474 — Identical expressions should not appear on both sides of an operator

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Both operands of a binary operator are the same expression, for an operator where that is either a
constant or a no-op.

```csharp
public bool IsValid(Rect rect) => rect.Width > 0 && rect.Width > 0;

public int Range(int[] values) => values[values.Length - 1] - values[values.Length - 1];
```

## Rule description

`x == x` answers a constant. `a && a` is `a`. `value - value` is zero. None of them is worth
writing, and none of them is usually what was typed on purpose — the second operand was meant to be
something else, and a copy-paste left it as a duplicate of the first. The bug hides well precisely
because the line looks symmetrical and deliberate.

```csharp
public bool IsValid(Rect rect) => rect.Width > 0 && rect.Height > 0;
```

### Which operators are reported

| Operator | With identical operands it is | |
| --- | --- | --- |
| `==` `<=` `>=` | always `true` | |
| `!=` `<` `>` | always `false` | |
| `&&` `\|\|` `&` `\|` | the operand itself | the second one does nothing |
| `-` `^` | zero | |
| `/` | one | or a divide-by-zero |
| `%` | zero | or a divide-by-zero |

`+` and `*` are **not** reported: `x + x` doubles and `x * x` squares, and both are perfectly
ordinary things to write. Shifts are not reported either — `value << value` is odd, but it is not
constant, and the rule only claims operators whose result is fixed by the duplication.

### Only side-effect-free operands

`Next() == Next()` calls two different things that happen to be spelled the same way, so it is left
alone. An operand is compared only when every part of it is known to compute nothing: a name, a
literal, `this`, a member-access chain, a cast, `!`/`~`/`-`/`+`, and any combination of those in
parentheses. An **invocation**, an `await`, a `++`/`--`, an assignment, an object creation, or an
**element access** anywhere inside an operand disqualifies the whole comparison.

Element access is excluded on purpose: `a[i] == a[i]` runs an indexer the rule cannot see, and
reading it twice is not provably the same as reading it once.

```csharp
public bool Advance() => Next() == Next();        // clean — two calls

public bool Cached(int i) => _items[i] == _items[i];  // clean — an indexer runs twice
```

### Floating point belongs to SST1473

`x != x` on a `double` is **not** a copy-paste mistake: it is the deliberate idiom for detecting
NaN, which is the one value that is not equal to itself. So an `==` or `!=` whose operands are a
`float` or a `double` is left to [SST1473](SST1473.md), which reports it with the fix that suits it
— `double.IsNaN(x)`. Reporting it here as well would state the same span twice, with the wrong
advice.

The deferral is limited to equality. `value < value` and `value - value` on a `double` are still
reported here: NaN or not, neither of them is an idiom.

`decimal` is not deferred. It is exact and has no NaN, so `price == price` on a decimal is the plain
mistake this rule is for.

## How to fix violations

There is **no code fix**. The rule has found that one operand is wrong, but only you know which one,
and what it was supposed to be. Read the line, work out which of the two was meant to differ, and
change it. If the answer is that neither was — that the expression really is constant — delete it.

The compiler's own CS1718 covers the narrowest slice of this (`x == x` on a simple variable). This
rule covers the operators CS1718 does not, and operands — member chains, casts, parenthesized
sub-expressions — that it does not look at.

## Configuration

None. There is nothing to tune: an operator with identical, side-effect-free operands is either a
constant or a no-op, and neither is ever intended.

## Performance

The clean path never binds and rarely walks. Two operands of different syntax kinds cannot be the
same expression, so the overwhelmingly common `a == b` is rejected on a single integer compare
before anything walks a subtree. The structural comparison runs only for same-kind operands, and it
is preceded by the purity test, which is cheaper and is what keeps `Next() == Next()` from being
walked at all. The one semantic query — the floating-point check that hands the NaN idiom to
SST1473 — is reached only by an operator whose operands have *already* been proved identical, so a
clean file never pays for it.
