# SST1475 — A condition should not be repeated

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A condition repeats one that was already tested — in the same `if`/`else if` chain, `switch`
statement, or `switch` expression, or in the `if` statement immediately before this one.

```csharp
if (value < 0)
{
    return Sign.Negative;
}
else if (value > 0)
{
    return Sign.Positive;
}
else if (value < 0)   // SST1475: repeats line 1; the branch it guards can never run
{
    return Sign.Zero;
}
```

## Rule description

The first matching condition wins. Once `value < 0` has been tested and rejected, testing it again
cannot succeed, so the branch it guards is dead code — and the value it was meant to return is never
produced. This is what a copy-pasted branch looks like when the condition never got edited: the
author meant `value == 0` and wrote `value < 0`.

The same reasoning covers a `switch`: two labels that select on the same case mean the second one is
unreachable, because the first already took the value.

```csharp
switch (state)
{
    case State.Ready when IsQueued(job):
        return Start(job);
    case State.Ready when IsQueued(job):   // SST1475: repeats the label above
        return Requeue(job);
    default:
        return Skip(job);
}
```

The diagnostic is reported on the **later**, repeated condition, and the message names the line
of the earlier one — that is the branch that actually runs, and the one to compare against when
working out what the duplicate was supposed to say.

### Two adjacent `if` statements

The same condition tested twice in a row is also reported — but it means something different, and
the message says so:

```csharp
var total = 0;
if (ready)
{
    total = 1;
}

if (ready)   // SST1475: repeats line 2; both tests run, so merge them — or one condition is wrong
{
    total = 2;
}
```

Nothing here is unreachable: both bodies run whenever `ready` is true. That is exactly what makes it
worth reporting. Either the two `if` statements were meant to be one — in which case the second
condition is noise the reader has to re-verify — or the second condition was meant to say something
else, and was never edited after the block was copied. The rule cannot tell you which; it can tell
you that one of the two is true.

This shape is judged much more carefully than a chain, because between two sequential `if` statements
something actually **runs**: the first one's body. If that body can change what the condition reads,
the second test is not a duplicate at all — it is a fresh question about a value that has since
moved. So an adjacent pair is reported only when the first `if` (body **and** `else`):

- contains no call, allocation or `await` anywhere. Any of these can reach code that writes any field
  the condition reads, and the rule will not try to prove otherwise; and
- writes nothing but plain, non-`ref` **local variables and parameters** whose names the condition does
  not mention. A write to a property runs a setter, which can do anything; a write through a `ref`
  local can be an alias for the very field the condition reads; and a write to a variable the
  condition reads is the clearest possible reason for the second test to answer differently.

Everything else stays quiet:

```csharp
// Not reported: Reset() may well set _ready to false.
if (_ready)
{
    Reset();
}

if (_ready)
{
    Finish();
}
```

```csharp
// Not reported: the first branch writes the very thing the condition reads.
if (ready)
{
    ready = false;
}

if (ready)
{
}
```

Only an **immediately adjacent** pair counts. Any statement between the two — even one the rule
could have analyzed — is a reason for the condition to have changed, and the rule does not go
looking for reassurance.

### What is compared

Conditions are compared **structurally**, ignoring whitespace and comments. `value==0` and
`value  ==  0` are the same condition; the formatting is not what makes them a duplicate.

Four constructs are examined:

- **`if` / `else if` chains.** Each condition is compared against every condition above it in the
  same chain. A nested `if` starts a chain of its own: it is guarded by the outer condition rather
  than sequenced after it, so repeating that condition is redundant but not unreachable, and is not
  reported here.
- **Adjacent `if` statements** in the same block, as described above.
- **`switch` statements.** Each case label is compared against every label above it, guard clause
  included. `case 1 when flag:` repeats only `case 1 when flag:`, never `case 1:` — those test
  different things.
- **`switch` expressions.** Each arm's pattern and guard are compared against those of the arms above
  it. The value an arm produces is not part of the comparison: two arms that select the same case are
  a bug whatever they return, because only the first one can run. The discard arm (`_`) is skipped —
  it is the catch-all, not a test.

### What is not reported

**A condition with side effects.** Evaluating a condition twice only provably answers the same way if
evaluating it cannot change anything:

```csharp
if (Reader.MoveNext())
{
    return Reader.Current;
}
else if (Reader.MoveNext())   // Not reported: the second call may well answer differently
{
    return Reader.Current;
}
```

A condition is disqualified when it contains an invocation, an `await`, an assignment, an increment
or decrement, or an object creation — anywhere inside it, including inside a `when` guard. Telling
the author that the second `Reader.MoveNext()` branch is unreachable would simply be wrong. This
applies to the adjacent-`if` shape too: `if (IsReady(x)) … if (IsReady(x)) …` is **not** reported,
because `IsReady` is a call and the rule cannot know that calling it twice answers the same way.

## How to fix violations

There is **no code fix**, deliberately — for either shape. The rule can prove the two conditions are
the same; it cannot know which of the two things you meant, and that is the only information that
matters.

For a chain or a `switch`, deleting the unreachable branch would silently discard the case the author
was trying to handle, which is worse than leaving the duplicate in place.

For an adjacent pair, merging the two bodies into one `if` looks safe and mostly is — but it is the
wrong default. It silences the warning while **cementing the likelier bug**: if the second condition
was meant to differ, the merge makes the mistake permanent and removes the evidence. (It is also not
mechanically free: merging two blocks merges two scopes, so two locals of the same name collide, and
a `return` in the first body turns the second body into unreachable code. Both break the build.)

The fix is one of:

1. **Correct the condition.** Almost always the intent: the duplicated branch was copied and its
   condition never updated.
2. **Merge or delete**, once you have established that the two really are the same test — the merge
   for an adjacent pair, the deletion for a chain.
3. **Reorder the chain**, if the duplicate is a narrower case that the earlier condition swallows.

## Configuration

None. The rule reports only what it can prove, so there is nothing to tune.

## Performance

The clean path is a pointer walk. Only the head of a chain is analyzed — an `else if` is the
statement of its predecessor's `else` clause and is rejected on a single parent check — so a
five-branch chain is walked once, not five times. The chain is then re-entered through its `else`
pointers rather than copied into a list, which keeps a two-branch `if`/`else if` (the overwhelmingly
common shape) allocation-free. The block scan for adjacent pairs is one indexed pass over a block's
statements that reads no condition at all until it finds two `if` statements side by side. The
structural comparison runs first and bails on the first difference; the side-effect scan runs only
behind it, and the semantic model is consulted only after two conditions have already been proven
identical — so clean code never touches it.
