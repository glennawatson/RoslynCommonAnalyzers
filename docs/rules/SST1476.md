# SST1476 — Conditional branches should not have identical bodies

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Every branch of a conditional construct has the same body, so the condition changes nothing about
what happens.

```csharp
if (user.IsAdmin)
{
    return Load(user, includeArchived: true);
}
else
{
    return Load(user, includeArchived: true);   // SST1476: the condition decides nothing
}
```

## Rule description

Code that branches is code that claims the branches matter. When they are identical, the claim is
false: either the condition is dead weight and should go, or — far more often — one branch was meant
to differ and the edit was never made. The `includeArchived: true` above was almost certainly
supposed to be `false` in the non-admin case, and the bug is invisible precisely because the code
*looks* like it handles both.

Five constructs are examined:

- **`if` / `else`** where both bodies are identical.
- **`if` / `else if` / … / `else` chains** where every body is identical.
- **Conditional (ternary) expressions** whose two arms are identical.
- **`switch` statements** where every section — `default` included — runs the same statements.
- **`switch` expressions** where every arm produces the same value.

The diagnostic is reported once, on the construct's keyword (the `if`, the `switch`, the `?`), not
on each duplicated branch: the construct is the thing that is wrong, not any one of its branches.

### Only exhaustive constructs are reported

An `if` with no `else`, and a `switch` with no `default`, are a **different shape** and are never
reported:

```csharp
if (value < 0)
{
    Log(value);
}
else if (value > 100)
{
    Log(value);
}
// No else. A value in 0..100 does nothing — which is not what the written branches do.
```

The unwritten branch does nothing, and doing nothing is not the same as doing what the other
branches do. The condition still decides something, so there is nothing to report. Only when the
construct covers every input — a trailing `else`, a `default` label — does "every branch is the
same" collapse into "the condition is irrelevant".

For a `switch` expression, exhaustiveness is the compiler's own answer rather than a search for a
discard arm, so a complete `bool` or enum switch counts even when it never spells out `_`:

```csharp
// Reported: both inputs are covered, and both produce the same value.
public string Describe(bool flag) => flag switch
{
    true => "unknown",
    false => "unknown",
};
```

### What is compared

Bodies are compared **structurally**, ignoring whitespace and comments. A body is read as its
statement list, so braces are not part of the comparison either — `if (c) Run(); else { Run(); }` is
the same duplicate as the fully braced pair.

## How to fix violations

There is **no code fix**, deliberately. Collapsing the construct is not a safe rewrite and is
usually not the right one:

- **It can change behaviour.** Removing the `if` also removes the evaluation of its condition, and a
  condition that calls a method, awaits, or assigns is doing work that the collapse would silently
  delete.
- **It is probably not what you want.** The overwhelmingly common cause is that one branch was
  supposed to differ. An analyzer that "fixed" the code by deleting the condition would erase the
  evidence of the bug rather than the bug — and it has no way to know which branch was meant to be
  the odd one out.

So the fix is a judgement:

1. **Make the branch that was meant to differ actually differ.** Start here; it is nearly always the
   answer.
2. **Delete the condition**, keeping one copy of the body, if the branches really are the same and
   the condition has no side effects.
3. **Merge the labels.** For a `switch`, `case 1: case 2: default:` in one section says
   deliberately what identical bodies say by accident.

## Configuration

```ini
# Rule-specific
stylesharp.SST1476.minimum_statements = 1

# Project-wide
stylesharp.minimum_statements = 1
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `minimum_statements` | positive integer | `1` | The smallest branch body, in statements, that counts as a duplicate. |

At the default of `1`, even a single-statement body counts — `if (c) return 0; else return 0;` is
reported, and so it should be. Raise it to `2` to suppress the trivially short duplicates, which is
worth doing in a codebase where one-line branches that coincide are common and usually intentional:

```ini
stylesharp.SST1476.minimum_statements = 2
```

A body that is a single expression — a conditional expression's arm, a `switch` expression's arm —
counts as one statement, so raising the minimum above `1` drops those shapes along with the one-line
`if`.

Two empty bodies never reach the default minimum, so `if (c) { } else { }` is not reported here; an
empty block is [SST1439](SST1439.md)'s business.

An unset or unparsable value falls back to the default, so a typo neither silences the rule nor
changes which constructs it reports.

## Performance

The clean path is a pointer walk. Exhaustiveness is settled first and syntactically — a trailing
`else`, a `default` label — so a non-exhaustive construct is rejected before a single body is read.
Only the head of an `if` chain is analyzed (an `else if` is the statement of its predecessor's `else`
clause and is rejected on a parent check), so a five-branch chain is walked once, not five times. The
body comparison is trivia-insensitive, bails on the first difference, and never copies a statement
list. Options are read, and the semantic model consulted for a `switch` expression's exhaustiveness,
only once a construct has already been proven duplicated — which means clean code neither allocates
nor binds.
