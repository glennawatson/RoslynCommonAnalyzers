# SST1477 — Integer division should not be assigned to a floating-point target

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

Two integers are divided, and the truncated result is then widened to `float`, `double` or `decimal`.

```csharp
public double SuccessRate(int passed, int total) => passed / total;
```

With 3 of 4 tests passing, this returns `0.0`.

## Rule description

`/` on two integers is integer division: it computes the quotient and throws the remainder away. The
widening happens **afterwards**, on a number from which the fraction has already been removed, so
there is nothing left for `double` to hold. `1 / 2` stored in a `double` is `0`, not `0.5`.

The mistake is easy to make because the *declaration* looks right — the variable really is a
`double`, the method really does return one — and nothing in the expression says otherwise. The type
of the target is not what decides the arithmetic; the types of the operands are.

Cast one operand and the other is promoted to match it, so the division itself runs in floating
point:

```csharp
public double SuccessRate(int passed, int total) => (double)passed / total;
```

### What counts as widening

The rule does not enumerate contexts. Every one of them is the same fact in the semantic model — the
division's converted type is a floating-point type while its own type is integral — so asking that one
question covers all of them:

```csharp
double a = x / y;                     // an initializer
_ratio = x / y;                       // an assignment to a double field
return x / y;                         // a return from a double member
Math.Sqrt(x / y);                     // an argument bound to a double parameter
total += x / y;                       // a compound assignment
flag ? x / y : 0.0;                   // a branch whose common type is double
(double)(x / y);                      // an explicit cast — it truncates before it widens
```

The explicit cast is worth calling out. `(double)(x / y)` casts the *quotient*, not an operand, so it
converts a number that has already been truncated. It is reported for exactly the same reason as the
implicit conversions, and the fix moves the cast to where it does some good.

### What is not reported

- **A division that already runs in floating point.** `x / 2.0`, `(double)x / y`, `x / (double)y` —
  one operand is floating point, so the other is promoted and nothing truncates.
- **A division that stays integral.** `int pages = total / size;` is integer division on purpose,
  which is fine — this rule fires on the widening, not on the division.
- **A user-defined `/`.** An operator on `BigInteger` or a custom numeric type has no built-in
  semantics the rule can reason about.
- **A nullable division.** `int? / int?` is a lifted operator with no special type; the rule does not
  measure it.
- **A native integer.** `nint` and `nuint` divide integrally, and are reported like any other integral
  type — their width does not matter here, only that they truncate.

### The boundary

The rule reports the expression the conversion is actually attached to. In `(a / b) + 1` assigned to
a `double`, the conversion applies to the *sum*, not to the division, so nothing is reported — even
though the truncation is just as real. This is deliberate: chasing a truncated value up through
arbitrary integer arithmetic means guessing at where the author expected precision to appear, and
guessing produces false positives. What the rule promises is precise and easy to reason about: if a
division is handed straight to something that wanted a fraction, it says so.

## How to fix violations

Cast one operand — the code fix casts the left one:

```csharp
// before
public double SuccessRate(int passed, int total) => passed / total;

// after
public double SuccessRate(int passed, int total) => (double)passed / total;
```

When the left operand is itself an operation, the fix keeps it grouped, so the arithmetic that was
already integral stays integral:

```csharp
// before
double scale = a * b / c;

// after — a * b still multiplies as integers; only the division changes
double scale = (double)(a * b) / c;
```

When the division was already wrapped in an explicit cast to the same type, the fix replaces that
cast rather than nesting a second one inside it, adding parentheses only where an operator could bind
to the promoted division more tightly than `/` does:

```csharp
// before
double average = (double)(total / count);
double scaled = 2.0 * (double)(total / count);

// after
double average = (double)total / count;
double scaled = 2.0 * ((double)total / count);
```

If the truncation was intentional, say so — assign the integer division to an integer first, and let
the widening happen on a value that has already been rounded on purpose:

```csharp
int whole = total / count;
double result = whole;
```

## Configuration

None. There is nothing to tune: a division either truncates into a floating-point target or it does
not.

## Performance

The clean path costs one bind. A real literal on either operand — `x / 2.0`, `x / 1e3`, `x / 2f` — is
rejected on the token's text without touching the semantic model, and a hexadecimal literal is thrown
out first so its digits are never mistaken for an exponent or a suffix. Everything else is bound
once, which yields the division's own type and its converted type together; a division that is not
being widened stops there. Only a division that really is widened pays for anything more, and only a
reported one allocates.
