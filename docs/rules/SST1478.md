# SST1478 — Shift counts should be within the operand's width

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `<<`, `>>` or `>>>` has a compile-time constant count that is `0`, negative, or at least as large as
the operand's width.

```csharp
public int ClearAll(int flags) => flags << 32;   // returns flags, unchanged
public int Nothing(int flags) => flags >> 0;     // does nothing at all
```

## Rule description

C# does not shift a value out of existence. Before the shift runs, the count is **masked** to the
operand's width — the low 5 bits for a 32-bit operand, the low 6 bits for a 64-bit one — so the count
wraps around instead of saturating:

| Written | On an `int` | Actually shifts by | Result |
| --- | --- | --- | --- |
| `x << 32` | `32 & 31` | `0` | `x`, unchanged |
| `x << 33` | `33 & 31` | `1` | `x << 1` |
| `x << -1` | `-1 & 31` | `31` | `x << 31` |
| `x << 0` | — | `0` | `x`, unchanged |

None of these do what they look like they do. `x << 32` reads as "shift every bit out and leave
zero", and instead hands back `x` untouched — a bug that survives review precisely because the code
says the right thing. A shift by `0` is a no-op, and usually means a loop bound or an offset is one
short.

### Width, and the promotion trap

The width that matters is the width of the operand **the shift actually runs on**, which is not
always the width of the operand as written. The built-in shift operators exist only for `int`,
`uint`, `long` and `ulong`. Anything narrower — `byte`, `sbyte`, `short`, `ushort`, `char` — is
promoted to `int` first, and is therefore 32 bits wide by the time it is shifted:

```csharp
byte b = 0x0F;

int wide = b << 8;    // fine. b is an int here, and 8 is well inside 32.
int wrong = b << 32;  // reported. 32 is out of range even for the promoted int.
```

This is the case the rule most easily gets wrong in the obvious direction. `b << 8` on a byte is not
"shifting an 8-bit value by 8" — the result is an `int`, all eight bits are still there, and the
expression is entirely correct. Only `long` and `ulong` get the wider 64-bit limit.

| Operand type | Width used |
| --- | --- |
| `sbyte`, `byte`, `short`, `ushort`, `char`, `int`, `uint` | 32 (promoted to `int`/`uint`) |
| `long`, `ulong` | 64 |
| `nint`, `nuint` | not reported |

`nint` and `nuint` are never reported: their width is a property of the process the code runs in, not
of the program, so the analyzer cannot know whether a count of 32 is in range.

### What is not reported

- **A computed count.** `x << count` and `x << (count + 1)` are only checkable at run time; the rule
  reports constants only.
- **A count inside the operand's width**, which is the overwhelming majority of shifts.
- **A native integer operand**, as above.

### Relationship to SST1481

`x << 0` and `x >> 0` are also identity operations, and SST1481 reports identity operands on `&`, `|`
and `^`. Zero shifts belong to **this** rule and this rule only: it knows the operand's width and can
say what the shift actually does. SST1481 does not register shift expressions at all, so the two never
report the same expression.

## How to fix violations

There is **no code fix**. The expression is wrong, but the analyzer cannot know what it was meant to
be — whether `x << 32` should have been `x << 31`, `0`, a `long` shift, or something else entirely
depends on what the code is for. Read the surrounding arithmetic and decide.

The usual causes:

1. **An off-by-one bound.** `for (var i = 0; i <= 32; i++) mask |= 1 << i;` — the last iteration
   masks back to `1 << 0` and sets bit 0 again.
2. **The wrong operand type.** A shift by 32 or more is often a `long` operation written on an `int`:
   `1L << 40` works, `1 << 40` does not.
3. **A no-op that was meant to be a real shift**, usually where the distance came from a table or a
   constant that turned out to be zero.

## Configuration

```ini
# Rule-specific
stylesharp.SST1478.allow_zero_shift = false

# Project-wide
stylesharp.allow_zero_shift = false
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `allow_zero_shift` | `true`, `false` | `false` | Allow a shift by a constant `0`. |

A shift by zero is reported by default because it does nothing, and the count it was meant to be is
usually one away. Set `allow_zero_shift = true` where a zero is written deliberately to keep a table
of shift distances visually regular:

```csharp
// with allow_zero_shift = true, the first term is left alone
int packed = (a << 0) | (b << 8) | (c << 16) | (d << 24);
```

The setting only affects a count of zero. A negative or out-of-range count is always reported: there
is no style in which those are deliberate.

An unset or unparsable value falls back to the default, so a typo does not silently turn half the
rule off.

## Performance

The clean path costs nothing at all in the common case. No operand this rule measures is narrower
than 32 bits, so a plain literal count of 1 through 31 is in range whatever it is shifting, and is
cleared on the token's digits without the semantic model ever being asked a question — which is what
almost every shift in real code looks like. Only a count outside that window is bound, only a count
that turns out to be constant is measured against the operand's width, and the `.editorconfig`
settings are read only for a count that has already turned out to be zero.
