# SST1479 — Count and length comparisons should be satisfiable

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes — replaces the comparison with the constant it folds to (supports Fix All) |
| Default | Enabled |

## Cause

A count or a length is compared against a value it can never take, so the comparison is decided before it
runs.

```csharp
if (items.Count >= 0)      // SST1479 — always true
{
    Process(items);
}

if (buffer.Length < 0)     // SST1479 — always false
{
    Recover();
}

if (name.Length == -1)     // SST1479 — always false
{
    Fail();
}
```

## Rule description

`Count`, `Length` and the result of `Enumerable.Count()` are never negative. That makes `count >= 0` a test
that always passes and `count < 0` a test that never does — the guard reads like a check, but it decides
nothing, and whatever it was meant to check is not being checked.

The mistake is almost always a near-miss:

```csharp
if (items.Count >= 0)   // meant: > 0, or >= 1
if (index < 0)          // meant: an index — a Count was pasted in
if (name.Length == -1)  // meant: IndexOf(...) == -1
```

That last one is the common one. `IndexOf` returns `-1` for "not found", so `-1` is a natural thing to
compare against — but a *length* is not an index, and the `-1` came from the wrong line.

### What counts as a count

The rule reads the members the framework guarantees to be non-negative, resolved through the semantic model
rather than by name:

- **`Array.Length`** and `Array.LongLength`.
- **`string.Length`**.
- **`Span<T>.Length`** and **`ReadOnlySpan<T>.Length`**.
- **The `Count` that satisfies `ICollection<T>` or `IReadOnlyCollection<T>`.** Asking the interface rather
  than the type is what makes `List<T>`, `Dictionary<,>`, `HashSet<T>`, an interface-typed variable, and any
  collection you write yourself all follow from one rule.
- **`Enumerable.Count()`** and `Enumerable.LongCount()`, in both their extension and static forms.

The operands may be written in either order: `0 <= items.Count` says exactly what `items.Count >= 0` says,
and is reported the same way.

### What is not reported

- **A comparison that still asks a question.** `count > 0`, `count == 0`, `count != 0` and every comparison
  against a positive value are left alone — those are the tests the reported ones were meant to be.
- **A `Count` that satisfies no collection interface.** Nothing says a user-defined `Count` is non-negative;
  a running balance or a delta is a perfectly reasonable thing to call `Count`, and it can be negative.
- **A count read through a conditional access.** `items?.Count >= 0` produces an `int?`, and a null one fails
  the comparison, so the fold does not hold.
- **Anything that merely looks like a count.** `text.IndexOf('a') >= 0` and `value >= 0` fold to a constant
  only if their operand is non-negative — which neither is — so they are never reported, and never bound.

## How to fix violations

The code fix replaces the comparison with the constant it always evaluates to:

```csharp
if (true)                 // was: if (items.Count >= 0)
{
    Process(items);
}
```

That is deliberately where the fix stops. Collapsing the surrounding `if` is a judgement about what the
author meant — the guard is usually salvageable (`>= 1`), not deletable — so the constant is left in place,
in the shape that makes the mistake obvious to whoever reads it next. Change it to the test you meant:

```csharp
if (items.Count > 0)
{
    Process(items);
}
```

## Configuration

None. There is nothing here to tune: a count is non-negative or it is not, and the members the rule trusts
are the ones the framework guarantees.

## Performance

The clean path never touches the semantic model. The bound is classified from the literal's token text —
digits only, so no value is boxed — and the comparison is folded on the operator alone, which rejects
`count > 0` and every other satisfiable test immediately. Only a comparison that *already* folds to a
constant is matched against the count-member names, and only one that clears that gate is bound. So
`value >= 0` and `text.IndexOf('a') >= 0` both fold, both fail the name gate, and neither reaches the
semantic model; `list.Count > 0`, the shape that actually appears everywhere, dies one step earlier still.
