# SST1480 — A constructed exception should be thrown

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes — inserts the missing `throw` (supports Fix All) |
| Default | Enabled |

## Cause

An exception is constructed as a whole statement, so its value is discarded.

```csharp
public void Process(Order order)
{
    if (order is null)
    {
        new ArgumentNullException(nameof(order));   // SST1480 — never thrown
    }

    order.Submit();   // runs anyway, on a null order
}
```

## Rule description

Constructing an exception does nothing. It allocates an object, captures no stack trace, and returns — and
then the statement ends and the object is collected. The guard above reads exactly like a guard, and does
exactly nothing: the `throw` was forgotten, the error path falls straight through, and the failure surfaces
later as a `NullReferenceException` somewhere that has no idea what went wrong.

It is an easy line to write. `throw` is the only word that separates a correct guard from a silent one, and
it is the one word that carries no meaning of its own.

```csharp
public void Process(Order order)
{
    if (order is null)
    {
        throw new ArgumentNullException(nameof(order));
    }

    order.Submit();
}
```

### What is reported

An object creation whose parent is an expression statement — the creation *is* the statement — and whose
type derives from `System.Exception`. That is the whole rule. Both the explicit `new SomeException(...)` and
the target-typed `new(...)` form are recognized.

### What is not reported

An exception whose value something consumes is not a mistake — it is an exception factory, and every shape of
one is left alone:

- **`throw new X()`**, obviously.
- **`return new X();`** — the classic factory method, and `Exception Build() => new X();` with it.
- **Assignment**, to a local, a field, or a property: `var error = new X();`.
- **An argument**: `Fail(new X())`, `tcs.SetException(new X())`, `new AggregateException(new X())`.
- **A lambda body**: `Func<Exception> factory = () => new X();`.
- **A collection initializer or an array**: `new List<Exception> { new X() }`.

Most of these fall out of the parent check for free — none of them is an expression statement — which is also
why the rule costs so little.

A discarded assignment, `_ = new X();`, is **not** reported either. The creation's parent is the assignment,
not the statement, and writing a discard is a deliberate act: the author put something on the left of the
exception, which is not the mistake this rule is about.

A target-typed `new(...)` cannot in fact stand alone as a statement — an expression statement offers it no
target type, so the compiler rejects it — but the rule handles the form anyway rather than depending on that
staying true.

## How to fix violations

The code fix inserts the missing `throw`, keeping the exception, its arguments, and the statement's
indentation:

```csharp
throw new ArgumentNullException(nameof(order));
```

The forgotten `throw` is the overwhelmingly likely intent, so the fix restores it rather than deleting the
statement. If the line really was dead, deleting it afterwards is a keystroke; if it was a guard, deleting it
would have lost the exception and the argument that names the parameter.

## Configuration

None. An exception that nothing consumes is a bug in every codebase.

## Performance

The clean path never touches the semantic model. The only test that runs for the overwhelming majority of
object creations is a type check on the parent node — is this creation the entire statement? — which is false
for virtually every `new` in a compilation. Only a creation that already looks like the bug is bound to
confirm it derives from `Exception`, and `System.Exception` itself is resolved lazily, so a compilation that
never reaches the bind never pays for the metadata lookup either.
