# SST1481 — Bitwise operations should not use identity operands

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes (except `x & 0`) |
| Default | Enabled |

## Cause

A bitwise operation has a constant operand that cannot change the result.

```csharp
public int Combine(int flags) => flags | 0;    // returns flags
public int Mask(int flags) => flags & 0;       // always returns 0
```

## Rule description

The operation reads as though it does something. It does not — and that gap is the point of the rule.
Nobody writes `flags | 0` on purpose; they write it because the constant on the right was supposed to
be a flag, or a mask, or a shifted bit, and something upstream produced a zero instead.

| Shape | Result | Reported |
| --- | --- | --- |
| `x \| 0`, `0 \| x` | `x` | yes |
| `x ^ 0`, `0 ^ x` | `x` | yes |
| `x & ~0` (that is, `x & -1`) | `x` | yes |
| `x & 0`, `0 & x` | `0`, always | yes, without a fix |

Only **constant** operands are examined. `flags & mask` is a real operation whatever `mask` turns out
to hold at run time, and is never reported.

### `x & 0` has no code fix, deliberately

`x & 0` is always zero. Rewriting it to `0` would preserve the behaviour exactly — and would take a
bug that the compiler is currently showing you and bake it into the source as though it were intended.
Whether the author meant a different mask, meant to remove the expression, or meant something else
entirely is not something an analyzer can know. So the rule reports it and stops. This is the one
shape it will not touch.

### All bits set, and the promotion trap

"All bits set" is decided against the type the **operation** runs on, not the type of the operand as
written. `&`, `|` and `^` promote their operands, so the only result types a built-in integral
operation can have are `int`, `uint`, `long` and `ulong`.

That matters most for the narrowing mask everybody writes:

```csharp
byte b = ReadByte();
int masked = b & 0xFF;   // not reported
```

`b` is promoted to `int` here, so the operation is a 32-bit one and all-bits-set for it is `-1`, not
`0xFF`. The mask is not an identity for the operation, and the rule leaves it alone.

The same reasoning makes a sign-extended constant work correctly across widths:

```csharp
long value = ...;
long same = value & -1;                 // reported: -1 is all 64 bits set here
ulong unsigned = ...;
ulong also = unsigned & ulong.MaxValue; // reported: the same constant, written the other way
```

### What is not reported

- **Booleans.** `flag | false` is a short-circuiting question, and belongs to
  [SST1468](SST1468.md). This rule is restricted to integral operations.
- **Enums.** `access | Access.None` has an enum result type, not an integral one.
- **A non-identity constant.** `x | 1`, `x & 0x0F`, `x ^ -1` (which is a bitwise NOT) all do
  something.
- **A non-constant operand.** `x & mask`, `x | Next()`, `x & masks[0]`.
- **Native integers.** `nint` and `nuint` have a platform-dependent width, so all-bits-set is not a
  constant the rule can name.
- **Zero shifts.** `x << 0` and `x >> 0` are the same kind of mistake, but they belong to
  [SST1478](SST1478.md), which knows the operand's width and can say what the shift actually does.
  This rule does not register shift expressions at all, so the two never report the same expression.

## How to fix violations

The code fix removes the operation and keeps the operand that survived it:

```csharp
// before
public int Combine(int flags) => flags | 0;

// after
public int Combine(int flags) => flags;
```

The surviving operand never needs parentheses added: it was already a direct operand of the removed
operator, so it binds at least as tightly as that operator did, and taking the operator's place
cannot change how anything around it groups. Parentheses that already wrapped the operation are
dropped only when what survives is a single primary expression, which nothing can regroup:

```csharp
// before                            // after
a & (b | 0)                          a & b
a & (b ^ c | 0)                      a & (b ^ c)   // a & b ^ c would mean (a & b) ^ c
```

For `x & 0` there is no fix — see above. Work out what the mask was meant to be.

## Configuration

None. An identity operand is an identity operand.

## Performance

The clean path is two cheap rejections. An operand that is a call, an index or a `new` can never be
constant, so an operation with no constant-shaped operand at all is dropped on syntax alone, without
the semantic model. The operation is then bound once to learn its type, which throws out every
boolean, enum, native-integer and user-defined operator before a constant value is ever asked for.
Only an operation that could plausibly have an identity operand pays for reading one, and only a
reported one allocates.
