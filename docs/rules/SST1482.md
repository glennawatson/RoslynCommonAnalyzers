# SST1482 — GetHashCode should not read mutable state

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `GetHashCode()` override reads state that can change after the object has been hashed.

```csharp
public class Order
{
    public int Id { get; set; }

    public string Customer { get; set; } = "";

    public override int GetHashCode() => HashCode.Combine(Id, Customer);
}
```

## Rule description

A hash-based collection puts an object in a bucket when it is added, and looks in *that* bucket when
it is asked for it again. If a field the hash reads is reassigned afterwards, the object's hash moves
but the object does not — and it can no longer be found. Not by an equal object, and not even by a
reference to itself:

```csharp
var order = new Order { Id = 1, Customer = "acme" };
var set = new HashSet<Order> { order };

order.Id = 2;              // the hash moved; the object did not

set.Contains(order);       // false — the object cannot find itself
set.Remove(order);         // false — and it can never be removed
```

The object is now a permanent leak in the collection: it is in there, it is reachable from there,
and nothing can get it back out. Nothing throws, so the damage shows up much later, somewhere else.

The fix is to hash only state that is fixed for the object's lifetime.

```csharp
public class Order
{
    public Order(int id, string customer)
    {
        Id = id;
        Customer = customer;
    }

    public int Id { get; }

    public string Customer { get; }

    public override int GetHashCode() => HashCode.Combine(Id, Customer);
}
```

If the object genuinely has to be mutable, hash a stable identity instead — a `readonly` key assigned
in the constructor — or do not put the type in a hash-based collection at all.

## What counts as mutable

The rule asks one question of each member the hash reads: **can it hold a different value later than
it did when the object was hashed?**

| Member | Mutable? |
| --- | --- |
| Field, neither `readonly` nor `const` | **Yes** — reported |
| `readonly` field | No |
| `const` field | No |
| `static readonly` field | No |
| Auto-property with a `set` accessor (including `private set`) | **Yes** — reported |
| Property with a `set` accessor of any kind | **Yes** — reported |
| Property with an `init` accessor | No — construction closes it |
| Get-only auto-property (`{ get; }`) | No |
| Computed get-only property (`=> ...`) | No |

`init` is the interesting one. An `init`-only property is writable exactly once, during construction
and object-initialization, and is closed from then on — which is precisely the guarantee the hash
needs. So `public int Id { get; init; }` is safe to hash, and a positional `record` — whose
properties are all `init`-only — is safe by construction.

A `static` field that is not `readonly` is judged the same way as an instance field. It is not this
object's state, but the hash it feeds is: reassign it and *every* object hashed against it is lost at
once. It is the same bug from further away.

## What is not reported

- **A parameter or a local.** Neither is state; a local computed inside the hash is fine.
- **State on another object.** `_next.Value` is that object's business, not this one's. Only
  `this` — implicitly, as `this.`, or as `base.` — names state this rule cares about.
- **`base.GetHashCode()` and `HashCode.Combine(...)`.** Neither is a field or a property, so the rule
  looks straight *through* them and inspects their arguments instead, which is exactly where the
  mutable reads hide. `HashCode.Combine(_id, _name)` reports `_id` and `_name`, not `Combine`.
- **`nameof(_field)`.** It yields a name at compile time and never reads the field.
- **An object initializer's member names.** `new Box { Width = _size }` names a member of the `Box`
  being built. The *value* is still inspected, so `_size` is reported if it is mutable.
- **A method not named `GetHashCode`, or one that is not an override.** Only the parameterless
  `public override int GetHashCode()` is analyzed.

### A known limit

The rule looks at the members the hash *names*, not at what those members go on to read. A computed
get-only property is therefore treated as fixed:

```csharp
private int _count;                         // mutable

public int Fingerprint => _count * 31;      // get-only, so treated as fixed

public override int GetHashCode() => Fingerprint;   // not reported
```

Chasing the read through the property body would mean following an arbitrary call graph from an
analyzer that runs on every keystroke. The rule stops at the first hop; the price is a miss on
indirection like this, and the benefit is that it costs nothing on code that is fine.

## How to fix violations

There is **no code fix**, deliberately. Every route out of this diagnostic is a design decision:

1. **Make the state `readonly`** — the usual answer, when the value never really needed to change.
   Not automatable: it may not compile (something assigns it) and it changes the type's contract.
2. **Hash a stable identity instead** — a `readonly` key assigned in the constructor. The analyzer
   cannot know which of the members is the identity.
3. **Make the property `init`-only**, if it was only ever set during construction.
4. **Keep the type out of hash-based collections**, and delete the override.

Which of these is right depends on what the type is *for*, and an automated rewrite that guessed
would either fail to compile or silently change equality semantics.

## Configuration

None. There is nothing to tune: the question of whether a member can change is answered by the
language, not by a threshold.

## Performance

The whole rule sits behind a syntactic prepass: the name is `GetHashCode`, the parameter list is
empty, and the `override` modifier is present. That is false for every method in a file but the one
hash override, and it is answered from tokens alone — no symbol is touched and nothing is allocated.

Only inside a real hash override does the walk run, and it is a hand-written preorder scan over
`ChildNodesAndTokens()` rather than `DescendantNodes()`, so there is no iterator to allocate. The
walk also decides what *not* to descend into, and that is what keeps the binds down: the member half
of an access on another object, the operand of a `nameof`, the member names of an object initializer
and a property pattern's name all name something other than this object's state, and none of them is
ever bound.
