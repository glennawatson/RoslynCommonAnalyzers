# SST1483 — Constructors should not call overridable members

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A constructor calls a member that a derived type can still override, dispatched on `this`.

```csharp
public class Control
{
    protected Control() => Render();     // dispatches to the derived override

    public virtual void Render()
    {
    }
}
```

## Rule description

Construction runs base-first. The base constructor finishes *before* a single derived field
initializer has run — but a virtual call made from it still dispatches to the derived override. The
override therefore runs against an object that does not exist yet:

```csharp
public class Button : Control
{
    private readonly string _caption;

    public Button(string caption)
    {
        _caption = caption;              // 3. …and only now is _caption assigned
    }

    public override void Render()
    {
        Console.WriteLine(_caption.Length);   // 2. NullReferenceException: _caption is null
    }
}

var button = new Button("OK");           // 1. Control's constructor calls Render()
```

`_caption` is `readonly`, and it is `null`. That is the tell: the invariant the type is built on is
not true yet, because the code that establishes it has not run. Nothing in `Control` looks wrong, and
nothing in `Button` looks wrong — the bug only exists in the seam between them, and only appears once
somebody derives.

The fix is to not do work in the constructor that a derived type gets to redefine. Move it to an
initialization method the caller invokes, seal the member, or seal the type.

```csharp
public class Control
{
    protected Control()
    {
    }

    public virtual void Render()
    {
    }
}

var button = new Button("OK");
button.Render();                         // fully built, then rendered
```

## What counts as overridable

Wider than `virtual`. The question is whether a type further down the chain can still replace the
implementation:

| Member | Overridable? |
| --- | --- |
| `virtual` | **Yes** — reported |
| `abstract` | **Yes** — reported (there is nothing else it *can* dispatch to) |
| `override` | **Yes** — reported; overriding a member does not close it |
| `sealed override` | No — `sealed` is what closes it |
| Any member of a `sealed` type | No |
| Any member of a `struct` or `record struct` | No — implicitly sealed |
| `private`, `static`, or plain non-virtual | No — the language does not allow it to be overridden |

An `override` being reported is the case people miss. `override void Render()` in a middle class is
still open: a type below it can override `Render` again, so calling it from the middle class's
constructor has exactly the original problem, one level down.

`object`'s own virtual members count too. `ToString()` in a constructor is a virtual call, and a
derived type that overrides `ToString` will see a half-built object.

## What is reported

- **A method call** on `this`, implicitly or explicitly: `Render()` and `this.Render()`.
- **A property read or write** on `this`. Naming a property *is* calling it — every read runs the
  getter and every write runs the setter, and both can be overridden.
- **An event subscription** (`+=` / `-=`) that runs an overridable accessor.
- **Any of the above in a constructor body**, block-bodied or expression-bodied.

## What is not reported

- **A `base.` call.** `base.Render()` is emitted as a *non-virtual* call: it always lands in the base
  implementation and can never reach a derived override. It is the fix, not the bug.
- **A call on another instance.** `other.Render()` is somebody else's object, fully built.
- **A method group.** `_handler = Render;` builds a delegate that dispatches when something later
  invokes it — which is not during construction.
- **A lambda or a local function declared in the constructor.** Both run when they are invoked, which
  may be long after the constructor has returned.
- **`nameof(Render)`.** It yields a name at compile time and dispatches nothing.
- **An object initializer's member names.** `new Box { Width = w }` names a virtual member of the
  `Box` being built, not of `this`.
- **A field-like event inside the type that declares it.** Within its declaring type, a field-like
  event *is* its backing field: `Changed += handler` combines delegates in the field directly and
  never calls `add_Changed`, so there is no dispatch for a derived type to intercept. The same event
  subscribed from a *derived* constructor does call the accessor, and is reported.
- **A `sealed` type**, including one whose `sealed` sits on a different `partial` half.
- **A static constructor**, which has no `this` to dispatch on.

## Primary constructors and initializers

Nothing to report — and not because the rule declines to look. The C# language makes it impossible:

```csharp
public class Widget(int size) : Control(Render())   // CS0120: an object reference is required
{
    private readonly int _area = Compute();         // CS0236: a field initializer cannot reference
                                                    //         the non-static member 'Compute'
}
```

A constructor initializer's arguments (`: base(...)` / `: this(...)`), a field or property
initializer, and a primary constructor's base arguments are all evaluated at a point where `this` does
not exist yet, so the compiler rejects *any* instance member there — CS0027, CS0120, CS0236. A lambda
in a field initializer is rejected the same way. There is consequently no virtual call for those
positions to make, and the rule does not walk them.

A primary constructor has no body, so it has nowhere to put the call either. A regular constructor
declared alongside one is an ordinary constructor and is analyzed normally.

That leaves the constructor *body* as the only place in the language where construction can dispatch
virtually — which is exactly what this rule walks.

## How to fix violations

There is **no code fix**, deliberately. The routes out are all design decisions:

1. **Seal the type.** If nothing was ever meant to derive from it, `sealed` states that and the
   problem disappears. A breaking change if the type is published.
2. **Seal the member.** `sealed override` closes one member without closing the type.
3. **Move the call out of the constructor** into an `Initialize()` method the caller invokes, or into
   a static factory that constructs and then initializes. This changes the type's usage contract.
4. **Make the member non-virtual** and give derived types a different extension point.

Which is right depends on why the member was virtual in the first place — a judgement no analyzer can
make, and an automated rewrite that guessed at it would silently change the type's inheritance
contract.

## Configuration

None. Whether a derived type can override a member is answered by the language, not by a threshold.

## Performance

The clean path is a modifier scan. A `sealed` class, a `struct`, a `record struct` and a static
constructor are all rejected from tokens alone, before any symbol is touched — and the `sealed` case
is the common one in code that has thought about inheritance at all. A `sealed` modifier can sit on a
different `partial` half, so that is confirmed against the constructor's symbol afterwards, which the
analyzer driver has already materialized and which therefore costs no bind.

Past the prepass the constructor body is walked with a hand-written preorder scan over
`ChildNodesAndTokens()` rather than `DescendantNodes()`, so there is no iterator to allocate, and the
walk decides what not to descend into: a lambda, a local function, the operand of a `nameof`, the
member half of an access on another object, an object initializer's member names and a property
pattern's name are all skipped without ever being bound.
