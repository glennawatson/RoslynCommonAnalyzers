# SST1484 — Declarations should not shadow an outer field or property

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A local, a parameter, or a pattern variable reuses the name of a field or property that is already visible
where it is declared — or a nested type declares a field or property that reuses the name of a containing
type's static member.

```csharp
public sealed class Order
{
    private decimal total;

    public void Recalculate(Line[] lines)
    {
        var total = 0m;

        foreach (var line in lines)
        {
            total += line.Amount;
        }

        // Nothing was recalculated. 'total' is the local; the field is still 0.
    }
}
```

## Rule description

Inside `Recalculate`, the name `total` means the local. Every unqualified use of it — including the one the
author meant for the field — silently addresses the local instead, and the method quietly does nothing. The
compiler is happy: both names are legal, and the assignment is a real assignment to a real variable. Only a
reader who already knows the field exists can see the bug.

That is the whole rule: a declaration that reuses a visible member's name makes the name ambiguous to a
reader and unambiguous, in the wrong direction, to the compiler.

```csharp
public sealed class Order
{
    private decimal total;

    public void Recalculate(Line[] lines)
    {
        var sum = 0m;

        foreach (var line in lines)
        {
            sum += line.Amount;
        }

        total = sum;
    }
}
```

### What is measured

Every declaration that introduces a name into a member's body:

- **Locals**, including `using`, `fixed`, `for` and `const` locals.
- **Parameters** — of a method, a constructor, an indexer, a local function, or a lambda.
- **Loop variables**, in both `foreach` forms.
- **Pattern variables**, **`out` variables**, and **deconstruction** variables.
- **Catch variables**.
- **Nested type members** — a field or property a nested type declares.

The name is compared against the fields and properties of the containing type **and of its base types**,
which is exactly the set C# resolves a simple name against. An inherited member is reported as an
`inherited field` or `inherited property`; a **private** member of a base type is invisible to the derived
type, so a name that only matches one of those is clean.

A nested type's member is measured against the **static** fields and properties its containing types make
visible, walking outward the way C# resolves the name — the nearest containing type that claims the name
answers. The confusion is the same one, moved to type scope:

```csharp
public class Registry
{
    public static int Count;

    public class Entry
    {
        private int Count;   // reported: inside Entry, 'Count' now means the entry's field

        public bool IsFull() => Count > 100;   // reads the entry's field, not the registry's counter
    }
}
```

### What is not reported

- **The constructor idiom.** `this.name = name;` is how C# construction is written — see below.
- **A primary constructor's parameters**, a positional record's included. The language scopes them over the
  whole type body deliberately, and backing a member is why they exist.
- **A local in a static member that reuses an instance field's name.** The field is not in scope there, so
  nothing is ambiguous. The same applies inside a `static` local function or a `static` lambda.
- **A name that only matches a member of an unrelated type.** `Defaults.Limit` is not in scope by simple
  name, so a local called `Limit` shadows nothing — and a sibling nested type's members are just as
  unrelated as any other type's.
- **A nested type's member that matches an instance member of a containing type.** An outer instance member
  is not reachable by simple name from a nested type, so nothing is ambiguous — the same reasoning as the
  static-member exemption above.
- **An explicit interface implementation.** It has no simple name to collide with.
- **A nested type's member marked `new` or `override`.** `new` says the hiding is deliberate, and an
  `override` keeps the name its base declared — neither name was this declaration's choice.
- **A discard.**
- **A field that hides an inherited field**, unless `check_base_types` is on — and never when it is marked
  `new`, which says the hiding is deliberate.

### The constructor idiom

This is the false positive the rule cannot afford, and it is exempted by asking one question: **does the
parameter feed the member it shadows?** If the declaration assigns the member, the parameter exists to fill
that member in, and the code is unambiguous about it.

```csharp
public sealed class Customer
{
    private readonly string name;

    // All of these are clean.
    public Customer(string name) => this.name = name;
}

public sealed class Account
{
    private readonly string name;
    private readonly int age;

    public Account(string name)
        : this(name, 0)                        // forwarded to another constructor — clean
    {
    }

    public Account(string name, int age)
        => (this.name, this.age) = (name, age);  // tuple assignment — clean
}
```

Phrasing the exemption this way rather than special-casing constructors means it also covers the shapes that
are the same idea under a different name: a setter (`void SetName(string name) => this.name = name;`), an
initialize method, and the bare `name = name;` spelling. What stays reported is a parameter that shadows a
member and never stores it — which is the case where a reader cannot tell which one the body meant.

## How to fix violations

There is **no code fix**, deliberately. The fix is a rename, and which name to change is a judgement call the
analyzer should not make for you: renaming the local may collide with something else in the method, and
renaming the field is an edit to the type's state that can reach beyond the file. Two ways out, in rough
order of preference:

1. **Rename the inner declaration.** `total` → `sum`, `name` → `candidateName`. The member keeps the name it
   earned; the local gets one that says what it is in this method.
2. **Qualify the member.** `this.total` says which one you meant and removes the ambiguity for a reader —
   but the name is still doing double duty, so prefer the rename where you can. For a nested type's member
   the qualification is the containing type's name: `Registry.Count`.

## Configuration

```ini
# Rule-specific
stylesharp.SST1484.check_base_types = false

# Project-wide
stylesharp.check_base_types = false
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `check_base_types` | `true`, `false` | `false` | Also report a field that hides a field of the same name inherited from a base type. |

`check_base_types` covers the *declaration* case rather than the *usage* case:

```csharp
public class Base
{
    protected int count;
}

public class Derived : Base
{
    private int count;   // reported only when check_base_types = true
}
```

Two fields, one name, and every unqualified `count` in `Derived` now means the derived one. It is off by
default because hiding is occasionally deliberate — and when it is, marking the field `new` says so, and the
rule stays quiet even with the option on.

Note that a **local** shadowing a visible inherited field is reported either way. The option is about a field
declaration hiding another field, not about whether inherited members participate at all.

An unset or unparsable value leaves the option off, so a typo cannot light up a whole hierarchy.

## Performance

The clean path never touches the semantic model. The set of names a type makes visible — its own fields and
properties, plus the ones it inherits and can still see — is built once per type declaration and cached on
the declaration node, so a local costs a parent walk to its type and two hash probes. Only the first
declaration inside a type pays for the type symbol at all; every later one is answered from the cache. Only a
name that actually collides pays for anything more: the static-context walk and the assignment scan that
clears the constructor idiom both run after the name has already matched, which in real code is almost never.
A nested type's field or property is measured against its containing types' cached tables the same way, and
a member of a non-nested type is rejected by a single parent probe before anything is looked up.
