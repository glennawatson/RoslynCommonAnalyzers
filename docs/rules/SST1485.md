# SST1485 — Members that must not throw should not throw

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A member that the runtime or the framework invokes on the caller's behalf throws an exception.

```csharp
public sealed class Session : IDisposable
{
    public void Dispose()
    {
        if (_transaction.IsOpen)
        {
            throw new InvalidOperationException("Commit or roll back before disposing.");
        }
    }
}
```

## Rule description

Nobody writes a `try` around the call, because nobody writes the call. A hash-based collection calls
`GetHashCode` and `Equals`; a debugger, a logger and an interpolated string call `ToString`; a `using` block
calls `Dispose`; the runtime calls a static constructor and a finalizer; and an implicit conversion runs at a
call site that does not even mention it. An exception from any of them surfaces somewhere the cause is no
longer visible, and the caller has nowhere to put the `catch`.

`Dispose` is the worst of them. A `using` block calls it **while an exception is already unwinding**, so a
throw there replaces the original exception with this one — the failure that actually mattered is gone, and
what gets logged is the cleanup complaining about it.

```csharp
public sealed class Session : IDisposable
{
    public void Dispose()
    {
        if (_transaction.IsOpen)
        {
            _transaction.Rollback();   // decide, do not throw
        }
    }
}
```

A static constructor is nearly as bad: its exception is wrapped in a `TypeInitializationException` and raised
at whichever unrelated line happened to touch the type first, which may be nowhere near the configuration
that was actually wrong. A finalizer is worse still — an unhandled exception on the finalizer thread ends the
process.

### What is measured

| Member | Why the caller cannot defend |
| --- | --- |
| `Equals(object)`, `Equals(T)` | Called by every collection, comparer and assertion. |
| `GetHashCode()` | Called on insert and on lookup, from inside the collection. |
| `ToString()` | Called by the debugger, by logging, and by string interpolation. |
| `Dispose()`, `DisposeAsync()` | Called by `using`, often while another exception is unwinding. |
| A static constructor | Called by the runtime; becomes a `TypeInitializationException` far from its cause. |
| `==`, `!=`, `<`, `>`, `<=`, `>=` | A comparison in the middle of a sort has nowhere to put a `catch`. |
| An **implicit** conversion operator | Runs at a call site that never mentions it. |
| A finalizer (`~T()`) | Runs on the finalizer thread; an exception there kills the process. |

Arity is part of the identity: `Dispose(bool disposing)` is the disposal pattern's helper, not the member a
`using` block calls, so it is left to the `Dispose()` that calls it. `ToString(string format)` is likewise an
ordinary overload the caller wrote out.

### What is not reported

- **`NotImplementedException` and `NotSupportedException`.** These do not report a failure; they state a
  contract. `NotImplementedException` is a placeholder, and `NotSupportedException` is how an implementation
  says a member does not apply to it. Both are idiomatic, and a type that derives from either says the same
  thing.
- **A throw inside a nested lambda or local function.** The delegate runs when it is invoked, which need not
  be during the member that declares it, so its throw is not the member's.
- **A bare `throw;` in a `catch`.** It propagates the exception that is already unwinding rather than
  originating one — the member is passing on a failure it did not cause, and swallowing that would be worse
  than the problem this rule guards against. Constructing a *new* exception in that same `catch` **is**
  reported: in `Dispose`, that is precisely how the original exception gets lost.
- **An explicit conversion operator.** The caller wrote the cast, so the conversion — and its failure — is
  visible at the call site. Only the implicit ones are invisible.
- **An ordinary member.** Throwing is how a normal method reports a bad argument or a broken invariant, and
  its caller can see the call and catch it.

## How to fix violations

There is **no code fix**, deliberately. What a member should do instead of throwing depends entirely on what
it means, and that is not something an analyzer can decide. The moves, in rough order of preference:

1. **Return a sentinel instead.** `Equals` returns `false`, `GetHashCode` returns a constant, `ToString`
   returns a placeholder. A member with no way to signal failure should answer, not fail.
2. **Make `Dispose` idempotent and total.** Release what is there and return. If a caller must decide
   something before cleanup, give them a method that says so — `Commit()`, `Rollback()` — and let `Dispose`
   clean up whatever state it finds.
3. **Move the validation out of the static constructor.** Fail where the caller can see it: at the first real
   call, or in an explicit initialization method.
4. **Make the conversion explicit**, or replace it with a `TryParse`-shaped method that returns success. If a
   conversion can fail, the caller should be able to see that it is happening.
5. **If the member genuinely cannot be implemented**, say so with `NotSupportedException` — which the rule
   allows, because that is a contract rather than a failure.

## Configuration

```ini
# Rule-specific
stylesharp.SST1485.additional_members = OnError, OnCompleted

# Project-wide
stylesharp.additional_members = OnError, OnCompleted
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `additional_members` | comma-separated names | *(empty)* | Extra member names to treat as must-not-throw. |

Use it for the members your own framework invokes implicitly — an observer's `OnNext`/`OnError`, an event
handler, a callback that runs on a thread with no catch above it. The names are matched exactly, on any
arity, so `additional_members = OnError` covers every `OnError` in the compilation.

## Performance

A member is rejected on its name. Almost every declaration in a compilation is not one of these members, and
answering that costs a string comparison against five names plus — when the option is set at all — one scan
of the configured list, which is in-place and allocates nothing. Only a member whose name matched has its
body walked, and the walk is an indexed child scan rather than `DescendantNodes()`, so it steps over lambdas
and local functions instead of descending into them. The thrown type is bound last: `throw new
NotImplementedException()` is recognized on its syntax before the semantic model is touched at all, so even
the common exemption usually costs nothing.
