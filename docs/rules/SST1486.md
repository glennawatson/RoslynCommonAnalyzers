# SST1486 — Repeated string literals should be named constants

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

The same string literal is written three or more times in one file.

```csharp
public sealed class OrderStore
{
    public Order? Find(IDictionary<string, Order> cache, string id)
        => cache.TryGetValue("orders/" + id, out var order) ? order : null;

    public void Save(IDictionary<string, Order> cache, Order order)
        => cache["orders/" + order.Id] = order;

    public void Remove(IDictionary<string, Order> cache, string id)
        => cache.Remove("orders/" + id);
}
```

## Rule description

A string written out several times has to be changed in several places, and a typo in one copy
compiles cleanly and fails at runtime — which is the whole problem, because the compiler cannot
tell `"orders/"` from `"order/"`. Naming the value once makes the change atomic and gives the
value a meaning that the literal alone does not carry.

```csharp
private const string OrderPrefix = "orders/";
```

This is the string counterpart of [SST1471](SST1471.md), and it works the same way: the literal is
not the problem, the *unnamed* literal repeated across the file is.

The rule reports **once**, on the **first** copy — the one whose value should become the constant —
and the message states how many copies exist. It does not squiggle every copy and say the same
thing N times.

Duplicates are counted **per file**. The same literal in two different files is two different
decisions about where the constant should live, and this rule does not make that call for you.

Literals are compared by **value**, not by spelling, so `"a\tb"` and `@"a<TAB>b"` and the raw-string
form of the same value are three copies of one value with three homes.

## What is not reported

A repeated literal is left alone wherever repeating it is idiomatic, or wherever it already has a
name:

- **A literal shorter than five characters.** A name for `""` or `","` buys nothing. Configure with
  `minimum_string_length`.
- **The empty string, and a literal that is nothing but whitespace** — whatever the minimum length
  is set to.
- **An attribute argument.** `[Obsolete("use CreateAsync instead")]` on twenty members is twenty
  members correctly documented, not twenty duplicates.
- **A `nameof` operand.** `nameof` produces a name, not a repeated value.
- **The initializer of a `const` field, a `const` local, or a `static readonly` field.** That
  declaration *is* the named constant this rule asks for; asking for it again would be circular. A
  plain instance field, a property initializer, and a default parameter value are **not** exempt:
  nothing has named those.
- **A `case` label, and a `switch` expression arm's pattern.** The switch already names the value
  structurally. The value an arm *produces* is not a label, so a repeated one is still reported.
- **An interpolated string's text segments.** `$"user {id} not found"` is a template, not a value
  that can be hoisted into a constant. A string literal written *inside* an interpolation hole —
  `$"{Lookup("user-key")}"` — is an ordinary literal and is counted like any other.
- **Anything in a preprocessor directive**, and anything in generated code.

A literal inside a lambda does not inherit the name of the field the lambda is assigned to, so
`static readonly Func<string> F = () => "value";` repeated three times still reports `"value"`.

## How to fix violations

Introduce a `const` (or `static readonly`) field and use it everywhere. Where to put it — the type,
a shared constants class, a resource file — is your call, which is why there is deliberately **no
code fix**.

## Configuration

```ini
# Rule-specific
stylesharp.SST1486.duplicate_string_threshold = 3
stylesharp.SST1486.minimum_string_length = 5

# Project-wide
stylesharp.duplicate_string_threshold = 3
stylesharp.minimum_string_length = 5
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `duplicate_string_threshold` | integer `>= 2` | `3` | The number of copies of one literal at which the file is reported. |
| `minimum_string_length` | integer `>= 1` | `5` | The shortest literal that counts. |

An unset or unparsable value falls back to the default, and so does a nonsensical one: a threshold
of `1` would report every literal in the file and a length of `0` would report every `""`, so
neither is accepted. A typo can never turn this rule into one that shouts at everything.

## Performance

The rule runs as a single syntax-tree action, because counting duplicates is inherently a per-file
aggregation and a per-node action would have to share mutable state across callbacks to do it.

The file is walked twice. The first walk is **allocation-free**: it counts the literals that could
be reported and folds each one's hash into a 64-bucket filter. If fewer literals survive the
exclusions than the threshold needs, or if no two of them ever landed in the same bucket, then
nothing in the file can repeat and the analyzer returns having allocated nothing at all. Only a
file that gets past that filter pays for the second walk and the dictionary it fills.

The filter is sound in the direction that matters: two copies of one value always collide, so an
empty collision mask *proves* every literal is distinct. It has false positives — sixty-four
buckets fill up once a file has a few dozen distinct literals — so a large file with many distinct
literals will still allocate one dictionary and count them properly. The cost is a wasted dictionary
on a big clean file, never a missed diagnostic.
