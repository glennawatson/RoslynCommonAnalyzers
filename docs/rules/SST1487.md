# SST1487 — Collection elements should not be overwritten before they are read

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

The same collection element is assigned in two adjacent statements, with nothing reading it in
between. The first value is thrown away.

```csharp
map["a"] = 1;
map["a"] = 2;   // SST1487 — the first assignment is lost

values[i] = ComputeX();
values[i] = ComputeY();   // SST1487
```

## Rule description

Writing the same slot twice in a row does nothing that writing it once would not do, so one of the
two statements is dead. That is almost never what was meant: the index or the key was supposed to
differ, and the counter that was supposed to advance did not.

```csharp
values[i] = ComputeX();
values[i + 1] = ComputeY();
```

The diagnostic lands on the **first** of the two writes — the one that does nothing — so the
squiggle marks the dead line rather than the line that killed it. Three writes to one slot report
two diagnostics, because two of the three writes are lost.

The rule is entirely **syntactic**. It reports only a literal, obvious repeat, and it never attempts
dataflow analysis:

- The two assignments must be **adjacent statements** in the same statement list — a block, or the
  body of a `switch` section.
- Both must be plain `=` assignments to an **element access** (`x[i]`).
- The **receiver** and every **index** must be the same side-effect-free expression in both.

## What is not reported

- **A compound assignment.** `sum[i] += x` and `cache[k] ??= v` read the element before they write
  it, so they can never throw a value away. Neither is an `=`, so neither is ever matched.
- **A side-effecting receiver or index.** `map[Next()] = 1; map[Next()] = 2;` writes two different
  slots as far as this rule can prove, and `values[i++] = 1; values[i++] = 2;` writes two slots for
  certain.
- **A right-hand side that reads the collection.** `v[i] = v[i] + 1;` is a read-modify-write, not a
  lost write. So, more conservatively, is `v[i] = v.Length;` and `v[i] = Sum(v);` — anything whose
  value could depend on what the first statement stored. Mentioning the collection anywhere in the
  second assignment's right-hand side is enough to keep the rule quiet.
- **A nested element access.** `grid[i][j] = 1; grid[i][j] = 2;` is left alone: the inner indexer is
  a call this rule cannot see through, so two spellings of it are not provably the same slot. This
  is a deliberate miss rather than a risky report.
- **Two writes that are only adjacent in this build.** A `#if` region between them means something
  was skipped that could have read the element, so the first write is not provably dead.
- **A repeated property or field assignment.** `target.Width = 1; target.Width = 2;` is out of
  scope. A property setter is a method, and a setter with a side effect — invalidating a layout,
  raising a change notification — is ordinary enough that reporting it would be noise. Only
  **indexers** are matched.

## How to fix violations

Look at the index. If it was supposed to advance, advance it. If the second value is the one you
wanted, delete the first statement. If the first value is the one you wanted, delete the second.

There is deliberately **no code fix**: which of the two assignments was intended is not knowable
from the code, and removing the wrong one would silently change behaviour.

## Configuration

None. There is nothing here to tune — the rule reports only the unambiguous shape.

## Performance

The scan is one indexed pass over each statement list. Every statement is classified once and the
result is carried into the next iteration, so a block of N statements costs N classifications rather
than the 2N a per-statement lookaround would cost.

The classification is a pair of type tests — is this an expression statement whose expression is a
simple assignment to an element access — which rejects essentially every statement in essentially
every block before anything else is looked at. Nothing else runs until two adjacent element writes
have been found, and no semantic model is ever touched: the rule binds nothing, allocates nothing on
the clean path, and works the same on code that does not compile.
