# SST1488 — Exception types should declare the standard constructors

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A type derives from `System.Exception` but does not declare all three of the constructors callers
expect: parameterless, message, and message plus inner exception.

```csharp
public class WidgetException : Exception // SST1488
{
}
```

## Rule description

Every exception in the framework can be constructed three ways, and code that throws exceptions is
written on that assumption. A type that only offers some of them forces its callers into contortions
at exactly the moment they are trying to report a failure.

The constructor that takes an inner exception is the one that matters most. Without it, the code
catching a low-level failure and rethrowing a domain-specific one has nowhere to put the original:

```csharp
try
{
    _store.Load(id);
}
catch (IOException error)
{
    // Nowhere to put 'error'. The cause is lost, and with it the stack trace that explains
    // what actually went wrong.
    throw new WidgetException($"Could not load widget {id}.");
}
```

Declaring the three constructors makes the wrapping throw natural, and the diagnostics that follow
from it complete:

```csharp
throw new WidgetException($"Could not load widget {id}.", error);
```

The constructors are matched by **parameter type, not by parameter name** — a constructor taking
`(string reason, Exception cause)` satisfies the rule as readily as one taking
`(string message, Exception innerException)`.

### What this rule does not ask for

It does **not** ask for the `protected T(SerializationInfo, StreamingContext)` constructor. That
constructor exists to serve `BinaryFormatter`, which modern .NET has obsoleted and removed. Adding it
back would be a step in the wrong direction — see [SST1489](SST1489.md), which reports it when your
target framework has obsoleted it.

### Accessibility

The constructors have to be reachable by the people who construct the type:

| Type | Expected |
| --- | --- |
| A normal exception | `public` constructors |
| An `abstract` exception | `protected` is correct — only derived types construct it |
| An exception not visible outside the assembly | `internal` already reaches everyone who can see it |

### Derived exception types

A derived type **does not inherit its base's constructors**, so every exception declares its own —
deriving from a well-formed exception buys nothing here, and a derived type with no constructors is
still reported.

## How to fix violations

Declare the missing constructors, each forwarding to `base`. The code fix does this for you,
including the XML documentation, because a project with the documentation rules on would otherwise
trade one diagnostic for another.

```csharp
public class WidgetException : Exception
{
    /// <summary>Initializes a new instance of the <see cref="WidgetException"/> class.</summary>
    public WidgetException()
    {
    }

    /// <summary>Initializes a new instance of the <see cref="WidgetException"/> class.</summary>
    /// <param name="message">The message that describes the error.</param>
    public WidgetException(string message)
        : base(message)
    {
    }

    /// <summary>Initializes a new instance of the <see cref="WidgetException"/> class.</summary>
    /// <param name="message">The message that describes the error.</param>
    /// <param name="innerException">The exception that is the cause of this exception.</param>
    public WidgetException(string message, Exception innerException)
        : base(message, innerException)
    {
    }
}
```

## Configuration

```ini
# .editorconfig
stylesharp.SST1488.require_parameterless = true
stylesharp.SST1488.include_non_public_types = true

# or project-wide, for every rule that reads the option
stylesharp.require_parameterless = true
stylesharp.include_non_public_types = true
```

| Option | Values | Default | Meaning |
| --- | --- | --- | --- |
| `require_parameterless` | `true`, `false` | `true` | Set `false` where every exception must carry a message. |
| `include_non_public_types` | `true`, `false` | `true` | Set `false` to check only exceptions visible outside the assembly. |

An unset or unparsable value falls back to the default, so a typo cannot quietly narrow the rule.

## When to suppress

Suppress on an exception type that is deliberately constructible only one way — a singleton sentinel,
or one built exclusively by a factory that owns the only valid arguments. Prefer
`require_parameterless = false` over a suppression when the objection is only to the parameterless
constructor.
