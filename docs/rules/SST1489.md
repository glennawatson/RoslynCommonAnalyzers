# SST1489 — Exception types should not carry formatter-based serialization members

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

An exception type declares one of the members that exist only to support formatter-based
serialization, on a target framework that has obsoleted them.

```csharp
[Serializable]
public class WidgetException : Exception
{
    // ...

    protected WidgetException(SerializationInfo info, StreamingContext context) // SST1489
        : base(info, context)
    {
    }

    public override void GetObjectData(SerializationInfo info, StreamingContext context) // SST1489
    {
        base.GetObjectData(info, context);
        info.AddValue(nameof(WidgetId), WidgetId);
    }
}
```

## Rule description

The `(SerializationInfo, StreamingContext)` constructor and the `GetObjectData` override exist to
serve `BinaryFormatter`. Modern .NET has obsoleted that whole mechanism and removed the runtime
support for it: **nothing calls these members any more.**

What is left behind is dead code that still costs. It has to be read, it has to be maintained, and —
worst of all — it has to be kept in step with every new field added to the exception, forever, in
service of a serializer that will never run. An exception type that gains a property and forgets to
add a matching `info.AddValue` looks broken to anyone reading the `GetObjectData` override, and is
not, because none of it does anything.

Deleting them is safe on these targets, and it is the only way the maintenance burden goes away.

## The target framework decides

**This rule does not fire on every framework, and that is the point.** On .NET Framework and
netstandard2.0 these members are still live — a `BinaryFormatter` round-trip really does call them,
and removing them would break real serialization.

Rather than test a framework version, the rule asks the framework itself: it looks at
`System.Exception.GetObjectData` and checks whether the framework has marked it `[Obsolete]`. Modern
.NET does; .NET Framework and netstandard2.0 do not. So:

| Target | Behavior |
| --- | --- |
| Modern .NET (the members are obsolete) | Reported |
| .NET Framework, netstandard2.0 (the members are live) | **Never reported** |

Asking the framework rather than hardcoding a version means the rule stays correct on targets that do
not exist yet, and costs nothing on a project where it does not apply — the whole rule is gated behind
that one probe.

A project multi-targeting both will see the diagnostic only in the modern target's compilation. Guard
the members with `#if` if you need to keep them for the older one.

## How to fix violations

Delete the member. The code fix does this.

```csharp
[Serializable]
public class WidgetException : Exception
{
    /// <summary>Initializes a new instance of the <see cref="WidgetException"/> class.</summary>
    public WidgetException()
    {
    }

    // ... the standard constructors, and nothing else.
}
```

The `[Serializable]` attribute is **left alone**. It still means something to other serializers, and
deciding whether your type should keep it is a judgement the fix has no business making.

## When to suppress

Suppress when the type is genuinely still round-tripped through a formatter you cannot remove yet —
for instance while a legacy component is being migrated. The suppression is the honest record that the
dependency exists.
