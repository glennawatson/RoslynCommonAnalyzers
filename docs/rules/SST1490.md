# SST1490 — Base lists should not state what is already implied

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A base list names an interface that another entry in the same list already brings in — either a base
class that implements it, or another interface that inherits it.

```csharp
public sealed class Bag<T> : IReadOnlyList<T>, IEnumerable<T>
{
}

public sealed class Worker : BackgroundService, IHostedService
{
}
```

`IReadOnlyList<T>` already inherits `IEnumerable<T>`, and `BackgroundService` already implements
`IHostedService`. Neither entry adds anything.

## Rule description

A base list is a claim about what a type *is*. When one of its entries is already implied by another,
the entry reads as though it means something — that this type implements the interface *directly*, or
that the interface matters here in a way it does not — and it hides the entries that do carry
information. The longer the list, the worse it gets: a reader who has to check three names to find
that only one of them tells them anything has been made to do work for nothing.

```csharp
public sealed class Bag<T> : IReadOnlyList<T>
{
}

public sealed class Worker : BackgroundService
{
}
```

The type still implements every interface it did before. Nothing about its public surface, its
overload resolution, or its interface dispatch changes.

### What is reported

Only an interface, and only when the reason is visible in the base list being read:

- **An interface a listed base class already implements.**
- **An interface another interface in the same list already inherits**, at any depth. In
  `: IList<T>, ICollection<T>, IEnumerable<T>` both `ICollection<T>` and `IEnumerable<T>` are
  reported, because `IList<T>` brings in both.

A base list with a single entry is never reported: there is nothing else in it that could imply the
entry.

### What is not reported

- **An explicit `object` base.** `class C : object` is reported by
  [SST1177](SST1177.md), which owns the compiler-implied base type (and an enum's implied `int`
  underlying type). It is not repeated here.
- **An interface re-implemented over a base class.** This is the one case where the redundant-looking
  entry is load-bearing:

  ```csharp
  public class Base : IRunnable
  {
      public void Run() => Console.WriteLine("base");
  }

  public sealed class Derived : Base, IRunnable   // not reported
  {
      public new void Run() => Console.WriteLine("derived");
  }
  ```

  Re-listing `IRunnable` **re-implements** it: the interface is re-mapped onto `Derived`'s own member,
  so `((IRunnable)derived).Run()` prints `derived`. Delete the entry and it prints `base` — a silent
  behavior change, not a cleanup. The same is true when the type declares an explicit implementation
  (`void IRunnable.Run()`), where deleting the entry would not even compile. So an interface implied by
  a base class is reported only when this type declares no implementation of its own.

  An **override** does not count as an implementation of its own: it is reached through the base class's
  mapping and keeps running once the entry is gone, so `Derived : Base, IRunnable` *is* reported when
  `Derived` merely overrides a virtual member.

- **Anything another part of a partial type says.** Each base list is judged on its own entries, so
  this stays clean:

  ```csharp
  public partial class C : IDerived { }

  public partial class C : IBase { }   // not reported, even though IDerived : IBase
  ```

  The rule's message says the entry is implied *by the rest of the base list*, and it means it. An
  entry whose implication lives in another file would send you looking at a list that does not explain
  itself.

## How to fix violations

Remove the entry. The code fix does it, keeping the entry that implied it; the base list always keeps
at least one entry, so it is never left dangling.

## Configuration

None.

## Performance

The clean path is a syntax test: a type with no base list, or with a single entry, is rejected without
touching the semantic model. Past that, entries are bound one at a time inside the comparison rather
than gathered into an array first, so the pass allocates nothing — base lists are two or three entries
long and the semantic model caches each bind. The interface-mapping check, which is the expensive part,
runs only for an entry that is *already known to be redundant* and only when the entry that implies it
is a base class.
