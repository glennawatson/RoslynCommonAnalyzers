# SST1491 — Modifiers should not restate the default

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A modifier names what the declaration already is.

```csharp
public interface IService
{
    public int Value { get; }        // interface members are public
    public abstract void Run();      // ...and abstract, with no body
    virtual int Fallback() => 0;     // ...and virtual, with one
}

public readonly struct Point
{
    public readonly int X => 0;      // every instance member of a readonly struct is readonly
}
```

## Rule description

A modifier that restates the default adds a keyword the reader must check and discard. Worse, it
invites the opposite mistake: `public abstract void Run();` inside an interface reads as though
`abstract` were a choice being made here, so a reader who later sees a member *without* it can
reasonably conclude the two differ. They do not.

It also hides a real coupling. `readonly int X => 0;` inside a `readonly struct` looks like a property
that was deliberately marked non-mutating — but if the struct's own `readonly` is removed, the member's
modifier silently changes from decoration to load-bearing.

```csharp
public interface IService
{
    int Value { get; }
    void Run();
    int Fallback() => 0;
}

public readonly struct Point
{
    public int X => 0;
}
```

## What is reported

A modifier is reported only when deleting it leaves code that **still compiles and still means the same
thing**. Four cases clear that bar:

| Modifier | Where | Why it is the default |
| --- | --- | --- |
| `public` | On any interface member | Interface members are public. |
| `abstract` | On an interface member with no body | An interface member with no body is abstract. |
| `virtual` | On an interface member with a body | An interface member with a body is virtual. |
| `readonly` | On an instance member of a `readonly struct` | Every instance member of a readonly struct is readonly. |
| `unsafe` | On a member already inside an `unsafe` type or member | The enclosing declaration supplies the context. |

The interface cases apply from C# 8, which is when the language first allowed an interface member to
state an accessibility at all. Below that they cannot occur.

## What is not reported

This is where the rule earns its keep, so the reasoning is worth spelling out.

### `static` on a member of a static class — because it does not compile without it

It looks like the textbook case, and it is not one:

```csharp
public static class Helpers
{
    public static int Twice(int value) => value * 2;   // NOT reported
}
```

Removing `static` here is a compile error: *cannot declare instance members in a static class*. The
keyword is **required**, not redundant. A rule that offered to delete it would offer to break the build.

### `private` on an interface member — because it is not the default

```csharp
public interface IService
{
    private int Compute() => 1;   // NOT reported
}
```

Interface members are *public* by default. Dropping `private` would not tidy the member; it would widen
it into the public contract.

### `abstract` and `virtual` on a **static** interface member — because they mean something there

```csharp
public interface ICounter<TSelf>
    where TSelf : ICounter<TSelf>
{
    static abstract TSelf Zero { get; }    // NOT reported
    static virtual int Step() => 1;        // NOT reported
}
```

A static interface member is not virtual by default, and one without a body is not abstract by default —
`static abstract` and `static virtual` each say something a bare `static` does not. Only `public` stays
redundant on a static interface member, and it is still reported.

### `sealed` — because another rule already reports it

A `sealed` that cannot do anything — on a member that is not an override, or on a member of a sealed
type — is [SST1419](SST1419.md)'s report, and the same modifier under two ids helps nobody. Note that
`sealed` on an interface member with a body is not redundant at all: it makes the member non-virtual.

### `unsafe` on a declaration with no unsafe syntax — because another rule already reports it

[SST1455](SST1455.md) removes an `unsafe` from a declaration that contains no pointer, `fixed`, or
`sizeof` syntax. This rule takes only the other case: a declaration that genuinely *is* unsafe and is
already sitting inside an `unsafe` type or member, so its own modifier changes nothing.

```csharp
public unsafe class Native
{
    public unsafe int First(int* values) => *values;   // reported: the class already supplies the context
    public unsafe int Nothing() => 0;                  // SST1455's report, not this one
}
```

### An access modifier on a class member

`private`, `internal` and the rest are never reported on a class or struct member. This codebase's
[SST1400](SST1400.md) *requires* an explicit access modifier, and a rule that asked you to delete what
another rule asks you to write is a rule that never converges.

## How to fix violations

Delete the modifier. The code fix does it, and removes each redundant modifier on a member
independently, so `public abstract void Run();` becomes `void Run();` in one pass.

## Configuration

None.

## Performance

Nothing on the clean path binds a symbol. Every judgment comes from the modifier list, the parent
declaration's own modifiers, and the parse options, so a member with no interesting modifier costs one
scan of a list that is almost always two tokens long. The only walk in the rule — proving that an
`unsafe` member really does contain unsafe syntax — runs solely for a member nested inside an unsafe
one, and uses the repo's allocation-free descendant traversal rather than `DescendantNodes()`.
