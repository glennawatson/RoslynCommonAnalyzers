# SST1492 — Do not test a value against what it is about to be assigned

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

An `if` guards an assignment of the very value its condition tested against.

```csharp
public void SetValue(int value)
{
    if (_value != value)
    {
        _value = value;
    }
}
```

The inverted spelling is the same thing:

```csharp
if (_value == value)
{
}
else
{
    _value = value;
}
```

## Rule description

`if (x != y) { x = y; }` does exactly what `x = y` does. The guard's only effect is to skip an
assignment that would have changed nothing — it decides nothing, and it costs a comparison and four
lines to decide it.

```csharp
public void SetValue(int value) => _value = value;
```

The shape is worth reporting because it is rarely written on purpose. It usually means the condition or
the assignment was mistyped — the guard was meant to test one thing and assign another, and one of the
two names is wrong. Finding out which is the actual fix; the analyzer's job is to make you look.

The other way it appears is as the fossil of a change-notification setter that lost its body:

```csharp
// what it was
if (_name != value)
{
    _name = value;
    OnPropertyChanged();
}

// what a refactor left behind
if (_name != value)
{
    _name = value;
}
```

## What is reported

- `if (x != y) { x = y; }`, with or without braces, and with the operands in either order —
  `if (y != x) { x = y; }` is the same mistake written the other way round.
- `if (x == y) { } else { x = y; }` — an empty branch with the assignment in the `else`.

## What is not reported

- **A guard that wraps more than the assignment.** The change-notification setter above, with its
  `OnPropertyChanged()` still intact, is doing real work and stays clean. Only a branch whose single
  statement is the assignment is reported.

- **A property with a hand-written accessor.**

  ```csharp
  public string Name
  {
      get => _name;
      set
      {
          _name = value;
          OnPropertyChanged();
      }
  }

  public void Rename(string value)
  {
      if (Name != value)   // not reported
      {
          Name = value;
      }
  }
  ```

  A setter written by hand can do anything behind the assignment — raise a notification, validate, dirty
  a flag — so the guard is deliberate: it is what stops the side effect from firing for a value that did
  not change. Only an **auto-implemented** property, whose setter can do nothing but store, is reported.
  A property from another assembly is left alone too: nothing in this compilation can prove what its
  setter does.

- **An operand with side effects.** `if (_value != Next()) { _value = Next(); }` stays clean. Skipping
  the assignment also skips the second call, so the guard is not doing nothing. The same reasoning
  excludes an element access — `a[i]` runs an indexer this rule cannot see through.

- **A compound assignment.** `if (_value != value) { _value += value; }` reads the target before it
  writes, so it is not the operation the condition tested and can never be a no-op.

## How to fix violations

Work out which of the two lines was wrong. If the answer is "neither, the guard is just pointless",
the code fix replaces the whole `if` with the bare assignment:

```csharp
_value = value;
```

If the answer is "the guard was meant to protect something", the fix is to put the something back —
and that is not a change any analyzer should make for you.

## Configuration

None.

## Performance

The clean path is three syntax tests — a binary condition of the right kind, a single-statement branch,
a simple assignment — and almost every `if` statement in a codebase fails the first or second. Only a
statement that has already matched the entire shape is bound to a symbol, which is what makes the
property-setter check affordable: it never runs for code that is not already a candidate report.
