# SST1493 — Methods should not return a constant

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A parameterless method's whole body is a constant.

```csharp
public int Limit() => 5;

public string Name()
{
    return "fixed";
}
```

## Rule description

A method promises a computation. `Limit()` says "ask me and I will work it out" — and then does not:
callers pay for a call, readers open the body looking for the logic, and nobody can tell from the call
site that the answer never changes. A value that is fixed should say so where it is used.

```csharp
public int Limit => 5;
```

Now `Limit` reads as what it is. The call site loses its empty parentheses, the reader loses the trip
into the body, and the next person who needs the value where a constant is required — a `case` label, an
attribute argument, a default parameter — can see immediately whether it is available.

## What is reported

A **parameterless, non-generic** method whose body is a single constant — an expression body, or a block
whose only statement returns one. The constant may be a literal, a named `const`, an enum member, or a
negated literal.

Parameters and type parameters are excluded because the rule's advice would not be followable: a
property can take neither, so telling you to expose `int Ignore(int value) => 1;` as a value would be
asking for something the language does not have. (A method that ignores its arguments and returns a
constant is often a bug, but it is a different bug, and a different rule's business.)

## What is not reported

The exclusions are most of this rule. A constant body is a *mistake* only when nothing else forced the
member to be a method.

- **An override.** A derived type answering a question with a constant is exactly how overriding works —
  it is the mechanism, not a smell:

  ```csharp
  public override bool IsReadOnly() => true;   // not reported
  ```

- **A `virtual` or `abstract` member.** The value is fixed *here*, and the member exists so that it is not
  fixed in a derived type.

- **An interface implementation**, implicit or explicit, and **an interface's own member.** The interface
  says this is a method; the implementation has no say in the matter.

- **A partial member.** Its shape belongs to the declaration it follows.

- **A method carrying any attribute.** Any at all — the rule does not try to judge which attributes
  matter. An attribute is often the entire reason the member exists as a method: a test case, a
  benchmark, a serialization or dependency-injection hook, a source-generator marker. The thing reading
  it is not in this file, and usually not in this compilation.

- **A method returning `null` or `default`.** That is a "nothing to give you" answer — a lookup that
  found nothing, a factory with no product — rather than a value that was supposed to be computed.

- **A local function**, which is not a member and has nothing to expose.

- **A body that computes anything at all**, including one that returns a field (`=> _value;`) or a
  `static readonly` (`=> string.Empty;`). Neither is a constant; the value can differ per instance or
  per run.

## How to fix violations

The code fix turns the method into a get-only property and drops the empty argument list from every call
site, across the solution:

```csharp
// before
public int Ceiling() => Max;
public bool IsOver(int value) => value > Ceiling();

// after
public int Ceiling => Max;
public bool IsOver(int value) => value > Ceiling;
```

The fix is **only offered when the whole change can be made correctly**. It is withheld — the diagnostic
stands, but no light bulb appears — when:

- **The method is used as a method group**: `Func<int> f = Weight;`. A property cannot be handed to a
  delegate.
- **A call is written as a statement of its own**: `Weight();`. A property read is not a statement.
- **The name is overloaded**: a property would collide with the other overload.

In each of those cases the rewrite would not compile, so it is not offered. Deciding what to do instead
is a design question — usually the answer is that the *caller* is the thing to change.

Whether the value then belongs as a `const` is a further step this fix does not take. A property keeps
the member's accessibility, its place in the type, and the shape of every call site; a field would change
all three, and whether that is right depends on what the value is for.

## Configuration

None.

## Performance

The clean path never reaches the semantic model. A method is rejected on the shape of its body —
anything but a lone return of a literal or a name is out — before the modifiers, the attributes, or the
signature are examined, and long before the constant is proven or the interfaces are searched. The
interface-implementation walk, the most expensive check in the rule, runs only for a method that already
looks exactly like a constant.
