# SST1494 — Do not pass an argument that repeats the parameter's default

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes — drops the argument, and the ones after it |
| Default | Enabled |

## Cause

A trailing argument at a call site has the value the parameter already defaults to.

```csharp
public void Configure(string name, int retries = 3, bool cache = true)
{
}

Configure("orders", 3, true);
```

## Rule description

Writing out the default adds nothing to the call and takes something away from it. `Configure("orders")`
says *use the defaults*; `Configure("orders", 3, true)` says *use 3 and true*, and only a reader who opens
the declaration can tell that those are the same thing today.

They may not be the same thing tomorrow. When the default changes — retries goes from 3 to 5 — every call
that omitted the argument moves with it, and every call that spelled the old value out silently keeps the
old behavior. That is the failure this rule exists to prevent: not noise, but a change that lands
everywhere except the places that had already made up their mind.

```csharp
Configure("orders");
```

### Only a trailing argument is reported

An argument in the middle of a positional list cannot be dropped on its own — the arguments after it would
slide into the wrong parameters. So the rule walks backwards from the last argument and stops at the first
one that is not redundant:

```csharp
Configure("orders", 3, false);   // clean: 'false' is not the default, so nothing before it can go either
Configure("orders", 3, true);    // both reported: the whole tail is redundant
```

A named argument is reported the same way, and dropping it is always safe:

```csharp
Configure("orders", cache: true);   // reported
```

### The value is compared, not the text

The comparison is on the constant the argument evaluates to, not on how it is written. All three of these
repeat a default of `0`:

```csharp
public void Skip(int count = 0)
{
}

private const int Zero = 0;

Skip(0);
Skip(0x0);
Skip(Zero);
```

The same applies across widths — an `int` literal `0` passed to a `long` parameter defaulting to `0` is the
same value — and to enum members and `null`.

### What is not reported

- **A caller-info parameter.** `[CallerMemberName]` and its siblings are filled in by the compiler, and
  passing one explicitly is a different mistake with a different fix — see [SST1448](SST1448.md).
- **A call inside an expression tree.** The language forbids omitting optional arguments there, so the
  argument this rule would remove is the one thing keeping the code compiling.
- **A `params` parameter.** It is never optional, so an argument bound to it is never a repeated default.
- **An argument whose omission would move the call.** Before anything is reported, the shortened call is
  bound and checked to still reach the same method. Where it would not, the argument is not redundant at
  all — it is what selects the overload:

  ```csharp
  public int Send(int size) => size;

  public int Send(int size, bool cache = false) => size;

  Send(1, false);   // clean: dropping 'false' would call the other Send
  ```

- **A target-typed `new(...)`.** Without its target type there is no way to bind the shortened form, and so
  no way to prove the omission safe. The rule says nothing rather than guess.

## How to fix violations

Delete the argument. The code fix does it, and deletes everything to its right at the same time — which is
only ever the rest of the redundant run, because that is all the rule reports. It re-binds the shortened
call before it offers the edit, so a Fix All pass over a file that has moved on cannot re-target a call.

## Configuration

None. There is nothing to tune: an argument either repeats its default or it does not.

## Performance

The clean path is an argument count. A call with no arguments is rejected on syntax alone, and one whose
method declares no optional parameter is rejected on the first bind — which is the same bind the compiler
has already done and cached. The constant comparison runs only over the trailing arguments, and stops at
the first one that differs. The two expensive steps — the expression-tree walk and the speculative bind of
the shortened call — run only after a redundant argument has been found, so a call site that is already
correct never pays for either.
