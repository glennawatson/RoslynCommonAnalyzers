# SST1495 — Do not compare with `==` when the type overrides `Equals`

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes — rewrites the comparison as `object.Equals` |
| Default | Enabled |

## Cause

`a == b` or `a != b`, where the operand is a reference type that overrides `Equals(object)` but does not
overload `operator ==`.

```csharp
public class Money
{
    public decimal Amount { get; init; }

    public override bool Equals(object obj) => obj is Money other && other.Amount == Amount;

    public override int GetHashCode() => Amount.GetHashCode();
}

bool same = left == right;   // compares references, not amounts
```

## Rule description

A type that overrides `Equals` has said what equality means for it. A type that does not *also* overload
`==` has left that operator meaning the default thing for a class: are these two the same object.

So the type ends up with two answers to the same question, and they disagree:

```csharp
var a = new Money { Amount = 5m };
var b = new Money { Amount = 5m };

a.Equals(b);         // true  — the definition the author wrote
a == b;              // false — the definition the author forgot
list.Contains(a);    // true  — the framework calls Equals
```

The one the author reaches for at the call site is usually `==`, because it is shorter and because it works
for `int` and `string`. It is the wrong one. And nothing catches it: the code compiles, the test that
compares two references to the same object passes, and the bug appears the day two equal values arrive from
different places.

Either call `Equals`, or overload the operator so that both agree. Overloading is usually the better answer
for a type whose value semantics are the point — but it is a change to the type, and this rule does not
presume to make it for you.

## What is not reported

- **A type that overloads `==`.** The operator then means whatever its author defined, which is the whole
  point of overloading it. This covers `string`, every `record` (the compiler writes both), the delegate
  types, and any type of your own that took the trouble.
- **A comparison against `null`.** `value == null` is a null check. It is always correct, and it is the one
  comparison that must not be rewritten — `object.Equals(value, null)` says the same thing and says it
  worse.
- **A value type.** There are no references to compare.
- **An operand typed as `object`.** Reaching for `object` is how a caller asks for reference semantics on
  purpose.
- **An interface operand.** The static type says nothing about how the implementation defines equality, and
  the comparison may well be an identity check between two known instances.
- **A type parameter.** `T == T` has no operator to reason about, whatever the constraints.

## How to fix violations

The code fix rewrites the comparison as a call to the static `object.Equals`:

```csharp
bool same = object.Equals(left, right);
bool differ = !object.Equals(left, right);
```

Two deliberate choices in that rewrite:

**The static form, not `left.Equals(right)`.** The static one is null-safe: it answers `true` for two nulls
and `false` when only one side is null, and it never throws. The instance call throws on a null receiver.
The operator being replaced never threw, so a fix that can is not a fix at all — it would turn a wrong
answer into a crash.

**Spelled `object.Equals`, not a bare `Equals`.** An unqualified two-argument `Equals` binds to whatever is
in scope, and a type that declares its own two-argument `Equals` would quietly capture the call — which is
exactly the class of silent mis-binding this rule exists to remove. Qualifying it makes the target
unambiguous, and the fix then binds the rewritten call speculatively and confirms it landed on
`System.Object.Equals(object, object)` before offering the edit at all.

The alternative fix — the one the analyzer cannot write for you — is to give the type the operator it is
missing, so that `==` and `Equals` finally agree:

```csharp
public static bool operator ==(Money left, Money right) => Equals(left, right);

public static bool operator !=(Money left, Money right) => !Equals(left, right);
```

## Configuration

None.

## Performance

The clean path stops at syntax: a comparison with a literal operand — the `null` check above all, which is
the single most common shape of `==` in real code — is rejected before the semantic model is touched. Only
a comparison between two named operands is bound, and the type is then filtered on three symbol flags
before either member walk runs. The `Equals` walk and the operator walk both stop at `object`, and the
operator walk runs only for a type that already overrode `Equals` — so a type with no equality of its own,
which is most of them, is rejected after one lookup.
