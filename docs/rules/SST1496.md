# SST1496 — Abstract types should declare something abstract

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes — drops `abstract`, and seals the type where that is provably safe |
| Default | Enabled |

## Cause

An `abstract class` declares no abstract member, and inherits no abstract member that it leaves
unimplemented.

```csharp
public abstract class Repository
{
    protected Repository()
    {
    }

    public int Timeout { get; set; }

    public void Save(Entity entity)
    {
    }
}
```

## Rule description

`abstract` on a class is a contract with two halves: *you cannot instantiate me*, and *you must supply
these members*. A class with nothing abstract in it has only the first half. It forbids instantiation and
then asks nothing in return — it is a base class wearing a contract's clothes.

That is worth reporting because of what a reader assumes when they see the keyword. They go looking for the
member the derived types have to write. There isn't one. Whatever `abstract` was doing here — marking the
type as "not for direct use", reserving a place in a hierarchy that never grew, or simply surviving a
refactor that moved its last abstract member elsewhere — it is not doing the thing its keyword says.

Two ways out, and the rule does not choose for you:

- **Give it the member.** If derived types really do have to supply something, say so:

  ```csharp
  public abstract class Repository
  {
      public abstract void Save(Entity entity);
  }
  ```

- **Drop `abstract`.** If nothing is missing, the type is a class. Seal it if nothing derives from it, and
  the reader stops looking for the contract that was never there.

## What is not reported

- **A type that inherits an abstract member it does not implement.** It is genuinely abstract — its base
  is still asking for something, and this type has not answered. That is what `abstract` means, and the
  rule walks the base chain to confirm it:

  ```csharp
  public abstract class Shape
  {
      public abstract double Area { get; }
  }

  public abstract class Rounded : Shape   // clean: Area is still unanswered
  {
      public double Radius { get; set; }
  }
  ```

  Once the last inherited abstract member has an override, the type is reported again — there is nothing
  left for a derived type to supply:

  ```csharp
  public abstract class Square : Shape    // reported: Area is answered, nothing is left abstract
  {
      public double Side { get; set; }

      public override double Area => Side * Side;
  }
  ```

- **A `partial` type with an abstract member in another part.** The rule judges the type once, from every
  part at once, so a member declared in a different file counts and the type is never reported twice.
- **A `static` class.** It is abstract and sealed in metadata, but it is not an abstract class.
- **A `record`.** An abstract record base with no abstract members is how a closed hierarchy is written,
  and the compiler supplies the members that make it work.

A type whose only constructor is `protected` **is** reported. That is a real pattern — a marker base you
are not meant to instantiate directly — but the protected constructor already says that, on its own, and
without the promise that `abstract` makes and does not keep.

## How to fix violations

Either add the abstract member the derived types owe you, or make the type concrete. The code fix does the
second, and takes it as far as it can prove is safe:

- **Nothing derives from it, and nothing in it was written to be inherited** — it is sealed:

  ```csharp
  public sealed class Repository
  ```

  The whole solution is searched for a derived class first, because sealing a type that something already
  derives from does not compile.

- **Something derives from it, or it declares a `protected` or `virtual` member** — only `abstract` is
  removed:

  ```csharp
  public class Repository
  ```

  A `virtual` member in a sealed type is an error, and a `protected` one is a warning: a class that holds
  either was written to be extended, whatever its `abstract` keyword was or was not doing.

- **It is `partial`** — no edit is offered. The modifiers are spread across parts the fix cannot see all
  of, and choosing which part should carry `sealed` is a guess. The diagnostic stands; the change is
  yours.

## Configuration

None.

## Performance

The rule runs as a symbol action, so it costs nothing per syntax node and sees a partial type once rather
than once per part. The filter is three symbol flags — class, abstract, not static — before anything is
enumerated, so the member scan runs only for abstract classes. The base-chain walk, the only part that
allocates, runs only for an abstract class that declared nothing abstract *and* has a base type other than
`object`, which is a small fraction of a small fraction.
