# SST1497 — Unused locals should be removed

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes — removes the variable, keeps what computing it did |
| Default | Enabled |

## Cause

A local variable is declared and never read.

```csharp
public void Run(Order order)
{
    var total = order.Total;       // never read
    Ship(order);
}
```

## Rule description

A local nobody reads is either dead code or a bug. Something was meant to happen to that value, and it does
not — the line below was supposed to use it, or the variable was left behind when the code around it moved
on. Either way, every future reader has to work out that it does not matter, and that is a cost with no
matching benefit.

### Written is not read

The distinction the rule turns on: a local that is *assigned*, even repeatedly, but never *read* is unused.
The value it holds goes nowhere.

```csharp
int state = 0;          // reported
if (flag)
{
    state = 1;          // a write, not a read
}

state = Compute();      // still a write; nothing ever asks what state is
```

A read is anything that asks for the value: passing it, returning it, testing it, `nameof`-ing it,
capturing it. And a read *inside a nested lambda or local function is still a read* — a captured local is
used exactly where it is used:

```csharp
public Func<int> Run(int value)
{
    var captured = value;    // clean: read inside the lambda
    return () => captured;
}
```

Some shapes are treated as reads even though they arguably are not — a compound assignment (`total += 1`),
an increment (`counter++`), an `out` or `ref` argument on an already-declared local. Each one is a place
where the value is loaded before it is stored, and unpicking them safely is not something a mechanical fix
can promise. The rule stays quiet rather than report something it cannot fix.

### What is not reported

Declarations whose lifetime is the point, or whose shape is not a choice:

- **A discard**, `_`. It already says the value is unwanted.
- **A `using` declaration.** The variable exists to be disposed at the end of the scope. Never reading it
  is the normal case.
- **A `ref` local.** The alias it holds is what the declaration is for.
- **A `foreach` variable**, and a **pattern variable** (`if (x is string text)`). Both are part of a
  construct's shape, not a value someone chose to name.
- **A `fixed` buffer.** Its scope is what pins the memory.

An **`out var`** *is* reported, and this is the one that surprises people. The callee assigns it, which is
not the caller reading it:

```csharp
if (int.TryParse(text, out var value))   // reported when nothing reads 'value'
{
    return true;
}
```

The fix turns it into `out _`, which says the same thing honestly.

## How to fix violations

Delete the variable — but not what computing it did. That is the whole difficulty of this fix, and it
decides case by case what happens to the expression that was being assigned:

| The assigned expression | What the fix does |
| --- | --- |
| Provably free of side effects (a literal, a name, a member access, arithmetic over those) | Deleted with the variable |
| A lambda or anonymous method | Deleted — writing one down creates a delegate and runs nothing |
| A call, a `new`, an `await`, an assignment, an increment | Kept as a statement: `Compute();` |
| Anything else that must still run | Kept as a discard: `_ = values[index];` |

```csharp
// before
var result = Compute();
var element = values[index];
var total = order.Total;

// after
Compute();
_ = values[index];
```

The dead writes go with the declaration, and get the same treatment — `state = Compute();` becomes
`Compute();`, and `state = 1;` disappears. All of it lands as one edit, because removing the declaration
without rewriting the writes would leave code that does not compile.

Every write is bound before it is touched. The analyzer finds them by name, which is enough to *report*;
it is not enough to *rewrite*, and an edit aimed at the wrong variable would be far worse than no edit at
all.

### When no fix is offered

The diagnostic stands but the edit is withheld where there is no safe one:

- **A variable declared alongside others**, whose initializer has to survive. Removing the declarator leaves
  the expression nowhere to go, and hoisting it out changes the order in which the initializers run.
- **A `stackalloc`.** It takes its type from what it is assigned to, so it cannot be discarded.
- **A language version below C# 7**, where the expression cannot stand as a statement. There is no discard
  to keep it alive with, so nothing is offered rather than something that will not compile.

In each case the local really is unused. Only the automatic edit is missing.

## Configuration

None.

## Performance

Reads are found by name, not by symbol: a plain identifier inside the local's scope can only mean that
local, because C# does not let a nested declaration shadow it, and every read of a local must spell its
name. So the whole rule runs without the semantic model. The scan is a preorder walk of the enclosing block
that stops at the first read, which is what happens almost immediately for a local that is used — and a
used local is nearly all of them. Only the code fix binds anything, and only for the local it is about to
remove.
