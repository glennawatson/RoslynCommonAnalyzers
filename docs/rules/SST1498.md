# SST1498 — Move a private member used only by a nested type into it

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes, for a static method |
| Default | Enabled |

## Cause

A private field, property or method of a type is referenced only from a nested type of that type,
and never from the type itself.

```csharp
public sealed class Parser
{
    private static int Double(int value) => value * 2; // reported: only Cursor uses it

    public sealed class Cursor
    {
        public int Advance(int value) => Double(value);
    }
}
```

## Rule description

A member that only a nested type touches is declared further out than it needs to be. Moving it in
narrows what a reader has to hold in their head when they change it — the answer to "who calls
this?" becomes the twenty lines they are already looking at, rather than the whole file — and it
stops the outer type from carrying a surface it does not use.

```csharp
public sealed class Parser
{
    public sealed class Cursor
    {
        public int Advance(int value) => Double(value);

        private static int Double(int value) => value * 2;
    }
}
```

## What is reported

Private **static** fields, properties and methods. A member is reported only when *exactly one*
nested type uses it and the outer type does not.

A member that calls itself is still a member only the nested type uses: a recursive call is not the
type using it.

## What is not reported

The rule is deliberately quiet wherever the move is not obvious.

- **An instance member.** Only a `static` member can actually move. An instance member belongs to an
  instance of the outer type, and the nested type reaches it through a reference it holds
  (`_owner._factor`); moving it would hand each nested instance its own copy, which is a different
  program.

  Reporting it anyway would not merely produce a diagnostic no one can act on — it would not
  terminate. The ordinary shape here is a subscription that unsubscribes itself by calling one
  private method on its owner. Move that method into the subscription, and the private helpers *it*
  called become nested-only in turn; move those, and the fields they touch follow, until the rule has
  asked for the outer type's whole state to be poured into the nested one.

- **A member two nested types share.** There is no single place to move it to, so there is nothing
  to say.
- **A member the outer type also uses.** It is not the nested type's to take.
- **A member with an attribute.** The attribute may be the only thing that uses it — a serialization
  callback, a test hook, something a generator or a framework reflects over — and none of that is
  visible to the compiler. Leaving these alone is the price of not being wrong about them.
- **A `partial` type.** Its other parts are other files, and a use in one of them would be invisible
  from here.
- **A member nothing uses at all.** That is a different question, and
  [SST1440](SST1440.md)'s.

## How to fix violations

The code fix moves the member into the nested type, and is offered for one shape only: a **static
method**. The method moves as a whole node, so its documentation comment and the comments around it
travel with it, and the formatter re-indents it for its new home.

The C# rule that makes that move safe is that a nested type still sees everything its enclosing type
declares. The moved method's body keeps binding to the outer type's other private members exactly as
it did before, and a call written as a bare `Helper(…)` inside the nested type keeps binding — now
to the method in its own type. Beyond `static`, the fix therefore requires all of:

- **Every nested use written unqualified.** A call spelled `Parser.Double(…)` names the type the
  method is leaving, and moving the method would leave that call pointing at nothing.
- **No other member of that name in the outer type, and none in the nested type.** C# name lookup
  stops at the first type that declares a name, so moving one overload of a family into the nested
  type would hide the whole rest of the family from every unqualified call inside it.
- **A nested class, struct or record with no base type, declared whole in one part.** A member
  declared in a type hides an inherited one of the same name — so a base class the nested type never
  mentions could be exactly where that name lives today.
- **A type free of preprocessor directives**, which a moved member would carry with it and
  unbalance.

Everything else the rule reports is left to a human, on purpose:

- **An instance member** would have to decide whose state it now is. The nested type has no implicit
  access to an instance of the type that encloses it; it reaches the member through a reference it
  holds. Moving the member means moving the state, and that is a design question.
- **A field** would change *when* its initializer runs, because static initialization happens per
  type. That is a behaviour change hiding inside a tidy-up, and the fix will not make it for you.

The move lands the method at the end of the nested type's members. If your layout wants it
elsewhere, put it there — the ordering rules will tell you.

## Performance

The analysis is whole-type, but the diagnostic is **local**: a private member cannot be named from
outside the type that declares it, so every use of it is inside the one declaration a syntax-node
action already hands over. Nothing is deferred to a compilation-end action — a diagnostic reported
there is non-local, and the IDE can neither refresh it as you type nor attach a code fix to it.

The clean path is a single indexed pass over the type's members that stops as soon as it knows the
type has no nested type: no allocation, no binding. Only a type that nests a type and declares a
private member goes further. The reference walk then matches names as text before it binds
anything, so a type whose nested code never mentions a private member's name never pays for a
symbol lookup.
