# SST1499 — Do not expose a mutable static field

| Item | Value |
| --- | --- |
| Category | Maintainability |
| Severity | Warning |
| Code fix | Yes, when `readonly` is all it takes |
| Default | Enabled |

## Cause

A static field that is visible outside its type can still be changed — either because it is not
`readonly`, or because it is a `readonly` array or mutable collection whose *contents* any caller can
rewrite.

```csharp
public static class Config
{
    public static int Timeout = 30;                          // any caller can reassign it

    public static readonly string[] Hosts = { "localhost" }; // any caller can rewrite Hosts[0]
}
```

## Rule description

A visible static field that is not `readonly` is global mutable state. Any code, on any thread, can
reassign it, and every other user of the type sees the change with no way to notice it happened —
no setter to breakpoint, no event, no log line. Tests that run in parallel start failing in ways
that depend on their order.

A `readonly` array or list is barely better. The reference cannot move, but the contents can:

```csharp
Config.Hosts[0] = "evil.example";  // compiles. Nothing about 'readonly' stops it.
Config.Retries.Clear();            // so does this.
```

So a visible static collection is reported too. What to expose instead depends on what you meant:

```csharp
public static class Config
{
    public static readonly ImmutableArray<string> Hosts = ["localhost"];  // nobody can change it

    private static string[] _hosts = ["localhost"];
    public static string[] GetHosts() => (string[])_hosts.Clone();        // callers get a copy

    public static IReadOnlyList<string> Names { get; } = _names.AsReadOnly();
}
```

## What is reported

A field is reported when it is **static**, **visible outside its own type**, and **mutable**.

Visibility is judged on the whole chain, not just the field: a `public` field inside a `private`
nested class is only reachable from the class that nests it, and is not global state.

Mutability means one of:

- The field is **not `readonly`** (and not `const`). Any caller can reassign it.
- The field is `readonly` but its type is an **array or a mutable collection** — `List<T>`,
  `Dictionary<,>`, `HashSet<T>`, `Queue<T>`, `Stack<T>`, `Collection<T>`, `ObservableCollection<T>`,
  the non-generic `ArrayList` and `Hashtable`, and the mutable interfaces `IList<T>`,
  `ICollection<T>`, `IDictionary<,>`, `ISet<T>`.

## What is not reported

- **`const`.** It is a value, not a variable.
- **A private field**, and any field inside a private nested type. Nothing outside can reach it.
- **`[ThreadStatic]`.** The state is deliberately per-thread, which is the opposite of the shared
  state this rule is about.
- **A `readonly` field of any other type.** This is the important one, and it is a deliberate
  choice: the rule holds a list of the collection types whose contents are known to be changeable,
  and reports nothing else. A string, a number, an immutable or frozen collection, a read-only
  wrapper, a record, a type you wrote — all pass. An analyzer cannot prove a hand-written type
  immutable, and the alternative to admitting that is reporting every singleton in the codebase.
- **The concurrent collections.** They are mutable, but they exist to be shared and mutated by many
  threads at once. A static `ConcurrentDictionary` cache is the pattern working, not the pattern
  failing.

The collection types are resolved **from the compilation itself**, by name, the first time the rule
needs them. A type your target framework does not have simply is not in the set, so the rule never
reports a field on the strength of a type that does not exist where your code runs.

## How to fix violations

The code fix adds `readonly`, and is offered for exactly one shape: the field is not `readonly` yet,
and **nothing in the solution reassigns it** outside its own initializer or its type's static
constructor. Then the keyword changes nothing about how the code runs and everything about what a
caller may do to it.

```csharp
public static int Timeout = 30;           // becomes:
public static readonly int Timeout = 30;
```

It is deliberately **not** offered when:

- **The field is already `readonly`** — the collection half of the rule. Swapping `string[]` for an
  immutable collection, a copy, or a method changes what the API promises and how it performs, and
  each choice reads differently. The diagnostic asks the question; it does not pretend to know the
  answer.
- **Something reassigns the field**, anywhere in the solution, from somewhere a `readonly` field may
  not be written — including through a lambda or a local function inside the static constructor,
  which can run after the constructor has returned.
- **The field is `volatile`.** `volatile readonly` is not legal C#.
- **The field is a mutable struct.** `Origin.X = 1` writes the field itself, and a call to one of the
  struct's non-readonly methods writes through `this`. Add `readonly` and the first stops compiling
  while the second quietly starts mutating a defensive copy.

## Configuration

```ini
# Rule-specific
stylesharp.SST1499.include_internal = true

# Project-wide
stylesharp.include_internal = true
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `include_internal` | `true`, `false` | `true` | Report a field visible only inside the assembly. |

Assembly-visible fields are included by default. `internal` is a boundary between teams' code, not a
boundary between threads: a static field the whole assembly can reassign is exactly as shared, and
exactly as hard to reason about, as a public one. Set `include_internal = false` to narrow the rule
to your assembly's public surface — reasonable if `internal` is where your codebase deliberately
keeps its shared caches.

An unset or unparsable value falls back to the default, so a typo neither disables the rule nor
turns it on where you turned it off.

## Performance

The clean path is a modifier scan. A field that is not `static`, or that declares no accessibility
keyword at all (and so is private), is rejected on syntax alone before the semantic model is
touched — which is almost every field in almost every file. Only a field that is already static and
already named as visible pays for a bind. The collection types are resolved lazily, once per
compilation, so a project that never declares a visible static field never pays to look up a single
metadata name.
