# SST1521 — Lines should not be too long

| Item | Value |
| --- | --- |
| Category | Layout |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A line is longer than the configured maximum, which defaults to 120 characters.

```csharp
public Order Create(string customer, Address address, Money money, string reference, DateTimeOffset placed, bool priority)
{
}
```

## Rule description

A line that runs past the edge of the window is read by scrolling, and a line read by scrolling is
read badly: the end of it disappears while the reader looks at the start. It also defeats
side-by-side diffs, which is where most code is actually read — a review pane is half a screen
wide, and a line that does not fit in it is a line nobody reviewed.

```csharp
public Order Create(
    string customer,
    Address address,
    Money money,
    string reference,
    DateTimeOffset placed,
    bool priority)
{
}
```

### What is measured

The characters of the line, excluding the line break. A tab counts as one character, not as an
indentation width: how wide a tab looks is an editor setting, and a rule whose verdict changes with
the reader's editor is not a rule.

### The unbreakable-line exemption

A line is **not reported when no wrap could shorten it**. That is decided in two steps, and both
have to hold:

1. **Nothing to wrap at.** The run of non-whitespace characters that crosses the limit is measured.
   If that run, placed on a line of its own at the line's own indentation, would *still* be over the
   maximum, then no break the author can insert will help; the only remaining "fix" would be to
   mangle the content.
2. **The run is content, not code.** The run has to sit inside a comment or inside a string or
   character literal — the two places a long unbreakable token legitimately occurs.

So both of these are clean:

```csharp
// See https://example.invalid/reference/a-very-long-canonical-url-that-nobody-can-shorten-and-splitting-it-breaks-the-link
private const string ConnectionString = "Server=a-very-long-host-name.example.invalid;Database=Orders;Encrypt=true;TrustServerCertificate=false";
```

And these are reported, because they *can* be wrapped:

```csharp
// A long comment made of ordinary words wraps onto a second line perfectly well and the rule says so.
public string Describe(string a, string b, string c) => a + b + c + Name + a + b + c + Name + a + b + c + Name;
```

A long *code* run with no whitespace in it — a hundred-character member chain — is still reported.
It is unbreakable only in the sense that nobody has broken it yet: `.` is a break opportunity, and
the analyzer declines to pretend otherwise.

## How to fix violations

There is **no code fix**. Where to break a line is a judgement about what the line says — which
argument groups with which, whether a chain reads better one call per line, whether the condition
wants extracting into a named local. An automatic wrap at column 120 would produce something that
compiles and that nobody would have written.

## Configuration

```ini
# Rule-specific
stylesharp.SST1521.max_line_length = 120

# Project-wide
stylesharp.max_line_length = 120
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `max_line_length` | positive integer | `120` | The most characters a line may contain. |

An unset or unparsable value falls back to the default.

## Performance

The clean path is one subtraction per line. The syntax tree is only asked for once a line is already
over the limit, so a file with no long lines never fetches the root and never touches a token.
