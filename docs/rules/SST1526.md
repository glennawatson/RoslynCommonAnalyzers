# SST1526 — Wrapped binary operators should sit on the configured side of the line break

| Item | Value |
| --- | --- |
| Category | Layout |
| Severity | Warning |
| Code fix | Yes |
| Default | Disabled |

## Cause

A binary expression is broken across lines and its operator sits on a different side of the break than
the configured style requires.

```csharp
// Default style is 'before': the operator leads the continuation line.
var total = subtotal +   // SST1526: the '+' trails the upper line
    shipping;
```

## Rule description

When a binary expression is broken across lines, the operator has two homes: the end of the upper line
or the start of the lower one. Either is fine; a file that mixes them is not, because the reader loses
the thread of where one operand ends and the next begins. Each operator in a wrapped chain is checked, so
the whole chain is held to one placement.

```csharp
var total = subtotal
    + shipping
    + tax;
```

Only an operator with an adjacent line break is checked; a single-line expression is never touched.

## How to fix violations

Move the line break to the configured side of the operator. The code fix does this in place.

## Configuration

```ini
# Project-wide
stylesharp.binary_operator_new_line = before

# Rule-specific
stylesharp.SST1526.binary_operator_new_line = before
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `binary_operator_new_line` | `before`, `after` | `before` | `before` leads the continuation line with the operator; `after` trails the upper line with it. |

The default `before` is the prevailing .NET convention. An unset or unrecognised value falls back to it.
