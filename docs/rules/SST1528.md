# SST1528 — A wrapped initializer '=' should sit on the configured side of the line break

| Item | Value |
| --- | --- |
| Category | Layout |
| Severity | Warning |
| Code fix | Yes |
| Default | Disabled |

## Cause

A field or local initializer wraps its value onto its own line and the `=` sits on a different side of
the break than the configured style requires.

```csharp
// Default style is 'after': the equals sign trails the name.
private static readonly int Limit
    = ComputeLimit();   // SST1528: the '=' leads the value line
```

## Rule description

A field or local whose initializer wraps onto its own line can keep the equals sign trailing the name or
lead the value line with it. The default trails the name.

```csharp
private static readonly int Limit =
    ComputeLimit();
```

Only a field or local (variable-declarator) initializer with an adjacent line break is checked. Property
initializers, parameter defaults, and single-line initializers are never touched.

## How to fix violations

Move the line break to the configured side of the equals sign. The code fix does this in place.

## Configuration

```ini
# Project-wide
stylesharp.equals_token_new_line = after

# Rule-specific
stylesharp.SST1528.equals_token_new_line = after
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `equals_token_new_line` | `after`, `before` | `after` | `after` trails the name with the equals sign; `before` leads the value line with it. |

An unset or unrecognised value falls back to the default.
