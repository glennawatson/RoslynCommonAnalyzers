# SST1529 — Wrapped call-chain operators should sit on the configured side of the line break

| Item | Value |
| --- | --- |
| Category | Layout |
| Severity | Warning |
| Code fix | Yes |
| Default | Disabled |

## Cause

A fluent member-access or conditional-access chain is broken across lines and one of its `.` / `?.`
operators sits on a different side of the break than the configured style requires.

```csharp
// Default style is 'before': each operator leads its continuation line.
var name = customer.
    Trim().          // SST1529: the '.' trails the upper line
    ToUpperInvariant();
```

## Rule description

A fluent call chain that wraps reads as a column of steps, one member access per line, each `.` or `?.`
leading its line. An operator that trails the upper line while the rest lead the lower one breaks the
column. Each chain link is checked once, so a wrapped chain is held to one placement.

```csharp
var name = customer
    ?.Trim()
    .ToUpperInvariant();
```

A conditional-access `?.` is moved as a unit, so the fix never splits the `?` from the `.`. A link with
no adjacent line break is never touched.

## How to fix violations

Move the line break to the configured side of the operator. The code fix does this in place.

## Configuration

```ini
# Project-wide
stylesharp.null_conditional_new_line = before

# Rule-specific
stylesharp.SST1529.null_conditional_new_line = before
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `null_conditional_new_line` | `before`, `after` | `before` | `before` leads each continuation line with the operator; `after` trails the upper line with it. |

An unset or unrecognised value falls back to the default.
