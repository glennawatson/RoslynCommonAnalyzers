# SST1633 — File should have a header

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Warning |
| Code fix | Yes (adds or replaces the header) |
| Config | `file_header_template` |

## Cause

A file does not begin with the header configured by `file_header_template`, and a
non-empty template is set.

## Rule description

When `file_header_template` is set in `.editorconfig`, every file must begin with
the rendered header. This is the same option the .NET SDK's file-header rule uses —
but SST1633 is a normal analyzer, so it runs in ordinary builds **without** turning
on `EnforceCodeStyleInBuild`.

When the template is `unset` or empty, the rule does nothing.

```ini
[*.cs]
file_header_template = Copyright (c) Contoso.\nLicensed under the MIT license.
```

```csharp
// Copyright (c) Contoso.
// Licensed under the MIT license.
namespace Contoso;
```

Each `\n` in the template starts a new `//` line, and `{fileName}` is substituted
with the file name. Leading preprocessor directives (`#if`/`#else`/`#endif`,
`#region`, `#pragma`, `#nullable`, …) are skipped when locating the header, so a
header that sits among or after such directives in the active compilation is still
recognised.

## How to fix violations

The code fix adds the rendered header at the top of the file. When the file already
begins with a header comment block that no longer matches — for example after the
copyright year is bumped in `file_header_template` — the fix **replaces** the stale
header in place rather than stacking a second one on top of it. A blank line ends the
header, so any comment past it is treated as ordinary code and left untouched.
