# SST1657 — Extension block documentation should reference a real parameter or type parameter

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Warning |
| Code fix | No |

## Cause

A `<param>` or `<typeparam>` element in a C# 14 extension block's documentation
names a parameter or type parameter the block does not declare.

## Rule description

A documentation reference that does not resolve is almost always a typo or a
left-over from a rename, and it produces a broken compiler-doc cross-reference:

```csharp
/// <summary>Adds projection helpers to an observable sequence.</summary>
/// <typeparam name="TSource">The type of the elements in the source sequence.</typeparam>
/// <param name="values">An observable sequence of elements to project.</param> // 👎 'values' is not a parameter
extension<TSource>(IObservable<TSource> source)
{
    public bool IsCold => true;
}
```

A `<param>` is matched against the block's value parameters and a `<typeparam>`
against its type parameters, so documenting a type parameter with `<param>` (or
vice versa) is also reported.

This is the StyleSharp counterpart of the analyzer's
[the rule](https://josefpihrt.github.io/docs/the analyzer/analyzers/the rule) for
extension blocks. The equivalent checks for ordinary members are already provided
by [SST1612](SST1612.md) (parameters) and [SST1620](SST1620.md) (type
parameters).

## How to fix violations

Correct the `name` attribute to match a declared parameter or type parameter, or
remove the stray element. There is no automatic fix because the intended target
cannot be inferred.
