# SST1659 — Comments should not be empty

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A comment carries no text at all.

```csharp
public sealed class Order
{
    //
    private int _total;

    /// <summary>Adds one to the total.</summary>
    /// <returns>The new total.</returns>
    public int Increment() => ++_total; /* */
}
```

## Rule description

A comment with no text is a leftover — a line someone meant to write, or the remains of one they
deleted. It costs a reader a glance and gives them nothing back. Four shapes are reported:

| Shape | Example |
| --- | --- |
| An empty line comment | `//` |
| A line comment holding only whitespace | `//␣␣␣` |
| An empty block comment | `/* */` |
| A documentation comment with nothing in it | a lone `///`, or `/** */` |

## What is not reported

### A blank line inside a documentation comment that says something

A `///` line with nothing on it is ordinary formatting — it is how you put a blank line between a
summary and what follows it. Consecutive `///` lines are **one** documentation comment, and that
whole comment is what the rule judges: it is reported only when *no* line in it has content.

```csharp
/// <summary>The running total.</summary>
///
/// <remarks>Reset by <see cref="Clear"/>.</remarks>
public int Total { get; }
```

### Commented-out code

`//// var disabled = 1;` keeps its extra slashes as text, so it is never empty — it falls out of
this rule without needing an exception. Whether commented-out code should be there at all is a
different question, and a different rule's.

### Directives

`#region`, `#if` and the rest are not comments. They are never reported, and the fix never touches
them: it only ever deletes lines that the comment had entirely to itself, which a directive's line
is not.

## How to fix violations

The code fix deletes the comment.

- A comment that **owns its line** takes the line with it, so no blank line is left behind.
- A comment that **trails code** is removed together with the whitespace that separated it from the
  code, and the code stays put.
- An empty documentation comment spanning several `///` lines is removed as a whole.

If the comment was a placeholder for something you still mean to write — write it.

## Performance

The rule is pure trivia. It registers a syntax-tree action, walks the tokens through the shared
allocation-free traversal helper, and reads characters straight out of the source text. It never
touches the semantic model, allocates nothing on the clean path, and never descends into a
comment's structure: a comment is classified from its own characters.
