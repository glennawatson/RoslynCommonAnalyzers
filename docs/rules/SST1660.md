# SST1660 — Parameter documentation should be ordered to match the parameters

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Info |
| Code fix | Yes (reorder the `<param>` elements) |
| Default | Enabled |

## Cause

A member documents exactly its parameters, but the `<param>` elements are written in a different order
than the parameters are declared.

```csharp
// 👎 reported — the <param> order does not follow the signature
/// <summary>Copies a range.</summary>
/// <param name="count">The number of items.</param>
/// <param name="source">The source array.</param>
public void Copy(int[] source, int count)
{
}
```

## Rule description

When the documentation and the signature list the parameters in the same order, the two read together
top to bottom. When they disagree, a reader has to hop back and forth to match each description to its
parameter.

The rule fires only when the documented names are already an exact, one-to-one match for the parameters:
every parameter is documented once and no extra `<param>` name appears. A member that is missing a
`<param>`, or documents a name that is not a parameter, is a different concern and is left to the rules
that own it, so this rule never competes with them.

## How to fix violations

The code fix reorders the `<param>` elements to match the parameter declaration order, moving only the
element text and leaving the surrounding `///` layout in place:

```csharp
/// <summary>Copies a range.</summary>
/// <param name="source">The source array.</param>
/// <param name="count">The number of items.</param>
public void Copy(int[] source, int count)
{
}
```

## Performance

The rule runs only on members that carry a documentation comment. It reads each member's `<param>`
elements once, compares them to the parameter list, and does no work at all when the documented set does
not match the parameters.
