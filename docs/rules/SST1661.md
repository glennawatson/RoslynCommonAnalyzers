# SST1661 — Code documentation tag should match its content

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Info |
| Code fix | Yes (swap `<c>` and `<code>`) |
| Default | Enabled |

## Cause

A snippet is wrapped in the wrong code tag: a single-line snippet uses the block `<code>` tag, or a
multi-line snippet uses the inline `<c>` tag.

```csharp
// 👎 reported — a one-line snippet in the block tag
/// <summary>Returns <code>value + 1</code>.</summary>

// 👎 reported — a multi-line snippet in the inline tag
/// <summary>
/// Runs <c>Configure();
/// Run();</c>.
/// </summary>
```

## Rule description

The two documentation code tags are for different shapes of content:

| Tag | Purpose |
| --- | --- |
| `<c>` | An inline fragment that sits on one line within a sentence. |
| `<code>` | A block sample spread across several lines. |

When a one-line snippet is wrapped in `<code>`, or a multi-line snippet in `<c>`, the tag and the content
disagree about which they are, and the rendered documentation formats the snippet against the author's
intent. The rule measures the snippet from its first to its last non-whitespace character, so wrapping
alone does not decide single versus multi-line.

An empty tag has no content to classify and is never reported.

## How to fix violations

The code fix swaps the tag, renaming both the start and end tags:

```csharp
/// <summary>Returns <c>value + 1</c>.</summary>

/// <summary>
/// Runs <code>Configure();
/// Run();</code>.
/// </summary>
```

## Performance

The rule inspects each `<c>` and `<code>` element once, comparing the source lines of its first and last
text characters, and does nothing for any other element.
