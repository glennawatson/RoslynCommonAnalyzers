# SST1662 — Thrown exceptions should be documented

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Warning |
| Code fix | Yes (add an `<exception>` skeleton) |
| Default | Disabled (opt-in) |

## Cause

A documented member throws an exception type its documentation does not describe with a matching
`<exception cref="...">` element.

```csharp
// 👎 reported — the throw is not described by an <exception> element
/// <summary>Withdraws an amount.</summary>
public void Withdraw(decimal amount)
{
    if (amount <= 0)
    {
        throw new ArgumentOutOfRangeException(nameof(amount));
    }
}
```

## Rule description

An `<exception>` element tells a caller what a member can throw and when, so they can decide what to catch
without reading the body. When a member throws an exception it does not document, that contract is
incomplete.

Only exceptions constructed and thrown directly in the member body — `throw new SomeException(...)` — are
considered. The rule does not follow method calls (no interprocedural analysis) and does not look inside
deferred lambda or local-function bodies, whose throws run in a different context. Matching is by the
exception's simple type name, so a thrown type is satisfied by any `<exception>` element whose `cref`
names the same type.

The rule only augments **documented** members: a member with no documentation comment is the coverage
rules' concern, not this one.

### Why it is off by default

A member with several throw sites — argument guards, state checks, resource failures — produces one
diagnostic per undocumented type, which is noisy for teams that do not document every throw. Enable it in
`.editorconfig` when you want thrown exceptions documented:

```ini
[*.cs]
dotnet_diagnostic.SST1662.severity = warning
```

## How to fix violations

The code fix appends an `<exception cref="...">` skeleton for each undocumented thrown type, ready for a
description:

```csharp
/// <summary>Withdraws an amount.</summary>
/// <exception cref="ArgumentOutOfRangeException"></exception>
public void Withdraw(decimal amount)
{
    if (amount <= 0)
    {
        throw new ArgumentOutOfRangeException(nameof(amount));
    }
}
```

## Performance

The rule runs only on documented members and scans the body once for direct `throw new` sites, skipping
nested lambda and local-function scopes. A member that throws nothing pays only the documentation lookup.
