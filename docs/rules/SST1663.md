# SST1663 — A summary comment should be a documentation comment

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Warning |
| Code fix | Yes (convert to `/// <summary>`) |
| Default | Disabled (opt-in) |

## Cause

A single `//` comment sits on the line immediately above a public member, with no blank line between
them, and reads like a summary.

```csharp
// 👎 reported — this describes the member but does not document it
// Gets the widget count
public int Count { get; }
```

## Rule description

A comment directly above a public member, with nothing separating them, is describing that member. That
is what a `/// <summary>` documentation comment is for — and only the documentation comment reaches the
generated docs and IntelliSense; a `//` comment is invisible to both.

To keep the noise down, the rule is deliberately narrow. It only reports a comment that:

- is a plain `//` comment (a `///` comment is already documentation, and `////` or more keeps its extra
  slashes as text — commented-out code);
- begins its own line and sits directly above the member with **no** blank line between them;
- starts with a letter, so section separators (`// ----`) and commented-out code are left alone;
- is a lone comment — a stacked block of `//` lines is not converted; and
- sits above a member that is not already documented.

### Why it is off by default

Not every adjacent comment is a summary — some are notes to the next maintainer, not a description of the
member. Enable it in `.editorconfig` when your team treats a comment above a public member as its
documentation:

```ini
[*.cs]
dotnet_diagnostic.SST1663.severity = warning
```

## How to fix violations

The code fix rewrites the comment as a `/// <summary>` documentation comment, escaping any XML-significant
characters so the result is well-formed:

```csharp
/// <summary>Gets the widget count</summary>
public int Count { get; }
```

## Performance

The rule reads only the member's leading trivia, and does no work once it sees a documentation comment is
already present or the comment is not a lone, prose-like `//` line.
