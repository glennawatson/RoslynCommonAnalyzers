# SST1664 — Summary paragraphs should use `<para>` elements

| Item | Value |
| --- | --- |
| Category | Documentation |
| Severity | Warning |
| Code fix | Yes (wrap paragraphs in `<para>`) |
| Default | Disabled (opt-in) |

## Cause

A `<summary>` separates its prose into paragraphs with blank documentation lines but has no `<para>`
elements to mark them.

```csharp
// 👎 reported — the blank line will not survive the documentation pipeline
/// <summary>
/// Stores the running total.
///
/// The total is reset when the batch closes.
/// </summary>
```

## Rule description

A blank `///` line is insignificant whitespace to the documentation pipeline: it collapses, and the two
paragraphs render as one unbroken run of text. A `<para>` element around each paragraph keeps the break
in the generated output.

To keep the fix safe, the rule only fires on a canonical, pure-prose summary:

- the `<summary>` and `</summary>` tags each sit alone on their line;
- the content between them is plain text with no nested elements (a summary that already contains an
  element such as `<see>` or `<para>` is left alone); and
- at least one blank documentation line separates prose from prose.

### Why it is off by default

Whether paragraph breaks belong in a summary at all is a matter of house style. Enable it in
`.editorconfig` when your team wants multi-paragraph summaries marked up:

```ini
[*.cs]
dotnet_diagnostic.SST1664.severity = warning
```

## How to fix violations

The code fix wraps each blank-line-separated paragraph in a `<para>` element and drops the now-redundant
blank lines:

```csharp
/// <summary>
/// <para>
/// Stores the running total.
/// </para>
/// <para>
/// The total is reset when the batch closes.
/// </para>
/// </summary>
```

## Performance

The rule inspects each `<summary>` element once over its physical lines and does nothing for a summary
that is single-line, contains nested elements, or has no blank-line paragraph break.
