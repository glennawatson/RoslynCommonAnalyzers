# SST1703 — Prefer extension blocks over extension methods

| Item | Value |
| --- | --- |
| Category | Extensions |
| Severity | Warning |
| Enabled by default | **No** (opt-in) |
| Code fix | No |
| Applies to | C# 14 or later |

## Cause

A classic `this`-parameter extension method is declared where a C# 14 extension-block member could be used.

## Rule description

Extension blocks group extension members by receiver type, support properties, indexers, and
operators, and read more cleanly than a flat list of `this`-parameter methods.

```csharp
public static class StringExtensions
{
    // 👎 reported (when enabled, on C# 14+)
    public static bool IsBlank(this string text) => text.Trim().Length == 0;

    // 👍
    extension(string text)
    {
        public bool IsBlank => text.Trim().Length == 0;
    }
}
```

## Why it is opt-in

The rule is **disabled by default**: adopting the new syntax is a deliberate, repo-wide migration,
and many codebases still target older language versions or prefer classic extension methods. It is
also gated on the compilation's language version being **C# 14 or later**, so it never fires where
extension blocks are unavailable. Enable it in `.editorconfig` when you are ready to migrate:

```ini
dotnet_diagnostic.SST1703.severity = warning
```

This rule has no the analyzer counterpart — it covers the new extension-member syntax introduced in C# 14.

## How to fix violations

Move the method into an `extension(Receiver) { … }` block, dropping the `this` parameter and
referring to the receiver by its name.
