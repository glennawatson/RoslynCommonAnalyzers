# SST1705 — Do not mix classic extension methods with extension blocks

| Item | Value |
| --- | --- |
| Category | Extensions |
| Severity | Warning |
| Code fix | No |
| Applies to | C# 14 extension blocks |

## Cause

A class declares both an `extension` block and a classic `this`-parameter extension method.

## Rule description

Once a class adopts extension blocks, declaring all of its extensions the same way keeps the file
consistent and the receiver type visible in one place. A stray classic extension method alongside
the blocks is easy to miss.

```csharp
public static class StringExtensions
{
    // 👎 reported — classic method mixed in with a block
    public static bool IsBlank(this string text) => text.Trim().Length == 0;

    extension(string text)
    {
        public int Words => text.Split(' ').Length;
    }
}
```

To flag *every* classic extension method (not only those mixed with blocks), enable the opt-in
[SST1703](SST1703.md) instead.

This rule covers the new extension-member syntax introduced in
C# 14.

## How to fix violations

Move the extension method into an `extension(Receiver) { … }` block.
