# SST1707 — Extension blocks should be ordered by receiver type

| Item | Value |
| --- | --- |
| Category | Extensions |
| Severity | Warning |
| Enabled by default | **No** (opt-in) |
| Code fix | No |
| Applies to | C# 14 extension blocks |

## Cause

A class declares its `extension` blocks in an order that is not sorted by receiver type.

## Rule description

When a class holds extension blocks for several receiver types, ordering the blocks by receiver type
makes a particular block easy to find.

```csharp
// 👎 reported (when enabled) — string before int
public static class SampleExtensions
{
    extension(string text) { /* … */ }
    extension(int value) { /* … */ }
}

// 👍
public static class SampleExtensions
{
    extension(int value) { /* … */ }
    extension(string text) { /* … */ }
}
```

Receiver types are compared as text (ordinal). The rule pairs with [SST1701](SST1701.md) (which keeps
one block per type) and [SST1702](SST1702.md) (which keeps the blocks contiguous).

## Why it is opt-in

The rule is **disabled by default** — the order of extension blocks is a matter of preference. Enable
it in `.editorconfig` if you want a consistent sort:

```ini
dotnet_diagnostic.SST1707.severity = warning
```

This rule has no the analyzer counterpart — it covers the new extension-member syntax introduced in
C# 14.

## How to fix violations

Reorder the extension blocks so their receiver types are in ascending order.
