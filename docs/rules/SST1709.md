# SST1709 — Static helpers in an extension container should be extensions

| Item | Value |
| --- | --- |
| Category | Extensions |
| Severity | Warning |
| Enabled by default | **No** (opt-in) |
| Code fix | Yes |
| Applies to | C# 14 or later |
| Analyzer | `Sst1709AlmostExtensionMethodAnalyzer` |

## Cause

A `public` or `internal static` method in a class whose name ends in `Extensions` takes a first
parameter with no `this` modifier — it reads like an extension that was never wired up.

```csharp
public static class StringExtensions
{
    // 👎 reported (when enabled, on C# 14+)
    public static bool IsBlank(string text) => text.Length == 0;

    // 👍
    extension(string text)
    {
        public bool IsBlank() => text.Length == 0;
    }
}
```

## Rule description

A method is reported when all of the following hold: it lives in a class named `…Extensions`; it is
`static` and `public` or `internal`; it is not generic; it has a body; and its first parameter is a
plain parameter (no `this`, `ref`, `out`, `in`, or `params`). Those bounds keep the offered conversion
always valid.

## Why it is opt-in

The rule is **disabled by default**: an `…Extensions` class legitimately holds plain static helpers too,
so treating every one of them as a missing extension would be noisy. It is also gated on the
compilation's language version being **C# 14 or later**, because the fix produces the extension-block
syntax introduced in that version. Enable it in `.editorconfig` once you have adopted extension blocks:

```ini
dotnet_diagnostic.SST1709.severity = warning
```

## How to fix violations

The fix rewrites the method into an `extension(Receiver) { … }` block: the first parameter becomes the
block receiver, the method drops `static` and that parameter, and the rest moves inside the block. The
rewritten member is re-parsed and the fix is withheld unless it forms a valid extension block, so a fix
that would not compile is never offered.
