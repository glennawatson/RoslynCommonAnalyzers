# SST1900 — Use System.Threading.Lock for a dedicated lock object

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled (where the type exists) |

## Cause

A `private readonly object` field is used only as a `lock` target, where
`System.Threading.Lock` (introduced in .NET 9) is the purpose-built type.

```csharp
// 👎 reported
private readonly object _gate = new();

public void Add()
{
    lock (_gate)
    {
        // ...
    }
}

// 👍
private readonly System.Threading.Lock _gate = new();
```

## Rule description

`System.Threading.Lock` is a dedicated lock type. When you `lock` on a `Lock`
instance, the C# 13 compiler binds to its `EnterScope` API (a typed, allocation-free
scope) instead of `Monitor.Enter`/`Monitor.Exit`, which is faster and harder to
misuse.

A field is reported only when **all** of the following hold, so changing the type
can never break another use:

- it is `private readonly`;
- its type is `object`;
- it is initialized with a parameterless `new()` / `new object()`; and
- every reference to it within the declaring type is the expression of a `lock`
  statement (and there is at least one).

A field that is also read, passed as an argument, or used with `Monitor` directly
is **not** reported. Partial types are skipped, because a use in another file
cannot be seen.

### Availability detection

This rule targets a **runtime type**, not a language feature, so it is gated on
`System.Threading.Lock` existing in the referenced framework (detected once per
compilation via `Compilation.GetTypeByMetadataName`). On pre-.NET 9 targets it
reports nothing rather than suggesting a type that does not exist. No target
framework string is parsed.

## How to fix violations

Change the field type to `System.Threading.Lock`. The provided code fix does this
and normalizes the initializer to `new()`.
