# SST1901 — Do not lock on a publicly accessible object

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `lock` statement targets a field or property that is reachable from outside the
declaring type (public, protected, or protected-internal).

```csharp
// 👎 reported
public readonly object Gate = new();

public void M()
{
    lock (Gate) { /* ... */ }
}

// 👍 — lock on private, dedicated state
private readonly object _gate = new();

public void M()
{
    lock (_gate) { /* ... */ }
}
```

## Rule description

When the object you lock on is visible outside the type, unrelated code can take a
lock on the same instance. Two components that each "own" the lock can then block
each other, producing deadlocks that are very hard to diagnose. Lock on a
`private` object that nothing else can reach (ideally a dedicated lock field — see
PSH1300 in PerformanceSharp.Analyzers for the faster `System.Threading.Lock` type).

Only fields and properties with public, protected, or protected-internal
accessibility are reported; locking on a `private` or `internal` member is not
flagged.

## How to fix violations

Introduce a `private readonly` lock object and lock on that instead of the
publicly accessible member. (No automatic code fix — choosing and threading the
dedicated lock object is a manual change.)
