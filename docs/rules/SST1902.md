# SST1902 — Do not lock on 'this', a Type, or a string

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | No |
| Default | Disabled (opt-in) |

## Cause

A `lock` statement targets `this`, a `System.Type` (e.g. `typeof(T)`), or a
string.

```csharp
// 👎 reported (when enabled)
lock (this) { /* ... */ }
lock (typeof(C)) { /* ... */ }
lock ("token") { /* ... */ }

// 👍
private readonly object _gate = new();
lock (_gate) { /* ... */ }
```

## Rule description

These targets have *weak identity* — they are reachable by code you do not
control:

- **`this`** can be locked by any external holder of the instance;
- **`Type`** objects are process-wide and shared by all callers; and
- **strings** may be interned, so two unrelated `lock ("x")` statements can take
  the same lock.

Any of these invites deadlocks. Lock on a `private` dedicated object instead.

**This rule is off by default** — locking on the instance is common in older
code, so turning it on is a decision a codebase makes deliberately. Enable it in
`.editorconfig` when you want it:

```ini
[*.cs]
dotnet_diagnostic.SST1902.severity = warning
```

## How to fix violations

Lock on a `private readonly` object dedicated to the lock. (No automatic code fix
— introducing the dedicated object is a manual change.)
