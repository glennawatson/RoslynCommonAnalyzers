# SST1903 — Do not lock on a newly-created object

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `lock` statement targets a freshly-created object such as `new object()`.

```csharp
// 👎 reported
lock (new object()) { /* ... */ }

// 👍
private readonly object _gate = new();

lock (_gate) { /* ... */ }
```

## Rule description

Locking on a newly-created object does not provide mutual exclusion, because
every evaluation creates a different instance. Each caller therefore takes a
different lock, and the protected region is not actually coordinated with
anything else.

Store the lock object in a dedicated field and reuse that same instance for
every `lock` statement that is meant to guard the same state.

## How to fix violations

Introduce a dedicated `private readonly` lock object field and lock on that
stored instance instead of constructing a new object inside the `lock`
expression. (No automatic code fix — choosing the correct shared lock object is
a manual change.)
