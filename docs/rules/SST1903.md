# SST1903 - Do not lock on a newly-created object

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `lock` statement targets an object that is fresh on every evaluation - `new object()` written inline,
or a local whose only initializer is a `new` and which never leaves the method.

```csharp
// reported
lock (new object()) { /* ... */ }

// reported: a fresh local that is never published
object gate = new();
lock (gate) { /* ... */ }

// not reported: the local aliases shared state
object local = _gate;
lock (local) { /* ... */ }
```

## Rule description

Locking on an object that is fresh on every call provides no mutual exclusion, because every caller
takes a different instance and the protected region is coordinated with nothing.

The rule reports the inline form (`lock (new object())`) and follows a local one hop to its initializer:
a local initialized with `new` that is never reassigned and never escapes the method - not stored to a
field, not passed as an argument, not captured by a lambda or local function - is a different instance on
every call, so it is reported.

A local that is an alias for shared state is correct and stays silent. A local initialized from a field,
a property, a parameter, or a method call is left alone, and so is a fresh object that is published
(`_gate = gate;`) - once it is stored where other callers can reach it, it is a legitimate lock target.

## How to fix violations

Store the lock object in a dedicated `private readonly` field and lock on that stored instance for every
`lock` that guards the same state.

```csharp
private readonly object _gate = new();

lock (_gate) { /* ... */ }
```

There is no automatic code fix: whether to promote the object to a field or to delete a no-op lock is a
design decision.

## When not to use the rule

There is no correct use of a lock that can never contend, so the rule has no routine exemption.
