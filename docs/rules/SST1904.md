# SST1904 - Do not lock on a non-readonly field

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | Yes (make the field readonly) |
| Default | Enabled |

## Cause

A `lock` statement targets an instance or static field of the declaring type that is not `readonly`.

```csharp
private object _gate = new();

public void M()
{
    lock (_gate) // SST1904
    {
    }
}
```

## Rule description

A lock only coordinates callers that take the same instance. When the lock target is a field that is
not `readonly`, any assignment to that field replaces the instance, and a caller that enters the lock
after the swap holds a different object than a caller already inside it. Both then run at once and the
state the lock was meant to protect is corrupted.

The rule reports a field that the declaring type owns and that is not externally accessible - a field
reachable from outside the type is reported by SST1901 instead, whose fix (a private dedicated object)
also closes this hole. A `readonly` field, a `const`, a local, a property, and a parameter are not
reported.

## How to fix violations

Declare the field `readonly` so its instance is fixed for the life of the type.

```csharp
private readonly object _gate = new();
```

On .NET 9 and later, with C# 13 or above, a dedicated lock field is better declared as a
`System.Threading.Lock`, which the compiler enters through a typed scope rather than `Monitor`. See
[PSH1300](PSH1300.md) for that upgrade once the field is `readonly`.

The code fix inserts `readonly`. It does not offer the fix when the field is assigned outside a
constructor - there the code is genuinely swapping the lock object, and the change is a decision for the
author.

## When not to use the rule

If a field is deliberately reassigned and every reassignment happens while no other thread can be inside
the lock, the rule is a false positive; suppress it at that field.
