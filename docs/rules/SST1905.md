# SST1905 - Do not use async void

| Item | Value |
| --- | --- |
| Category | Concurrency |
| Severity | Warning |
| Code fix | Yes for a method or local function (return Task) |
| Default | Enabled |

## Cause

An `async void` method, lambda, or local function that is not a genuine event handler.

```csharp
public async void Run() // SST1905
{
    await LoadAsync();
}

Action a = async () => await LoadAsync(); // SST1905
```

## Rule description

An `async void` member looks awaitable and looks catchable and is neither. The caller gets control back
at the first `await` with no task to wait on, and any exception the body throws is raised on whatever
thread resumed the method, with no one positioned to catch it. On the thread pool that ends the process.

The exemption is decided by shape, not declaration kind. A member whose delegate is the standard
event-handler shape - void-returning with an `(object sender, TEventArgs e)` parameter list where
`TEventArgs` derives from `EventArgs` - has no return value to hand a `Task` back through, and is left
alone. A method that overrides or implements a void member cannot change that signature and is left
alone. Every other void-returning delegate - `Action`, `Action<T>`, a custom void delegate - is the
fire-and-forget shape that crashes processes, and is reported. Local functions are reported too.
`async Task`, `async ValueTask`, and `async IAsyncEnumerable<T>` are never reported.

## How to fix violations

Return `Task` (or `ValueTask`) so a caller can await the work and observe its failures.

```csharp
public async Task Run()
{
    await LoadAsync();
}
```

For genuine fire-and-forget, make the discard explicit and log the failure rather than letting it
escape - `_ = RunAsync().ContinueWith(t => Log(t.Exception), TaskContinuationOptions.OnlyOnFaulted)`.

For a real event handler, keep `async void` and wrap the body in `try`/`catch`, because there is
nowhere else for the exception to go.

The code fix changes `void` to `Task` for a method or a local function. It is not offered for the lambda
form, whose delegate type belongs to the caller.

## When not to use the rule

If a project drives fire-and-forget work through a framework that owns the exception (some UI command
frameworks do), the crash argument does not apply; turn the rule off there.
