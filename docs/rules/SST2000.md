# SST2000 — Use ArgumentNullException.ThrowIfNull

| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled (where the helper exists) |

## Cause

A hand-written null check throws `ArgumentNullException`, where
`ArgumentNullException.ThrowIfNull` (introduced in .NET 6) would do the same in
one line.

```csharp
// 👎 reported
if (value is null)
{
    throw new ArgumentNullException(nameof(value));
}

// also reported (== null form, single-line throw)
if (value == null) throw new ArgumentNullException(nameof(value));

// 👍
ArgumentNullException.ThrowIfNull(value);
```

## Rule description

`ArgumentNullException.ThrowIfNull(argument)` checks for null and throws with the
correct parameter name — inferred from the argument expression via
`CallerArgumentExpression` — replacing the boilerplate `if`/`throw`.

The rule only fires when the matched constructor would not lose information:

- no constructor argument, or
- a single `nameof(value)` / `"value"` argument that already names the checked
  expression.

A constructor carrying a **custom message** (the `(paramName, message)` overload)
is left alone, because `ThrowIfNull` cannot preserve that message.

### Availability detection

This rule targets a **runtime API**, not a language feature, so it is gated on
`ArgumentNullException.ThrowIfNull` actually existing in the referenced framework
(detected once per compilation via `Compilation.GetTypeByMetadataName`). On
.NET 5 / .NET Framework and earlier — where the helper does not exist — the rule
reports nothing rather than suggesting code that would not compile. No target
framework string is parsed.

## How to fix violations

Replace the guard with `ArgumentNullException.ThrowIfNull(value);`. The provided
code fix does this automatically.
