# SST2010 — Read the clock through a TimeProvider

| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |
| Code fix | No |
| Default | **Disabled** (opt-in) |

## Cause

A type reads the machine clock directly: `DateTime.Now`, `DateTime.UtcNow`, `DateTimeOffset.Now` or
`DateTimeOffset.UtcNow`.

```csharp
public sealed class Session
{
    private readonly DateTime _started = DateTime.UtcNow;   // SST2010

    public bool Expired => DateTime.UtcNow - _started > TimeSpan.FromMinutes(20);   // SST2010
}
```

## Rule description

A type that calls `DateTime.UtcNow` reaches out of itself for a value nobody can control. The
behaviour that depends on it cannot be tested without waiting for real time to pass — or without
changing the machine's clock, which is not a test, it is a threat.

`TimeProvider` is the seam. The production caller passes `TimeProvider.System`; the test passes a
fake and moves time wherever it likes.

```csharp
public sealed class Session
{
    private readonly TimeProvider _time;
    private readonly DateTimeOffset _started;

    public Session(TimeProvider time)
    {
        _time = time;
        _started = time.GetUtcNow();
    }

    public bool Expired => _time.GetUtcNow() - _started > TimeSpan.FromMinutes(20);
}
```

### Why it is off by default

Introducing the seam is a **design change**, not a cleanup. It changes how the type is constructed
and therefore how every caller constructs it, and not every type wants that: a logger stamping a
line, a diagnostic dump, a one-shot script — for these the clock is not part of the logic and the
indirection buys nothing.

Turn it on where time *is* part of the logic — expiry, scheduling, retry backoff, rate limiting,
anything with a deadline — and leave it off elsewhere:

```ini
dotnet_diagnostic.SST2010.severity = warning
```

### The .NET 8 gate

`TimeProvider` arrived in .NET 8. The rule resolves `System.TimeProvider` once per compilation and
**registers nothing at all when it is absent**, so a project targeting a framework that has no
`TimeProvider` never sees a diagnostic it could not act on — and pays nothing for the analysis.

### What is reported

A read of one of the four clock properties from inside a type declaration. A read outside any type —
a top-level statement — is not reported: there is no constructor to inject into.

Reading through an injected provider (`_time.GetUtcNow()`, `_time.GetTimestamp()`) is, of course,
clean.

## How to fix violations

There is **no code fix**. The fix adds a constructor parameter and rewrites every construction of
the type — potentially across projects, potentially on a public API. A diagnostic that makes you
look is the honest outcome.
