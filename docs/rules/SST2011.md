# SST2011 ŌĆö Record instants in UTC

| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

An instant is recorded from the **local** clock: a `DateTime.Now`, `DateTimeOffset.Now`,
`DateTime.Today` or `DateTimeOffset.Now.DateTime` read that is stored in a field or property, or
returned.

```csharp
public sealed class AuditEntry
{
    private readonly DateTime _recorded = DateTime.Now;   // SST2011

    public DateTime Recorded => _recorded;
}
```

## Rule description

Local time is a presentation format, not a point in time. An instant stored as local time is
ambiguous twice a year ŌĆö the hour that repeats when the clocks go back genuinely maps to two
different moments ŌĆö and it means something different again the moment the process moves to another
machine. Record UTC; convert to local only where a human reads it.

```csharp
public sealed class AuditEntry
{
    private readonly DateTime _recorded = DateTime.UtcNow;

    public DateTime Recorded => _recorded;

    public string ForDisplay() => _recorded.ToLocalTime().ToString("f");
}
```

### The four reads that carry a local instant

| Read | What it hands back |
| --- | --- |
| `DateTime.Now` | the local time, `DateTimeKind.Local` |
| `DateTimeOffset.Now` | the local time, with the machine's current offset |
| `DateTime.Today` | local midnight, `DateTimeKind.Local` ŌĆö the offset is discarded exactly as `Now` discards it |
| `DateTimeOffset.Now.DateTime` | the local time with the offset **thrown away**, `DateTimeKind.Unspecified` |

`DateTimeOffset.Now.DateTime` is the worst of the four. The value on the left of the `.DateTime` was
carrying the one piece of information that made the instant unambiguous, and the read drops it ŌĆö
leaving a `DateTime` that does not even record which zone it came from.

These two are **not** reported, because they already carry the UTC instant:

```csharp
private readonly DateTime _fromLocalClock = DateTimeOffset.Now.UtcDateTime;   // ok
private readonly DateTime _fromUtcClock = DateTimeOffset.UtcNow.DateTime;     // ok
```

### What "recorded" means

The clock read has to *be* the value being kept ŌĆö not part of an expression that consumes it. So
these are reported:

| Position | Example |
| --- | --- |
| A field initializer | `private DateTime _at = DateTime.Now;` |
| A property initializer | `public DateTime At { get; set; } = DateTime.Now;` |
| A store into a field or property | `_at = DateTime.Now;` |
| A returned expression | `return DateTime.Today;` |
| An expression-bodied member | `public DateTime At => DateTimeOffset.Now.DateTime;` |

And these are clean, because the value never escapes the expression that read it:

```csharp
public string Greeting() => DateTime.Now.Hour < 12 ? "morning" : "afternoon";   // ok
public string Stamp() => DateTime.Now.ToString("t");                            // ok
public DayOfWeek Weekday() => DateTime.Today.DayOfWeek;                         // ok
```

Displaying local time is what local time is for. A store into a **local variable** is also
not reported: it does not outlive the statement it was written for.

### The other rules that read a clock

[SST1451](SST1451.md) asks whether a `DateTime` being **constructed** states its `DateTimeKind` ŌĆö
`new DateTime(2024, 1, 1)` is `Unspecified`, and later conversions guess. This rule asks whether an
instant **read from the clock** came from the right clock. They never overlap: SST1451 only looks at
object creations and never at a property read; SST2011 only looks at property reads and never at an
object creation.

[SST2017](SST2017.md) reports a `.Date` read that proves a `DateTime` was only ever a date. It steps
aside for a read taken straight off the clock, so neither `DateTime.Today` nor the
`DateTime.UtcNow.Date` this rule's fix produces is reported by it.

[SST2016](SST2016.md) reports a `DateTime` on a **visible signature** ŌĆö the declared type, never an
expression ŌĆö so it never lands on the same span as this rule, though a public `DateTime` member whose
body reads a local clock trips both, on the type and on the expression.

[SST2010](SST2010.md) asks a different question again: not *which* clock, but whether the type should
be reading a clock at all rather than taking a `TimeProvider`. It reports every direct read ŌĆö
`Now` and `UtcNow` alike ŌĆö so a local-clock read that this rule reports is reported by SST2010 too
when SST2010 is turned on. It is **off by default** for exactly that reason: adopting a
`TimeProvider` is a design change, and where a codebase has not made it, only this rule speaks.
`DateTime.Today` is not one of the reads SST2010 recognizes, so it is reported by this rule alone.

## How to fix violations

The code fix moves the read to the UTC clock.

| Reported | Rewritten to |
| --- | --- |
| `DateTime.Now` | `DateTime.UtcNow` |
| `DateTimeOffset.Now` | `DateTimeOffset.UtcNow` |
| `DateTime.Today` | `DateTime.UtcNow.Date` |
| `DateTimeOffset.Now.DateTime` | `DateTimeOffset.UtcNow.UtcDateTime` |

```diff
- private readonly DateTime _recorded = DateTime.Now;
+ private readonly DateTime _recorded = DateTime.UtcNow;
```

This **changes the recorded value** ŌĆö that is what the rule is asking for. If the stored value is
later formatted for a user, add the `ToLocalTime()` at the point of display, which is where it
belongs. If some other code round-trips the value through a local-time format, fix that too; the
diagnostic has found a real bug, not a style problem.

### Why `UtcDateTime` and not `DateTime`

`DateTimeOffset.Now.DateTime` becomes `DateTimeOffset.UtcNow.UtcDateTime`, not
`DateTimeOffset.UtcNow.DateTime`. The two agree on the ticks ŌĆö `UtcNow` has an offset of zero, so
its local time *is* its UTC time ŌĆö but they disagree on the `DateTimeKind` they carry:

- `.DateTime` hands back `DateTimeKind.Unspecified`, which is the very ambiguity the rule exists to
  remove. A later `ToLocalTime()` or `ToUniversalTime()` on it guesses.
- `.UtcDateTime` hands back `DateTimeKind.Utc`, so every later conversion knows what it is holding.

### The rewrite is proved before it is offered

The rewritten read is bound speculatively and its type is compared with the type of the read it
replaces. A `Now` on a type of your own is never rewritten into a `UtcNow` that does not exist, and a
rewrite that would hand back a different type is never offered.
