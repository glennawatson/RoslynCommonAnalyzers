# SST2012 — Use Guid.Empty for the empty GUID

| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A GUID is constructed with the parameterless constructor.

```csharp
private Guid _id = new Guid();   // SST2012
```

## Rule description

`new Guid()` does **not** make a new GUID. It makes the all-zero one —
`00000000-0000-0000-0000-000000000000` — which is the opposite of what the code appears to say. The
reader who wanted a fresh identifier has to already know that `Guid.NewGuid()` is the one that
generates, and that this one does not.

`Guid.Empty` says what the value is:

```csharp
private Guid _id = Guid.Empty;
```

If a *fresh* identifier was what you meant, the fix is not this rule's — it is `Guid.NewGuid()`, and
only you can know which you wanted. That is precisely why the diagnostic exists: the code as written
cannot tell the difference.

### What is not reported

```csharp
private Guid _fresh = Guid.NewGuid();                                       // ok — generates
private Guid _seeded = new Guid("00000000-0000-0000-0000-000000000001");    // ok — means it
private Guid _named = Guid.Empty;                                            // ok — already named
private Guid _defaulted = default;                                           // ok — says "default"
```

A parameterless construction of some **other** type is not reported, even if that type is called
`Guid`: the construction is bound, and only `System.Guid` matches.

A target-typed `new()` in a `Guid`-typed position *is* reported — it has exactly the same meaning
and exactly the same problem.

## How to fix violations

The code fix replaces the construction with `Guid.Empty`, spelled the way the construction was:

```diff
- private Guid _id = new Guid();
+ private Guid _id = Guid.Empty;

- private System.Guid _id = new System.Guid();
+ private System.Guid _id = System.Guid.Empty;
```

A target-typed `new()` has no spelling to borrow, so `Guid.Empty` is tried first and the fully
qualified name is the fallback for a file with no `using System`. Each candidate is bound
speculatively at the site it would occupy, and the fix is only offered once one of them resolves to
`System.Guid.Empty` — so the fix never produces code that does not compile.
