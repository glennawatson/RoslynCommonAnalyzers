# SST2015 — Do not bury an increment inside a larger expression

| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `++` or `--` is used inside a larger expression, so its value and its side effect happen in the
middle of something else.

```csharp
array[i++] = Compute(i);       // SST2015
Send(x++ + y);                 // SST2015
var pair = values[i++] + values[i++];   // SST2015, twice
```

## Rule description

`array[i++] = Compute(i)` has a value that depends on which side is evaluated first, and readers
reliably guess wrong. (C# evaluates the left, so `Compute` sees the *incremented* `i` — which is
almost never what the author meant, and is certainly not what the line looks like it says.)

The increment is doing two jobs — producing a value *and* advancing a variable — and only one of
them is visible where it is written. Give it its own statement:

```csharp
array[i] = Compute(i);
i++;
```

The cost is a line. The gain is that the code means one thing.

### What is not reported

An increment that **is** the expression is not buried, and is never reported. In all of these the
increment happens once, in a place with nothing to be ordered against:

```csharp
i++;                                  // a statement of its own
for (var i = 0; i < n; i++) { }       // the loop's incrementor
var next = _index++;                  // the whole value being stored
_index = counter++;                   // the whole right-hand side of a store
return _next++;                       // the whole returned value
public int Next() => _next++;         // the whole expression body
Action bump = () => i++;              // the whole lambda body
```

A **compound** assignment is reported, though — `total += i++` reads its target as well as writing
it, so the order is back in question.

## How to fix violations

There is **no code fix**, and that is not an omission.

Hoisting the increment into its own statement **changes when it runs** relative to the rest of the
expression — which is the whole point of the diagnostic. `array[i++] = Compute(i)` and

```csharp
var index = i;
i++;
array[index] = Compute(i);
```

do not compute the same thing as

```csharp
array[i] = Compute(i);
i++;
```

and only the author knows which one was intended. A fix that silently picked one would be changing
behaviour while claiming to clarify it. The rule's job is to make you look at the line; deciding
what it should have said is yours.
