# SST2016 — Expose DateTimeOffset rather than DateTime

| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `DateTime` is the declared type of an **externally visible** field, property, parameter or return type —
the places a moment leaves the assembly.

```csharp
public sealed class Booking
{
    public DateTime Start { get; set; }             // SST2016

    public DateTime? Cancelled { get; set; }        // SST2016

    public DateTime Ends(DateTime start) => start;  // SST2016 twice — return type and parameter
}
```

## Rule description

A `DateTime` says what the clock read. It does not say where the clock was standing.

Inside one method that is survivable — everybody looking at the value can see where it came from. The
moment it crosses a boundary it is no longer survivable: a caller in another time zone, a row in a
database, a JSON payload on a wire. The receiver has to guess an offset, and the guess is *its* local
zone rather than the writer's. `DateTimeKind` is not a rescue: it is one of three values, two of which
mean "unspecified" in practice, and it is routinely lost by serializers.

`DateTimeOffset` carries the offset with the value, so the instant it names is the same instant
everywhere:

```csharp
public sealed class Booking
{
    public DateTimeOffset Start { get; set; }

    public DateTimeOffset? Cancelled { get; set; }

    public DateTimeOffset Ends(DateTimeOffset start) => start;
}
```

### What is reported

Only the **declared type**, and only at a boundary:

- a field, a property, a parameter, or a return type;
- on a member that is visible outside the assembly (`public`, `protected`, `protected internal`, inside
  types that are themselves visible), including a positional record parameter and a delegate signature.

### What is not reported

- **Clock reads.** `DateTime.Now`, `DateTime.UtcNow`, `DateTime.Today` — those are expressions, and the
  question of where the time came from belongs to [SST2010](SST2010.md). This rule never looks at an
  expression, so the two can never argue about the same code.
- **Locals.** A value that never leaves the method cannot be misread by anybody else.
- **`private` and `internal` members.** The hazard is the boundary. Inside the assembly the type can be
  changed by whoever needs it changed.
- **An `override`, or an implementation of an interface member.** The type was chosen by the declaration
  above; that declaration is what gets reported (if it lives in this compilation), and it is the only
  place a change would work.

### Divergence from the analyzer the rule

the rule ("Prefer using `DateTimeOffset` instead of `DateTime`") reports **expressions that produce a
`DateTime`** — `new DateTime(...)`, `DateTime.Now`, `DateTime.UtcNow`, `DateTime.Today`,
`DateTime.MinValue`, `DateTime.UnixEpoch` — and says nothing about declared types at all. SST2016 does
the opposite on purpose:

- the clock reads the rule flags are already SST2010's, and two rules shouting at one line is a defect;
- `new DateTime(...)` with an explicit `DateTimeKind` is flagged by the rule anyway, which is noise; and
- the expression is not where the damage is done. A `DateTime` local converted to a `DateTimeOffset`
  before it is stored is fine. A `public DateTime` property is not, no matter how carefully it was
  computed.

So the two rules do not overlap and are not substitutes: SST2016 reports the signature, the rule reports
the value.

## When to suppress

- **A codebase that lives in one fixed zone**, by design and by contract — an on-premise line-of-business
  application where every machine, every user and every database sits in the same zone, and always will.
  The offset is a constant there and writing it down buys nothing.
- **Interop.** An API you do not own takes or returns `DateTime` (many ORMs, `System.Data`, older
  contracts). Converting at the edge is right; converting in the middle of a call chain just to satisfy a
  rule is not.

```ini
dotnet_diagnostic.SST2016.severity = none
```

## Limits

`DateTimeOffset` records an **offset, not a time zone**. `+01:00` does not say *which* zone was +1, and a
zone's offset changes across the year. That is enough for a past instant — the offset was whatever it was
— but it is **not** enough for a *future* local time: "09:00 in Berlin on 3 March next year" cannot be
stored as an offset today, because the rules for that day may change. Store the moment *and* the IANA
zone id (`Europe/Berlin`) alongside it, and resolve with `TimeZoneInfo` when the time comes.

`DateTimeOffset` is the right default. It is not a complete time-zone strategy.

## The gate

`System.DateTimeOffset` is resolved once per compilation. If it is absent the rule **registers nothing at
all** — no diagnostic that cannot be acted on, and no analysis cost.
