| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |

## Cause

A `DateTime` is reduced to half of itself and stays a `DateTime`:

```csharp
public DateTime Birthday(DateTime born) => born.Date;          // SST2017

public TimeSpan Opening(DateTime when) => when.TimeOfDay;      // SST2017
```

## Rule description

`.Date` does not throw the time away â€” it sets it to midnight and hands back another `DateTime`, `Kind`
and all. `.TimeOfDay` hands back a `TimeSpan`: a *duration*, standing in for a clock face.

Neither type says the missing half is meaningless, so nothing stops the next line from treating it as
real. A birthday stored as a `DateTime` compares unequal to the same birthday that came back from a
database with a different `Kind`; it shifts a day when it crosses a time zone; it is one careless
`AddHours` away from being a different date. None of that is possible if the type simply has no time
component:

```csharp
public DateOnly Birthday(DateTime born) => DateOnly.FromDateTime(born);

public TimeOnly Opening(DateTime when) => TimeOnly.FromDateTime(when);
```

`DateOnly` has no time to accidentally compare, no `Kind`, and nothing for a time-zone conversion to
move. `TimeOnly` is a time on a clock face rather than an elapsed span, so `23:00 + 2h` wraps to `01:00`
instead of becoming 25 hours.

### What is reported

Exactly two property reads, and only when the receiver is a `System.DateTime`:

| Written | Suggested |
| --- | --- |
| `value.Date` | `DateOnly.FromDateTime(value)` |
| `value.TimeOfDay` | `TimeOnly.FromDateTime(value)` |

These are the only two places the code has already *proved* what it meant. The rule does **not** guess
from names: a property called `ExpiryDate` may well need a time, and a rule that read intent out of an
identifier would be a false-positive machine.

### What is not reported

- **A clock read as the receiver** â€” `DateTime.Now.Date`, `DateTime.UtcNow.Date`. That expression belongs
  to [SST2010](SST2010.md), which asks the type to take a `TimeProvider`; one line does not need two
  rules arguing over it. Fix the clock read first and the `.Date` on the result is reported like any
  other.
- **A `DateTimeOffset` receiver.** `offset.Date` and `offset.TimeOfDay` exist too, but naming *the date*
  of an instant means deciding whose date â€” the offset's, UTC's, or some zone's â€” and that is a decision
  the rule has no business making for you.
- **Any other member.** `.Day`, `.Hour`, `.Year` and friends are components, not a value that is standing
  in for something it is not.
- **A `Date` or `TimeOfDay` on a type of your own.** Only `System.DateTime`'s properties bind.

## How to fix violations

There is **no code fix**. Changing the expression alone would just move the problem â€” the field,
property, or parameter holding the result usually wants to change type too, and that is a decision about
your model, not a mechanical rewrite. Follow the value: give it a `DateOnly`/`TimeOnly` home, then
convert at the edges where an API still insists on `DateTime`
(`birthday.ToDateTime(TimeOnly.MinValue)`).

## When not to use it

- **A value that really is an instant with the time zeroed** â€” a "start of day" boundary you are about to
  compare against real `DateTime` timestamps in the same zone. Converting to `DateOnly` there just means
  converting back again.
- **A framework or ORM that cannot map `DateOnly`.** Older EF Core providers, some serializers, and many
  legacy contracts only understand `DateTime`. Keep `DateOnly` in the domain and convert at that edge â€”
  or, if the edge is everywhere, turn the rule off:

```ini
dotnet_diagnostic.SST2017.severity = none
```

## The gate

`System.DateOnly` and `System.TimeOnly` (both .NET 6) are resolved **independently**, once per
compilation. With neither present, the rule registers nothing at all: a project targeting .NET Framework
or .NET Standard 2.0 never sees a diagnostic it could not act on, and pays nothing for the analysis.
With only one present, only the half you can actually write is reported.
