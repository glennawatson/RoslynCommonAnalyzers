# SST2018 — A null check beside a type pattern is redundant

| Item | Value |
| --- | --- |
| Category | Modernization |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

An explicit null test is combined with an `is` type pattern on the same value.

```csharp
public bool M(object o) => o != null && o is string; // SST2018
```

## Rule description

The type pattern already answers the null question: `o is T` is false for a null `o` for every `T`, including a
nullable value type whose `HasValue` is false. So the null test decides nothing and can be dropped, leaving
`o is T`.

Three shapes are recognised, all reducing to a single pattern test:

- `o != null && o is T` and `o is not null && o is T` reduce to `o is T`.
- `o == null || o is not T` reduces to `o is not T`.
- the combinator form `o is not null and T` reduces to `o is T`.

A test that keeps a non-null value which is *not* a `T` (`o != null && o is not T`) is a different, real check —
it distinguishes "a non-null non-`T`" from "null" — and is left alone. So is "null or a `T`"
(`o == null || o is T`).

## How to fix violations

Drop the null test and keep the pattern.

```csharp
public bool M(object o) => o is string;
```

The code fix collapses each recognised shape to its pattern test: the `&&` and combinator forms to `o is T`, the
`||` form to `o is not T`.

## Performance

The clean path is a syntax-kind test that rejects every ordinary `&&` and `||` before anything binds. Operand
equivalence is a token compare, not a semantic one, so the two sides are matched without consulting the model.
