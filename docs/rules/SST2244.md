# SST2244 — Numeric literal suffixes should be upper case

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Enabled |

## Cause

A numeric literal's type suffix is written with a lower-case letter.

```csharp
long ticks = 1l;
ulong mask = 1ul;
float ratio = 1.5f;
decimal price = 19.99m;
```

## Rule description

In most programming fonts a lower-case `l` and the digit `1` are all but the same glyph. `1l` is
"one, long" but it reads as eleven, and `10l` reads as one hundred and one. There is nothing to
puzzle out with `1L`.

That is the whole argument, and it only applies to `l`. The other suffixes — `u`, `f`, `d`, `m` — are
not ambiguous with anything. They are included so that the rule is one rule rather than a special
case, and so a file does not end up with `1L` next to `1.5f`. If you disagree with that half, this
rule is not for you; the `l` half is the part worth having.

```csharp
long ticks = 1L;
ulong mask = 1UL;
float ratio = 1.5F;
decimal price = 19.99M;
```

## The suffix, and only the suffix

The rule works on the literal's token text and never looks at what the number means. It finds the
trailing run of suffix characters and reports it if any of them is lower case. Everything before that
run is left exactly as written.

That distinction matters for hexadecimal:

```csharp
long mask = 0xffl;   // SST2244 — the 'l' is the suffix
long mask = 0xffL;   // fixed — 'ff' are digits and are none of the rule's business
```

**Hex digit casing is a separate concern, and this rule takes no position on it.** `0xff` and `0xFF`
are the same value and the same number of characters; whether your codebase prefers one is a
different argument from whether `1l` looks like eleven. The fix will never touch them.

The rule knows which characters can be a suffix in which base, so it does not mistake digits for
suffixes:

| Literal | Suffix | Reported |
| --- | --- | --- |
| `0xff` | none — `f` and `f` are hex digits | No |
| `0xffd` | none — `d` is a hex digit | No |
| `0xffL` | `L` | No |
| `0xffl` | `l` | Yes |
| `1.5e3f` | `f` — the exponent is not a suffix | Yes |
| `1_000l` | `l` — separators are digits | Yes |

Two-character suffixes are reported and fixed as a unit, so `1lu` becomes `1LU` and a half-cased
`1Lu` becomes `1LU`.

## How to fix violations

Upper-case the suffix. The code fix does exactly that and nothing else — the digits, any digit
separators, the exponent, and the `0x`/`0b` prefix are copied through character for character.

```csharp
var a = 1l;        // becomes 1L
var b = 1ul;       // becomes 1UL
var c = 0xffl;     // becomes 0xffL   (not 0xFFL)
var d = 1_000_000d;// becomes 1_000_000D
```

## Language versions

Every suffix this rule touches has existed since C# 1.0, and upper-casing one is valid in every
language version and on every target framework. There is nothing to gate: the rule reads a token and
rewrites a token, and it never consults the semantic model.

## Performance

The clean path is one scan backwards from the end of the literal's token text, stopping at the first
character that cannot be part of a suffix — for a literal with no suffix, that is the first character
it looks at. A literal whose suffix is already upper case costs one extra pass over one or two
characters. Nothing is allocated unless a diagnostic is reported, and the semantic model is never
touched.
