# SST2245 — Use while when a for loop has no initializer or increment

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Enabled |

## Cause

A `for` statement has a condition but no initializer and no incrementor.

```csharp
for (; reader.Read(); )
{
    Process(reader);
}
```

## Rule description

A `for` loop makes three promises in its header: here is where the counter starts, here is when we
stop, here is how we step. When two of the three are empty, the header still asks the reader to check
them — the eye has to land on `(;` and on `;)` and confirm there is nothing there — and then arrive
at the same place a `while` would have said outright.

```csharp
while (reader.Read())
{
    Process(reader);
}
```

The two loops compile to the same thing. `while` says it in fewer words, and it says it without the
two empty clauses that look, at a glance, like something was deleted from them.

## What is not reported

**`for (;;)` — the infinite loop.** With no condition either, `for (;;)` is the language's idiomatic
"loop forever". There is no condition to move into a `while`, and rewriting it to `while (true)`
would trade one idiom for another with nothing gained. It is left alone.

```csharp
for (;;)                      // not reported
{
    if (!TryPump()) break;
}
```

**Any loop that still uses a clause it declared.** If the header carries an initializer, a
declaration, or an incrementor, it is doing the job a `for` exists for:

```csharp
for (var i = 0; i < count; i++)  { }   // not reported
for (; i < count; i++)           { }   // not reported — the incrementor is real
for (i = 0; i < count;)          { }   // not reported — the initializer is real
for (int i = 0, j = n; i < j; i++) { } // not reported
```

## How to fix violations

The code fix rewrites the loop as a `while`, keeping the condition and the body exactly as they are.

```csharp
// before
for (; queue.Count > 0; )
{
    Handle(queue.Dequeue());
}

// after
while (queue.Count > 0)
{
    Handle(queue.Dequeue());
}
```

The rewrite is behaviour-preserving, including for `continue`: in a `for` with no incrementor,
`continue` jumps to the (empty) incrementor clause and then re-tests the condition, which is exactly
what `continue` does in a `while`.

The fix reuses the loop's own tokens rather than building fresh ones, so the condition, the body, the
indentation, and any comments come through untouched.

## Language versions

`while` has been in the language since C# 1.0, so the rewrite is valid on every language version and
every target framework. The rule is pure syntax and never binds a symbol, so there is nothing to gate.

## Performance

The check is four field reads on the `for` statement node — is there a condition, is there a
declaration, are the initializer and incrementor lists empty — and no allocation on the clean path.
The semantic model is never consulted.
