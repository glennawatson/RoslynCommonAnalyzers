# SST2250 — Join a declaration with its first assignment

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Enabled |

## Cause

A local is declared without a value, and the very next statement is a straight-line assignment that
gives it its first value.

```csharp
int total;
total = seed * 2;
```

## Rule description

Splitting a local's declaration from its first value makes the reader hold the name across a gap that
carries no meaning: the declaration says a `total` exists but not what it is, and the answer only
arrives on the next line. Putting the value on the declaration keeps the name and its first value in
one place.

```csharp
int total = seed * 2;
```

The rule reports only when the assignment is the **immediately following** statement, at the same
block level, writing the same local by a plain `name = value`. Because nothing sits between the
declaration and that assignment, nothing can read the local in between, and joining the two moves
neither the order in which anything is evaluated nor the point at which the local becomes definitely
assigned.

## What is not reported

**A declaration that already has a value.** The rule only joins a *bare* declaration; a declaration
that is initialized and then overwritten is a different shape.

```csharp
int x = 0;   // not reported
x = 5;
```

**A first write that is conditional or nested.** When the next statement is an `if`, a loop, a `try`,
or anything other than a plain assignment at the same block level, the first write is not a
straight-line one and joining could change what runs.

```csharp
int x;                       // not reported
if (flag)
{
    x = 5;
}
else
{
    x = 6;
}
```

**A use of the local before the assignment.** When the local is passed by `out`, `ref`, or otherwise
touched before the plain assignment, that earlier statement — not the assignment being looked at — is
the real first write, so the two are left apart.

```csharp
int x;                       // not reported
int.TryParse(text, out x);
x = 10;
```

**A multi-variable declaration**, a declaration that is the last statement in its block, a following
assignment to a *different* local, and a following assignment to a field or property (`this.x = …`)
are all left alone.

## How to fix violations

The code fix moves the assigned value onto the declaration and removes the separate assignment
statement. The declaration keeps its position, its type, and its surrounding trivia; only the value
travels.

```csharp
// before
// Compute the running total.
int total;
total = seed * 2;

// after
// Compute the running total.
int total = seed * 2;
```

Because the assignment already compiled against the declared type, the joined initializer binds the
same way, so the merged declaration compiles whenever the original did.

## Language versions

Declaring a local with an initializer has been in the language since C# 1.0, so the fix is valid on
every language version and every target framework. There is nothing to gate.

## Performance

The check runs on each local declaration: it confirms a single uninitialized declarator, finds the
declaration's slot in its block, and inspects the one following statement. It is pure syntax, allocates
nothing on the clean path, and never consults the semantic model. Inside the block the local's name
cannot be shadowed, so the assignment target is matched by name without binding a symbol.
