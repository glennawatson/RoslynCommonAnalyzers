# SST2251 — Let inference supply the type arguments

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Enabled |

## Cause

A method call names its type arguments explicitly, and inference would supply the same ones from the
call's arguments.

```csharp
var box = Create<int>(42);
```

## Rule description

`Create<int>(42)` and `Create(42)` bind to the same constructed method: the compiler infers `T` is
`int` from the argument `42`, so writing `<int>` says again what the call already establishes. The
type-argument list is noise the reader has to check against the arguments to confirm it agrees.

```csharp
var box = Create(42);
```

The rule is reported only when it can prove the two forms are the same call. It binds the invocation
once as written, then binds it again with the type-argument list removed, and reports only when the
second form resolves to the identical constructed method — the same original definition and the same
inferred type arguments. If the shortened call would bind to anything else, or fail to bind, nothing
is reported.

## What is not reported

**Type arguments inference cannot reproduce.** When the type parameter does not appear in a parameter,
or the call has no arguments to infer from, dropping the list is a compile error, so the call is left
alone.

```csharp
T Default<T>() => default;          // T comes from nowhere at the call site
void Log<T>(object value) { }       // T is in no parameter

var x = Default<int>();             // not reported — <int> is required
Log<int>("text");                   // not reported — inference has nothing to work with
```

**Calls whose meaning would change.** If removing the list moves the call to a different overload, or
lets inference pick different type arguments, the two forms are not the same call and the rule stays
silent.

```csharp
string Pick<T>(T value) => "generic";
string Pick(int value) => "specific";
T Identity<T>(T value) => value;

Pick<int>(42);        // not reported — Pick(42) binds to the non-generic overload
Identity<object>("text"); // not reported — inference would pick string, not object
```

**Type arguments that differ only in nullability.** The comparison includes nullable reference annotations,
so an explicit argument the inferred one would re-annotate is not treated as redundant — dropping it would
change the expression's type.

```csharp
#nullable enable
Task<object?> Wrap(object value) => Task.FromResult<object?>(value);
// not reported — dropping <object?> infers Task<object> from the non-null argument
```

**Type arguments that belong to a type.** Only the invoked method's type arguments are considered. Type
arguments on a receiver type, on an object creation, or on an attribute are usually required and are
never touched.

```csharp
Box<int>.Make(42);    // not reported — <int> belongs to Box, not Make
new List<int>();      // not an invocation
```

## How to fix violations

The code fix removes the type-argument list from the invoked name and keeps the identifier, the
receiver, the arguments, and the surrounding trivia exactly as they were.

```csharp
// before
var box = repository.Create<int>(42);

// after
var box = repository.Create(42);
```

## Language versions

Generic method type inference has been in the language since C# 2.0, so the rewrite is valid on every
language version and every target framework. The safety of the fix comes from binding the shortened
call, not from a version check.

## Performance

The clean path is a single syntax check: only a call whose invoked name carries a type-argument list
reaches the semantic model at all. The speculative bind — bind the call, then bind it once more without
the list — runs only for those candidates, so a project with no explicitly-parameterized calls pays
nothing beyond the syntax test.
