# SST2252 — Avoid nesting switch statements

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `switch` statement lexically contains another `switch` statement inside one of its sections.

```csharp
switch (transport)
{
    case Transport.Http:
        switch (method)
        {
            case Method.Get:
                return HandleHttpGet();
            case Method.Post:
                return HandleHttpPost();
            default:
                return NotSupported();
        }

    default:
        return NotSupported();
}
```

## Rule description

Two `switch` statements stacked at one point make the reader hold two governing values and both sets of
cases in their head at once. Every branch of the outer switch is now read through the lens of "and which
inner case are we in", and the section that hosts the inner switch grows into a procedure of its own.

Pulling the inner switch out restores one decision per place. It can become its own method:

```csharp
switch (transport)
{
    case Transport.Http:
        return HandleHttp(method);

    default:
        return NotSupported();
}
```

a `switch` expression when each arm is a value:

```csharp
var result = (transport, method) switch
{
    (Transport.Http, Method.Get) => HandleHttpGet(),
    (Transport.Http, Method.Post) => HandleHttpPost(),
    _ => NotSupported(),
};
```

or a lookup keyed on the pair. Each of these keeps one branch readable on its own.

The check is purely syntactic. Each `switch` statement is a candidate inner switch, and its enclosing
statements are walked outward. The first enclosing `switch` statement found means the inner one sits in
that switch's section — a statement can live nowhere else inside a `switch` — and the inner `switch`
keyword is reported. In a three-deep nest, the middle and the innermost switches are each reported against
the switch that directly encloses them.

## What is not reported

**A `switch` expression.** The `switch` expression is the preferred compact form and is never reported —
neither as the inner nor the outer side of a nesting. Only a `switch` *statement* is a candidate.

```csharp
switch (transport)
{
    case Transport.Http:
        return method switch      // not reported — a switch expression, not a nested statement
        {
            Method.Get => HandleHttpGet(),
            _ => NotSupported(),
        };

    default:
        return NotSupported();
}
```

**Sibling switches.** Two `switch` statements one after another in the same method are not nested, so
neither is reported.

**A switch inside a lambda or local function declared in a section.** A `switch` that lives inside a
lambda or a local function belongs to that body, not to the enclosing switch, so the nesting search stops
at the lambda or local-function boundary.

```csharp
switch (outer)
{
    case 1:
        int Local(int value)
        {
            switch (value)      // not reported — this switch is the local function's own concern
            {
                case 10:
                    return 100;
                default:
                    return 0;
            }
        }

        return Local(inner);

    default:
        return -1;
}
```

## How to fix violations

There is **no code fix** — turning nested control flow into a method, a `switch` expression, or a lookup
is a design decision, and the safe rewrite depends on what each branch does. Lift the inner switch into
its own method, collapse the pair into a `switch` expression when the arms are values, or replace the
mapping with a keyed lookup.

## Configuration

None.

## Performance

The rule runs from each `switch` statement and walks its ancestors, stopping at the first enclosing
`switch`, lambda, local function, or member declaration. A `switch` with no enclosing `switch` is rejected
after that short walk with no allocation and no semantic model, so a method full of flat, unrelated
switches pays only a handful of parent-pointer steps each.
