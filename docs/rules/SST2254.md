# SST2254 — Name the created type in an object creation

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Disabled (opt-in) |

## Cause

A target-typed object creation (`new(...)` or `new() { ... }`) leaves the constructed type to be recovered
from the target of the expression instead of naming it at the creation site.

```csharp
Connection connection = new();
```

## Rule description

`new(...)` reads the constructed type from wherever the expression lands — the declared variable, the field
or property, the method return type, the parameter. This rule takes the opposite view: it asks the creation
to name its own type, `new Type(...)`, so what is being built is visible where it is built, at the cost of
repeating the type name.

```csharp
Connection connection = new Connection();
```

This is a minority preference and the **inverse** of the default target-typed-`new` suggestion (SST2202),
so it ships **disabled**. A codebase enables at most one of the two directions:

- Leave **SST2202** enabled (the default) to steer toward `new()`.
- Disable **SST2202** and enable **SST2254** to steer toward `new Type()`.

Enabling both would report the same code from both directions, so pick one.

The rule reports only when it can name the type safely. It resolves the created type from the creation's
type information, splices the type's minimally-qualified name back between `new` and the argument list, and
binds that `new Type(...)` speculatively. The diagnostic is raised only when the rebuilt form resolves to the
identical type, so a name that would collide, fail to bind, or construct a different type is never reported.

## Examples

The rule fires wherever a target-typed `new` has a resolvable, source-expressible type:

```csharp
Connection field = new();                    // typed local / field / property initializer
public Connection Property { get; } = new();
public Connection Make() => new();            // return with an explicit return type
Accept(new());                               // argument bound to a typed parameter
List<int> values = new() { 1, 2 };           // arguments and initializer are kept
```

## What is not reported

**A type with no plain source name.** The created type must be expressible as a type name that can stand
between `new` and the arguments. A tuple, an anonymous type, a pointer or function pointer, `dynamic`, and an
error type are all left alone.

```csharp
(int, int) pair = new();   // not reported — a tuple has no 'new (int, int)()' spelling here
```

**A type the explicit form would not reconstruct.** If binding `new Type(...)` would resolve to a different
type, fail to bind, or not compile, nothing is reported and no fix is offered.

## How to fix violations

The code fix inserts the created type's minimally-qualified name between `new` and the argument list, keeping
the arguments, the initializer, and the surrounding trivia exactly as they were.

```csharp
// before
Connection connection = new();

// after
Connection connection = new Connection();
```

## Enabling the rule

The rule is off by default. Turn it on in `.editorconfig`:

```ini
[*.cs]
dotnet_diagnostic.SST2254.severity = warning
```

A team that adopts SST2254 will usually also turn off the opposite direction:

```ini
[*.cs]
dotnet_diagnostic.SST2202.severity = none
dotnet_diagnostic.SST2254.severity = warning
```

## Language versions

Target-typed `new` is a C# 9 feature, so a `new(...)` expression only parses when the language version already
supports it; there is nothing to report on older versions. The explicit `new Type(...)` form is valid on every
language version, and the safety of the fix comes from binding the rewritten expression rather than from a
version check.

## Performance

The rule is registered on the target-typed-creation node alone, so a source with no `new(...)` expressions
never reaches the semantic model. For each candidate it resolves the type once and rejects anything without a
source name before allocating a display string; the speculative bind runs only for the survivors.
