# SST2264 — Name the enum member instead of casting a number

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Enabled |

## Cause

A numeric literal is cast to an enum type instead of naming the member.

```csharp
var options = (RegexOptions)1;
```

## Rule description

A numeric literal cast to an enum hides which member it means behind the underlying value, so
the reader has to know the enum's numbering to read the code. Naming the member states the
intent and survives a renumbering of the enum.

```csharp
var options = RegexOptions.IgnoreCase;
```

The rule resolves the literal to its single matching named member and rewrites the cast to
name it. The rewrite is bound speculatively so it is only offered when `Type.Member` resolves
back to the same value.

## What is not reported

**A value that combines members.** `(RegexOptions)3` names no single member (it is a
combination), so the rule leaves it alone rather than guess at an OR of members.

**A value that matches no member.** `(RegexOptions)512` with no member at that value has no
name to offer, so it is left alone.

**A cast of a non-literal expression.** `(RegexOptions)value`, where `value` is not a literal,
is not a numbering the rule can resolve and is left alone.

## How to fix violations

The fix replaces the cast with the named member.

```csharp
// before
Configure((FileMode)2);

// after
Configure(FileMode.Create);
```

## Language versions

Named enum members are valid on every language version and target framework, so the fix
applies wherever the enum is defined.

## Performance

The rule registers on cast expressions, matches a numeric-literal-to-enum shape, and only
then walks the enum's members to find the single match. The clean path allocates nothing.
