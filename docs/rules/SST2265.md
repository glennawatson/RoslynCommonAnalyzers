# SST2265 — Fold consecutive fluent calls into one chain

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Disabled (opt-in) |

## Cause

Two or more consecutive statements each call a member on the **same receiver** and get that receiver back,
so together they are a single fluent chain written out as separate statements.

```csharp
builder.Append("a");
builder.Append("b");
builder.Append("c");
```

## Rule description

A fluent API returns its receiver so calls can be chained. When each call is written as its own statement,
the receiver name is repeated on every line and the reader has to notice that it is the same object each time.
Folding the run into one chain names the receiver once.

```csharp
builder.Append("a").Append("b").Append("c");
```

This is a house-style preference — separate statements are perfectly clear, and some teams prefer them — so
the rule ships **disabled**. Enable it in `.editorconfig` for a codebase that wants fluent runs chained.

The rule reports only when the fold is safe:

- **The receiver is side-effect-free.** It must be an identifier, `this`, or a member-access chain of those,
  so evaluating it once in the chain matches evaluating it once per statement. A receiver that is itself a call
  (`Get().Append(...)`) is left alone, because the chain would evaluate it once instead of on every line.
- **Every call returns the receiver's own type.** Each statement's call must return the same type as the
  receiver, which is what makes the next `.Method(...)` bind. A call that returns `void` or a different type —
  `list.Add(1); list.Add(2);` — is not a fluent run and is left alone.

## What is not reported

```csharp
first.Append("a");
second.Append("b");    // not reported — different receivers

list.Add(1);
list.Add(2);           // not reported — Add returns void, not the receiver

Get().Append("a");
Get().Append("b");     // not reported — the receiver is a call, evaluated per statement
```

## How to fix violations

The code fix folds the run into one statement, re-parenting each later call onto the growing chain. The first
statement keeps its leading trivia, the last keeps its trailing trivia, and the statements in between are
removed.

## Enabling the rule

The rule is off by default. Turn it on in `.editorconfig`:

```ini
[*.cs]
dotnet_diagnostic.SST2265.severity = warning
```

## Performance

The rule matches the `receiver.Method(args);` statement shape syntactically before consulting the semantic
model, so a statement that is not such a call never pays for a bind. The receiver's purity is a pure syntax
check.
