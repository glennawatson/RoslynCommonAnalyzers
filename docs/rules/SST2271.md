# SST2271 — Keep the var-versus-explicit choice consistent

| Item | Value |
| --- | --- |
| Category | ModernSyntax |
| Severity | Warning |
| Code fix | Yes (supports Fix All) |
| Default | Disabled (opt-in) |

## Cause

A local declaration or `foreach` variable names its type in the form the codebase did not choose: `var` where
the configured style wants the type named, or an explicit type where the style wants `var`.

```csharp
List<int> numbers = new List<int>();
```

## Rule description

A local declaration and a `foreach` variable can name their type explicitly or use `var`. This rule normalizes
to the one choice the project makes. Whether `var` or the type reads better is a house-style call, so the rule
ships **disabled**.

```csharp
var numbers = new List<int>();
```

This is the code-style space that is otherwise unowned; enabling it lets a codebase enforce a single
convention.

- **Explicit-to-`var`** is offered only when `var` would infer the identical type — the initializer's natural
  type must equal the declared type — so a base- or interface-typed local such as
  `IList<int> x = new List<int>()` is left alone, and a target-typed initializer (`new()`, a collection
  expression, or `default`) that `var` cannot infer is never converted.
- **`var`-to-explicit** names the inferred type, gated on that type having a source-expressible name and on the
  name binding back to it.

A `foreach` has no initializer to judge, so the `when_obvious` mode leaves loop variables alone; `always` and
`never` still normalize them.

## Configuration

```ini
# Rule-specific
stylesharp.SST2271.use_var = when_obvious

# Project-wide
stylesharp.use_var = when_obvious
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `use_var` | `always`, `never`, `when_obvious` | `when_obvious` | Which type spelling declarations are normalized to. |

- `always` — use `var` wherever it infers the same type.
- `never` — always name the type.
- `when_obvious` — use `var` only when the right-hand side names the type (a `new`, a cast, or a literal); name
  the type otherwise.

An unset or unrecognized value falls back to `when_obvious`.

## How to fix violations

The code fix replaces the type node with `var` or the inferred type name, keeping the surrounding trivia. The
inferred type is re-resolved and its name re-bound before the explicit form is offered.

```csharp
// with stylesharp.use_var = always
int value = 1;                       // becomes:  var value = 1;

// with stylesharp.use_var = never
var value = 1;                       // becomes:  int value = 1;

// with stylesharp.use_var = when_obvious
List<int> x = Load();                // left alone — the type is not named on the right
var x = Load();                      // becomes:  List<int> x = Load();
```

## Enabling the rule

The rule is off by default. Turn it on in `.editorconfig`:

```ini
[*.cs]
dotnet_diagnostic.SST2271.severity = warning
```

## Performance

The option is read once per declaration, and the semantic model is consulted only to confirm the rewrite is
type-preserving. A declaration already in the configured style is dropped before any name is built.
