# SST2300 — Implement the disposal pattern correctly

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Partial — the two mechanical clauses (supports Fix All) |
| Default | Enabled |

## Cause

A class implements `IDisposable` but builds only half of the pattern.

```csharp
public class Connection : IDisposable
{
    private readonly Socket _socket;

    ~Connection() => _socket.Close();

    public void Dispose()
    {
        _socket.Close();
    }
}
```

## Rule description

Disposal has two callers, not one: the code that owns the object, and — if the object holds unmanaged
state — the finalizer, running later, on another thread, when the managed objects it references may
already be gone. The pattern exists to give both callers one place to go while telling them apart:

```csharp
public class Connection : IDisposable
{
    private bool _disposed;

    ~Connection() => Dispose(false);

    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed)
        {
            return;
        }

        if (disposing)
        {
            // Managed state. Safe to touch: a real caller is on the stack.
        }

        // Unmanaged state. Safe from the finalizer too.
        _disposed = true;
    }
}
```

`protected` so a derived type can add its own cleanup, `virtual` so it can, and `Dispose()` chains into it
and then takes the instance off the finalizer queue — which it has earned the right to do, because the
cleanup has just run.

### What is reported

The first of these clauses that fails, and only the first: they overlap, and stacking four squiggles on
one type would say the same thing four times. Fix the one you are shown and the next appears.

On the type that first signs the contract, four things are checked:

1. **An unsealed type with no `protected virtual void Dispose(bool disposing)`.** A derived type has
   nowhere to put its cleanup that does not either replace the base's or run at the wrong time.
2. **A public `Dispose(bool)`.** The `bool` is the pattern's private switch between "a caller is disposing
   me" and "the finalizer is". It is not a decision to hand to callers.
3. **A `Dispose()` that never calls `Dispose(true)`.** The type moved its cleanup into the overload and
   then never ran it.
4. **A finalizable type whose `Dispose()` never calls `GC.SuppressFinalize(this)`.** Every instance is
   promoted, queued, and resurrected for a finalizer with nothing left to do.

A type whose base already implements `IDisposable` inherits the pattern, so it is checked for one thing
only:

5. **A derived type that hides the base's `Dispose()` with a new parameterless one.** `public new void
   Dispose()` (or an implicit hide) does not run when a caller holds the base type:

   ```csharp
   using (Connection c = new PooledConnection()) { }  // calls Connection.Dispose(), not PooledConnection.Dispose()
   ```

   A `using` binds `Dispose()` statically on the variable's type, so the derived cleanup silently never
   runs. An `override` of a *virtual* base `Dispose()` is correct and is not reported.

### What is not reported

- **A sealed class with no finalizer.** This is the important exclusion. Nothing can derive from it, and
  nothing but the owner will ever call its cleanup, so a plain `Dispose()` is not an incomplete pattern —
  it is the *whole* pattern, correctly. Demanding `protected virtual void Dispose(bool)` from a sealed
  type is asking for a virtual method nothing can override and a parameter that is always `true`.
- **A derived type that overrides `Dispose(bool)`.** The pattern belongs to the type that first signed the
  contract; the derived type's job is to override `Dispose(bool)`, and *that* is what the rule wants it to
  be able to do. Re-listing `IDisposable` in the derived base list changes nothing. Only a derived type that
  *hides* the base's parameterless `Dispose()` (clause 5 above) is reported.
- **A record.** The compiler generates members for it, and disposal is not one of them; the rule leaves
  records alone rather than guess at how a generated equality contract and a hand-written finalizer should
  interact.
- **A struct.** It cannot be derived from and cannot have a finalizer, so a plain `Dispose()` is complete
  for the same reason a sealed class's is.
- **A type that implements only `IAsyncDisposable`.** Asynchronous disposal is a different pattern with a
  different shape (`DisposeAsync()` / `DisposeAsyncCore()`), and it is out of this rule's scope. A type
  that implements both contracts is measured on the synchronous one.
- **A `Dispose()` with no body** — an abstract declaration, or one that lives in metadata. It states
  nothing to disagree with.

The chained call is matched on syntax, so `Dispose(true)` and `this.Dispose(true)` both count, and both
still count from inside a guard.

## How to fix violations

The code fix repairs the two mechanical clauses:

- **Add `GC.SuppressFinalize(this)`** to a `Dispose()` on a finalizable type. An expression-bodied
  `Dispose()` becomes a block, because it now has two things to say. The call is emitted through the
  global namespace and simplified back down only where the short form binds to the same method, so it
  compiles in a file with no `using System;` and in a file that already has a `GC` of its own.
- **Change a public `Dispose(bool)`** to `protected virtual` — or to `private` on a sealed type, where
  nothing can override it and `protected` is a contradiction.

Both support Fix All.

The remaining clauses have **no code fix**, deliberately. Synthesizing a `Dispose(bool)` means deciding
which of the type's fields are managed, which are not, and which the finalizer path is allowed to touch —
touching a managed field from a finalizer is exactly the bug the pattern's `bool` exists to prevent. That
is the design work the diagnostic is asking for; a fix that guessed at it would write a body that compiles
and disposes nothing. The hidden-`Dispose()` clause has no fix either: the right repair depends on intent —
override a virtual base `Dispose()`, override `Dispose(bool)` instead, or drop the redundant member — and
only the author knows which.

## Configuration

None. The pattern is not a matter of taste.

## Performance

The clean path is one walk of the type's interface list, which is empty for most types and never contains
`IDisposable` for almost all of the rest. Only a type that signs the contract — and whose base does not
already own the pattern — reaches the member scan, and only one that declares a `Dispose()` with a body
reaches the syntax walk, which is an allocation-free preorder scan that stops as soon as it has found both
of the calls it is looking for. The `GC.SuppressFinalize` lookup that proves the suggestion is available
sits behind every other test, so it runs only for a finalizable disposable type that is about to be
reported.
