# SST2301 — Types implementing IEquatable&lt;T&gt; should be sealed

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Yes — seals the type (supports Fix All) |
| Default | Enabled |

## Cause

A class implements `IEquatable<T>` where `T` is the class itself, and can still be derived from.

```csharp
public class Money : IEquatable<Money>
{
    public decimal Amount { get; init; }

    public bool Equals(Money other) => other is not null && other.Amount == Amount;
}
```

## Rule description

`IEquatable<Money>` is a promise: *I decide equality against a `Money`.* The promise is kept right up
until someone writes this:

```csharp
public class Coin : Money
{
    public string Mint { get; init; }
}
```

Now `Coin` inherits `Equals(Money)`, which compares an amount and nothing else. Two coins from different
mints with the same amount are equal. Worse, equality is no longer symmetric the moment `Coin` overrides
it to include the mint: `money.Equals(coin)` answers true through the base's comparison while
`coin.Equals(money)` answers false through the derived one. Every `Dictionary`, `HashSet` and
`Contains` built on that type now depends on which operand happened to be on the left — and the bug is
not a crash, it is a wrong answer, in a lookup, sometimes.

Seal the type. That is what the contract already said.

### What is reported

A non-abstract, non-sealed **class** that implements `IEquatable<TSelf>` — including a generic one,
`Box<T> : IEquatable<Box<T>>`.

### What is not reported

- **Abstract classes.** An abstract type cannot be instantiated, so it is never the thing being compared;
  a base class that implements `IEquatable<TSelf>` is usually handing its concrete descendants a shared
  field comparison rather than claiming to be the final word on equality. It is the *leaves* that have to
  keep the contract, and each leaf is reported on its own — which is the diagnostic you can act on.
- **Structs.** Nothing can derive from them, so the asymmetry cannot arise.
- **Records.** The compiler's generated equality already carries the type check (`EqualityContract`) that
  keeps a hierarchy honest: a `Coin` and a `Money` with the same amount are never equal, in either
  direction.
- **A class implementing `IEquatable<TOther>` for some other type.** It makes no claim about itself.
- **A derived class that merely inherits the interface.** `Coin : Money` implements `IEquatable<Money>`,
  not `IEquatable<Coin>`; the report belongs on `Money`, and that is where it goes.

## How to fix violations

The code fix adds `sealed` (in front of `partial`, which stays last). It supports Fix All.

If the class is **already derived from**, sealing it will not compile: the hierarchy and the contract
were never compatible. The choices then are to move the shared
state into a base that does not implement `IEquatable<T>` and let each leaf implement it for itself, or —
if the hierarchy really does define one notion of equality — to drop `IEquatable<T>`, override
`Equals(object)` and `GetHashCode()`, and make the type check explicit:

```csharp
public override bool Equals(object obj)
    => obj is Money other && other.GetType() == GetType() && other.Amount == Amount;
```

## Configuration

None.

## Performance

The clean path is a walk of the type's interface list — empty for most types — and a name comparison. The
contract is matched structurally, on the interface's name, arity and namespace, so no well-known type is
ever resolved: a compilation whose types never mention equality pays for nothing at all.
