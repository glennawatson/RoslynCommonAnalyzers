# SST2302 — Overload operators in their complete set

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A type overloads an operator without the rest of the set that operator belongs to.

```csharp
public class Money
{
    public decimal Amount { get; init; }

    public static bool operator ==(Money left, Money right) => left?.Amount == right?.Amount;

    public static bool operator !=(Money left, Money right) => left?.Amount != right?.Amount;
}
```

`money == other` says one thing. `money.Equals(other)` — which is what `List.Contains`, `Dictionary`, and
every generic comparison in the framework actually calls — says another. Both compile. Only one of them is
used by the code you did not write.

## Rule description

C# enforces part of the set itself. It will not compile a type that declares `==` without `!=`, `<`
without `>`, or `<=` without `>=`. **A rule that repeats those would only ever fire on code that does not
build, so this one does not.** What it reports is the part of the set the language leaves open:

| Reported | Why the compiler does not catch it |
| --- | --- |
| `==` without an `Equals(object)` override | The language pairs `==` with `!=` and stops there. |
| `==` without a `GetHashCode()` override | Same. A type whose hash does not follow its equality silently loses items in every hash-based collection. |
| `<`/`>` without `<=`/`>=` (or the reverse) | The compiler pairs each operator with its *mirror*. It never requires the other pair, so a caller can be able to write `a < b` and not `a <= b`. |
| `<` or `<=` on a type that does not implement `IComparable<T>` | The operators are not what `Sort`, `Max`, `OrderBy` or `SortedSet` use. A type that can be compared with `<` but not sorted is a type that answers half the question. |
| A public class with `+ - * / %` but no `==`, `Equals(object)` or `GetHashCode()` | An arithmetic type is a value. Without value equality, `a + b == c` uses *reference* equality — always false for two freshly computed results — and the type cannot be a dictionary key or unit-tested for the value it computed. The compiler is happy; the type is not usable. |

### Where it is reported

Once per gap, on one site each — the mirror operators (`!=`, `>`, `>=`) are never the reporting site, so a
complete pair never says the same thing twice:

- the equality gap, on the `==` declaration (naming both missing overrides in one message when both are
  missing);
- the ordering gaps, on the `<` declaration — or on `<=` when the type has no `<` — with the missing pair
  and the missing `IComparable<T>` named together;
- the arithmetic gap, on the first arithmetic operator the type declares in the order `+ - * / %`, so a
  type with several is reported once rather than per operator.

The arithmetic gap is checked only on a **public class**: a `struct` already has value equality, a `record`
declares its own, and a non-public type is not a surface callers depend on. A unary `+` or `-` does not make
a type an arithmetic type for this purpose. A class that declares *any* one of `==`, `Equals(object)` or
`GetHashCode()` is left to the equality rows above.

An override **inherited from a base class does not count**. A base that overrides `Equals(object)` decides
equality over the base's state; a derived type that adds state and an `==` of its own has to say how that
state participates, and inheriting the answer is exactly the bug.

The `IComparable<T>` gap is reported only when `IComparable<T>` actually exists in the compilation being
analyzed — a rule that suggests an API the target framework does not have would be handing you a
diagnostic you cannot fix. The non-generic `IComparable` is accepted as an implementation of the contract.

## How to fix violations

There is **no code fix**. A generated body could not know what to contain:

- **`Equals(object)` and `GetHashCode()`** have to name the fields that define identity. That is the same
  decision the `==` body already made, but an analyzer cannot read it back out of an arbitrary expression
  — and a stub that returned `base.GetHashCode()` or `ReferenceEquals` would compile, look done, and be
  wrong in exactly the way the diagnostic was warning about. A wrong `GetHashCode` is worse than a missing
  one: the missing one is a warning, the wrong one is a lost dictionary entry.
- **`<=` from `<`** is *not* `!(a > b)`. On any type with a floating-point component that identity fails
  for NaN, where all four comparisons are false and negating one of them is not the same as evaluating the
  other. A "mechanical" rewrite here would be a correctness bug the compiler cannot see.
- **`IComparable<T>`** needs a `CompareTo` — a total order over the type's state, including what happens
  with `null`. That is design, not transcription.

Write the set out:

```csharp
public sealed class Money : IComparable<Money>
{
    public decimal Amount { get; init; }

    public int CompareTo(Money other) => other is null ? 1 : Amount.CompareTo(other.Amount);

    public override bool Equals(object obj) => obj is Money other && other.Amount == Amount;

    public override int GetHashCode() => Amount.GetHashCode();

    public static bool operator ==(Money left, Money right) => Equals(left, right);

    public static bool operator !=(Money left, Money right) => !Equals(left, right);

    public static bool operator <(Money left, Money right) => left is null ? right is not null : left.CompareTo(right) < 0;

    public static bool operator >(Money left, Money right) => left is not null && left.CompareTo(right) > 0;

    public static bool operator <=(Money left, Money right) => left is null || left.CompareTo(right) <= 0;

    public static bool operator >=(Money left, Money right) => left is null ? right is null : left.CompareTo(right) >= 0;
}
```

## Configuration

None.

## Performance

The rule is driven from the operator declaration, not from the type: a compilation that overloads no
operators — which is nearly all of them — never runs a single check. A type that does overload one pays
for a handful of name-keyed member lookups and, at most, one walk of its interface list. The
`IComparable<T>` lookup that proves the suggestion is available runs only on that same rare path, and only
for a type that does not already implement it.
