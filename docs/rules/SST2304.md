# SST2304 — Events should use the standard handler signature

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An event's delegate type is not `EventHandler` or `EventHandler<TEventArgs>` — either its shape is not
`void (object sender, TEventArgs e)`, or it is a hand-written delegate that duplicates that shape.

```csharp
public sealed class MovedEventArgs : EventArgs
{
}

public delegate void ValueChanged(int oldValue, int newValue);

public delegate void MovedHandler(object sender, MovedEventArgs e);

public class Slider
{
    public event ValueChanged Changed;   // wrong shape

    public event MovedHandler Moved;     // right shape, bespoke delegate
}
```

## Rule description

The `(object sender, TEventArgs e)` shape is a protocol, not a style. Everything written to work with
events in general is written against it: the code that forwards one event to another, the weak-event
subscription that has to unhook itself, the binding that wires a handler up at runtime, the test helper
that records the last event raised, the designer that lists what a control can tell you. All of it is
generic over `TEventArgs` and takes the sender as `object`.

A delegate of any other shape works — until one of those things has to handle it, and then it cannot be
handled at all. The two parameters of `ValueChanged` above cannot be passed to anything that does not
already know what a `ValueChanged` is, and the subscriber cannot find out who raised it.

A bespoke delegate that already has the right shape — `MovedHandler` above — has a different problem: it
is `EventHandler<MovedEventArgs>` under another name. Two events carrying the same payload through two
delegate types cannot share a handler variable, a stored subscription, or a helper that re-raises them,
because delegate types do not convert to one another even when their signatures are identical. The
bespoke type is extra public API surface that buys nothing the framework delegate does not already
provide, so the message names the exact replacement.

The other half of the argument is versioning. Adding a value to an event that carries an `EventArgs` is a
new property on the payload; adding one to a custom delegate is a new parameter, which breaks every
subscriber that has ever compiled against it.

```csharp
public sealed class ValueChangedEventArgs : EventArgs
{
    public int OldValue { get; init; }

    public int NewValue { get; init; }
}

public class Slider
{
    public event EventHandler<ValueChangedEventArgs> Changed;

    public event EventHandler<MovedEventArgs> Moved;
}
```

### What is reported

Any event whose delegate type is a custom delegate. The message names the tightest replacement the
delegate's shape allows:

- a **custom delegate that already matches the shape** is told the exact construction to move to —
  `EventHandler<MovedEventArgs>` for `delegate void MovedHandler(object sender, MovedEventArgs e)`,
  plain `EventHandler` when the payload is `EventArgs` itself, and `EventHandler<T>` over the same type
  parameter for a generic delegate constrained to `EventArgs`;
- a **custom delegate with the wrong parameters** — anything that is not exactly two, the first typed
  `object`, both by value — is told to move to `EventHandler<TEventArgs>`, where the arguments type is
  still to be designed;
- the same for a **second parameter that is not an `EventArgs`**, so there is nowhere to put the next
  thing the event needs to carry, and for a handler that **returns a value**. An event has no one to
  return it to: with several subscribers, only the last one's answer survives, and which one is last is
  not something the raiser controls.

### What is not reported

- **`EventHandler` and `EventHandler<T>`.** They are the goal, by definition, and are never examined.
- **An event that overrides one, or implements an interface member.** Its type is dictated by the
  declaration it follows, so the fix belongs there — and *that* declaration is the one reported, whether it
  is the abstract event on the base class or the event on the interface.
- **Generated code.**
- **Anything, if the compilation has no `EventHandler<T>`.** The rule suggests an API, so it proves the API
  is there first rather than handing you a diagnostic your target framework cannot satisfy.

## How to fix violations

When the delegate already matches the shape, the change is a swap: replace the event's type with the
handler construction the message names, and delete the delegate. Raise sites keep compiling as they are,
and so does every subscriber written as a method group or lambda. The one thing to sweep for is a
subscriber that names the delegate type — `new MovedHandler(OnMoved)`, or a field or local declared as
`MovedHandler` — which must lose the wrapper or change its declared type, because identical signatures do
not make two delegate types interchangeable.

When the shape is wrong, the change is a redesign, worth making by hand:

1. Add an `EventArgs`-derived type carrying what the event needs to say, with room for what it will need to
   say later.
2. Change the event's type to `EventHandler<T>`.
3. Raise it with `Changed?.Invoke(this, new ValueChangedEventArgs { ... })`.
4. Delete the delegate.

If the event genuinely is not an event — if the subscriber's return value matters, or there can only ever
be one of them — then it is a callback, and it should be a property of a delegate type or an interface,
not an `event` at all.

There is **no code fix**, deliberately — for either flavour. The wrong-shape rewrite means inventing a
type that does not exist yet, deciding which of the delegate's parameters become properties on it and what
they are called, rewriting every raise site to construct that type, and rewriting every subscriber. The
matching-shape swap looks mechanical, but it is only mechanical inside the declaring file: any subscriber
anywhere in the solution that names the bespoke delegate type stops compiling the moment the event's type
changes, and a fix that edits one document cannot see those. Each flavour therefore ends in either a
design decision or a solution-wide sweep, and a fix that got either wrong would produce a compilation
error in a file the user never opened.

## Configuration

None.

## Performance

The rule runs on event symbols, so a compilation that declares no events never does any work. The common
case — an `EventHandler<T>` — is settled by two string comparisons before anything else is looked at, and
the interface walk that exempts an implementing event runs only for an event, which is rare. The
`EventHandler<T>` lookup that proves the suggestion is available — and the shape walk that picks between
the exact replacement and the placeholder — sit behind every other test, so only an event that reports
pays for them.
