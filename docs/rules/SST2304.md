# SST2304 — Events should use the standard handler signature

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An event's delegate does not have the shape `void (object sender, TEventArgs e)`.

```csharp
public delegate void ValueChanged(int oldValue, int newValue);

public class Slider
{
    public event ValueChanged Changed;
}
```

## Rule description

The `(object sender, TEventArgs e)` shape is a protocol, not a style. Everything written to work with
events in general is written against it: the code that forwards one event to another, the weak-event
subscription that has to unhook itself, the binding that wires a handler up at runtime, the test helper
that records the last event raised, the designer that lists what a control can tell you. All of it is
generic over `TEventArgs` and takes the sender as `object`.

A delegate of any other shape works — until one of those things has to handle it, and then it cannot be
handled at all. The two parameters of `ValueChanged` above cannot be passed to anything that does not
already know what a `ValueChanged` is, and the subscriber cannot find out who raised it.

The other half of the argument is versioning. Adding a value to an event that carries an `EventArgs` is a
new property on the payload; adding one to a custom delegate is a new parameter, which breaks every
subscriber that has ever compiled against it.

```csharp
public sealed class ValueChangedEventArgs : EventArgs
{
    public int OldValue { get; init; }

    public int NewValue { get; init; }
}

public class Slider
{
    public event EventHandler<ValueChangedEventArgs> Changed;
}
```

### What is reported

An event whose delegate type does not match the shape:

- a **custom delegate with the wrong parameters** — anything that is not exactly two, the first typed
  `object`;
- a **second parameter that is not an `EventArgs`**, so there is nowhere to put the next thing the event
  needs to carry;
- a handler that **returns a value**. An event has no one to return it to: with several subscribers, only
  the last one's answer survives, and which one is last is not something the raiser controls.

### What is not reported

- **`EventHandler` and `EventHandler<T>`.** They are the shape, by definition, and are never examined.
- **A hand-written delegate that already has the right shape.** `delegate void ValueChangedHandler(object
  sender, ValueChangedEventArgs e)` is exactly what the rule is asking for; the rule is about the shape a
  consumer can handle, not about the name of the delegate. A generic delegate is judged by what its
  constraint promises, so `delegate void Handler<T>(object sender, T e) where T : EventArgs` is fine too.
- **An event that overrides one, or implements an interface member.** Its type is dictated by the
  declaration it follows, so the fix belongs there — and *that* declaration is the one reported, whether it
  is the abstract event on the base class or the event on the interface.
- **Generated code.**
- **Anything, if the compilation has no `EventHandler<T>`.** The rule suggests an API, so it proves the API
  is there first rather than handing you a diagnostic your target framework cannot satisfy.

## How to fix violations

There is **no code fix**, deliberately. Rewriting `event ValueChanged Changed` to
`event EventHandler<ValueChangedEventArgs> Changed` means inventing a type that does not exist yet
(`ValueChangedEventArgs`), deciding which of the delegate's parameters become properties on it and what
they are called, rewriting every `Changed?.Invoke(oldValue, newValue)` raise site to construct that type,
and rewriting every subscriber — including the ones in other projects and the ones written as lambdas
against the old parameter list. Each of those steps is a design decision or a cross-assembly source
change, and a fix that got any of them wrong would produce a compilation error in a file the user never
opened.

The change is worth making by hand:

1. Add an `EventArgs`-derived type carrying what the event needs to say, with room for what it will need to
   say later.
2. Change the event's type to `EventHandler<T>`.
3. Raise it with `Changed?.Invoke(this, new ValueChangedEventArgs { ... })`.
4. Delete the delegate.

If the event genuinely is not an event — if the subscriber's return value matters, or there can only ever
be one of them — then it is a callback, and it should be a property of a delegate type or an interface,
not an `event` at all.

## Configuration

None.

## Performance

The rule runs on event symbols, so a compilation that declares no events never does any work. The common
case — an `EventHandler<T>` — is settled by two string comparisons before anything else is looked at, and
the interface walk that exempts an implementing event runs only for an event, which is rare. The
`EventHandler<T>` lookup that proves the suggestion is available sits behind every other test, so only an
event that would otherwise be reported pays for it.
