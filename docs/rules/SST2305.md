# SST2305 — Collection properties should not be settable

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Yes — remove the setter |
| Default | Enabled |

## Cause

A property whose type is a mutable collection declares a caller-visible `set` accessor.

```csharp
public sealed class Basket
{
    public List<Item> Items { get; set; } = new List<Item>();
}
```

## Rule description

A setter on a collection property does not let a caller change the collection — a get-only property
already does that, because the collection itself is mutable. What the setter adds is the power to
replace it:

```csharp
basket.Items.Add(item);      // works with a get-only property
basket.Items = newList;      // this is what the setter adds
```

That second line swaps the instance out from under everything still holding a reference to the old
one: the type's own fields, the loop that captured it a line earlier, the `CollectionChanged`
subscription attached to it, the code in another thread halfway through iterating it. Nobody asked
for that, and it is nearly always a bug waiting for a maintenance change to trigger it.

```csharp
public sealed class Basket
{
    public List<Item> Items { get; } = new List<Item>();
}
```

If the collection genuinely has to be replaceable, say so with a method that names the operation
(`ReplaceItems(...)`), so the cost of the swap is visible at the call site.

### What is a mutable collection

A type is mutable here if the reference a caller holds can be used to add to it or clear it — that
is, if it is an array, or if it carries `ICollection<T>`, `IList<T>`, `IDictionary<K, V>`, or
`ISet<T>` (or their non-generic forms). That covers `List<T>`, `T[]`, `Dictionary<K, V>`,
`HashSet<T>`, `Collection<T>`, `ObservableCollection<T>`, and the interfaces themselves.

The types whose mutating interfaces throw are not: `ImmutableArray<T>` and everything else in
`System.Collections.Immutable`, the frozen collections, and the `ReadOnly*` wrappers in
`System.Collections.ObjectModel`. Replacing one of those is a value assignment, and there is nothing
being torn away from anyone. `IEnumerable<T>` and `IReadOnlyList<T>` are not mutable either, so a
settable property of those types is not reported.

## What is not reported

- **An `init` accessor.** The object is built once and then settled — that is the design this rule
  is asking for, expressed in the language.
- **A `private set`.** The type keeps control of when the collection is replaced, and can be
  trusted to know what it is doing.
- **A `required` property.** The setter is what an object initializer satisfies; removing it would
  not compile.
- **Anything carrying an attribute** — on the property, or on the type that declares it. A
  serialization contract, a data-transfer object, a designer-generated model: these need their
  setters, and the attribute is how they say so. The rule cannot know every attribute that means
  it, so it steps back from all of them. This is deliberate under-reporting: it costs a missed
  report on a decorated type and saves a stream of false ones.
- **A positional record's members.** They are the constructor's parameters, not settable
  properties, and never appear as property declarations at all.
- **An `override`, an explicit interface implementation, and an interface implementation.** The
  shape comes from the base type or the interface, and *that* declaration is what the rule reports
  instead — which is also the only place the setter can actually be removed.

## How to fix violations

The fix removes the setter. Where the property was assigned from outside its own type, that
assignment stops compiling — which is the point: each one is a place the collection was being torn
out from under its holders, and each one becomes an `Add`, a `Clear`, or a named method on the type.

## Related rules

- **[PSH1017](PSH1017.md)** looks at the *getter* — a property that allocates a fresh copy of a
  collection on every read. SST2305 looks only at the *setter*, and never inspects what the getter
  returns. A property can trip both, and each has its own separate cause and fix.

## Performance

The clean path is syntax only: a property with no `set` accessor, an `init` accessor (a different
syntax kind entirely), a `private set`, `required`, `override`, an explicit interface implementation,
or any attribute is rejected before the semantic model is touched. Only a property that really does
hand a caller a setter pays for a symbol bind, and only one that also turns out to hold a mutable
collection pays for the interface-implementation walk.
