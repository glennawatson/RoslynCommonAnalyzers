# SST2306 — Return an empty collection instead of null

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Yes — return the empty collection |
| Default | Enabled |

## Cause

A member whose declared return type is a collection hands back `null`.

```csharp
public IReadOnlyList<Order> FindOrders(string customer)
{
    if (!_byCustomer.TryGetValue(customer, out var orders))
    {
        return null;
    }

    return orders;
}
```

## Rule description

A collection type already has a value that means "there is nothing": the empty collection. Returning
`null` invents a second one, and it is the one that every caller has to remember to check for:

```csharp
foreach (var order in service.FindOrders(name))   // NullReferenceException on the day there are none
{
}
```

An empty collection reads the same way at every call site — you can iterate it, count it, and pass
it on — and it costs no allocation at all: `Array.Empty<T>()` and the empty collection expression
both hand back a shared instance.

```csharp
public IReadOnlyList<Order> FindOrders(string customer)
    => _byCustomer.TryGetValue(customer, out var orders) ? orders : Array.Empty<Order>();
```

## What is reported

A `null` in return position of a member whose return type is a collection:

- a `return null;` statement,
- an expression body (`=> null`), on a method, a property, or an indexer,
- either branch of a returned conditional (`return found ? items : null;`),
- a getter — a property or indexer `get` is a return like any other,
- a local function.

A collection here is an array, or any non-`string`, non-ref-like type that carries `IEnumerable`.

## What is not reported

- **A nullable return type.** `IEnumerable<T>?` is the author saying, in the signature, that null is
  a value in this API. The rule takes them at their word — this is the supported way to opt out of
  it for one member.
- **`string`.** A string satisfies `IEnumerable<char>`, but nobody returns one meaning "a sequence
  of characters", and rewriting `return null` to an empty string across every string-returning method
  in a codebase would be both wrong and intolerable. This exclusion is not an edge case; it is
  essential to the rule being usable at all.
- **`Task<T>` and `ValueTask<T>`.** A task is not a collection. A `null` handed back where a task is
  expected is a different bug with a different answer, and belongs to the rules that own
  asynchronous returns.
- **A lambda or anonymous method.** Its shape comes from the delegate it is assigned to.
- **An interface or abstract member**, which has no body to return from.
- **A generic `T`**, where no empty value can be constructed.

## How to fix violations

The fix replaces the `null` with the empty collection for that return type. What it writes is
decided per compilation, never assumed from a target framework:

| Return type | Replacement |
| --- | --- |
| `T[]`, on C# 12 or later | `[]` |
| `T[]`, below C# 12 | `Array.Empty<T>()`, or `new T[0]` where the compilation has no `Array.Empty` |
| `IEnumerable<T>`, `ICollection<T>`, `IList<T>`, `IReadOnlyCollection<T>`, `IReadOnlyList<T>` | `Array.Empty<T>()` — an empty array satisfies all of them, on every framework |
| `ISet<T>`, `IReadOnlySet<T>` | `new HashSet<T>()` |
| `IDictionary<K, V>`, `IReadOnlyDictionary<K, V>` | `new Dictionary<K, V>()` |
| A concrete type with a public parameterless constructor (`List<T>`, `Dictionary<K, V>`, …) | `new List<T>()` |
| Anything else | *No fix* — the diagnostic is reported and left for you |

Every name in the replacement is written the way the file's own imports require: without
`using System;`, the fix emits `System.Array.Empty<T>()` rather than a name that would not resolve.
And before the fix is offered at all, the expression it intends to write is speculatively bound at
the exact position it would be written — so a suggestion that names an API this compilation does not
have, or a name this file cannot reach, is dropped instead of producing code that will not build.

The empty collection expression `[]` is written only on C# 12 and later, and only for an array
target, where its meaning is exact.

One case is reported without a fix on purpose: a `null` in a **conditional branch** below C# 9. The
replacement only compiles there because the language target-types the branches from the return type,
and before C# 9 it did not — so the diagnostic stands and the rewrite is left to you.

## Performance

The clean path is syntax only: an expression with no `null` in it, a lambda, an async member, a
nullable return type. None of them touch the semantic model. Only a member that really does return
`null` is bound; only one whose return type really is a collection has a replacement computed for it;
and no well-known type is resolved until a violation has already been found.
