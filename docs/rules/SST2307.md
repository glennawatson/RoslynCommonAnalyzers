# SST2307 — Generic method type parameters should be inferable from the parameters

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An externally visible generic method declares a type parameter that appears in none of its parameters,
so the compiler cannot infer it and every call site has to name it.

```csharp
public static class Registry
{
    public static void Register<TService>() { }               // TService is inferable from nothing

    public static TResult Create<TResult>() => default!;      // the return type does not infer

    public static void CopyInto<TSource, TItem>(TSource source)
        where TSource : IEnumerable<TItem> { }                // constraints do not infer either
}
```

## Rule description

C# infers a method's type arguments from the arguments the call supplies, and from nothing else. It does
not look at the return type, and it does not walk the constraints. A type parameter that appears in no
parameter is therefore never inferable, and the burden lands on every caller:

```csharp
Registry.Register<IClock>();
var order = Registry.Create<Order>();
Registry.CopyInto<List<int>, int>(numbers);   // both arguments, every time
```

The `CopyInto` call is the one that shows the cost most clearly. `TSource` could have been inferred on
its own; because `TItem` cannot be, the caller loses inference for *both* — C# infers all of a method's
type arguments or none of them. One un-inferable type parameter makes the whole call site explicit.

The fix is to take the type as a parameter, so that a value pins it down:

```csharp
public static void Register<TService>(TService service) { }

public static void CopyInto<TItem>(IEnumerable<TItem> source) { }
```

Or to drop the type parameter and accept the type itself, when the method really is about the type
rather than a value of it:

```csharp
public static void Register(Type serviceType) { }
```

A type parameter counts as inferable wherever it appears inside a parameter's type, however deeply
nested — `T[]`, `List<T>`, `Func<int, T>`, `(T, string)`, `ref T`, and an extension method's receiver all
pin it down.

## When a factory is the point

`T Create<T>()` and `T Deserialize<T>(string json)` are reported: the caller does have to write
`Create<Order>()`. Sometimes that is the design — the type argument *is* the input, and there is no value
to infer it from. Turn the rule off rather than reshape a signature that is right as it stands:

```ini
dotnet_diagnostic.SST2307.severity = none
```

## How to fix violations

Give the type parameter a home in the parameter list, drop it in favour of a `Type` argument, or — where
the explicit type argument is genuinely the API — turn the rule off.

## When not to use the rule

Turn it off for a codebase built around type-argument-driven factories, resolvers, or serializers, where
naming the type at the call site is the intended way to call in.

## Not reported

- A method whose signature is not its author's to change: an `override`, an explicit interface
  implementation, or a method implementing an interface member. The fix belongs at the declaration that
  sets the shape, and the interface or base declaration is reported there instead.
- Anything not externally visible. An `internal` or `private` method's callers all live in the assembly
  that can change it freely.
- A type parameter on the *type* rather than the method. `class Cache<T>` names `T` once, at construction.
