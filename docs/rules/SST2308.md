# SST2308 — Obsolete attributes should explain what to use instead

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An `[Obsolete]` attribute carries no message, or a message that is empty or only whitespace.

```csharp
[Obsolete]
public void Send(string body) { }

[Obsolete("")]
public void Publish(string body) { }

[Obsolete(message: null, error: true)]
public void Broadcast(string body) { }
```

## Rule description

The message is the whole value of the attribute. Without it, the compiler tells a caller that their
code is wrong and then stops talking:

```text
warning CS0618: 'Mailer.Send(string)' is obsolete
```

The caller now goes looking: through the source if they have it, through the release notes if they
can find them, through a search engine if they cannot. Every one of them repeats that search. The
author, at the moment of writing the attribute, already knows the answer — and the message is the one
place they can hand it over, at exactly the point the reader needs it.

```csharp
[Obsolete("Use SendAsync(Message) instead; this overload drops the message headers.")]
public void Send(string body) { }
```

A good message says what replaced it and, when it is not obvious, why. It is read far more often
than it is written.

## What is reported

- `[Obsolete]` — no arguments at all.
- `[Obsolete(DiagnosticId = "…")]`, `[Obsolete(UrlFormat = "…")]` — a property initializer is not a
  message.
- `[Obsolete("")]`, `[Obsolete("   ")]`, `[Obsolete(null)]` — a message that says nothing, including
  one supplied by a constant that turns out to be blank.
- `[Obsolete(message: null, error: true)]` — the error flag is not an explanation. (There is no
  constructor taking the flag on its own, so this is the only way to write one without a message.)

The attribute is reported wherever it can be written: types, methods, constructors, properties,
fields, events, delegates, enum members, indexers, operators, parameters, and local functions.

## What is not reported

- **Any message at all**, however it is supplied: positionally, as `message:`, alongside the error
  flag, or through a constant. The rule asks for a message, not for a good one — it cannot judge
  prose, and does not try.
- **A non-constant message expression**, which the rule cannot evaluate and does not guess about.
- **An attribute of the same name that is not the framework's.** The name is matched first because
  it is free, but a declaration is only reported once the attribute has been bound to
  `System.ObsoleteAttribute`.
- **Generated code.**

## How to fix violations

There is **no code fix**, deliberately. The message is the one thing an analyzer cannot write: only
the author knows what replaced the member, why it went, and what a caller has to do about it. A
generated placeholder ("This member is obsolete.") would satisfy the rule and help nobody — it would
be worse than the missing message, because it would look like an answer.

Write the migration:

```csharp
[Obsolete("Use SendAsync(Message) instead. Removed in 5.0.")]
```

## Performance

The clean path is the attribute's simple name, read straight off the syntax — which rejects every
other attribute in the file for free. A message that is a string literal is read off the token, so
a correctly-written `[Obsolete("…")]` never touches the semantic model either. Only a non-literal
argument costs a constant evaluation, and only a declaration that is about to be reported is bound at
all.
