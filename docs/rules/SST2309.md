# SST2309 — Use an overload instead of an optional parameter

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An externally visible method or constructor declares a parameter with a default value.

```csharp
public sealed class Client
{
    public Client(Uri endpoint, int retries = 3) { }

    public Task<Response> Send(Request request, TimeSpan timeout = default) => ...;
}
```

## Rule description

The default is not stored in the method. The compiler copies it into every call site that omits the
argument, at the moment *that caller* is compiled:

```csharp
client.Send(request);          // the caller's IL actually reads: client.Send(request, default)
```

Ship version 2 with `retries = 5` and every assembly already built against version 1 keeps passing `3`.
It does not see the new default, it will not see it after an upgrade, and nothing warns anybody — the old
value is baked into their binary. Only a recompile picks it up, which is exactly the thing a library
author cannot make their consumers do.

An overload keeps the default inside the method, in one place, where changing it reaches every caller the
moment they load the new assembly:

```csharp
public Client(Uri endpoint)
    : this(endpoint, DefaultRetries)
{
}

public Client(Uri endpoint, int retries) { }
```

Optional parameters also introduce overload-resolution ambiguity, and they are not usable from every
.NET language.

## How to fix violations

Replace the optional parameter with an overload that supplies the value, and put the default in a named
constant inside the type.

## When not to use the rule

Turn it off for an application, or any assembly that ships and rebuilds together with all of its callers.
The hazard is a *stale* default in a separately compiled caller; when everything recompiles at once,
there is no staleness to protect against, and optional parameters are much less typing.

```ini
dotnet_diagnostic.SST2309.severity = none
```

## Not reported

- **Caller-info parameters.** `[CallerMemberName]`, `[CallerFilePath]`, `[CallerLineNumber]` and
  `[CallerArgumentExpression]` only work on an optional parameter — the default is the slot the compiler
  writes the caller's details into. Demanding an overload there would demand code that does not compile.

  ```csharp
  public void Log(string message, [CallerMemberName] string caller = "") { }   // fine
  ```

- **A signature its author cannot change:** an `override`, an explicit interface implementation, or a
  method implementing an interface member. The interface and abstract declarations that set the shape are
  reported instead.
- **A positional record's primary constructor.** Its parameter list is the record's definition, not a
  signature you could write an overload for.
- **`params` arrays.** There is no default value to bake in; omitting the arguments builds an empty array
  at the call site.
- **Anything not externally visible.** Inside an assembly every caller recompiles together, so an
  `internal` or `private` default can never go stale.
