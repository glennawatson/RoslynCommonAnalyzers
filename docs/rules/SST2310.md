# SST2310 — Deprecated code should be removed

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Any `[Obsolete]` attribute, on any declaration — **including one that carries a message**.

```csharp
[Obsolete]                                   // reported
public void Send(string body) { }

[Obsolete("Use SendAsync(Message).")]        // reported
public void Publish(string body) { }

[Obsolete("Use SendAsync(Message).", DiagnosticId = "ACME001")]   // reported
public void Broadcast(string body) { }
```

## Rule description

**This is a nag rule, on purpose.** It does not ask you to write a better `[Obsolete]`; it asks you to
delete the code the attribute is on, and it will keep asking until you do.

Deprecating a member is the first half of removing it. The second half is the one that pays. Until the
member is gone it is still compiled, still tested, still maintained, still binary surface you cannot
change, and still the first thing a newcomer finds when they open the type. A deprecation that nobody
ever finishes is just a comment with a compiler warning attached.

So this rule exists to keep the list of unfinished removals in front of you. It is the right rule for a
codebase that is **actively retiring API** — an application, an internal library, a service — where
"obsolete" is a step on the way out and the warning count is supposed to go down.

## When this rule is wrong

**It is the wrong rule for a library that must keep its obsolete members for compatibility.** If you
ship a public package with a support policy, your `[Obsolete]` members are load-bearing: they are how
you keep faith with callers on the old API while pointing them at the new one. You cannot delete them,
so a rule telling you to is pure noise on every build, forever.

In that case, turn it off:

```ini
[*.cs]
dotnet_diagnostic.SST2310.severity = none
```

and let the two rules that police the *contents* of the attribute do the work instead — they ask for
things you can actually give:

- **[SST2308](SST2308.md)** — the attribute must carry a message.
- **[SST2314](SST2314.md)** — that message must carry a `DiagnosticId`.

Those two, with SST2310 off, are the right configuration for a long-lived library. All three together
are the right configuration for a codebase in the middle of a cleanup.

## What is reported

- **Every `[Obsolete]`**, whether or not it has a message, an error flag, a `DiagnosticId`, or a
  `UrlFormat`. A well-written deprecation is still a deprecation.
- On every declaration the attribute can be written on: types, methods, constructors, properties,
  fields, events, delegates, enum members, indexers, and operators.
- On an **`override`** that carries its own `[Obsolete]`. The attribute is written on the overriding
  member, so it can be deleted there — unlike a rule that would demand a different member shape, this
  one asks for something the derived type can do on its own.
- Whatever the accessibility. A private obsolete member is still unfinished work.

## What is not reported

- **An attribute of the same name that is not the framework's.** The name is matched on syntax first
  because it is free, but nothing is reported until the attribute binds to `System.ObsoleteAttribute`.
- **`[Obsolete]` on a member of a referenced assembly** — the rule reports declarations, not uses. A
  *call* to an obsolete member is the compiler's CS0618, which is already telling you.
- **Generated code.**

## How to fix violations

Delete the deprecated member and its callers. That is the whole rule, and it is why there is **no code
fix**: removing a member is not a mechanical edit, because every call site has to be migrated first, and
only you know whether that migration has happened.

If the member cannot go yet, that is a scheduling answer, not a code answer — suppress it at the site
with a note, or turn the rule off for the project as described above.

## Related rules

- **[SST2308](SST2308.md)** — an `[Obsolete]` with no message at all.
- **[SST2314](SST2314.md)** — an `[Obsolete]` with a message but no `DiagnosticId`.

The three overlap by design, because they want different things: a message, an id, and — eventually —
the code gone. A bare `[Obsolete]` is reported by SST2308 *and* by this rule.

## Performance

The clean path is the attribute's simple name, read straight off the syntax, which rejects every other
attribute in the file without binding anything. Only a declaration that is about to be reported is
bound at all — which is the step that tells a real `System.ObsoleteAttribute` from a type of the same
name.
