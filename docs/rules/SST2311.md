# SST2311 — Visible constants should be static readonly

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |
| Replaces | the analyzer [the rule](https://rules.the analyzer.com/csharp/RSPEC-2339/) |

## Cause

A `const` field that code in **another assembly** can read — that is, a `public`, `protected` or
`protected internal` constant declared in a type the outside world can see.

```csharp
public class Limits
{
    public const int Maximum = 100;        // reported
    protected const int Minimum = 1;       // reported
}
```

## Rule description

A `const` is not read at run time. The compiler copies its **value** into every call site, at the moment
that call site is compiled.

Inside one assembly that is harmless: everything is rebuilt together, so every copy is refreshed at the
same time. Across an assembly boundary it is not. Ship version 2 of your library with
`Maximum = 200`, and every caller that was compiled against version 1 keeps using **100** — no error, no
warning, no rebuild, just the old number, silently, until someone recompiles them. The constant you
"changed" never reached them.

A `static readonly` field does not have this problem. It is a real field, read from the declaring
assembly at run time, so a caller picks up the new value the moment it loads the new binary.

```csharp
public class Limits
{
    public static readonly int Maximum = 100;   // a change reaches every caller
}
```

## `static readonly` is not a drop-in replacement

**Be honest about this before you apply the fix.** A `const` is a compile-time value, and the language
requires a compile-time value in several places. In every one of them, `static readonly` **will not
compile**:

```csharp
public const int Maximum = 100;

[Range(0, Maximum)]                                  // attribute argument — needs a const
public int Value { get; set; }

public void Check(int v)
{
    switch (v)
    {
        case Maximum:                                // case label — needs a const
            break;
    }
}

public void Send(int limit = Maximum) { }            // default parameter value — needs a const
```

A constant that feeds an attribute argument, a `case` label, a default parameter value, another `const`,
or an array size in a `stackalloc`/fixed buffer **has to stay a `const`**. This rule cannot see those
uses across assemblies, so it does not try to guess: it reports the shape and leaves the judgement to
you. When the value is genuinely needed at compile time, the right answer is to suppress the rule at the
declaration and move on — not to contort the code.

The rule is most valuable on the constants that are *not* in that bucket: tuning numbers, limits,
version strings, default sizes — values that look like they might change one day, and whose whole
problem is that when they do, nobody finds out.

## What is reported

- A **`public`** `const` in a type visible outside the assembly, including one nested in a public or
  protected type, and including a type that is `sealed` or a `struct`.
- A **`const` in an `interface`**, which is implicitly `public`.
- A **`protected`** or **`protected internal`** `const` in a type another assembly can derive from — the
  derived type bakes the value in exactly the same way.
- **Each declarator separately** when one declaration introduces several constants
  (`public const int Min = 1, Max = 100;` reports twice), because each is copied into its callers on its
  own.

## What is not reported

- **`private`, `internal`, and `private protected` constants.** No caller outside the assembly can read
  them, so no caller outside the assembly can hold a stale copy. Inside the assembly, everything is
  rebuilt together.
- **Any constant — however public — declared inside a type the outside world cannot see**: an `internal`
  class, a `private` nested class. The containing chain has to be visible for the hazard to exist.
- **A `protected` constant in a `sealed` or `static` type.** Nothing can derive from it, so nothing can
  reach the constant.
- **Local constants** (`const int x = 1;` inside a method). They never leave the method.
- **Enum members**, which are not fields and are a different design decision entirely.
- **`static readonly` fields**, which are what the rule is asking for.

## How to fix violations

There is **no code fix**, deliberately: as the section above explains, the rewrite is not always legal,
and an automated fix that produced code which does not compile — or which quietly changed a compile-time
constant into a run-time read in a hot loop — would be worse than the warning.

When the value is not needed at compile time:

```diff
-public const int Maximum = 100;
+public static readonly int Maximum = 100;
```

When it is needed at compile time, keep the `const` and suppress the rule where it is declared.

## Divergence from the rule

the analyzer's the rule reports **only `public`** constants; a `protected const` in a public, unsealed
class does not trigger it. SST2311 reports that case too, because the hazard is identical — a derived
type in another assembly copies the value in at its own compile time, and a later change never reaches
it. Every site the rule reports is also reported here, so existing suppressions map across one for one; you
may see additional warnings on `protected` constants, and they are real.

## Performance

The `const` keyword is matched on the declaration's modifier list, which is a token scan and rejects
every ordinary field for free. Only a constant is ever bound, and only then is its accessibility and its
containing type's visibility examined.
