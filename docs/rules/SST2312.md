# SST2312 — Types should be declared in a named namespace

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |
| Replaces | the analyzer [the rule](https://rules.the analyzer.com/csharp/RSPEC-3903/) |

## Cause

A type is declared in the **global namespace** — in no namespace at all.

```csharp
public class Widget          // reported: no namespace
{
}
```

## Rule description

A type in the global namespace is visible from every file of every project that references the assembly,
and there is nothing a consumer can do about it. They cannot `using` their way around a name they never
asked for, and they cannot avoid a collision with a type of their own — a `Widget` in the global
namespace and their `Widget` in the global namespace is an ambiguity they have to resolve with an alias,
in every file that touches either.

A namespace is the one tool the language gives for this, and it costs one line:

```csharp
namespace Vendor.Widgets;

public class Widget
{
}
```

## What is reported

- A **class, struct, interface, enum, record, or record struct** declared directly in the compilation
  unit, with no enclosing namespace.
- **Regardless of accessibility** — an `internal` type in the global namespace still collides with
  another `internal` type in the same assembly, and still crowds the global name table for every file
  in the project.
- **Each part of a partial type separately**, because each part is a declaration that has to be moved.

## What is not reported

- **A delegate** declared in the global namespace. This keeps the rule's firing set identical to the
  the rule it replaces, so existing suppressions map across one for one.
- **A nested type.** Its enclosing type is the one that needs a namespace; reporting both would report
  the same problem twice.
- **A type in any namespace**, block-scoped or file-scoped. This rule only asks that a namespace exist.
- **The `Program` that top-level statements generate** — see below.
- **Generated code.**

## Top-level statements

A file of top-level statements produces an implicit `Program` class in the global namespace, and this
rule never reports it. It has no declaration of its own to move, so there is nothing to fix.

Neither is the explicit companion a project writes next to it — the shape integration tests use to reach
the entry point:

```csharp
// Program.cs
public partial class Program;      // not reported
```

That one **must** stay in the global namespace. Moving it into a namespace does not fail to compile; it
does something worse. The generated `Program` is in the global namespace, so a `partial class Program`
inside a namespace is simply a *different type* — the two stop merging, the members quietly go missing
from the entry-point class, and `WebApplicationFactory<Program>` no longer refers to what you think it
does. A rule that demanded that change would be demanding a bug, so it does not.

Both cases are recognised the same way: the compiler points the symbol's declaration back at a whole
compilation unit, which is something only the top-level-statements type does.

## This is not SST2237

[SST2237](SST2237.md) converts a **block-scoped namespace** into a **file-scoped** one. It is about how a
namespace that already exists is written. This rule is about a namespace not existing at all.

The two can never report the same declaration — a file with no namespace has nothing for SST2237 to
convert, and a file with one has nothing for this rule to report — so enabling both is safe and neither
duplicates the other.

## How to fix violations

There is **no code fix**. Wrapping the type in a namespace requires knowing *which* namespace, and an
analyzer does not: the project's root namespace is a guess, the folder path is a guess, and getting it
wrong means a rename that ripples through every consumer. Pick the namespace, add the declaration:

```diff
+namespace Vendor.Widgets;
+
 public class Widget
 {
 }
```

Use a file-scoped namespace (C# 10 and later), which is what [SST2237](SST2237.md) would ask for anyway.
On an older language version, use a block-scoped one.

## Performance

A type declaration's parent is checked first: anything nested in a namespace or in another type is
rejected on syntax alone, without touching the semantic model. Only a type that really is in the global
namespace — which is the reported set — is bound, and that is only to recognise the top-level-statements
`Program`.
