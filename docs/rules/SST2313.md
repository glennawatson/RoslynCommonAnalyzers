# SST2313 â€” Enums should use an allowed storage type

| Item | Value |
| --- | --- |
| Category | Design |

## Cause

An enum declares an underlying type that is not on the project's allowed list. The list defaults to
`int`.

```csharp
public enum Level : byte      // reported by default
{
    Low,
    High,
}
```

## Rule description

An enum that names no underlying type is stored as `int`, and `int` is what everything downstream
assumes unless told otherwise â€” a reader skimming the type, a serializer choosing a wire width, an
interop signature matching a native `enum`. Naming a different one is a real decision, and this rule
exists to make sure it was a decision rather than a habit.

It is often a deliberate decision, which is what the configuration is for.

## When this rule is wrong

A `byte`- or `long`-backed enum is a legitimate, deliberate choice:

- **`byte` or `short` to pack a struct.** An enum field inside a struct that you allocate by the million
  costs exactly its storage type. Dropping from `int` to `byte` is three bytes a copy, and it is the
  whole reason to do it.
- **`long` to carry more than thirty-two flags.** A `[Flags]` enum with thirty-three members has no
  choice.
- **A fixed width to match a wire format or a native signature.** If the protocol says one byte, the
  enum is one byte, and no amount of style advice changes that.

If your project does any of these deliberately, **widen the allowed list â€” do not suppress the rule.**
The list is the rule's whole design:

```ini
[*.cs]
# Allow the packing sizes this project actually uses.
stylesharp.SST2313.allowed_enum_storage = int, byte, long
```

The rule then keeps working: it still catches the `ushort` somebody reached for by accident, while
staying silent on the `byte` you chose on purpose.

## Configuration

| Key | Values | Default |
| --- | --- | --- |
| `stylesharp.SST2313.allowed_enum_storage` | comma-separated type names | `int` |
| `stylesharp.allowed_enum_storage` | comma-separated type names | `int` |

The rule-specific key wins over the general one. Types may be named as C# keywords or as CLR names, and
matching is case-insensitive â€” `long`, `Int64` and `INT64` all mean the same thing.

```ini
[*.cs]
stylesharp.allowed_enum_storage = int, byte
```

An unset or empty value falls back to the default, so a typo in the key does not turn every enum in the
project into a diagnostic.

## What is reported

- An enum whose underlying type is not on the allowed list: by default `byte`, `sbyte`, `short`,
  `ushort`, `uint`, `long`, and `ulong`.
- **Regardless of where it is declared or what it is marked with** â€” a nested enum, a `private` enum, an
  `internal` enum, and a `[Flags]` enum are all held to the same list. `[Flags]` is not an exemption:
  a flags enum with four members has no more need of a `long` than any other.

## What is not reported

- **An enum with no base list.** It names no underlying type, so it is already an `int` â€” and it is
  rejected on syntax alone, without binding anything.
- **An enum whose underlying type is on the allowed list**, including one that spells `int` out
  explicitly (`enum Level : int`). Writing the default out is redundant, but it is not this rule's
  business.

## How to fix violations

Either drop the underlying type, when it was not a considered choice:

```diff
-public enum Level : byte
+public enum Level
 {
     Low,
     High,
 }
```

â€¦or, when it was, tell the project so:

```ini
stylesharp.SST2313.allowed_enum_storage = int, byte
```

There is **no code fix**: only you know which of those two the answer is, and an analyzer that guessed
would either undo a deliberate packing decision or entrench an accidental one.

## Performance

Most enums name no underlying type at all, and those are rejected on the presence of a base list â€” a
null check on a syntax node. Nothing else happens: no semantic model, no option read. An enum that does
name a storage type costs one symbol bind and one scan of the configured list, which is matched straight
out of the `.editorconfig` string without allocating a set.
