# SST2314 — Obsolete attributes should carry a DiagnosticId

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |
| Requires | `ObsoleteAttribute.DiagnosticId` — .NET 5 or later |

## Cause

An `[Obsolete]` attribute carries a message but no `DiagnosticId`.

```csharp
[Obsolete("Use SendAsync(Message).")]      // reported: a message, but no id
public void Send(string body) { }
```

## Rule description

The message tells a caller **what to do**. The `DiagnosticId` is what lets them **do it**.

Without an id, every deprecation in every library a project references collapses into the same warning:

```text
warning CS0618: 'Mailer.Send(string)' is obsolete: 'Use SendAsync(Message).'
```

One number, `CS0618`, for all of them. So a caller who has read your message, agreed with it, and
scheduled the migration for next quarter has exactly one lever: suppress `CS0618`. That silences your
deprecation *and* every other deprecation, from every other library, including the ones they have not
read yet. The blunt instrument is the only instrument.

Give the attribute an id of your own and the warning becomes something a caller can actually hold:

```csharp
[Obsolete(
    "Use SendAsync(Message).",
    DiagnosticId = "ACME001",
    UrlFormat = "https://acme.dev/deprecations/{0}")]
public void Send(string body) { }
```

Now the caller gets `warning ACME001`, and with it:

- **A suppression that means one thing.** `#pragma warning disable ACME001` — or a severity entry in
  `.editorconfig` — silences *this* migration and nothing else.
- **A severity they choose.** `dotnet_diagnostic.ACME001.severity = error` in the module that has
  finished migrating, `= none` in the one that has not.
- **A link.** `UrlFormat` turns the id into the URL of your migration guide — the compiler surfaces it,
  the IDE makes it clickable, and the caller reads the instructions instead of searching for them.

A message alone is a note. An id is an affordance.

## Framework requirement

`ObsoleteAttribute.DiagnosticId` and `UrlFormat` arrived in **.NET 5**. On netstandard2.0, .NET
Framework, and any older target the properties simply do not exist, and writing one would not compile.

**The rule is gated on the property actually being there.** It resolves `System.ObsoleteAttribute` in the
analyzed compilation once, at compilation start, checks that the type really has a settable
`DiagnosticId`, and if it does not it **registers nothing at all** — no syntax callback, no per-file
cost, and no possibility of the rule firing. A project that cannot take the advice never hears it, and
pays nothing for its absence. The check is on the resolved symbol, not on a target-framework string, so
a polyfill or a multi-targeted build is judged on what it can actually bind.

## What is reported

- `[Obsolete("…")]` — a message with no `DiagnosticId`.
- `[Obsolete("…", true)]` — the error flag is not an id.
- `[Obsolete("…", UrlFormat = "…")]` — a `UrlFormat` without a `DiagnosticId` is inert: the format's
  `{0}` is the id, so with no id there is no URL.
- Wherever the attribute can be written: types, methods, constructors, properties, fields, events,
  delegates, enum members, indexers, and operators.

## What is not reported

- **`[Obsolete]` with no message at all** — `[Obsolete]`, `[Obsolete("")]`, `[Obsolete(null)]`. That is
  [SST2308](SST2308.md)'s diagnostic, and it is the bigger problem; the two rules never report the same
  attribute.
- **Any attribute that sets `DiagnosticId`**, which is what the rule is asking for.
- **A compilation whose `ObsoleteAttribute` has no `DiagnosticId`** — see above. The rule is not
  registered.
- **An attribute of the same name that is not the framework's.**
- **Generated code.**

## How to fix violations

There is **no code fix**. The id is a naming decision — it belongs to your package, it has to be unique
within it, and it has to stay stable once callers start suppressing it. An analyzer inventing `ACME001`
for you would be inventing part of your public contract.

Pick a prefix for the package and number the deprecations:

```diff
-[Obsolete("Use SendAsync(Message).")]
+[Obsolete(
+    "Use SendAsync(Message).",
+    DiagnosticId = "ACME001",
+    UrlFormat = "https://acme.dev/deprecations/{0}")]
 public void Send(string body) { }
```

Document the ids alongside the package, and point `UrlFormat` at that page.

## Related rules

- **[SST2308](SST2308.md)** — the attribute has no message at all.
- **[SST2310](SST2310.md)** — the deprecated code is still here. That rule reports every `[Obsolete]`,
  including a fully specified one, because its ask is removal rather than documentation. If you ship a
  library that must keep its obsolete members, turn SST2310 off and keep this rule on.

## Performance

The attribute's simple name is matched on syntax, which rejects every other attribute in the file for
free, and the `DiagnosticId` initializer is found by scanning the attribute's own argument list — no
binding. Only an attribute that has a message and no id is bound, which is the step that confirms it is
really `System.ObsoleteAttribute`. On a target without the property the analyzer registers no callback,
so the cost is a single symbol lookup per compilation.
