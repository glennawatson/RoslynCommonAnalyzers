# SST2315 - A type that owns a disposable should be disposable

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Yes for the field and property shapes (add IDisposable and a Dispose) |
| Default | Enabled |

## Cause

A type creates and keeps a disposable but does not implement `IDisposable` or `IAsyncDisposable`.

```csharp
public sealed class Reader // SST2315
{
    private readonly Stream _stream = File.OpenRead("data.bin");
}
```

## Rule description

A type that builds a disposable and holds onto it has taken ownership of a resource - a handle, a
socket, an event subscription. The only place that resource can be released is a disposal method the
type does not have, so no owner, container, or `using` ever cleans it up, and it leaks for the life of
the process.

The rule reports only the ownership shapes that a plain `new` on a field does not already surface:

- a field or auto-property whose initializer is a **static factory call** - `File.OpenRead`, a `Create`
  method, an extension such as `CreateScope`;
- an **auto-property initialized with `new`**;
- a **collection the type creates and then fills with newly created disposable elements**
  (`private readonly List<Res> _all = new();` fed by `_all.Add(new Res())`).

It does not report a member assigned from a constructor parameter or a primary-constructor parameter -
that is an injected dependency the caller owns and must not be disposed here. An instance call on an
injected dependency, a `static` field, and a `Task`-typed field are left alone too.

## How to fix violations

Implement `IDisposable` (or `IAsyncDisposable` when the owned type is async-disposable) and release the
owned members. For a collection, loop over it in `Dispose` or hold the elements in a composite disposer.

```csharp
public sealed class Reader : IDisposable
{
    private readonly Stream _stream = File.OpenRead("data.bin");

    public void Dispose() => _stream.Dispose();
}
```

The code fix adds `IDisposable` and a `Dispose()` that releases each owned member. It is offered only
when every owned member is synchronously disposable; a collection, or a member that is only
async-disposable, is a design decision the fix does not make.

## When not to use the rule

Turn it off in code where ownership is deliberately handed elsewhere by a convention the rule cannot see
- for example a factory type that returns the disposables it builds without keeping them.
