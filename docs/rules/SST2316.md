# SST2316 - A disposal method needs its disposal interface

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Yes (add the interface) |
| Default | Enabled |

## Cause

A type declares a public `Dispose()` or `DisposeAsync()` but does not implement the matching interface.

```csharp
public class Cache // SST2316
{
    public void Dispose()
    {
    }
}
```

## Rule description

A public `Dispose()` or `DisposeAsync()` on a type that does not implement the matching interface is
cleanup nobody runs. Dependency-injection containers, service scopes, and composite disposers all
dispose through `IDisposable` and `IAsyncDisposable`; a type that has the method but not the interface
is registered, held for the life of the process, and never cleaned up - silently, because the code
compiles and looks correct.

The exemptions are the point of the rule:

- a **`ref struct`** is never reported: it disposes by pattern (`using` binds to the method by
  duck-typing), and it may not be permitted an interface on the language version in use;
- a **duck-typed enumerator**, with `MoveNext()` and `Current`, is disposed by `foreach` through the
  pattern;
- an **explicit interface implementation**, and a `Dispose()` **inherited from a base** that already
  implements the interface, both leave the type implementing it, so they never match.

The `DisposeAsync` half is silent where the framework has no `IAsyncDisposable`.

## How to fix violations

Implement the interface the method already matches.

```csharp
public class Cache : IDisposable
{
    public void Dispose()
    {
    }
}
```

The code fix adds `IDisposable` or `IAsyncDisposable` to the base list.

## When not to use the rule

If a type exposes a `Dispose()` purely as a domain operation and is never handed to an owner that would
call it, the rule is a false positive; rename the method or suppress the rule there.
