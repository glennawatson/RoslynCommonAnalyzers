# SST2317 - Wrap an owned native handle in a SafeHandle

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A disposable type owns a native resource in a raw `IntPtr` field, releases it only on the disposal path,
and has no finalizer.

```csharp
public sealed class Buffer : IDisposable // SST2317 on _handle
{
    private IntPtr _handle = Marshal.AllocHGlobal(1024);

    public void Dispose() => Marshal.FreeHGlobal(_handle);
}
```

## Rule description

A disposable holding an owned native resource in a raw `IntPtr` field leaks that resource whenever
`Dispose` is not called - an exception before disposal, a caller that forgets, a finalizer the type does
not have. A hand-written finalizer is not the answer: it runs on the finalizer thread with no ordering
guarantee, can resurrect the object, and can race an in-flight P/Invoke.

The fix is a `SafeHandle`: a critical-finalization object with ref-counted release, guaranteed run
order, and P/Invoke marshalling that closes the use-after-free window. The concrete on-ramps are
`SafeFileHandle`, `SafeWaitHandle`, or a small `SafeHandleZeroOrMinusOneIsInvalid` subclass; where the
field is raw memory rather than an OS handle, `NativeMemory.Alloc`/`Free` (.NET 6+) fits.

The rule reports only the no-finalizer case, where the leak is real, and only when the field is passed
to a call on the disposal path - the proof that it is an owned resource rather than an opaque cookie. A
type that already has a finalizer, or whose native state is already a `SafeHandle`, is left alone. The
rule stays silent unless `System.Runtime.InteropServices.SafeHandle` resolves, so the suggestion always
compiles.

## How to fix violations

Replace the `IntPtr` field with a `SafeHandle`.

```csharp
public sealed class Buffer : IDisposable
{
    private readonly SafeBuffer _handle = /* allocate */;

    public void Dispose() => _handle.Dispose();
}
```

There is no code fix: moving an `IntPtr` owner to a `SafeHandle` changes the field layout, the P/Invoke
signatures, and the release semantics, and a partial rewrite would be worse than none.

## When not to use the rule

If the `IntPtr` genuinely wraps a handle whose lifetime a `SafeHandle` cannot model - a handle owned by
another process or library - suppress the rule at that field.
