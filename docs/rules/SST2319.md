# SST2319 -- An optional parameter whose default can never be used

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method declares an optional parameter, but a same-named overload takes exactly the parameters that come
before it.

```csharp
public sealed class Logger
{
    public void Log(string message) { }

    public void Log(string message, bool echo = false) { }   // SST2319: echo's default is dead
}
```

## Rule description

Overload resolution prefers the candidate that does not rely on an omitted optional argument. Every call
that writes `Log("hi")` therefore binds `Log(string)`, not `Log(string, bool)`, so the `echo = false`
default is never reached. It is dead surface: it appears in completion and documentation, promising a
behaviour no caller can actually get.

The rule takes each ordinary method that declares an optional parameter, reads the required parameters
before the first optional one as a prefix, and looks for a same-named overload whose whole parameter list
is exactly that prefix -- same count, same types, same ref kinds. When one exists, the optional parameter is
reported.

An optional parameter with no shorter overload behind it is fine and is not reported, and overloads whose
prefixes differ in type do not shadow each other.

## How to fix violations

Remove the default so the parameter becomes required, or change the two signatures so the shorter overload
no longer matches every call that omits the argument. There is no code fix: deleting a default changes the
emitted metadata and the callable surface, so the author decides which method to keep.

## Performance

A method with no optional parameter -- nearly every method -- is skipped after a single parameter walk,
before any overload set is scanned. The prefix text for the message is built only after a shadowing overload
is found.
