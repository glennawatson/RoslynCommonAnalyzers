# SST2321 - Do not terminate the process from library code

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A class library calls `Environment.Exit` or `Environment.FailFast`.

```csharp
public sealed class Uploader
{
    public void Upload(Stream data)
    {
        if (data is null)
        {
            Environment.Exit(1); // SST2321
        }
    }
}
```

## Rule description

A class library does not own the process it runs in. The same assembly may be loaded into a web server
that is serving other requests, a test host that is running other tests, or a long-lived tool that only
wanted to call one method. `Environment.Exit` and `Environment.FailFast` end that whole process, and they
make the decision on the host's behalf - it never gets a say.

`Environment.FailFast` is the sharper of the two. It skips every pending `finally` block on its way out and
writes a crash dump before the process dies, so nothing gets the chance to flush a buffer, roll back a
transaction, or release a handle. `Environment.Exit` runs finalizers but still takes the whole host down.

A library that reaches an unrecoverable condition should throw. An exception travels up to whoever called
the library and lets that caller decide what to do: retry, log and continue, fail the one request, or - if
it really is an application that owns the process - shut down deliberately. Only the top of the stack has
the context to make that call.

The rule fires only when the compilation is a **class library**. An application (a console app or any other
executable output) does own its process, so ending it there is a legitimate choice, and the rule registers
nothing at all for an executable build.

Where the host is a generic-host application, requesting an orderly shutdown through the host's application
lifetime is the cooperative equivalent of what the terminating call did abruptly; when that lifetime type
is available, the diagnostic message points at it.

## How to fix

Throw instead of terminating, and let the caller decide.

```csharp
public sealed class Uploader
{
    public void Upload(Stream data)
    {
        if (data is null)
        {
            throw new ArgumentNullException(nameof(data));
        }
    }
}
```

There is **no code fix**. What should happen when the failure is hit is a design decision the analyzer
cannot make for you: which exception type carries the right meaning, whether the condition is truly
unrecoverable, and whether a hosted application should instead request a graceful shutdown. Replacing a
process kill with a thrown exception changes control flow, and that belongs to the author.

## What is reported

- `Environment.Exit(...)` and `Environment.FailFast(...)`, however the call is qualified, in a library
  compilation.

## What is not reported

- The same calls in an executable (a console or other application output). An application owns its process.
- An `Exit` or `FailFast` method declared on some other type. Only `System.Environment` is meant.
- A compilation with no `System.Environment` to bind. The rule registers nothing.

## Performance

The rule is registered only for a library whose compilation can bind `System.Environment`; an application
build, and a build without the type, pay a single check at compilation start and nothing more. Each
invocation is filtered on the called member's name before the semantic model is touched, so only the handful
of calls that could be `Exit` or `FailFast` are bound to confirm the containing type.
