# SST2322 — Readonly fields should not expose a mutable collection

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A non-private instance `readonly` field has a mutable collection type — a `List<T>`, `Dictionary<,>`,
`HashSet<T>`, `Collection<T>`, an array, and the like.

```csharp
using System.Collections.Generic;

public class Order
{
    public readonly List<string> Lines = new();
}
```

`readonly` reads as "settled", and here it is not. The reference cannot be reassigned, but the collection
behind it can be changed by anyone who can see the field:

```csharp
var order = new Order();
order.Lines.Add("injected");
order.Lines.Clear();
```

The `readonly` keyword froze the wrong half. Nothing stops a caller reaching straight past `Order` and
rewriting the data it owns.

## Rule description

The rule reports a field only when all of the following hold:

- it is **`readonly`** — a non-readonly field can also have its reference swapped, which is a plainer
  problem left to other rules;
- it is an **instance** field — a `static` mutable field is shared global state, a separate concern;
- it is **not private** — a `private` field (and a `private protected` one) keeps the collection under the
  type's own control, so it is left alone;
- its type is one of the **mutable collection** shapes. An immutable or frozen collection, a read-only
  wrapper, an `IReadOnlyList<T>`, a scalar, or any user type passes without a word — `readonly` on those
  already means what it says.

The clean path is a modifier scan: a field that is not `readonly`, that is `static`, or that names no
accessibility beyond private is rejected on syntax alone, before the semantic model is consulted.

## How to fix violations

There is **no code fix** — what to expose instead is a design decision.

Expose the collection through a read-only type, keeping the mutable one private:

```csharp
using System.Collections.Generic;

public class Order
{
    private readonly List<string> _lines = new();

    public IReadOnlyList<string> Lines => _lines;
}
```

Callers can read the collection and the owning type can still add to it, but no one outside `Order` can
rewrite the data. Where a snapshot is wanted instead, hand back a copy so a caller's changes do not touch
the original.

## Configuration

None.

## Performance

The rule is driven from the field declaration. The overwhelming majority of fields — private ones,
non-readonly ones, static ones — are rejected on their modifiers alone, with no bind. Only a non-private
instance `readonly` field pays for a single semantic lookup, and only to learn whether its type is one of
the mutable collection shapes, which are resolved once per compilation and only when the first such field
is seen.
