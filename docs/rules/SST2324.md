# SST2324 — Do not declare a member more accessible than its containing type

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A member declares an accessibility wider than its containing type can deliver.

```csharp
internal class Cache
{
    public void Clear()   // 'public' — but nothing outside the assembly can reach Cache
    {
    }

    public class Entry    // 'public' — likewise capped at internal by Cache
    {
    }
}
```

## Rule description

A member is only ever as reachable as the type that contains it. `Cache` is `internal`, so no code outside
the assembly can name it — and therefore none can reach `Cache.Clear` or `Cache.Entry`, whatever modifier
those carry. The `public` is dead: it promises a reach the container never grants, and it misleads a reader
into taking the member for part of the public surface.

The containing type's **effective** accessibility is what matters, and it is computed by walking every
enclosing type. A `public` member of a `public` type nested in an `internal` type is still effectively
internal, so both the nested type and its member are reported:

```csharp
internal class Outer
{
    public class Middle          // reported: effectively internal
    {
        public void Method()     // reported: effectively internal
        {
        }
    }
}
```

Accessibility is compared by the set of callers it admits, not by a single ordering. `protected` and
`internal` admit disjoint caller sets — a derived type in another assembly versus a non-derived type in this
one — so neither is "wider" than the other, and a `protected` member of an `internal` class is **not**
reported. A `protected internal` member of an `internal` class **is**: its `internal` half is preserved, but
its `protected` half — reachable only by deriving across an assembly boundary — is dead, because an
`internal` class cannot be derived from outside its assembly.

### What is not reported

- A member declared no wider than its container, or less accessible than it.
- Every member of a `public` top-level type: it caps nothing.
- A member whose accessibility a contract fixes and the author cannot narrow: an `override` matches its base
  member, and an explicit or implicit interface implementation matches the interface member. Narrowing any of
  these would not compile, so reporting them would hand you a diagnostic you cannot act on.
- Interface members (implicitly public) and enum members (implicitly public): there is no author-written
  modifier to narrow.
- A member a test framework requires to be `public`. A TUnit lifecycle hook (`[Before]`, `[After]`,
  `[BeforeEvery]`, `[AfterEvery]`) is rejected by the framework at any lesser accessibility, so narrowing it —
  the only fix this rule could offer — would not build. Such a member is left alone even inside an `internal`
  type.

## How to fix violations

Declare the member no wider than its container's effective accessibility. For the opening example:

```csharp
internal class Cache
{
    internal void Clear()
    {
    }

    internal class Entry
    {
    }
}
```

If the member genuinely needs to be part of the public surface, the fix is the other direction: widen the
containing type, so the modifier and the reach agree.

## Configuration

None.

## Performance

The rule is symbol-driven and its clean path never touches syntax. A type whose effective accessibility is
`public` — nearly all of them — returns before its members are scanned; every other type pays a short
member scan of cheap accessibility comparisons. Only once a widening member is found is any further work done:
the interface-implementation check and the syntax read that locates the offending modifier both run on that
rare path alone.
