# SST2325 ŌĆö Validate an async method's arguments before its first await

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An `async` method checks one of its arguments after its first `await`.

```csharp
public async Task SaveAsync(Stream target) // SST2325
{
    await _gate.WaitAsync();

    if (target is null)
    {
        throw new ArgumentNullException(nameof(target)); // SST2325
    }

    // ...
}
```

## Rule description

An `async` method starts running the moment it is called, but only up to its first `await`. Everything past
that runs on a continuation, once the returned task is awaited. So a guard placed after the first `await`
does not throw at the call site, where the caller's stack still says who passed the bad value ŌĆö it surfaces
later, when the task is awaited, and the bad argument has already been accepted.

```csharp
var task = SaveAsync(null); // no exception here ŌĆö the argument looked fine
// ... the task is stored, combined, handed on ...
await task;                 // ArgumentNullException here, far from the call that was wrong
```

The guard is not wrong. It is on the wrong side of the `await`.

### What is reported

Within an `async` method that returns something other than `void`, a guard that appears after the method's
first `await` and checks one of the method's parameters:

- an `if` that throws an `ArgumentŌĆ”Exception` ŌĆö `ArgumentNullException`, `ArgumentException`,
  `ArgumentOutOfRangeException`, or any type named the same way ŌĆö whose condition or thrown exception names a
  parameter;
- a runtime throw-helper call, `ArgumentNullException.ThrowIfNull(parameter)` or any
  `ArgumentŌĆ”Exception.ThrowIfŌĆ”(parameter)`, whose argument names a parameter.

The "first await" is a statement that awaits the moment it runs: `await X();`, `x = await X();`, or
`T x = await X();`. An `await` buried in an `if` or a loop is not treated as the boundary, because whether it
runs before the guard is a runtime question this rule does not answer. A guard placed *before* the first
`await` runs synchronously at the call site and is never reported. A guard, or an `await`, inside a nested
lambda or local function belongs to that function, not to the method around it, and is left alone. An
`async void` method is a separate problem and is not reported here.

## How to fix violations

Move every argument check ahead of the first `await`, so it runs synchronously when the method is called:

```csharp
public async Task SaveAsync(Stream target)
{
    if (target is null)
    {
        throw new ArgumentNullException(nameof(target));
    }

    await _gate.WaitAsync();
    // ...
}
```

When the validation has to precede work that is genuinely asynchronous, split the method: a synchronous
method that validates and then calls a private `async` implementation, so the check throws at the call site
and only the real work is deferred.

```csharp
public Task SaveAsync(Stream target)
{
    if (target is null)
    {
        throw new ArgumentNullException(nameof(target));
    }

    return SaveCoreAsync(target);

    async Task SaveCoreAsync(Stream stream)
    {
        await _gate.WaitAsync();
        // ...
    }
}
```

## Configuration

None.

## Performance

The clean path is a look at a method's modifiers: only a method carrying `async` and returning something
other than `void` pays for a single scan of its own top-level statements, and nothing in the rule binds a
symbol ŌĆö an `await` and a guard are both syntactic facts. The parameter-reference walk runs only once a
candidate `ArgumentŌĆ”Exception` guard has been found after the first `await`.
