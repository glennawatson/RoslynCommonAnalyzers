# SST2326 — Do not narrow an interface reference to a concrete implementation type

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Info |
| Code fix | No |
| Default | Enabled |

## Cause

A value whose static type is an interface is narrowed to a concrete class that implements it — through an
explicit cast, an `as`, or an `is` type test.

```csharp
void Draw(IShape shape)
{
    var circle = (Circle)shape;        // explicit cast
    var maybe = shape as Circle;       // as
    if (shape is Circle c) { }         // is, with a declared variable
    if (shape is Circle) { }           // is, a bare type test
}
```

## Rule description

An interface exists so callers depend on the contract, not on any one class behind it. Reaching past the
interface to a named implementation throws that away: the code now assumes a specific type is on the other end.
Substitute another implementation — a decorator, a proxy, a test double, a future rewrite that returns a
different class — and the cast throws `InvalidCastException`, the `as` quietly yields `null`, or the `is` test
takes the wrong branch. None of that shows up until run time, and it shows up precisely when someone tries to
use the abstraction the way an abstraction is meant to be used.

The rule fires only when it can be sure the narrowing is real:

- the operand's static type is a genuine interface, and
- the target is a **non-abstract class** that actually implements that interface.

```csharp
IReader reader = OpenReader();
var file = (FileReader)reader;         // reported: FileReader implements IReader
file.FlushToDisk();                    // the member that tempted the cast
```

### What is not reported

- **Interface to interface.** Widening or moving between contracts keeps you on abstractions.

  ```csharp
  IEnumerable<int> items = GetItems();
  var list = (IList<int>)items;        // not reported: IList<int> is still an interface
  ```

- **A cast to an abstract base**, even one that implements the interface: an abstract class is itself an
  abstraction, not the concrete implementation this rule is about.

- **A cast to a struct** that implements the interface: the target is not a class.

- **An unrelated narrowing** whose target does not implement the interface. The compiler either rejects it or
  leaves it open for a subclass to satisfy; either way it is not this rule's concern.

- **An operand that is not an interface** — `object`, `dynamic`, a concrete class, or a **type parameter** even
  when it is constrained to an interface. A type parameter can be a value type, and its own identity, not the
  interface, is what the code is written against.

- **An `is` pattern that is not a plain type match** — `is not`, a constant pattern, a recursive pattern —
  because there is no single concrete type being singled out.

## How to fix violations

Keep working through the interface. If you reached for the concrete type to call a member the interface does not
expose, that member belongs on the interface — or on a further interface the value can be typed as — not fetched
by narrowing to the implementation:

```csharp
interface IReader
{
    void Flush();                      // promote the member to the contract
}

void Save(IReader reader) => reader.Flush();
```

If a single, local, deliberate narrowing really is the pragmatic answer — a fast path keyed on a known runtime
type, for instance — this rule is a suggestion, not an error, and can be left in place or silenced at that site.

## Configuration

None.

## Performance

The clean path allocates nothing. A cheap syntactic dispatch on the node kind rejects everything that is not one
of the four narrowing shapes before the semantic model is touched, and an `is` pattern that is not a declaration
pattern binds nothing. The interface and concrete-type display strings that fill the message are built only once
a violation is confirmed.
