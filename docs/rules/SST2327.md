# SST2327 — Do not branch on a type's own runtime type

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An instance tests its own runtime type against a specific named class — `this is Derived`, `this as Derived`,
or `this.GetType() == typeof(Derived)`.

```csharp
public abstract class Shape
{
    public double Area()
    {
        if (this is Circle circle)          // reported: the base branches on which subtype it is
        {
            return 3.14159 * circle.Radius * circle.Radius;
        }

        if (this.GetType() == typeof(Square))  // reported: same smell, spelled with GetType
        {
            var square = (Square)this;
            return square.Side * square.Side;
        }

        return 0;
    }
}
```

## Rule description

A base type that checks which of its own subtypes it "really" is defeats polymorphism. The base takes on
knowledge of its derivations, and every new subtype forces this branch — and every other one like it — to be
revisited. Miss one and the type silently returns the wrong answer.

The behaviour that varies by type belongs in a virtual member the base declares and each derived type
overrides. The runtime then dispatches to the right implementation on its own, and a new subtype supplies its
own behaviour without touching the base.

Three shapes are reported when their operand is the instance itself:

- `this is Derived` and `this is Derived name` — a type test or declaration pattern.
- `this as Derived` — a cast test.
- `this.GetType() == typeof(Derived)` — and the `!=` form, and either operand order.

### What is not reported

- A capability check against an **interface** (`this is IDisposable`). Feature detection against a contract a
  subtype may or may not implement is a distinct, accepted pattern, so it is left alone.
- A test against a **type parameter** (`this is T`): there is no specific named type being singled out.
- A test of some **other value** rather than the instance (`other is Derived`) — the smell is specific to a
  type inspecting *itself*.
- A **negated or property pattern** (`this is not Derived`, `this is { }`): neither singles out a named type in
  the way this rule targets.
- A `GetType()` comparison whose receiver is **not an explicit `this`** (`GetType() == typeof(Derived)`, or a
  call on another object). Only the explicit `this.GetType()` form is matched, keeping detection syntactic.

## How to fix violations

Move the type-specific behaviour into a virtual member and override it per subtype:

```csharp
public abstract class Shape
{
    public abstract double Area();
}

public sealed class Circle : Shape
{
    public double Radius { get; init; }

    public override double Area() => 3.14159 * Radius * Radius;
}

public sealed class Square : Shape
{
    public double Side { get; init; }

    public override double Area() => Side * Side;
}
```

## Configuration

None.

## Performance

The clean path is purely syntactic. An `is`/`as` test or a declaration pattern is looked at only when its
operand is literally `this`; an equality is looked at only when one side is a `typeof(...)` and the other is a
zero-argument `this.GetType()` call. The semantic model is consulted once, and only after that rare syntactic
shape matches, to confirm the tested type is a named class.
