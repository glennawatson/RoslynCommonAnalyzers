# SST2328 — Do not expose a raw native pointer handle

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A visible instance field or property whose type is a raw native pointer — `IntPtr`, `UIntPtr`, `nint`, or
`nuint` — hands the bare handle out across the type boundary.

```csharp
public sealed class Window
{
    public IntPtr Handle;                 // reported: a visible native handle field
    public IntPtr Buffer { get; set; }    // reported: a visible native handle property
}
```

## Rule description

A raw `IntPtr`/`UIntPtr` is just an address. Whoever can read one can pass it to a native free, hand it to
unmanaged code, or keep it past the moment the owning object releases it; whoever can write one can point the
object at memory it does not own. Exposing that value lets a caller reach straight past the type that owns the
native memory and corrupt it, double-free it, or use it after free — the kind of defect that surfaces as a
process crash far from its cause.

A member is reported when both of the following hold:

- Its declared accessibility is one the outside world can reach — `public`, `protected`, or
  `protected internal`. An `internal` or `private protected` member keeps the handle inside the assembly, and a
  `private` one keeps it inside the type; none of those is reported.
- It is an instance member. A `static` handle is not part of an instance's surface and is left to other rules.

Both the field form and the property form are reported, and for the same reason: they expose the same value.
Turning a flagged field into a `public` property therefore does not clear the diagnostic — the property is
reported too. A get-only property is enough on its own, because reading the address is all a caller needs to
free or corrupt the memory behind it.

### What is not reported

- A `private`, `internal`, or `private protected` field or property: the handle never leaves the assembly, or
  never leaves the type.
- A `static` field or property.
- A member whose accessibility a contract fixes and the author cannot narrow: an `override` matches its base
  member, and an explicit interface implementation is private and reachable only through the interface.
- An indexer: it hands back a computed value rather than exposing a stored handle.
- A field or property of any type other than `IntPtr`, `UIntPtr`, `nint`, or `nuint`.
- Members of an interface: an interface declares a contract and owns no memory, so there is nothing there to
  move behind a private field.

## How to fix violations

Keep the raw value in a `private` field and expose the resource through a
[`SafeHandle`](https://learn.microsoft.com/dotnet/api/system.runtime.interopservices.safehandle). A `SafeHandle`
is a critical-finalization object with ref-counted release: it keeps the handle valid for exactly as long as it
is in use, closes the use-after-free window, and marshals correctly across a native call, so callers hold the
wrapper rather than the bare address.

```csharp
public sealed class Window
{
    private readonly SafeWindowHandle _handle;

    public Window(SafeWindowHandle handle) => _handle = handle;

    // Callers work through the SafeHandle; the raw address never leaves the type.
}
```

`SafeFileHandle`, `SafeWaitHandle`, or a small `SafeHandleZeroOrMinusOneIsInvalid` subclass are the usual
on-ramps. Where the field is raw memory rather than an OS handle, keep it private and release it through
`NativeMemory.Alloc`/`Free` (.NET 6+) or your allocator's own free routine.

## Configuration

None.

## Performance

The rule is symbol-driven and its clean path never touches syntax. A type that is not a concrete `class` or
`struct`, or is `static`, returns before its members are scanned; every other type pays a short member scan of
cheap checks — kind, static flag, accessibility, and special type. No syntax is read, and nothing is allocated
until a violation is reported.
