# SST2329 — Flags enums should declare a zero value

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Yes (add `None = 0`) |
| Default | Enabled |

## Cause

An enum marked `[Flags]` declares no member whose value is zero.

```csharp
using System;

[Flags]
public enum Access   // SST2329
{
    Read = 1,
    Write = 2,
}
```

## Rule description

A flags enum models a set, and every set has an empty member. Without a zero value there is no name for
"no flags set", so the empty state has to be written as a magic `(Access)0`, a default-initialized field
prints as an empty string rather than a meaningful name, and there is nothing to compare against when you
mean "none selected". Declaring a zero member — `None = 0` by convention — gives the empty set a name the
same way every other combination has one.

This is the companion of the distinct-bit-values rule, and the two do not overlap: that rule is about the
other members owning their own bits and treats a zero member as the expected empty set, while this rule is
only about the empty set being missing.

## What is reported

- A `[Flags]` enum whose members never include a constant value of zero, however the members are numbered.

## What is not reported

- A `[Flags]` enum that already has a zero-valued member, whatever its name (`None`, `Empty`, …).
- A default-numbered flags enum whose first member is therefore zero.
- An enum without the `[Flags]` attribute: it makes no set promise, so a missing zero is not this rule's
  concern.

## How to fix violations

Add a zero-valued member — the code fix inserts `None = 0` as the first member:

```diff
 [Flags]
 public enum Access
 {
+    None = 0,
     Read = 1,
     Write = 2,
 }
```

## Performance

The clean path is a type-kind check, so a compilation's non-enum types cost one comparison each and an enum
without the attribute costs one attribute scan; only a flags enum reaches the member scan.
