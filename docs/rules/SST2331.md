# SST2331 — Give enum members explicit values

| Item | Value |
| --- | --- |
| Category | Design |
| Severity | Warning |
| Code fix | Yes (assign explicit values) |
| Default | Disabled (opt-in) |

## Cause

An enum has at least one member with no `= value`, so its number is the position it happens to sit in.

```csharp
public enum Color   // SST2331
{
    Red,
    Green,
    Blue,
}
```

## Rule description

When a member has no explicit value, its number is decided by declaration order — and that number is real:
it is what gets persisted, sent on the wire, and compared against. Insert, remove, or reorder a member above
it and the number shifts, silently repointing already-stored data at a different name. Assigning every member
an explicit value pins the mapping so a later edit to the list cannot move it.

This is an opinionated, house-style rule. Many enums are never persisted and read perfectly well as an
ordered list, so it is **disabled by default** and opt-in through `.editorconfig`.

## Opting in

```ini
[*.cs]
dotnet_diagnostic.SST2331.severity = warning
```

## What is reported

- An enum with one or more members that have no explicit initializer. The diagnostic is reported once, on the
  enum name.

## What is not reported

- An enum whose every member has an explicit value.
- An empty enum: it has no member to number.

## How to fix violations

The code fix fills each member that lacks an initializer with the value the compiler currently gives it, so no
runtime value changes:

```diff
 public enum Color
 {
-    Red,
-    Green,
-    Blue,
+    Red = 0,
+    Green = 1,
+    Blue = 2,
 }
```

Members that already carry an explicit value are left untouched.

## Performance

The check is purely syntactic — a scan of the enum's members for a missing initializer — and reads no semantic
model on the clean path.
