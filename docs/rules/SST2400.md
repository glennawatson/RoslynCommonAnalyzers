# SST2400 — Arguments should be passed in the parameter's order

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

An argument's name matches a parameter of the call — but a different one, and the argument in *that*
position names *this* one. The two have been transposed.

```csharp
public void Copy(string source, string target)
{
}

public void Run(string source, string target)
{
    Copy(target, source); // SST2400
}
```

## Rule description

The types line up, so nothing stops it: the call compiles, runs, and copies the wrong way round. This is the
mistake a pair of same-typed parameters invites, and it is nearly invisible in review — the line reads
perfectly well, because both names are right there and both are spelled correctly.

Being wrong about this would be worse than saying nothing, since the fix reorders a call that currently
works. So the rule only reports a genuine transposition, and defines that as narrowly as it can. Every one of
the following must hold:

- **Every argument is positional.** One named argument anywhere in the list and the order is already explicit
  at the call site; there is nothing to be mistaken about.
- **The argument is a bare identifier.** Its name is the only thing being read, so it has to be a name and
  nothing else. `Copy(GetTarget(), source)` is not measured.
- **The identifier names a parameter the call really has**, at a different position.
- **The argument in that position names this parameter.** A two-cycle, not a chain. A rotation —
  `Take(second, third, first)` against `(first, second, third)` — has no unambiguous repair, and reading it as
  a mistake would be a guess.
- **The two parameters have the same type and ref kind.** This is what makes the report safe to act on: a
  reordered call binds to the same method it did before and still compiles. It is also the only shape that
  could have gone wrong silently — had the two parameters differed in type, the transposed call would not have
  compiled in the first place.

Case is ignored when comparing a name with a parameter's, so a `Source` property or a `Target` local still
reads as the parameter it is named after. A `params` tail ends the analysis: an argument's position no longer
maps to a parameter's, so a name landing in the "wrong" slot proves nothing.

## How to fix violations

Put the two arguments back in the parameters' order.

```csharp
Copy(source, target);
```

The code fix does exactly that, and nothing else: only the two reported positions move, and only their
expressions do. Each argument keeps the trivia around it, so a call spread over several lines keeps its shape
and only the two names change places.

If the call really was meant to be that way round — a deliberate transpose, a rotation of coordinates — then
the names are lying about it, and the fix is to say so at the call site:

```csharp
Copy(source: target, target: source);
```

A named argument silences the rule, because the order is then explicit.

## Performance

The clean path is one scan of the argument list, counting bare identifiers and looking for a named argument.
A call with fewer than two bare-identifier arguments never reaches the semantic model, which is most call
sites; only the ones that could be a transposition are bound.
