# SST2401 — Do not catch NullReferenceException

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `catch` targets `NullReferenceException`, either by naming it in the clause or by reaching it through an
exception filter.

```csharp
try
{
    Process(order.Customer.Address);
}
catch (NullReferenceException) // SST2401
{
    return false;
}
```

## Rule description

A `NullReferenceException` is not a condition to recover from. It is the runtime reporting that the code
dereferenced something it never checked — a bug, in this method or in the one that handed it the value.

Catching it converts a crash that comes with a stack trace pointing straight at the mistake into silent,
arbitrary behavior. And it does not catch *one* mistake: it catches every future instance of the same
mistake, anywhere inside the `try` block, including the ones nobody has written yet. The next null
dereference three call levels down, in code added a year from now, disappears into this handler too.

The filter form is the same handler written the long way, so the rule follows the name into the filter:

```csharp
catch (Exception error) when (error is NullReferenceException) // SST2401
```

A type of the project's own that happens to be called `NullReferenceException` is not reported — only the
framework's.

## How to fix violations

Guard the value instead of catching the failure. Which value, and what to do when it is null, is the question
the exception was trying to ask:

```csharp
if (order.Customer?.Address is not { } address)
{
    return false;
}

Process(address);
```

There is **no code fix**. The fix is to check the thing that was null, and an analyzer cannot know which of
the values inside the `try` block that was — the exception does not say, which is precisely the problem with
catching it. Removing the handler and leaving the crash is a legitimate answer too; automating that would be
deleting error handling, which is not something a fix should do quietly.

Where a null really is expected and legitimate, catching is still the wrong tool: `?.`, a null check, or a
`TryGet` pattern all say so at the point where it matters, and cost nothing.

## Performance

The clean path is a token-text comparison. A clause whose type is not spelled `NullReferenceException`, and a
filter that does not mention the name anywhere, are both rejected before the semantic model is touched. The
bind that follows only confirms the name really is the framework's type.
