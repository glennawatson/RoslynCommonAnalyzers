# SST2402 — Constructors should not write to static fields

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An **instance** constructor assigns a **static** field of its own type.

```csharp
public sealed class Connection
{
    public static string Endpoint = string.Empty;

    public Connection(string endpoint)
    {
        Endpoint = endpoint; // SST2402
    }
}
```

## Rule description

An instance constructor runs once per object; a static field exists once per type. Assigning one from the
other means the last object constructed silently redefines the field for every object that already exists —
and every object that reads it afterwards. Two connections, and the first one's endpoint is gone.

In a threaded program it is worse than "the last one wins": *which* one wins is a race, and the field can be
read half-written by a thread that never saw either constructor run.

If the value belongs to the type, set it where the type is initialized — a static constructor or a field
initializer, which run once, before anything can observe the field. If it belongs to the object, it is an
instance field.

### What is not reported

Four shapes mean something other than "the newest object redefines the field", and none of them is reported:

- **A static constructor.** That is the correct place to set static state.
- **A compound assignment or an increment** — `Created++`, `_total += cost`. It accumulates rather than
  overwrites. An instance counter is a deliberate design (a racy one, but deliberate), and the value the last
  constructor wrote is not the whole story.
- **A lazily-initialised guard.** `Instance ??= this;`, and any assignment whose enclosing `if` tests the same
  field, are the author saying "only the first one wins" — the opposite of the bug this rule describes.
- **A `[ThreadStatic]` field**, which is per-thread state deliberately kept off the instance.

A write inside a lambda or a local function declared in the constructor is not reported either: that code runs
when the delegate is invoked, not while the object is being built.

## How to fix violations

Decide who the value belongs to.

```csharp
public sealed class Connection
{
    // Per type: set once, where the type is initialized.
    public static string Endpoint { get; } = LoadDefaultEndpoint();

    // Per object: the constructor's business.
    public Connection(string endpoint) => _endpoint = endpoint;

    private readonly string _endpoint;
}
```

There is **no code fix**. The two repairs — moving the value onto the instance, and moving the assignment into
a static initializer — mean opposite things, and the choice between them is the design decision the diagnostic
is asking you to make. Either one changes what other code sees.

## Performance

The clean path binds nothing. The constructor's assignments are found on syntax; the containing type is
resolved only once one is present; and an assignment target is bound only after its name has matched a static,
writable field the type actually declares — which the `this._value = value` of an ordinary constructor never
does.
