# SST2402 — Instance members should not write to static fields

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An **instance** member — a constructor, method, finalizer, or a property, indexer, or event accessor —
assigns a **static** field of its own type.

```csharp
public sealed class Connection
{
    public static string Endpoint = string.Empty;

    public Connection(string endpoint)
    {
        Endpoint = endpoint; // SST2402
    }

    public void Reconnect(string endpoint)
    {
        Endpoint = endpoint; // SST2402
    }

    public string Address
    {
        get => Endpoint;
        set => Endpoint = value; // SST2402
    }
}
```

## Rule description

An instance member acts for one object; a static field exists once per type. Assigning one from the other
means the last instance to run silently redefines the field for every object that already exists — and
every object that reads it afterwards. Two connections, and the first one's endpoint is gone.

It does not matter which member does the writing. A constructor makes the newest object win; a method or a
setter makes the most recently *used* object win; a lambda or local function the member creates carries the
same write into whenever the delegate runs. In a threaded program it is worse than "the last one wins":
*which* one wins is a race, and the field can be read half-written by a thread that never saw any of them
run.

If the value belongs to the type, set it where the type is initialized — a static constructor, a field
initializer, or a static member, which act for the type rather than for one instance. If it belongs to the
object, it is an instance field.

### What is not reported

Four shapes mean something other than "one instance redefines the field for all the others", and none of
them is reported:

- **A static member.** A static constructor, method, or accessor is the correct place to set static state:
  it acts for the type, not for one instance.
- **A compound assignment or an increment** — `Created++`, `_total += cost`. It accumulates rather than
  overwrites. An instance counter is a deliberate design (a racy one, but deliberate), and the value the
  last instance wrote is not the whole story.
- **A lazily-initialised guard.** `Instance ??= this;`, and any assignment whose enclosing `if` tests the
  same field, are the author saying "only the first one wins" — the opposite of the bug this rule
  describes.
- **A `[ThreadStatic]` field**, which is per-thread state deliberately kept off the instance.

A write inside a lambda or a local function declared in a **constructor** is not reported either: that code
runs when the delegate is invoked, not while the object is being built. In any other instance member the
same write *is* reported — the delegate is part of how that member mutates type-wide state, whenever it
runs.

## How to fix violations

Decide who the value belongs to.

```csharp
public sealed class Connection
{
    // Per type: set once, where the type is initialized.
    public static string Endpoint { get; } = LoadDefaultEndpoint();

    // Per object: the instance's business.
    public Connection(string endpoint) => _endpoint = endpoint;

    public void Reconnect(string endpoint) => _endpoint = endpoint;

    private string _endpoint;
}
```

There is **no code fix**. The two repairs — moving the value onto the instance, and moving the assignment into
a static initializer — mean opposite things, and the choice between them is the design decision the diagnostic
is asking you to make. Either one changes what other code sees.

## Performance

The clean path binds nothing. A member's assignments are found on syntax; the containing type is resolved
only once one is present; and an assignment target is bound only after its name has matched a static,
writable field the type actually declares — which the `this._value = value` of an ordinary member never
does.
