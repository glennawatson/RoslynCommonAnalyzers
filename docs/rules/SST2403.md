# SST2403 — Do not let 'this' escape from a constructor

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Inside a constructor, `this` is passed as an argument, stored somewhere that outlives the object, or carried
out inside a closure subscribed to somebody else's event.

```csharp
public sealed class Widget
{
    private readonly string _name;

    public Widget(Registry registry, Service service)
    {
        registry.Add(this);                              // SST2403
        service.Updated += (sender, args) => Redraw();   // SST2403
        _name = "widget";
    }
}
```

## Rule description

Handing `this` to anything before the constructor returns publishes an object whose fields are not all set.
`_name` above is still null when `registry.Add` runs, and it is `readonly`, so the receiver reads a value the
type says can never be there. Field initializers of a derived class have not run at all yet.

If the receiver stores the reference somewhere another thread can see, that thread can use the object while it
is still being built — and no amount of locking inside the type can help, because the object was published
before its own constructor finished.

The lambda counts, and is the form this bug usually takes. `service.Updated += (s, e) => Redraw();` hands the
closure — and the half-built object inside it — to whatever holds that event, which is free to raise it
immediately.

### What is not reported

- **Member access on itself** — `this.Field = value`, `this.Method()`, `this[i]`. The object is talking to
  itself; nothing leaves.
- **Storing it in the object's own instance state**, or in a local. Neither reference outlives the object it
  points at.
- **Subscribing to the object's own event.** The delegate goes into the object's own field.
- **A constructor initializer.** The language does not allow `this` there at all, so the question does not
  arise.
- **A `this` inside a local function.** A local function is not a closure until it is converted to a delegate,
  and the conversion — not the declaration — is where the object would escape. That is a chain this rule does
  not follow.

Only a written `this` is seen. A lambda that captures the object through an unqualified member name
(`() => Redraw()` rather than `() => this.Redraw()`) captures it just as surely, but there is no `this` in the
source to point at, so the rule stays silent rather than report a location that is not there.

## How to fix violations

Publish the instance *after* construction, from a factory or an explicit initialization step:

```csharp
public sealed class Widget
{
    private Widget(string name) => _name = name;

    public static Widget Create(Registry registry, Service service)
    {
        var widget = new Widget("widget");
        registry.Add(widget);
        service.Updated += (sender, args) => widget.Redraw();
        return widget;
    }
}
```

There is **no code fix**. Moving the escaping call out of the constructor means deciding *where* it goes — a
factory, an `Initialize` method the caller must remember, a two-phase build — and every one of those changes
the type's public contract and every call site that constructs it. That is a design decision, and an
automated rewrite would be a silent API change with a good chance of being wrong.

## Performance

The clean path binds nothing. Each `this` is classified by walking up from it on syntax alone, and an
assignment target is bound only once `this` is known to be what is being stored. A constructor with no `this`
in it at all costs a single walk that finds nothing.
