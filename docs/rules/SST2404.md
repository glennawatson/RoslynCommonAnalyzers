# SST2404 — Validate an iterator's arguments before it starts yielding

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A method containing `yield` opens with a guard that checks its arguments.

```csharp
public IEnumerable<int> Doubled(int[] values) // SST2404
{
    if (values is null)
    {
        throw new ArgumentNullException(nameof(values));
    }

    foreach (var value in values)
    {
        yield return value * 2;
    }
}
```

## Rule description

The body of a method containing `yield` does not run when the method is called. Calling it builds a state
machine and returns; the body starts on the first `MoveNext`. So the guard at the top does not fire at the
call site — it fires later, from inside whatever `foreach` eventually consumed the sequence, on a stack that
no longer says who passed the bad value.

```csharp
var sequence = Doubled(null); // no exception here
// ... the sequence is stored, passed around, returned ...
foreach (var value in sequence) // ArgumentNullException here, and this frame is innocent
{
}
```

The guard is not wrong. It is in the wrong method.

### What is reported

The guard has to check an argument: an iterator that opens by checking its own state, and nothing it was
handed, is doing something this rule has no opinion about. Once one argument check is there, though, every
leading guard is part of the report — a disposed check standing in front of a null check is stranded behind
the same `MoveNext`, and splitting the method has to take both with it.

Both guard forms are recognized: an `if` that throws, and a runtime throw-helper call
(`ArgumentNullException.ThrowIfNull(values)`).

## How to fix violations

Split the method in two: a normal method that validates and returns the iterator, and a private iterator that
does the yielding.

```csharp
public IEnumerable<int> Doubled(int[] values)
{
    if (values is null)
    {
        throw new ArgumentNullException(nameof(values));
    }

    return Iterator();

    IEnumerable<int> Iterator()
    {
        foreach (var value in values)
        {
            yield return value * 2;
        }
    }
}
```

The outer method is no longer an iterator — its `yield` statements have all moved inside the local function —
so its body runs when it is called, and the guard throws at the call site, where the caller can see it.

The code fix does this rewrite. The local function needs no parameters: it closes over the method's, which is
why the guards can move without any of the arguments moving with them. Type parameters are in scope for the
same reason, so a generic iterator splits without a signature change. If the name `Iterator` is already taken
in the method, the next free one is used.

### The one case that is reported but not fixed

An **`async` iterator** is reported and left alone. Its cancellation token is bound to the iterator by
`[EnumeratorCancellation]` on the parameter; moving the `async` body into a local function would leave the
attribute on a parameter of a method that is no longer the iterator, and the token would quietly stop being
honoured. Splitting one of those correctly means deciding how the token is threaded through, which is the
author's call, not the fix's — so the fix does not offer itself, and the split is done by hand:

```csharp
public IAsyncEnumerable<int> DoubledAsync(int[] values, CancellationToken cancellationToken)
{
    ArgumentNullException.ThrowIfNull(values);
    return Iterator(values, cancellationToken);

    static async IAsyncEnumerable<int> Iterator(
        int[] values,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // ...
    }
}
```

## Performance

The clean path is a look at the body's first statement. Only a method that opens with a guard pays for the
walk that looks for a `yield`, and nothing in the rule binds a symbol at all: an iterator is a syntactic fact,
and so is a guard.
