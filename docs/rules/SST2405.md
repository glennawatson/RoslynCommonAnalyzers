# SST2405 — DebuggerDisplay should reference members that exist

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `[DebuggerDisplay]` string names a member the type does not declare, and does not inherit.

```csharp
[DebuggerDisplay("Order {Total}")] // SST2405 — the type has Amount, not Total
public sealed class Order
{
    public decimal Amount { get; }
}
```

## Rule description

The expression inside the braces is resolved by the debugger, not by the compiler. A typo, or a member that
was renamed six months ago, survives the build happily and shows up as an error string in the watch window —
at exactly the moment someone is trying to debug something else. Nothing else in the file is checked less by
the toolchain and read more under pressure.

The rule parses the braces out of the display string and checks each name against the type and every base
type, so an inherited member is found and stays silent.

### What is checked

Only an expression the rule can be sure about:

- a bare member name — `{Amount}`
- with a format specifier — `{Amount,nq}`
- a called member — `{Describe()}`, `{Describe(),nq}`

Anything else is left alone: a chain (`{Items.Count}`), an indexer (`{Items[0]}`), arithmetic. A name inside
one of those may be resolved against something other than this type, and a diagnostic that is only probably
right is worse than none.

The attribute is only checked on a **type declaration**. On a field or a property it resolves against that
member's type, which is a different question.

## How to fix violations

Name a member the type has.

```csharp
[DebuggerDisplay("Order {Amount}")]
public sealed class Order
{
    public decimal Amount { get; }
}
```

There is **no code fix**. The rule knows the name is wrong; it does not know which member you meant. Offering
the nearest-spelled member would be guessing, and a display string that shows the wrong value is worse than
one that shows an error — at least the error tells you to look.

## Performance

The clean path is one string comparison against the attribute's name. Only a `[DebuggerDisplay]` is parsed,
and the parse walks the literal's own text once, which is also how the diagnostic lands on the member name
inside the string rather than on the whole of it.
