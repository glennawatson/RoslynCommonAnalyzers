# SST2406 — A loop's stop condition should be able to change

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `while` or `for` condition reads only variables that nothing in the loop ever writes.

```csharp
var index = 0;
while (index < limit) // SST2406
{
    Console.WriteLine(items[index]);
}
```

## Rule description

The condition has already decided its answer before the first iteration. Either the loop never runs, or it
never stops — and the variable that was supposed to advance is the one that was forgotten. In the example
above, `index++` is missing, so the loop spins forever printing the same element.

Being wrong here is easy, so the rule refuses to guess. It stays silent whenever anything could make the
condition change, or could end the loop without it changing:

- **A `break`, `return`, `throw`, `goto` or `yield break` anywhere in the body.** That is a real way out, and
  a condition that never changes is then a deliberate one — `while (true)` written the long way. Even a jump
  that belongs to a nested loop or switch is enough to stop the report: the rule would rather miss a spinning
  loop than accuse a working one.
- **A condition that is not built purely from locals, literals and operators.** A call may return something
  different each time. A field or a property may be written by another thread, or by something the loop calls
  — `while (_running)` is a perfectly good way to wait for another thread, and reading a property is a call.
  An element or a member access may be anything at all.
- **A lambda or a local function in the body**, which could write the variable somewhere the scan cannot
  follow — as could a `ref` alias to it.
- **A variable the body writes**, in any of the ways a variable is written: an assignment, a compound
  assignment, an increment or decrement, or being passed by `ref`, `out` or `in`. A `for` loop's incrementors
  count as part of the loop.

What is left is a condition over locals and parameters that the loop demonstrably never assigns, never
increments and never passes by reference. Nothing else can change a local, so the condition is fixed.

`while (true)` is not reported: it reads no variables, and it says what it means.

## How to fix violations

Advance the thing the condition reads, or stop reading a thing that cannot change.

```csharp
var index = 0;
while (index < limit)
{
    Console.WriteLine(items[index]);
    index++;
}
```

There is **no code fix**. The rule can prove the loop is stuck; it cannot know what was meant to move it. An
increment is the usual answer, but so is a `break`, a different condition, or the realization that the loop
should not be there at all — and inserting the wrong one would turn an obvious hang into a subtle wrong
answer.

## Performance

The clean path never binds. The condition's shape rejects most loops outright — anything with a call, a
member access or an element access in it — and a loop whose body writes a condition variable, which is nearly
all of them, is rejected on syntax before the locals are resolved.
