# SST2407 — Declared events should be raised

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A field-like event is declared, and nothing in the compilation ever raises it.

```csharp
public sealed class Service
{
    public event EventHandler? Started; // SST2407

    public void Start() => _running = true; // the event is never raised
}
```

## Rule description

An event nobody raises is a promise the type never keeps. Callers subscribe, the handler never runs, and there
is nothing to see at the point of failure — the bug is the *absence* of code, which is the hardest kind to
notice. The subscriber looks correct, the event exists, the type compiles.

Only the declaring type can raise a field-like event: outside it, the language allows nothing but `+=` and
`-=`. So an event with no raise anywhere is one the type has forgotten to fire.

### What is not reported

Events that cannot be raised where they are declared:

- an **`abstract`** or **`extern`** event, and an event in an **interface** — there is no body to raise it in;
- an **`override`**, and an event **implementing an interface member** — the declaration exists because
  something else says it must;
- a **custom event** with `add`/`remove` accessors, whose raise goes through whatever backing store the
  accessors chose, under a name of its own;
- an event whose backing field carries attributes (`[field: …]`), for the same reason.

### What silences it

The rule errs towards silence. An event is reported only when its name appears **nowhere** in the compilation
except on the left of a `+=` or a `-=` — that is, nowhere it could be used as a value. So an event raised
through a copy of the delegate is not reported:

```csharp
var handler = Started;   // Started is used as a value here
handler?.Invoke(this, EventArgs.Empty);
```

and neither is one whose name is shared with an unrelated member somewhere else in the compilation. That is
the trade this rule makes deliberately: it would rather miss a never-raised event than accuse one that is
raised.

The diagnostic is reported on the event's own declaration, live in the editor, like any other local
diagnostic — the whole-compilation knowledge it needs is gathered separately and shared by every event in the
compilation.

## How to fix violations

Raise it, or remove it.

```csharp
public sealed class Service
{
    public event EventHandler? Started;

    public void Start()
    {
        _running = true;
        Started?.Invoke(this, EventArgs.Empty);
    }
}
```

There is **no code fix**. Where the event should be raised is the whole question — it is the line of code that
was never written, and only its author knows which method was supposed to end with it. Deleting the event
instead is just as plausible a repair, and just as much a decision.

## Performance

The compilation's names are indexed once, on the first event that survives the cheap checks, and shared by
every event after it. The index is purely syntactic: binding every name in the compilation to decide whether
it is *this* event would cost far more than the rule is worth, and the answer would only ever make the rule
report more — which is the direction it does not want to be wrong in.
