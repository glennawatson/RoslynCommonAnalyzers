# SST2408 — A StringBuilder that is filled should be read

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A local `StringBuilder` is appended to, and its contents are never read.

```csharp
public void Report(string[] lines)
{
    var builder = new StringBuilder(); // SST2408
    foreach (var line in lines)
    {
        builder.AppendLine(line);
    }
    // the ToString that was supposed to collect it is missing
}
```

## Rule description

Every `Append` in the method did real work and then threw it away. The code looks like it builds a string and
does not, and nothing at runtime says so — no exception, no wrong value, just a report that never gets
written. It is usually the tail end of an edit: the `return builder.ToString();` went somewhere else, or the
method's `void` return type is the thing that is wrong.

### What counts as a read

Anything that could look at the contents, and the rule leans on that deliberately:

- calling anything that is not one of the mutating methods — `ToString()`, `CopyTo(…)`;
- reading a property — `builder.Length`;
- passing the builder to something, or returning it;
- interpolating it, or reassigning the local.

Only a **discarded** call to `Append`, `AppendLine`, `AppendFormat`, `AppendJoin`, `Insert`, `Remove`,
`Replace` or `Clear` leaves the builder unread — a whole statement whose value nobody takes, chained or not:

```csharp
builder.Append("name: ").Append(name).AppendLine(); // still no read
```

At least one of them has to be an append, or there is nothing in the builder to have lost.

### What is not measured

Only locals. A field or a property may be read by any other member of the type, and a builder handed in as a
parameter belongs to the caller.

## How to fix violations

Read the thing you built.

```csharp
public string Report(string[] lines)
{
    var builder = new StringBuilder();
    foreach (var line in lines)
    {
        builder.AppendLine(line);
    }

    return builder.ToString();
}
```

There is **no code fix**. The repair is to use the string — return it, write it, assign it — and where it was
supposed to go is exactly what the code does not say. Appending a `ToString()` call that nothing consumes
would silence the diagnostic without fixing anything, which is worse than leaving it.

## Performance

The clean path is a look at the declared type's name. Nothing binds unless the declaration says
`StringBuilder`, and the search for references only binds the names that already spell the local's — so a
builder that is obviously read costs one scan of the block it lives in.
