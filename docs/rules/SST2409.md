# SST2409 — Do not throw a general exception type

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

`throw new Exception(…)`, `throw new SystemException(…)` or `throw new ApplicationException(…)`.

```csharp
public Order Load(string path)
{
    if (!File.Exists(path))
    {
        throw new Exception("order file not found"); // SST2409
    }

    // ...
}
```

## Rule description

`Exception`, `SystemException` and `ApplicationException` say only that something went wrong. They give a
caller nothing to catch selectively: to handle *this* failure, it has to catch all of them — including the
ones it has never heard of. So the `catch` that was meant for a missing file also swallows the null
dereference three frames down, and a bug becomes a handled error.

The message is not a substitute. Nobody catches on a message.

Only the three general types themselves are reported, never a type derived from them: deriving from
`Exception` is exactly how a project names its own failures, and that is the fix, not the problem. A type of
the project's own that happens to be called `Exception` is not reported either — only the framework's.

Both forms are checked, the throw statement and the throw expression:

```csharp
var value = source ?? throw new Exception("missing"); // SST2409
```

## How to fix violations

Throw the type that names the failure. Most of the time the framework already has it:

```csharp
throw new FileNotFoundException("order file not found", path);
```

and where it does not, the project's own type is a few lines:

```csharp
public sealed class OrderFormatException : Exception
{
    public OrderFormatException(string message)
        : base(message)
    {
    }

    public OrderFormatException(string message, Exception innerException)
        : base(message, innerException)
    {
    }
}
```

There is **no code fix**. Which exception type is right is the entire content of the decision: `ArgumentException`,
`InvalidOperationException`, `FileNotFoundException` and a type of your own all mean different things to a
caller, and picking one for you would be putting words in the code's mouth. A rewrite that guessed would
compile, look fixed, and still be lying about what went wrong.

## Performance

The clean path is a comparison of the thrown type's name. Only one of the three names reaches the semantic
model, where the bind confirms it really is the framework's type.
