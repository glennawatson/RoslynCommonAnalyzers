# SST2410 — A created disposable should be disposed

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes (`using` declaration) |
| Default | Enabled |

## Cause

A local variable is initialized with a newly created `IDisposable` (or `IAsyncDisposable`), and
nothing ever disposes it.

```csharp
public void Save(byte[] bytes)
{
    var file = new FileStream("out.bin", FileMode.Create); // SST2410
    file.Write(bytes, 0, bytes.Length);
    // the handle stays open until the finalizer gets round to it, if it ever does
}
```

## Rule description

The method built the thing, kept the only reference to it, and returned without releasing it. Nothing
fails where the leak is: the cost arrives later, as a file that stays locked, a socket that is never
given back, a timer that keeps firing.

## What this rule is not

**This is a syntactic ownership check, not dataflow analysis.** It answers one question — *is this
value obviously never disposed and obviously never given to anybody who could dispose it?* — and it
deliberately goes quiet the moment the value is passed anywhere, because the receiver may now be the
owner. `new StreamReader(stream)` owns the stream it wraps; disposing the reader disposes the stream,
and a rule that did not know this would be wrong about it.

A local is reported **only** when every one of these holds:

- it is initialized with an object creation — `new Foo(…)` (a factory such as `File.OpenRead(…)` is
  not reported: what it hands back may be owned elsewhere);
- the created type is a **reference type** implementing `IDisposable` or `IAsyncDisposable`;
- it is not a `using` declaration, and not the subject of a `using`/`await using` statement;
- `Dispose()` / `DisposeAsync()` is never called on it;
- every reference to it is a plain member access on it — `stream.Write(…)`, `stream.Length`.

It says nothing as soon as the value:

| goes | example |
| --- | --- |
| out of the method | `return stream;` |
| into a field or a property | `_stream = stream;` |
| into another variable | `Stream other = stream;` |
| into any method | `Consume(stream);` |
| into any constructor | `new StreamReader(stream)` — the reader owns it now |
| into a collection | `list.Add(stream);` |
| out of an iterator | `yield return stream;` |
| into a closure | `Action a = () => stream.Write(…);` |

### Types that are never reported

- **Structs**, such as `CancellationTokenRegistration` — disposing a copy is not a decision this rule
  can make.
- **`Task`** (and `Task<T>`), which implements `IDisposable` but
  [should not be disposed](https://devblogs.microsoft.com/pfxteam/do-i-need-to-dispose-of-tasks/).

## How to fix violations

Make the local a `using` declaration, so it is disposed when the scope ends.

```csharp
public void Save(byte[] bytes)
{
    using var file = new FileStream("out.bin", FileMode.Create);
    file.Write(bytes, 0, bytes.Length);
}
```

For a type that implements `IAsyncDisposable`, use `await using`.

```csharp
await using var connection = new SqlConnection(connectionString);
```

The **code fix** turns the local into a `using` declaration, and into an `await using` declaration when
the type is asynchronously disposable and the body is `async` — where `await` would not compile, no fix
is offered rather than one that does not build. A declaration of several variables at once is left
alone, since a `using` declaration would cover all of them.

SST2410's only claim is that the value is not disposed **at all**. Which disposal syntax to prefer once
there *is* one belongs to [SST2236](SST2236.md) (a simple `using` declaration) and
[PSH1310](PSH1310.md) (in PerformanceSharp.Analyzers, `await using`) — neither of which fires on the
code this rule reports, because there is no `using` there yet.

## Replaces

the analyzer **the rule** ("IDisposables should be disposed"). Two differences worth knowing when you
retarget suppressions:

- **SST2410 sees more types.** the rule only tracks a fixed list of types (`FileStream`, `StreamReader`,
  `StreamWriter`, `TcpClient`, …) and says nothing about a `MemoryStream`, an `HttpClient`, or a
  disposable you wrote yourself. SST2410 reports any created reference type that implements the
  interface, so it can fire where the rule did not.
- **SST2410 sees fewer places.** the rule also tracks *fields*, and reports a local captured by a lambda.
  SST2410 covers locals only, and treats a capture as an escape.

## Performance

The clean path is syntactic: a local declaration with no `using` keyword whose initializer is an
object creation. Nothing binds until both hold, and the disposal interfaces are resolved once per
compilation. A local that is obviously handed away costs one scan of the block it lives in, stopped at
the first reference that escapes.
