# SST2411 — A for loop's counter should be advanced

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `for` loop introduces a counter, reads it in the stop condition, and then neither the incrementer nor the body
ever writes to it.

```csharp
for (var i = 0; i < limit;) // SST2411
{
    Console.WriteLine(items[i]);
}
```

## Rule description

The counter cannot move, so the condition keeps the answer it started with — the loop runs forever or not at
all. The usual cause is a forgotten step, or a step written against the wrong variable
(`for (var i = 0; i < 10; total++)`).

Because the mistake is a declared, tested counter that is never advanced, the report holds even when the body
can `break` out: the counter was still meant to drive the loop and does not. The rule stays silent where the
loop could legitimately be doing something else:

- **The counter is stepped somewhere** — in the incrementer or the body — so it does move.
- **The condition reads a field or property**, which something the loop calls, or another thread, could change;
  only a condition over locals and parameters is provably fixed.
- **A lambda, a local function, or a `ref` alias** in the body could advance the counter out of the scan's
  sight.

Where the counter only walks a collection, a `foreach`, a `for` over a span, or a range makes the intent
explicit and removes the counter that was left unadvanced.

## How to fix violations

Advance the counter, or restate the loop so no manual counter is needed.

```csharp
for (var i = 0; i < limit; i++)
{
    Console.WriteLine(items[i]);
}
```

There is **no code fix**. The rule can prove the counter never moves; it cannot know what was meant to move it —
an increment, a step against a different variable, a `foreach`, or a wholly different condition are all
plausible, and inserting the wrong one would turn an obvious hang into a subtle wrong answer.

## Performance

The clean path never binds. A well-formed stepping loop is rejected from the incrementer before any allocation,
so it pays only a token comparison; the locals are resolved only after the syntactic checks find a tested,
never-stepped counter.
