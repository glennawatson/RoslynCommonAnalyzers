# SST2413 — A for loop's body should be able to run

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

The counter starts at a compile-time constant and is compared against another compile-time constant, and the
comparison is already false before the first iteration.

```csharp
for (var i = 10; i < 5; i++) // SST2413
{
    Console.WriteLine(i);
}
```

## Rule description

The body is dead code — it never runs once. The step direction is consistent with the comparison, so the loop
is not stepping the wrong way; the two constants are simply on the wrong sides of each other, usually a swapped
start and bound. An inclusive bound that still folds false (`i = 5; i <= 4`) is reported the same way.

A step that disagrees with the comparison side is the separate spinning-loop shape and is reported there
instead. The rule stays silent where the emptiness is not a mistake:

- A **condition that reads a collection's count** (`i < items.Count`) is never reported: a collection that is
  empty today is not a bug.
- A **non-constant bound** cannot be folded, so nothing is claimed about it.

## How to fix violations

Put the start and bound on the correct sides.

```csharp
for (var i = 5; i < 10; i++)
{
    Console.WriteLine(i);
}
```

There is **no code fix**. Swapped constants have no single obvious repair — the start and the bound could be
exchanged, or the comparison could be the error — and picking the wrong one would turn an obvious no-op into a
subtle wrong answer.

## Performance

The constants are folded only after the cheap structural checks: a single constant-signed step, a relational
comparison naming the counter once, and a body that does not re-step the counter. A loop with a non-constant
start or bound never reaches the fold.
