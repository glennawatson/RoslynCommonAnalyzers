# SST2415 — A boolean guard should short-circuit

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A single `&` or `|` joins two boolean operands, and the right operand does real work — a call, an assignment, an
increment, an element access, an object creation.

```csharp
public bool M(bool ready) => ready & Compute(); // SST2415
```

## Rule description

The `&` and `|` operators evaluate both sides every time, while `&&` and `||` stop as soon as the left operand
decides the answer. When the left operand is a null test or a readiness check, it reads like a guard — but the
eager operator ignores it, so `Compute()` runs even when `ready` is false. Writing `&&` makes the guard hold.

This is the case where replacing the operator changes behaviour, because the right operand's work is exactly
what the guard was meant to prevent. It is separated from the tidy-up case (SST1468), where the right operand is
side-effect-free and switching to the short-circuiting form changes nothing observable.

An integer `&` or `|` is a bitwise operation, not a boolean guard, and is not reported. A candidate inside a
lambda converted to an expression tree is skipped, because the rewrite would change the produced tree shape.

## How to fix violations

Use the short-circuiting operator so the left operand can guard the right.

```csharp
public bool M(bool ready) => ready && Compute();
```

The code fix replaces `&` with `&&` and `|` with `||`, and nothing else.

## Performance

All boolean `&` / `|` operators are analyzed in a single walk. The two ids are the two arms of one
side-effect check the analyzer already computes for the operand, so choosing between them costs nothing extra.
