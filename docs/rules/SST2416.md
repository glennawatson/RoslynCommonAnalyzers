# SST2416 — A remainder test should account for negative values

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A remainder is compared against a non-zero constant on a signed operand — the classic `x % 2 == 1` odd-number
test.

```csharp
public bool M(int n) => n % 2 == 1; // SST2416
```

## Rule description

The remainder operator keeps the sign of the dividend, so `-3 % 2` is `-1`, not `1`. Comparing the remainder
against a non-zero constant therefore silently fails for every negative value: an odd negative number is never
reported as odd. The `!=` form (`n % 2 != 1`) has the same defect.

Comparing against zero (`n % 2 == 0`) is unaffected and is not reported. An operand that cannot be negative — a
`Length`, a `Count`, or an absolute value — is left alone, because the sign problem cannot arise there. An
unsigned operand is likewise safe.

The diagnostic reports on every target framework; only the suggested fix depends on what the framework offers.

## How to fix violations

Test in a way that is correct for negative values.

```csharp
public bool M(int n) => int.IsOddInteger(n);
```

The code fix promotes the generic-math parity helper — `int.IsOddInteger` for `== 1`, `int.IsEvenInteger` for
`!= 1` — where the operand's type provides it (.NET 7 and later). Where that helper is absent, it degrades to a
zero comparison:

```csharp
public bool M(int n) => n % 2 != 0;
```

## Performance

The clean path is a syntax-kind test: register on `==` / `!=` and bail unless one side is a remainder. Only then
are the two constants and the operand's type resolved. An operand that is provably non-negative is rejected on
the syntax, before binding.
