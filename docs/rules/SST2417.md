# SST2417 — An assignment should not read as a transposed operator

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

The `=` is written flush against a following unary `+`, `-`, or `!`, with a space after that operator.

```csharp
x =+ 1; // SST2417
```

## Rule description

The spacing mimics a compound assignment whose two characters were typed in the wrong order. `x =+ 1` assigns
the positive value `+1`, but reads as the `+=` the author almost certainly meant. The tokens for `x =+ 1` and
`x = +1` are the same and differ only in where the space falls, so the whitespace asymmetry is the whole signal.

The rule reports only that asymmetric spacing. It stays silent on:

- the deliberate unary form `x = +1`, where the space follows the `=`;
- a real compound assignment `x += 1`;
- the closed-up form `x =+1`, with no space after the sign.

The companion spacing rules stay silent on the reported span, so their fix does not close the gap and cement the
fake compound operator.

## How to fix violations

Decide which was meant and space it accordingly. The two readings compete and neither is preferred, so the code
fix offers both: close the gap into the compound operator, or open it so the unary intent is plain.

```csharp
x += 1; // read as a compound assignment
x = +1; // read as assigning the unary value
```

For a transposed `!`, only the unary reading is offered (`flag = !other`), since there is no `!=` compound
assignment.

## Performance

Detection is purely lexical — no semantic model. The clean path is a syntax-kind test on the assignment
followed by two trivia reads to check where the spaces fall.
