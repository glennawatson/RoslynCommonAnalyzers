# SST2418 — The result of an immutable value's method should be used

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method is called on an immutable value as a whole statement, and its return value is thrown away.

```csharp
public void M(DateTime date)
{
    date.AddDays(1); // SST2418
}
```

## Rule description

Because the receiver cannot mutate, the method's only effect is the value it returns. `date.AddDays(1);`
computes a new date and drops it, leaving `date` unchanged. The mistake comes from expecting the call to modify
the receiver in place, as it would on a mutable object.

The rule recognises the immutable receiver from its type rather than a fixed list of methods, so it covers a
readonly struct, a span or memory slice, a well-known immutable value type, a stateless numeric helper class,
and a user-defined `readonly record struct`:

```csharp
public readonly record struct Money(int Amount)
{
    public Money Add(Money other) => new Money(Amount + other.Amount);
}

money.Add(other); // SST2418
```

Shapes that a discarded-object or discarded-string diagnostic already reports are left alone: `string`
receivers, LINQ, `[Pure]`-marked methods, `TryParse`, and object creation. A void method, and a call whose
result is used, are not reported.

## How to fix violations

Use the value the call produces — store it, return it, or pass it on.

```csharp
date = date.AddDays(1);
```

There is **no code fix**. The rule can prove the result is discarded; it cannot know whether the value was meant
to be assigned back, handed to another call, or the statement removed altogether, and guessing wrong would hide
the defect rather than surface it.

## Performance

The clean path is two syntax-kind tests; nothing binds until a statement is a bare member call. The span and
memory types are resolved once at compilation start, not per call.
