# SST2419 — A collection operation should not be applied to itself

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A set or list operation is given the very collection it is called on.

```csharp
public void M(HashSet<int> set) => set.UnionWith(set); // SST2419
```

## Rule description

Depending on the operation, the result is a no-op, a constant, or a silent wipe:

- a set unioned or intersected with itself does not change;
- `ExceptWith` against itself clears the set;
- a subset, superset, or equality test against itself is always true;
- a list that adds or inserts its own range doubles its contents.

Each reads as though it does something, and almost always came from a copy-and-paste where the second argument
should have named a different collection. The rule works from the `ISet<T>` and `IList<T>` interfaces, so it
applies to every implementation, and reports only when the receiver and the argument are the same
side-effect-free expression. A call-valued receiver (`Get().UnionWith(Get())`) may return two different
collections and is left alone.

## How to fix violations

Pass the collection that was actually meant.

```csharp
public void M(HashSet<int> a, HashSet<int> b) => a.UnionWith(b);
```

There is **no code fix**. Whether the wrong argument or the doubling was the mistake is genuinely ambiguous, so
the repair is left to the author.

## Performance

The clean path is a switch on the method-name token that rejects essentially every call before the receiver's
type is resolved. The set and list interfaces are resolved once at compilation start.
