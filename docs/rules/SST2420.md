# SST2420 — An index-of test should not skip the first position

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A search that returns the index of a match has its result tested with `> 0`.

```csharp
public bool M(string s) => s.IndexOf('a') > 0; // SST2420
```

## Rule description

The search returns 0 for a match at the first element and -1 for no match, so `> 0` treats a match at position 0
as though nothing was found. The test that answers "was it found" is `>= 0` or `!= -1`; `> 0` quietly excludes
the first position. The habit comes from C, where a non-zero return meant success.

The rule reports both the `IndexOf(...) > 0` and reversed `0 < IndexOf(...)` shapes, on a string, an array, a
span, or a list. It leaves the correct forms alone:

- `>= 0` and `!= -1`, which include the first position;
- `>= 1`, which is the same expression written on purpose — a deliberate "found beyond the first position".

The diagnostic reports on every target framework; only the suggested fix depends on what the framework offers.

## How to fix violations

A membership test is clearest written as `Contains`:

```csharp
public bool M(string s) => s.Contains('a');
```

The code fix rewrites to `Contains` where a matching overload exists. Where it is absent — the
`string.Contains(char)` overload does not exist on netstandard2.0 or .NET Framework — it degrades to the
corrected comparison instead:

```csharp
public bool M(string s) => s.IndexOf('a') >= 0;
```

## Performance

The clean path is a syntax test: register on `>` / `<`, and bail unless one side is the literal `0` and the
other an `IndexOf` / `LastIndexOf` call. Only then is the method bound to confirm it is a real index search
returning `int` on a supported container. The `>= 1` form is not registered, so a deliberate test is never
touched.
