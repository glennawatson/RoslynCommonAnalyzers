# SST2423 - Do not return a disposable owned by a using

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes for a using declaration (transfer ownership) |
| Default | Enabled |

## Cause

A value owned by a `using` is returned out of the `using` scope.

```csharp
public Stream Open()
{
    using var stream = File.OpenRead("data.bin");
    return stream; // SST2423
}
```

## Rule description

A `using` disposes its value the moment control leaves the scope - including on the way out through a
`return`. Returning that value hands the caller an object that was disposed during the return, and the
failure surfaces later, somewhere else, as an `ObjectDisposedException` with a stack that points nowhere
near the cause.

The rule reports a local declared by a `using` declaration or a `using` statement (including
`await using`) that is returned - `return local;`, an element of `return (local, ...);`, or
`yield return local;`. Returning a member of the local (`return local.Value;`), or a different object,
is not reported.

## How to fix violations

Transfer ownership: drop the `using` and let the caller dispose the value. This is the opposite of the
instinct to wrap it in a `try`/`finally`.

```csharp
public Stream Open()
{
    return File.OpenRead("data.bin");
}
```

When the method is `async` and the value is `IAsyncDisposable`, the caller then disposes it with
`await using`.

The code fix drops the `using` from a `using` declaration. Unwrapping a `using` statement block is left
to the author.

## When not to use the rule

There is no case where returning a value the current scope disposes is correct, so the rule has no
routine exemption.
