# SST2424 — An override should not change a parameter's default value

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An `override` declares a parameter default that differs from the base: a different value, one the base does
not have, or dropping one the base has.

```csharp
public class Base
{
    public virtual int Go(int a, int b = 1) => b;
}

public class Derived : Base
{
    public override int Go(int a, int b = 2) => b; // SST2424
}
```

## Rule description

A default value is not stored on the method. It is chosen at the call site from the static type of the
receiver. So the same object receives different arguments depending on which reference type holds it:

```csharp
Derived d = new Derived();
d.Go(0);          // b is 2  (Derived's default)
((Base)d).Go(0);  // b is 1  (Base's default)
```

Nothing warns, and the two call sites quietly disagree. The reader has no way to see that `Go(0)` means two
different things.

## How to fix violations

Declare the same default the base does, so a call means the same thing whichever reference type holds the
object. There is no code fix, because restoring the base's default value or removing the derived one changes
behaviour and the author should choose deliberately.

If the intent was a different default for the derived type, that is a separate method, not an override of the
same one.

## Performance

The clean path is a single modifier check: a method with no `override` keyword is rejected before the
semantic model is touched. Only an override fetches its overridden method and walks the parameter lists once.
