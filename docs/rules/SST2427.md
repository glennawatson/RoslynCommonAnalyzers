# SST2427 - A derived overload should not hide a base overload with a more specific parameter

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A derived class declares an overload whose parameter is a base type of a same-named base-class overload's
parameter.

```csharp
public class Base
{
    public virtual void Handle(string message) { }
}

public class Derived : Base
{
    public void Handle(object message) { } // SST2427
}
```

## Rule description

Overload resolution does not search the whole inheritance hierarchy for the best match. It stops at the
first type, walking up from the most derived, that has an applicable member of that name. Once it finds one,
it never looks at the base class.

So when `Derived` declares `Handle(object)`, that overload is applicable for *every* argument the base
`Handle(string)` could take - a `string` is an `object` - and it lives on the more derived type. A call
through `Derived` binds to it and stops:

```csharp
new Derived().Handle("literal"); // binds Derived.Handle(object); Base.Handle(string) is unreachable here
```

The caller passed a `string`, expecting the `string` handler, and silently got the `object` one instead.
Usually that means a value boxed or widened and the specialised code skipped - a bug that compiles cleanly,
because the types line up, and is invisible in review.

The same trap applies when the derived parameter is a base *interface* of the base parameter (an
`IEnumerable<int>` overload hiding a `List<int>` one), and when the widening is a boxing conversion (an
`object` overload hiding an `int` one).

## How to fix

Pick the shape that matches your intent:

- **Add the specific overload too.** If the derived type should handle both, declare `Handle(string)` on it
  as well, so the specific call reaches specific code.
- **Make it an `override`.** If the derived method is meant to replace the base behaviour, it should share
  the base signature and override it, not introduce a wider one.
- **Rename it.** If the derived method is a genuinely different operation, give it a name that says so, and
  the base overload stops being shadowed.

If the hide is deliberate and understood, declaring the derived method `new` says so, and the rule leaves it
alone.

There is **no code fix**. Which of the above is correct depends on what the type is meant to do - the
analyzer cannot know whether you wanted to specialise, override, or rename - so the fix is left to the
author.

## What is reported

- An ordinary method on a class whose parameter, at some position, is a proper base class or base interface
  of a same-named, same-arity base method's parameter, with every other parameter equal or likewise more
  general.

## What is not reported

- An `override`, whose signature is fixed by the base and cannot hide it.
- A method declared `new`, where the author has acknowledged the hide.
- An overload whose parameters are the same as, or more specific than, the base overload's - it hides
  nothing a caller was relying on.
- An unrelated parameter type (`Handle(int)` against a base `Handle(string)`), where neither call reaches
  the other.

## Performance

The rule runs per named type. A class whose base is `object` is rejected immediately, and for every other
class the check starts from a declared method's own parameters and climbs the base chain only for a
same-named, same-arity method. An `override` is filtered from its symbol; the `new` modifier is read from
the declaration syntax only once an actual hiding relationship has been found, so the syntax is never
fetched on the clean path.
