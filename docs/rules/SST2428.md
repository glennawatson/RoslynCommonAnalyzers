# SST2428 — A static field initializer should not read a static field declared later

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A static field's initializer reads another static field of the same type that is declared later in the
source.

```csharp
public static class Endpoints
{
    public static readonly string Full = "root" + Suffix; // SST2428: Suffix is null here -> "root"
    public static readonly string Suffix = "/v1";
}
```

## Rule description

Static field initializers run in the textual order the fields are declared. When `Full` runs, `Suffix` has
not been assigned yet, so it still holds its default — `null` for a reference type, `0` for a number. The read
does not fail; it quietly uses the default and keeps the result forever. `Full` ends up as `"root"`, not
`"root/v1"`, and nothing at the point of use hints at why.

Only a real ordering hazard is reported. The read must bind to a `static`, non-`const` field of the *same*
type, declared after the field being initialized. Reading a field declared earlier is fine — it has already
run. A `const` is a compile-time value with no run order to get wrong. An instance field, or a static field of
a different type, is not this rule's concern.

When the two fields are declared in different files of the same `partial` type, the order the initializers run
in is not merely reversed — it is undefined. The read can see the default no matter how the two declarations
are arranged, and the diagnostic says so.

## How to fix violations

For fields in the same file, move the field being read above the field that reads it, so it runs first:

```csharp
public static class Endpoints
{
    public static readonly string Suffix = "/v1";
    public static readonly string Full = "root" + Suffix;
}
```

For fields split across parts of a `partial` type, reordering cannot help, because the order between parts is
undefined. Compute the value in a static constructor, where the order is explicit, or make the dependent field
lazy:

```csharp
public static class Endpoints
{
    public static readonly string Suffix = "/v1";
    public static readonly string Full;

    static Endpoints() => Full = "root" + Suffix;
}
```

There is no code fix: moving a declaration or introducing a static constructor changes the type's shape in a
way the author should choose.

## Performance

The clean path never touches the semantic model. A field that is not `static`, is `const`, or has no
initializer is rejected on syntax alone, which is nearly every field. Only a static field with an initializer
has its initializer's identifiers bound, and only to confirm which field a name really is.
