# SST2429 — This accessor never uses its value

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `set`, `init`, `add`, or `remove` accessor has a body that never mentions the contextual `value` parameter.

```csharp
public int Height
{
    get => _h;
    set => _height = _width; // SST2429: value is thrown away
}
```

## Rule description

Every write accessor is handed the incoming value through the contextual `value` parameter. A `set` or `init`
receives the assigned value; an `add` or `remove` receives the delegate being subscribed or removed. An
accessor whose body never reads `value` discards it: the caller's assignment or subscription runs, compiles,
and changes nothing the caller can observe.

The usual cause is a body that reads the wrong member — `_height = _width` instead of `_height = value` — so
the property looks writable but silently ignores every write.

A local named `value` cannot be declared inside such an accessor, so the absence of the `value` token is
conclusive on its own. Deliberately inert accessors are left alone:

- An empty body, `set { }`, is a recognised no-op.
- A body that only throws, `set => throw new NotSupportedException();`, is refusing the write on purpose.
- An auto-implemented accessor, `set;`, has no body and stores `value` for you.

## How to fix violations

Read `value` in the accessor's body — usually by assigning it to the field the accessor is meant to back:

```csharp
public int Height
{
    get => _h;
    set => _height = value;
}
```

If the accessor really should reject writes, make that explicit by throwing; if it should ignore them, an
empty body says so. There is no code fix, because the analyzer cannot know which member the discarded `value`
was meant to reach.

## Performance

The whole check is a token scan of the accessor body, with no semantic model. It stops as soon as it sees a
`value` token, and an auto-implemented or empty accessor is rejected before the scan even starts.
