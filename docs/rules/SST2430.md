# SST2430 — A serialization callback has the wrong signature

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method marked with `[OnSerializing]`, `[OnSerialized]`, `[OnDeserializing]`, or `[OnDeserialized]` does not
have the signature the serializer invokes.

```csharp
using System.Runtime.Serialization;

public class Order
{
    [OnDeserialized]
    private void AfterLoad(int version) // SST2430: wrong parameter type
    {
    }
}
```

## Rule description

The serializer calls a serialization callback only when the method has the exact shape it expects: a
non-generic, non-static method that returns `void` and takes a single `StreamingContext` parameter. A method
that deviates in any of those is silently skipped at run time. The state the callback was meant to fix up, or
the resource it was meant to reattach, is simply never touched, and there is no error to point at.

Each of these is reported, because the serializer will pass over all of them:

```csharp
[OnDeserialized] private void M(int x) { }                 // wrong parameter type
[OnSerializing] public static void M(StreamingContext c) { } // static
[OnDeserialized] private void M() { }                       // no parameter
[OnDeserialized] private int M(StreamingContext c) => 0;    // non-void
```

The rule is gated on the serialization attributes and `StreamingContext` existing in the compilation, so a
target framework without them is never flagged.

## How to fix violations

Give the callback the required signature: an instance method returning `void` with a single `StreamingContext`
parameter.

```csharp
[OnDeserialized]
private void AfterLoad(StreamingContext context)
{
}
```

There is no code fix, because correcting the signature can mean changing the parameter list, the return type,
or the `static` modifier, and the body may need to be adjusted to match.

## Performance

The four attributes and `StreamingContext` are resolved once per compilation; a compilation without them
registers nothing. Only a method that carries one of the attributes has its signature examined, so a type with
no serialization callbacks pays nothing.
