# SST2433 — A caller-info parameter must come last and have a default

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A parameter marked with a caller-info attribute (`[CallerMemberName]`, `[CallerFilePath]`,
`[CallerLineNumber]`, or `[CallerArgumentExpression]`) is followed by an ordinary parameter, or has no default
value.

```csharp
void Write([CallerMemberName] string caller = "", string message = "") { } // SST2433 on caller
```

## Rule description

A caller-info parameter is filled in by the compiler from the call site, so callers leave it out. That only
works when the parameter is last and optional.

Put an ordinary parameter after it and the call silently does the wrong thing:

```csharp
Write("hello"); // "hello" binds to caller; message keeps its default ""
```

The single positional argument lands in `caller`, because that is the first parameter, and `message` — the one
the caller meant to set — takes its default. The types line up, so nothing stops it.

A caller-info parameter with no default forces every caller to pass it explicitly, which is the one thing the
mechanism exists to avoid. The C# compiler already rejects that outright (a caller-info attribute may only be
applied to a parameter with a default value), so in practice the valuable, compiler-invisible case is the
ordering one above.

The rule reports on the offending parameter. Two or more caller-info parameters in a row at the end, each with
a default, are correct and are not reported.

## How to fix violations

Move the caller-info parameters to the end and give each a default:

```csharp
void Write(string message, [CallerMemberName] string caller = "") { }
```

There is **no code fix**. Reordering the parameters of an existing method silently re-maps every positional
call site — the very failure this rule is about — so the safe migration is to add a new overload with the
corrected order and leave the old signature (or mark it obsolete) rather than edit it in place.

## Performance

The four caller-info attributes are resolved once at compilation start; `[CallerArgumentExpression]` arrived in
.NET 6, so that clause runs only where the attribute exists and the other three keep working when it does not.
The rule is a symbol action gated on a method that actually declares a parameter carrying an attribute, so a
method with no attributed parameters costs a single length check.
