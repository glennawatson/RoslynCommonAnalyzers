# SST2437 — Avoid recursive generic inheritance

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A generic type inherits from a base type that instantiates the type with a type argument that again
contains the type.

```csharp
public class Base<T>
{
}

public class Recursive<T> : Base<Recursive<Recursive<T>>> // SST2437
{
}
```

## Rule description

When a type is nested inside its own base's type arguments, the runtime has to build an ever-larger chain
of constructed types to load it. The source compiles clean and reports nothing, and then the type throws a
`TypeLoadException` the moment it is loaded. Any tool that walks the constructed type graph of the type runs
out of stack on the same shape.

The self-referential base used for fluent builders and comparable types is depth one and is legitimate, so
it is not reported:

```csharp
public class Fluent<T> : Base<Fluent<T>> // fine
{
}
```

Only a type nested inside its own base's type arguments (a `Recursive<Recursive<...>>`) is reported. A type
that appears inside a different generic, but not inside its own arguments, is also fine.

## How to fix violations

Break the self-nesting. Pass the type parameter itself, or a fixed type, as the base's argument instead of
re-wrapping the declaring type:

```csharp
public class Recursive<T> : Base<T>
{
}
```

There is no code fix; the correct base type is a design decision.

## Performance

Detection is purely syntactic and scans only the base list's type-argument syntax, which is a small finite
tree. The rule never asks the semantic model to expand the base type, so it cannot itself run out of stack on
the shape it reports.
