# SST2438 - A caught exception should be passed to the logger

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A catch block logs at error or critical level but never hands the caught exception to the logger.

```csharp
try
{
    Process(order);
}
catch (Exception ex)
{
    logger.LogError("failed to process the order"); // SST2438: ex is thrown away
}
```

## Rule description

Every logging method has an overload that takes the exception as a dedicated argument, and only that argument
is projected into the exception fields a structured sink reads - the type, the stack trace, the inner
exceptions. When a catch logs at error or critical level without passing the exception, all of that is lost:
the record says something failed but not what, and the one piece of data that would let someone diagnose it is
gone.

Two shapes are reported:

- the exception is logged only through one of its properties, such as its message text - a degraded copy that
  drops everything else;
- the exception is not passed at all and nothing in the catch rethrows it.

To keep false positives out, the rule reports only when the exception is provably discarded. It stays silent
when:

- the level is below the configured floor (error by default);
- the catch declares no exception variable, so there is nothing to pass;
- the catch rethrows - the exception is not being swallowed;
- the log already passes the exception as its exception argument;
- the exception is used elsewhere in the catch, so it was not simply forgotten.

## How to fix violations

Pass the caught exception as the log's first argument. The code fix does this, dropping a redundant projection
and the placeholder it filled.

```csharp
catch (Exception ex)
{
    logger.LogError(ex, "failed to process the order");
}
```

## Configuration

```ini
# Rule-specific
stylesharp.SST2438.minimum_level = error
```

| Option | Values | Default | Effect |
| --- | --- | --- | --- |
| `minimum_level` | `trace`, `debug`, `information`, `warning`, `error`, `critical` | `error` | The lowest level at which a discarded exception is reported. |

Error is the default because a lost stack trace at error or critical level is a real operational problem, while
the same shape at lower levels is common and usually harmless. Lower the floor to opt the noisier levels in. An
unset or unrecognized value falls back to error.

## Performance

The clean path finds its catch by walking up parents from the call, bounded by the enclosing method or lambda,
and reads its options only once per file. The level check and the catch shape are settled before the exception
references are resolved, so a log outside a catch, or below the floor, costs almost nothing.
