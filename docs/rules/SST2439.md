# SST2439 - An exception should be passed as the exception argument

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

An exception is passed as one of a log message's values instead of as its exception argument.

```csharp
logger.LogError("processing failed for {Error}", ex); // SST2439
```

## Rule description

When an exception is passed as a template value, it is formatted into text and filed under a property name,
while the dedicated exception argument stays empty. A structured sink then has no stack trace, no exception
type, and no inner-exception chain to index, because those come only from the exception argument. The message
still reads plausibly, so the loss is easy to miss.

Every logging method offers an overload whose first argument is the exception. The rule reports when an
argument whose type derives from the exception base type is passed as a template value and that overload
exists. A value that is merely a projection of an exception - its message text, for instance - has type
`string`, not an exception type, and is not reported.

## How to fix violations

Move the exception into the exception argument. The code fix hoists it and removes the placeholder it filled.

```csharp
logger.LogError(ex, "processing failed");
```

## Performance

The template is bound only after a syntactic filter passes: the invoked member starts with `Log` or is
`BeginScope`, and one argument is a string literal. The exception check runs over the template's value
arguments only, and the whole family shares one bound call and one parsed template.
