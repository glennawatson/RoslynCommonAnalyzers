# SST2440 - Log values should follow the order of the template placeholders

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

Two values named after the template's placeholders sit in each other's slots.

```csharp
logger.LogInformation("{UserId} {UserName}", userName, userId); // SST2440
```

## Rule description

A message template binds its values to placeholders by position: the first value fills the first placeholder,
and so on. When two values are named after the placeholders but transposed, the call compiles and files each
value under the wrong property name. In a structured store this is worse than a cosmetic swap - every later
query keyed on `UserId` reads user names, and the data lies without ever looking wrong.

Only a genuine two-way swap is reported, and only when:

- each value is a bare name or a simple member access, so its name is the only thing being read;
- the placeholder names are all distinct;
- the name in each slot matches the other slot's placeholder - a two-cycle, not a longer rotation.

A three-way rotation has no unambiguous repair and is left alone. Values already in their own slots, computed
values, and templates with a repeated placeholder name are not reported here.

## How to fix violations

Swap the two values into the placeholders' order. The code fix moves only the two reported values.

```csharp
logger.LogInformation("{UserId} {UserName}", userId, userName);
```

## Performance

Nothing binds until the syntactic filter passes and the template is parsed. The swap search runs only when the
number of values equals the number of placeholders and every placeholder is a distinct name, which rules most
calls out before any value name is read.
