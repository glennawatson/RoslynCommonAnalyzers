# SST2442 - A message template should not repeat a placeholder name

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Two placeholders in one template share a name.

```csharp
logger.LogInformation("{Id} then {Id}", first, second); // SST2442
```

## Rule description

A structured sink keys the logged payload by placeholder name, so two placeholders that share a name collapse
to a single property: the last value wins and the earlier one is lost. The rendered message still reads
plausibly, which is why the loss goes unnoticed, and any query on that name returns only one of the values that
were meant to be recorded.

Names are compared without regard to case, because a case-only difference is the same key to most sinks and is
itself a bug. The second and each later use of a repeated name is reported. A purely numeric placeholder is
positional and is not treated as a name, so a reused index is not reported here.

Only whole-name equality counts: `{User}` and `{UserName}` are different names and are not a duplicate.

## How to fix violations

Give each placeholder a distinct name.

```csharp
logger.LogInformation("{FirstId} then {SecondId}", first, second);
```

There is no code fix, because renaming a placeholder is a semantic choice.

## Performance

Names are compared directly over the parsed template's offsets, with no substring allocated until a duplicate
is reported. The comparison runs only on the named placeholders the single template scan already found.
