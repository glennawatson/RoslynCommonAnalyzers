# SST2443 - A typed logger's category should be the type that logs through it

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A field, property, or parameter of type `ILogger<T>`, or a `CreateLogger<T>()` / `CreateLogger(typeof(T))`
call, names a type other than the enclosing one.

```csharp
public sealed class OrderService
{
    private readonly ILogger<PaymentService> _logger; // SST2443
}
```

## Rule description

A generic logger's type argument becomes the log category - the string that per-namespace level filters and
sink routes match against. When a type holds or creates a logger categorized as some other type, usually the
result of copying a field or a creation call, its own configured filters and routes silently do nothing, and
its entries appear under a category that has nothing to do with where they came from. The code runs and logs;
it is only the filtering that fails, which makes this one of the harder logging faults to notice.

The following are deliberate and are not reported:

- the category is a base type or an implemented interface of the enclosing type - a base logging on behalf of
  its derived types;
- the category is a dedicated marker: an empty type, or one whose name ends in `Category` or `Logs`;
- the enclosing type is nested inside the category;
- the enclosing type is generic and the category is its own constructed self.

## How to fix violations

Use the enclosing type as the category. The code fix rewrites it, carrying the type parameters when the
enclosing type is generic.

```csharp
private readonly ILogger<OrderService> _logger;
```

## Performance

The rule is gated on the generic logger type resolving in the compilation; a project that does not use it
registers no action. Each candidate is filtered on its written identifier before it is bound.
