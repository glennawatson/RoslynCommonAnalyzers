## Cause

A custom date/time format string contains an unquoted `/` or `:`, and it is formatted with a
culture-sensitive provider.

```csharp
public string Stamp(DateTime d) => d.ToString("dd/MM/yyyy", CultureInfo.CurrentCulture); // SST2445
```

## Rule description

In a custom date/time format string, an unquoted `/` does not mean a literal slash â€” it means "the current
culture's date separator". An unquoted `:` means "the current culture's time separator". So a format written
as `dd/MM/yyyy` renders with dots under one culture and dashes under another, and a value meant for a wire
format, a file name, or a database key silently changes shape the moment the process runs somewhere else.

## What is reported

The defect is reported only when a culture-sensitive provider is in play:

- an explicit current-culture provider (`CultureInfo.CurrentCulture` or `CultureInfo.CurrentUICulture`) on a
  call such as `ToString(format, provider)`, `ParseExact(...)`, or `TryParseExact(...)`;
- the implicit current culture of a plain interpolated string, `$"{d:dd/MM/yyyy}"`, that is used as a
  `string`.

It covers `DateTime`, `DateTimeOffset`, `DateOnly`, `TimeOnly`, and `TimeSpan`. The following are correct and
are never reported:

| shape | why it is silent |
| --- | --- |
| `d.ToString("dd/MM/yyyy", CultureInfo.InvariantCulture)` | the invariant culture fixes the separators |
| `d.ToString("dd/MM/yyyy")` | no provider is supplied â€” out of scope |
| `d.ToString("dd'/'MM'/'yyyy", CultureInfo.CurrentCulture)` | the separators are already quoted |
| `d.ToString("O", CultureInfo.CurrentCulture)` | a standard specifier carries no `/` or `:` |
| `d.ToString("dd.MM.yyyy", CultureInfo.CurrentCulture)` | `.` is a literal, not a culture separator |
| `string.Create(CultureInfo.InvariantCulture, $"{d:dd/MM/yyyy}")` | the interpolation is built invariantly |

## How to fix violations

For a fixed shape â€” a wire format, a file name, a key â€” quote the separator or pass the invariant culture. For
a value shown to a user, use a standard specifier such as `"d"` or `"g"`, which is already localized; a
hardcoded `dd/MM/yyyy` is the worst of both, being neither fixed nor localized.

Two **code fixes** are offered:

- *Quote the separators* wraps each unquoted `/` and `:` in single quotes, so the format keeps its shape under
  any culture. It applies to both the method-call and interpolated-string shapes.
- *Use the invariant culture* replaces the current-culture provider with `CultureInfo.InvariantCulture`. It
  applies to the method-call shape.

## Performance

The clean path is a token compare and a character scan of the literal for a `/` or `:`; nothing binds until a
candidate call or interpolation carries one. The date/time and provider types are resolved once per
compilation, and `DateOnly` and `TimeOnly` â€” absent before `.NET 6` â€” simply never match on an older target.
