# SST2446 — The number of bytes read from a stream should not be discarded

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes (`ReadExactlyAsync`), where the API exists |
| Default | Enabled |

## Cause

A stream's asynchronous read is awaited as a statement and the count it returns is thrown away, in one
of two shapes: a read awaited through a configured awaiter, or a read stored in a local that is then
awaited.

```csharp
public async Task Fill(Stream stream, Memory<byte> buffer)
{
    await stream.ReadAsync(buffer).ConfigureAwait(false); // SST2446: the count is discarded
}
```

## Rule description

A single asynchronous read is allowed to return fewer bytes than were asked for. A short read is normal
on a network stream, a pipe, or a file still being written, and the count the read returns is the only
signal of how much of the buffer was actually filled. Awaiting the read and discarding that count treats
every read as complete, so the buffer keeps stale or default bytes past the point the read reached, and
whatever was not read is silently lost or misparsed.

## What this rule reports

Only two shapes, both of which discard the count while looking like ordinary asynchronous code:

```csharp
// a read awaited through a configured awaiter
await stream.ReadAsync(buffer).ConfigureAwait(false);
await stream.ReadAsync(buffer).ConfigureAwait(true);

// a read stored in a local and then awaited as a statement
var read = stream.ReadAsync(buffer);
await read;
```

A read whose count is put to use is correct and is never reported:

| shape | example |
| --- | --- |
| the count is assigned | `int n = await stream.ReadAsync(buffer).ConfigureAwait(false);` |
| the count is returned | `return await stream.ReadAsync(buffer).ConfigureAwait(false);` |
| a plain awaited read | `await stream.ReadAsync(buffer);` |
| a write | `await stream.WriteAsync(buffer).ConfigureAwait(false);` |
| a read on something that is not a stream | a channel's own `ReadAsync` |

The rule is gated on `System.IO.Stream` resolving, and the reported call must bind to `Stream.ReadAsync`
or an override of it.

## How to fix violations

Where the framework offers it (`.NET 7` and later), read the buffer completely with `ReadExactlyAsync`,
which fills the buffer or throws — there is no count left to discard:

```csharp
await stream.ReadExactlyAsync(buffer).ConfigureAwait(false);
```

The **code fix** rewrites the configured-await shape to `ReadExactlyAsync`, and only where that method
resolves in the compilation. A read stored in a local is reported without a fix, because changing its
initializer would change the local's type — rewrite it by hand. On frameworks without `ReadExactlyAsync`,
loop until the buffer is filled, checking the count each time, or act on the returned count directly.

## Performance

The clean path is a syntactic prepass: the await must be an expression statement, and after unwrapping a
configured awaiter the read must be a `ReadAsync` call reached through that awaiter or a local. Nothing
binds until that holds, and the stream type is resolved once per compilation.
