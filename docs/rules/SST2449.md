# SST2449 — Unsubscribing with a new lambda removes nothing

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `-=` whose right operand is a lambda or anonymous method, applied to an event or to a delegate-typed
field, local, parameter, or property.

```csharp
public sealed class Watcher
{
    public void Start(FileSystemWatcher watcher)
    {
        watcher.Changed += (sender, args) => Reload();
    }

    public void Stop(FileSystemWatcher watcher)
    {
        watcher.Changed -= (sender, args) => Reload(); // SST2449
    }
}
```

## Rule description

Removing a handler with `-=` compares delegates for equality, and a lambda or anonymous method creates a
brand-new delegate every time it is evaluated — even one whose text matches the subscription character for
character. The two lambdas compile to two different methods (or two different closure instances), so they
never compare equal. The subtraction finds no match and removes nothing.

The code *looks* like a clean subscribe/unsubscribe pair, which is what makes the bug expensive: the
original handler keeps firing after the "unsubscribe", and the event's handler list keeps the handler's
target alive — a leak that grows with every subscribe/unsubscribe cycle.

The same delegate equality governs `-=` on a plain delegate value, so subtracting a lambda from a
delegate-typed field, local, parameter, or property is reported for the same reason.

### What is not reported

- **A right operand that is not a freshly created delegate** — a method group (`Saved -= OnSaved`), a
  stored delegate (`Saved -= _handler`), or any other expression. A method group compares equal by method
  and target, and a stored delegate can be the very reference that was added; whether either one matches is
  their business, not this rule's.
- **A left side that is not an event or delegate.** A custom subtraction operator that happens to accept a
  delegate argument is combining values, not managing a handler list.
- **A left side with no simple name**, such as a delegate array element. The described shape is an event or
  a delegate-typed field, local, parameter, or property.

## How to fix violations

Keep the delegate that was subscribed, and unsubscribe with that same reference.

```csharp
public sealed class Watcher
{
    private FileSystemEventHandler _onChanged;

    public void Start(FileSystemWatcher watcher)
    {
        _onChanged = (sender, args) => Reload();
        watcher.Changed += _onChanged;
    }

    public void Stop(FileSystemWatcher watcher)
    {
        watcher.Changed -= _onChanged;
    }
}
```

Or subscribe with a method group in the first place — `+= OnChanged` / `-= OnChanged` pairs match by method
and target with nothing to store.

There is **no code fix**. The repair is not a rewrite of the `-=` line: the delegate has to be captured at
the *subscription* site — in a field or local whose lifetime spans both operations — and where that state
lives is a design decision the diagnostic is asking you to make.

## Performance

The clean path binds nothing. The right operand's syntax kind is checked first, and almost every `-=` in
real code subtracts a number or a stored delegate, not a lambda. The left side is bound only after a lambda
or anonymous method is already on the right — which is the violation, minus the one custom-operator shape
the bind exists to excuse.
