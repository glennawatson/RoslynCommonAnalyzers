# SST2450 — Debug.Assert conditions must not have side effects

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

The condition argument of a `System.Diagnostics.Debug.Assert` call performs a side effect — an assignment, an
increment or decrement, or a call that changes a collection or advances an enumerator.

```csharp
public void M(List<int> items, int x) => Debug.Assert(items.Remove(x)); // SST2450
```

## Rule description

`Debug.Assert` carries `[Conditional("DEBUG")]`. In a release build the compiler removes the entire call —
condition included — so the condition is never evaluated there. Any state the condition changes therefore
happens only in debug builds and silently disappears in release, and the program behaves differently between
the two configurations. The classic form is a mutation whose result the author also wanted to check:

```csharp
Debug.Assert(set.Add(item));   // in release the item is never added
Debug.Assert(queue.TryDequeue(out var head)); // in release nothing is dequeued
Debug.Assert(index++ < length); // in release the index never advances
```

Move the work into its own statement, which runs in every build, and assert the result:

```csharp
var added = set.Add(item);
Debug.Assert(added);
```

## What this rule reports

Only shapes that are a side effect from syntax alone, so ordinary asserts stay quiet:

- an assignment (`=`, `+=`, `??=`, …) anywhere in the condition;
- a pre or post increment or decrement (`++`/`--`);
- a call to a well-known state-changing collection or enumerator member on a receiver — `Add`, `Remove`,
  `Pop`, `Dequeue`, `MoveNext`, and the `Try*` mutators (`TryAdd`, `TryRemove`, `TryTake`, `TryPop`,
  `TryPush`, `TryDequeue`, `TryUpdate`).

A condition that only reads is never reported:

| shape | example |
| --- | --- |
| a comparison or boolean operator | `Debug.Assert(count > 0 && items != null)` |
| an ordinary predicate call | `Debug.Assert(IsValid(x))` |
| a query call | `Debug.Assert(items.Contains(x))` |
| the idiomatic try-get shape | `Debug.Assert(map.TryGetValue(key, out var value) && value > 0)` |
| an `is`/pattern check | `Debug.Assert(o is int n && n > 0)` |
| a `nameof` | `Debug.Assert(nameof(value) != null)` |

The call is bound before anything is reported, so a `Trace.Assert` or a same-named `Assert` method of the
project's own is left alone; only `System.Diagnostics.Debug.Assert` — any overload — is reported.

Because purity is judged from the method name rather than the bound symbol, two shapes are deliberately not
reported: the pure `Add`/`Remove` of an immutable collection (which returns a new collection instead of
mutating), and a mutation reached through a conditional access (`list?.Remove(x)`). Both are rare inside an
assert, and staying silent keeps false positives near zero.

## How to fix violations

There is **no code fix**. Hoisting the side effect out of the condition, or dropping it, changes behaviour in a
way only the author can choose, so the repair is left by hand: run the side-effecting statement unconditionally
and assert its result.

## Performance

The clean path is syntax only. Every invocation is seen, but all but calls named `Assert` are rejected at once,
and the first argument's subtree is scanned with an allocation-free preorder walk. Nothing binds until a side
effect is found inside an `Assert`-named call, and the `Debug` type is resolved once per compilation.
