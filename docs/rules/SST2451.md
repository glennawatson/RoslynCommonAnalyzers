# SST2451 — A class only its own constructors can create must actually create itself

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Every constructor of a non-static, non-abstract class is private, and no member of the type ever
creates an instance.

```csharp
public sealed class Session // SST2451
{
    private readonly string _user;

    private Session(string user) => _user = user;

    public string User => _user;
}
```

## Rule description

Making every constructor private is a promise: the type takes sole responsibility for creating its
instances. That is how a singleton exposes `Instance`, how a factory method validates before it
constructs, how a set of well-known values stays closed.

When no member follows through with a `new` expression, the promise is never kept. The class still
compiles, its instance members still bind, callers can still declare variables of the type — but no
instance can ever exist, so none of that code can ever run. Usually a static `Create` or `Instance`
member was planned and never written, or the one call site that constructed the type was deleted and
left the rest behind.

The scan accepts any creation the type can reach: a static field or property initializer, a method,
a lambda or local function (a `Lazy<T>` initializer's `() => new Session(...)` counts), and the
members of nested types — the whole accessibility domain of a private constructor. A nested type
that *derives* from the class also counts, because constructing the subclass chains the private
constructor.

### What is not reported

- **A static or abstract class.** Neither expects direct instantiation; an abstract class with
  private constructors and nested subclasses is a deliberately closed hierarchy.
- **A partial class.** The creation may live in another part, in a file this rule does not see.
- **An open class with `private protected` constructors.** A derived class anywhere in the assembly
  can chain them. On a `sealed` class no derived class can exist, so the constructor collapses to
  private and the class is analyzed.
- **A constructor a framework calls by reflection.** Two shapes mark one: an attribute whose name
  ends in `Constructor` (the designated entry point of a deserializer or object mapper), and the
  classic `(SerializationInfo, StreamingContext)` deserialization signature. Such a type *is*
  created — just not by a `new` expression the type could contain.
- **A struct or a record.** A struct's default value always exists, and a record's primary
  constructor is part of its public shape.

Creation by reflection from outside the type — an activator, a dependency-injection container, a
test harness reaching for a private constructor — is invisible to a per-declaration analysis and out
of scope. A type built that way should mark the constructor it exposes with the framework's
designated attribute, or keep an accessible constructor.

## How to fix violations

Keep the promise, or stop making it.

```csharp
public sealed class Session
{
    private readonly string _user;

    private Session(string user) => _user = user;

    public string User => _user;

    // Either discharge the type's creation duty…
    public static Session Open(string user) => new Session(user);
}

// …or hand it back to the caller.
public sealed class Session
{
    public Session(string user) => _user = user;
}
```

There is **no code fix**. The two repairs — adding the static member that creates the instance, and
widening a constructor — change the type's public surface in opposite directions, and choosing
between them is the design decision the diagnostic is asking you to make.

## Performance

The clean path binds nothing. Constructor accessibility is decided from modifiers alone, and almost
every class leaves at its first non-private constructor. Only an all-private-constructor class walks
its declaration, and only a `new` expression or base-list entry whose written name could be the
class is bound to confirm the match.
