# SST2456 — A field-like event that overrides or hides an inherited event splits its subscriber list

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A field-like event — one with no `add`/`remove` accessors — is declared `override`, or `new` hiding an
inherited event.

```csharp
public class Publisher
{
    public virtual event EventHandler Changed;
}

public class LoudPublisher : Publisher
{
    public override event EventHandler Changed; // SST2456
}
```

## Rule description

A field-like event compiles to a hidden delegate field plus compiler-generated accessors that mutate it.
When a derived type redeclares the event with a field-like declaration of its own — `override`, or `new`
that hides an inherited event — the derived declaration gets a hidden backing field *of its own*. From that
point the hierarchy holds two subscriber lists:

```csharp
public class Publisher
{
    public virtual event EventHandler Changed;

    public void Raise() => Changed?.Invoke(this, EventArgs.Empty); // reads Publisher's field
}

public class LoudPublisher : Publisher
{
    public override event EventHandler Changed; // a second, unrelated field
}

var publisher = new LoudPublisher();
publisher.Changed += OnChanged;   // binds LoudPublisher's field
publisher.Raise();                // reads Publisher's field — OnChanged never runs
```

Which list a `+=` lands in depends on the static type it binds through, and which list a raise walks depends
on where the raising code lives. Handlers subscribed through one type are invisible to raises through the
other. Nothing throws and nothing is logged — the notification just silently disappears.

An `override` always splits: the override brings storage distinct from the base's. A `new` field-like event
splits only when it actually hides an inherited event; that is the case reported.

### What is not reported

- **A plain field-like event with neither modifier.** There is exactly one compiler-generated field and no
  way to split it. A plain `virtual` field-like event with no `override` or `new` is reported elsewhere and is
  not repeated here.
- **An `abstract` event**, including an `abstract override` that re-abstracts. It declares no backing field at
  all — the storage decision belongs to whoever implements it.
- **An event with explicit `add`/`remove` accessors.** Its subscribers live wherever the author chose, and
  every type in the hierarchy that delegates to that storage shares it.
- **An interface event.** It declares no storage; a base interface's event lives outside the class base
  chain, so a `new` interface event hides nothing that could split.
- **A `new` that hides nothing.** With no inherited event of the same name the keyword is redundant, the
  compiler already says so, and no split can occur.

## How to fix violations

Decide whether the derived type really needs its own declaration of the event.

If it does not — the common case — drop the `override`/`new` and reuse the single inherited event. A derived
type that wants to react can override a raising method instead of the event itself:

```csharp
public class Publisher
{
    public event EventHandler Changed;

    protected virtual void OnChanged() => Changed?.Invoke(this, EventArgs.Empty);
}

public class LoudPublisher : Publisher
{
    protected override void OnChanged()
    {
        Log("changed");
        base.OnChanged();
    }
}
```

If the event genuinely must be redeclared, give the whole hierarchy explicit accessors so the storage is
named, shared, and under your control — the override then has something concrete to delegate to:

```csharp
public class Publisher
{
    private EventHandler _changed;

    public virtual event EventHandler Changed
    {
        add => _changed += value;
        remove => _changed -= value;
    }

    protected void Raise() => _changed?.Invoke(this, EventArgs.Empty);
}
```

There is **no code fix**. The two repairs — dropping the redeclaration, and moving to explicit accessors —
change the type's contract in opposite directions, and the accessor form needs a storage decision only the
author can make.

## Performance

The clean path is one scan of the declaration's modifier list and binds nothing. The declared symbol is
resolved only for a declaration that spells `override` or `new`; `IEventSymbol.IsOverride` confirms an
accepted override, and a base type is walked for a hiding `new` only after the symbol proves it is not an
override.
