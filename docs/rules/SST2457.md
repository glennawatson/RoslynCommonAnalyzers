# SST2457 — Wrapping a sequence sum in unchecked does not stop it throwing

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An integer overload of the sequence `Sum` operator — `int`, `long`, or their nullable and selector
forms — is called inside an `unchecked` expression or block.

```csharp
using System.Linq;

public sealed class Ledger
{
    public int Total(int[] amounts)
    {
        return unchecked(amounts.Sum()); // SST2457
    }
}
```

## Rule description

An `unchecked` context is lexical: it changes the arithmetic *written inside it*, not the arithmetic
inside the methods it calls. The integer overloads of the sequence `Sum` operator accumulate in a
checked context of their own, so the call throws `OverflowException` on overflow no matter what
wraps it.

That makes the wrapper worse than useless. A reader — and the author — takes `unchecked(…Sum())` as
"this total is allowed to wrap around", and writes the surrounding code for a value that always
arrives. At runtime the sum still throws, on exactly the inputs the wrapper was supposed to absorb,
and it throws from a line that visibly claims it cannot.

### What is not reported

- **The `float`, `double`, and `decimal` overloads.** Floating-point addition never throws on
  overflow, and decimal arithmetic throws with or without a wrapper — neither is a wraparound claim
  the operator silently breaks in the same way.
- **A call whose nearest wrapper is `checked`.** `checked` claims nothing about wrapping around,
  even when an `unchecked` block encloses it further out.
- **A user-defined method named `Sum`.** Only the sequence operator is known to check internally;
  what someone else's `Sum` does is its own business.

## How to fix violations

Decide what you actually want from overflow.

```csharp
using System.Linq;

public sealed class Ledger
{
    // To really wrap around: accumulate in your own loop, where unchecked applies.
    public int WrappingTotal(int[] amounts)
    {
        var total = 0;
        unchecked
        {
            foreach (var amount in amounts)
            {
                total += amount;
            }
        }

        return total;
    }

    // To make overflow survivable instead: sum into a wider type.
    public long WideTotal(int[] amounts) => amounts.Sum(amount => (long)amount);
}
```

If the sum genuinely cannot overflow, drop the wrapper: it documents an escape hatch the code does
not have.

There is **no code fix**. The two repairs — a manual loop that really wraps, and a wider accumulator
that no longer needs to — mean opposite things, and choosing between them is the design decision the
diagnostic is asking you to make.

## Performance

The clean path binds nothing. The invoked name must be `Sum` and an enclosing `unchecked` construct
must be found — both on syntax, walking no further up than the containing member — before the
invocation is bound to confirm it is the sequence operator. `System.Linq.Enumerable` itself is
resolved lazily, on the first candidate only, so a compilation with no `Sum` inside `unchecked`
never pays for the lookup.
