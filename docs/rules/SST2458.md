# SST2458 — Bitwise operations should not be applied to a non-flags enum

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A bitwise operator — `|`, `&`, `^`, `~`, or their compound assignments — is applied to an enum that
is not declared `[Flags]`.

```csharp
public enum Color
{
    Red,    // 0
    Green,  // 1
    Blue,   // 2
}

var mixed = Color.Green | Color.Blue; // SST2458 — the value is 3, which is no Color at all
```

## Rule description

An enum without `[Flags]` is a list of alternatives. The compiler numbers its members sequentially —
0, 1, 2 — so the members share bits instead of owning one each. Bitwise operators assume the
opposite: that each member is an independent bit that can be set, cleared, and tested on its own.

Apply them to a sequential enum and the arithmetic still runs, but the answers are wrong:

- `Color.Green | Color.Blue` is `3` — a value no member names. It prints as `"3"`, matches no
  `switch` arm, and round-trips through serialization as a number nothing can interpret.
- `Color.Red | Color.Green` is `Color.Green` — or-ing with the zero member does nothing, silently.
- `value & Color.Blue` examines bit 1 of `value`, a bit that was never assigned a meaning. The test
  answers a question nobody asked, so `(value & Color.Blue) == Color.Blue` is `true` for `Blue` *and*
  for any value whose low bits happen to overlap — with sequential values, that is most of them.

Every one of those still compiles and runs. That is what makes it a correctness bug rather than a
style choice: each branch executes, some inputs even produce the expected result, and the rest
quietly do not.

### What is reported

- `|`, `&`, `^` where an operand is a non-flags enum, including the lifted operators on a nullable
  enum.
- `~` on a non-flags enum value.
- `|=`, `&=`, `^=` where the assigned variable is a non-flags enum.

A chain of bitwise operations over the same enum — `a | b | c`, `value & ~mask` — is one mistake, so
only the outermost operation of the chain is reported.

A masked value compared to a named member (`(value & mask) == member`) and a masked zero test
(`(value & mask) != 0`) are deliberately still reported. Both borrow the flag-testing idiom, and the
idiom only tells the truth when each member owns a bit. If the enum genuinely is a bit set written
without the attribute, the repair is to say so — see below — not to keep the tests looking
plausible.

### What is not reported

- Any operation on an enum declared `[Flags]`. Whether its members actually occupy distinct bits is
  a different question, owned by SST2303.
- Equality comparisons — `==` and `!=` are the supported way to use a non-flags enum.
- Operands explicitly cast to a numeric type: `(int)color | 0x40` says "raw number" in the source,
  and raw numbers combine however the author likes.
- Bitwise work on integers, and `|`/`&`/`^` on booleans, where they are the ordinary
  non-short-circuiting operators.

## How to fix violations

Decide what the enum is.

If its values genuinely combine, it is a flag set — declare it as one, with one bit per member:

```csharp
[Flags]
public enum Color
{
    None = 0,
    Red = 1,
    Green = 2,
    Blue = 4,
}

var mixed = Color.Green | Color.Blue; // a real combination, prints "Green, Blue"
```

If its values are alternatives, the bitwise operation itself is the bug — the code meant a
comparison, a collection, or a second field:

```csharp
if (color == Color.Green || color == Color.Blue) { }   // instead of (color & …)
```

There is **no code fix**. The two repairs point in opposite directions — `[Flags]` changes the
enum's contract for every consumer, and rewriting the logic changes what this call site computes —
and choosing between them is exactly the design decision the diagnostic is asking for.

## Performance

Only the seven bitwise operator kinds are examined, and the clean path binds one operand at most. A
literal or numeric-cast operand is rejected on syntax alone, so `(int)a | (int)b` and `0x1 | 0x2`
never touch the semantic model; for everything else one operand is bound, and a non-enum answer
settles the whole operation, because no built-in bitwise operator pairs an enum with anything but
itself. The flags attribute is only inspected once an enum operand has actually been found.
