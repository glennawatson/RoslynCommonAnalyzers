# SST2459 — [Optional] cannot make a ref or out parameter optional

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

`[Optional]` (`System.Runtime.InteropServices.OptionalAttribute`) is applied to a `ref` or `out` parameter.

```csharp
using System.Runtime.InteropServices;

public sealed class Session
{
    public void Load([Optional] ref int value) => value = 1; // SST2459
}
```

## Rule description

A `ref` or `out` argument must name a variable at every call site, so a C# caller can never omit it — a
call that leaves it out does not compile. `[Optional]` does not change that, so the optionality it
advertises does not exist.

The attribute is not inert, though. It flips the parameter's reflection-visible `IsOptional` flag, so
late-bound and reflection-based callers are told they may omit an argument that every compiled call must
supply — the opposite of the truth.

`[Optional]` on a by-value parameter is honest: there the runtime fills in the default and callers really
can omit it. So are `in` and `ref readonly`, which stay omittable, and neither is reported. Members of a
COM-imported type are also left alone, because there the compiler really does let callers omit an
`[Optional]` by-reference argument.

## How to fix violations

Remove the attribute. If the parameter should be omittable, pass it by value or by `in`, or add an
overload without it.

```csharp
public void Load(ref int value) => value = 1;
```

The code fix removes the `[Optional]` attribute, keeping any others in the same list.

## Performance

The clean path is syntax-only: a parameter with no attribute list, or with no `out` and no bare `ref`, is
rejected without ever touching the semantic model. Only an attribute whose written name is `Optional` is
bound, which is what confirms it is the framework attribute and not a type of the same name.
