# SST2462 — A member hidden with 'new' should not reduce its accessibility below the member it hides

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A member declared with the `new` modifier is less accessible than the inherited member it hides.

```csharp
public class Base
{
    public void M()
    {
    }
}

public class Derived : Base
{
    private new void M() // SST2462: 'M' is private but the hidden 'Base.M' stays public
    {
    }
}
```

## Rule description

The `new` modifier declares a member that hides an inherited one of the same name. Hiding is resolved at
compile time against the static type of the reference, not the runtime object, so declaring the hiding
member with a narrower accessibility than the member it hides does not make the inherited member any less
reachable.

```csharp
Base b = new Derived();
b.M();      // binds to Base.M — public, and still reachable
```

A caller holding the base type binds to the more accessible inherited member; a caller holding the derived
type sees the narrower one. The two members diverge, and the accessibility that looks reduced is an
illusion — the type ends up exposing two members of the same name at different visibilities.

## What this rule reports

Any member kind that can carry `new` — a method, property, event, field, or nested type — when the derived
member is **strictly less accessible** than the accessible inherited member it hides. Methods are matched by
signature; the other kinds by name.

```csharp
public class Base
{
    public void M() { }
    public int P { get; set; }
    public int F;
}

public class Derived : Base
{
    private new void M() { }              // SST2462
    internal new int P { get; set; }      // SST2462
    protected new int F;                  // SST2462
}
```

The rule stays silent when the change is not a reduction, when nothing is actually hidden, and on overrides:

| shape | why it is clean |
| --- | --- |
| `public new void M()` over a `public` base | equal accessibility — nothing is narrowed |
| `public new void M()` over a `protected` base | wider accessibility |
| `internal new void M()` over a `protected` base | `protected` and `internal` are incomparable, neither narrower |
| `public override void M()` | an override, not a hide |
| `private new void M(string s)` over `void M(int i)` | a different signature hides nothing (the compiler warns separately) |
| a `new` member hiding a `private` base member | a private base member is not inherited, so nothing is hidden |
| a narrower member declared without `new` | a compiler diagnostic already reports the unintended hide |

## How to fix violations

Give the hiding member the same or wider accessibility than the member it hides, so the visibility a reader
sees is the visibility that is enforced. If the intent was to stop exposing the inherited member, hiding it
does not achieve that — remove the `new` member and `override` or rename it so a single member governs
access.

## Performance

The walk is symbol-based and its clean path never touches syntax. A type whose base is `object` is rejected
outright, and for every other type each declared member is matched against an accessible inherited member up
the base chain. The declaration's `new` modifier is read only once a strictly-more-accessible hidden member
has already been found, so syntax is fetched only on the verge of reporting.
