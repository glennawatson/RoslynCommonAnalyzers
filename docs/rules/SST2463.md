# SST2463 — A field should not differ from an inherited field only by case

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A derived type declares an instance field whose name matches an accessible field it inherits from a base
type when the two names are compared without regard to case, yet the names are not identical.

```csharp
public class Base
{
    protected int _value;
}

public class Derived : Base
{
    private int _Value;   // SST2463: differs only by case from the inherited _value
}
```

## Rule description

The compiler keeps the base field and the derived field as two separate storage locations. Both are
reachable by an unqualified name inside the derived type, so a reference that gets the case wrong binds to
the other field and compiles with no warning. A value written through one name is then read back as stale or
default through the other, and the two names sit close enough that the mistake is easy to make and hard to
see in review.

## What this rule reports

The reported field is the derived type's own instance field; the message names the inherited field and the
type that declares it.

```csharp
public class Base
{
    public int Value;
}

public class Derived : Base
{
    public int VALUE;     // SST2463: differs only by case from the inherited Value
}
```

The base field is matched at any depth of the base chain, and at any accessibility the derived type can
see — `protected`, `internal`, `protected internal`, or `public`.

A field that does not fit that shape is never reported:

| shape | example | why it is clean |
| --- | --- | --- |
| identical names | base `_value`, derived `new int _value` | an exact match is deliberate hiding, a different concern |
| a private base field | base `private int _value`, derived `_Value` | the base field is invisible to the derived type |
| more than a case difference | base `_value`, derived `_values` | two ordinary, distinct fields |
| a case-clash inside one type | `_value` and `_Value` in the same class | no inheritance is involved |
| a static field on either side | base `static int _value`, derived `_Value` | the storage kinds differ, so an unqualified reference is not ambiguous the same way |
| an inherited property's backing field | base `int Value { get; set; }`, derived `int value` | the compiler-generated field is not a source field |

## How to fix violations

Rename the derived field so it no longer collides with the inherited one — usually to the name it was meant
to be, or to a distinct name if it is genuinely a separate value. If the derived field is meant to replace
the inherited storage, remove it and use the inherited field directly.

## Performance

The rule runs on the type symbol. A type that is not a class, or that derives directly from `object`, is
rejected before any member is inspected, and the base chain is walked only for a type that declares at least
one own instance field. Nothing but symbol names is compared.
