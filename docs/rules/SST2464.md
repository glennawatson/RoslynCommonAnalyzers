# SST2464 — A value-equality operator should not be declared on a mutable reference type

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `class` that still exposes settable instance state declares a user-defined `operator ==`, giving the
reference type value semantics that depend on state which can change after construction.

```csharp
public class Point
{
    public int X { get; set; } // settable — the instance is mutable

    public static bool operator ==(Point a, Point b) => a.X == b.X; // SST2464
    public static bool operator !=(Point a, Point b) => !(a == b);
}
```

## Rule description

A user-defined equality operator gives a reference type value semantics: two distinct instances compare
equal when their state matches, and a matching hash sends them to the same bucket. That is only safe while
the state cannot change.

When the class still exposes a settable field or property, an instance can be mutated *after* it has been
stored as a key in a `Dictionary` or `HashSet`. Its hash then no longer matches the bucket it was placed
in, so the key can never be found again, `Remove` and `ContainsKey` silently fail, and lookups near it can
be corrupted. The same instance is simultaneously "in" the collection and unreachable through it.

Make the type immutable — every field `readonly`, every property get-only or `init`-only — or drop the
value-equality operator and let instances compare by reference identity.

## What this rule reports

The `operator ==` declaration, once, when its containing type is a **mutable class**: a `class` that
declares at least one instance field that is not `readonly`/`const`, or an instance property with a `set`
accessor that is not `init`-only. Binding merges partial declarations, so state declared in a different
part than the operator is still counted.

Only `operator ==` is reported; the paired `operator !=` is left alone so one mutable class yields one
diagnostic.

These are never reported:

| shape | why |
| --- | --- |
| an immutable `class` — every field `readonly`, every property get-only or `init`-only | its equality can never go stale |
| a `struct` or `record struct` | a value type is copied, not keyed by reference identity |
| a `record` or `record class` | records own their value equality by design |
| an operator other than `==` (for example `+`) | the defect is value equality, not any operator |

## How to fix violations

Prefer making the type immutable:

```csharp
public sealed class Point
{
    public Point(int x) => X = x;

    public int X { get; } // get-only — the instance can no longer be mutated

    public static bool operator ==(Point a, Point b) => a.X == b.X;
    public static bool operator !=(Point a, Point b) => !(a == b);
}
```

If value semantics are what you want and the data is naturally immutable, a `record` expresses it directly
and generates the equality for you. If the type must stay mutable, remove the `operator ==`/`operator !=`
pair (and any matching `Equals`/`GetHashCode`) so instances compare by reference, which stays correct under
mutation.

There is no automatic code fix: the safe rewrite — sealing the state as immutable or removing equality
altogether — changes the type's contract, so it is left to a person.

## Performance

The clean path is syntactic: the node must be an `operator ==` declaration whose containing type is a plain
`class`. A `struct`, a `record`, and a `record struct` are a different syntax node and never reach the bind.
Only then is the type bound and its members scanned once for settable state.
