# SST2465 — A for loop's body reassigns a variable its condition depends on

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `for` loop's body reassigns the counter, or the local its condition compares the counter against, so the
loop runs a different number of times than the header states.

```csharp
public void Fill(List<int> list)
{
    var n = list.Count;
    for (var i = 0; i < n; i++)
    {
        Process(list[i]);
        n = 0; // SST2465: the bound the condition tests is reset every iteration
    }
}
```

## Rule description

A `for` header reads as a promise about how many times the body runs: the counter starts, the condition
bounds it, and the incrementer steps it. When the body itself reassigns one of the variables the condition
reads, that promise is broken — the loop runs forever, stops early, or never runs — and the mistake hides
because the header still looks right. Resetting the bound (`n = 0`) or moving the counter (`i = 5`) inside
the body are the two shapes this catches.

## What this rule reports

Only the unambiguous counted-loop shape, and only an assignment that runs on every iteration:

```csharp
// the counter is reassigned in the body
for (var i = 0; i < 10; i++)
{
    i = 5; // SST2465: never terminates
}

// the bound is reassigned in the body
for (var i = 0; i < n; i++)
{
    n = 0; // SST2465: the loop stops after the first pass
}
```

For a loop to be considered at all:

- the condition is a single relational or equality comparison (`<`, `<=`, `>`, `>=`, `==`, `!=`) built only
  from identifiers, literals, and arithmetic;
- the loop has exactly one incrementer that steps a single counter with `++`, `--`, or a compound
  assignment, and the condition tests that counter.

Within such a loop, an assignment (`=`, `+=`, `-=`, `++`, `--`) to a variable the condition reads is reported
when it binds to a local or a parameter and runs unconditionally.

A write the rule leaves alone:

| shape | why it is not reported |
| --- | --- |
| `if (done) i = n;` | guarded — it may be an intended early advance the rule cannot disprove |
| a write inside a nested loop, `switch`, `try`, or lambda | the same: not an every-iteration reassignment |
| `_n = 0;` where `_n` is a field or property | something else may own the value, and the write may be intended |
| the header's own `i++` | only the body is scanned |
| `for (var i = 0; i < n;) { i++; }` | the header delegates stepping to the body, so the body write is the step |

When in doubt the rule stays silent: it prefers to miss a real defect over reporting a deliberate one.

## How to fix violations

Decide which the header means. If the counter or bound really should change mid-loop, a `while` loop states
that honestly; if it should not, remove the stray assignment from the body.

```csharp
for (var i = 0; i < n; i++)
{
    Process(list[i]); // no reassignment of i or n in the body
}
```

## Performance

The clean path never binds. The condition shape and the single-counter incrementer reject nearly every loop
up front, and a candidate write's symbol is resolved only after its identifier has already matched a
condition variable — which is rare — so a well-formed counted loop pays only a handful of syntactic
comparisons.
