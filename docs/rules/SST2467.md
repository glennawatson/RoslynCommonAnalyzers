# SST2467 — A params overload should not be silently bypassable by a more specific overload

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A type declares a `params` overload and a same-named, same-arity overload whose last parameter is a
**more specific** type than the array's element type, so a call passing a single argument of that type
binds to the specific overload instead of the params array.

```csharp
public class Log
{
    public void Write(string format, params object[] args) { } // SST2467: bypassed by the sibling below
    public void Write(string format, Exception error) { }
}

// The author expects formatting; the call quietly runs the Exception overload:
log.Write("connection failed: {0}", ex);
```

## Rule description

A `params` array parameter is a catch-all: any single trailing argument that can be stored in one array
element is accepted. When the same type also declares a same-named overload of the same arity whose last
parameter is a stricter type than the array's element type — a subclass of it, an interface it implements,
or a value type that would otherwise box — overload resolution prefers that overload for every call passing
exactly that type. An exact match beats the conversion the params array's expanded form would need, so the
call binds to the sibling, not the params overload, and runs different code.

The classic trap is a `params object[]` logging or formatting method next to an overload taking a specific
type such as `Exception`: `Write("failed {0}", ex)` silently drops the formatting and calls the `Exception`
overload. The compiler does not warn about this, and the difference only shows up at runtime.

## The exact shape reported

The rule is decided entirely from the two declarations — no call site is inspected. It reports the `params`
overload `M(..., params E[])` of arity `n` when the same type declares a **non-`params`** sibling `M` such
that:

- the sibling has the same arity `n` and the same static-ness;
- its leading `n - 1` parameters have identical types and ref kinds;
- its final by-value parameter type `T` is **strictly more specific** than the element type `E` — a proper
  subclass of `E`, an interface that `E` is (an interface `T` derives from), a type that implements the
  interface `E`, or a value type that boxes to `E`. For `params object[]`, every non-`object` type qualifies.

## What this rule does not report

| shape | example | why |
| --- | --- | --- |
| the element type itself | `Write(string, object)` beside `params object[]` | the deliberate allocation-avoiding overload — same behaviour, no array allocated |
| a base type of the element | `Write(string, object)` beside `params string[]` | the params overload wins for a `string`, so nothing is bypassed |
| a different arity | `Write(string)` beside `Write(string, params object[])` | captures only the zero-argument call, a common intentional overload |
| a different leading parameter | `Write(int, Exception)` beside `Write(string, params object[])` | not applicable to the same calls |
| two `params` overloads | `Write(string, params string[])` beside `params object[]` | neither is the fixed sibling that peels a single argument |
| an unrelated sibling type | `Write(string, string)` beside `params int[]` | a `string` is not convertible to `int`, so both are never applicable at once |

## How to fix violations

There is no automatic fix, because the right change depends on intent:

- If the specific overload is meant to be reachable, rename it (for example `WriteError`) so callers pick it
  deliberately.
- If the params overload is the intended entry point, remove the specific overload, or give it a distinct
  leading parameter so the two no longer collide.
- If both must coexist under the same name, be aware which one a single-argument call selects and steer
  callers accordingly.

## Performance

The clean path is symbol-only. A type's members are scanned once, and the sibling lookup runs only for a
method that actually ends in a `params` array. No syntax is walked and nothing is bound, so a type with no
`params` overload pays a single member scan and returns.
