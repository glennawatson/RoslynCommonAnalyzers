# SST2470 — String literals concatenated without a space should not fuse a SQL keyword

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes (regular string literals) |
| Default | Enabled |

## Cause

Two string literals are concatenated with `+`, and because there is no space between them the seam fuses a
SQL keyword into the adjacent token.

```csharp
var sql = "SELECT * FROM t" + "WHERE id = 1"; // runs as "SELECT * FROM tWHERE id = 1"
var more = "... WHERE a = 1" + "AND b = 2";    // runs as "... WHERE a = 1AND b = 2"
```

## Rule description

A SQL statement assembled from concatenated string literals is one continuous string at runtime, so the seam
between two literals is invisible. When a literal ends in a word character and the next begins with a SQL
keyword — or a literal ends with a keyword and the next begins with a word character — the two run together
into a single token. The keyword disappears, and the database either rejects the query or, worse, runs a
different one than the code appears to build.

## What this rule reports

A `+` of **two string literals** (regular, verbatim, or raw) whose seam fuses a keyword, in either direction:

```csharp
// a keyword fused onto the left operand's tail
"SELECT * FROM t" + "WHERE id = 1";
"... WHERE a = 1" + "AND b = 2";

// a keyword fused onto the right operand's head
"SELECT id FROM" + "users";
```

The fusion must actually merge two tokens, so the character on each side of the seam must be a word character
(an ASCII letter, digit, or underscore); a keyword bounded by a space, punctuation, or the string end stays a
separate token and is not reported. The curated keyword set is `WHERE`, `AND`, `OR`, `FROM`, `JOIN`, `INNER`,
`LEFT`, `RIGHT`, `ON`, `ORDER`, `GROUP`, `HAVING`, `SELECT`, `SET`, `VALUES`, `INTO`, `UNION`, `LIMIT`,
`OFFSET`, `RETURNING` (case-insensitive).

To keep ordinary prose quiet the rule also requires the **left literal to read as SQL**, by containing a strong
SQL keyword as a delimited token (`SELECT`, `FROM`, `WHERE`, `JOIN`, `INTO`, `VALUES`, `UNION`, `HAVING`,
`RETURNING`, `INSERT`, `UPDATE`, `DELETE`). This is what keeps the rule near zero false positives.

None of the following is reported:

| shape | example |
| --- | --- |
| a space, tab, or newline at the seam | `"SELECT * FROM t" + " WHERE id = 1"` |
| a punctuation token boundary at the seam | `"SELECT a," + "FROM t"` |
| a keyword that is only a prefix of a longer word | `"SELECT * FROM t" + "WHEREVER it goes"` |
| prose whose left side is not SQL | `"click here" + "OR press escape"` |
| an interpolated or otherwise non-literal operand | `$"SELECT * FROM {t}" + "WHERE id = 1"` |

Only a direct pair of adjacent literals is inspected; a keyword fused across the outer seam of a longer
concatenation chain (where one side is itself a concatenation) is not reported.

## How to fix violations

Add a space at the seam so the keyword stays a separate token:

```csharp
var sql = "SELECT * FROM t" + " WHERE id = 1";
```

The **code fix** gives the right literal a single leading space. The concatenated text is identical wherever
the space lands, so the edit is unambiguous. It is offered only when the right operand is a regular string
literal; a verbatim or raw right literal is reported without a fix, because rewriting it would change its form
rather than just its content — place the space by hand there.

## Performance

The clean path is purely syntactic. A `+` whose two operands are not both string literals returns at once, and
the keyword scans run only after a cheap seam check holds, so no work and no allocation happen until a real
fusion is found. Nothing binds to the semantic model.
