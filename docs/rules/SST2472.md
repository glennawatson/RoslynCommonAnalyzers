# SST2472 — An exported contract type must be implemented by the exporting type

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A type is exported for a contract it neither implements nor inherits. The contract is named only inside
the export attribute, so the mismatch compiles cleanly and surfaces only when the part is composed.

```csharp
using System.ComponentModel.Composition;

public interface IService { }

[Export(typeof(IService))] // SST2472: Widget does not implement IService
public class Widget { }
```

## Rule description

An export declares the contract a part is offered under: the container hands the part to every import of
that contract. When the exporting type is not actually assignable to the declared contract, nothing can
satisfy the import at run time — composition fails, or the cast the container performs throws — even
though the compiler saw nothing wrong. The compiler cannot check the relationship because the contract
appears only as a `typeof(...)` argument, never in the type's base list where it would be verified.

The rule reports the export when the declared contract type is not the exporting type, one of its base
types, or one of the interfaces it implements. The named-argument form is treated identically:

```csharp
[Export(contractType: typeof(IService))]
public class Widget { }
```

Both MEF flavors are covered: `System.ComponentModel.Composition.ExportAttribute` and
`System.Composition.ExportAttribute`.

## What this rule reports

Only an export whose contract the type cannot supply. These are correct and are never reported:

| shape | why it is fine |
| --- | --- |
| `[Export(typeof(IService))] class Widget : IService` | the contract is implemented |
| `[Export(typeof(IService))] class Widget : ServiceBase` where `ServiceBase : IService` | satisfied through a base type |
| `[Export(typeof(Widget))] class Widget` | a type exported as itself is assignable to itself |
| `[Export] class Widget` | no contract type is named, so there is nothing to check |
| `[Export(typeof(IService))]` on a type in a project that references no MEF assembly | the rule is gated off |

## How to fix violations

Implement the contract, or export the type under a contract it actually satisfies:

```csharp
[Export(typeof(IService))]
public class Widget : IService { } // implement the contract

// or export it under the right contract
[Export(typeof(IWidget))]
public class Widget : IWidget { }
```

## Performance

The rule is gated at compilation start on a MEF export attribute resolving; a project that references no
MEF assembly registers nothing. On the hot path each attribute is rejected on its written simple name
before any binding, and only an attribute named like an export is bound to confirm it is the marker type
and to read its contract argument.
