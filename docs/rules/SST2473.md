# SST2473 — A shared export part should be obtained from the container, not constructed with 'new'

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `new` expression directly constructs a type that is itself a shared (single-instance) export part.
Constructing it bypasses the container that was meant to own its single instance.

```csharp
using System.ComponentModel.Composition;

[Export(typeof(IService))]
[PartCreationPolicy(CreationPolicy.Shared)]
public class Service : IService { }

public class Consumer
{
    public IService Create() => new Service(); // SST2473: bypasses the container's single instance
}
```

## Rule description

A part marked shared is meant to exist once: the container creates a single instance and hands the same
one to every importer. Constructing that type with `new` builds a second instance the container knows
nothing about, so the single-instance guarantee the part was designed around no longer holds. Any state
it caches, any resource it owns, and any identity comparison against the container's instance quietly
diverge — and none of it shows up at compile time, because `new` on an exported type is perfectly legal.

A part is treated as shared when it carries a MEF export and is marked single-instance:

- `System.ComponentModel.Composition` — `[PartCreationPolicy(CreationPolicy.Shared)]`.
- `System.Composition` — `[Shared]`.

## What this rule reports

Only the direct construction of a type that is both exported and shared. These are never reported:

| shape | why it is fine |
| --- | --- |
| `new Service()` where `Service` is `[PartCreationPolicy(CreationPolicy.NonShared)]` | a non-shared part has no single-instance guarantee to break |
| `new Service()` where `Service` has no `[Export]` | not a composed part |
| `new Plain()` on an ordinary type | not a part at all |
| constructing a shared part in a project that references no MEF assembly | the rule is gated off |

## How to fix violations

Obtain the part from composition instead of constructing it — import it where it is needed, or resolve
it from the container:

```csharp
public class Consumer
{
    [Import]
    public IService Service { get; set; }
}
```

If the type does not actually need to be a single shared instance, reconsider its creation policy rather
than constructing it directly.

## Performance

The object-creation scan is registered only when both an export attribute and a shared marker resolve in
the compilation, so a project that references no MEF assembly, or references it but never marks a part
shared, pays nothing extra. When it is active the created type is resolved once per `new` and its
attributes are read only to confirm it is a shared export part.
