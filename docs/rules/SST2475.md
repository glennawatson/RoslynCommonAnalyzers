# SST2475 — An entity's primary key should not be a temporal type

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No (changing a key's type is a schema decision) |
| Default | Enabled |

## Cause

An entity's primary key property is typed `DateTime` or `DateTimeOffset` (or their nullable forms). The
key is identified either by an explicit `[Key]` attribute, or by the primary-key naming convention on a
type that is used as an entity through a `DbSet<T>`.

```csharp
using System;
using System.ComponentModel.DataAnnotations;

public class Reading
{
    [Key]
    public DateTime CapturedAt { get; set; } // SST2475: a temporal primary key
}
```

## Rule description

A primary key stakes row identity on a value, and a wall-clock value is the wrong thing to identify a row
by:

- **Collisions.** Two rows created in the same tick get the same key, and the insert fails or silently
  overwrites — timestamps are not unique.
- **Not a stable identifier.** The value shifts with the clock, the time zone, and daylight saving, so the
  identifier means different things in different contexts.
- **Poor clustering.** A temporal clustered key orders and fragments the table by insertion time rather
  than by identity, which is not what a key is for.
- **Imprecise round-trip.** Providers store `DateTime` and `DateTimeOffset` at different resolutions and
  offsets, so a key that matched on write can fail to match on read.

Use a stable surrogate key — an integer identity or a GUID — and keep the timestamp as an ordinary column.

## What this rule reports

The key property is found two ways:

```csharp
// explicit: the property carries the key attribute
[Key]
public DateTimeOffset CapturedAt { get; set; }   // reported

// convention: a public read-write Id / <TypeName>Id on a type exposed by a DbSet<T>
public class Order
{
    public DateTime Id { get; set; }             // reported
}

public class ShopContext : DbContext
{
    public DbSet<Order> Orders { get; set; } = null!;
}
```

These shapes are **not** reported:

| shape | why |
| --- | --- |
| a non-temporal key (`int Id`, `Guid Id`) | the key is already a stable identifier |
| a convention-named temporal property on a type not exposed by any `DbSet<T>` | the type is not used as an entity |
| a read-only, static, or non-public convention-named property | it is not the shape the framework maps as a key |
| any property in a project that references neither the key attribute nor Entity Framework | there is no entity model to have a key |

## How to fix violations

Give the entity a stable surrogate key and demote the timestamp to an ordinary column:

```csharp
public class Reading
{
    [Key]
    public int Id { get; set; }

    public DateTime CapturedAt { get; set; } // just data now
}
```

There is no automatic code fix: which surrogate to use (integer identity, sequence, GUID) and how to
migrate existing data are schema decisions the tool cannot make safely.

## Performance

The whole rule is gated at compilation start: it registers nothing unless the key attribute resolves (for
the explicit path) or Entity Framework's `DbContext`/`DbSet` resolves (for the convention path), so a
project referencing neither pays nothing. When the framework is present, the entity set is walked once per
compilation over the source assembly's own types; the per-property callback then does a cheap type check
and returns before any allocation for every non-temporal property.
