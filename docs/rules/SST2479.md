# SST2479 — A delegate that outlives a loop should not capture the loop's changing variable

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A lambda, anonymous method, or local function captures a variable that a `for`, `while`, or `do` loop
steps through, and is then stored where it outlives the loop iteration rather than being run in place.

```csharp
var handlers = new List<Action>();
for (int i = 0; i < 3; i++)
    handlers.Add(() => Console.WriteLine(i)); // SST2479: every handler prints 3
```

## Rule description

A `for`/`while`/`do` loop keeps one shared storage slot for the variable it advances — unlike `foreach`,
it does not make a fresh copy per iteration. A closure captures that single slot, so it reads whatever value
the variable holds at the moment the closure runs, not the value it held when the closure was created.

That is harmless while the closure runs in place inside the same iteration. It becomes a defect the moment
the closure is stored where it outlives the iteration: subscribed to an event, added to a collection,
assigned to a field, property, or array element, handed to a deferred runner such as `Task.Run`, or produced
by `yield return`. Each stored delegate closes over the same slot, so when they finally run they all read the
variable's final value — every handler behaves as if it were the last.

## What this rule reports

A capturing delegate that escapes the loop body through one of these local shapes:

```csharp
for (int i = 0; i < items.Count; i++)
{
    button.Click += (s, e) => Use(i);          // subscribed to an event
    handlers.Add(() => Use(i));                // added to a collection
    slots[i] = () => Use(i);                   // assigned to an array element
    tasks.Add(Task.Run(() => Use(i)));         // handed to a deferred runner
}

IEnumerable<Action> Make()
{
    for (int i = 0; i < 3; i++)
        yield return () => Use(i);             // produced by yield return
}
```

The same applies to an anonymous method and to a local function captured by method group and stored. A local
stepped by a `while` or `do` body — declared outside the loop and assigned or incremented inside it — is
treated the same as a `for` header variable.

The following are correct and are never reported:

| shape | why it is safe |
| --- | --- |
| a `foreach` iteration variable | the language gives it a fresh copy each iteration |
| a per-iteration copy (`int copy = i;` captured) | the copy is a fresh local each iteration |
| a captured local that never changes across the loop | there is no per-iteration value to lose |
| a delegate invoked in place inside the loop | it reads the current value, which is correct |
| a delegate `return`ed from inside the loop | `return` ends the loop, so no delegates share a final value |
| a captured parameter or field | it is not stepped by the loop |

Escape is decided from local syntax only; no dataflow or interprocedural analysis runs, so an unrecognized
escape shape is left silent rather than guessed at.

## How to fix violations

Copy the loop variable into a variable declared **inside** the loop body and capture that copy, so each
iteration closes over its own value:

```csharp
for (int i = 0; i < 3; i++)
{
    int captured = i;
    handlers.Add(() => Console.WriteLine(captured)); // each handler prints its own index
}
```

There is no code fix: choosing the copy's name and placement is a judgment left to the author.

## Performance

The clean path is syntactic. Every candidate escape site — a collection-store call, a deferred-runner call,
an event subscription, a storage assignment, or a `yield return` — is rejected without binding unless it
actually stores a delegate expression. Only then are the delegate's captures bound and its enclosing loops
walked.
