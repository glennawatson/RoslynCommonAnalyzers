# SST2481 — GetHashCode should not fold in the base object identity hash

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `GetHashCode` override on a type that derives straight from `object` mixes `base.GetHashCode()` into a
hash otherwise built from the object's fields.

```csharp
public sealed class Money
{
    private readonly decimal _amount;
    private readonly string _currency;

    public override bool Equals(object obj)
        => obj is Money other && other._amount == _amount && other._currency == _currency;

    public override int GetHashCode()
        => base.GetHashCode() ^ _amount.GetHashCode() ^ _currency.GetHashCode(); // SST2481
}
```

## Rule description

When a type's base is `object`, `base.GetHashCode()` is the runtime identity hash — a value fixed to the
reference the instance lives at, unrelated to the fields it carries. Folding it into a hash built from those
fields ties the whole hash to identity: two instances that `Equals` reports as equal produce different hashes.

A hash-based collection places an object in a bucket derived from its hash. Look-up recomputes the hash and
searches that one bucket. Once identity leaks into the hash, an object stored under one instance lands in a
bucket an equal instance never visits, so the value cannot be found under a value-equal key — the exact
contract `Equals` and `GetHashCode` exist to uphold is broken.

## What this rule reports

Only a `base.GetHashCode()` that is **combined with other state** inside a parameterless `GetHashCode`
override whose base is `object`:

```csharp
public override int GetHashCode() => base.GetHashCode() ^ _value.GetHashCode();          // reported
public override int GetHashCode() => System.HashCode.Combine(base.GetHashCode(), _value); // reported

public override int GetHashCode()
{
    int seed = base.GetHashCode(); // reported — the identity hash is still mixed in below
    return seed ^ _value;
}
```

These shapes are correct and are never reported:

| shape | example | why it is clean |
| --- | --- | --- |
| a base class's own value hash | `class D : B { public override int GetHashCode() => base.GetHashCode() ^ _z; }` where `B` overrides `GetHashCode` | the base call is a real value hash, so chaining it is legitimate |
| pure delegation | `public override int GetHashCode() => base.GetHashCode();` | the base call is the whole result; it is a reference-hash concern, reported elsewhere |
| a field-only value hash | `public override int GetHashCode() => System.HashCode.Combine(_a, _b);` | no identity hash is folded in |
| a struct | `struct S { public override int GetHashCode() => base.GetHashCode() ^ _x; }` | `ValueType.GetHashCode` is deterministic on the fields; mixing it is a performance concern, not a lookup bug |

The base call is bound to confirm it reaches `System.Object`'s implementation; a call that resolves to a base
class's own override is left alone.

## How to fix violations

Hash only the fields that `Equals` compares, and drop the `base.GetHashCode()` term:

```csharp
public override int GetHashCode() => System.HashCode.Combine(_amount, _currency);
```

There is no code fix: which fields belong in the hash — and how they should be combined — is the author's
design, and it must stay in step with `Equals`.

## Performance

The clean path is a syntactic prepass. Every invocation in the file is checked against a single pattern — a
`base.` receiver whose member is `GetHashCode` with no arguments — and all but that shape return at once. Only a
`base.GetHashCode()` that is not the member's whole result, sitting inside a parameterless `GetHashCode`
override, is bound, and the bind runs once to confirm the base is `object`.
