# SST2484 — A raw handle read through DangerousGetHandle is not reference-counted

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Code calls `SafeHandle.DangerousGetHandle()` and uses the returned `IntPtr`.

```csharp
public IntPtr Read(SafeHandle handle)
{
    IntPtr raw = handle.DangerousGetHandle(); // SST2484
    return raw;
}
```

## Rule description

A `SafeHandle` exists to keep an operating-system handle alive exactly as long as something references the
wrapper, and to release it exactly once. `DangerousGetHandle()` steps outside that guarantee: it returns the raw
`IntPtr` **without taking a reference count**. The instant it returns, nothing keeps the handle alive.

If another thread disposes the safe handle, or the finalizer runs because the safe handle is no longer reachable,
the operating system frees the underlying handle. The freed numeric value can then be reused for an unrelated
resource. Code still holding the raw `IntPtr` reads or writes a handle that has been freed — a crash — or one
that now belongs to a different object, silently corrupting it. The window is exactly the kind of race that
passes every local test and fails in production under load.

## How to fix violations

Hold a counted reference across the whole time the raw handle is in use, and release it in a `finally`:

```csharp
public void Read(SafeHandle handle)
{
    bool added = false;
    try
    {
        handle.DangerousAddRef(ref added);
        IntPtr raw = handle.DangerousGetHandle();
        UseRawHandle(raw);
    }
    finally
    {
        if (added)
        {
            handle.DangerousRelease();
        }
    }
}
```

Better still, avoid the raw value entirely: pass the `SafeHandle` itself to the P/Invoke or API that consumes
it, so the marshaller manages the reference count for you.

There is no code fix: the correct remedy is a reference-counting protocol whose shape depends on how long the
raw handle is needed and where, not a mechanical rewrite of the call.

## What this rule reports

Any call whose bound target is `SafeHandle.DangerousGetHandle` — through an instance, the abstract base type, a
null-conditional access, or an unqualified call to the inherited member from within a derived handle. A
same-named method on an unrelated type, or one that takes an argument, is not this API and is never reported.

## Performance

The clean path is a syntactic prepass: an invocation is considered only when it has no arguments and its invoked
member is named `DangerousGetHandle`; every other call returns before any binding. Only then is the call bound
once, to confirm it resolves to `SafeHandle`'s method. The whole rule is gated at compilation start on
`System.Runtime.InteropServices.SafeHandle` resolving, so a compilation with no safe handle type pays nothing.
