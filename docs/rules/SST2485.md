# SST2485 — A NotImplementedException should not be left in shipped code

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A member throws `new NotImplementedException(...)`, the placeholder the compiler accepts for code that has
not been written yet.

```csharp
public int Balance()
{
    throw new NotImplementedException(); // SST2485
}
```

## Rule description

`throw new NotImplementedException()` is a stub: it compiles, so the surrounding code builds and the type
looks complete, but the runtime has nothing to run. Any path that reaches the throw fails on the spot. Left
in a shipped build it turns an unfinished member into a crash waiting for the first caller — and because it
compiles cleanly, nothing else flags it.

Finish the member so it does the work its signature promises, or remove it until it is ready rather than
shipping a stub that only fails.

A `throw new NotSupportedException(...)` is a different thing and is **never** reported. It is a deliberate,
permanent statement that an operation is genuinely unsupported on this type — a read-only collection's
mutating members, an interface member that does not apply — not an unfinished stub.

## What this rule reports

A `throw` — statement or expression — whose operand is a `new` of the framework type
`System.NotImplementedException`. The message text passed to the constructor does not matter; a message
does not make an unimplemented member implemented.

These are never reported:

| shape | why |
| --- | --- |
| `throw new NotSupportedException(...)` | a deliberate, permanent "this operation is unsupported" signal |
| a rethrow — `throw;` | it re-raises an existing exception, it does not create one |
| `throw error;` where `error` is an existing value | the operand is not a `new` object creation |
| a project's own `NotImplementedException` in another namespace | the bind confirms it is a different type from the framework's |
| a different or generic exception type | the defect is the framework's `NotImplementedException`, nothing else |

## How to fix violations

Implement the member:

```csharp
public int Balance() => _entries.Sum(e => e.Amount);
```

If the member is not ready to be part of the public surface, remove it until it is, rather than shipping a
stub that throws.

There is no automatic code fix: only the author knows what the member is meant to do, so the correct body
cannot be generated.

## Performance

The clean path is syntactic: the throw's operand must be an object creation whose type, as written, has the
simple name `NotImplementedException`. A rethrow, a throw of an existing value, or any other exception name
returns before the semantic model is touched. Only a name match is bound, once, to confirm the framework's
`System.NotImplementedException`.
