# SST2486 — Prefer Assembly.Load over the path and partial-name load APIs

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes (LoadWithPartialName only) |
| Default | Enabled |

## Cause

An assembly is loaded through `Assembly.LoadFrom`, `Assembly.LoadFile`, or `Assembly.LoadWithPartialName` instead
of `Assembly.Load` with a full assembly display name.

```csharp
using System.Reflection;

var a = Assembly.LoadFrom("plugin.dll");
var b = Assembly.LoadFile(@"C:\plugins\plugin.dll");
var c = Assembly.LoadWithPartialName("Plugin");
```

## Rule description

Each of these three APIs loads an assembly outside the default, name-based binding context, and that is a
recurring source of duplicate-identity bugs that only surface at runtime:

- **`LoadFrom`** resolves through a separate *LoadFrom* binding context. A type it returns is a different identity
  from the same type loaded by the default context, so a cast, an `is` check, or an equality comparison between the
  two silently fails even though both came from the same file.
- **`LoadFile`** loads a file with no binding context at all. Two loads of one assembly produce two separate,
  non-unified identities, and its dependencies do not resolve through the normal probing path.
- **`LoadWithPartialName`** resolves a partial name to whichever installed assembly the runtime happens to find
  first. The result is nondeterministic across machines, and the API is deprecated.

`Assembly.Load` taking a full display name resolves through the default context, so the loaded assembly unifies
with everything else already loaded and its types compare equal as expected.

## What this rule reports

A call whose member name is `LoadFrom`, `LoadFile`, or `LoadWithPartialName` and whose bound method's containing
type is `System.Reflection.Assembly`. The message names the API and why it bites:

```csharp
Assembly.LoadFrom("plugin.dll");            // reported
Assembly.LoadFile("/opt/plugin.dll");       // reported
Assembly.LoadWithPartialName("Plugin");     // reported (and fixable)
```

A method of the same name on any other type is never reported, because the call is bound and its containing type
checked before anything is flagged. Only the qualified `Assembly.<Method>(...)` member-access form is inspected.

## How to fix violations

Load through the default context with a full display name:

```csharp
var assembly = Assembly.Load("Plugin, Version=1.0.0.0, Culture=neutral, PublicKeyToken=abc1234567890def");
```

The **code fix** is offered only for `LoadWithPartialName`, where the swap to `Assembly.Load` keeps the same string
argument and always compiles — `Assembly.Load(string)` exists on every target framework. `LoadFrom` and `LoadFile`
are reported without a fix, because moving off them changes the load context and can change behaviour: supply the
right full display name (and, where needed, an `AssemblyDependencyResolver` or an `AssemblyLoadContext`) by hand.

## Performance

The clean path is a syntax check. An invocation whose member name is not one of the three returns before anything
binds, and `System.Reflection.Assembly` is resolved once per compilation with the whole rule gated on it, so no
work and no allocation happen until a candidate call is found.
