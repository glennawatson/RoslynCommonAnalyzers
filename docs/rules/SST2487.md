# SST2487 — A [ConstructorArgument] should name a constructor parameter

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A property carries `[ConstructorArgument("name")]`, but the name matches no parameter of any constructor of
the declaring type.

```csharp
using System.Windows.Markup;

public class BindingExtension
{
    public BindingExtension(string path) => Path = path;

    [ConstructorArgument("paht")] // SST2487: no constructor takes 'paht'
    public string Path { get; set; }
}
```

## Rule description

A markup extension remembers which constructor argument a property was built from by tagging that property
with `[ConstructorArgument("name")]`. When the markup is written back out, the serializer looks up a
constructor parameter with that name and emits the property's value as the positional argument in its place —
that is what lets `{Binding path}` round-trip instead of expanding to a longer property form.

The name is a plain string. The compiler never resolves it against anything, so a typo, a renamed parameter,
or a copy from another type leaves it pointing at a parameter that does not exist. Nothing fails at build
time; the round-trip simply stops working, and the value is dropped or written the long way the next time the
object is serialized. The break only surfaces at run time, in whichever tool reads the markup back.

The rule reports a name that matches no parameter of any of the declaring type's constructors, comparing
exactly — the same comparison the serializer makes. A type whose only constructor is the implicit
parameterless one has no parameter for the name to map to, so it is reported too. The name is checked against
every constructor, so a name that belongs to any overload is accepted.

The check runs only when the markup attribute is present in the project and only against that exact attribute:
a same-named attribute from another namespace is left alone. There is no code fix, because the correct
parameter name is a judgement the tool cannot make — the intended target may be a renamed parameter, a
misspelling, or a constructor that still needs to be added.

## How to fix violations

Spell the name to match the constructor parameter it is meant to map to.

```csharp
[ConstructorArgument("path")]
public string Path { get; set; }
```

If the parameter was renamed, update the string to the new name. If no constructor takes the value yet, add
the parameter the property is meant to round-trip through.

## Performance

Nothing runs unless the markup attribute is present in the compilation, so a project that does not use it pays
nothing. For a project that does, the clean path is one comparison of the attribute's written name against a
constant and a check that its argument is a string literal; only an attribute that passes both is bound to
confirm its type and to read the declaring type's constructors.
