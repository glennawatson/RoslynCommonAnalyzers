# SST2488 — Do not log and rethrow the same exception

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `catch` block's only effect is to log the caught exception and then re-raise it with a bare `throw;`.

```csharp
try
{
    Process(order);
}
catch (Exception ex)
{
    _logger.LogError(ex, "Processing failed"); // SST2488
    throw;
}
```

## Rule description

This handler does two jobs at once, and neither completely. It logs the failure here, then re-raises the
exception unchanged so it keeps travelling up the stack — to be logged again by whatever finally handles it,
and again by anything in between that catches, logs, and rethrows the same way.

The result is one failure recorded as several. A single fault produces a cascade of near-identical entries,
each from a different layer, and the log that should point at the problem instead buries it. The stack trace
in each entry is the same trace, so the duplication adds noise without adding information.

A `catch` should own the exception or pass it on — not both:

- **Own it.** If this is the layer that decides what to do about the failure, log it and recover: return a
  fallback, retry, surface a friendly result. Do not rethrow.
- **Pass it on.** If a layer further out is the one that decides, let the exception propagate untouched. Do
  not log it here; the outer handler will, once, with the full context it has that this layer does not.

The rule reports a `catch` whose body is a logging call — or several — followed by a bare `throw;`, and
nothing else. A logging call is one made on a logger-typed value, or a `Log…` call handed the caught
exception. A handler that does real work between the log and the rethrow is left alone: that is handling, not
double-logging.

Two neighbouring shapes are deliberately **not** reported here, because each is its own defect:

```csharp
catch (Exception ex)
{
    throw; // a catch that only rethrows adds nothing — a separate concern
}

catch (Exception ex)
{
    _logger.LogError(ex, "failed");
    throw ex; // re-raising the variable resets the stack trace — a separate concern
}
```

## How to fix violations

Decide which handler owns the exception, then keep only that half.

Recover here, and drop the `throw;`:

```csharp
catch (Exception ex)
{
    _logger.LogError(ex, "Processing failed");
    return Result.Failed;
}
```

Or let it propagate to a single outer handler, and drop the logging:

```csharp
catch (Exception ex) when (Cleanup(ex))
{
    throw;
}

// ...far out, where the failure is actually decided:
catch (Exception ex)
{
    _logger.LogError(ex, "Processing failed");
    return Result.Failed;
}
```

There is **no code fix**: which half to keep is a design decision the analyzer cannot make. Only the code
knows whether this layer is meant to recover from the failure or defer to one above it.

## Performance

The clean path is purely syntactic. A `catch` is rejected unless its last statement is a bare `throw;` and
every statement before it is a call whose member name is a logging name — checks that touch only tokens. The
semantic model is consulted only for that rare shape, to confirm each call really is a logging call before the
diagnostic is raised.
