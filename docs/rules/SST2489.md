# SST2489 — Relational comparison is decided by the operand's type

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A relational comparison (`<`, `<=`, `>`, `>=`) tests an integer value against the edge of its own type's
range, so the result is fixed before the code runs.

```csharp
public bool IsInRange(uint length)
    => length >= 0; // SST2489 — a uint is never negative, so this is always true
```

## Rule description

Every integer type carries a fixed range. An unsigned value is never negative, so `x >= 0` is always true,
`x < 0` is always false, and `x > 0` can only be false when the value is zero — an `!= 0` test written as a
bounds check. A value compared to its own type's maximum is settled the same way: `b <= 255` is always true for
a `byte`, and `b > 255` is always false.

These comparisons read as real range checks, which is what makes them dangerous. A validation meant to reject
out-of-range input passes everything through; a branch meant to run for a boundary case never runs. The
comparison compiles, runs, and quietly does nothing.

Only a comparison whose constant sits **exactly** on the operand type's minimum or maximum is reported. A
constant past that edge — `b < 256` for a `byte` — is already a compiler diagnostic and is left to it. An
interior constant such as `u > 5` asks a real question and is never reported.

The bound may sit on either side, so `0 <= u` reads the same as `u >= 0`.

## What this rule reports

| Shape | Example | Verdict |
| --- | --- | --- |
| unsigned at least zero | `u >= 0` | always true |
| unsigned below zero | `u < 0` | always false |
| unsigned above zero | `u > 0` | really `u != 0` |
| at most the type maximum | `b <= 255`, `i <= int.MaxValue` | always true |
| above the type maximum | `b > 255` | always false |
| at the signed minimum | `i >= int.MinValue` | always true |
| below the signed minimum | `i < int.MinValue` | always false |
| above the signed minimum | `i > int.MinValue` | really `i != int.MinValue` |

`byte`, `sbyte`, `short`, `ushort`, `int`, `uint`, `long`, and `ulong` are reasoned about on both edges.
`nuint` is reasoned about only on its minimum (zero); its maximum is platform-dependent and not a compile-time
constant, and `nint` has neither edge fixed, so it is skipped.

## What this rule does not report

These shapes are correct — or are decided by something other than the operand's type — and are never reported:

| Shape | Example | Why it is clean |
| --- | --- | --- |
| a signed value against zero | `i >= 0`, `i < 0` | a signed value can be negative, so both are real tests |
| an interior bound | `u > 5`, `b <= 100` | the constant is inside the range and asks a real question |
| an unsigned at-most-zero | `u <= 0` | this is an `== 0` test, left to the equality it really is |
| a floating-point operand | `d >= 0.0` | `NaN` makes even a comparison against a constant meaningful |
| a constant past the range | `b < 256` | already reported by the compiler |
| a constant operand | `const uint z = 0; z >= 0` | the compiler already folds the whole comparison |

## How to fix

There is no automatic fix, because the intended bound is ambiguous — only the author knows which real check the
dead one stood in for. Decide what the comparison was meant to guard:

- If it was meant to be an equality or inequality test, write that: `u > 0` becomes `u != 0`.
- If it was meant to bound the value against something other than the type edge, use that real limit.
- If the type genuinely cannot fall out of range, the check is redundant and can be removed.

## Configuration

This rule has no configurable options.
