# SST2490 — Adjacent try statements with identical handling should be merged

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

Two `try` statements sit next to each other in the same block, and the second carries exactly the same
catch/finally handling as the first.

```csharp
try
{
    LoadPrimary();
}
catch (IOException ex)
{
    _log.Warn(ex);
}

try // SST2490
{
    LoadSecondary();
}
catch (IOException ex)
{
    _log.Warn(ex);
}
```

## Rule description

The two `try` statements guard different bodies but repeat the identical handling: the same catch clauses —
count, caught type, exception filter, and handler body — and the same finally clause. That duplication is a
maintenance trap. A later edit to one copy that misses the other leaves the two guarded regions protected
differently even though they read as a matched pair, and the repetition hides that both bodies were meant to
run under one guard.

Collapsing the pair into a single `try` that wraps both bodies keeps one copy of the handling:

```csharp
try
{
    LoadPrimary();
    LoadSecondary();
}
catch (IOException ex)
{
    _log.Warn(ex);
}
```

## What this rule reports

Only two `try` statements that are **immediate siblings** in the same block, whose handlers are
**syntactically identical** ignoring trivia, and whose shared handler does **real work**. In a run of three or
more identical `try` statements, each one after the first is reported against the `try` before it.

These shapes are left alone:

| shape | why it is clean |
| --- | --- |
| a different caught type, filter, catch body, catch count, or finally body | the handlers are not identical, so there is nothing to fold together |
| catches that differ only in the exception variable name | the two handlers are not syntactically the same, so a merge is not a mechanical rewrite |
| a `finally` present on only one of the two | the handling is not the same |
| two adjacent `try {} finally {}` with an empty finally | the finally guards nothing, so merging buys nothing |
| a bare `try {} catch {}` with no caught type, filter, or body | the catch guards nothing, so merging buys nothing |
| two identical `try` statements with a statement between them | they are not adjacent |

## How to fix violations

Wrap both bodies in one `try` and keep a single copy of the catch/finally handling.

There is no code fix. Merging the two bodies is not a mechanically safe edit: a local declared in the first
body and read in the second, or handling that must run between the two regions, can make the combined `try`
behave differently. The decision to merge — and how to reconcile the two bodies — is left to the author.

## Performance

The rule registers on the `try` statement and anchors on the first `try` of a pair, looking one statement
forward. The clean path — a `try` whose following sibling is not another `try` — is a parent-kind test and one
index compare and allocates nothing. The trivia-stripped handler comparison runs only for the rare
adjacent-`try` pair.
