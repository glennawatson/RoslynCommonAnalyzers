# SST2491 — Do not return a pending task from inside a using, lock, or try/finally

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A method that is not `async` returns a task directly while a `using`, a `lock`, or a `try`/`finally`
still governs a resource. The teardown runs the instant the method returns — before the returned task
completes.

```csharp
public Task<int> ReadAsync()
{
    using var stream = File.OpenRead(path);
    return JsonSerializer.DeserializeAsync<int>(stream).AsTask(); // SST2491
}
```

## Rule description

Returning a task hands control back to the caller immediately. Only then does the caller await it — but
by then the `using` has already disposed `stream`, the `lock` has already been released, or the `finally`
has already run. The asynchronous work is left to run against a disposed object, or without the lock it
was written to hold. It compiles, and it usually passes a quick test, because the race only shows when
the awaited operation actually suspends.

Making the method `async` and awaiting the call keeps the method on the stack until the work finishes, so
the resource is disposed — and the lock released — after completion, where it was meant to be:

```csharp
public async Task<int> ReadAsync()
{
    using var stream = File.OpenRead(path);
    return await JsonSerializer.DeserializeAsync<int>(stream).AsTask();
}
```

### What is reported

- A `return` of a `Task`, `Task<T>`, `ValueTask`, or `ValueTask<T>` from a non-async method or local
  function, when the `return` is lexically inside a `using` statement, a `using` declaration, a `lock`,
  or the body of a `try` that has a `finally`.

### What is not reported

- An already-`async` method — awaiting there already orders the teardown correctly.
- A return of an already-completed task: `Task.CompletedTask`, `Task.FromResult(...)`,
  `Task.FromException(...)`, `Task.FromCanceled(...)`, a `new ValueTask(...)`, or a `null`/`default`.
  A completed task has nothing pending to outrace the teardown.
- A return that is not inside any of those teardown scopes.

## How to fix violations

The fix makes the method `async` and awaits each returned task. A generic `Task<T>`/`ValueTask<T>` return
becomes `return await X;`; a non-generic `Task`/`ValueTask` return becomes `await X; return;`.

The fix is **not** offered when the return sits inside a `lock` body, because `await` is not allowed
there. That case is a genuine design problem — the work was never actually protected by the lock — and
the repair (capture the task, leave the lock, then await, or move the async work out of the locked
region) is left to the author.

## Performance

The rule registers on the `return` statement. The clean path is syntactic: a `return` with no
expression, or one whose expression is an already-completed task shape, is dropped without binding, and a
single upward walk finds both the enclosing function and any teardown scope. Only a return that clears
all of that is bound once, to confirm the function is a non-async method returning a task type. The task
types are resolved once per compilation, so a project without `System.Threading.Tasks` pays nothing.
