# SST2493 — Compare an unconstrained type parameter with 'is null', not '== null'

| Item | Value |
| --- | --- |
| Category | Correctness |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

A value of an unconstrained type parameter is compared to null with `==` or `!=`.

```csharp
public static bool IsMissing<T>(T value) => value == null; // SST2493
```

## Rule description

Because the type parameter has no `class`, `struct`, or `notnull` constraint, it can be substituted with
a value type — and there the operator does not mean what it reads. The compiler resolves `value == null`
to a reference comparison, which is always false for a non-nullable value type, however the value was
produced. So `IsMissing(0)` and `IsMissing(default(int))` both return false: the check silently answers
the wrong question for half the types it accepts.

The constant pattern `is null` / `is not null` is correct for every substitution — it compares against
the default for value types and against a null reference for reference types — and never boxes:

```csharp
public static bool IsMissing<T>(T value) => value is null;
```

### What is reported

- `x == null` and `x != null` (in either operand order) where `x` binds to a type parameter that has no
  `class`, `struct`, `notnull`, or `unmanaged` constraint and no class base-type constraint — that is, a
  parameter a value type could be substituted for.

### What is not reported

- A parameter constrained with `class`, `notnull`, or a class base type: it is always a reference type,
  where `== null` is a correct reference comparison.
- A parameter constrained with `struct`/`unmanaged`, or any non-generic operand. These are a different
  question and are left to the operators the author chose.
- `.Equals(null)` and comparer-based comparisons, which are about boxing rather than the operator's
  meaning.

## How to fix violations

The fix rewrites `x == null` to `x is null` and `x != null` to `x is not null`, keeping the operand.

## Performance

The rule registers on equality expressions. The clean path is syntactic: an equality whose other operand
is not the null literal is dropped without binding. Only a genuine null comparison binds its non-null
operand once, and reports only when that operand's type is a type parameter a value type could satisfy.
