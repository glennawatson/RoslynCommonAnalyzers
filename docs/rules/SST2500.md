# SST2500 — A test method should verify something

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method carrying a test attribute contains no assertion and declares no expected exception, so it
runs, passes, and verifies nothing.

```csharp
[Fact]
public void Adds() // SST2500: this test asserts nothing
{
    var result = 1 + 1;
}
```

## Rule description

A test that never asserts and never declares an expected exception always passes. It gives the coverage
report a green tick and the suite a passing count, but it locks in no behaviour: the code it appears to
cover can regress freely and the test stays green. That is worse than no test, because it reads as
protection that is not there.

A test method is one carrying a test attribute from a supported framework: xUnit (`Fact`, `Theory`),
NUnit (`Test`, `TestCase`, `TestCaseSource`, `Theory`), MSTest (`TestMethod`, `DataTestMethod`), or TUnit
(`Test`). The rule is gated on one of these attributes resolving in the compilation; a project that
references no supported framework is never analysed.

## What this rule reports

The rule is deliberately conservative to keep false positives near zero. Proving that a body verifies
nothing would otherwise require following every method it calls (interprocedural analysis), which is not
done here. Instead the rule reports **only** when it can prove the body verifies nothing from the body
alone: every invocation and object creation in the body must bind to a non-verifying platform (BCL)
method, or the body must contain no calls at all.

A call counts as a non-verifying platform call when its declaring assembly is a platform assembly —
`mscorlib`, `netstandard`, `System.Private.CoreLib`, or an assembly named `System` or beginning
`System.` — and it is **not** one of the in-BCL verification helpers `System.Diagnostics.Debug`,
`System.Diagnostics.Trace`, or `System.Diagnostics.Contracts.Contract`.

```csharp
[Fact] public void A() { }                                  // reported: empty body
[Fact] public void B() { var x = 1 + 1; _ = x; }            // reported: only local computation
[Fact] public void C() { System.Console.WriteLine("hi"); }  // reported: only a platform call
[Fact] public void D()                                      // reported: only platform construction
{
    var builder = new System.Text.StringBuilder();
    builder.Append("x");
}
```

Anything the rule cannot prove is a harmless platform call keeps the method **silent**:

| shape | why it is silent |
| --- | --- |
| `Assert.Equal(2, sum)`, `Assert.That(...)`, `Assert.Throws<T>(...)`, `Record.Exception(...)` | the framework's assertion assembly is not a platform assembly |
| `result.Should().Be(2)`, `result.ShouldBe(2)` | a third-party fluent assertion is not a platform call |
| `Helper.CheckInvariants()` | a call into the user's own code might itself assert |
| `_ = new MyType()` | constructing a non-platform type might run verifying logic |
| `throw new NotImplementedException()` | a deliberately pending test does not pass, so it is not "asserts nothing" |
| `[ExpectedException(typeof(...))]` | the test verifies by asserting that an exception is thrown |
| a call that does not bind | the rule cannot prove what an unresolved call does |
| any method with no test attribute | it is not a test |

Because a single user or third-party call silences the whole method, the rule catches empty and
setup-only tests rather than every conceivable non-asserting test — a deliberate trade for near-zero
false positives.

## How to fix violations

Add an assertion for the behaviour the test is meant to pin down, declare the exception it expects, or
delete the empty test if it is a leftover stub.

```csharp
[Fact]
public void Adds()
{
    var result = 1 + 1;
    Assert.Equal(2, result);
}
```

## Performance

The clean path is a syntactic prepass: a method is skipped unless one of its attributes is spelled with a
known test-attribute simple name, so an ordinary method never binds. The framework marker types are
resolved once per compilation. Only a method that clears the name check is bound and has its body walked,
and that walk stops at the first call it cannot prove is a platform call.
