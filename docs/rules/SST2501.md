# SST2501 — An equality or identity assertion should not compare an expression with itself

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An equality or identity assertion is handed the same expression on both sides, so both operands always
evaluate to the same value. A positive assertion (`Equal`, `StrictEqual`, `AreEqual`, `Same`, `AreSame`, or
`Assert.That(x, Is.EqualTo(x))`/`Is.SameAs(x)`) then passes for every possible value; a negated assertion
(`NotEqual`, `NotSame`, `AreNotEqual`, `AreNotSame`) fails for every possible value.

```csharp
[Fact]
public void Total_is_correct()
{
    var total = Sum(order);
    Assert.Equal(total, total); // SST2501: always passes and verifies nothing
}
```

## Rule description

A test earns its keep by distinguishing a correct result from a wrong one. An assertion whose two operands
are the same expression cannot do that: because both sides read the same value, its outcome is fixed before
the code under test ever runs. That almost always means one operand was meant to be a different value — the
expected constant, a second instance, or the result of a separate call — and the mistake left the test either
permanently green (a positive assertion) or permanently red (a negated one). Either way the assertion is
noise, and a permanently-green one is the more dangerous because it looks like coverage.

The two operands are compared syntactically, so the value is irrelevant — only that both sides are the same
expression.

## What this rule reports

The equality and identity assertions of the common test frameworks, when both compared operands are the same
expression:

```csharp
// xUnit
Assert.Equal(x, x);
Assert.StrictEqual(x, x);
Assert.Same(a, a);
Assert.NotEqual(x, x);   // always fails
Assert.NotSame(a, a);    // always fails

// NUnit (classic and constraint model)
Assert.AreEqual(total, total);
Assert.AreSame(a, a);
Assert.That(x, Is.EqualTo(x));
Assert.That(a, Is.SameAs(a));

// MSTest
Assert.AreEqual(5, 5);
Assert.AreSame(a, a);
Assert.AreNotEqual(x, x); // always fails
Assert.AreNotSame(a, a);  // always fails
```

A member or element access counts as the same expression on both sides (`Assert.Equal(order.Total, order.Total)`),
because a field or property read yields the same value each time.

The rule stays silent whenever the two sides need not agree, or when the shape is not a framework assertion:

| shape | why it is not reported |
| --- | --- |
| two different operands | `Assert.Equal(expected, actual)` is the normal, correct case |
| a call on either side | `Assert.Equal(Next(), Next())` — two calls need not return the same value |
| an object creation on either side | `Assert.Same(new Box(), new Box())` — each side is a distinct instance |
| an `await`, assignment, or increment on either side | the operand has a side effect or can vary between reads |
| a non-`EqualTo`/`SameAs` constraint | `Assert.That(x, Is.GreaterThan(x))` is a real comparison |
| named arguments | skipped, because a name can reorder the operands |
| a same-named method that is not a framework `Assert` | the call does not bind to a resolved `Assert` type |

The whole rule is gated on at least one framework `Assert` type resolving, so a project that references no test
framework is never analyzed.

## How to fix violations

Replace one operand with the value it was meant to compare against — the expected constant, the value produced
by the code under test, or a separately-constructed instance:

```csharp
var total = Sum(order);
Assert.Equal(6, total);          // compare against the expected value
Assert.NotSame(original, copy);  // compare two distinct instances
```

## Performance

The clean path is syntax only: every invocation is seen, but all but the equality/identity assertion names are
rejected before anything binds, and the call is bound only once the two operands are syntactically equivalent.
The framework `Assert` types are resolved once per compilation, and a call, object creation, `await`, or
assignment on either side stops the report without binding.
