# SST2502 — The expected value should be the first argument to an equality assertion

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled |

## Cause

An equality assertion is given a compile-time constant as its actual (second) argument and a computed value
as its expected (first) argument, so the two have been reversed.

```csharp
[Fact]
public void M(int result)
{
    Assert.Equal(result, 42); // SST2502: the constant belongs first
}
```

## Rule description

These assertions take the expected value first and the actual value second, and print a failure as
"expected *first*, actual *second*":

- `Assert.Equal(expected, actual)` — xUnit
- `Assert.AreEqual(expected, actual)` — MSTest and the classic NUnit form

The value a test knows in advance — a literal, a `const`, a `nameof`, an enum member — is the expected value
and belongs in the first slot. The value the code produced is the actual value and belongs in the second. When
the constant sits second and a computed value sits first, the two are almost certainly the wrong way round: a
failing test then claims it *expected* the value the code produced and *got* the constant it was written
around, which is backwards and sends the reader looking in the wrong place.

Being wrong about this would be worse than saying nothing, since the fix reorders a call in a passing test. So
the rule reports only the one shape that is a genuine reversal, and stays silent everywhere else:

- **The actual (second) argument is a constant** — a literal, a `const` reference, a `nameof`, an enum member,
  or any other compile-time constant expression.
- **The expected (first) argument is not a constant** — it is a computed value.

Every other shape is left alone:

| shape | example | why it is clean |
| --- | --- | --- |
| the constant is already first | `Assert.Equal(42, result)` | the correct, common form |
| both arguments are constant | `Assert.Equal(1, 2)` | nothing marks one side as the actual |
| both arguments are computed | `Assert.Equal(first, second)` | neither is the value the test knew in advance |
| the arguments are named | `Assert.Equal(expected: result, actual: 42)` | the order is explicit at the call site |
| the fluent, actual-first form | `Assert.That(result, Is.EqualTo(42))` | actual comes first here by design — not a reversal |

The rule is gated on one of the assertion host types resolving in the compilation; a project that references
none of them is never analysed. A call to a same-named method on any other type is not an assertion and is not
reported.

## How to fix violations

Put the expected value first.

```csharp
Assert.Equal(42, result);
```

The **code fix** swaps the two arguments and nothing else: each keeps the trivia around it, so a call spread
over several lines keeps its shape. The two arguments already bind to the same expected and actual parameters
whichever order they are in, so the reordered call still compiles.

If the second argument really was meant to be the expected value, name the arguments to make the order
explicit at the call site, which also silences the rule.

## Performance

The clean path is a syntactic prepass — a two-argument, positional call whose invoked name is `Equal` or
`AreEqual` — and the constant shape is checked before the call is ever bound, so the correct constant-first
assertions are pruned without overload resolution. The assertion host types are resolved once per compilation.
