# SST2503 — An equality assertion against a boolean literal should use the boolean assertion

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | Yes (the framework's boolean assertion) |
| Default | Enabled |

## Cause

An equality assertion is handed a boolean literal — `true` or `false` — as one of its two operands, comparing
a value against the literal instead of asserting the condition directly.

```csharp
Assert.AreEqual(true, order.IsPaid); // SST2503: use the boolean assertion
Assert.Equal(false, list.Contains(x)); // SST2503
```

## Rule description

Comparing a value to a boolean literal states the test's intent through the wrong shape. The dedicated boolean
assertion — `Assert.IsTrue`/`Assert.IsFalse` on MSTest and NUnit, `Assert.True`/`Assert.False` on xUnit — says
directly that the value under test must hold. When it fails it reports the *condition*, whereas the equality
form reports an expected-versus-actual value mismatch (`Expected: True, Actual: False`) that reads as a plain
comparison and hides what was really being checked.

A `true` literal maps to the affirmative assertion and a `false` literal to its negation. The literal is
recognised in either argument position, so both `Assert.Equal(true, x)` and the transposed `Assert.Equal(x, true)`
are reported.

## What this rule reports

The call must resolve to a recognised test framework's `Assert` type and compare a boolean value against a
boolean literal:

| framework | equality call | becomes |
| --- | --- | --- |
| xUnit | `Assert.Equal(true, x)` | `Assert.True(x)` |
| xUnit | `Assert.Equal(false, x)` | `Assert.False(x)` |
| MSTest / NUnit | `Assert.AreEqual(true, x)` | `Assert.IsTrue(x)` |
| MSTest / NUnit | `Assert.AreEqual(false, x)` | `Assert.IsFalse(x)` |

The following are never reported:

| shape | reason |
| --- | --- |
| `Assert.AreEqual(3, count)` | neither operand is a boolean literal |
| `Assert.AreEqual(true, o)` where `o` is not a boolean | the value under test is not a boolean, so the boolean assertion would not apply |
| `Assert.AreEqual(true, b, "message")` | only the two-argument equality form is rewritten |
| a same-named call on an `Assert` type outside a recognised namespace | it is not a test-framework assertion |

## How to fix violations

Call the framework's boolean assertion with the value under test and drop the literal:

```csharp
Assert.IsTrue(order.IsPaid);
Assert.False(list.Contains(x));
```

The **code fix** rewrites the call, replacing the equality method with the affirmative or negative boolean
assertion and keeping only the value under test. It is offered only when that boolean assertion resolves in the
compilation, so the rewritten call always binds.

## Performance

The whole rule is gated at compilation start on at least one recognised `Assert` type resolving, so a project
that references no test framework registers nothing. The clean path is a syntactic prepass: the invoked name must
be `Equal` or `AreEqual`, the call must have exactly two arguments, and one of them must be a `true`/`false`
literal. Nothing binds until that holds.
