# SST2504 — A test class should declare at least one test method

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A concrete class is marked as a test fixture but declares no test method, so the runner loads and
instantiates it yet has nothing to run.

```csharp
[TestClass]
public class OrderServiceTests // SST2504: no test method
{
    private readonly OrderService _service = new();

    public void SetUp() { }
}
```

## Rule description

Marking a class as a test fixture — MSTest's test-class attribute or NUnit's test-fixture attribute —
tells the runner to discover it, instantiate it, and reflect over its methods for tests. A fixture that
declares no test method gives it nothing to do: the runner finds no test and moves on. Such a class is
almost always a leftover. The tests it once held were deleted or moved, a base class was mixed up, or the
per-method test attributes were dropped in an edit — and it lingers as dead weight that reads like
coverage while verifying nothing.

The rule is scoped to the frameworks whose test classes are marked explicitly. It is gated on at least
one of those class-attribute markers resolving in the compilation, so a project that references neither
pays nothing.

| Framework | Class marker | Method markers that count as a test |
| --- | --- | --- |
| MSTest | test-class attribute | test-method and data-test-method attributes |
| NUnit | test-fixture attribute | test, test-case, test-case-source, and theory attributes |

xUnit has no test-class attribute — an xUnit test class is implicit in containing a fact or theory — so
this rule does not apply to xUnit.

## What this rule reports

A concrete class carrying a test-class marker that declares no test method of its own and inherits none:

```csharp
[TestClass]
public class Fixture // reported
{
    public void Helper() { }
}

[TestFixture]
public class OtherFixture // reported
{
}
```

These shapes are deliberately left alone:

| shape | why |
| --- | --- |
| an abstract fixture | a legitimate shared base others derive their tests from |
| a fixture with a test method | it has something to run |
| a fixture deriving from a base with test methods | it is exercised through the inherited tests |
| a fixture deriving from another test-fixture base | it specializes a shared fixture rather than being a leftover |
| a class with no test-class attribute | not in scope, including every xUnit class |

The base chain is walked, and a class whose base type either carries a test-class attribute or declares a
test method is treated as exercised. When the shape cannot be told apart cheaply, the rule stays silent
rather than risk a false report.

## How to fix violations

Add the test that belongs in the fixture, or — if the class is genuinely a leftover — delete it. If the
class only holds shared setup for derived fixtures, make it `abstract` so it reads as a base rather than a
runnable fixture.

## Performance

The clean path is a syntactic prepass: a class that does not carry a test-class attribute by name, or that
is abstract, is dismissed before anything binds. Only a class that syntactically looks like a fixture is
bound — its attribute confirmed against a resolved marker, and its own members and base chain examined —
and the class-attribute markers are resolved once per compilation.
