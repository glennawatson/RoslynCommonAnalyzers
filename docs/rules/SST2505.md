# SST2505 — A parameterized test declares no data source

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A test method carries a test attribute and declares one or more parameters, but nothing supplies those
parameters — there is no data source on the method or on any of its parameters.

```csharp
[Theory]
public void Adds(int a, int b) // SST2505: no [InlineData]/[MemberData]/[ClassData]
{
    Assert.Equal(3, a + b);
}
```

## Rule description

A test method with parameters is data-driven: the runner needs a data source to fill those parameters
for each case. When a parameterized test has no data source, there is nothing to run it with, so
depending on the framework the case is skipped, errors, or silently never executes. Either way the
assertions inside are never reached, and the test appears to pass while checking nothing.

The data source each framework expects differs:

| Framework | A data source is | Recognized by |
| --- | --- | --- |
| xUnit | `[InlineData]`, `[MemberData]`, `[ClassData]`, or a custom data attribute | deriving from the xUnit data-attribute base |
| NUnit | `[TestCase]` / `[TestCaseSource]` on the method, or `[Values]` / `[Range]` / `[Random]` on a parameter | implementing the NUnit test-builder or parameter-data-source interface |
| MSTest | `[DataRow]` or `[DynamicData]` | implementing the MSTest test-data-source interface |
| TUnit | `[Arguments]`, `[MethodDataSource]`, `[ClassDataSource]`, or a data-source generator | implementing the TUnit data-source interface |

## What this rule reports

Only a method that carries a recognized test attribute, declares at least one parameter, and has no
recognized data source anywhere on it. These are reported:

```csharp
[Theory] public void A(int value) { }                    // xUnit theory, no data attribute
[Fact]   public void B(int value) { }                    // xUnit fact should have no parameters
[Test]   public void C(int value) { }                    // NUnit test, no [TestCase]/[Values]
[TestMethod] public void D(int value) { }                // MSTest, no [DataRow]/[DynamicData]
[DataTestMethod] public void E(int value) { }            // MSTest data test, no row source
```

A parameterless test, or a parameterized test that already has any data source, is never reported:

```csharp
[Fact]   public void A() { }                             // no parameters
[Theory] [InlineData(1)] public void B(int value) { }    // xUnit data attribute present
[TestCase(1)] public void C(int value) { }               // NUnit case supplies the argument
[Test]   public void D([Values(1, 2)] int value) { }     // NUnit per-parameter source
[DataTestMethod] [DataRow(1)] public void E(int value) { } // MSTest row source present
```

The rule is gated on at least one test-attribute type resolving in the compilation, and each attribute
that decides the outcome is bound before it is trusted — a same-named attribute from an unrelated
namespace does not count as a test, and does not count as a data source.

## How to fix violations

Give the parameters a data source appropriate to the framework, or remove the parameters if the test was
never meant to be data-driven:

```csharp
[Theory]
[InlineData(1, 2, 3)]
public void Adds(int a, int b, int expected)
{
    Assert.Equal(expected, a + b);
}
```

## Performance

The clean path is a syntactic prepass: a method must declare a parameter and carry an attribute written
with a known test-attribute name before anything binds. Only then are the attributes bound to confirm a
real test and the absence of a data source. The framework types are resolved once per compilation, and
when no test framework is referenced the rule registers no callbacks at all.
