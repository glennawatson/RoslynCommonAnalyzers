# SST2506 — A test should not pause on a fixed real-time delay

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A test method's body calls `System.Threading.Thread.Sleep`, pausing the thread for a fixed real-time delay.

```csharp
[Fact]
public void Processes_the_queue()
{
    _queue.Enqueue(item);
    System.Threading.Thread.Sleep(100); // SST2506: a fixed delay races the wall clock
    Assert.Empty(_queue);
}
```

## Rule description

A test that sleeps pauses for a fixed real-time delay, and that delay is spent on every run. A suite full of
these sleeps grows slower with every test that adds one, whether or not the work being waited on is already
done.

Worse, the delay is a guess. It is either longer than the work needs — wasted time — or, on a loaded or slow
machine, shorter than the work needs, at which point the test observes a half-finished state and fails
intermittently. A sleep is the classic source of a flaky test, because it races a wall clock instead of waiting
for the thing the test actually depends on.

Wait for that condition directly instead: poll it, await the task or signal that completes it, or advance a fake
clock the code under test reads. The test then proceeds the instant the work is done and never a moment before.

## What this rule reports

A call that binds to `System.Threading.Thread.Sleep` inside a method marked as a test, including a call nested in
an `if`, a loop, a lambda, or a local function:

```csharp
[Fact]
public void Retries()
{
    if (!ready)
    {
        Thread.Sleep(500); // reported — the nesting does not hide it
    }
}
```

The method is recognised as a test by its attribute — xUnit (`[Fact]`, `[Theory]`), NUnit (`[Test]`,
`[TestCase]`, `[TestCaseSource]`, `[Theory]`), MSTest (`[TestMethod]`, `[DataTestMethod]`), or TUnit (`[Test]`) —
and the whole rule stays silent unless one of those frameworks is referenced.

Shapes that are never reported:

| shape | why |
| --- | --- |
| `Thread.Sleep` in a non-test method | production code may legitimately sleep |
| a call to a same-named `Sleep` method of your own | it is bound and found not to be `Thread.Sleep` |
| any test when no test framework is referenced | the marker attributes do not resolve, so the rule does nothing |

## How to fix violations

Replace the fixed delay with a wait for the actual condition. There is no automatic fix, because the right wait
depends on what the test is waiting for:

- await the `Task`, `TaskCompletionSource`, or signal that the work completes.
- poll the condition with a bounded timeout (for example, a spin-wait helper that returns as soon as it is true).
- inject a controllable time source (a fake clock / `TimeProvider`) and advance it, so no real time passes at all.

## Limitations

- The rule matches an attribute by its simple name (`Fact`, `Theory`, `Test`, `TestCase`, `TestCaseSource`,
  `TestMethod`, `DataTestMethod`), then binds it to confirm the framework type. A custom attribute that derives
  from one of those markers but is applied under a *different* name is not recognised.
- Only `Thread.Sleep` is reported. An awaited `Task.Delay` with a constant delay is the same defect in
  asynchronous form, but it is intentionally left to a separate rule so this one stays a single, unambiguous
  shape.

## Performance

The whole rule is gated at compilation start on `System.Threading.Thread` resolving and at least one test-marker
attribute resolving; a project with no test framework registers nothing. Per method, the clean path is syntax
only: a method is skipped unless one of its attributes is spelled like a test marker, and its body is then walked
once for a `Sleep`-named call. Nothing binds until such a call is found — only then is the marker attribute bound
to confirm the method is a test and the call bound to confirm it is `Thread.Sleep`.
