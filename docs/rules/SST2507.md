# SST2507 — A test's expected exception should be asserted on the specific operation

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A test method declares its expected failure with an expected-exception attribute — MSTest's
`[ExpectedException(typeof(T))]` or NUnit's legacy `[ExpectedException]` — instead of asserting the
specific operation with `Assert.Throws<T>`.

```csharp
[TestMethod]
[ExpectedException(typeof(InvalidOperationException))] // SST2507
public void Withdraw_over_balance_throws()
{
    var account = new Account(balance: 100);
    account.Deposit(50);          // if THIS threw, the test would still pass
    account.Withdraw(200);
}
```

## Rule description

An expected-exception attribute asserts only that *some* statement, anywhere in the whole method body,
threw the named exception. It does not say which statement threw, and it does not say the operation
under test was the one that threw. A setup line, an arrange step, or a mistyped call that happens to
throw the same exception type passes the test just as well, so the test keeps reporting green while
asserting nothing about the behaviour it was written to pin down.

The precise form wraps the single operation expected to fail in `Assert.Throws<T>(() => operation)`,
which asserts that that call — and only that call — threw:

```csharp
[TestMethod]
public void Withdraw_over_balance_throws()
{
    var account = new Account(balance: 100);
    account.Deposit(50);

    Assert.ThrowsException<InvalidOperationException>(() => account.Withdraw(200));
}
```

## What this rule reports

The attribute is reported on any method the compilation also recognises as a test — a method carrying
`[TestMethod]`, `[DataTestMethod]`, `[Fact]`, `[Theory]`, `[Test]`, `[TestCase]`, `[TestCaseSource]`, or
`[TUnit.Core.Test]`:

```csharp
[TestMethod]
[ExpectedException(typeof(ArgumentNullException))]                 // reported
public void A() { ... }

[Test]
[ExpectedException]                                                 // reported (NUnit legacy)
public void B() { ... }
```

The following are never reported:

| shape | why |
| --- | --- |
| a test using `Assert.Throws` / `Assert.ThrowsException` | the specific operation is already asserted |
| a non-test method carrying the attribute | it is not a test |
| an unrelated attribute that merely shares the `ExpectedException` name | it is not the framework's attribute |
| any code when no expected-exception attribute type is referenced | modern NUnit (3+) removed the attribute, so the rule does nothing |

The whole rule is gated on an expected-exception attribute type resolving in the compilation, so a
project that references neither MSTest's attribute nor NUnit's legacy one pays nothing.

## How to fix violations

Remove the attribute and wrap the single operation expected to fail in `Assert.Throws<T>` (xUnit,
NUnit) or `Assert.ThrowsException<T>` (MSTest). No code fix is offered: turning the attribute into an
assertion means choosing which statement in the body is the one expected to throw, and only the test's
author knows that.

## Performance

The clean path is a syntactic prepass: a method must syntactically carry an attribute named
`ExpectedException` before anything binds. Only then is the method symbol resolved and its attributes
inspected. The expected-exception and test-marker attribute types are resolved once per compilation, and
a compilation with no expected-exception attribute registers no per-method callback at all.
