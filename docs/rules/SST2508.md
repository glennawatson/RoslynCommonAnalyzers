# SST2508 ‚Äî A fluent assertion is started but never completed

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An expression statement is a bare `value.Should()` call ‚Äî a fluent assertion that names its subject but
never chains a check onto it, so it verifies nothing.

```csharp
[Fact]
public void Balance_is_updated()
{
    var account = new Account(balance: 100);
    account.Deposit(50);

    account.Balance.Should();   // SST2508 ‚Äî names the subject, asserts nothing
}
```

## Rule description

A fluent assertion is written in two halves: `value.Should()` names the subject, and a chained check ‚Äî
`Be`, `Contain`, `BeTrue`, `BeEquivalentTo`, and the like ‚Äî states what must hold. Only the chained check
asserts anything. A `value.Should();` written as a whole statement stops at the subject: it compiles,
runs, and passes while checking nothing, so a regression in the code it appears to cover slips through
green.

The completed form chains the check that was intended:

```csharp
[Fact]
public void Balance_is_updated()
{
    var account = new Account(balance: 100);
    account.Deposit(50);

    account.Balance.Should().Be(150);
}
```

## What this rule reports

Only the proven-incomplete shape: an expression statement whose whole expression is a `Should()`
invocation with nothing chained after it, once the call binds to a subject type from a fluent-assertion
library (its type's root namespace is `FluentAssertions` or `AwesomeAssertions`).

```csharp
result.Should();                 // reported
result.Should().Be(expected);    // clean ‚Äî a check is chained
```

The following are never reported:

| shape | why |
| --- | --- |
| `value.Should().Be(...)` (any chained check) | the subject already carries a check |
| a `Should()` result that is stored or returned (`var a = x.Should();`) | it is not a dangling statement |
| a `Should()` from any other library or a project's own extension | it is not a fluent-assertion subject |
| any code when no fluent-assertion library is referenced | there is nothing to complete |

The whole rule is gated on a fluent-assertion library being referenced ‚Äî the `AssertionExtensions` type
that hosts the `Should()` extension methods, under either the `FluentAssertions` or the
`AwesomeAssertions` namespace, must resolve ‚Äî so a project that references neither pays nothing.

## How to fix violations

Chain the check the subject was meant to carry (`.Be(...)`, `.Contain(...)`, `.BeTrue()`, and so on), or
delete the dangling statement if it was left over. No code fix is offered: only the test's author knows
which check the subject was meant to state.

## Performance

The clean path is syntax only: the statement's expression must be an invocation whose invoked member is
named `Should` ‚Äî which, being the top-level invocation, also proves nothing is chained after it ‚Äî before
anything binds. Only then is the call bound to confirm its return type is a fluent-assertion subject. A
compilation that references no fluent-assertion library registers no per-statement callback at all.
