# SST2509 — A test method has a shape the runner cannot execute

| Item | Value |
| --- | --- |
| Category | Testing |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A method carries a test attribute but has a signature the runner will not execute, so it is discovered as a
test and then silently skipped — it looks green because it never runs and its assertions are never reached.

```csharp
[Fact] private int Adds() => 1 + 2; // SST2509: not public, and returns int
```

## Rule description

A test runner discovers a method by its test attribute and then decides whether it can execute it. Three
shapes stop it from running, and every one of them fails quietly — the method appears in no failure report
because its body never runs:

| Shape | Why the runner skips it |
| --- | --- |
| Not public | xUnit, NUnit, and MSTest discover only public test methods; a non-public one is never collected. |
| Generic with no parameters | There is no argument from which the type parameter can be inferred, so no framework can close and run the method. |
| Returns something other than void or an awaitable | Only `void`, `Task`, `ValueTask`, `Task<T>`, and `ValueTask<T>` are run or awaited; any other return value is left unobserved and the case does not run. |

`static` is **not** reported: some frameworks run static test methods, so a static test method is a valid
shape here.

The public requirement is applied only for xUnit, NUnit, and MSTest. TUnit is exempt from the public check
because a public method is not a universal requirement there; its generic and return-type shapes are still
checked.

## What this rule reports

```csharp
[Fact]   private void A() { }            // not public — never collected
[Test]   internal void B() { }           // not public — never collected
[Fact]   public void C<T>() { }          // parameterless generic — T cannot be inferred
[Fact]   public int D() => 0;            // returns int — not void/Task/ValueTask
[Test]   public string E() => "";        // returns string — not void/Task/ValueTask
```

These are never reported:

```csharp
[Fact]   public void A() { }                                   // the runnable shape
[Fact]   public static void B() { }                            // static is allowed
[Fact]   public System.Threading.Tasks.Task C() => default;    // awaitable return
[Fact]   public System.Threading.Tasks.ValueTask<int> D() => default; // awaitable return
[Fact]   public void E<T>(T value) { }                         // generic with a parameter — T can be inferred from data
[Test]   private void F() { }                                  // TUnit is exempt from the public check
```

A generic test method that declares parameters is left alone: its type argument can be inferred from a data
source, and whether it actually has one is a separate concern.

The rule is gated on at least one test-attribute type resolving in the compilation, and the attribute that
marks a method as a test is bound before it is trusted — a same-named attribute from an unrelated namespace
does not count as a test.

## How to fix violations

Give the method the shape the runner runs: make it `public`, drop the type parameter (or move it onto a
data-driven parameter the framework can infer), and return `void`, `Task`, or `ValueTask`:

```csharp
[Fact]
public async System.Threading.Tasks.Task Adds()
{
    Assert.Equal(3, 1 + 2);
    await System.Threading.Tasks.Task.CompletedTask;
}
```

## Performance

The clean path is a syntactic prepass: a method must carry an attribute written with a known test-attribute
name and must not already be public, non-generic, and `void`-returning before anything binds. Only a method
that looks like a test with a suspect shape is bound to confirm a real test attribute and classify the
violation. The framework types are resolved once per compilation, and when no test framework is referenced
the rule registers no callbacks at all.
