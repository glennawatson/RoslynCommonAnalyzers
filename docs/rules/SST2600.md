## Cause

Application output is written through `System.Diagnostics.Trace.Write`, `WriteLine`, `WriteIf`, or
`WriteLineIf`, in a project that already has a structured-logging abstraction (`ILogger`) available.

```csharp
public void Process(Order order)
{
    Trace.WriteLine("processing order " + order.Id); // SST2600: legacy tracing, not structured logging
}
```

## Rule description

Legacy tracing emits a flat string with no severity, no category, and no structured fields, routed only to
whatever `TraceListener` happens to be attached. Structured logging through `ILogger` keeps a level a sink can
filter on, a category a sink can route on, and named values a sink can index and query, so the same message
survives as data rather than a line of text:

```csharp
public void Process(Order order)
{
    _logger.LogInformation("Processing order {OrderId}", order.Id);
}
```

The rule reports a `Trace.*` output call only when structured logging is actually available in the
compilation, so the suggestion is one the project can act on. It is gated at compilation start on
`Microsoft.Extensions.Logging.ILogger` resolving; a project that references no structured-logging abstraction
has nothing to migrate to and is never reported. The reported call is bound to confirm its containing type
really is `System.Diagnostics.Trace`, so a same-named method on another type is never reported.

## What this rule reports

Only the four legacy tracing output methods, and only when they bind to `System.Diagnostics.Trace`:

| shape | example |
| --- | --- |
| write a line | `Trace.WriteLine("...")` |
| write | `Trace.Write("...")` |
| conditional write | `Trace.WriteIf(flag, "...")` |
| conditional write line | `Trace.WriteLineIf(flag, "...")` |
| through `using static` | `WriteLine("...")` after `using static System.Diagnostics.Trace;` |

The following are never reported:

| shape | why |
| --- | --- |
| any `Trace.*` call when no `ILogger` is available | the suggestion would not be actionable |
| `Trace.TraceInformation`/`TraceError`/`TraceWarning`, `Trace.Flush`, â€¦ | not one of the four output methods this rule scopes to |
| a `WriteLine` on some other `Trace` type | the call does not bind to `System.Diagnostics.Trace` |
| `Debug.Write*` | see below |

## Why `Debug.Write*` is not reported

`System.Diagnostics.Debug`'s output methods carry `[Conditional("DEBUG")]`, so the compiler omits the entire
call from a release build. It never runs in production and is a debug-time aid rather than an application log
sink â€” a far weaker signal that structured logging was ever the intent. `Trace.*`, by contrast, is not
conditional and runs in release, so a `Trace.Write*` call genuinely is application output going through legacy
tracing, which is what this rule reports. Reporting `Debug.*` as well would nag legitimate debug-only tracing,
so this rule scopes to `Trace.*` only and leaves `Debug.*` silent.

## How to fix violations

Inject an `ILogger` (or `ILogger<T>`) and log through it with a message template, choosing a level and naming
the values so the sink records them as structured state:

```csharp
_logger.LogInformation("Processing order {OrderId}", order.Id);
```

There is **no code fix**: converting to `ILogger.Log*` needs a logger instance the analyzer cannot supply and
a message template with named placeholders the author must choose.

## Performance

The clean path is syntax only: every invocation is filtered to the four tracing method names before anything
binds, and the `ILogger` and `Trace` types are each resolved once per compilation. A project with no `ILogger`
available registers no per-node work at all.
