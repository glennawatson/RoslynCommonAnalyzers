# SST2601 — Logger field or property should follow the logger naming convention

| Item | Value |
| --- | --- |
| Category | Logging |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

An `ILogger` or `ILogger<T>` field or property is named against the conventional logger name, so loggers are
named inconsistently from one type to the next.

```csharp
public sealed class OrderService
{
    private readonly ILogger log; // SST2601: a private instance logger should be named '_logger'
}
```

## Rule description

A logger read consistently by name is a logger a reader finds instantly. The convention:

| Member | Expected name |
| --- | --- |
| a private instance field | `_logger` (or `_log`) |
| any non-private or static field or property | `Logger` |

A field or property typed as `Microsoft.Extensions.Logging.ILogger` or a constructed `ILogger<T>` is checked
against the pattern that applies to its accessibility and static-ness, and reported on its identifier when the
name does not match:

```csharp
public sealed class OrderService
{
    private readonly ILogger _logger; // matches the private instance convention
}

public abstract class ServiceBase
{
    protected ILogger Logger { get; } // matches the non-private convention
}
```

The rule is gated on `Microsoft.Extensions.Logging.ILogger` resolving in the compilation, so a project that
references no such abstraction is never reported and the rule costs it nothing. Each candidate is bound to
confirm its type really is the logger abstraction, so a same-named member of an unrelated type is left alone.

## What this rule reports

| shape | example | reported |
| --- | --- | --- |
| private instance field off-convention | `private readonly ILogger log;` | yes |
| private instance field on-convention | `private readonly ILogger _logger;` | no |
| private instance field on-convention | `private readonly ILogger _log;` | no |
| non-private field or property off-convention | `public ILogger MyLogger { get; }` | yes |
| non-private field or property on-convention | `protected ILogger Logger { get; }` | no |
| static logger not named `Logger` | `private static readonly ILogger _logger;` | yes |
| explicit-interface property | `ILogger IHasLogger.Tracer => ...;` | no (its name is fixed by the interface) |
| a same-named member of an unrelated type | `private readonly Other.ILogger log;` | no (not the logger abstraction) |

## Configuration

The accepted names for a private instance logger field are configurable. The value is a comma-, semicolon-, or
whitespace-separated list; the default is `_logger, _log`.

```ini
# .editorconfig

# Accept only '_logger' for a private instance logger field.
stylesharp.SST2601.fieldname = _logger
```

The non-private convention (`Logger`) is not configurable.

## How to fix violations

Rename the member to the conventional name for its accessibility — `_logger` for a private instance logger,
`Logger` for a non-private or static one — or set `stylesharp.SST2601.fieldname` to the convention your
codebase already uses.

There is **no code fix**: a rename ripples across every reference to the member and is a judgement the author
should make, not one the analyzer applies automatically.

## Performance

The clean path is syntax only: a field or property is skipped unless the simple name of its declared type is
`ILogger`, so nothing binds for the overwhelming majority of members. The logger abstraction is resolved once
per compilation, and the configured field name is read only after a member is confirmed to be a private
instance logger.
