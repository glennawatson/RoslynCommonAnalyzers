# SST2700 — A route template must not contain a backslash

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | Yes |
| Default | Enabled (where the ASP.NET Core routing types exist) |

## Cause

The route template of a routing attribute — a verb attribute deriving from
`Microsoft.AspNetCore.Mvc.Routing.HttpMethodAttribute` (`[HttpGet]`, `[HttpPost]`, …) or
`[Microsoft.AspNetCore.Mvc.RouteAttribute]` — contains a backslash.

```csharp
[HttpGet(@"api\products")] // SST2700
public IActionResult List() => Ok();
```

## Rule description

URL paths are separated by the forward slash `/`. A backslash `\` in a route template is a mistaken path
separator: ASP.NET Core routing treats it as a literal character inside a single segment, not as a boundary
between segments, so the endpoint is never matched by the intended request and the action becomes unreachable.

The rule inspects the *decoded* template value, so every way of writing the backslash is caught:

```csharp
[HttpGet(@"api\products")]     // verbatim string
[Route("api\\products")]        // escaped backslash
[HttpPost(template: @"a\b")]    // named template argument
```

The attribute is bound and only the argument that maps to the route-template parameter is examined. A backslash
in an unrelated argument — for example the route `Name` property — is left alone, because that value is not a URL
path:

```csharp
[Route("api/products", Name = @"legacy\name")] // not reported — Name is not the template
```

An escape sequence that decodes to something other than a backslash (a `\t` tab, for instance) is not a path
separator and is not reported.

### Availability detection

The rule is gated on the ASP.NET Core routing types resolving in the referenced framework (detected once per
compilation via `Compilation.GetTypeByMetadataName`). A project that does not reference ASP.NET Core MVC reports
nothing. No target-framework string is parsed.

## How to fix violations

Separate the route segments with a forward slash.

```csharp
[HttpGet("api/products")]
public IActionResult List() => Ok();
```

The code fix replaces every backslash in the template with a forward slash, re-emitting the corrected value as an
ordinary string literal.

## Performance

The clean path is a cheap syntactic scan: the attribute's arguments are checked for a string literal whose decoded
value contains a backslash before anything is bound. Only when such a literal is present is the attribute bound and
its route-template parameter resolved.
