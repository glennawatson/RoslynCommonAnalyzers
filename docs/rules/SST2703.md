# SST2703 — A route constraint must match the component parameter type

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A routable component's route template constrains a segment to one type, while the `[Parameter]` property of the
same name is a different type.

```csharp
using Microsoft.AspNetCore.Components;

[Route("/user/{id:int}")]
public class UserPage
{
    [Parameter]
    public string Id { get; set; }   // constrained to int, received as string
}
```

## Rule description

A typed route segment such as `{id:int}` does two things: it tells the router to accept the segment only when
it parses as that type, and it hands the parsed value to the component parameter of the same name. Those are two
halves of one contract, and when they disagree — `{id:int}` feeding a `string`, `{key:guid}` feeding an `int` —
the value either fails to bind or is coerced against the parameter's real type when the component is navigated
to. The template compiles (it is just a string), so nothing catches the mismatch until runtime.

The constraints mapped to a CLR type are:

| Constraint | CLR type |
| --- | --- |
| `int` | `int` |
| `long` | `long` |
| `guid` | `Guid` |
| `bool` | `bool` |
| `datetime` | `DateTime` |
| `decimal` | `decimal` |
| `double` | `double` |
| `float` | `float` |

Other constraints (`alpha`, `regex`, `length`, `minlength`, `min`, `range`, and the like) constrain the text
without fixing a numeric or temporal type, so they are not checked. Only typed `{name:constraint}` segments are
examined; an untyped `{name}` segment is ignored. A parameter whose type is the `Nullable<>` form of the
constrained type matches, since `{id:int?}` and an `int?` parameter agree.

### What is reported

A component carrying `[Route(...)]` whose template has a typed segment `{name:constraint}` — where the
constraint is one of the mapped keywords above — for which a `[Parameter]` property of the same name
(case-insensitive), on the component or a base component, has a CLR type that is neither the mapped type nor its
`Nullable<>` form. Catch-all (`{*name:constraint}`) and multi-constraint (`{name:int:min(1)}`) segments are
handled; the first constraint keyword fixes the type.

### What is not reported

- A segment whose constraint matches the parameter type, or its nullable form.
- An untyped segment, or a constraint the rule does not map to a CLR type.
- A typed segment with no matching parameter, or a same-named property that is not a `[Parameter]`.
- Anything in a project that does not reference the components assembly; the rule resolves both the route and
  parameter attribute types and stays silent when either is absent.

## How to fix

Make the constraint and the parameter type agree — usually by giving the parameter the type the route already
promised.

```csharp
using Microsoft.AspNetCore.Components;

[Route("/user/{id:int}")]
public class UserPage
{
    [Parameter]
    public int Id { get; set; }
}
```

## Configuration

None.
