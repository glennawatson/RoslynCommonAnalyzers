# SST2704 — An API controller action should declare an HTTP verb

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | No |
| Default | Enabled (where the ASP.NET Core MVC types exist) |

## Cause

A public instance method on a type annotated `[ApiController]` has no HTTP-verb attribute.

```csharp
[ApiController]
public class WidgetController : ControllerBase
{
    public IEnumerable<Widget> List() => _store.All(); // SST2704
}
```

## Rule description

A public method declared on a controller is treated as an action. When that action carries no HTTP-verb attribute
— none deriving from `Microsoft.AspNetCore.Mvc.Routing.HttpMethodAttribute`, such as `[HttpGet]` or `[HttpPost]`,
and nothing else that supplies the verbs — it answers **every** HTTP method. That is rarely intended: it widens
the surface (a `GET` action silently also handles `POST`, `DELETE`, and the rest), defeats method-specific
routing, and can produce an ambiguous-match failure at runtime against a sibling action that *does* declare a verb
on the same route.

The rule is deliberately narrow to keep false positives near zero:

- It reports only on types annotated `[ApiController]` (directly or through a base type).
- A method that already declares a verb — through a `[Http*]` attribute or any attribute that supplies the verbs
  (for example `[AcceptVerbs]`) — is not reported.
- A method marked `[NonAction]` is not reported.
- A static method, a property accessor, a constructor, an operator, a generic method, and an override inherited
  from `object` (`ToString`, `Equals`, `GetHashCode`) are all excluded — they are not actions.

```csharp
[ApiController]
public class WidgetController : ControllerBase
{
    [HttpGet]
    public IEnumerable<Widget> List() => _store.All();   // declares a verb — clean

    [NonAction]
    public Widget Normalize(Widget w) => w;              // opted out — clean

    public override string ToString() => "widgets";      // object override — clean
}
```

### Availability detection

The rule is gated on `Microsoft.AspNetCore.Mvc.ApiControllerAttribute`,
`Microsoft.AspNetCore.Mvc.ControllerBase`, and `Microsoft.AspNetCore.Mvc.Routing.HttpMethodAttribute` all
resolving in the referenced framework (detected once per compilation). A non-web project reports nothing.

## How to fix violations

Add the HTTP-verb attribute that matches what the action does, and give it its route:

```csharp
[HttpGet("widgets")]
public IEnumerable<Widget> List() => _store.All();
```

If the method is genuinely not an action — a helper that happens to be public — mark it `[NonAction]` or reduce
its accessibility.

**There is no automatic code fix**, because the intended verb (`GET`, `POST`, `PUT`, `DELETE`, …) cannot be
inferred from the method.
