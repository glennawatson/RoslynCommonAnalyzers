# SST2706 - Mark the Windows Forms entry point as single-threaded apartment

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | Yes (adds `[STAThread]`) |
| Default | Enabled |

## Cause

A program that references Windows Forms has an entry point `Main` that carries neither
`[System.STAThread]` nor `[System.MTAThread]`.

```csharp
public static class Program
{
    public static void Main() // SST2706
    {
        System.Windows.Forms.Application.Run(new MainForm());
    }
}
```

## Rule description

Windows Forms leans on COM for a set of everyday UI features - the clipboard, drag-and-drop, the common
file/color/font dialogs, and some ActiveX and shell integrations. COM's single-threaded apartment (STA) is
initialized on the thread that runs the program's entry point, and which apartment that thread joins is
driven by the apartment attribute on the entry point.

When `Main` declares no apartment, the runtime starts the thread in a multithreaded apartment, and those
COM-backed features fail or misbehave at runtime: a clipboard call throws, a file dialog hangs or returns
nothing, drag-and-drop silently does nothing. Because the failure only shows up when that feature is
exercised, a quick smoke test rarely catches it. Stating the apartment explicitly removes the ambiguity.

The rule fires only on the compilation's actual entry point, resolved through the compiler, and only when
`System.Windows.Forms.Application` is referenced - so a class library or a non-UI program is never reported.
A `Main` method that is not the entry point, or an entry point that already declares `[STAThread]` or
`[MTAThread]`, is left alone. Because the suggested attribute is a standard runtime type, the rule stays
silent (and the fix is never offered) on a target framework that does not provide it.

## How to fix

Mark the entry point with the apartment it needs. For Windows Forms that is the single-threaded apartment:

```csharp
public static class Program
{
    [System.STAThread]
    public static void Main()
    {
        System.Windows.Forms.Application.Run(new MainForm());
    }
}
```

The code fix inserts `[System.STAThread]` on its own line above `Main`. If a program genuinely needs the
multithreaded apartment, declaring `[System.MTAThread]` instead also satisfies the rule.

## When to suppress

Suppress only when the entry point does not host a UI message loop and none of the COM-backed features are
used - for example a console tool that merely references a Windows Forms assembly for one helper type. In
that case declaring an apartment is harmless, so adding the attribute is still the simpler resolution.
