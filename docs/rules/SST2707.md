# SST2707 - Do not capture HttpContext in a fire-and-forget Task.Run

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | No |
| Default | Disabled (opt-in) |

## Cause

Inside a type deriving from `Microsoft.AspNetCore.Mvc.ControllerBase`, a `Task.Run(...)` whose returned task
is discarded - not awaited, returned, or assigned - starts a delegate that captures the request's
`HttpContext` (or a value read from it).

```csharp
public class OrdersController : ControllerBase
{
    public IActionResult Submit()
    {
        Task.Run(() =>
        {
            var user = HttpContext.User; // SST2707 - runs after the request ends
            Audit(user);
        });

        return Ok();
    }
}
```

## Rule description

`HttpContext`, and the services hanging off it, are scoped to a single request. As soon as the response is
sent, the framework disposes those objects and may recycle the context instance for the next request.

A fire-and-forget `Task.Run` outlives the request that started it. By the time its delegate finally touches
the captured `HttpContext`, the context has been disposed or reused, so the background work reads torn state
or throws `ObjectDisposedException`. And because nothing observes the discarded task, that exception is
swallowed - the work silently fails with no trace in the response.

Offloading to `Task.Run` inside a request handler rarely helps anyway: it moves synchronous work onto another
pool thread without freeing the request thread, so it adds scheduling overhead without the throughput it
looks like it should buy.

The rule is a heuristic. It fires only on the discarded shape - the `Task.Run(...)` call is the whole
expression of a statement, or the right-hand side of a `_ = ...` discard assignment - so an awaited,
returned, or assigned task is never reported. The delegate must be a lambda or anonymous method that
references an `HttpContext`-typed value, and the enclosing type must derive from `ControllerBase`. The whole
rule is gated on both `HttpContext` and `ControllerBase` being referenced, so a non-web project pays nothing.

## How to fix

Copy the values the background work needs out of `HttpContext` into locals before starting the work, and
hand long-running work to a component with its own lifetime - a hosted background service or a queue - rather
than borrowing the request's scope.

```csharp
public class OrdersController : ControllerBase
{
    public OrdersController(IBackgroundQueue queue) => _queue = queue;

    private readonly IBackgroundQueue _queue;

    public IActionResult Submit()
    {
        var user = HttpContext.User; // read while the request is still alive
        _queue.Enqueue(() => Audit(user)); // owns its own lifetime

        return Ok();
    }
}
```

## Configuration

The rule is off by default because the fire-and-forget shape it keys on is a heuristic. Enable it in
`.editorconfig` where it fits your codebase:

```ini
[*.cs]
dotnet_diagnostic.SST2707.severity = warning
```

## When to suppress

Suppress when the captured value is safe to use after the request ends - for example, the delegate reads an
immutable value type out of `HttpContext` that was copied by value before the task starts, or the work is
guaranteed to complete before the response is sent. Prefer restructuring the code so the capture is not
needed at all.
