# SST2709 - Do not request a render while disposing

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A component calls `StateHasChanged()` from its `Dispose` or `DisposeAsync` body.

```csharp
using System;
using Microsoft.AspNetCore.Components;

public class Panel : ComponentBase, IDisposable
{
    public void Dispose()
    {
        StateHasChanged(); // SST2709
    }
}
```

## Rule description

Disposal runs while the renderer is tearing the component down: its place in the render tree is already
gone. Asking for a render at that point has nothing to render into, so the renderer rejects it and
throws. The call reads as a harmless refresh but faults at runtime every time the component is disposed.

The rule looks only at `StateHasChanged()` called on `this`/`base` from a method named `Dispose` or
`DisposeAsync` - including the `Dispose(bool)` pattern - and confirms the enclosing type is a component
and the call really binds to the framework's `StateHasChanged`. A call that sits inside a lambda or
local function is left alone, because that delegate may run at some later, valid point.

## How to fix violations

Remove the call. A component that is going away does not need to re-render; disposal should release
resources, not request UI updates.

```csharp
using System;
using Microsoft.AspNetCore.Components;

public class Panel : ComponentBase, IDisposable
{
    public void Dispose()
    {
        // release resources here; do not request a render
    }
}
```

## When not to use the rule

There is no supported case for requesting a render during disposal, so the rule has no intended
exceptions. If a diagnostic fires on a method named `Dispose` that is not really a disposal path,
rename it.
