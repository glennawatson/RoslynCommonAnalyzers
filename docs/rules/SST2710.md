# SST2710 - Marshal a timer callback's render request onto the dispatcher

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A component calls `StateHasChanged()` directly from a `System.Threading.Timer` or `System.Timers.Timer`
callback, instead of marshalling it through `InvokeAsync(...)`.

```csharp
using System.Threading;
using Microsoft.AspNetCore.Components;

public class Ticker : ComponentBase
{
    private Timer? _timer;

    protected override void OnInitialized()
    {
        _timer = new Timer(_ => StateHasChanged(), null, 0, 1000); // SST2710
    }
}
```

## Rule description

Both timer types fire their callback on a thread-pool thread, not on the renderer's synchronization
context. Touching component state from there races the renderer, and the renderer guards against exactly
that: it throws when a render is requested off its own dispatcher.

The rule works forward from the timer wiring - the callback delegate of a `new Timer(callback, ...)` and
the handler on a `timer.Elapsed += handler` - into the callback body, and reports every
`StateHasChanged()` call there that is not already an argument of an `InvokeAsync(...)` call. A lambda or
anonymous-method callback is scanned in place; a method-group callback is followed to its declaration
when that declaration is in the same file. The already-safe method-group form
`InvokeAsync(StateHasChanged)` is never flagged.

## How to fix violations

Route the render request through `InvokeAsync`, which marshals the work back onto the dispatcher before
the render is requested.

```csharp
using System.Threading;
using Microsoft.AspNetCore.Components;

public class Ticker : ComponentBase
{
    private Timer? _timer;

    protected override void OnInitialized()
    {
        _timer = new Timer(_ => InvokeAsync(StateHasChanged), null, 0, 1000);
    }
}
```

## When not to use the rule

If the callback never touches component state and the reported `StateHasChanged` is on an unrelated type
that merely shares the name, the diagnostic does not apply; the rule already confirms the call binds to
the framework's `StateHasChanged`, so this is rare.
