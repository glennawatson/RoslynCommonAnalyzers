# SST2713 — A DotNetObjectReference must be stored so it can be disposed

| Item | Value |
| --- | --- |
| Category | Frameworks |
| Severity | Warning |
| Code fix | No |
| Default | Enabled |

## Cause

A `DotNetObjectReference.Create(...)` result is passed straight into an interop call, or otherwise dropped,
instead of being stored where the component can dispose it.

```csharp
using Microsoft.JSInterop;

public class Map
{
    private readonly IJSRuntime _js;

    public Map(IJSRuntime js) => _js = js;

    public async Task InitAsync() =>
        await _js.InvokeVoidAsync("map.init", DotNetObjectReference.Create(this));
}
```

## Rule description

`DotNetObjectReference.Create` wraps a .NET object so JavaScript can call back into it, and the runtime holds
that wrapper alive in a per-circuit map, keyed by id, until the reference is disposed. A reference that is
created and passed inline — or created and discarded — is never assigned anywhere the component can reach, so its
`Dispose` is never called and the map entry lives for the whole life of the circuit. The component and everything
it captures leak, one navigation at a time.

Storing the reference in a field, passing that field to interop, and disposing it when the component is disposed
breaks the leak: the map entry is released and the wrapped object becomes collectable.

### What is reported

A `DotNetObjectReference.Create(...)` call whose result is dropped:

- passed inline as an argument to a method call (the common interop case),
- created as a bare statement, or
- assigned to a discard (`_ = ...`).

### What is not reported

- A reference assigned to a field or property — it is stored, and whether the owning type disposes it is covered
  by the disposable-ownership rules, not this one.
- A reference bound to a local, a `using` declaration, or a field initializer, or returned to the caller — in
  each case the value stays reachable, or ownership is handed off, so reporting it would be a false positive.
- Anything in a project that does not reference the JavaScript-interop assembly. The rule resolves
  `DotNetObjectReference` in the compilation and stays completely silent when it is absent.

## How to fix

Promote the reference to a field, pass the field to interop, and dispose it from the component's disposal method.
There is no automatic fix, because where the field belongs and how the type should dispose it are design
decisions.

```csharp
using System;
using Microsoft.JSInterop;

public class Map : IDisposable
{
    private readonly IJSRuntime _js;
    private DotNetObjectReference<Map>? _self;

    public Map(IJSRuntime js) => _js = js;

    public async Task InitAsync()
    {
        _self = DotNetObjectReference.Create(this);
        await _js.InvokeVoidAsync("map.init", _self);
    }

    public void Dispose() => _self?.Dispose();
}
```

## Configuration

None.
